Registry

Every signed artifact, content-addressed.

A public index of .kolm artifacts published with their receipts. Any engineer can resolve a CID to a receipt body that an auditor can verify offline, without trusting us, without trusting the publisher, and without downloading the model weights.

What lives in the registry

Receipt CIDs only, not weights. The registry is a lookup table from artifact CID to the JSON receipt that the compiler signed. Storage is small (a few KB per entry); trust is large (the receipt names every input that produced the artifact).

Fields in every registry entry

artifact_cid
The content-addressed hash of the .kolm zip. Anyone with the artifact file can recompute and compare.
base_model
The open-weight base the student distilled from, plus its SHA-256 (for example Qwen/Qwen2.5-7B-Instruct).
teacher
The teacher endpoint that produced training data: a frontier model id, a self-hosted vLLM URL, or self-distill for trace-based distillation.
eval_pack
The CID of the eval pack the K-score was computed against. The pack is itself content-addressed and reproducible.
k_score
Composite numeric (0–100) plus the five axes (faithfulness, coverage, calibration, honesty, cost).
signed_at
UTC timestamp when the compiler signed the receipt.
public_key
Ed25519 fingerprint of the signing key. Verifiers match this against the artifact's embedded key.
publisher
Tenant id of the team that published; opaque hash unless they opt to publish a display name.

Search

Filter on base model, teacher, eval pack, K-score thresholds, or freeform tags. Search runs against the public index; you can pipe results into kolm pull to fetch any receipt for offline verification.

# find recent finance artifacts from a Qwen2.5 base with K above 85
$ kolm registry search "qwen2.5 base finance K>85" --limit 10

# JSON output for piping
$ kolm registry search "qwen2.5 base finance K>85" --json | jq '.results[].artifact_cid'

# resolve and verify each match without ever fetching weights
$ kolm registry search "qwen2.5 base finance K>85" --json \
  | jq -r '.results[].artifact_cid' \
  | xargs -n1 kolm verify --cid

Query syntax accepts base:qwen2.5-7b, teacher:claude-opus-*, eval:ks-finance-*, K>85, honesty>=90, since:2026-05-01. Tags carry over from kolm publish --tag.

Resolve a CID

Three equivalent paths. All return the same signed receipt body; the verifier is happy with any of them.

CLI · pull

$ kolm pull bafkreieq...4f7c
→ saved to ~/.kolm/receipts/
$ kolm verify --cid bafkreieq...4f7c
ok · signature valid
ok · K-score 94.1 ship

Pulls the receipt body and validates the Ed25519 signature against the embedded public key fingerprint. Offline once cached.

Web · verify route

https://kolm.ai/verify/<cid>
→ renders the receipt JSON
→ shows base, teacher,
   eval pack, K-score
→ links to publisher

Human-readable view for compliance reviewers. Same data; renders verdict pill and the five K-score axes inline.

HTTP · curl

$ curl -s kolm.ai/v1/registry/<cid>
{
  "version": "kolm-receipt-1",
  "artifact_cid": "bafkrei...",
  "k_score": { ... }
}

Raw JSON for automation. Cacheable, content-addressed; CDN-friendly. No auth required for resolves.

Publish to the registry

Opt-in only. The compiler never publishes for you; you run a single command after you have read your own receipt.

# publish a built artifact (receipt CID only; weights stay on your disk)
$ kolm publish ./artifacts/finance-q2.kolm
ok artifact_cid bafkreieq4f7c...
ok registry    https://kolm.ai/registry
ok verify      https://kolm.ai/verify/bafkreieq4f7c...

# publish with tags and a public display name
$ kolm publish ./artifacts/finance-q2.kolm \
    --tag finance --tag q2-2026 \
    --display "Acme Finance Triage v3"

  • What gets uploaded

    PAYLOAD · receipts only

    The Ed25519 signature, the receipt JSON body (a few KB), and your chosen tags. The .kolm zip stays on your filesystem; the model weights never leave your network unless you also push them to your own object store.

  • Why publishing is optional

    OFFLINE-VERIFY · always

    Anyone with the artifact zip can still verify offline with kolm verify path/to/artifact.kolm. The registry just makes a receipt discoverable; it never changes whether an artifact is verifiable.

Takedown policy

Anyone who can prove ownership of the signing key can request the receipt CID be removed from the public index. The receipt itself remains verifiable as long as someone holds the artifact zip.

  • Filing a takedown

    HOW-TO · live

    Email trust@kolm.ai with the receipt CID and a signature over a takedown nonce we provide. The CID drops from /registry search and the /v1/registry/<cid> resolve within 24 hours.

  • What takedown does not erase

    HONEST SCOPE · provenance survives

    The receipt is signed and content-addressed. A downstream verifier who pulled it before takedown still has a valid receipt body; kolm verify on their copy still passes. Takedown removes discovery, not provenance.

  • Policy-driven removal

    POLICY · live

    We will remove a CID for malware, sanctions-list publishers, or court order. The receipts policy is in trust. We never remove for an unfavorable K-score; that is the verifier's job to read.

  • Takedown receipts

    AUDIT-TRAIL · tombstone

    Every takedown writes a tombstone receipt with the date and reason class. Auditors can prove the CID was once present and what category of removal applied without the receipt body being re-served.

Federation

The public registry at kolm.ai/registry is one mirror. Enterprise customers run private mirrors that are protocol-compatible with the public one; the same CLI talks to both.

Public

kolm.ai/registry

Anyone can resolve any published CID. Free, rate-limited per IP. The CDN in front is content-addressed, so cache hits are cheap. Backed by an object store with object-lock to keep receipts immutable.

  • No auth for resolves
  • Rate-limited 60 req/min/IP
  • Receipts cached at the edge
Private mirror

Self-hosted registries

For regulated and air-gapped teams. Runs as a static service inside your VPC, behind your SSO. Receipts you publish do not propagate to the public mirror unless you opt in per-CID. Federation across mirrors is on the roadmap.

  • Terraform module in enterprise
  • S3 or GCS-compatible storage
  • Compatible with kolm pull --registry <url>

Honest scope

The registry is a discovery index, not the source of truth. The source of truth is the signed receipt embedded in the artifact zip. We want anyone, including someone who does not trust kolm.ai, to be able to verify the chain.

  • The receipt is the source of truth

    TRUTH-OF-RECORD · embedded

    Each .kolm zip carries its own signed receipt. kolm verify path/to/artifact.kolm validates the receipt against the embedded public key, recomputes the artifact hash, and checks the K-score axes. The registry adds nothing the artifact does not already prove.

  • What we do not promise

    NON-CLAIM · plumbing not adjudication

    The registry does not host model weights. It does not run the eval pack for you (use kolm verify --recompute for that). It does not certify a K-score is honest; only an independent re-run does. It is plumbing, not adjudication.
