kolm.ai  /  drift detection + supersession chain

A frozen artifact does not stay correct forever.

A compiled artifact is frozen; production is not. kolm tracks staleness, scheduled rebuilds, drift reports, and supersession as hash-bound manifest blocks so a verifier can prove when an artifact should keep running or be retired.

Lifecycle ladder M+1 through M+4 · reviewed 2026-05-17 · wave 171 dedicated surface

The four-tier lifecycle ladder.

Four lifecycle tiers keep deployed artifacts reviewable over time. Each one is opt-in, additive, and adds a new manifest block rather than mutating an existing one. Older artifacts continue to verify; when a lifecycle block is absent, the verifier reports an informational pass instead of breaking legacy deployments.

M+1 · shipped

Stale notification

Artifact age vs ship date with a configurable age threshold. The compile-time manifest carries ship_date; the verifier surfaces a stale pill when the artifact is older than the recipe's stale_after_days. Cheap heuristic, no traffic required.

YES · manifest.ship_date + recipe.stale_after_days
M+2 · shipped

Re-distillation cadence

Configurable per-recipe cadence (weekly, monthly, quarterly) that schedules routine rebuild even in the absence of detected drift. Cadence emits a scheduled_rebuild supersession block on every cycle so the chain stays continuous.

YES · recipe.cadence + supersession reason='scheduled_rebuild'
M+3 · wave 167

Drift detection cron

Snapshot the current K-score axes, compare against a baseline snapshot, emit one signal per axis with status within / drift / breach. Cron config block emits a crontab line a tenant drops into their scheduler. Verifier check #24 gates ship on verdict.

YES · manifest.drift_report + verifier check #24
M+4 · wave 167

Supersession chain

The successor's manifest carries supersession_provenance naming the predecessor's artifact_hash, the retirement reason, the supersession_date, and (when reason='drift_detected') the breached drift signals inlined. A verifier walks the chain backward to genesis.

YES · manifest.supersession_provenance + verifier check #23

The two halves are tightly coupled by design. A drift detection job produces a DriftReport; when the report's status is breach, a tenant re-distills the recipe and ships a new artifact carrying a SupersessionBlock that names the predecessor's artifact_hash plus the reason (drift_detected) plus the failing drift signals. A third-party verifier walking the chain can confirm: the predecessor existed and verified at the time it was superseded, the new artifact is a legitimate successor, and the reason given for supersession is accompanied by enough evidence to justify retiring the predecessor.

The six supersession reasons.

A successor declares exactly one reason for retiring the predecessor. The enum is frozen so a verifier can reason about the chain without parsing free text. Block hashes bind into artifact_hash_input.supersession_hash.

reasonwhen this firesevidence required
drift_detectedM+3 detected breach; predecessor's K-score axes moved past tolerance against a tenant baseline.drift_signals array or drift_report_hash (one or both); builder rejects without evidence.
scheduled_rebuildM+2 cadence triggered routine re-distillation. No drift detected; just the calendar.None beyond the cadence config block already on the manifest.
security_patchUpstream model or runtime CVE; retire stale weights. Independent of drift.notes recommended (CVE id + advisory URL); authorized_by recommended.
recipe_revisionspec.json changed shape: new input field, new output schema, new redactor class. Not a pure retrain.notes recommended (link to spec diff); changed spec.json hash is captured separately in the new artifact's seeds/recipe chain.
policy_changeLegal or compliance policy update: new redactor class added, license terms shifted, regulator request.authorized_by recommended (compliance-ops identifier); notes recommended (policy reference).
tenant_requestExplicit tenant or operator command to retire the artifact (e.g., wrong recipe shipped, customer requested rebuild).authorized_by recommended (operator identifier); notes recommended (ticket reference).

Reason-specific guard: reason === 'drift_detected' REQUIRES either drift_signals or drift_report_hash. The builder throws at compile time; the verifier rejects at ingest. The retirement claim is auditable.

Default drift tolerances.

Every drift signal classifies its delta against two thresholds: warn (movement worth flagging) and fail (movement worth retiring the artifact). Tolerances are configurable per recipe; the defaults below ship in src/drift-supersession.js as the frozen DEFAULT_TOLERANCES table.

axiswarn bandfail bandcomparison mode
eval_score± 0.02± 0.05numeric delta (normalized scalar in [0, 1])
k_score.composite± 0.03± 0.08numeric delta (sum of weighted axes; tolerates 1e-4 float noise across architectures)
k_score.<axis>± 0.05± 0.10numeric delta (every named axis R/F/E/Z/T/A/S/L/C/V present in both snapshots)
external_holdout_hashexact match requiredany byte change in the external corpus identity is a breach
tenant_shadow_corpus_hashexact match requiredany byte change in the tenant's residency-bounded shadow corpus identity is a breach
artifact_hashinformationala new artifact ALWAYS has a new hash; reported but does not gate ship

Tolerances pass through to detectDrift(baseline, current, tolerances) and are persisted into the drift_report manifest block so a verifier replaying the comparison can confirm the same thresholds were used.

The CLI flow, end to end.

Three commands compose the lifecycle: kolm drift detect compares two artifacts, kolm drift cron emits the scheduler config, kolm compile --supersession-of retires the predecessor with the drift evidence bound in.

01 · detect drift

kolm drift detect

Loads two .kolm artifacts, snapshots their manifests, runs the comparator, prints a signal table tagged [BREACH] / [drift] / blank. Writes a hash-stamped drift-report-v1 JSON. Exits 2 on breach so a CI step fails loud.

$ kolm drift detect acme-claims-may.kolm \
 --baseline acme-claims-jan.kolm \
 --out drift-report-2026-05-17.json

02 · schedule the cron

kolm drift cron

Validates the cadence (5-field crontab expression, regex per field), builds a drift-cron-v1 config block, emits the crontab line. The wrapped kolm drift detect exits non-zero on breach so cron mail surfaces the alert without needing a webhook.

$ kolm drift cron \
 --baseline ./baselines/acme-claims-genesis.kolm \
 --current ./builds/acme-claims-latest.kolm \
 --cadence "0 6 * * MON" \
 --out drift-cron.json \
 --alert-webhook https://hooks.acme/health \
 --alert-on breach,drift

 config: drift-cron.json
 hash: bafyabcdef12…

 # crontab line emitted, drop into /etc/cron.d/kolm-drift
 0 6 * * MON kolm drift detect "./builds/acme-claims-latest.kolm" \
 --baseline "./baselines/acme-claims-genesis.kolm" \
 --out "./reports/drift-$(date +%Y-%m-%d).json"
03 · ship the successor

kolm compile --supersession-of

Loads the predecessor, extracts artifact_hash + cid, builds a supersession-v1 block, embeds it in the new artifact's manifest. When --supersession-reason=drift_detected, the breached + drifted signals are inlined alongside drift_report_hash so the supersession is self-contained.

$ kolm compile acme-claims.spec.json \
 --supersession-of acme-claims-jan.kolm \
 --supersession-reason drift_detected \
 --supersession-drift-report drift-report-2026-05-17.json \
 --supersession-authorized-by compliance-ops-2026 \
 --supersession-notes "Q2 distribution shift, ICD-10 2026 update"

The receipt chain, in order.

Lifecycle blocks append after auditor attestation. A verifier can walk from the current artifact back to genesis and prove the predecessor, supersession reason, evidence, and chain integrity.

spec → seeds → split → train → recipes → evals → external_holdout → tenant_shadow → auditor_attestation → supersession → drift_report → export → signatures → rekor
  • supersession · appended after auditor attestation. Binds into artifact_hash_input.supersession_hash. Reason-specific evidence guard enforced at builder and re-checked at verifier.
  • drift_report · appended after supersession. Binds into artifact_hash_input.drift_report_hash. Embedded snapshots are themselves validated drift-snapshot-v1 objects with their own hashes.
  • artifact_hash · the final hash over the canonical artifact_hash_input object (all upstream block hashes concatenated in declared order). Any tamper drifts artifact_hash and breaks every downstream signature including the Rekor transparency log entry.
  • Backward compatibility · every pre-Wave-167 artifact lacks both lifecycle blocks. Verifier checks #23 (a) and #24 (a) emit informational pass with upgrade hints. No legacy .kolm regresses; both checks only activate when the artifact opts in at compile time.

Why two checks, not one.

Supersession and drift are separate receipts. That lets auditors distinguish a routine rebuild, a warning-band drift event, and a breach-driven successor without reading one overloaded status.

scenario A

Scheduled rebuild, no drift

A cadence-triggered weekly rebuild ships a new artifact carrying supersession with reason scheduled_rebuild (no drift evidence required) plus a drift report for the same period (verdict within). The binder shows both green; the auditor sees the chain stays continuous even when nothing moved.

supersession: pass · scheduled_rebuild drift_report: pass · within

scenario B

Warn-band movement, ship anyway

Drift detection finds movement past the warn band but inside the fail band. The artifact ships with a drift report verdict of drift. Check #24 passes with a WARN prefix and the hint "schedule a re-distillation against current production traffic before the next cadence interval." No supersession required.

supersession: pass · absent drift_report: pass · WARN drift

scenario C

Breach, retire and ship successor

Drift detection finds fail-band breach on at least one axis. The cron exits 2; the alert fires. A new artifact is built with --supersession-of + --supersession-reason=drift_detected + --supersession-drift-report. The breached signals are inlined; the predecessor's artifact_hash is bound in.

supersession: pass · drift_detected + evidence drift_report: pass · within (new)

scenario D

Forged supersession (rejected)

An attacker tries to ship a successor claiming drift_detected without attaching evidence. The builder throws at compile time with supersession.reason='drift_detected' requires drift_signals or drift_report_hash. If the attacker hand-crafts the manifest to skip the builder, the verifier rejects at ingest with the same message and the recomputed block hash mismatch.

supersession: fail · missing evidence

Plain scope: what kolm ships, what the tenant owns.

kolm ships the manifest blocks, verifier semantics, and scheduler scaffolding. The tenant owns the actual cron, Airflow, k8s, or GitHub Actions schedule.

kolm ships

Manifest blocks + verifier semantics + scheduler scaffolding

  • src/drift-supersession.js module with four spec-version constants, six supersession reasons, default tolerance table, builder + validator pairs, comparator, snapshot extractor, disk round-trip helpers, crontab-line emitter
  • CLI kolm drift <detect|cron|verify> + compile-time flags --supersession-of, --supersession-reason, --supersession-drift-report, --supersession-authorized-by, --supersession-notes, --drift-report
  • Verifier checks #23 Supersession chain + #24 Drift report, both emit on every artifact (legacy informational pass, post-actionable)
  • Manifest blocks supersession_provenance and drift_report bound into artifact_hash_input so any tamper breaks every downstream signature including the Rekor anchor
tenant owns

The scheduler + the redistillation pipeline + the alert routing

  • The OS scheduler that runs the cron line on cadence: cron, systemd timer, k8s CronJob, Airflow, GitHub Actions schedule, tenant picks
  • The build pipeline that runs kolm compile with the new seeds and the supersession flag pair when a breach fires
  • The alert routing: cron mail to operators by default; optional webhook + alert-on filter for tenants who prefer push
  • The baseline policy: which artifact is "the baseline," how often it rotates, whether the baseline itself ages out

Why cron config emission, not a daemon. kolm does not own a long-running scheduler process. Every regulated tenant already has one. kolm drift cron builds a hash-stamped drift-cron-v1 config block and emits the crontab line; the tenant drops the line into their scheduler. The wrapped kolm drift detect exits non-zero on breach so cron mail delivery surfaces the alert to the operator without needing a webhook. The webhook + alert-on filter exist for tenants who prefer push notification but they are layered on top of the same exit-code contract, not replacements for it.
