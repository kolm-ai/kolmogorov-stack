kolm.ai  /  training

Training, distillation, compilation. One evidence path.

Distill top models into deployable specialists.

Start with approved examples, teacher traces, or captured AI calls. Kolm chooses the smallest method that passes the holdout: rule, synthesized rule, distilled student, LoRA/QLoRA, quantized bundle, or composed specialist set. Same CLI, same receipt chain, same signed .kolm artifact.

The training ladder

Each rung is a tier. The CLI is the same. The artifact gets bigger; the trust chain gets longer; the K-score gets calibrated against an ever-larger holdout.

K-score figures are typical holdout outcomes after Q+2 train/holdout gating, not promises. Your seeds set the ceiling.

Six tiers, one CLI

Each tier names the smallest recipe class that can pass its K-score gate. You can always stay at a lower tier if your seeds are clean and your task is narrow.

What it is. A hand-authored JavaScript function. Same input always returns the same output. There is no model file because there is no model. Five seeds is enough when the task is regex-shaped: phone-number normalization, SSN redaction, ICD-10 lookup, denial-code classification with explicit rules.

How you make one. Write your seeds. Hand the spec to kolm compile. The CLI normalizes the legacy {prompt, completion} shape, splits 80/20 by row hash, lets you author the rule against the train pairs, then scores it against the held-out 20% as independent ground truth. K-score is exact-match unless the recipe declares a different comparator.

kolm seeds bootstrap phone-normalizer
kolm compile recipes/phone-normalizer.spec.json --class rule
kolm verify dist/phone-normalizer.kolm
kolm run dist/phone-normalizer.kolm "+1 (415) 555-2671"

Where it runs. Any JavaScript runtime. Node, Bun, browser, Cloudflare Workers, Deno Deploy, V8 sandbox inside a Postgres procedure. The artifact is a zip with a manifest and a function; no model weights, no GPU dependency, no platform-specific binaries.

What you get. Sub-millisecond latency. Bit-identical output across machines. Bundle so small it can ship inside a JSON column. Receipt chain: seeds_hash → train_hash → holdout_hash → recipe_hash → eval_result_hash. Ed25519-signed.

tier 1 Synthesized rule
data50 seeds
classsynthesized_rule
artifact120–400 KB
latency1–3 ms
teacherrequired
K-score A0.85–0.97
LLM-authored code AST-validated no weights

What it is. A frontier model emits candidate JavaScript that satisfies the spec. The candidate is parsed into a constrained AST (no I/O, no FFI, bounded loops), fuzzed against the train split, and gated on K-score against the holdout. The final artifact contains rule code, not weights, but the rule code was authored by an LLM, not by a human. Verifiable distinction: this is rule code generation, not knowledge distillation.

Why a step up from tier 0. Fifty seeds give the synthesizer enough coverage to find edge cases (e.g., extension numbers, vanity letters, international prefixes) that a five-seed hand-author misses. The teacher_vendor + teacher_model lands in the manifest so a verifier can replay.

kolm compile recipes/denial-code-classifier.spec.json \
 --class synthesized_rule \
 --teacher anthropic:claude-sonnet-4-6
kolm verify dist/denial-code-classifier.kolm
# manifest.lineage.source = "synthesis"
# manifest.lineage.teacher_vendor = "anthropic"
# manifest.lineage.teacher_model = "claude-sonnet-4-6"

Where it runs. Same JavaScript runtimes as tier 0. The artifact is still pure code; the teacher only touches the build pipeline, never inference time.

What you get. Single-digit-millisecond latency. Deterministic output across machines. AST guarantee: provably terminating, no I/O side effects, bounded resource use. Manifest carries teacher provenance + train/holdout hashes.

tier 2 Distilled, sub-billion
data500 seeds
classdistilled_model
artifact250–500 MB
latency20–80 ms
teacherfrontier API
K-score A0.88–0.94
real weights INT4 quantized on-device

What it is. A real neural network. The teacher (Claude Sonnet, GPT-5-mini, Gemini Flash) generates training pairs from your train split, the PHI redactor strips Safe Harbor identifiers before the call, the reinjector restores them after. A small open-weight base (Qwen2.5-0.5B, SmolLM2-360M, or Llama-3.2-1B) is LoRA-fine-tuned, GPTQ/AWQ-quantized to INT4, and exported as GGUF.

When it earns its keep. When the task has too many edge cases for a rule to cover (free-text classification, intent extraction, summarization with strict structure) and you want the artifact to run on the user's device. iPhone 15 Pro, M3 Mac, Jetson Orin Nano, all run a quantized 0.5B inline.

kolm distill recipes/prior-auth-extractor.spec.json \
 --teacher anthropic:claude-sonnet-4-6 \
 --student-base qwen2.5-0.5b \
 --distillation-method lora \
 --redact phi \
 --eval-gate 0.88 \
 --quantize int4-awq

Where it runs. Onnxruntime, llama.cpp, candle. Web via WASM build. Mobile via CoreML/NNAPI export. The same .kolm contains every target.

What you get. Sub-100ms first-token on a phone. Receipt chain extension: redaction_map_hash → teacher_call_log_hash → reinjection_log_hash → teacher_vendor → teacher_model → student_base → distillation_method. Verifier check #15 rejects the artifact if any of those four lineage fields are missing.

What it is. Same pipeline as tier 2, larger base. Llama-3.2-1B, Llama-3.2-3B, Qwen2.5-1.5B, Phi-3.5-mini, or Gemma-2-2b. Training shards run in parallel; adapter weights are merged before quantization. The student now has enough capacity to match the teacher on the long tail of your distribution.

When it earns its keep. When tier 2 misses on rare cases and you have the seed volume to teach those cases. Documents with multi-page context, transactions that need cross-row reasoning, summaries that must hold style across paragraphs.

kolm distill recipes/claims-summarizer.spec.json \
 --teacher anthropic:claude-opus-4-7 \
 --student-base llama-3.2-1b \
 --distillation-method lora \
 --lora-rank 16 \
 --redact phi \
 --eval-gate 0.92 \
 --quantize int4-gptq

Where it runs. M3 Mac, Jetson Orin AGX, RTX 4090 desktop, server-class CPU with AVX-512. Same artifact runs every target via runtime detect.

What it is. QLoRA on a 7B-class open-weight base (Qwen2.5-7B, Mistral-7B-v0.3, Yi-1.5-6B). Multi-GPU training; checkpoint sharding to fit in 24GB cards. The student now closes the A/T fidelity ratio above 0.97 on most tasks. This is the tier that replaces a frontier API call entirely.

When it earns its keep. When the per-call frontier-API spend dwarfs the one-time distillation spend, and you need the result inside your own perimeter. Health insurers, banks, government, regulated SaaS.

Where it runs. 24GB consumer GPU, dual H100 server, on-prem accelerator pool. Vendor-neutral: same artifact runs on RTX 4090, MI300X, Gaudi 3, TPU v5e via ONNX.

data4× tier-3
classcomposed
artifact2–6 GB
latencyrouted < 300 ms
teacherper-expert
K-scoreper-expert + composite
router composed one bundle

What it is. The Kimi/Qwen pattern: many small experts, each LoRA-fine-tuned on a narrow slice, glued together with a learned router so the bundle behaves like a much larger model. Each expert is itself a tier-2 or tier-3 distill. kolm compose links them, fuzzes the router on the union of holdouts, and emits a single .kolm.

When it earns its keep. Health-insurance back-office (PA + denial coding + eligibility + appeals), legal triage, finance KYC. Different tasks, different teachers, different bases, one artifact that ships under one signature.

kolm compose dist/bundle.kolm \
 --expert phi-redactor=dist/phi-redactor.kolm \
 --expert denial-coder=dist/denial-coder.kolm \
 --expert prior-auth=dist/prior-auth.kolm \
 --expert eligibility=dist/eligibility-resolver.kolm \
 --route-by class \
 --router-train holdouts/router.jsonl

Where it runs. Same targets as the heaviest expert it contains. The router itself is a sub-millisecond classifier; the dispatch cost is dominated by the activated expert.

The receipt chain, same shape at every tier

Each box hashes the box before it. The verifier walks the chain backward and refuses on any break. Wave 156 made the build step itself reproducible; kolm verify --rebuild recompiles from the recorded toolchain and confirms binary_hash matches.

Frontier methods

The six methods below are the post-training techniques the kolm pipeline runs in production. The full literature dump, with citations to every paper, repo, and vendor blog, lives at /research/methods-2026-q2. The status pill in the last column links each method to where it is wired in src/.

method what it does sample efficiency status in kolm
Knowledge Distillation
Hinton et al., 2015		 Train a smaller student model on teacher logits (KL on soft targets at temperature T). Order-of-magnitude smaller student with comparable accuracy on in-distribution tasks. shipped src/tune.js
QLoRA (4-bit NF4 + LoRA)
Dettmers et al., NeurIPS 2023		 Fine-tune large open-weight bases on consumer GPUs by quantizing the base to 4-bit and adapting via LoRA. LIMA shows 1,000 high-quality pairs can beat 50,000 mediocre ones; 65B fits a single 48 GB GPU. shipped scripts/tune-step.py
On-Policy Distillation
Thinking Machines, Oct 2025		 Student samples its own trajectories; teacher returns dense per-token reverse-KL supervision on those rollouts. Closes the off-policy gap for chat and reasoning; O(N) bits per episode vs. RL's O(1). shipped src/distill-onpolicy.js
Preference distillation (SimPO / KTO / ORPO)
2024 - 2025		 Distill RLHF-style preferences without an explicit reward model; SimPO and ORPO drop the reference model entirely. Comparable to PPO + reward-model with roughly 10x less compute; KTO works on binary thumbs. shipped src/distill-preference.js
Speculative decoding (EAGLE-3 / Medusa)
Li et al. + Cai et al., 2024		 Inference-time speedup: a small draft head proposes 1..k tokens; the base verifies and accepts in parallel. 2 to 6x throughput on autoregressive decode with no accuracy loss; KL between target and draft is the objective. shipped src/spec-decode.js
Self-Instruct + Evol-Instruct
Wang et al. 2022; Xu et al., ICLR 2024		 Synthetic instruction-data bootstrap: hand-seed 175 instructions, prompt a teacher to evolve them via depth and breadth operators. Self-Instruct: 175 seeds expand to 52k pairs (Stanford Alpaca). WizardLM (Evol-Instruct) beat ChatGPT by 7.9 pts win-rate on hard prompts. shipped src/synthesis.js

Full citations, anchor papers, and the deferred methods (SpQR, Variational Spec Decoding, Quiet-STaR rationale pretrain) live at /research/methods-2026-q2. The source module name links each method to the path where it ships.

How much data do I actually need?

Volume requirements scale with recipe class, not with use case. A class-0 EDI parser ships with zero seed examples; a class-3 letter generator needs at least a thousand. The numbers below come from the frontier-methods sweep; full citations and the dataset references live at /research/methods-2026-q2.

CLASS_RANK 0

rule

lower bound
0 examples. Deterministic transform; seeds.jsonl can be empty.
sweet spot
10 to 30 examples just for eval cases (regression detection on the holdout).
upper bound
100 examples = comprehensive eval suite; more does not change the rule, only the K-score precision.

Worked example. An EDI 837 parser or an MLR-rebate calculator: 0 seed examples because the transform is deterministic. pack.rules covers the IG dictionary; the holdout exists only to catch regressions.

CLASS_RANK 1

synthesized_rule

lower bound
30 to 100 examples. Enough to cover the edge cases the teacher must satisfy during synthesis; Magpie or Evol-Instruct can grow these if the human seed is small.
sweet spot
200 to 500 examples. HEDIS measures and CMS Star contributors land here; AST validation + verifier loop catches mistakes the seeds did not.
upper bound
2,000 examples. Past this, you are mostly fuzzing coverage on coded combinations the teacher already handles.

Worked example. The HEDIS CBP measure and the CMS Star contributor extractor: teacher emits a JS rule, AST-validated against the constrained DSL, K-score-gated on the holdout. No weights ever materialize.

CLASS_RANK 2

compiled_rule

lower bound
Same as synthesized_rule. The native-compile step does not change data requirements; it lowers an existing rule to C / Rust / WASM.
sweet spot
Same as synthesized_rule plus a benchmark suite so kolm verify --rebuild can confirm binary_hash against the pinned toolchain.
upper bound
Same as synthesized_rule. Speed-up is from native code, not from more data.

Worked example. Any rule or synthesized_rule whose hot path needs native speed (per the wave-163 compiled-rule class). Differential tests against the source rule are the gate; the compiled artifact ships alongside the source so a verifier can rebuild.

CLASS_RANK 3

distilled_model

lower bound
1,000 high-quality examples. The LIMA finding (Meta, NeurIPS 2023) is the floor for getting meaningful capability via fine-tune; sub-1k drifts into few-shot territory.
sweet spot
10,000 to 50,000 examples. UltraFeedback (OpenBMB, ~64k preference pairs) and Tulu-3 (Allen AI, ~939k SFT) bracket the production-grade range.
upper bound
1M examples. Past this, base-model selection and data quality matter more than raw volume; you are paying teacher tokens for diminishing K-score gains.

Worked example. A denial-appeal-letter or prior-auth-letter generator, or the phi-redactor companion: LoRA on teacher pairs, INT4-quantized, GGUF-exported, shipped as one signed .kolm.

Quality beats volume past a floor. LIMA (Meta, NeurIPS 2023) showed 1,000 carefully curated examples can outperform 50,000 mediocre ones. The kolm K-score gate at compile time enforces this in code: a recipe trained on 50,000 bad seeds will score worse on the holdout than one trained on 1,000 good ones, and the gate will refuse the build before the artifact is signed.
