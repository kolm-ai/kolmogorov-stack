Continuous integration

A K-Score gate for every pull request.

Add kolm-ai/quality-gate-action@v1 to your workflow. The action compiles the new model, runs your eval set, compares against the K-Score on main, and fails the PR if it regresses past the floor.

Drop in

One YAML block. Two required inputs. Done.

Save as .github/workflows/kolm-gate.yml. The action installs the kolm CLI, runs the compile + eval, and posts the K-Score diff as a PR comment.

Workflow file
.github/workflows/kolm-gate.yml

One job, three steps: checkout, run gate, report. The action self-installs the kolm CLI so the runner needs nothing pre-baked.

Required inputs
api-key + eval-set

api-key from secrets.KOLM_API_KEY, eval-set a path to your eval.jsonl in the repo.

Optional gate floor
min-k-score: "0.90"

Hard floor below which the action fails the PR. Default is the K-Score recorded on the merge base (regression-only gate).

Optional runner offload
cloud-compile: "true"

Compile happens on a kolm-managed RunPod GPU, eval runs on the GitHub runner. Useful for > 13B without a self-hosted runner.

The full workflow

Copy this. Replace the eval set path. Ship.

Triggers on pull-request, runs once per PR, reports as a PR comment + a status check named kolm-gate.

GitHub Actions

.github/workflows/kolm-gate.yml

name: kolm-gate
on: [pull_request]
jobs:
  gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: kolm-ai/quality-gate-action@v1
        with:
          api-key:     ${{ secrets.KOLM_API_KEY }}
          eval-set:    eval/support-tickets.jsonl
          min-k-score: "0.90"
          recipe:      recipes/support-bot.toml
          # cloud-compile: "true"   # uncomment to offload > 13B builds
What the gate produces

A PR comment, a status check, and a signed receipt.

All three reference the same artifact SHA, so a green merge is the same artifact the fleet later runs.

1

PR comment

in: PR · out: K-Score diff comment

Markdown table: K-Score base vs head, latency p95 base vs head, top 5 eval items that regressed.

2

Status check

in: gate result · out: kolm-gate ✓ / ✗

Set as required in branch protection. Blocks merge on regression. Same shape as a test or lint check.

3

Signed receipt

in: gate run · out: rcpt_01KZ...

Pinned to the .kolm SHA. The fleet's kolm fleet deploy can --require-gate-receipt so only PR-gated artifacts roll.

Beyond GitHub

Same gate, three other runners.

The action wraps kolm gate run. That command is the gate — call it from any CI you already use.

GitLab CI
.gitlab-ci.yml job

Single job that runs kolm gate run --eval-set eval.jsonl --min-k-score 0.90. Exit code = gate result.

CircleCI
orb: kolm-ai/quality-gate@1

Same inputs as the GitHub Action, packaged as a CircleCI orb command.

Buildkite
kolm-gate-pipeline.yml

Plugin invocation. Buildkite's annotations render the same K-Score diff comment as a build annotation.

Bare shell
kolm gate run --json

Returns JSON with K-Score base, K-Score head, gate result, receipt id. Pipe into anything.
