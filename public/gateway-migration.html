Gateway logs are training data

Your gateway is sitting on a distill corpus.

Portkey, Helicone, LiteLLM, OpenRouter — export the traces, run kolm import, ship a model that does the same job at one-tenth the inference cost. The gateway already captured the inputs, the outputs, the model tags, and the request metadata. That is the dataset. kolm reads it, distills a student against your chosen teachers, and hands back a portable artifact you run on your own GPU.

Four importers, no glue code.

Each importer reads the gateway's native export, normalizes the request/response pairs into a kolm capture corpus, and skips malformed rows with a per-row error log. Zero-dep Node, runs from any shell. The output is a single JSONL file that downstream distill specs and Cerebras teacher runs consume without further transformation.

Portkey

Shipped

Portkey Logs export (JSONL). Reads request body, response choices, model tag, and latency — emits capture pairs keyed by trace id.

$kolm import --from portkey ./logs.jsonl

Helicone

Shipped

Helicone Requests CSV/JSONL export. Pulls prompt and completion fields, model column, and the custom properties you set on the call.

$kolm import --from helicone ./logs.jsonl

LiteLLM

Shipped

LiteLLM proxy callback logs (JSONL) or DB dump. Handles the OpenAI-compatible shape and the proxy's metadata block.

$kolm import --from litellm ./logs.jsonl

OpenRouter

Shipped

OpenRouter activity export (JSONL). Carries the routed model id and the upstream provider tag so multi-teacher attribution survives.

$kolm import --from openrouter ./logs.jsonl
End to end

From log to distilled model in five commands.

One copyable block. Swap helicone for any of portkey, litellm, or openrouter. Resumable at every step, signed receipts at every step.

terminal · bash Copy & run 
# 1. Import the gateway log as a kolm capture corpus.
$ kolm import --from helicone ./helicone-logs.jsonl

# 2. Draft a distill spec: base model + teachers.
$ kolm distill spec

# 3. Collect teacher completions over the corpus.
$ kolm distill collect

# 4. Train the student (LoRA or full-merge).
$ kolm distill train

# 5. Deploy the portable artifact to your runtime of choice.
$ kolm deploy

Why gateways alone are not enough.

A gateway routes, meters, and logs. It does not produce a model you own. The three rows below are the gap that kolm closes — cryptographic provenance via signed receipts, a portable artifact that survives outside any vendor, and training on the hardware you already pay for.

Feature Gateway kolm
Model lineage receipts Trace id, request log; no per-call signature Ed25519-signed receipt on every distill, eval, and inference call
Portable artifact None — routes to upstream APIs HuggingFace weights, GGUF (Q4_K_M / Q5_K_M / Q8_0), Ollama Modelfile, vLLM, TGI, TRT-LLM
BYOC training Not in scope Train on your GPU (RTX 5090, H100, A100), your cloud, or air-gap — no callbacks

What we don't import yet.

The four importers above cover the gateways customers asked for first. The list below is what is not in the box today.

  • ·Kong AI Gateway — plugin-shaped Kong logs. Not in tree.
  • ·TrueFoundry and custom in-house gateways — bring a schema, we will quote an importer.
