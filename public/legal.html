How privilege stays intact

What makes this work-product-safe.

No vendor-speak. The exact mechanism for each property, with the CLI verb you can verify yourself when you install kolm.

01

No document text leaves the firm at run time

The .kolm runs on the partner's laptop or the firm's secure compute. Counterparty drafts, deposition transcripts, and matter files stay on the device that opened them. There is no inference API to log, throttle, or subpoena.

02

The teacher model never sees production matter

Compile uses your gold examples (de-identified or synthetic on the first pass, real once the firm signs off). The teacher sees only the training set during compile. Production matter is never sent to a third party.

03

Per-matter artifacts respect conflict walls

Corporate redline artifacts never see litigation files; litigation discovery artifacts never see corporate matter. Distribution is via the firm's MDM. The artifact has no cross-matter memory because it has no remote storage.

04

Frozen model state, defensible at deposition

The model weights, recall index, and recipe pack are pinned inside the artifact at compile time. The firm controls when the artifact updates. If opposing counsel asks which model produced a memo, the answer is a sha256 with a date, not a vendor release note.

05

Ed25519 receipt per output

Every run writes a content-addressed receipt: input hash, output hash, recall sources, model identity. The receipt lives in the firm's matter-management system. Re-verification is byte-exact on the same artifact.

06

Encryption at rest and in transit

TLS 1.3 in transit. AES-256-GCM at rest with a per-tenant key in your KMS (AWS KMS, GCP KMS, Azure Key Vault). Keys rotate quarterly by default; you can force rotation any time.

07

Right-to-erasure and matter close-out

kolm erase --matter matter_id removes the matter from captures, eval sets, and the audit-log searchable index (the immutable log retains the redaction receipt, not the data). Compatible with the firm's close-out policy and outside-counsel guidelines.

Why legal AI is different

Duties cloud AI cannot satisfy by default.

Cloud-LLM bans at firms are usually not vendor friction. They are a mismatch between confidentiality duties and a third-party inference API that may log prompts.

A

ABA Model Rule 1.6 confidentiality

A lawyer must not reveal information relating to representation. A cloud prompt is a disclosure to a third-party processor, and most firms cannot get the informed consent required for every matter touched. Architecture that keeps document text on firm hardware sidesteps the consent question entirely.

B

Work-product doctrine

Mental impressions, legal theories, and case strategy live in the prompts associates write. If those prompts persist on a vendor's logging plane, the firm has a discoverability problem that did not exist before the tool was adopted. A local artifact has no remote log.

C

Conflict and matter walls

Information barriers between matters are an ethical obligation, not a feature flag. Cloud inference does not respect firm walls by default. A compiled artifact per matter type runs only against the documents the lawyer hands it and writes its receipt to the firm's matter-management system.

Honest scope

What kolm does, and what we tell every firm up front.

No "AI lawyer" claims, no replacement narrative. Read this before the rest of the page becomes interesting.

A

Kolm does not practice law

The CLI does not give legal advice. Our distilled artifacts redline, extract, summarize, structure, and prioritize. Anything that crosses into legal judgment is the lawyer's call, the lawyer's name on it, and the firm's regulatory path.

B

We do not provide a privilege opinion

Whether a specific deployment preserves privilege in a specific jurisdiction is a question for the firm's GC and ethics counsel. We document the architecture, the data flow, the receipts; we do not write your privilege memo.

C

Models will be wrong sometimes

Our distilled models drop K-score by under 3 points after quantization. They still hallucinate, miss edge cases, and exhibit bias. Every kolm workflow ships with a defer-on-low-confidence path and a human-in-the-loop gate by default; turning that off is a choice the firm makes on purpose with a partner's name on it.

D

We will not let you ship a bad artifact

Compile enforces a K-score ship floor against your hold-out evaluation. If the artifact does not meet it, the compile fails. Override is available, but the override is logged with the user, timestamp, and signed reason, in the same audit log opposing counsel will eventually look at.

Reference architectures

How firms actually deploy kolm.

Same product, several placements. Pick the one that matches your IT estate and your matter-management workflow.

A. Partner laptop

MDM-distributed signed artifact

Single signed .kolm file deploys via Intune, Jamf, or Workspace ONE. The artifact runs locally with zero network calls. Receipts persist to the matter file via NetDocuments or iManage when the laptop next reaches the network.

$ kolm run nda-redline.kolm \
--in counterparty.docx \
--out redlined.docx \
--matter MAT-2026-0123
B. Firm-hosted compute

Inside the firm's cloud account

Helm chart installed in the firm's AWS / GCP / Azure account. Residency pinned to your selected region (US, UK, EU, APAC). Matter files flow through your existing VPC peering; kolm never sees an external network.

helm install kolm kolm/legal \
--set residency=us-east-1 \
--set kms.arn=arn:aws:kms:... \
--set egress=audit-only
C. On-prem with DMZ

Bare metal in the firm's data center

kolm runs in the firm's data center on Kubernetes or bare metal. The only outbound path is the audit-log stream to the firm's SIEM. No vendor cloud is in the path. Outside-counsel guidelines stay green.

$ kolm install --on-prem \
--bundle kolm-2026-05.tar.zst \
--signature kolm-2026-05.sig \
--egress audit-only
D. In-house legal SaaS

Dedicated SaaS, per-client tenancy

If you are an ALSP or in-house team serving many clients each with their own confidentiality posture, we give you a dedicated single-tenant environment per client. You sign a master DPA; clients sign downstream DPAs with you.

A real matter walkthrough

NDA redline, from intake to signed output.

Concrete example. A senior associate receives an inbound NDA from a counterparty on a corporate matter. The firm has a redline policy compiled as nda-redline-v2026.02.kolm. Here is the run.

01

Intake

Associate opens the matter file in NetDocuments. The counterparty draft (12.4 KB .docx) is checked into the matter. Conflict check is already cleared by the firm's intake workflow. No documents have left the firm boundary.

02

Compile (done once, last quarter)

The redline policy was compiled from 142 historical NDAs reviewed and signed off by the corporate practice group. K-score 0.81 cleared the ship floor of 0.74. The artifact is pinned in the firm's MDM and distributed to every partner laptop.

03

Review

Associate runs kolm run nda-redline-v2026.02.kolm --in counterparty.docx --out redlined.docx. The artifact returns a tracked-changes file with 23 markup proposals: indemnity scope tightened, IP-assignment carve-out added, MFN deleted, term capped at 24 months. Recall surfaces the playbook section behind each change. Runtime 14.2 seconds on M3 Pro, zero network calls.

04

Ship

Partner reviews each tracked change, accepts or rejects as a human lawyer, and sends the redlined draft back to the counterparty under the firm's letterhead. The receipt (receipt-9b110d56.json) attaches to the matter file. If the deal closes and the agreement is later challenged, the firm can re-verify which policy version produced which markup on which date.

Common questions

What partners and GCs ask first.

Direct answers. If the answer is "depends on the firm's GC," we say that.

Privilege waiver

Does running kolm on client documents waive privilege?

That is a determination for the firm's GC and ethics counsel. The architecture is designed to avoid the third-party-disclosure event most privilege-waiver analyses flag. The .kolm runs on firm-controlled hardware. No document text is transmitted to a third-party API at run time. The receipt is local. We provide the architecture and the audit trail; the privilege opinion is the firm's call.

Protective orders

Can the firm use this for matters under a protective order?

Most protective orders restrict disclosure to third parties and require secure handling. A local artifact does not transmit document content to anyone. The firm should still review the specific order language with the GC. We provide a deployment that produces receipts the firm can attach to the matter record.

Discovery

Can opposing counsel demand the model?

The firm produces what was used. Because every run writes a receipt with the artifact's sha256, the firm can identify exactly which model state generated which output and provide that artifact for inspection if required. The model is not a remote service the firm cannot freeze in time.

Training

Does the firm need to train its own model?

No. Compile uses a base model the firm selects (Qwen, Llama, Mistral, or a frontier model via API key at compile time only). The firm's gold examples and playbook tune the artifact during specialization. The teacher model sees the training set during compile, not production matter at run time.

Conflicts

How does this fit conflict checks and matter walls?

Per-matter or per-practice-group artifacts are the default deployment pattern. A corporate redline artifact never sees litigation files; a litigation discovery artifact never sees corporate matter. Distribution is handled by the firm's MDM. The artifact has no cross-matter memory because it has no remote storage.

Model deprecation

What does the firm do when a frontier model deprecates?

Nothing immediately. The artifact's behavior is frozen at the version compiled. Frontier deprecations affect future compiles, not in-flight artifacts. The firm decides when to recompile. Old artifacts remain re-verifiable indefinitely because the model state is inside the .kolm.
