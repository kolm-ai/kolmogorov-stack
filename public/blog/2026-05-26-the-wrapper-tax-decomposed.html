The gateway overhead, decomposed.

2026-05-26 · ~10 minute read

Every gateway adds latency. The question is how much, where it goes, and what you get in exchange. We ran 10 identical prompts through the kolm Wrapper against Anthropic in production and decomposed every phase. Here is the breakdown: 703 milliseconds of overhead on a 2478 millisecond call, made up of about 250 milliseconds of pipeline work that buys you the audit trail and about 450 milliseconds of one cross-host hop you can eliminate in the self-hosted configuration.

The headline number

10 identical prompts, two legs, run live against https://kolm.ai with claude-haiku-4-5. The prompt was, "In two short sentences, explain what an LLM gateway does." Wall clock, milliseconds:

Legp50p95mean
Direct (teacher proxy to Anthropic)165026421774
Kolm gateway to Anthropic (managed)189977042478
Kolm gateway to local trinity-500 (projected)124014501240

The mean delta is 703 milliseconds, or about 40 percent over a direct call. The p95 tail (7704 ms) is a Vercel cold start on the proxy function and clusters near 1500 to 2200 ms once the function is resident. The local artifact leg is faster than direct because there is no provider round trip at all.

Where every millisecond goes

The Wrapper does seven things per call. Here is the per-phase shape on the managed configuration (Vercel-proxied so Railway does not need provider keys), with the upstream call (chain_dispatch) sitting in the middle as the only phase that does the real model work.

So of the 703 ms of overhead, roughly 250 ms is the pipeline (the work that buys you the audit trail and the capture row) and roughly 450 ms is one extra cross-host network hop that exists only on the managed kolm.ai because Anthropic credentials live on Vercel and the gateway runs on Railway.

Three pipeline phases worth a closer look

PII pass

Input is scanned before it leaves your machine: regex sweeps for emails, phones, SSNs, credit card patterns, IPs, and named entities (with an optional spaCy backend when high-fidelity is needed). Mode is configurable per namespace: detect (log only), redact (replace with tokens), block (return 4xx), or off. About 50 ms on input, slightly more on output because output is usually longer. PII events land in a separate log so you can prove the redaction happened on a given call without grepping receipts.

Receipt sign

Every call gets one kolm-audit-1 receipt: 19 canonical fields in a fixed key order, hashed with sha256, signed with Ed25519. About 25 ms because Ed25519 is genuinely fast and the canonical-ordering step is one Object.keys sort. The signing key id is stamped on the receipt so you can rotate without breaking old verifications. GET /v1/verify/<receipt_id> returns the receipt plus a verification result with no auth required.

Capture write

The request, response, route decision, confidence, and capture-eligible flag are written to your namespace lake as one row. About 40 ms because we batch flushes; the row is queryable inside a second. The capture-eligible flag is the one that matters: every fallback (frontier had to take over because local confidence was low) gets capture_eligible:true, and that is what you query when you build the next distill set.

The Vercel hop is optional

The 400 to 500 ms cross-host hop is a deployment choice, not a fundamental cost. Two configurations:

Configurationchain_dispatch (mean)gateway overhead (mean)end-to-end
Railway-direct (provider key on the gateway host)~1500 to 1800 ms~10 to 50 ms~1550 to 1850 ms
Vercel-proxy (managed kolm.ai)~1900 to 2200 ms~400 to 700 ms~2300 to 2900 ms

If you self-host or deploy your own kolm with provider keys on the same machine, you pay the pipeline cost (10 to 50 ms) and nothing else. The managed deployment proxies through Vercel because the Vercel function holds the customer's Anthropic and OpenAI keys (Railway is treated as untrusted for provider keys to keep the threat model tight). Setting ANTHROPIC_API_KEY directly on Railway removes the hop.

The Wrapper does not mark up tokens. The 12-token output delta (572 vs 560) is per-call provider variance, not a fee. When work routes to a local artifact, marginal cost goes to zero on a per-call basis; you pay for the electricity that runs the GPU, which on a 5090 is roughly $140 a month at residential rates for the equivalent throughput.

What the 703 ms buys you

  • One Ed25519 receipt per call, signed and verifiable, including on upstream failures.
  • PII redaction on input and output with separate event logs.
  • Namespace-aware routing with confidence-gated fallback.
  • A capture row per call with capture_eligible:true on every fallback. That flag is the next distill set.
  • Provider-key isolation: managed customers do not push their Anthropic or OpenAI key to the Railway gateway. The Vercel proxy holds the key and the gateway never sees it.
  • Receipt verification at the edge: GET /v1/verify/<receipt_id> works with no auth, so a third party (auditor, customer, regulator) can verify any receipt you produced.

What is next

The follow-up wave instruments every dispatch with a per-phase latency_breakdown attached to the receipt as an additive top-level field (not covered by the signature, so existing receipts still verify clean). Once that ships to production, the table above moves from "estimated from the leg total" to "measured per call". The two prod-deployed dashboards (gateway and bench) will show the breakdown live.

Two follow-ups we expect to ship inside V1:

  • Railway-direct as a managed option. Customers who would rather we hold the provider key on the same host (and skip the hop) can flip a switch and lose the 400 ms. The tradeoff is the same key now sits on Railway too; some compliance teams prefer that, some do not.
  • Receipt batching. When the gateway sees a burst of calls inside a few milliseconds, it can sign one receipt that covers the batch with a Merkle root over the individual call hashes. The per-call sign cost falls toward zero. The first version of this is in the draft.
