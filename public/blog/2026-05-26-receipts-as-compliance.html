Receipts as compliance: kolm-audit-1 and what auditors do with it.

2026-05-26 · ~9 minute read

Every gateway call in kolm emits one Ed25519-signed receipt under the kolm-audit-1 schema. 19 fields, deterministic key order, verifiable at the edge with no auth. The receipt turns "we ran an AI call" into "we ran a specific call, with this input hash, against this provider, with this output, signed by this key." Here is the schema, the signing process, and what SOC 2, HIPAA, and EU AI Act auditors do with it.

The schema

The full kolm-audit-1 receipt has 15 required fields and 5 optional fields, ordered for canonical signing. Required fields fail the receipt if missing; optional fields can be present-but-null (for example, artifact_id is null when the route decision was frontier):

FieldTypeRequired
schemastring literal "kolm-audit-1"yes
receipt_idulid prefixed with rcpt_yes
timestampRFC 3339 UTCyes
namespace_idcustomer-scoped stringyes
route_decision"local" or "frontier"yes
provideranthropic, openai, etc.yes
modelmodel slug, e.g. claude-haiku-4-5yes
artifact_idsha256-prefixed kolm idoptional (set on local routes)
confidencefloat [0,1]optional (set on local routes)
fallback_reasonenum: low_confidence, class_mismatch, upstream_timeout, upstream_429, upstream_5xx, poison_detected, or nulloptional
input_hashsha256:<16-hex> of request bodyyes
output_hashsha256:<16-hex> of response bodyyes
capture_eligiblebooleanyes
capture_idulid of capture lake rowoptional (set when capture_eligible)
redaction_appliedarray of PII categories redactedoptional
input_tokensintyes
output_tokensintyes
cost_usdfloat, 6 decimal placesyes
signing_key_id32-hex public key fingerprintyes
verify_urlfull URL to /v1/verify/<receipt_id>yes

The signature_ed25519 sub-object holds the algorithm name, the full public key (PEM), and the base64url signature. It sits at the end of the wire receipt and is excluded from the canonical-signing payload (the signer hashes everything else, signs that, then appends the signature).

How the signature works

Ed25519 was chosen for three reasons: signatures are 64 bytes (fits anywhere), signing is about 25 ms per receipt (fast enough for every call), and the verification primitive is in every modern crypto library (Node, Python, Rust, C, Go, WebCrypto) so a third party never needs a kolm-specific tool.

The canonical payload is built by sorting the receipt keys (according to ALL_FIELDS), serializing as compact JSON, and hashing with sha256. The hash is the input to ed25519.sign. Verifiers re-canonicalize the same way, hash, and pass to ed25519.verify with the public key from the receipt. One byte different, signature fails.

A worked example:

# Fetch the receipt (no auth required)
curl https://kolm.ai/v1/verify/rcpt_01KYC1ZVTGDCW3FX06JQSC

# Response includes the full receipt plus a verification result
{ "ok": true,
  "receipt_id": "rcpt_01KYC1ZVTGDCW3FX06JQSC",
  "receipt": {
    "schema": "kolm-audit-1",
    "route_decision": "frontier",
    "provider": "anthropic",
    "model": "claude-haiku-4-5",
    "input_hash": "sha256:02c6ce4fdf6bb19eafc04ff687a9efce",
    "output_hash": "sha256:843ac01149cced785dfebd0028d3b03b",
    "signing_key_id": "c2e942e24061e9314790d73ce7b3f351",
    "signature_ed25519": {
      "alg": "ed25519",
      "public_key": "-----BEGIN PUBLIC KEY-----\nMCowBQYDK2VwAyEA...",
      "signature": "DkBAG4fUoMtElFvx_iKWOKenRUKNsqdDGvjfFOdmiNrvi2DITI..."
    }
  } }

Verify with any Ed25519 library; the kolm SDKs include a one-call helper that fetches and re-verifies locally so CI can fail closed on a tampered receipt.

Why every call

The temptation is to receipt only the "important" calls. Auditors care about completeness, integrity, and verifiability. Skipping receipts on some calls breaks completeness. The discipline costs about 25 ms of signing plus 40 ms of capture write (see the gateway-tax post) and is worth it.

Receipt-on-failure is part of the contract: when upstream returns 5xx, the gateway still writes a receipt with fallback_reason: "upstream_5xx" and zero-token cost. Auditors specifically ask about failed calls; an audit trail that goes dark on failure is a suspect trail.

What SOC 2 auditors do with them

SOC 2 Type 2 audits cover five trust services criteria; receipts map to three:

  • Security (CC6, CC7). The receipt proves the call was authenticated (it carries the namespace tied to the bearer), authorized (route_decision reflects the namespace policy), and logged (the receipt is the log). Key rotation is recorded in the signing-key audit log.
  • Processing integrity (PI1). Input and output hashes let an auditor sample receipts, request the matching capture rows, hash them, and confirm match. Tampering is detectable.
  • Confidentiality (C1). The redaction_applied array names which PII categories were redacted on a given call. Combined with the PII event log, this proves email, SSN, credit card, etc. were scrubbed before the call left the gateway.

kolm assurance export --workspace produces an SOC 2 control map (each control with the receipts or other evidence demonstrating it) that an auditor ingests directly.

What HIPAA auditors do with them

Two touchpoints. First, the BAA-signed relationship requires PHI flowing through the gateway is either redacted at ingress or processed under the Privacy Rule. The redaction_applied field plus the PII events log proves the redaction. Second, the Security Rule asks for audit controls (164.312(b)) and integrity controls (164.312(c)(1)); the signed receipt covers both.

The HIPAA mapping page at /hipaa-mapping lists every Privacy and Security Rule citation, the kolm feature that addresses it, and the receipt field or event auditors can sample. Typical sample: "show me 30 random calls from your support namespace last month with verification and PII redaction proof." One CLI call:

kolm receipts sample --namespace support --last 30d --count 30 \
  --include-pii-events --verify --format pdf

What EU AI Act auditors do with them

Two articles touch the gateway directly:

  • Article 12 (record-keeping). High-risk systems must automatically record relevant events over the lifetime of the system, with technical features that ensure data integrity. The kolm receipt is that record; the Ed25519 signature is the integrity feature.
  • Article 13 (transparency). Providers must supply technical documentation including input, output, model, and routing logic on demand. The receipt names the provider, model, input hash (which keys to the capture row holding the full prompt), output hash, route decision, and confidence. kolm assurance export --framework eu-ai-act bundles the receipts with the model card and evidence DAG into the Annex IV pack.

Article 26 (logging by deployers) extends the obligation to the customer running the gateway. The same receipts satisfy both parties on the same artifact.

Three implementation choices that matter

Hashes, not bodies. The receipt carries input_hash and output_hash; the bodies live in the capture lake under your control. Auditors verify that the hashes match the bodies you produce on request; unredacted bodies are not replicated to the receipt store.

Public key per receipt. The public key travels on the receipt. A verifier never fetches a separate key endpoint and never trusts DNS for it. Key rotation is signaled by a new signing_key_id; old receipts verify against the old key embedded in them.

Verify URL on the receipt. Every receipt carries verify_url pointing back to /v1/verify/<receipt_id>. That endpoint requires no authentication. An external auditor can paste any receipt into a browser and get the verification result. Same pattern Sigstore certificates use for software supply chains.

What it does not do

Receipts prove a call happened with a given input and produced a given output; they do not prove the output was correct. K-Score evaluation and the capture-row review queue cover correctness. Receipts do not encrypt anything either; they are signed and integrity-protected, but the payload is open. Confidentiality is the capture lake's job.

They are also not a regulatory pre-clearance. An auditor accepting receipts depends on the auditor agreeing that the receipt covers the control. The control map in kolm assurance export bridges receipts to controls; certification still requires an auditor in the loop.

Caveats. The kolm-audit-1 schema is V1; we expect to ship a V2 with batched-receipt support (Merkle root over a small window of calls, one signature) inside the next two waves. Existing receipts will continue to verify under V1 forever. The Ed25519 keypair is generated per tenant at signup; rotation is supported but not yet automatic on a schedule. SOC 2 Type 2 attestation is in progress with our auditor and is not the same thing as the assertion that the receipts technically satisfy the controls.
