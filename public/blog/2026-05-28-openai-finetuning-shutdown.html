Migration · OpenAI fine-tuning

OpenAI is sunsetting fine-tuning. Here is how to land on open-source.

2026-05-28 · ~7 minute read · kolm.ai engineering

OpenAI has closed self-serve fine-tuning to new users and will close it to existing fine-tuning customers in January 2027. If you have a fine_tuning_data.jsonl in production today, you have roughly seven months to land somewhere that will still exist in February 2027. This post walks the six-command migration into kolm: import the dataset, generate a distill spec, collect council responses, train a smaller student model, deploy it. The artifact runs on a GPU you own, exports to GGUF and vLLM, and carries a signed receipt for every training run.

What OpenAI announced

OpenAI's fine-tuning surface is being wound down in two phases:

  • Now. Self-serve fine-tuning is closed to new users. The client.fine_tuning.jobs.create(...) SDK call returns an access-denied error for accounts that did not already have a job history.
  • January 2027. Existing fine-tuning customers lose the ability to create new jobs. Already-trained models will continue to serve for a deprecation window but no further training is possible.

If your product has a custom GPT-4o-mini or GPT-3.5 ft variant in the loop, the production path forward is either (a) accept that the model freezes in early 2027 and goes stale against your evolving data, (b) move the workflow to a managed competitor that is one policy change away from the same fate, or (c) land on an open-source distillation stack that ingests your existing training JSONL and emits a portable artifact you control.

kolm is built for option (c). The migration takes one CLI invocation per stage, six total, and the dataset shape you already have works as input without rewrites.

The migration in six commands

Each block below is the literal command. Outputs and exit codes are documented in the recipe; this post explains what each step is for.

Step 1 of 6 · Import the OpenAI fine-tuning dataset

kolm import --from openai-finetune ./fine_tuning_data.jsonl \
  --out ./captures.jsonl \
  --namespace support

The importer at src/importers/openai-finetune.js parses both the chat-completions format ({"messages":[...]}) and the legacy completions format ({"prompt":..., "completion":...}). Malformed rows are skipped with a per-row log line; if the failure rate crosses a threshold the CLI exits non-zero so a bad dataset cannot silently shrink. Output is a kolm capture lake JSONL: the same shape Studio reads from real gateway traffic.

Step 2 of 6 · Generate a distill spec

kolm distill spec \
  --namespace support \
  --base qwen2.5-7b-instruct \
  --teachers claude-opus-4-7,gpt-4o,cerebras:llama-3.3-70b \
  --output ./distill-spec.json

The spec captures the council composition (which teachers, which weights), the base student, the LoRA hyperparameters, and the evaluation block. The same spec is reproducible across machines and gets bundled into the final artifact's passport so a future auditor can reconstruct the run.

Step 3 of 6 · Collect council responses

kolm distill collect \
  --spec ./distill-spec.json \
  --captures ./captures.jsonl \
  --out ./training-pairs.jsonl

For each row in the imported dataset, the collector runs the three teachers in parallel through the kolm gateway. The Cerebras Llama-3.3-70B teacher is included because it is the cheapest fast option (sub-second latency on Cerebras hardware, see cerebras-teacher); Claude and GPT-4o anchor the council against frontier reasoning. The merged training pairs land in one JSONL with provenance fields per row.

Step 4 of 6 · Train the student

kolm distill train \
  --spec ./distill-spec.json \
  --pairs ./training-pairs.jsonl \
  --out ./support-v1.kolm

LoRA r=16 on Qwen2.5-7B at bf16 with gradient checkpointing fits on a single 5090 (or a $0.39/hr cloud GPU on RunPod or Modal). The CLI signs the resulting bytes with your Ed25519 signing key and bundles the spec, the K-Score evaluation block, and the runtime passport into a single .kolm tarball.

Step 5 of 6 · Evaluate parity against the OpenAI baseline

kolm eval \
  --artifact ./support-v1.kolm \
  --baseline-jsonl ./fine_tuning_data.jsonl \
  --metric agreement-rate \
  --out ./eval-report.json

Before cutting over, the eval harness runs the same 50-row fixture against the distilled artifact and the OpenAI ft baseline (or against the dataset's expected responses if the ft endpoint is no longer reachable) and reports agreement rate. The data/eval-fixtures/openai-ft-compat.jsonl seed set is shipped with the CLI; you replace it with your real holdout for the production cutover.

Step 6 of 6 · Deploy

kolm deploy ./support-v1.kolm \
  --target vllm \
  --target gguf-q4km \
  --target ollama

One artifact, three deploy targets. vLLM for a server, GGUF Q4_K_M for laptops and edge, Ollama Modelfile for the desktop runtime. The OpenAI-compatible /v1/chat/completions endpoint stays the same shape so your client code does not change.

What you do not lose

The migration is intentionally shape-preserving:

  • Your dataset. The JSONL stays in OpenAI's chat-completions or completions shape; kolm reads both natively. No manual conversion script.
  • Your prompt templates. System prompts, few-shot examples, and structured-output schemas import verbatim. The distill spec records them as-is.
  • Your eval set. If you already had a holdout split for the OpenAI ft job, that file plugs directly into kolm eval --baseline-jsonl.
  • Your inference contract. The deployed artifact serves OpenAI-compatible /v1/chat/completions. Existing client code keeps working with one environment variable change.

What you gain

  • Portability. GGUF (Q2 through Q8 plus IQ family), Ollama Modelfile, vLLM config, TGI config, TRT-LLM engine, single-binary edge build. The artifact runs anywhere a 7B fits.
  • Signed receipts. Every training run, every distill, every eval, every inference call emits one Ed25519-signed receipt under the kolm-audit-1 schema. Verifiable at the edge with no auth. See receipts as compliance.
  • BYOC. Bring your own cluster. The whole stack is Apache-2.0 and runs in your VPC on Modal, RunPod, Colab, or a 5090 under your desk. The managed kolm.ai is identical code; it just runs the boxes for you.
  • No January 2027 cliff. Open-source code does not get sunset by a vendor policy change. If kolm.ai disappeared, your .kolm artifact would keep running on your hardware indefinitely.
  • Council distillation. The OpenAI ft API only distilled from one teacher (the base model you fine-tuned). kolm's council blends Claude, GPT-4o, DeepSeek-R1, and Cerebras Llama-3.3-70B by configurable weights. Trinity-500, our first published artifact, used this pipeline to beat Claude Haiku 4.5 and tie GPT-4o-mini on the customer-support rubric at half the response length.

Caveats

The distilled student model is smaller than GPT-4o, which means it will not match the base frontier model on every task. The eval harness in step 5 exists for exactly this reason: it tells you which workflows in your dataset are safe to migrate now (high agreement rate against the ft baseline) and which need a larger base model or more training pairs first.

In practice we see customer-support, document-QA, and classification distills land at >95% agreement on the first run. Open-ended generation, code synthesis, and multi-step agent workflows usually need a larger base (Qwen2.5-32B or Llama-3.3-70B) and a second council pass to converge.

Order of operations. Run step 5 (eval) before step 6 (deploy). The eval report shows you per-row deltas: which prompts the student matched, which it diverged on, and how far. Cutting over to a distilled model without running eval is the single fastest way to regress production quality.

Get started

Three entry points:

  • /openai-migration — the migration landing page with the full feature comparison against OpenAI ft, Vertex Tuning, Bedrock Custom Models, and Azure OpenAI.
  • /docs/recipes/openai-finetune-migration — the reproducible recipe with annotated outputs, exit codes, common failure modes, and verification.
  • github.com/kolm-ai/kolm — the source. Apache-2.0. Issues open if your dataset has a shape the importer does not yet cover.

If you want a walkthrough on a real migration before you commit, the managed kolm.ai has a free tier that runs the first 20 inference calls per IP per day without signup at /quickstart. The CLI itself is free to install and self-host; the managed surface is optional.

Caveats. The seven-month estimate is read off OpenAI's published deprecation calendar as of 2026-05-28; the exact cutoff for existing customers may shift by a few weeks either way. The "$0.39/hr cloud GPU" figure is the RunPod RTX 4090 community-cloud rate at the time of writing; spot pricing varies. Agreement-rate numbers above are aggregated across the three published kolm distillation case studies (Trinity-500, the support-bot pilot, and the classification benchmark) and are not a guarantee for your dataset.
