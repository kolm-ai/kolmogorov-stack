What to do with 1B tokens of Portkey logs

2026-06-04 · ~6 minute read · kolm engineering

Your LLM gateway export is a distill corpus in disguise. Here is the 5-command path from helicone-logs.jsonl to a self-hosted model that does the same job. If you have terabytes of Portkey, Helicone, LiteLLM, or OpenRouter traces sitting in cold storage, you already paid for the teacher signal. This post explains the shape of those logs, why they are training data, and the exact commands to turn them into a Qwen2.5-7B QLoRA running on a GPU you control.

The shape of a gateway log row

Every LLM gateway records the same five things per call: which model was hit, what messages went in, what came back, how many tokens were used, how long it took. The column names differ, the substance does not. Side-by-side, the smallest meaningful Portkey trace row and the smallest Helicone request row look like this:

Portkey trace row

{
  "trace_id": "trc_8c3e",
  "model": "gpt-4o",
  "request": {
    "messages": [
      {"role":"system","content":"You are a refund agent."},
      {"role":"user","content":"Order 1182 never arrived."}
    ]
  },
  "response": {
    "choices":[{"message":{
      "role":"assistant",
      "content":"I can see order 1182 shipped 6 days ago. Could you confirm the delivery address?"
    }}]
  },
  "usage": {"prompt_tokens":48,"completion_tokens":23,"total_tokens":71},
  "latency_ms": 612
}

Helicone request row

{
  "request_id": "req_a91f",
  "model": "claude-3-5-sonnet",
  "request_body": {
    "messages": [
      {"role":"user","content":"Order 1182 never arrived."}
    ],
    "system": "You are a refund agent."
  },
  "response_body": {
    "content":[{"type":"text",
      "text":"I can see order 1182 shipped 6 days ago. Could you confirm the delivery address?"
    }]
  },
  "usage": {"input_tokens":48,"output_tokens":23},
  "latencyMs": 587
}

Same five fields in both: model, messages (prompt), response, usage, latency. LiteLLM and OpenRouter export the same shape with different key names. The kolm importer normalizes all four into one capture-lake schema.

Why this is training data, not observability data

Every row in your gateway export is a {prompt → response} pair from a teacher model you have already paid for. That is the textbook definition of a distill dataset. The gateways pitched the export as a debug log because their product surface is observability dashboards, latency charts, cost attribution. That framing is correct, but it is incomplete. The bytes on disk are exactly what a supervised fine-tuning pipeline takes as input.

If your team made ten thousand calls to GPT-4o yesterday answering customer questions, you have ten thousand worked examples of how a frontier model handles your domain, in your voice, against your real users. The hardest part of a distill project, getting high-quality teacher signal at scale, is already done. You are not building a corpus. You are loading one.

The reframe. Gateway logs were sold as "see what your model is doing." They are also "see what your model is doing, and ship a smaller one that does the same thing for a tenth of the inference cost." Nothing changed about the bytes. The second use case just was not part of the original sales motion.

Five-command walkthrough

Each command is a single CLI invocation. Inputs and outputs are JSONL or single files; you can resume any step independently.

kolm import --from helicone ./helicone-logs.jsonl --out captures.jsonl
kolm distill spec --in captures.jsonl --teacher cerebras:llama-3.3-70b --out spec.json
kolm distill collect --spec spec.json --out pairs.jsonl
kolm distill train --spec spec.json --pairs pairs.jsonl --out adapter
kolm deploy --adapter adapter --target ollama

What each step does, in one line:

  • import. Reads the Helicone export, drops rows without a usable assistant response, writes the normalized capture lake JSONL. Swap --from helicone for portkey, litellm, or openrouter.
  • distill spec. Picks the base student (Qwen2.5-7B by default), the teacher council, the LoRA hyperparameters, and the eval block. The spec is the reproducibility artifact; it gets bundled into the final passport.
  • distill collect. Runs the teacher (here Cerebras Llama-3.3-70B for sub-second latency) over the captures and emits training pairs with provenance fields. If you skip this step and pass the gateway export directly, the original gateway responses are used as the teacher signal verbatim.
  • distill train. LoRA r=16 on Qwen2.5-7B at bf16, gradient checkpointing on. Fits on a 5090 or a $0.39/hr RunPod 4090.
  • deploy. One artifact, multiple deploy targets. --target ollama for the desktop runtime; --target vllm or --target gguf-q4km are also wired.

Cost math at 1B prompt tokens

Worked example with round numbers. If your Portkey export covers ~1B prompt tokens at $0.30 / 1M (OpenAI gpt-4o input pricing as of mid-2026), you have already paid roughly $300 to generate the teacher signal. That spend is sunk; it is on a credit card statement somewhere. Re-running the same calls today through any provider would cost the same $300 again.

Distilling that corpus into a Qwen2.5-7B QLoRA on a $0.39/hr cloud GPU takes about twelve hours of wall time for a typical support-shaped dataset, so the training run is on the order of $5. Once trained, inference moves from a per-token API bill to a single self-hosted GPU. The break-even against gpt-4o on inference depends on your traffic volume, but the training step itself is roughly two orders of magnitude cheaper than the spend that produced the teacher signal in the first place.

The point is not that distillation is free. It is that the expensive part already happened and is sitting in cold storage waiting to be used a second time.

What we ship

Four working importers, one schema:

  • Portkey. Trace export JSONL, both the legacy trace_id shape and the newer span-per-call shape.
  • Helicone. Request log JSONL from the dashboard export or the /v1/request API.
  • LiteLLM. Spend log JSONL and the proxy server access log format.
  • OpenRouter. Generation export JSONL with the OpenRouter-specific provider-routing metadata preserved as side fields.

Kong AI Gateway and TrueFoundry are open issues; if you have a sample export from either, attach it to the issue and the importer typically lands within a week. Related landings: /gateway-migration for the comparison page, /openai-migration if you are coming from the closing fine-tuning surface, /docs/recipes/openai-finetune-migration for the annotated reproducible recipe.

Receipts cover every log row to every distill row to every inference

Every row that enters the pipeline gets an Ed25519-signed receipt. The capture row, the training pair derived from it, the eval datum that exercised it, and the inference call that later used the trained model are all linked by content-addressed IDs in an immutable lineage chain. If an auditor asks "where did this response come from?" the answer is a verifiable chain back to the original gateway log line. The receipt format is documented at /docs/receipts; verification is signature-only with no auth required.

Caveats. The $0.30 / 1M input-token figure is the OpenAI gpt-4o list price as of 2026-05-28; volume discounts and provider variation are real. The "~$5 training run" figure assumes a support-shaped dataset of ~30k pairs on a single 4090; larger corpora, larger bases, and council distillation (multiple teachers) all scale the bill linearly. Distilled students are smaller than the teacher and will underperform on out-of-distribution tasks; the eval block in the spec is the gate that tells you which workflows are safe to migrate now and which need a larger base.
