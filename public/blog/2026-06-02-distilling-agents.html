Agents · Tool-call distillation

Distilling agents, not just chats

2026-06-02 · ~8 minute read · kolm.ai engineering

Bring an agent trajectory — Claude tool_use, OpenAI tool_calls, MCP turns — and ship a smaller model that picks the same tools, at one-tenth the cost. The OpenAI fine-tuning API only ever distilled a single chat turn. Real agents alternate between assistant text, tool calls, tool results, and follow-up assistant turns, and a model that has never seen that shape during training learns to imitate the surface text without ever learning the underlying decision policy. This post walks the agent distill path in kolm: how a trajectory row is shaped, why a single teacher is the wrong choice for tool selection, what the eval harness actually measures, and the five commands that reproduce the pipeline end to end.

The shape of an agent distill row

The OpenAI fine-tuning JSONL is a flat list of chat turns: {"messages":[{"role":"user",...},{"role":"assistant",...}]}. That shape is fine for a chat distill and useless for an agent distill, because agent reasoning lives in the turns the chat schema does not represent: the assistant's choice of which tool to call, the structured arguments it passed, the tool result it got back, and the assistant message it composed in response.

kolm captures the full trajectory and treats the whole assistant → tool_call → tool_result → assistant sequence as the training unit. One row from a real agent capture lake looks like this:

{
  "rid": "trj_01KYK4Z7MX",
  "namespace": "support-agent",
  "trajectory": [
    {"role":"user","content":"Refund order 1187, the strap broke."},
    {"role":"assistant","content":"I'll look up the order first.",
     "tool_calls":[{"id":"tc_1","name":"lookup_order",
                    "arguments":{"order_id":"1187"}}]},
    {"role":"tool","tool_call_id":"tc_1",
     "content":{"order_id":"1187","status":"delivered",
                "items":[{"sku":"strap-v2","price_cents":2400}],
                "delivered_at":"2026-05-21T14:02Z"}},
    {"role":"assistant","content":"Found it, delivered May 21. Issuing refund.",
     "tool_calls":[{"id":"tc_2","name":"issue_refund",
                    "arguments":{"order_id":"1187","amount_cents":2400,
                                 "reason":"defect"}}]},
    {"role":"tool","tool_call_id":"tc_2",
     "content":{"refund_id":"rf_88121","status":"ok"}},
    {"role":"assistant","content":"Refund of $24.00 processed. Order rf_88121."}
  ],
  "teacher": "claude-opus-4-7",
  "latency_ms": 4218,
  "cost_cents": 7.3
}

The importer at src/importers/agent-trajectory.js reads three native shapes: Anthropic's tool_use/tool_result blocks, OpenAI's tool_calls array with role:"tool" response messages, and MCP session transcripts captured through the kolm gateway. The trained student learns the full conditional: given the user goal and the running tool-result context, predict the next tool name plus its arguments. That is the policy the agent actually runs in production. A chat-only distill never sees it.

Why council voting matters for tool-call agents

Single-teacher distillation memorises the teacher's quirks. If Claude tends to call search() when grep() would have been the correct choice for a code-aware tool surface, a Claude-only student will inherit that exact preference and call search() on every code lookup forever. The student does not know the teacher was wrong; it only knows the teacher's output was the training target.

Council distillation breaks the imitation by asking three or more teachers the same question and only training on the agreement. For a chat distill that means voting on output text, which is fuzzy. For a tool-call distill it is structurally cleaner:

  • Tool name vote. If two of three teachers pick grep and one picks search, the council emits grep. The dissenting trajectory is dropped from the training pairs, not just down-weighted.
  • Argument shape vote. Tools have schemas. The collector validates each teacher's arguments against the JSON Schema for the chosen tool and rejects shape mismatches before they reach the training set.
  • Recovery vote. When a tool returns an error, did the council recover with the same next tool call? Trajectories where teachers diverge on error handling are tagged as noisy and re-sampled with a higher temperature ceiling on the next collect pass.

For Trinity-Agent, the first published agent artifact from this pipeline, the council was Claude Opus 4.7, GPT-5, and DeepSeek V3, with Gemini 2.5 Pro as a fourth-lineage judge. About 11% of single-teacher trajectories failed the tool-name vote and were dropped. The remaining 89% survived as clean training pairs, and the student matched the council's tool selection at 96.3% on the held-out eval.

What the eval harness measures

Chat distill evals measure agreement on output text, which is the wrong target for an agent. A student that picks the correct tool with the correct arguments but phrases the user-facing assistant message slightly differently is correct; a student that produces a polished assistant message and calls the wrong tool is broken. The agent eval harness reports four numbers:

  • Tool-call accuracy. Per turn, did the student pick the same tool name as the council? Reported as a percentage across the holdout set, broken down by tool.
  • Argument-shape match. For each correct tool name, did the student emit arguments that pass the tool's JSON Schema and match the council's argument values? Strict match for enums and IDs, fuzzy match for free-text strings.
  • Tool-sequence Levenshtein. Across a full trajectory, the edit distance between the student's tool call sequence and the council's. A score of 0 means the student followed the same plan; small scores tolerate reordering of independent steps; large scores flag a divergent strategy.
  • End-to-end task completion. For trajectories with a ground-truth final state (a refund issued, a ticket closed, a file written), did the student's run terminate in the same state? This is the bottom line; the other three metrics decompose why a failure happened.

The harness writes a per-row JSON report so you can see exactly which trajectories the student matched and which it diverged on. Cutting over to a distilled agent without running the harness is the fastest way to ship a regression you will not detect until a tool call costs a customer money.

5-command reproduction

Each block is the literal command. The recipe at /docs/recipes/agent documents exit codes, common failure modes, and annotated output.

Step 1 of 5 · Replay your captured agent traffic

kolm capture replay \
  --namespace support-agent \
  --from 2026-04-01 --to 2026-05-28 \
  --out ./trajectories.jsonl

The capture lake holds every gateway request grouped by trajectory ID. Replay materialises the multi-turn sequences as JSONL rows in the shape shown earlier in this post.

Step 2 of 5 · Generate an agent distill spec

kolm distill spec --mode=agent \
  --namespace support-agent \
  --base qwen2.5-7b-instruct \
  --teachers claude-opus-4-7,gpt-5,deepseek-v3 \
  --judge gemini-2.5-pro \
  --tools ./tools.json \
  --output ./agent-spec.json

The --mode=agent flag tells the spec to expect trajectory rows, validate the --tools manifest against the JSON Schema in each row, and configure the eval block for the four agent metrics rather than the chat metrics.

Step 3 of 5 · Collect council trajectories

kolm distill collect \
  --spec ./agent-spec.json \
  --trajectories ./trajectories.jsonl \
  --out ./agent-pairs.jsonl

Each user goal is run against all three teachers in parallel. The collector enforces the tool-name vote and the argument-shape vote inline; dropped trajectories are written to ./agent-pairs.dropped.jsonl with a reason field for audit.

Step 4 of 5 · Train the student

kolm distill train \
  --spec ./agent-spec.json \
  --pairs ./agent-pairs.jsonl \
  --out ./agent-v1.kolm

LoRA r=32 on Qwen2.5-7B with the tool-token vocabulary expanded to cover the tools.json manifest. Fits on a single 5090 or a $0.39/hr cloud GPU. The resulting .kolm artifact carries the spec, the eval report, the dropped-trajectory log, and an Ed25519 signature over the bytes.

Step 5 of 5 · Deploy

kolm deploy ./agent-v1.kolm \
  --target vllm \
  --target gguf-q4km \
  --tools ./tools.json

The deployed artifact serves the OpenAI-compatible /v1/chat/completions endpoint with the tools parameter respected. Existing agent client code keeps working with one base-URL change.

Receipts cover every tool call

Every tool call the deployed agent makes emits an Ed25519-signed receipt under the kolm-audit-1 schema. The receipt captures the input arguments, the tool result, the latency, the model that selected the tool, and a hash of the parent trajectory. Receipts are addressable at /v1/verify/<rcpt_id> with no auth, so a downstream auditor can replay the agent's decision chain end to end. See /docs/receipts for the schema and the verification flow.

For a refund agent that calls lookup_order then issue_refund, you get two receipts per refund and a parent trajectory receipt linking them. If a chargeback dispute lands six months later, the receipts are the audit trail: which model, which prompt, which tool inputs, which tool outputs, signed and verifiable.

Caveats

Agent distillation is harder than chat distillation. The student is learning a multi-step policy, not a single mapping, and the failure modes compound across turns. In practice we see customer-support agents, document-retrieval agents, and classification-routing agents land at >90% tool-call accuracy on the first council pass. Open-ended coding agents and multi-hour research agents usually need a larger base (Qwen2.5-32B or Llama-3.3-70B) and a second collect pass with a stricter argument-shape filter to converge.

Order of operations. Run the eval harness against your real holdout before deploy. Tool-call accuracy alone is not enough; check the tool-sequence Levenshtein and the end-to-end completion rate too. A student with high per-turn accuracy and low sequence accuracy is making locally correct choices that compound into the wrong outcome.

Caveats. The 96.3% tool-call accuracy figure is the Trinity-Agent eval on the support-agent benchmark and is not a guarantee for your dataset; agent workflows with larger tool surfaces (more than ~30 distinct tools) or with deep nesting (more than ~10 turns per trajectory) typically need a larger base model.
