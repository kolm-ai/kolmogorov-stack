kolm vs Anthropic

Claude is the right call when your constraint is frontier reasoning and your deployment target is Anthropic's hosted API. kolm is for repeated, narrow workflows you want to own, prove, and run outside a vendor endpoint.

They can pair: use Claude as the teacher inside a kolm compile, then ship the repetitive workflow as a portable .kolm artifact. This page shows when Claude wins, when kolm wins, and when they stack.

Where they actually differ.

The matrix above is the executive summary. The five sections below are the substance. Anthropic is genuinely strong on the reasoning surface; kolm is genuinely strong on the ownership and on-device surface. The interesting buyers use both.

Caching vs distillation.

Prompt caching takes a repeated prefix (your system prompt, your tool definitions, the document you are reasoning over) and amortizes its cost across many calls. It is the right answer for workloads where the prefix is large and the variable suffix is short, because the cache write happens once and the cache reads are cheap. It does not change the latency floor (you still pay the network roundtrip and the generation step) and it does not change ownership (the cache lives in Anthropic's infrastructure). Distillation takes a different cut: it learns the input-to-output mapping on your task distribution and bakes it into student weights that run locally. The cache helps the next thousand calls; the distilled student helps every call forever and runs in your VPC. They solve different shapes of the same cost problem and are complementary, not substitutes.

Per-call audit chain.

Anthropic provides usage logs, console history, and (on Enterprise) ZDR plus a BAA on the Healthcare SKU. That is sufficient documentation for many regulators. For tenants whose auditor wants to know what model, what prompt template, what tool definitions, and what eval gate produced each inference, the per-call vendor log is not the same as a compile-time receipt chain. kolm signs the artifact at compile, records the 14-step provenance chain (spec, seeds, split, train, recipes, evals, external holdout, tenant shadow, auditor, supersession, drift, sigs, rekor), and the runtime verifier confirms the bytes on disk match the bytes that were signed. When the auditor asks "what trained this, on what data, with what gate, signed by whom" the answer is a single file plus a Sigstore entry, not a console screenshot.

On-device inference vs API roundtrip.

The Anthropic surface assumes a network. That is not a complaint: the Messages API plus tool use plus extended thinking plus the file-handling surface is excellent and depends on Anthropic's infrastructure to work. The trade is that every inference is one network hop away, p50 latency sits in the 500 ms to 3 s range, p99 spikes during peak hours, and an outage on Anthropic's side is an outage on yours. kolm's runtime is in-process: a 7B INT4 student on a 3090 generates at well under 100 ms for typical task lengths, p99 is predictable, and an Anthropic outage does not touch a deployment that runs the .kolm artifact. For interactive UX (chat, autocomplete, in-the-loop tooling) the latency difference is the difference between feeling instant and feeling laggy. For batch or background workloads the latency floor matters less; the cost crossover matters more.

Cost structure: token-based vs amortized GPU.

Token-based pricing scales linearly with usage. Cache reads at 90% off the input rate flatten the curve substantially for prefix-heavy workloads, but the curve is still linear. Amortized GPU cost is a step function: you pay for the hardware whether you use it or not, and the per-call cost is power plus depreciation divided by throughput. Below the duty-cycle threshold (a few thousand requests per day, depending on prompt length), the token-priced API with caching is cheaper than owning a GPU. Above the threshold (typically 50k to 200k requests per day on a stable task), the amortized GPU plus a kolm student is cheaper by an order of magnitude and the savings compound monthly. The decision is throughput and predictability, not a moral judgment about hosted vs local.

When Anthropic + kolm work together (the layered pattern).

The pattern that wins for high-volume regulated workloads in 2026 is layered. Use Claude (Opus or Sonnet 4.x) as the teacher inside kolm distill --teacher=anthropic:claude-sonnet-4-7 on your captured production traffic. The compile step calls Anthropic once per seed, applies the PHI redactor (W157) before the call, and bakes the teacher's behavior on your task distribution into a small student. The .kolm artifact ships to your runtime and handles the repetitive 80% to 95% of inference at local cost and local latency. The remaining 5% to 20% (the long tail that genuinely needs frontier reasoning) routes back to the Anthropic API directly. The bill drops by an order of magnitude, the auditor reads a receipt chain, and Claude is doing the work it is uniquely good at: hard reasoning on the long tail.

What to use kolm for.

  • Regulated data with a strict perimeter. HIPAA PHI, financial KYC, legal privilege, classified workflows. Inference runs in your VPC; the teacher call at compile is redacted; the auditor reads a signed file.
  • Edge, on-device, or air-gapped deploy. Phones, retail kiosks, factory PLCs, ship-borne networks, classified enclaves. The .kolm file ships through any pipe that ships a binary and runs without a network.
  • Repetitive prompts. Form normalizers, denial-code classifiers, schema-aware extractors, retrieval rerankers, intent routers. If a substantial share of traffic looks similar in shape, distillation captures the mapping and runs forever at local cost.
  • Interactive latency budgets. Sub-200 ms p99 for in-the-loop tooling (autocomplete, voice agents, real-time triage). The network hop to Anthropic is the floor; the on-device student does not have one.
  • Cost predictability at volume. Once you cross the duty-cycle threshold for owning a GPU, amortized hardware plus a kolm artifact is cheaper than any token-based SKU and the line is flat instead of linear.

What to use Anthropic for.

  • Long-tail reasoning that does not distill cleanly. Hard novel reasoning, agentic multi-step planning, code synthesis on unfamiliar codebases. The frontier capability of Opus or Sonnet 4.x is what you are paying for and a 7B student plateaus below that bar.
  • Hosted-only deploy with moderate volume. Below the cost crossover, token-based pricing with prompt caching is cheaper than owning a GPU. If your buyer does not require local inference and your volume is in the few-thousand-requests-per-day band, skip the rest.
  • Safety and constitutional posture. Buyers whose risk team specifically asked for Claude's safety posture, refusal behavior, or constitutional alignment as a procurement requirement.
  • Frontier-grade tool use and extended thinking. The Messages API surface (tool use, parallel tools, extended thinking, file handling) is excellent and is what you would build anyway if you owned the runtime. Use it directly when the runtime owns the loop.
Licensing note. Anthropic's API terms restrict using outputs to develop "competing" general-purpose models. A narrow-task kolm artifact (form normalizer, denial-code classifier, single-vertical extractor) is not a competing general model; the legal status of the narrow-task carve-out is unsettled. The always-safe option is --teacher=local:<open-weight-model> on your own GPU. See RS-1 §7.7 for the posture.
