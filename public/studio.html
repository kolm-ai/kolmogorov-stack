01 · Teachers

Pick a frontier teacher. Any of them.

Anthropic, OpenAI, vLLM, SGLang, TGI, TRT-LLM, your own kolm runtime. Studio treats them all the same: a teacher that answers, scored by k-score, ready to label your captured data. Switch teachers without rewriting the distill recipe. Cost-attribute per teacher per namespace.

  • Claude, GPT-5, DeepSeek-R1, Qwen2.5, your own
  • vLLM, SGLang, TRT-LLM, TGI adapters
  • Switch teachers, recipe stays the same
  • Per-teacher cost and latency attribution
  • Teacher fallback, resume from any step

02 · Distill

Train a smaller student on your data.

Studio reads from your capture log. Teacher labels what the student misses. Student trains on the diff. Every step writes a checkpoint plus a progress row. Resume from any step. Fail over to a backup teacher. Output: a smaller, faster, narrower model that beats the teacher on your task.

  • Reads tenant-fenced captures, no external upload
  • Teacher-fallback list, resume from any step
  • Per-step checkpoints, per-step progress row
  • On-policy, preference, and DPO objectives
  • Run on RTX 5090, A100, H100, or rented compute
Distill docs → Open distill runs 
# distill 7 days of inbox captures into qwen2.5-3b.
$ kolm distill from-captures \
    --teacher claude-sonnet-4-6 \
    --student qwen2.5-3b-instruct \
    --since 7d --namespace inbox --resume auto

03 · Quantize

BF16 to INT4. Same quality. A third the VRAM.

NF4 plus double quantization via bitsandbytes 0.49.2. DeepSeek-R1 32B: 61.0 GB to 17.9 GB in 125.3 s on one RTX 5090. Worker-isolated so a crash in the quantizer does not take the orchestrator down. Same pipeline works for every size in the matrix above.

  • NF4 + double quantize via bitsandbytes 0.49.2
  • Worker-isolated, crash-safe
  • k-score gate, refuses to ship a regression
  • Re-quantize as new techniques land
  • Proven on 4 sizes, 0.5B to 32B, single 5090
k-score gate → Quantize benchmarkscompare 
# quantize the distilled student to INT4.
$ kolm quantize ./ckpt-300 --to int4 --method nf4-double

04 · Compile

One artifact. Every target. Byte-reproducible.

Compile spits one .kolm file you can ship to any runtime: CUDA, ROCm, Metal, CPU, or WASM. The build is hash-pinned end to end, including the redactor rule set and the receipt verifier. Run kolm verify in CI to detect a tamper before deploy. Sigstore-attested optional.

  • Targets: CUDA, ROCm, Metal, CPU, WASM
  • Byte-reproducible build, hash-pinned
  • Verifier in 6 SDK languages
  • Sigstore + in-toto attestation, optional
  • Embeds Ed25519 receipt key in the artifact
Compile docs → Offline mode 
# compile the quantized student into one .kolm file.
$ kolm compile ./inbox-router.q4.kolm-shard/ \
    --target cuda --sign \
    --out ./inbox-router.kolm

05 · k-score

The regression gate. No magic.

k-score is a 0 to 100 composite of exact-match, semantic, latency, cost, and PII leak per row, then averaged over a fixed frozen eval set. Studio refuses to ship an artifact whose k-score regresses against the previous artifact in the same namespace. You can override, but you have to type the word.

  • Composite of exact + semantic + latency + cost + leak
  • Frozen eval set, namespace-scoped
  • Refuses to compile on regression, opt-in override
  • Same metric in CI, deploy gate, drift watcher
  • Spec lives at /k-score, 0 magic
k-score spec → Open build runs 
# compile blocks a k-score regression.
$ kolm compile ./inbox-router.q4.kolm-shard/ --target cuda
× blocked: k-score regressed
   previous artifact: 0.87
   new artifact:      0.83
   Δ                  -0.04

   override with --i-accept-regression and the reason flag.

$ kolm compile ./inbox-router.q4.kolm-shard/ --target cuda \
    --i-accept-regression \
    --reason "rolling back to test capture diff"
✓ overridden by ks_4b7bc3b...
✓ reason logged to /v1/audit-log

06 · Registry

Every artifact, immutable, signed, browsable.

The registry holds every artifact you compile, with its full provenance: dataset hash, teacher hash, student hash, quantize method, k-score, signed by your tenant key. Pull from anywhere with one command. Push to your own air-gapped registry. Or use the public hub for open weights.

  • Immutable, content-addressed, signed
  • Full provenance per artifact
  • Push to private or air-gapped registry
  • Public hub for shared open-weight artifacts
  • kolm pull / kolm push, one command each
Registry CLI → Browse hub 
# publish to your private registry, then pull elsewhere.
$ kolm registry push ./inbox-router.kolm \
    --registry registry.acme.internal \
    --tag inbox/router:1.4.0
✓ uploaded 187 MB
✓ tagged registry.acme.internal/inbox/router:1.4.0

$ kolm pull inbox/router:1.4.0 \
    --registry registry.acme.internal \
    --out /opt/kolm/artifacts/

✓ signature verified ed25519
✓ ready to run with kolm runtime serve
