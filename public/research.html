01 / The approach

Start from the reviewer's desk.

The moment a CISO has to vet an autonomous agent, a questionnaire stops clearing it. A reviewer wants a method they can examine, not a claim they have to take on faith. Everything below is built backward from that desk.

Inspect, not trust

The verifier is open

The verification library, the canonicalization, and the CLI are open and reproducible. A reviewer checks the result and the thing that produced it, in their own environment.

Mapped, not invented

Standards reviewers already cite

Every check translates to SOC 2, ISO 42001, NIST AI RMF, the EU AI Act, OWASP and MITRE ATLAS. ASR is deliberately not a rival certification.

Scope over spin

Limits stated plainly

What is assessed is named. What is tested and reported, not warranted, is named too. The scope statement travels inside the signed report, next to the findings.

02 / The ASR framework
02

Eight controls, written down once.

Agent Security Readiness is the checklist a buyer's review runs, set down openly. It does not compete with the frameworks a reviewer enforces; it maps into them, so passing it means something to the people who sign off.

ASR-2tamper-evident audit trail

ASR-5supply-chain provenance
ASR-6verifiable evidence signed

One yardstick, eight readings

Each control is a question a reviewer already has: does the agent hold more access than it uses, can the trail be edited, where does data go, can the agent be hijacked, what is in the supply chain, can the evidence be verified, can retrieval be poisoned, and what can a sub-agent do that its caller cannot.

ASR → SOC 2 TSC · ISO 42001 · NIST AI RMF · EU AI Act Art.12/14 · OWASP LLM & Agentic Top 10 · MITRE ATLAS

Not a rival credential

ASR is published CC0. Use it, fork it, map your own program to it. We would rather raise the floor for agent security than own a gate in front of it.

03 / Adversarial testing

We attack the agent on purpose.

The first question about an autonomous agent is whether it can be hijacked into acting for someone else. So injection is not a checkbox here. It is a battery, run against the agent's own tools and context, with every result reproduced before it is written down.

LLM01 · direct

Instruction hijack

The agent is told to ignore its instructions and act for the attacker. We probe the system prompt and the tool layer for the boundary that should hold and the one that does not.

OWASP LLM01

LLM01 · indirect

Poisoned context

A payload hides in retrieved content, a tool result, or a document the agent reads. The agent never sees a malicious user, only data that carries an instruction.

OWASP LLM01 · MITRE ATLAS

ATLAS · bypass

Guardrail and jailbreak

Known jailbreak patterns and obfuscations run against the system prompt and the filters in front of it, to see what slips through and what the agent does with it.

ATLAS AML.T0051

01 · reproduce

Run it again

Every finding ships with the exact input and the agent's response, so a reviewer can rerun it rather than take our word.

02 · score

Judge the action

Each case is rated by what the agent did, not by whether a refusal string appeared. Then it is mapped to OWASP LLM01 and MITRE ATLAS.

03 · report

Seal it with its limits

The battery and its limits go into the signed report, where the scope statement sits next to the findings.

Scope is contractual. Permission posture, redaction and audit-trail integrity are assessed. Injection is tested and reported, not warranted.

04 / The signing design
04

Findings a buyer verifies without us.

A report is only evidence if the reader can check it themselves. Findings are canonicalized to exact bytes, signed with Ed25519, and recorded in an append-only Merkle log. The buyer verifies in their own browser, against the key inside the report, with no kolm server in the trust path.

Canonical bytes

Edit one field, the seal breaks

Findings serialize to key-sorted, whitespace-free JSON, hashed with SHA-256. The signature covers those exact bytes, so a downgraded finding or an inflated score is self-evident.

Embedded key

Nothing needed from us

The signature and its public key travel together inside the report. The verifier runs offline against the key in the file.

Transparency log

No quiet backdating

Each issuance enters an append-only Merkle log (RFC 6962 style, Ed25519 over SHA-256), so a report cannot be silently swapped. A transparency log, not a chain.

Two checks, both offline. Tier 1 confirms the signature covers the bytes. Tier 2 confirms the signing key is the one the buyer's keyring expects. A rogue key clears Tier 1 and fails Tier 2.
