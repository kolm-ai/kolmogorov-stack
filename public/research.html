01 / The approach

Start from the reviewer's desk.

A six-figure deal does not fail in security review. It stalls. The moment a CISO has to vet an autonomous agent, a one-week review runs four to eight, and a questionnaire no longer clears it. A reviewer wants a method they can examine, not a claim they have to take on faith. Everything below is built backward from that desk.

Inspect, not trust

The verifier is open

The verification library, the canonicalization, and the command-line tool are open and reproducible. A reviewer can check the result and the thing that produced it, in their own environment.

Mapped, not invented

Standards reviewers already cite

We do not ask a buyer to learn a new credential. Every check translates to OWASP, MITRE ATLAS, NIST AI RMF and the EU AI Act. ASR is deliberately not a rival certification.

Scope over spin

Stated plainly

What is tested is named. What is reported, not warranted, is named too. The scope statement is part of the signed report, so a reviewer reads the limits next to the findings.

02 / The ASR control framework

Six controls, written down once.

Agent Security Readiness is a seed-stage readiness checklist for agentic products: the concrete controls a buyer's review looks for, set down openly. It does not compete with the frameworks a reviewer enforces. It maps into them, so passing it means something to the people who sign off.

ASR-1least-privilege scopes
ASR-2tamper-evident audit trail
ASR-3data egress and redaction
ASR-4injection resistance reported, not warranted
ASR-5supply-chain provenance
ASR-6verifiable evidence signed

One yardstick, six readings

Each control is a question a reviewer already has: does the agent hold more access than it uses, can the trail be edited after the fact, where does data go, can the agent be hijacked, what is in the supply chain, and can any of it be verified.

03 / Adversarial testing

We attack the agent on purpose.

For an autonomous agent, the first thing a reviewer asks is whether it can be hijacked into acting for someone else. So injection is not a checkbox. It is a battery, run against the agent's own tools and context, with every result reproduced before it is written down.

LLM01 · direct

Instruction hijack

The agent is told to ignore its instructions and act for the attacker. We probe the system prompt and the tool layer for the boundary that should hold and the one that does not.

OWASP LLM01

LLM01 · indirect

Poisoned context

A payload hides in retrieved content, a tool result, or a document the agent reads. The agent never sees a malicious user, only data that carries an instruction.

OWASP LLM01 · MITRE ATLAS

ATLAS · bypass

Guardrail and jailbreak

Known jailbreak patterns and obfuscations are run against the system prompt and the filters in front of it, to see what slips through and what the agent then does with it.

04 / The signing and transparency design

Findings a buyer can verify without us.

A report is only evidence if the person reading it can check it themselves. So a finding is canonicalized to exact bytes, signed with Ed25519, and recorded in an append-only log. The buyer verifies the signature in their own browser, against the key inside the report, with no kolm server in the trust path.

Canonical bytes

Findings are serialized to key-sorted, whitespace-free JSON, then hashed with SHA-256. The signature covers those exact bytes, so a downgraded finding or an inflated score is self-evident.

Ed25519, embedded key

The signature and the public key it was made with travel together inside the report. The buyer needs nothing from us: the verifier runs offline against the key in the file.

Append-only Merkle log

Each issuance is recorded in an append-only Merkle log (RFC 6962 style, Ed25519 over SHA-256), so a report cannot be quietly backdated or swapped. It is a transparency log, not a chain.

Two checks, both offline. Tier 1 confirms the signature covers the bytes. Tier 2 confirms the signing key is the one the buyer's keyring expects. A rogue key clears Tier 1 and fails Tier 2.
