Why so many SDKs

One compiler. Runtime adapters per ecosystem.

kolm is a single product: capture API traffic, distill, quantize, compile to a signed .kolm artifact you own. The SDKs are thin clients over the same /v1 routes the CLI uses, plus the loaders each ecosystem needs. Node and Python wrap every route. MCP and VS Code wrap the verbs a Claude-resident developer reaches for. C and Rust exist so regulated and embedded deployments can verify a .kolm artifact without a heavy runtime. Swift and Kotlin are coming for on-device artifact loading on phones and edge devices.

Hello world

The same four-line whoami across every binding

Every kolm SDK starts the same way: read KOLM_API_KEY from the environment, hit /v1/whoami, print the authenticated tenant. If this works, every other route works.

Node · @kolm/sdk
import { Kolm } from "@kolm/sdk";
const client = new Kolm({ apiKey: process.env.KOLM_API_KEY });
const me = await client.whoami();
console.log(me.tenant_id, me.plan);
Python · kolm-py
from kolm import Kolm
client = Kolm(api_key=os.environ["KOLM_API_KEY"])
me = client.whoami()
print(me.tenant_id, me.plan)
Rust · kolm crate v0.1.0
use kolm::Client;
let c = Client::from_env()?;
let me = c.whoami()?;
println!("{} {}", me.tenant_id, me.plan);
C · kolm.h (stb-style)
#define KOLM_IMPLEMENTATION
#include "kolm.h"
kolm_client_t *c = kolm_client_from_env();
kolm_response_t r = kolm_whoami(c);
printf("%d %s\n", r.status, r.body);
One contract · every binding

The promises that hold regardless of language

Six shipped bindings, one contract. The shape an engineer relies on stays constant.

envelope

Honest error envelope

Every response surfaces raw HTTP status plus the structured body {ok, code, error, data}. No binding hides a 4xx behind a success boolean. The C and Rust bindings expose status and body directly so callers see what the server said.

auth

x-kolm-api-key header

One header, one key. Read from KOLM_API_KEY by default. Override with explicit constructor args when calling from a multi-tenant runner. The CLI writes the key to ~/.kolm/credentials on kolm login.

base-url

KOLM_BASE_URL override

Point any SDK at a self-hosted gateway by setting KOLM_BASE_URL=https://kolm.internal. The hosted default is https://kolm.ai. Useful for VPC-resident installs and air-gapped CI.

verifier

Verify without a runtime

Every binding can call /v1/verify/:cid against a public CID. The C and Rust bindings exist precisely so a regulated deployment can verify a .kolm artifact from inside an audit pipeline without a Node or Python dependency.

CLI is the canonical surface

What an install actually does

Every SDK is a thin client over the same /v1 routes the CLI talks to. Install once, use any binding.

From zero to a signed receipt
# 1. Install the CLI (npm-distributed; Node 18+ required)
npm install -g @kolm/sdk

# 2. Authenticate and write credentials to ~/.kolm/credentials
kolm signup
kolm whoami
→ tenant_4d3d8db6c820 plan=hacker

# 3. Capture a call through the gateway
kolm capture log --input "draft a cover letter" --output "..."

# 4. Distill, compile, score, verify (CLI verbs == SDK methods)
kolm distill from-captures --base qwen2.5-3b-instruct
kolm compile --target gguf --precision int4
kolm score --evals ks-finance-2026-q2
kolm verify ./cm-claims-v4.kolm
