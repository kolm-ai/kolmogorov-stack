kolm.ai  /  k-score · the compile-time quality gate

A K-score gate decides whether a .kolm ships.

K-score is the compile-time quality contract. Each artifact is scored against a frozen holdout; if the gate clears it signs and ships, and if it fails no .kolm bytes are written.

The four K-score axes.

The composite is a weighted sum of normalized axes. Weights travel in the manifest, so a verifier replaying the score uses the same policy the builder used.

axis 01

groundedness

Fraction of the artifact's outputs whose claims can be re-derived from inputs or from the recipe's declared corpus. Hallucination is the negation of groundedness: a free-text answer that asserts a fact absent from inputs and absent from the corpus scores zero on this axis. For rule recipes the score is structural (rules cannot hallucinate by construction); for distilled-model recipes the score is judged row-by-row against the holdout.

default weight 0.25 · configurable via k_score_components · RS-1 §7.6
axis 02

coverage

Fraction of declared input cases on which the artifact returns a usable output. Coverage subsumes raw accuracy: a recipe that returns a correct answer on 80 percent of inputs and refuses-with-route-to-human on 20 percent scores 1.00 on coverage; a recipe that errors out on 20 percent of inputs scores 0.80. Coverage is computed against the recipe's eval rows as declared at compile time.

default weight 0.25 · configurable via k_score_components · RS-1 §7.6
axis 03

safety

Fraction of holdout rows on which the artifact's output respects the recipe's declared safety contract: PHI/PII redactor rules, route-to-human on low confidence, license-restricted refusal classes. A safety failure is not a low-quality answer; it is the wrong shape of answer (raw email in a redacted field, treatment recommendation when the recipe says route-to-human). One safety failure on a regulated workload is enough to fail the axis.

default weight 0.25 · configurable via k_score_components · RS-1 §7.6
axis 04

behavioral_drift

Delta between the candidate artifact's holdout behavior and the recipe's declared reference behavior (teacher outputs for distilled recipes; canonical-rule outputs for rule recipes; previous-shipped-artifact for cadence rebuilds). Drift is measured per row and aggregated; a small drift is expected and tolerated, a large drift indicates the artifact is no longer doing the job the recipe was authored for.

default weight 0.25 · configurable via k_score_components · RS-1 §7.6

The four axes are independent by design. An artifact that scores 0.95 on coverage and 0.40 on safety should not ship: averaging high accuracy with low safety hides the failure mode that matters most on a regulated workload. The composite is the weighted sum, but the per-axis floor (configurable per recipe) lets the operator say "fail the build if any single axis is below 0.50" so a strong-on-three-weak-on-one artifact gets rejected before it reaches the gate.

Default K-score gate by recipe class.

Each of the four recipe classes (defined in src/recipe-class.js, surfaced in the docs) carries a different default gate. The reasoning is debugability: a class whose outputs cannot be debugged at runtime needs a stricter compile-time floor; a class whose outputs are inherently fuzzy needs a relaxed floor with a per-axis weight reshuffle.

recipe_classdefault gaterationale
rule0.88Hand-authored deterministic JS. Same input always returns the same output. The artifact can be debugged step-by-step at runtime; a strict-but-not-strictest gate catches the case where the rule code is logically wrong (correct shape, wrong answer) without requiring perfection on every fuzzy edge.
synthesized_rule0.90LLM-emitted rule code, AST-validated against a constrained DSL. Strictier than rule because the teacher may emit subtly wrong code that passes the AST gate but fails the holdout on an edge case the author did not enumerate. The 0.90 floor forces the synthesis loop to iterate until the candidate clears the bar.
compiled_rule0.92Strictest gate. A compiled native binary cannot be inspected at runtime without a debugger and the source mapping; a regulated tenant deploying a compiled rule to a production endpoint has no in-band remediation if the binary is wrong. The 0.92 floor pushes the compile pipeline to re-run synthesis or hand-edit the DSL until the artifact is the right answer at compile time.
distilled_model0.85Relaxed gate. A neural network's outputs are inherently fuzzier than rule outputs; demanding 0.92 on a distilled model rejects most viable artifacts. The 0.85 floor matches the baseline composite gate shipped in src/kscore.js (GATE = 0.85). The groundedness axis weighted at 0.20 in k_score_components when class is distilled_model gives slightly less penalty for the residual hallucination LM outputs always carry; the remaining 0.05 redistributes across the other three axes.

The class default is recovered from manifest.k_score.gate at verify time. A recipe can override the default by setting spec.min_k; the override travels with the artifact bytes and is signed inside the same receipt body the K-score block lives in, so a downstream verifier cannot strip the gate without invalidating the signature.

What happens when the gate fails.

Gate failure is a hard stop, not a warning. The effects below fire together, so a tenant cannot accidentally end up with a half-built artifact whose K-score was checked but whose signature was skipped.

effect a

Compile aborts

The kolm compile process exits non-zero ($? = 2). Any CI step that wraps the compile sees the failure; cron mail surfaces the alert without needing a webhook. No partial bytes are emitted to the output path.

effect b

Verifier emits gate_failed

If the operator bypasses the compile and hand-crafts a manifest with a sub-gate composite, the verifier check #4 K-score gate rejects at ingest with reason='gate_failed'. The receipt records the rejection alongside the composite and the gate that failed.

effect c

Artifact is not signed

The Ed25519 signer never runs on a sub-gate artifact. The signing pipeline's require_k_score_pass guard blocks the key from touching unverified bytes. No transparency-log anchor is created; no Rekor entry exists for a gate-failed build.

effect d

No .kolm file is written

The output path is left untouched. If a previous build exists at the same path, the build pipeline does NOT overwrite it; the prior artifact stays on disk, the new gate-failed build is discarded, and the operator sees the failure plus the prior artifact's hash for context.

The four effects are tightly coupled by design. The K-score gate, the signing pipeline, and the file-write step share the same precondition: k_score.composite ≥ k_score.gate. Skipping the gate is not a config flag; it requires editing src/binder.js and re-installing from source. That is intentional: the verifier check #4 exists so a third-party auditor never has to trust that the operator did not bypass the gate.

The frozen-eval invariant.

A K-score is only meaningful against a holdout the artifact has never seen. The eval set is hash-pinned at compile time and persisted as a manifest field; the verifier recomputes the K-score against the same hash at ingest. Any swap of the holdout for an easier corpus drifts the eval-set hash and the verifier rejects.

The split is enforced by the seeds module (src/seeds.js, surfaced as the train/holdout-independence check, shipped wave 141 · Q+2 · N+1.5). At seed-loading time the rows are split by deterministic hash of the row's canonical JSON; the same input row always lands on the same side of the split across reseeding. The verifier replays the split and confirms train_hashes ∩ holdout_hashes = ∅; non-empty intersection is a hard fail.

The K-score is computed against the holdout side of the split and recorded with the holdout's content hash. Re-running kolm eval on the same artifact against the same holdout reproduces the same composite to four decimal places (the round4 contract in src/kscore.js); any non-determinism in the computation is itself a verifier failure. The frozen-eval invariant is what lets the gate be a contract instead of a marketing number.

Why these axes, not just accuracy.

Single-axis accuracy is the wrong primitive for a compile-time gate on a regulated workload. The four-axis shape catches four failure modes that single-axis accuracy hides; each axis covers a different way an artifact can be wrong.

  • Accuracy is one component of coverage, not its complement. An artifact that returns the correct answer on 80 percent of inputs and crashes on the other 20 percent is high-accuracy and low-coverage; for a production deployment, the crashes matter more than the accuracy on the rows where the function returned at all. Folding accuracy into coverage makes the artifact's usable-output-fraction the headline number.
  • Safety catches failures accuracy misses. The HIPAA Safe Harbor redactor either holds or it does not; an artifact whose redactor leaks a raw email in one row out of one thousand is still high-accuracy by every other measure, but is not shippable. The safety axis is binary in spirit: any safety failure on a regulated workload drops the axis fast, and the per-axis floor catches the case before the composite hides it.
  • Groundedness catches the hallucination class single-axis accuracy cannot. A distilled-model recipe whose teacher routinely embellishes correct answers with plausible-sounding but unsourced detail scores high on raw accuracy (the answer is correct) and low on groundedness (the supporting detail is not in the inputs or the corpus). Groundedness is the axis a buyer cares about when the artifact will be deployed where a confident-but-fabricated detail has compliance cost.
  • Behavioral_drift catches the rebuild regression nothing else does. A re-distilled artifact whose K-score composite is within tolerance of the predecessor's but whose row-by-row behavior has shifted (different cases pass, different cases fail) is a different artifact masquerading as a refresh. Behavioral_drift makes the row-level continuity an axis the gate enforces, not a property the operator has to spot-check by hand. The LIMA finding (Zhou et al., 2023, cited at /research/methods-2026-q2) that small-curated-dataset training produces strong generalist behavior translates to a high prior that even a small holdout will detect material drift; the drift axis is what consumes that signal.

The CLI flow, end to end.

Compute, gate-check, auto-gate at compile, read-back from a shipped receipt. The exit-code contract is uniform: zero on pass, two on gate failure, one on other error.

01 · compute

kolm eval

Runs the compiled artifact across the frozen holdout, scores each row on the four axes, weights into a composite. Writes the k_score block; does not gate. Exit zero on success, one on eval error.

$ kolm eval prior-auth.kolm

 rows scored: 412
 groundedness 0.942
 coverage 0.917
 safety 0.988
 behavioral_drift 0.901

 composite 0.937 (weights 0.25 / 0.25 / 0.25 / 0.25)
 gate 0.92 (recipe_class=compiled_rule)
 verdict PASS composite ≥ gate
02 · gate-check

kolm verify (check #4)

Loads the artifact's manifest, recomputes the composite from the stored axes, compares against the stored gate. Verifier check #4 K-score gate in src/binder.js emits pass with the composite-vs-gate detail or fail with the gate-failed reason.

03 · auto-gate

kolm compile

Runs eval as a build step, gates on the result, exits non-zero on failure before the signer touches the bytes. The require_k_score_pass guard inside the build pipeline blocks the Ed25519 key from signing a sub-gate artifact.

$ kolm compile prior-auth.spec.json

 recipes → evals
 eval verdict: GATE_FAIL
 composite 0.881 below gate 0.92
 failing axes: safety=0.74 (one redactor leak)
 [abort] artifact not signed; no .kolm written
 $? = 2

Reads the frozen K-score from the receipt without re-running eval. Useful when the operator wants to confirm what the binder PDF will show without recomputing from the holdout (which may not be available on the deploy host).

Where the K-score lands in the receipt chain.

The K-score lives in the evals receipt block and hashes into artifact_hash_input. Editing the score, axes, or gate changes the artifact identity and breaks downstream signatures.

spec → seeds → split → train → recipes → evals → external_holdout → tenant_shadow → auditor_attestation → supersession → drift_report → export → signatures → rekor

The composite that the verifier check #4 compares against the gate is the same composite the binder PDF renders to the auditor, the same composite the inference endpoint reads when reporting the artifact's score, the same composite the registry indexes when listing the artifact. One number, one source-of-truth, one verifier-recomputed value. The receipt-chain binding is what makes the K-score a contract instead of a label.

Where this fits in the wider stack.

spec · canonical

RS-1 v2.1 §7.6

The normative spec section for the K-score axes, weights, and the verifier check #4 K-score gate semantics. The four-axis shape, the per-axis weight field, the gate-by-class table, the recompute-at-ingest contract.

 frozen eval · invariant

Frozen-eval invariant

The hash-pinned holdout that the K-score is computed against. Train/holdout disjointness is enforced by the seeds module; the verifier replays the split and rejects any artifact whose train and holdout rows intersect.

 drift · downstream

Drift detection + supersession

The K-score axes are the load-bearing signal in the drift report. When production distribution shifts past tolerance against the baseline snapshot, the drift cron exits non-zero and a successor artifact retires the predecessor with the breached signals inlined.

 classes · gate-by-class

Recipe-class taxonomy

The four classes (rule / synthesized_rule / compiled_rule / distilled_model) that drive the default-gate table. Source-of-truth is src/recipe-class.js; the per-class default gate is recovered from the manifest at verify time.
