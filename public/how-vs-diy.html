kolm.ai  /  compare  /  how kolm relates to the DIY LoRA stack

kolm is shrink-wrap around the DIY stack you already use.

The DIY LoRA stack fine-tunes brilliantly. kolm keeps that stack and adds the production pieces around it: receipt chain, eval gate, signed artifact, rule fallback, teacher selection, and PHI redaction. Drop back to raw DIY any time.

Wave 169 · reviewed 2026-05-17 · shrink-wrap, not replacement

What the DIY stack already does.

  • LLaMA-Factory / Axolotl — declarative YAML configs for SFT, DPO, ORPO, GRPO, and a long tail of fine-tuning methods. Supports most open-weight bases.
  • Unsloth — 2-5× faster training and lower VRAM via custom Triton kernels and 4-bit quantization. The fastest path from "I have a GPU" to "I have a LoRA."
  • TRL (HuggingFace) — trainer classes for SFT, DPO, RLHF, GRPO. The reference implementation other tools build on.
  • PEFT — the underlying parameter-efficient-fine-tuning library. LoRA, QLoRA, IA3, prefix tuning.
  • Cost — in 2026, a LoRA job on a 7B model runs under $2 on cloud GPUs (per public comparison docs).

The DIY stack is the foundation, not the competitor. kolm uses it under the hood for the Wave 157+ ML pipeline. Where it stops being enough: when you need to ship the artifact through a procurement review, a HIPAA audit, an air-gapped deploy, or a third-party verifier.

kolm with a drop-out to raw DIY at any time.

The contract: kolm never hides the underlying stack. If you need to drop down to raw LLaMA-Factory or Unsloth, the .kolm manifest tells you exactly which trainer was invoked and what config. You can re-run the same train outside kolm and get bit-identical weights (deterministic build flags are part of the spec).

# Default kolm path — everything wired
$ kolm distill phone-normalizer \
 --teacher=anthropic:claude-opus-4-7 \
 --student-base=qwen2.5-0.5b \
 --backend=unsloth \
 --redact=phi --eval-gate=0.85

# Inspect the actual call kolm made to the DIY stack
$ kolm inspect phone-normalizer.kolm --field=training_config
 backend: unsloth
 base: Qwen/Qwen2.5-0.5B
 trainer: SFTTrainer
 lora_rank: 16, lora_alpha: 32
 ... full config

# Drop out: re-run the training outside kolm using the dumped config
$ kolm extract phone-normalizer.kolm --target=training-config --out=./train.yaml
$ llamafactory-cli train --config ./train.yaml
 # — same trainer, same flags, bit-identical weights

This is by design. The .kolm spec is Apache-2.0; the receipt chain documents which DIY tools were invoked. If kolm goes away, the DIY stack still runs your config.

When raw DIY is the right answer.

You're a research team iterating on training methods, your buyer doesn't ask for receipts, you have a GPU, and the shortest path is the rawest one. The DIY stack is free, mature, and well-documented. Don't pay kolm tax for a workflow whose value is the training experiment itself.

When kolm-wrapping the DIY stack is the right answer.

You need the production-readiness pieces the DIY stack doesn't ship: receipt chain for procurement, PHI redactor for HIPAA, signed artifact for auditor, cross-vendor manifest for license compliance, drift cron for lifecycle. kolm gives you those pieces without forcing you to abandon LLaMA-Factory or Unsloth — it wraps them.
