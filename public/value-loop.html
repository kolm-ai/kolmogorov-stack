Try it now

Post a prompt + response to the public /v1/loop/try demo endpoint. No key. No write. The receipt body shows demo:true and the x-kolm-capture-durable: false header so it cannot lie about persistence. The shape matches what a real authenticated POST /v1/bridges/observe emits.






Why a loop, not a pipeline


A pipeline is one-shot: input goes in, output comes out, you're done. A loop reads from the artifact you just shipped, captures the new prompts it sees in production, and feeds them back into the next compile. The version 2 of your recipe is informed by the failure modes of version 1. The version 3 by version 2. That is the difference between fine-tuning a model once and compiling a model over and over against your live workload.

Rung 1: capture


Every prompt/response pair from your production traffic posts to /v1/capture/log with a namespace, a model identifier, and an optional latency reading. The server persists the row and returns 201 with the x-kolm-capture-durable: true receipt header. If the storage driver is misconfigured or the row could not be written, the response is 503 with error: 'capture_store_unavailable' and the receipt header is absent. No silent swallow.


curl -X POST https://kolm.ai/v1/capture/log \
 -H "authorization: Bearer $KOLM_API_KEY" \
 -H "content-type: application/json" \
 -d '{"namespace": "prod", "items": [{"input": "...", "output": "..."}], "model": "gpt-4o-mini"}'

# Response (201):
# x-kolm-capture-durable: true
# x-kolm-namespace: prod
# {"ok": true, "namespace": "prod", "count": 1, "ids": ["cap_..."]}




Rung 2: health


Probe the storage subsystem before you bother sending captures. /v1/capture/health returns the driver name, whether durability is currently asserted, subscriber count for the SSE fan-out, and the threshold catalog (100 / 500 / 1000 pairs by default).


curl -H "authorization: Bearer $KOLM_API_KEY" https://kolm.ai/v1/capture/health
# {"ok": true, "driver": "sqlite", "durable": true,
# "subscriber_count": 0, "thresholds": {"alert": [100, 500, 1000]}}



Rung 3: bridges


Read your captures back without scanning the whole table. /v1/bridges/observations is a tenant-scoped, namespace-filterable, indexed SQL read that returns the most-recent rows for the namespace you ask for. The index is enforced by the findByTenant primitive in src/store.js so cross-tenant leakage is impossible.


curl -H "authorization: Bearer $KOLM_API_KEY" \
 "https://kolm.ai/v1/bridges/observations?namespace=prod&limit=50"
# {"observations": [...], "total": 50, "namespaces": ["prod", "staging"], "ready": []}



Rung 4: distill


Promote a namespace to a recipe with /v1/distill/from-captures. The server groups captures by template hash, picks the largest cluster (at least four rows by default), and synthesizes a recipe via the in-tree synthesizer. Recipes ship in 200 (accepted) or 422 (rejected); both are contract-valid. Above 1000 pairs the mode flips to specialist and forwards to a configured trainer bridge.


curl -X POST https://kolm.ai/v1/distill/from-captures \
 -H "authorization: Bearer $KOLM_API_KEY" \
 -H "content-type: application/json" \
 -d '{"namespace": "prod", "mode": "recipe", "min_pairs": 4}'
# 200: {"mode": "recipe", "namespace": "prod", "accepted": true, "job_id": "...", "synth": {...}}
# 400 (too few rows): {"error": "not_enough_captures", "count": 2, "min_pairs": 4}
# 400 (no shared cluster): {"error": "no_cluster", "best_cluster_size": 1}
# 503 (store down): {"error": "capture_store_unavailable"}



Rung 5: replay


Close the loop. /v1/replay picks N captures from a namespace, runs each prompt through the freshly compiled artifact (no cache), and emits a per-row diff: jaccard k_score against upstream, latency delta, cost delta. One screen, one click, the whole observe, optimize, compile ladder visible.


curl -X POST https://kolm.ai/v1/replay \
 -H "authorization: Bearer $KOLM_API_KEY" \
 -H "content-type: application/json" \
 -d '{"namespace": "prod", "version_id": "v_abc123", "limit": 25}'
# 200: {"rows": [...], "summary": {"avg_k_score": 0.87, "median_latency_delta_us": -240000}}
# 400 (no concept/version): {"error": "concept_id_or_version_id_required"}
# 404 (artifact missing): {"error": "artifact_not_found", "concept_id": "..."}



Rung 6: audit · receipts


Every rung above emits a receipt. Capture returns x-kolm-capture-durable: true. Distill, replay, and compile each chain into the same RS-1 receipt format with parent hash, signer, and SLSA provenance attestation. The artifact ships with its SBOM. Walking the chain backward from the deployed <recipe>.kolm to the first captured pair is a one-liner: kolm verify <artifact>. There is no silent step. RS-1 spec, /security, and /sbom document the chain end to end.


curl -s https://kolm.ai/v1/receipts/e176bc2f720b94a8/public > receipt.json
curl -s https://kolm.ai/v1/receipts/verify \
 -H "content-type: application/json" \
 -d @receipt.json
# 200: {"ok":true,"verified":true,"chain":[...]}



Rungs 1 through 3 are tenant-isolated by primary key in src/store.js. Rungs 4 and 5 inherit that isolation through findByTenant. Rung 6 inherits it through the receipt-chain join, so an audit query against another tenant's chain returns 404. Cross-tenant leakage is asserted as a unit test in tests/wave297-value-loop-happy-path.test.js and as a CLI smoke in tests/wave298-doctor-loop.test.js.
