Capture · step 1 of 4 · tenant-scoped event lake

Every LLM call lands in your event lake.

A developer points an OpenAI-compatible SDK at http://localhost:11434/v1 and every prompt, completion, tool call, and latency number is written to a local event store at ~/.kolm/events. Tenant-scoped, namespace-partitioned, and ready to feed distill, compile, and the K-score gate. No data ever leaves your boundary unless you explicitly route it elsewhere.

Drop-in OpenAI-compatible proxy

Run the proxy once. Change one base URL in your code. Every request your app already makes gets captured, scored, and made replayable. No SDK rewrites, no wrapper libraries.

# 1. Install the CLI (Mac, Linux, Windows; Node 18+)
$ npm install -g github:kolm-ai/kolm

# 2. Start the local proxy. Listens on 11434, writes to ~/.kolm/events
$ kolm capture serve
kolm capture · listening on http://localhost:11434
events dir       /Users/you/.kolm/events
tenant           tenant_f33c240034c5
namespace        default

# 3. Point any OpenAI-compatible SDK at it. That is the whole integration.
$ export OPENAI_BASE_URL=http://localhost:11434/v1
$ export OPENAI_API_KEY=ks_4b7bc3b...
$ python my_agent.py     # or node, ruby, go, curl, anything

# 4. Inspect what was captured
$ kolm capture ls --namespace default --limit 5

Why capture before you distill

A distilled specialist is only as good as the rows it was trained on. Capture is how you turn real production traffic into training rows you trust; the rows that came from real users, real prompts, real tool outputs, real edge cases.

  • Real prompts beat synthetic prompts

    REAL EVIDENCE · from your traffic

    Synthetic data covers what an engineer imagined; capture covers what a user actually typed. The distill step picks the rows that matter; capture is what makes those rows exist in the first place.

  • Every row gets a receipt

    VERIFY CHAIN · Ed25519 to artifact

    Each captured row writes a content-addressed event with a row hash. The hash flows into the distilled dataset, the dataset hash flows into the artifact receipt, and the artifact receipt is Ed25519-signed. The whole chain is verifiable offline.

  • You learn what is actually expensive

    COST LENS · per-row tokens + latency

    Capture writes the model, the token count, and the latency on every row. Before you pay an engineer to optimize, you can see which prompt families cost the most and which a distilled model could cover.

  • You catch upstream drift early

    DRIFT WATCH · event store is source-of-truth

    When OpenAI swaps the silent default behind gpt-4o, your latency and K-score numbers move. Capture is the source of truth for that detection; the drift dashboard reads from the event store, not from a vendor status page.

Where capture sits in the pipeline

Capture is step 1. The other three steps (distill, compile, verify) all read from the event store written here. You can stop at step 1 and still get a queryable event lake; you can also walk to step 4 and ship a signed specialist.

  1. capture

    Land traffic

    Proxy at localhost:11434 writes rows to ~/.kolm/events. Tenant-scoped, namespace-partitioned.

  2. distill

    Pick and train

    Cluster captures by template, pick the dominant cluster, generate teacher labels, train a smaller student.

  3. compile

    Quantize and sign

    Quantize weights to INT4 or INT8, build the .kolm zip, write the Ed25519-signed receipt, bind to artifact_cid.

  4. verify

    Run and prove

    Load the .kolm in vLLM, llama.cpp, MLX, or TRT-LLM. The signed receipt is verifiable on a clean laptop.

On-disk layout

Captures are plain JSONL files under a stable directory. An engineer can grep them, an analyst can load them into DuckDB, and a compliance reviewer can verify them in place. No proprietary database, no vendor lock-in.

Default paths (override with KOLM_DATA_DIR)

~/.kolm/events
Root of the event lake. One subdirectory per tenant, one per namespace, then JSONL files rotated by day.
~/.kolm/events/<tenant>/<ns>/YYYY-MM-DD.jsonl
Daily roll. Each line is one captured request and response, with hashes for the prompt body, the completion body, and the tool calls.
~/.kolm/events/<tenant>/index.sqlite
Optional index built by kolm capture index. Makes kolm capture search fast; safe to delete and rebuild.
~/.kolm/keys
Ed25519 signing key for receipts. Backed up by your dotfile manager; rotate with kolm keys rotate.
~/.kolm/audit.log
Operational audit log. Tail it into Splunk, Datadog, or your SIEM. Receipts are the durable record; this is the operational one.

Optional redaction at capture time

If a regulated workload moves through the proxy, configure the redaction worker to scrub PHI, secrets, and other sensitive tokens before the row is persisted. The worker is opt-in and isolated so the root install stays light.

  1. Configurable token list

    text PHI · live

    The redactor reads a tenant-scoped pattern list (regex + literal). Patches matched substrings to a stable hash so analytics still work but the raw value never reaches disk.

  2. Image and audio scrub

    media · live

    Optional workers/media-redact and workers/multimodal-redact-image workers blur faces, mask plates, and strip voiceprints from media captures. Heavy deps stay outside Node.

  3. No silent passthrough

    loud fail · live

    If the redactor is configured but the optional dep is missing, the capture fails with error:no_detector_installed and an install hint. The proxy never silently lets unredacted media through.

Tenant and namespace scoping

A namespace is the unit a distill run targets. A tenant is the unit a billing line and a retention policy apply to. Both are enforced at the proxy; nothing in the codebase trusts a client-supplied tenant string.

HeaderMeaningDefault
x-kolm-tenantTenant for the captured row. The proxy verifies this matches the API key and rejects the call if not.From API key
x-kolm-namespaceNamespace partition for the row. Pick one per application so distill jobs read a coherent corpus.default
x-kolm-routeOptional route tag. Useful for splitting a multi-tool agent into per-tool corpora.auto
x-kolm-trace-idOptional trace ID for cross-row correlation. Pass through from your observability stack.Generated
x-kolm-capture-durableReturned on every captured response. The event hash, useful for proving a row was persisted.Set

A team can name namespaces by feature (chat, code-review, support-triage) or by environment (prod, staging). Distill jobs are namespace-scoped, so the partition you pick at capture time determines what a future distilled specialist learns from.

Proxy capacity and overhead

A drop-in proxy is only useful if it does not become the slow part of the path. The capture proxy is a single Node process by default; it pages writes to the event store in a background flush and adds a small constant overhead per request.

  1. Sub-millisecond overhead per call

    single process · live

    The proxy parses the request, computes the messages hash, and forwards. The added wall-clock per call sits between 0.4 ms and 1.2 ms on a developer laptop; the upstream API latency (200 ms to 2 s) dwarfs it.

  2. Writes never block the response

    background flush · live

    Captured rows enqueue to an in-memory ring and a background worker flushes them in batches of 64 lines or every 250 ms. A reply returns before disk fsync; durability is best-effort by default with x-kolm-capture-durable reporting the row hash.

  3. --workers N for higher fan-out

    multi-worker · live

    Run kolm capture serve --workers 8 to spawn a pool sharing a Unix socket; tested up to 6500 requests per second on a single 32-core box without losing rows. Each worker holds its own ring; the rotator handles concurrent appenders.

  4. SSE and websocket streams

    stream-safe · live

    OpenAI-style server-sent event streams are proxied byte-for-byte. The proxy reassembles the completion in the background and writes the row only after the upstream closes the stream; partial captures are never persisted.

  5. Disk backpressure is plain

    backpressure · live

    If the local disk fills, the proxy refuses new captures with HTTP 507 and the row hash in the response body. A regulated team can wire that to an alert and rotate retention manually before the upstream call completes; a developer can read the error and free space without a silent data loss path.

  6. Replay-ready rows

    replay · live

    Every captured row carries enough fields (model, parameters, messages hash, response hash) to be replayed against the same upstream or a distilled successor. The bakeoff harness reads from the event store directly; you do not have to re-export to a separate corpus.

What a captured row actually looks like

A captured row is one line of JSONL on disk. Hashes, timestamps, model id, tool calls, tokens, and the upstream cost all live in fields a developer can grep and a compliance reviewer can read. There is no proprietary binary format.

// one line from ~/.kolm/events/<tenant>/default/2026-05-22.jsonl
{
  "ts": "2026-05-22T18:14:02.317Z",
  "row_hash": "sha256:f48a4b22e2b1c7a3...e2b1",
  "tenant": "tenant_f33c240034c5",
  "namespace": "default",
  "route": "chat.completions",
  "upstream": { "vendor": "openai", "base_url": "https://api.openai.com/v1" },
  "request": {
    "model": "gpt-4o-mini",
    "messages_hash": "sha256:9c4f...e801",
    "prompt_tokens": 812,
    "tools_present": true
  },
  "response": {
    "completion_hash": "sha256:2dba...77c4",
    "completion_tokens": 1030,
    "finish_reason": "stop",
    "tool_calls": [
      { "name": "search_kb", "args_hash": "sha256:a3c1...d10b" }
    ]
  },
  "timing": { "latency_ms": 421, "ttft_ms": 189 },
  "cost": { "micro_usd": 5340, "source": "openai-published-rate-2026-05" },
  "redact": { "applied": true, "hit_count": 0 },
  "trace": { "trace_id": "01HFG3...J9Q", "span_id": "a8c2d1..." }
}

An analyst can load a day of rows into DuckDB with one read_json_auto call. The hash columns make joins across days deterministic; the cost column makes per-namespace billing roll-ups one SQL query. The row_hash is the field that flows into the distill seeds file and from there into the artifact receipt; it is the link between a production prompt and the signed specialist eventually trained from it.

Export to Datadog, Grafana, Splunk

The capture proxy emits OpenTelemetry traces and metrics on the standard OTLP gRPC and HTTP ports. Point an OTel collector at it; the captured rows themselves stay on disk, but the latency, cost, and error counters become first-class signals in your existing observability stack.

# Send OTLP traces to Datadog Agent on localhost
$ kolm capture serve \
    --otlp-endpoint http://localhost:4318 \
    --otlp-headers "DD-API-KEY=$DD_API_KEY"

# Resource attributes the exporter adds automatically
service.name              kolm.capture
service.namespace         default
kolm.tenant               tenant_f33c240034c5
kolm.row_hash             sha256:f48a...e2b1
kolm.upstream             openai
kolm.model                gpt-4o-mini
kolm.latency_ms           421
kolm.prompt_tokens        812
kolm.completion_tokens    1030
kolm.cost_micro_usd       5340

A site-reliability engineer can dashboard cost-per-namespace, p99 latency-per-model, and error rate per route without changing application code. The same OTLP stream feeds Grafana, Honeycomb, New Relic, and any other vendor that speaks OTLP.

CLI reference

Every capture operation is reachable from the kolm capture verb. Run kolm capture --help for the full surface; the most common subcommands are below.

CommandWhat it does
kolm capture serveStart the OpenAI-compatible proxy on :11434. Add --port to move it.
kolm capture lsList recent captures. Filters: --namespace, --model, --since, --limit.
kolm capture searchSubstring or regex search over captured prompts. Uses the optional SQLite index if present.
kolm capture indexBuild or rebuild the SQLite index. Safe to delete and rerun.
kolm capture statsPer-namespace counts, cost, latency p50/p95/p99, top models.
kolm capture exportStream captures to a file. JSONL, CSV, or Parquet via the optional workers/parquet-export.
kolm capture deleteSubject deletion. Removes raw rows and writes a tombstone receipt so an auditor can prove the deletion happened.
kolm capture redact-doctorCheck the optional redaction worker is installed and able to scrub. Loud fail with an install hint if not.
kolm namespace set-retentionSet per-namespace retention in days. The local rotator deletes older rows; receipts that reference them stay verifiable.
