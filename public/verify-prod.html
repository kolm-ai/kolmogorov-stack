Verify a .kolm

Drop a model, check the receipt.

A .kolm artifact is a signed zip. The manifest names the teacher, the base model, the eval pack, and the K-score. The receipt is a chain of Ed25519 signatures over the manifest hash. This page parses an artifact in your browser, recomputes the content-addressed hash, walks the signature chain, and reports the verdict. No upload, no signup, no telemetry. Engineers, procurement, and compliance can run the same check in 30 seconds.

Verification report

What the verifier actually checks

Each step is independently re-runnable from the CLI. A reviewer can reproduce every row in this report on a disconnected machine.

  1. 01

    Parse the manifest

    Open the .kolm zip, read manifest.json, and confirm every required field is present: artifact_cid, compiler_version, teacher, base_model, eval_pack, k_score, axes, signed_at.

  2. 02

    Recompute the CID

    Hash the canonical manifest payload with SHA-256 and check the result matches the artifact_cid in the manifest. A mismatch means the manifest was tampered with after signing.

  3. 03

    Verify the signature chain

    For each signer in the chain, fetch the Ed25519 public key and verify the signature against the manifest hash. The chain ends at a kolm release key signed in CI on a tagged commit.

  4. 04

    Replay the K-score gate

    Read the eval-pack name and the recorded score. Confirm the score meets the gate threshold that the eval pack declares. A failing K-score is reported as the verdict, not silently passed.

  5. 05

    Cross-reference the registry

    If the artifact has a published CID, the page fetches the registry index entry and confirms the receipt CID matches. A mismatch points at a forked artifact that was not re-published, the verifier flags it as cid-not-in-registry.

  6. CLI

    Reproducible at the CLI

    The same five steps are exposed at kolm verify ./model.kolm. Same result, same exit code. Anything you see on this page can be re-run in CI or in an air-gapped environment.

Verify at the CLI

For engineers or for CI pipelines. The CLI uses the same five-step verifier as this page.

# install (Apache-2.0, single binary)
npm install -g kolm

# verify a local artifact
kolm verify ./model.kolm
  ok: true
  artifact_cid: bafy2bzace...
  k_score: 0.842
  signatures: 2/2 valid
  registry: matched

# verify a published CID without download
kolm verify --cid bafy2bzace...

# exit code is the contract: 0 = pass, 3 = invalid, 1 = error
kolm verify ./model.kolm && echo OK

# pin a release key fingerprint and refuse to verify against any other
kolm verify --signer-fingerprint ae21f3... ./model.kolm

Anatomy of a receipt

A .kolm receipt is a small JSON object. Real fields, real example. Nothing magical, nothing hidden.

Receipt fields

artifact_cid
The SHA-256 content-addressed hash of the canonical manifest. The hash the receiver recomputes in step 2.
compiler_version
The version string of the kolm compiler that produced the artifact, including the commit SHA. kolm@1.4.7+rev.a12bc.
teacher
The teacher model name and provider. claude-opus-4-7@anthropic or gpt-5@openai or any other identifier the distillation used.
base_model
The base model the artifact was distilled from or quantized over. Qwen2.5-7B-Instruct, Llama-3.1-8B, etc.
eval_pack
The eval pack name, version, and content hash. medical-claims-v3@bafy2bzace....
k_score
The composite K-score the eval-pack run produced. A single number between 0 and 1. Mid-80s and above is production-ready for the named eval pack.
axes
Per-axis breakdown of the K-score. Accuracy, refusal, latency, robustness, calibration. Procurement reads this row.
signed_at
ISO 8601 timestamp the signature was applied. Used to detect stale receipts against rotated keys.
signature
Ed25519 signature over the canonical manifest payload. One entry per signer in the chain.

Why a receipt and not a vibes claim

The argument for the receipt format, written for the developer that owns the model in production.

  • The team that shipped it can prove it

    VENDOR · signs

    A claim that says "our distilled model is 92% as good as the teacher" can be made by anyone. A signed receipt says the named eval pack scored 0.842 on the named base model on the named compiler version. The reviewer can replay.

  • The procurement reviewer can audit it

    REVIEWER · verifies

    Receipts are content-addressed and signed. The reviewer does not need to trust the vendor that compiled the artifact. The verifier runs on a disconnected laptop and the exit code is the contract.

  • The regulator can replay it

    REGULATOR · replays

    A receipt is reproducible. Give a regulator the .kolm, the eval pack, and the compiler version, and they get the same K-score. No proprietary harness, no black-box benchmark.
