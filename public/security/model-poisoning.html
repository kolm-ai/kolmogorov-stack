kolm.ai / security / model-poisoning

Open-source AI workbench

Model poisoning detection and quarantine.

A distilled student inherits whatever its captures contain. Poison the captures, you poison the student. ships four detection layers: statistical anomaly on capture telemetry, a copyright fingerprint heuristic, cryptographic binding of every capture to a verified teacher response, and an orchestrator that combines them into a four-level risk verdict. Quarantine is audit-trailed and reversible. Stamped w761-v1.

Why poisoning is the #1 threat to distilled models

The distill loop reads captures from production traffic and trains a smaller student to mimic the teacher. That loop has a critical assumption: the captures faithfully represent what the teacher said. Break that assumption and you control the student's behavior without ever touching the training code.

An attacker who can inject adversarial captures can:

  • Bias the student toward a target output the teacher would never have produced. (Imagine a refund-bot that silently approves any refund over $1000 because the captures said so.)
  • Force the student to leak training data from a different tenant by smuggling the data through capture text.
  • Degrade the student's K-Score until the operator gives up on distillation entirely and falls back to the more expensive teacher.

Conventional ML guardrails do not catch this. The student is doing exactly what it was trained to do: match the captures. The problem is the captures, not the model. attacks the problem at the source.

Four detection layers

Layer 1 ·

Statistical capture anomaly

Per-tenant per-namespace running mean and standard deviation on four axes: output length, vocabulary entropy, response time, and token overlap to the teacher's typical reply. Any axis more than 3 sigma from baseline flags. Cold-start protected: refuses to flag until there are at least 8 promoted captures in the namespace.

Layer 2 ·

Copyright fingerprint heuristic

Local regex pack matching well-known Disney character names, recognizable song titles, code copyright headers, and SPDX license identifiers. Catches the class of poisoned captures that try to smuggle protected works into the training corpus. Heuristic only — non-zero score means "looks like it may contain copyrighted content," not "confirmed violation."

Layer 3 ·-3

Teacher-response HMAC binding

The strongest signal. Every capture row carries a SHA-256 HMAC computed at the moment the teacher response was received, keyed by a tenant-controlled secret. MITM injection and cache poisoning both fail this check because the attacker does not have the key. Any post-bind mutation of the response body invalidates the binding.

Layer 4 · orchestrator

Combined four-level verdict

The orchestrator runs all three primitives and returns one of four risk levels. The ladder is monotone — any single layer can ONLY raise the risk, never lower it. HMAC failure ALWAYS escalates to rotate_teacher_key; it never falls through to safe even if the other two layers are silent.

Teacher-response HMAC binding

The cryptographic binding is the load-bearing signal in the four-layer stack. The hash chain is:

response_hmac = HMAC-SHA256(
  KOLM_TEACHER_HMAC_KEY,
  teacher_id || ":" || request_hash || ":" || sha256(response_body) || ":" || timestamp_ms
)

The binding envelope persisted alongside each capture also carries a 16-character key fingerprint — a SHA-256 hash of the active HMAC key, truncated. The fingerprint lets the verifier detect key rotation explicitly: if the active key's fingerprint differs from the binding's fingerprint, verification surfaces hmac_key_mismatch_post_rotation rather than ambiguously reporting "signature mismatch."

Verification uses crypto.timingSafeEqual — never plain equality on HMAC bytes — so a high-frequency attacker cannot mount a timing side-channel against the verifier.

Verification reasonWhat it means
valid_signatureBody matches the bound HMAC. Capture is from the configured teacher.
signature_mismatchBody was mutated post-binding or a different body was passed for verification.
response_body_mutatedAlias for signature_mismatch with a more specific diagnostic.
binding_missing_fieldsThe binding envelope is incomplete or absent on the capture row.
hmac_key_mismatch_post_rotationThe active key fingerprint does not match the binding's fingerprint.

Quarantine flow

Quarantine is audit-trailed and reversible. The orchestrator never deletes a capture — it appends a poisoning.capture_quarantined event to the per-tenant audit chain. The same chain that kolm audit verify walks. A subsequent poisoning.capture_released event records the release with the actor identity and the analyst's reason.

Quarantine is idempotent. Re-quarantining the same (capture_id, reason) tuple no-ops — the orchestrator returns the existing event without writing a duplicate. Release is owner/admin-gated at the route layer; the orchestrator itself only enforces the audit-trail contract.

POST /v1/poisoning/quarantine
{
  "capture_id": "ev_8f3a...",
  "reason": "hmac:invalid_signature",
  "evidence": ["anomaly:flagged[output_length,vocab_entropy]"],
  "confirm": true
}
→
{
  "ok": true,
  "capture_id": "ev_8f3a...",
  "reason": "hmac:invalid_signature",
  "quarantined_at": "2026-05-24T...",
  "audit_event_id": "aud_...",
  "version": "w761-v1"
}

Risk levels and what triggers each

safe

safe

HMAC verified (or no binding in legacy rows where it cannot be verified), no statistical anomaly, no copyright hit. The capture is safe to promote into the distill corpus.

review

review

Single 3-sigma anomaly axis OR copyright heuristic firing weakly OR the row carries no HMAC binding at all. No single signal is conclusive, but an analyst should look before promotion.

quarantine

quarantine

Two or more anomaly axes flagged simultaneously OR copyright heuristic firing above the 0.25 threshold. Hold the row outside the distill corpus until an analyst clears it.

rotate_teacher_key

rotate_teacher_key

HMAC verification FAILED. The capture's bound signature does not match the body, or the binding was signed under a key that no longer exists. This is the strongest possible poisoning signal — rotate the HMAC key and re-verify the window.

No-silent-pass contract

HMAC binding requires KOLM_TEACHER_HMAC_KEY set to at least 32 bytes of random data. Without it, every binding call returns {ok:false, error:'hmac_key_not_configured'} — we do NOT silent-pass with an empty key. Keys shorter than 32 bytes are refused with hmac_key_too_short.

Generate a key with: openssl rand -hex 32

HMAC verification failure ALWAYS escalates to rotate_teacher_key. The orchestrator NEVER returns safe when HMAC fails. A row whose binding cannot be re-derived is by definition not from the configured teacher — treat it as the strongest poisoning evidence the system can produce.
