SBOM and supply-chain pinning.

A signed receipt proves a .kolm artifact's bytes are the bytes that were minted — but a signature tells you nothing about what was in the build chain when those bytes were minted. A compromised dependency in the build pipeline produces a perfectly valid signature on a tainted artifact. W763 emits a Software Bill of Materials (CycloneDX 1.5 or SPDX 2.3) alongside every .kolm artifact and every kolm release so an auditor can verify the supply chain matches a known-good snapshot. Stamped w763-v1. Built by src/sbom-emit.js + apps/export/sbom.py.

Why SBOM matters for compiled AI artifacts

Every distilled student carries a hidden dependency graph: the tokenizer the teacher used, the calibration corpus tooling, the quantization kernels, the inference runtime, every transitive package those needed. Most of these are out-of-process and out-of-frame in the artifact-hash — the artifact has the weights, but not the chain that produced them.

An SBOM closes that gap. The same way the signed receipt proves the bytes you downloaded match the bytes that were compiled, the SBOM proves the dep graph that compiled them matches the snapshot you signed off on. When a CVE drops next month on, say, @anthropic-ai/sdk — you can ask: did any of my shipped artifacts come from a build with the vulnerable version? Without an SBOM, the only direct answer is "we don't know." With one, the answer is grep.

CycloneDX and SPDX

We support both major formats so downstream tooling (Snyk, Dependency-Track, scancode, syft consumers, FedRAMP attestation packets, EU CRA conformance dossiers) can pick the one they prefer. The Node + Python emitters round-trip every component through both shapes.

CycloneDX 1.5

Industry-standard SBOM published by OWASP. Strong tooling around vuln scanning + license compliance.

Format key: cyclonedx-json

Hash field: components[].hashes[].alg

SPDX 2.3

ISO/IEC 5962:2021. Required by some US-federal procurement contracts and a common ask in EU regulated industries.

Format key: spdx-json

Hash field: packages[].checksums[].algorithm

GitHub workflow for SBOM-on-release

.github/workflows/sbom.yml runs on every push to main and on every release event:

  1. actions/checkout@v4 + setup Node 20 with npm cache
  2. npm ci --omit=optional --ignore-scripts — verify the lock resolves cleanly (and that no postinstall hook touches the runner)
  3. npx @cyclonedx/cyclonedx-npm — canonical CycloneDX generation with validate-mode on
  4. Stdlib-Python backstop (apps/export/sbom.py) emits both CycloneDX + SPDX so we always have an SBOM even when cyclonedx-npm trips on a parse edge
  5. actions/upload-artifact@v4 archives the three SBOM files for 90 days
  6. If the CycloneDX step fails, an issue is opened with the run URL and the supply-chain+sbom+w763 labels — so a release is never silently shipped without an SBOM

Hash pinning recommendation

The SBOM emit path is read-only. We deliberately did not rewrite package-lock.json or requirements.txt as part of W763 — pinning every transitive dep with strict hash verification is a sustained audit, not a single ship. Here is the recommended hardening path; we will tighten across W763-followup waves.

  • npm. npm ci (not npm install) on every build — verifies package-lock.json integrity hashes resolve exactly. --ignore-scripts in CI to block postinstall hooks. Future: npm audit signatures as a gate.
  • pip. Use pip-tools's pip-compile --generate-hashes to produce a fully hash-pinned requirements.txt, then pip install --require-hashes -r requirements.txt. Without --require-hashes, a malicious mirror can serve a different wheel under the same version number.
  • cargo. Cargo.lock already carries integrity checksums per crate. Commit it. Use cargo install --locked. cargo audit as a CI gate.
  • GitHub Actions. Pin actions to a SHA (uses: actions/checkout@<sha>) rather than a tag. This repo is on the W763-followup roadmap for SHA pinning.
Plain scope: Strict hash pinning of every transitive dependency is a sustained audit, not a single ship. We emit SBOMs today; we will progressively tighten hash pinning across W763-followup waves. The SBOM surfaces the audit gap (it carries an unhashed_count on Python deps and a scope per npm dep) so an auditor can see exactly which lines need tightening.

Snyk + Dependabot integration

Dependabot is wired via .github/dependabot.yml — weekly PRs on the npm, github-actions, and pip ecosystems, all carrying the supply-chain+w763 labels so they land in the same review queue as SBOM tracking issues. Snyk is a customer-side choice: once you've enabled Snyk on the GitHub repo, the SBOM JSON uploaded by .github/workflows/sbom.yml is the input it scans.

Caveats note: Snyk and Dependabot integration via .github/workflows/sbom.yml runs on every push to main and on release events — not on tagged pre-releases that don't fire the release event. If you need an SBOM for a pre-release, run the workflow manually via workflow_dispatch.

CI workflow.github/workflows/sbom.yml — push to main + on release
Dependabot.github/dependabot.yml — weekly npm/actions/pip PRs

Every SBOM emitter (Node, Python, HTTP) stamps the same w763-v1 version field so a downstream consumer can verify which generator produced the file. Callers MUST regex-match (/^w763-/) rather than literal-compare so a v1.x bump in the same wave does not break their parser (the W604 anti-brittleness contract). Component counts match across emitters: a fresh checkout reports the same component count from kolm sbom repo as from python3 apps/export/sbom.py --package-lock.

Next steps

Start with kolm sbom repo --format cyclonedx-json > kolm-stack-sbom.json to see the SBOM of your running install. Run kolm sbom emit --manifest path/to/your-artifact-manifest.json to generate an SBOM for any compiled .kolm. Wire .github/workflows/sbom.yml into your fork so every release ships with an SBOM artifact attached.
