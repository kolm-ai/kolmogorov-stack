kolm.ai / security / threat model

Open-source AI workbench

Threat model.

The honest version of a threat model. We name every asset, every adversary, every threat, and every mitigation — and we name the parts we have not yet proven. The point of this page is to make it cheap for a security reviewer to find the gaps and challenge them, not to assert a defended boundary we have not built. See /security/posture for the controls inventory and /security for the overview.

Asset inventory

The things worth protecting on the kolm.ai stack. Anything not on this list is either out of scope or downstream of one of these.

  • Captures · tenant-private Raw prompts, completions, tool outputs, retrieval contexts, and run metadata logged by the tenant's calls. The lake holds them per tenant per namespace; they never cross the tenant boundary except by explicit publish. Leak of a tenant's captures is the highest-severity data event we plan for.
  • Distilled artifacts · tenant-private .kolm artifacts produced by the compiler against a tenant's captures and chosen teacher. The artifact contains quantized weights, the signed receipt, the eval pack reference, and the K-score axes. Tenant-private by default; only the receipt CID becomes public if the tenant publishes.
  • API keys · auth credentials Per-tenant API keys minted at signup and rotation. We store only the SHA-256 hash in the control plane; the plaintext exists only on the client. A leaked key authenticates the holder until rotated; an audit log entry on every issued request makes the leak detectable.
  • Event-store rows · audit trail Append-only event rows covering auth, key issuance, namespace mutation, plan change, artifact publish, takedown, and admin action. Used by SOC-2 evidence collection and tenant-side audit pulls. Tampering with this trail is the high-severity integrity event we plan for.
  • Source code · public The kolm stack at github.com/kolm-ai/kolmogorov-stack. Apache-2.0. Source disclosure is by design — the same code an auditor reviews is the code that runs. The risk to manage here is supply-chain on the build pipeline, not source-reading.

Adversary model

The actors we plan against. Each adversary has a capability boundary; the threats below pair an asset with the adversary that could reach it.

  • External attacker (no credentials). Network-reachable internet adversary with no kolm account. Capability: probe public endpoints, attempt auth bypass, run vulnerability scanners, attempt CDN/edge attacks. Cannot present a valid API key.
  • Low-privilege tenant user. Holds a valid API key for tenant A. Capability: legitimate API calls within tenant A's plan + quota; may attempt cross-tenant reads by spoofing tenant ID, namespace, or row ID. Cannot reach the control-plane admin surface.
  • Compromised dependency (npm / pypi / cargo / actions). A transitive package or GitHub Action used in the build pipeline ships a backdoor. Capability: execute arbitrary code on a build runner; influence the bytes of a released artifact; insert a backdoor that lies dormant until activated.
  • Insider with op access. A Kolm operator with legitimate access to a subset of production systems (e.g. on-call SRE). Capability: read storage volumes, mutate config, deploy code. The intent is malicious or coerced; the control framework treats them as a separate adversary even though their access is legitimate.
  • Regulator (audit). Not an attacker in the adversarial sense; an actor who requires evidence that the controls we claim are in fact in place. Capability: subpoena, audit, request reproducible evidence, demand data-residency proofs. We model them as an adversary because they generate the same evidence requirements an attacker discovers under incident response.

Out of scope

Threats we do not defend against on this stack. We list them honestly so a downstream operator can decide whether their environment closes the gap.

  • Physical-server compromise. An attacker with physical access to the host running the hosted control plane is out of scope — we depend on the hosting infrastructure (data-center physical controls, hardware tamper-evidence, on-premise customer controls) for that boundary. The same applies to on-premise deployments: the operator owns physical security of their hardware.
  • Attacks via the configured teacher API. The kolm distill path calls a teacher endpoint the tenant chose (Anthropic, OpenAI, a self-hosted vLLM/SGLang/TGI). We trust the configured teacher to behave as specified. A malicious teacher could return distillation data designed to poison the student; we do not defend against that. Tenants that need to defend against a malicious teacher should pin to a known-good teacher and treat distillation output as untrusted.
  • User-device compromise. The user's laptop, browser, terminal, or CI runner is out of scope. If the API key on the user's laptop is exfiltrated by malware on that laptop, no server-side control can prevent it. Rotation, audit-log review, and per-tenant rate limits are the recovery path.
  • Side-channel attacks on the inference runtime. Timing, power, and cache side channels against a deployed .kolm running on the tenant's hardware are out of scope. Operators in regulated settings should run the inference path on hardware with the relevant side-channel mitigations enabled.
  • Social engineering of operators or tenants. Phishing, pretexting, and pressure on a person to grant access are out of scope of the kolm stack. We document the runbook (rotation, audit-log review, account lock); we do not promise to prevent the social-engineering attempt itself.

Open questions

Things we are still figuring out. Each item is a real engineering question we have not yet resolved to a signed proof; we list them on the public site so a security reviewer can challenge our current reasoning.

  • Cross-tenant model-leak via fine-tune. If tenant A's captures train a distilled artifact, and tenant B later runs a fine-tune that includes membership-inference probes against the same artifact, can tenant B recover capture content from tenant A? We believe the W411 tenant fence + per-row defense-in-depth holds because tenant A's artifacts are not reachable by tenant B in the first place — but the formal proof against membership-inference under a shared base-model assumption is still in progress. See /security/membership-inference for the in-progress write-up.
  • Receipt forgery under a quantum-capable adversary. Ed25519 receipts are signed today. A quantum-capable adversary in the future could break Ed25519 retroactively. The forward path is post-quantum signing (Dilithium / SPHINCS+); we have not yet shipped a PQC migration. Open question: at what point do we migrate, and how do we keep historical receipts verifiable through the migration?
  • Eval-pack poisoning. If a tenant publishes an eval pack that the wider ecosystem references, can a malicious tenant craft an eval pack that grades unsafe outputs as high-K-score? The K-score gate is a tenant-chosen contract; a malicious tenant can grade their own artifact however they like. The open question is whether public-registry eval packs need a separate trust signal beyond the CID hash.
  • Compromised teacher detection. Today we trust the configured teacher (see Out of scope). Open question: should the distill worker run a sanity check on teacher output (e.g. canary-prompt detection) and refuse to compile if the teacher fails it? We have not yet shipped this and have not yet decided on the policy.
  • Audit-log offsite hash chain. The event store is append-only at the application layer. A sufficiently privileged insider with database access could in principle rewrite history. The mitigation is a periodic offsite hash chain that an external observer can verify; we have not yet shipped that publication step. Open question: cadence (per-hour? per-day?) and where to publish.
If you find an item here that you think we have answered (or one we should add), email security@kolm.ai with a writeup. We acknowledge within 24 hours and aim to triage within 72.
