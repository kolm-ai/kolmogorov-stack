Security posture.

Honest security posture. What we do today, what we have committed to, and the public disclosures that bind us to those commitments. This page is a deeper dive than the security overview — it names the controls in production and the controls still on the roadmap with target dates. If a row is missing here it is because the control is still on the roadmap; we do not list aspirational controls.

Architecture

Kolm separates three planes so an attacker who breaches one does not automatically compromise the others. The gateway holds auth and rate limits. The control plane holds tenant metadata, namespaces, and API keys. The data plane holds the captures, the distilled artifacts, the event-store rows, and the worker pools (distill, eval, redact). The diagram below shows the default hosted shape; the same shape ships on-premise with the gateway and control plane co-located.

What we do

The controls below are live in the codebase today and exercised on every request.

  • Auth · API key SHA-256 hash, never plaintext API keys are presented once at signup, then we store only a SHA-256 digest in the control plane. A database snapshot does not leak useful credentials. Lost-key recovery is rotation, not retrieval, because we cannot retrieve a hash. See src/auth.js and the signup flow at /signup.
  • Tenant fence · W411 invariant + per-row defense in depth Every query against the event store, the artifact registry, and the capture lake filters by tenant ID at the query layer (W411 invariant). Each row also carries the tenant ID inside the row payload, and every read path re-checks it before returning the row to the caller (defense in depth). A bug in the query layer alone cannot leak cross-tenant data.
  • Encryption in transit · TLS 1.3 enforced at the gateway The hosted gateway terminates TLS 1.3 only. HTTP/1.1 plaintext is refused. HSTS with max-age=63072000; includeSubDomains; preload. On-premise deployments inherit the operator's TLS config; the recommended baseline is the same.
  • Encryption at rest · storage-layer provided The event store, artifact registry, and capture lake live on managed storage volumes whose at-rest encryption is provided by the underlying storage layer (AES-256-GCM on the hosted side; whatever the operator configures on-premise). We do not layer a separate at-rest cipher on top — see What we don't yet do for that honesty.
  • Audit log · W770 append-only event sink Auth events, key issuance, key rotation, namespace mutation, plan change, artifact publish, takedown, and admin action all land in an append-only audit log. Tenant operators can pull their own log via kolm audit; SOC-2-bound enterprise customers can stream it to their SIEM. See /security/sbom for the supply-chain side and src/audit.js for the schema.
  • SBOM · W763 CycloneDX 1.5 / SPDX 2.3 per release Every .kolm artifact and every kolm release ships with a Software Bill of Materials. CycloneDX 1.5 and SPDX 2.3 supported. Emitted by src/sbom-emit.js + apps/export/sbom.py + .github/workflows/sbom.yml. See /security/sbom for the full surface.
  • Supply-chain pinning · W763 lockfile integrity npm ci on every build (verifies package-lock.json hashes resolve exactly), --ignore-scripts to block postinstall hooks, Dependabot weekly on npm + actions + pip with the supply-chain + w763 labels. Hash pinning of every transitive dependency is a sustained audit — we are tightening across W763-followup waves and the SBOM surfaces the gap.
  • Artifact hash binding · W460 tamper evidence Every .kolm artifact embeds a content-addressed artifact hash inside its signed receipt. Editing weights, swapping an eval pack, or rewriting the K-score after the fact invalidates the signature. The verifier is offline (kolm verify on any artifact zip).

What we don't yet do

Honest list of controls we have not yet shipped. Each item has a roadmap reference or a stated reason.

  • End-to-end encryption of capture payload at rest Capture payload bodies (prompt text, completion text, tool outputs) are encrypted only by the storage layer cipher (AES-256-GCM on the hosted side). We do not layer a tenant-held key on top, so an attacker with full storage-layer access could read decrypted capture text. Tenant-managed envelope encryption is on the W670+ roadmap; until then, regulated tenants should run the on-premise data plane.
  • SOC 2 Type II report signed SOC 2 Type I is available under NDA (W767). The Type II observation window is underway with the same auditing firm. The Type II report is not signed yet — do not treat an interim controls letter as a signed Type II. Expected end of observation window: Q4 2026. See /security for the Type I detail and the procurement contact.
  • Full BYOC (bring-your-own-cloud) option W742 ships the local-only mode — the entire kolm stack on the operator's own hardware with no outbound connection required. Full BYOC, where the control plane runs inside the operator's cloud account but with a Kolm-managed update plane on top, is on the roadmap and not yet shipped. Until then, the choice is hosted (kolm-managed) or local (operator-managed).

Report a vulnerability

Email security@kolm.ai with a writeup. We acknowledge within 24 hours and aim to triage within 72.

Scope

  • The hosted gateway and control plane at kolm.ai and kolm.ai/v1.
  • The kolm CLI and the Node, Python, Rust, C, and MCP SDKs in the repo.
  • The verifier (kolm verify) and the signed receipt format.
  • The artifact registry and the public CID resolver.

Out of scope

  • Customer-resident deployments (on-premise, air-gapped) — report those to the operating tenant directly.
  • Third-party teacher endpoints (Anthropic, OpenAI) — report to the relevant vendor.
  • Findings that require physical access to a host we do not operate.

Disclosure policy

90-day coordinated disclosure window from a verified report. We will request an extension in writing if a fix requires longer (rare, and we will not request more than 30 additional days). Credit acknowledged in the changelog if the reporter wants it; anonymized otherwise. Bug bounty is selective and offered case-by-case for impactful findings.
