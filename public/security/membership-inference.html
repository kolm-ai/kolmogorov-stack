Membership inference and the right to be forgotten.

Membership inference asks a sharp question of any distilled model: can an attacker prove that a specific captured row was in the training corpus? If yes, that capture has leaked. This page documents the four-attack harness we ship (kolm mit), the ten-category PII scanner that rides alongside bakeoff, and the audit-trailed kolm forget mechanism that backs the GDPR right-to-erasure and the CCPA right-to-delete.

1. What membership inference is.

A trained model that memorizes individual training rows is a privacy hazard. Membership inference is the canonical attack: an adversary holds a candidate prompt/response pair and asks the model whether that exact pair was in the training set. If the model answers reliably (high confidence that the row was a member), the model has memorized the row instead of learning the underlying distribution.

For a regulated buyer the implications are concrete. Every captured row in a distill corpus might contain a customer name, a support-ticket number, an internal URL, or a code snippet pulled from a private repository. If the resulting .kolm artifact will replay any of those strings under interrogation, the artifact is leaking the corpus.

The harness in src/membership-inference-test.js turns this question into a measurable score. For each capture in the corpus we run four attack kinds through the artifact and score the Jaccard 5-gram overlap between the artifact's response and the capture's original response. Overlap above the threshold (default 0.85) flags the row as leaked. The route POST /v1/mit/run returns {extraction_rate, by_attack_kind, leaked_captures}.

2. The four attack kinds.

The harness ships with four canonical attacks, frozen in MIT_ATTACK_KINDS (re-order is a deliberate breaking change because attribution dashboards key on the array index):

KindWhat it probesWhat a leak looks like
exact_prompt_replayCaptured prompt fed back verbatim.Artifact emits something close to the original response token-for-token.
paraphrase_promptNaive word-shuffle of the captured prompt.Artifact still emits the memorized response even though the surface prompt differs.
partial_prompt_completionFirst half of the prompt only; artifact completes the rest.The completion reproduces the captured response, proving the prompt is enough of a key.
unique_token_probeRarest long token from the response is checked in an unrelated probe prompt.Artifact emits the same rare token despite no contextual cue, indicating verbatim memorization of a fingerprint string.

Each attack reports its own count in by_attack_kind. A capture flagged by unique_token_probe is a stronger signal than one flagged only by exact_prompt_replay, because the unique-token attack does not feed the original prompt back at all.

3. PII scanning during bakeoff.

Membership inference measures memorization. PII scanning measures something complementary: when the artifact emits text, does it leak personal data regardless of whether that exact row was in the corpus? The W764-2 scanner runs over every bakeoff response and flags hits across ten categories.

CategoryMethod
emailRFC 5322-lite regex.
phone_us(NNN) NNN-NNNN and dotted/dashed US formats.
phone_intlLeading + country code with 7-14 digit body.
ssn_usStrict NNN-NN-NNNN with first triplet not 000/666/9XX.
credit_card_luhn12-19 digit run that passes the FULL Luhn mod-10 algorithm. Regex alone is not enough.
aws_access_keyAKIA prefix + 16 uppercase alphanumerics.

name_likely HEURISTICCapitalized adjacent words. Flagged with heuristic:true so downstream pipelines do NOT auto-redact.

The route POST /v1/mit/scan-pii runs the scanner over arbitrary text. The bakeoff entry point runPiiBakeoffScan({prompts, runOnArtifact}) runs prompts through the artifact and aggregates hits by category and a pii_rate across the prompt set.

4. The kolm forget mechanism.

Marking a capture as forgotten writes a durable, audit-trailed event into the canonical event store. The marker carries the capture id, the reason, the requester, and the timestamp. Future distill builds that pull from the event store will filter the marked rows out via filterForgottenCaptures(captures) before training begins.

# Mark a single capture for exclusion from future builds.
$ kolm forget --capture-id evt_4b7bc3b... \
              --reason gdpr_erasure_request \
              --confirm
{ "ok": true,
  "capture_id": "evt_4b7bc3b...",
  "forgotten_at": "2026-05-24T18:42:11Z",
  "audit_event_id": "forget_a91f...",
  "requires_redistill": true,
  "version": "w764-v1" }

# List every forget marker for this tenant.
$ kolm forget list --namespace customer-support

Idempotency is built in. Re-running markCaptureForgotten for the same (tenant_id, capture_id) returns the original audit_event_id unchanged. A retry loop never writes a duplicate audit row.

Defense-in-depth tenant fences live at three layers: the route reads req.tenant_record.id, listEvents filters by tenant_id in the query, and the per-row loop body re-checks row.tenant_id === tenant_id so a future schema bug cannot silently leak markers across tenants.

6. Honesty note.

Forgetting a capture marks it for exclusion from future builds — existing artifacts trained with it are NOT modified retroactively. Re-distill (kolm distill --rebuild) is required to actually purge.

Distilled artifacts are static binaries; we cannot remove a row from a model once training is complete. The forget mechanism guarantees that the row will not be in the next training run, and the audit trail proves to a regulator that the request was received and acted on at a specific timestamp. To satisfy a true erasure request end-to-end the operator must re-distill and re-deploy any downstream artifact built from the marked corpus.

We chose this design over a silent filter for two reasons. First, an audit trail is a regulatory requirement under GDPR Art. 30; a silent filter cannot produce one. Second, an existing artifact that was already shipped to a customer is outside the workbench's control — surfacing the re-distill requirement loudly is the only honest way to handle that gap.
