High-risk verticals · runtime envelope · signed receipts

Liability and high-risk verticals.

Medical, legal, and financial outputs carry consequences beyond software. Kolm injects a runtime disclaimer envelope when an artifact's manifest declares a high-risk vertical, signs every run with Ed25519, and keeps the burden of professional judgment with the operator. Here is what kolm does, and what it does not.

01The runtime disclaimer envelope

When you set vertical on your spec to one of medical, legal, or financial, spec-compile writes a disclaimer_injection field into the manifest. The runtime reads that field and wraps every generated output with the exact disclaimer text, verbatim. The user cannot suppress it from the wrapping; they can only suppress it by recompiling without the vertical declared.

# Spec snippet for a medical-vertical artifact
{
  "job_id": "job_soap_redactor",
  "task": "Redact PHI from clinical SOAP notes",
  "vertical": "medical",
  "recipes": [ ... ]
}

# Resulting manifest contains:
"vertical": "medical",
"disclaimer_injection": "This output was generated by a distilled AI model. It is not a substitute for professional medical advice. Verify with a qualified professional before acting."
02What kolm provides
CapabilityWhat you get
Deterministic compile pathThe same spec, the same teacher, the same corpus yields the same artifact hash. Audits can re-derive your build.
Signed receipts (Ed25519)Every run produces a receipt signed by the local keypair. The receipt proves which artifact ran on which input at which time on which device.
Manifest vertical stampmanifest.vertical is hash-bound. The runtime cannot strip it without invalidating kolm verify.
Disclaimer envelopeInjected automatically when manifest.vertical is in the high-risk set. The text is canonical; downstream consumers can parse it programmatically.
Open compile + open runtimeThe compile bridge and the runtime are both source-available. Your counsel can review the exact code path that produced the output.
03What kolm does not do

Kolm is not a medical device. Kolm is not law-firm counsel. Kolm is not a registered investment adviser. The disclaimer envelope is a hygiene measure, not a guarantee. The operator who deploys the artifact is the responsible party for any downstream decision the artifact's output informs.

Kolm does not warrant that the disclaimer text satisfies any specific regulatory regime (HIPAA, GDPR, SEC Rule 206(4)-7, state bar rules, FDA SaMD framework, or others). The text is a reasonable default. Your counsel may require additional language; you can extend the envelope in your runtime wrapper.

04The audit story

When something goes wrong in production, the question is "what model produced this output, when, on what input." Kolm answers all three from a single receipt. The receipt carries the artifact hash, the vertical stamp, the disclaimer text that was injected, the input hash, the output hash, the device fingerprint, and the timestamp. The keypair signature ties the row to the device.

$ kolm verify ./run-receipt-2026-05-23T14-22-08.json
artifact_hash : a91d4f0c...
vertical      : medical
disclaimer    : (verbatim injected text)
input_hash    : 7c3e...
output_hash   : 9b18...
device        : kolm-rtx5090-prod-01
timestamp     : 2026-05-23T14:22:08Z
signature     : ed25519:valid
ok            : true
05Bug bounty and responsible disclosure

If you find a security issue, a verifier escape, a signature-forgery path, or a way to strip the disclaimer envelope without invalidating kolm verify, please report it directly to dev@kolm.ai. We acknowledge reports within 72 hours and aim to ship a fix in the next release wave.

Please do not file public issues for security reports. We credit reporters in /hall-of-fame after fixes ship, unless you ask to remain anonymous.
