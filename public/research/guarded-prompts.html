Guarded prompts

Guarded prompts separate trusted instructions from untrusted content. kolm uses that boundary so user messages, tool results, and retrieved documents cannot quietly rewrite the operator's intent.

What it is

Prompt injection (Perez and Ribeiro, 2022; Greshake et al., 2023) is the failure mode where attacker-controlled text in a tool result or retrieved document overrides the system prompt. Guarded prompts attack the problem at the template layer: every untrusted slot is wrapped in a sentinel-delimited block, every system instruction reasserts what the slot is and what the model should do with it, and the template explicitly names which instructions are authoritative. Hines et al. (2024) call the encoding trick spotlighting: transform the untrusted text (base64, datamarking, encoding) so that any instructions inside it look obviously out-of-distribution at inference time. Wallace et al. (2024) propose instruction hierarchy: train the model to assign priority levels to messages from system, developer, user, and tool sources, with explicit rules for resolving conflicts. None of these eliminate injection; they raise the cost.

How kolm uses it

Guarded prompts are an opt-in capture transform in kolm v0.2.6. When enabled on a recipe (guards: ["spotlight", "hierarchy-tags"]), the proxy wraps untrusted slots in named XML-style tags before recording the capture and before forwarding to the upstream model. The wrapper template is stored on the recipe, not in each prompt, so a downstream auditor can verify what shape every captured prompt had at training time. The compile loop trains the student on the same wrapped form; the runtime applies the same wrapper at inference. Instruction-hierarchy training (Wallace et al.) ships in v0.3 alongside the GRPO recipe. The K-score includes an optional injection-pass-rate gate that runs a small adversarial probe set against the candidate before signing.
