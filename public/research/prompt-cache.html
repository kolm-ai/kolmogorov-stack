Prompt caching

Prompt caching can mean provider KV reuse or semantic reuse through a cache. Both reduce repeated teacher calls, and both matter when a capture pipeline sees the same task many times.

What it is

KV-cache prefix sharing is a server-side optimization. When two requests start with the same tokens (a system prompt, a long document, a few-shot block), the attention key-value tensors for that prefix can be computed once and reused. vLLM's PagedAttention (Kwon et al. 2023) and SGLang's RadixAttention (Zheng et al. 2024) generalized the idea across requests at the engine level. Anthropic, OpenAI, and DeepSeek expose it as a per-message cache_control hint with a published discount (cached input is ~10 percent of full input). Semantic prompt caching is a client-side optimization: embed the prompt, look it up in a vector index of past prompts, and if the cosine distance to the nearest is below a threshold, return the cached response. GPTCache (Bang 2023) is the reference open-source implementation. Semantic caches trade exactness for hit rate; on production traces with repeated questions, hit rates of 20 to 60 percent are common, but every false positive returns the wrong answer.

How kolm uses it

KV-prefix caching is wired into the distillation stage. The synthesis pipeline batches teacher calls by stable prompt prefix (system prompt + recipe-fixed few-shot block) and emits the cache-control hint, so the typical distillation run pays the ten-percent rate for the prefix on every call after the first. The receipt records cache_hit_rate and cache_savings_usd so the buyer can verify the bill. Semantic prompt caching stays separate from the capture loop because the lake needs every request as a training example. Buyers who want semantic runtime caching can layer GPTCache or LiteLLM in front of the kolm runtime; the recipe is documented in /docs/optimizer. The first-party semantic-cache track preserves tenant isolation and emits cache-hit events to the capture log.
