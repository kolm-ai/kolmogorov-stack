kolm.ai / research / sbom

SBOM for .kolm artifacts

An ML SBOM records the corpus hashes, base model, library versions, and license posture behind a build. Every signed .kolm ships with one downstream consumers can validate.

What it is

SBOMs entered the public-policy mainstream with the 2021 US Executive Order 14028, which required all federal software vendors to provide a Bill of Materials. The two dominant formats are CycloneDX (OWASP, 2017) and SPDX (Linux Foundation, 2011, ISO/IEC 5962:2021). Both enumerate components by identifier, version, supplier, and license; both support attestation linkage to in-toto and SLSA provenance documents. The CycloneDX 1.5 spec added explicit ML-BOM support: enumeration of training datasets, model lineage, evaluation results, and considerations. The SPDX 3.0 AI/ML profile covers similar ground. For ML artifacts the items a buyer wants to enumerate are the source corpus, the teacher model used during distillation, the base student model, the optimizer and trainer versions, and any third-party adapters merged in.

How kolm uses it

Every .kolm artifact ships with both a CycloneDX 1.5 and an SPDX 3.0 SBOM next to the manifest. The SBOM enumerates: the corpus content-address and row count, the teacher model id and provider, the base student model id and weights hash, the trainer git revision and config hash, the python-package lockfile hash, and the per-component license string. CycloneDX's ML extensions are used to attach the holdout K-score and the receipt-chain root. The CLI kolm sbom <artifact> prints both formats; the runtime exposes them at /v1/artifacts/<id>/sbom. Per-component license posture is computed from the recipe-declared license of each input plus the trainer's transitive package licenses; conflicts (a GPL package on an Apache base model with an MIT distribution) are flagged at compile and block signing unless the recipe explicitly resolves them. Continuous vulnerability scanning of the SBOM against the OSV database ships in v0.3.
