kolm-meta -- the meta-distillation model.

Given your capture statistics and a chosen student architecture, kolm-meta predicts the K-Score you should expect, the compile time, and the failure modes most likely to bite you. XGBoost-style gradient boosting trained on aggregated run telemetry. Below 1000 training rows we fall back to the TAAS rule ladder with an loud-fail envelope.

Predicts K-Score + compile time + failure modes Tenant-scoped aggregates only, never raw text Falls back to TAAS ladder under n=1000
What kolm-meta predicts

The numbers, before you spend the GPU hours.

The use case is "should I distill this corpus into this architecture, or do I need more captures first?" The model answers that without you paying the compile cost.

  • Predicted K-Score

    scalar in 0..1 + 80% CI

    A scalar prediction with an 80 percent confidence interval. Tight CIs indicate kolm-meta has seen runs like yours before; wide CIs are the plain "we do not know" signal.

  • Predicted compile time

    seconds at chosen hardware

    Predicted wallclock seconds at the hardware you select (5090, H100, B200, M-series Mac). Used to estimate cost before you start.

  • Likely failure modes

    ranked by probability

    Ranked list of failure modes seen on runs with similar fingerprints. Examples: distribution-shift on long-tail intents, overfit on under-represented language, judge-disagreement on edge cases.

Training data spec

One row per completed distill run.

Each row carries the run's capture-corpus statistics, the chosen architecture, and the observed outcome. No raw capture text is ever included; only aggregate statistics that cannot reconstruct the underlying captures.

{ "capture_stats": { "n_captures": 14200, "avg_input_tokens": 187, "avg_output_tokens": 95, "intent_entropy": 2.34, "language_dist_top3": ["en", "es", "fr"], "tool_call_pct": 0.18 }, "chosen_arch": "qwen2.5-3b-int4", "observed_kscore": 0.91, "compile_time": 13.85, "observed_failure_modes": ["long_tail_intent_shift", "low_confidence_on_es"] }
Architecture

XGBoost-style gradient boosting (initial v0).

For v0 we chose gradient-boosted trees because the feature space is small (10s, not 1000s), the target is well-behaved scalars, and the model is interpretable -- you can read the splits and know why kolm-meta predicted what it did.

  • One model per target

    K-Score, time, failure-mode classifier

    One regression model per scalar target (K-Score and compile time) and a multi-label classifier for failure modes. Separate models let each tune independently and let the failure classifier carry per-mode calibration.

  • Interpretable splits

    SHAP attributions per prediction

    Each prediction ships with SHAP attributions so you can see which capture-stat features pushed the predicted K-Score up or down. No black box; the answer comes with its receipts.

  • v1 candidates

    tab-transformer, simple MLP

    v1 will compare gradient-boosted trees against a tab-transformer and a vanilla MLP on the same training rows. We will publish the head-to-head with K-S divergence on held-out runs.

Fallback and privacy

Under n=1000 we use the TAAS rule ladder.

When the per-tenant or per-vertical training set is below 1000 rows, kolm-meta refuses to predict and falls back to the W716 TAAS rule ladder. The response envelope carries meta_insufficient_data so the caller never confuses a heuristic with a learned prediction.

{ "predicted_kscore": null, "predicted_compile_time_s": null, "predicted_failure_modes": null, "fallback_used": "taas_rule_ladder_v716", "fallback_predicted_kscore_range": [0.82, 0.94], "envelope": "meta_insufficient_data", "training_rows_seen": 412 }

  • Tenant-scoped aggregates only

    never raw capture text

    Training rows are aggregate capture statistics keyed by tenant and vertical. We never ship raw capture text or model outputs into the kolm-meta training set. Statistics chosen so a single capture cannot be reconstructed.

  • Opt-in

    per tenant, per namespace

    Tenants opt in per namespace. Without opt-in we still produce TAAS-ladder predictions but the tenant's runs do not contribute to kolm-meta training. Symmetric: contributors and non-contributors get the same predictions.

  • Federation-friendly

    aggregates cross tenant boundary

    Because rows are aggregates by design, kolm-meta is compatible with the W461 federated approval-row sharing model: only hashed, noise-protected fingerprints cross the tenant boundary.

Loud-fail envelope. When kolm-meta cannot predict, the response is meta_insufficient_data with a clear count of training rows seen so far. We do not silently emit a heuristic prediction styled as a learned one. Every kolm-meta prediction ships with its SHAP attributions, the per-target confidence interval, and a pointer to the training-row cohort it generalized from.
