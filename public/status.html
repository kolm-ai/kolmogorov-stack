DEPLOY SEQUENCE · backend first, then the site
# from repo root
npm.cmd run lint:refs
npm.cmd run verify:claims-scope
npm.cmd run ui:audit:critical
npm.cmd run local:surfaces:deep -- --json

# deploy backend API target used by Vercel rewrites
railway up

# deploy public site shell and static assets
npx.cmd vercel --prod

# post-deploy gate
curl https://kolm.ai/ready
curl https://kolm.ai/v1/product/graph

A green uptime light means the system is running - not that every feature is finished. We keep those two things apart on purpose, so a working service is never confused with one we still call done.

What's still in the works

We show what isn't done yet.

No buried roadmap. The work that's still in progress - packaged releases, outside verification, published benchmarks - sits right here in the open, next to everything that's already live.

Each item is held behind a readiness gate (a release blocker we won't pass until the proof exists)A readiness gate is a checkpoint a feature must clear before we call it shipped. We keep it closed until there is evidence you can verify, so a claim never gets ahead of the proof. The live count is read straight from /product-readiness-closeout.json.. Right now 8 of them are still open.

Still being finished
