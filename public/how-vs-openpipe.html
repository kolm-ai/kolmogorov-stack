kolm.ai  /  compare  /  how kolm relates to OpenPipe

Both capture the same traffic. Only one ships you the file.

OpenPipe wraps your OpenAI/Anthropic SDK calls and logs every prompt/completion pair. Then it fine-tunes from the logged traffic and serves the result as a hosted endpoint. kolm captures the same traffic into seeds.jsonl and compiles a signed artifact you keep. Same input shape. Different output shape.

Wave 169 · reviewed 2026-05-17 · SDK wrap vs. file emit

What OpenPipe ships in 2026.

  • SDK wrap, drop-in replacement for OpenAI/Anthropic SDK that logs every prompt/completion pair to OpenPipe's backend.
  • Hosted fine-tune, built on TRL + HuggingFace; base models include Llama 3.1. Claim: "fine-tuned Llama 3.1 outperforms GPT-4o" on captured-traffic tasks.
  • Hosted serve, deploy the tuned model on OpenPipe's infrastructure, or BYO inference.
  • 2025-2026 pivot, repositioning toward "RL for Agents", reinforcement-learning agent fine-tuning beyond pure prompt → completion distillation.
  • Pricing, 30-day free trial; usage-based pricing afterward.

What OpenPipe doesn't ship: a portable signed file (the output is a hosted endpoint or downloadable LoRA weights without a receipt chain), an independent eval gate (their eval is what the captured-traffic dataset says), a PHI redactor before the teacher call, a cross-vendor teacher catalog with provenance manifest, or air-gapped/offline deployment.

Migrate the OpenPipe capture loop to kolm in one session.

# 1. Replace OpenPipe SDK base URL with kolm-proxy
# (kolm-proxy is OpenAI-compatible; same client code keeps working)
$ kolm proxy --capture-to=./seeds.jsonl --upstream=https://api.openai.com/v1
 LISTEN 127.0.0.1:7117 · capturing → ./seeds.jsonl

# 2. Export your existing OpenPipe dataset to seeds.jsonl format
$ kolm import openpipe --project=my-project --out=./seeds.jsonl
 WROTE 12,847 rows · canonical format {"input","output","tags"}

# 3. Distill, with the same teacher you were already using
$ kolm distill my-task \
 --teacher=openai:gpt-5 \
 --student-base=llama-3.2-1b \
 --redact=phi --eval-gate=0.85

 PASS K-score 0.892 · train/holdout disjoint · PHI redactor clean
 WROTE my-task.kolm · 184 MB · Ed25519 + Sigstore signatures

# 4. Verify, then run on target
$ kolm verify my-task.kolm
 PASS 24/24 checks
$ kolm run my-task.kolm --input "..." · # local inference, no network

When OpenPipe is the right answer.

You want hosted inference, not a file. Your team doesn't need a signed artifact or a cryptographic receipt chain. The "drop-in SDK wrap + auto-fine-tune + hosted endpoint" flow is the shortest path from production traffic to a cheaper-than-GPT-4 endpoint, and OpenPipe is the cleanest implementation of that pattern. The 2025 RL-for-Agents pivot is interesting and orthogonal to anything kolm is doing.

When kolm is the right answer.

You need the artifact to be a file: for air-gap deployment, for compliance receipts, for residency control, for cross-vendor distillation provenance, or because your buyer's auditor asked. kolm captures the same traffic shape OpenPipe captures, but the output is a portable signed binary instead of a hosted endpoint. The migration is one CLI session.
