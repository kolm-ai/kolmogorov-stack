02 / Intake Packet

Send enough context to prove the loop without exposing unnecessary data.

The best first contact is concrete: one workflow, one owner, one evidence destination, and the current risk that blocks production.

1. Source

Where behavior starts

Provider logs, gateway events, agent traces, product telemetry, support tickets, warehouse rows, CI events, repository context, or custom API events.

2. Risk

What must not regress

Customer action, data boundary, workflow failure, latency target, cost ceiling, policy violation, escalation path, or reviewer concern.

3. Gate

How release is blocked

Eval threshold, regression set, security review, procurement evidence, CI status, runtime compatibility, or buyer control mapping.

4. Target

Where artifact runs

Hosted GPU, local runner, browser worker, edge function, private cloud, restricted fleet, offline verifier, or deployment check.

5. Export

Where proof must go

Receipt, eval report, evidence bundle, JSON API, webhook, warehouse export, SIEM/GRC event, ticket, or CI deployment status.

6. Caveat

What cannot be claimed yet

Benchmark, certification, package-channel, partner-adoption, and standards-stewardship claims stay readiness-gated until external proof exists.

03 / Owner Questions

The first conversation should identify owners and control objects.

If the workflow has no source owner, policy owner, runtime owner, evidence owner, or failure owner, it is too vague to control.

Send Owner Map Enterprise workflow

AI Owner

Security Owner

Which retention, redaction, vault, routing, least-privilege, unknown-schema, and export policies apply before reuse?

Evidence Owner

Who receives the receipt, governance packet, SIEM/GRC event, warehouse export, ticket, or deployment status?
