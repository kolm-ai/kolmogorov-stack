First-class runtime adapters

Each adapter takes a path to a .kolm, picks the right weight shard for the target, and starts the upstream binary with the right flags. None of these forks the runtime; we pin a commit and shell out.

  • vLLM

    SERVER GPU · live

    Continuous batching, paged attention, OpenAI-compatible HTTP. Best fit for high-throughput chat and tool-use; AWQ and GPTQ INT4 land cleanly. NVIDIA and AMD ROCm. kolm runtime up --target vllm support-triage.kolm

  • SGLang

    SERVER GPU · live

    RadixAttention prefix-cache and constrained decoding. Best fit for structured-output workloads where the system prompt is long and stable. Same weight format as vLLM. kolm runtime up --target sglang support-triage.kolm

  • TGI

    SERVER GPU · live

    Hugging Face Text Generation Inference. Best fit for teams already running the HF stack; the .kolm loads as a normal safetensors model with the receipt sidecar verified on start. kolm runtime up --target tgi support-triage.kolm

  • TRT-LLM

    NVIDIA AOT · live

    NVIDIA ahead-of-time engine plans for H100, H200, B100, B200, RTX 4090/5090. The compile step can emit a TRT-LLM plan as a sidecar so the first start is fast. kolm runtime up --target trt-llm support-triage.kolm

  • llama.cpp

    CPU · portable · live

    GGUF Q4_K_M, Q5_K_M, Q8_0. Runs on Linux, macOS, Windows; CPU AVX2/AVX-512, CUDA, ROCm, Vulkan, Metal. The widest portability target in the matrix. kolm runtime up --target llama.cpp support-triage.kolm

  • MLX

    APPLE SILICON · live

    Apple's MLX framework; runs on M1, M2, M3, M4. Best fit for on-device demos, internal Mac fleets, and CI on Apple runners. The MLX weight shard is a sidecar in the .kolm. kolm runtime up --target mlx support-triage.kolm

One verb, every runtime

The CLI hides the adapter wiring. kolm runtime up verifies the .kolm, picks the right weight shard for the requested target, and starts the upstream binary on a free port. Add --port to pin one.

# Verify the artifact first (offline; no network)
$ kolm verify support-triage.kolm
ok: true · artifact_cid bafkreieq…4f7c

# Start an OpenAI-compatible server on a free port
$ kolm runtime up --target vllm support-triage.kolm
[receipt]   verified ok · K-score 86.3 · verdict ship
[shard]     selected weights/model.safetensors.index (vllm-awq)
[spawn]     vllm 0.6.4 · pinned commit 7f3a…b210
[port]      listening on http://0.0.0.0:8000

# Test from any OpenAI-compatible SDK
$ curl -sS http://localhost:8000/v1/chat/completions \
    -H 'Content-Type: application/json' \
    -d '{"model":"support-triage","messages":[{"role":"user","content":"hi"}]}'

The compile worker lives at workers/runtime-build/. Each successful build emits a build-receipt.json next to the binary (target slug, toolchain, CMake args, source commit, SHA-256 of the artifact). Use kolm runtime build-from-source <target> to run the pinned recipe locally.

How to pick a runtime

The default is llama.cpp because it runs anywhere. The other five are for cases where you actually need the extra capability. Plain scope: this is a heuristic, not a benchmark; run your own K-score on the candidate runtime before committing to it.

  • High-throughput chat & tool use

    PICK · vLLM

    Continuous batching means many concurrent requests share GPU memory cleanly; first-token latency drops vs. naive batching. Pair with a fronting reverse proxy if you need TLS or rate limits.

  • Constrained decoding

    PICK · SGLang

    RadixAttention shares prefix KV across requests with the same system prompt; constrained decoding (regex, JSON schema, BNF) is first-class. Good for an agent with a long shared prompt scaffold.

  • Already on Hugging Face

    PICK · TGI

    Same serving model your team already runs; the .kolm loads as a safetensors directory with a receipt sidecar the adapter verifies at start.

  • Latency-critical on NVIDIA

    PICK · TRT-LLM

    Ahead-of-time engine plans win on per-token latency at the cost of longer build time; the compile step can emit the plan as a sidecar.

  • Run anywhere

    PICK · llama.cpp

    GGUF runs on Linux, macOS, Windows; CPU AVX2/AVX-512, CUDA, ROCm, Vulkan, Metal. Default for laptops, CI runners, and air-gapped deploymentes without a vendor SDK.

  • On-device on a Mac

    PICK · MLX

    Native to Apple Silicon; integrates with Swift and SwiftUI for on-device demos. Internal Mac fleets get sub-second cold start once the weights are cached.

Speed flags — opt-in, measured

Five flags layer on top of the runtime: speculative decoding, Shard KV-cache compression, prompt caching, continuous batching, and a Blackwell-gated NVFP4 export. Each is independently measurable with kolm bench --speed. Defaults stay safe; --optimize picks hardware-aware values for all four.

  • Speculative decoding

    1.5–2.4× tok/s · HF + vLLM

    A small draft model proposes k tokens; the target verifies in one pass. --speculative auto picks a tokenizer-matching draft from the same family; cross-tokenizer drafts are refused. Acceptance is reported in kolm bench --speed --json · spec_accept_rate.

  • Shard KV compression

    ~10× KV memory · HF only

    A transformers.Cache subclass: K-PCA + V-VQ256 per layer. Fresh cache per request for concurrent safety. vLLM/SGLang/TGI/TRT-LLM ship their own KV managers and ignore external caches; the serve loop logs a warning and falls back to standard.

  • Prompt caching

    TTFT win · vLLM

    vLLM enable_prefix_caching — system prompts and few-shot prefixes hit cache instead of recomputing. Big win on RAG with fixed scaffolds, no-op on RAG with rotating context.

  • Continuous batching

    Throughput knob · vLLM

    --max-batch N sets vLLM max_num_seqs. Default 8; raise to 16/32/64 for big-VRAM GPUs and many lightweight requests. Tail latency goes up with depth; the serve loop refuses to start if the projected peak exceeds VRAM and suggests a lower N.

  • NVFP4 on Blackwell

    sm_100+ · gated

    4-bit float with native tensor-core support on B100/B200/RTX 5090. kolm export --target nvfp4 probes nvidia-smi and refuses on Hopper/Ada (where the runtime emulates FP4 on FP16 cores and is slower than FP8). Pass --force for calibration runs; the runtime passport records blackwell_gate.forced:true.

Receipt verification at load time

Every adapter calls kolm verify before it spawns the upstream binary. If the signature does not match, the bytes hash does not match, or the K-score gate is below the threshold the artifact claims, the runtime refuses to start. There is no --skip-verify flag.

# What the runtime adapter does on every cold start
[receipt]   load receipt.json · receipt.sig · receipt.pubkey
[bytes]     sha256(artifact bytes) == artifact_bytes_sha256 ok
[canon]     sha256(JCS(receipt)) == signed hash ok
[sign]      Ed25519 verify ok
[k-score]   composite 86.3 / threshold 80 · verdict ship ok
[target]    requested target llama.cpp in receipt.targets[] ok
[spawn]     llama-server · pinned commit 2026-05-17-frontier-pin

# Build receipt for the runtime binary

Air-gapped serving

Every adapter can run with no network access. The .kolm is the entire dependency; the runtime binary is pre-built; the receipt verifies offline. A regulated team can move artifacts in on a sealed disk and stand up a serving cluster with no egress.

  • Static runtime binaries

    NO EGRESS · live

    For each target slug the runtime worker produces a static binary with the pinned commit. Drop it next to the .kolm; the adapter shells out to it without touching any package manager at run time.

  • Offline verifier

    NO EGRESS · live

    The verifier is a single binary. No network, no truthfulness service, no registry lookup. An auditor on a clean laptop validates the chain end-to-end and sees the same ok:true a production node sees.

  • TEE attestation pinning

    ATTESTATION · live

    If the compile step embedded a TEE attestation report, the adapter checks the runtime hardware matches the attested platform before serving. Mismatch refuses to start. See the compile attestation section.

  • Real failure modes

    PLAIN SCOPE · live

    If the upstream binary segfaults, the adapter forwards the exit code and the last 64 KB of stderr to the operator. We do not swallow runtime crashes; the failure is yours to read and act on.

CLI reference

Every runtime operation is reachable from the kolm runtime verb. Run kolm runtime --help for the full surface; the most common subcommands are below.

CommandWhat it does
kolm runtime targetsList every build target the worker knows about. JSON with --json.
kolm runtime info <slug>Show the CMake flags and SDK requirement for one target.
kolm runtime doctor --target <slug>Check the toolchain on this box can build the target. Loud fail with install hints.
kolm runtime build-from-source --target <slug>Clone the upstream runtime, check out the pinned commit, run CMake, install the binary.
kolm runtime up --target <adapter> <.kolm>Verify the artifact and spawn the upstream serving binary.
kolm runtime downStop the spawned runtime cleanly. Reports exit code.
kolm runtime statsLive token-per-second, queue depth, first-token latency. Reads from the runtime metrics socket.
kolm verify <.kolm>Offline verify of the artifact. The adapter runs this before every cold start.
