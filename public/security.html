02 / Data handling

Redact. Process in scope. Retain only the artifact.

One short lifecycle, and the controls that hold each stage.

01 · redact

You strip it first

Field-level redaction removes secrets and customer identifiers on your side, before anything reaches our processing.

02 · process

We work in scope

The redacted input is processed inside the assessed scope: TLS 1.3 in transit, AES-256 at rest, isolated per tenant.

03 · retain

We keep the report

The signed report and its log entry persist. Raw uploads do not outlive the run. Minimal retention by default, not by request.

Encryption

AES-256 at rest, TLS 1.3 in transit

Hosted control plane data is encrypted with AES-256 (AWS KMS, per-tenant data keys). Client traffic uses TLS 1.3 with HSTS.

Isolation

Per-tenant, per-namespace, per-row

Tenant and namespace isolation, plus per-row tenant fences inside every query handler. Defense in depth, not a single boundary.

Access

SSO-gated, MFA-enforced, reviewed quarterly

Engineering access to the control plane is SSO-gated and MFA-enforced. Production access is written to an immutable audit store.

Scope is contractual. Permission posture, redaction and audit-trail integrity are assessed. Injection is tested and reported, not warranted.

03 / Signing keys
03

One Ed25519 key signs every report. The private half stays put.

The signature your buyer checks is only as good as the key behind it. We keep that key small, single-purpose and rotated, and publish the fingerprint.

  • Reports are canonicalized to kolm canonical JSON (keys sorted, no whitespace; format spec), then signed with Ed25519
  • The private key stays in the signing service; only the public key fingerprint travels in the report
  • Editing any field after signing breaks the signature, in front of the reviewer
  • Rotation runs via kolm keys rotate on a defined cadence; prior reports stay verifiable under the fingerprint they carry

What an attacker would have to beat, tier by tier, and what we do not defend, is written down in one place.

04 / Transparency log

Every signature lands in a Merkle log you can audit.

A signature you cannot situate in a record is a signature taken on faith. Each signed report is appended to a public log a reviewer audits independently.

Entries are SHA-256 hashed into a Merkle tree structured after RFC 6962, so nothing can be edited or removed without changing the root. Every report ships with an inclusion proof.

This is a Merkle log, not a blockchain: an append-only structure with no tokens, no wallets, no distributed ledger. The log is public and anyone can recompute the root.

05 / Responsible disclosure
05

Found something? Tell us, and you are covered.

Report issues in our hosted services, signing pipeline, CLI or evidence format to dev@kolm.ai with a writeup, reproduction steps and assessed impact.

  • 24 hours to acknowledge, 72 hours to triage
  • 90-day coordinated disclosure window from a verified report, with proactive notice if a fix needs longer
  • No legal action against researchers acting in good faith under this policy

Dependency hygiene

Critical CVEs in 7 days

Critical CVEs patched within 7 days, high severity within 30. A CycloneDX SBOM ships with every CLI release, signed against the release tag.

Supply chain

Signed, reproducible releases

CLI releases are signed and built from a reproducible script in the public repository, so consumers can audit the tree and confirm the bytes.

Incident response

On-call, with a written follow-up

An on-call rotation covers the hosted gateway and signing pipeline. Post-incident reviews reach affected enterprise customers within 14 days of closure.
