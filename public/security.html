Security · privacy · compliance

Your data, your weights, your boundary.

Kolm is an open-source compiler. It runs inside your VPC, on-prem, or in an air-gapped enclave. Every artifact ships with a signed receipt that names the teacher, the base model, the eval pack, and the K-score axes. A security engineer, a compliance reviewer, or an auditor can verify the chain years later without trusting us.

Data-flow boundary

The default deployment never sends customer prompts, completions, or model weights to kolm.ai infrastructure. The compiler runs on your machines; only signed receipts and (optionally) public registry pointers leave the boundary.

What stays inside your network

Captures
Raw prompts and completions land in ~/.kolm/events or whichever path KOLM_DATA_DIR points to. Tenant-scoped by namespace and never replicated upstream.
Distilled datasets
Teacher-student pairs are generated by your distill workers against the teacher endpoint you configure. We do not see them.
Model weights
Quantized INT4/INT8 weights are written next to your captures. The .kolm artifact is a zip on your filesystem.
Eval pack runs
K-score evaluations execute against the artifact locally. The numeric scores and axes are computed in-process.
Logs and traces
Trace IDs, request bodies, and tool outputs are in your observability stack. Use the OTLP exporter if you want them in Datadog or Grafana.

What does cross the boundary (only if you opt in)

Receipt CIDs
If you publish to the public registry, a content-addressed hash of the receipt (no weights, no captures) becomes resolvable at kolm.ai/verify/<cid> for downstream verifiers.
Telemetry (off by default)
Opt-in usage counters help us prioritize. Set KOLM_TELEMETRY=0 to disable; the CLI never sends prompts or completions either way.
Anthropic / OpenAI calls
Only if you use the hosted teacher path. Bring-your-own-key and self-hosted vLLM, SGLang, and TGI teachers keep all traffic in your network.

Signing and verification

Every .kolm artifact ships with an Ed25519 signature over a content-addressed receipt. The receipt names the inputs that produced the artifact and the K-score that gated it. Verification is offline and reproducible.

algorithmLive

Ed25519 over the receipt CID

The receipt is canonicalized to JCS, hashed (SHA-256), and signed. The signature, the public key fingerprint, and the receipt body all sit in the artifact zip. Any developer or auditor with the artifact can verify without contacting us.

offlineLive

kolm verify works air-gapped

The verifier is a static binary. No network, no registry lookup, no honesty service in the middle. Hand a .kolm file to an auditor on a USB stick and they can verify the chain on a laptop.

key-rotationLive

Per-tenant signing keys

The CLI generates an Ed25519 keypair on first use and stores it at ~/.kolm/keys. Rotate with kolm keys rotate; previous receipts remain verifiable because the public key fingerprint is embedded.

tamper-evidenceLive

Artifact hash binds to receipt

The artifact hash is computed over the canonical zip and embedded in the receipt. Editing weights, swapping an eval pack, or rewriting the K-score after the fact invalidates the signature.

Deployment modes

Pick the boundary that matches your regulator. The same compiler, the same CLI, the same receipts; only the network shape changes.

VPC-resident

Inside your AWS, GCP, or Azure

Compiler workers run in a private subnet. Teacher calls go to Anthropic, OpenAI, or your self-hosted endpoint over your VPC routes. Receipts written to S3/GCS with object-lock.

  • Default for SaaS teams with vendor data
  • BAA-eligible when paired with HIPAA-aware teacher path
  • One-VPC reference Terraform in enterprise
On-prem

Your own hardware

CLI installed on workstations, distill runs on an RTX 5090 or H100 box, artifacts written to a CIFS or S3-compatible store you operate. No outbound calls if you bring your own teacher.

  • Default for regulated payors and providers
  • Works with self-hosted vLLM, SGLang, TGI
  • See healthcare for HIPAA path
Air-gapped

No network at all

CLI and runtimes are static binaries. Move artifacts and eval packs in on a sealed disk; the verifier works offline. Used by defense and intelligence pilots.

  • kolm verify is a single static binary
  • Eval packs are CIDs, not URLs
  • Receipts verifiable on a clean laptop

Retention and deletion

Captures, receipts, and eval runs are tenant-scoped and namespace-scoped. You set retention; we never apply a default lifetime to your data.

capturesLive

Per-namespace retention

Set retention_days per namespace via kolm namespace set-retention <ns> --days 90. Older captures are deleted by the local rotator. Receipts that reference deleted captures remain verifiable because they only embed hashes.

right-to-deleteLive

Subject deletion

Pass a deletion request to kolm capture delete --subject <id>. The CLI removes raw rows and writes a tombstone receipt so auditors can prove the deletion happened without recovering the data.

artifactsLive

Artifact lifecycle

.kolm artifacts live in your object store. The compiler never deletes them. Use your bucket lifecycle to age artifacts; the receipt CID stays verifiable as long as the receipt is preserved (the receipt is a few kilobytes).

registryLive

Registry takedown

If you publish a receipt CID to the public registry, you can request takedown at trust contact. Takedown removes the CID from the public index; downstream verifiers with the receipt zip continue to verify.

Threat model

What the compiler does and does not defend against, with the mitigation that handles each class.

Tampered weights post-compile

An operator swaps the INT4 weights in a deployed artifact and tries to ship the same receipt.

mitigation · artifact hash bound to signed receipt; kolm verify fails

Spoofed K-score

A team edits the K-score JSON to claim a higher gate verdict than the eval pack produced.

mitigation · K-score and axes signed in receipt; verifier recomputes if eval pack present

Stolen API key

A leaked CLI API key is used to publish to the hosted registry or call hosted teachers.

mitigation · rotate with kolm keys rotate; per-tenant rate limits; audit log

Captures exfiltrated from disk

An attacker with filesystem access reads ~/.kolm/events directly.

mitigation · out of scope for the compiler; use OS-level encryption (LUKS, BitLocker, FileVault)

Supply-chain on the CLI

A compromised release pushes a backdoored kolm binary.

mitigation · releases signed; SBOM published; reproducible build script in repo

Prompt injection on captures

A captured prompt contains instructions that try to manipulate downstream distill or scoring.

mitigation · eval pack runs are sandboxed; teacher calls are scoped per row

Operational security

How we run kolm.ai the service; how you should run the compiler in your environment.

internal

Least-privilege access

Engineering access to the hosted control plane is SSO-gated, MFA-enforced, and reviewed quarterly. Production access is logged to an immutable audit store.

internal

Vulnerability response

Critical CVEs in dependencies patched within 7 days; high within 30. Disclosure window is 90 days from a verified report. Email security@kolm.ai with a writeup.

internal

Incident response

On-call rotation for the hosted gateway and registry. Post-incident reviews shared with affected enterprise customers within 14 days.

your-side

Network egress

For VPC deployments, allowlist only the teacher endpoint(s) you use and (optionally) the public registry. Everything else stays inside.

your-side

Key custody

Signing keys live in ~/.kolm/keys by default. For multi-engineer teams, move them to your KMS (AWS, GCP, Azure, HashiCorp Vault) and configure kolm sign --kms.

your-side

Audit log shipping

The CLI emits structured JSON to ~/.kolm/audit.log. Tail it into Splunk, Datadog, or your SIEM. Receipts are the durable record; the audit log is the operational one.
