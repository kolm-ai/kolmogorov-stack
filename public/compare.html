04 / How Kolm becomes the best product

Eight rules that beat category sprawl.

This is the product standard. Any feature that does not strengthen one of these should be treated as noise.

1. API control center first

Enterprise buyers need ingress, egress, capture, redaction, provider vault, budgets, retention, evals, compile state, targets, and export policy in one console.

2. Import every trace source

Trace exports, eval dashboards, OpenTelemetry spans, gateway logs, JSONL, webhooks, GraphQL/RPC, queues, warehouses, SIEM drains, MCP, A2A, browser events, files, and custom adapters.

3. Compile to an artifact

The output cannot be just a dashboard score. It must be a signed `.kolm` package with manifests, examples, evaluators, hashes, receipts, and target instructions.

4. Verify without us

Receipts, public keys, hashes, and exported packets should remain useful when a buyer, auditor, or platform team checks them outside the Kolm UI.

5. Runtime-neutral targets

Kolm should not pretend every deployment target is the same. It should package clean target recipes for hosted, local, edge, mobile, and restricted fleets.

6. Honest readiness scope

The site must say what is shipped, local-certified, partner-gated, benchmark-gated, and package-release-gated. Overclaiming weakens trust.

7. Product UI over marketing

Every main page should show the actual machine: routes, data channels, policies, jobs, receipts, targets, and next actions. No generic AI fluff.

8. One loop, many integrations

Gateways, evals, security, GRC, training, RAG, and runtime tools become sources, controls, or deployment targets for the compiler loop.
