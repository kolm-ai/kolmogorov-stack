kolm.ai  /  quickstart  /  natural language

From idea to a verifiable, signed .kolm recipe in 60 seconds, using only natural language.

Four CLI verbs compose into one flow: kolm nl drafts a recipe from a free-text request, kolm seeds new scaffolds candidate examples, kolm distill runs the actual training, and kolm verify gates the ship on K-score and the receipt chain. Every step works offline today. No black box anywhere in the path.

Wave 200 dedicated surface · Shift 5 of the to plan · reviewed 2026-05-17

Start at step 1 → CLI quickstart (non-NL) → Methods research →

Prereqs.

Install with npx kolm on Node 20 or newer. This walkthrough defaults to KOLM_AIRGAP=1, so no API key is required and nothing leaves your laptop.

The four-step natural-language flow.

Each CLI verb feeds the next one, and every artifact carries provenance: request, examples, teacher/student pair, and verifier verdict. A third party can replay the chain from the receipts inside the .kolm.

step 01 Scaffold from a free-text request kolm nl shipped wave 197

Type your recipe idea in plain English. kolm nl infers the recipe class (rule, synthesized_rule, compiled_rule, distilled_model), proposes a K-score gate, drafts ten starter seed examples, and prints a JSON scaffold plus a next-steps list. The output is deterministic in air-gap mode so every developer on your team gets the same scaffold for the same prompt.

$ KOLM_AIRGAP=1 kolm nl "I need a recipe that classifies denial appeal CARC codes 50/197/204/16 and writes a member-friendly appeal letter." --json

{
 "ok": true,
 "suggested_slug": "denial-appeal-letter",
 "recipe_class": "distilled_model",
 "class_inference_basis": "keyword:appeal+letter+member",
 "suggested_k_score_gate": 0.85,
 "suggested_task_description": "Classify a CARC denial reason code and draft a member-friendly appeal letter that cites the code.",
 "network_status": "air_gap",
 "suggested_seed_examples": [ /* 10 prompt+completion pairs, drafted offline */ ],
 "next_steps": [
 "kolm seeds new denial-appeal-letter --count=12 --out=seeds.jsonl",
 "edit seeds.jsonl: replace candidate rows with your real examples",
 "kolm distill --local-worker --spec ./denial-appeal.spec.json --seeds ./seeds.jsonl --out ./denial-appeal.kolm --student-base=Qwen/Qwen2.5-7B-Instruct --teacher=anthropic:claude-opus-4-7",
 "kolm verify ./denial-appeal.kolm --binder report.html"
 ]
}

note: the deterministic air-gap path is the default. Pass --network
 to opt into the hosted-LLM drafter. See kolm nl --help.

The recipe_class drives downstream choices (compile vs distill vs synthesize) and the suggested_k_score_gate becomes the floor a successor must clear before kolm verify passes. Both are starting points; override either with a flag if you know better than the keyword inference.

step 02 Generate candidate seed examples kolm seeds new shipped · air-gap default, opt into --network

Drop the NL slug into kolm seeds new. The verb scaffolds a starter seeds.jsonl with candidate prompt+completion rows you replace with your real examples. These are candidates, not labels: kolm seeds new never claims a row is ground truth. The CLI output makes that explicit every time.

Use natural language to scaffold the first candidate rows, then replace or approve them with real examples before compile. If you need volume, kolm seeds generate expands reviewed rows deterministically so the holdout gate still has a clean split.

step 03 Distill into a recipe kolm distill shipped · local-worker mode

Pass your seeds plus a base student model and a teacher to kolm distill --local-worker. The worker runs the actual training inside an isolated package so heavy ML deps never bleed into the root install. Qwen2.5-7B-Instruct is the default student for the appeal-letter use case; substitute any model in the catalog (run kolm distill --local-worker --list-catalog). The K-score gate from step 1 enforces ship/no-ship.

$ kolm distill --local-worker \
 --spec=denial-appeal.spec.json \
 --seeds=seeds.jsonl \
 --out=denial-appeal.kolm \
 --student-base=Qwen/Qwen2.5-7B-Instruct \
 --teacher=anthropic:claude-opus-4-7 \
 --redact=phi

The training run is real: it loads the student base, applies LoRA against the seeds, scores against the dev split, gates on the dev K-score, and writes the artifact only if the gate passes. The exit code is non-zero if it fails so a CI step surfaces the rejection.

step 04 Verify the artifact kolm verify shipped · 24 offline checks

Run kolm verify against the freshly-built .kolm. The 24-check verifier walks the receipt chain, recomputes every block hash, validates the signature against the manifest, replays the K-score gate, and confirms the redactor class matches the recipe class. Every check runs offline; nothing is trusted.

What just happened.

Four small verbs, one composable flow. The truthfulness hooks at each step keep the artifact auditable end to end.

  • Step 1 (nl): the scaffold is a starting point, not a finished spec. kolm nl proposes a class, a gate, and ten example pairs drafted offline. You override any of them with a flag if you know better. The next-steps array tells you exactly which verb runs next.
  • Step 2 (seeds new): rows in seeds.jsonl are candidates, not labels. The CLI marks starter rows explicitly so nobody treats templated rows as training data. You replace them with your real examples; the K-score becomes meaningful only against real captured or reviewed rows.
  • Step 3 (distill): the worker runs the actual training, not a stub. Heavy ML deps live in workers/distill so they never bleed into the root install. The training loop scores against the dev split and rejects ships that do not clear the K-score gate.
  • Step 4 (verify): the verifier gates the ship with 24 offline checks. No step in the chain ships a black box; every claim resolves to a manifest field and a verifier check that re-derives it. The binder HTML is the artifact your auditor reads.

The composed flow is intentionally local-first. Steps 1, 2, and 4 require nothing but the CLI. Step 3 needs a base student and a teacher you control; pass --teacher=local:<model> if you want to keep the teacher on-disk too. The artifact at the end is portable, signed, and verifiable by anyone with the .kolm file and the open-source verifier.

The KOLM_AIRGAP=1 story.

Every step in the flow above runs offline today. kolm nl takes the deterministic keyword-inference branch (Wave 197 air-gap-first design). kolm seeds new emits templated candidates locally. kolm distill --local-worker trains against your seeds with whatever student and teacher you have on-disk. kolm verify recomputes every check from the bytes of the artifact. No step ever needs to call out to the kolm cloud or any third party; the only required network call is whatever your --teacher flag points at, and you control that too.

The networked LLM path (where kolm nl calls out to a hosted model for richer drafts) is opt-in: pass --network or unset KOLM_AIRGAP=1 to enable it, and the scaffold output carries network_status: "wired" instead of "air_gap" so a downstream tool can detect which branch ran. The CLI POSTs the request to /v1/nl/scaffold and the response is tagged with the x-kolm-nl-source header so a caller can confirm which branch served. The air-gap branch continues to exist, continues to be deterministic, and continues to be the default for tenants who want byte-for-byte reproducibility across developer laptops. See the design pattern in /research/methods-2026-q2.

What this flow does NOT do.

Three boundaries are worth naming up front so nobody walks into the next conversation with the wrong expectation.

  • Does NOT replace human eval. The K-score gate scores against your dev split, not against ground truth in the world. You still need to look at the candidates in step 2, look at a sample of the model's outputs in step 3, and decide whether the artifact is fit for purpose. kolm nl does not know whether your real users will accept the appeal-letter tone; only you do.
  • Does NOT promote unverified examples to training data. The K-score gate enforces a floor before the artifact writes to disk. If your seeds are noisy and the dev split looks bad, the artifact is not produced. There is no quiet fallback that ships a degraded model under a green light; the only way an artifact reaches your filesystem is by clearing the gate you set.
  • Does NOT do online learning. Artifacts are frozen at ship time. If production traffic drifts six months later, the answer is supersession via /drift: detect drift on a cadence, rebuild a successor with --supersession-of, retire the predecessor with reason and evidence. The supersession chain stays auditable. No artifact ever rewrites itself in place.
