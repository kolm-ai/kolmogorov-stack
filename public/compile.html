Compile · step 3 of 4 · spec to signed .kolm

Spec in. Signed .kolm artifact out.

A compiler engineer feeds a spec (the base model, the distilled weights, the K-score gate, the eval pack, the teacher trace) into kolm compile. The output is a single .kolm zip with an Ed25519 signature over a JCS-canonical SHA-256 hash of the receipt. The artifact_cid binds the weights, the receipt, and the K-score into one object an auditor can verify offline three years later, on a clean laptop, with no network and no honesty service in the middle.

What you give kolm. What kolm gives back.

A clean exchange. Everything on the right is reproducible from everything on the left. Nothing leaves your machine that you didn’t ship in.

You provide

Task · examples · evals · key.

  • taskOne sentence. The behavior the artifact has to match.
  • examples10 to 50,000 rows. JSONL, CSV, a folder of files, or a corpus namespace.
  • evalsHeld-out set. Synthesized if absent. This artifact’s K-score is measured against this set, not against the base model.
  • api keyBYO. Anthropic, OpenAI, Google, Mistral, Together, Fireworks, or your own gateway.
  • basePick one from the open list, or let kolm choose by size and target.
kolm returns

One signed .kolm. Nothing else.

  • artifactSingle zip. Model, LoRA, recipes, recall index, tests, manifest, signature.
  • k-scoreOne number for this .kolm (not for the base model). Below the gate, the artifact does not ship.
  • receiptEd25519 chain. Bound in the artifact receipt. Tamper-detectable.
  • runtimeSame file runs through the CLI, the MCP server, and native iOS / Android adapters. One signature, every target.
  • costFrontier bill is yours. Local runtime is unlimited on every paid tier.

One verb produces a verifiable artifact

The compile verb takes a path to a distill run (or a spec file) and writes a .kolm with a signed receipt. The receipt names every input that produced the artifact and binds them into a single content-addressed id.

# Compile a previous distill run into a verifiable artifact
$ kolm compile ~/.kolm/runs/2026-05-22T18-14-02Z \
    --eval-pack ks-finance-2026-q2 \
    --target llama.cpp \
    --quantize int4

[load]      run dir ok · distill done at 2026-05-22T18:14:02Z
[eval]      eval pack ks-finance-2026-q2 cid bafkreif5…8a21
[score]     re-score against eval pack · K-score 86.3 · verdict ship
[pack]      build .kolm zip · weights + receipt + sidecars
[hash]      artifact bytes sha256:3f29…a01e
[canon]     JCS-canonical receipt sha256:2dba…77c4
[sign]      Ed25519 signature ok
[cid]       artifact_cid bafkreieq…4f7c

artifact    ~/.kolm/artifacts/support-triage.kolm
size        5.4 GB
verify      kolm verify support-triage.kolm

A developer hands the .kolm zip to an auditor. The auditor runs kolm verify support-triage.kolm on an air-gapped laptop. The verifier reads the receipt, recomputes the artifact hash, verifies the Ed25519 signature against the embedded public key fingerprint, and reports ok:true. No network call; no honesty service; no trust in us.

What the signed receipt names

The receipt is the durable record. It names every input that produced the artifact and binds them into a single content-addressed id. Editing any field invalidates the signature; deleting the captures the artifact was trained from does not invalidate the receipt because the receipt only embeds hashes.

  "base_model": { "id": "Qwen/Qwen2.5-7B-Instruct", "sha256": "a3c1...d10b" },
  "distill_run": {
    "seeds_sha256": "f48a...e2b1",
    "row_count": 2400,
    "teacher_attempts": 1
  },
  "quantize": { "scheme": "nf4-double", "library": "bitsandbytes-0.49.2" },
  "eval_pack": { "id": "ks-finance-2026-q2", "cid": "bafkreif5...8a21" },

Inside the compile pipeline

Four sub-steps, every one of which writes intermediate state into the run directory. A failure in any sub-step aborts the compile; nothing partial is ever signed.

  1. 01 · PACK

    Build the zip

    Weights, tokenizer, eval pack snapshot, distill receipt chain, and target-specific sidecars (GGUF metadata, MLX manifest).

  2. 02 · HASH

    Content address

    SHA-256 over the canonical zip bytes; embedded in the receipt as artifact_bytes_sha256. Any post-compile edit breaks this.

  3. 03 · CANON

    JCS-canonical receipt

    The receipt JSON is JCS-canonicalized (RFC 8785), then hashed. The hash is the input to the Ed25519 signature.

  4. 04 · SIGN

    Ed25519 over the hash

    Signed with the tenant key in ~/.kolm/keys. Signature, public-key fingerprint, and receipt body all sit in the artifact zip.

What is inside a .kolm zip

A .kolm is a plain zip with a stable layout. An engineer can unzip -l it; an auditor can mount it read-only and inspect every byte. There is no proprietary container, no DRM, and no opaque blob.

support-triage.kolm
├── weights/
│   ├── model.q4_k_m.gguf          # the quantized weights for the target runtime
│   └── model.safetensors.index    # shard map (multi-file weights)
├── tokenizer/
│   ├── tokenizer.json
│   └── tokenizer_config.json
├── eval/
│   ├── pack.json                  # frozen snapshot of the eval pack used at compile time
│   └── score.json                 # per-axis K-score numbers, also embedded in the receipt
├── trace/
│   ├── distill.jsonl              # per-row teacher attribution: endpoint, model id, hash
│   └── seeds.sha256               # hash of seeds.jsonl, also in receipt
├── receipt.json                  # the JCS-canonicalized signed receipt
├── receipt.sig                   # the Ed25519 signature bytes
└── receipt.pubkey                # the signing key fingerprint

Target-aware packing

A .kolm can land on more than one runtime. Pass --target at compile time to pre-pack the weights for the format the runtime expects; the receipt records every target so a verifier knows which weights to inspect.

--targetWeight formatRuntime
llama.cppGGUF (Q4_K_M, Q5_K_M, Q8_0)CPU and GPU; widest portability.
vllmsafetensors + AWQ / GPTQGPU serving (NVIDIA, AMD ROCm).
sglangsafetensors + AWQ / GPTQGPU serving with RadixAttention prefix cache.
tgisafetensorsHugging Face Text Generation Inference.
trt-llmTRT-LLM engine planNVIDIA GPUs; ahead-of-time compiled.
mlxMLX weight shardsApple Silicon (M-series).
onnxONNX with QDQ INT8Cross-platform inference engines.
wasmGGUF + WebAssembly runtime hintBrowser and edge.

A single .kolm can carry weights for multiple targets if you pass --target llama.cpp --target mlx; the receipt records both and the runtime adapter picks the right shard at load time. Cross-runtime parity is gated by the K-score on each target.

PRECISIONS

Every bit-width. The compiler handles the calibration.

From 1-bit ternary BitNet to FP8 and NVFP4. Activation-aware, imatrix-calibrated, K-score-gated. Cross-reference /forge for runtime numbers and /benchmarkscompare for end-to-end traces.

1.0bit

BitNet b1.58 (binary)

Ternary {-1, 0, +1} packed at 1.0-bit average. AngelSlim recipe; matmul becomes additions on CPU.

research
1.25bit

Sub-2-bit IQ1_M

llama.cpp IQ1 family with imatrix calibration. Heavy K-score discount; preview only.

preview
1.58bit

BitNet ternary

log₂(3) = 1.58 bits per weight. Matmul-free inference path; CPU-friendly inference without specialized accelerators.

research
2.0bit

IQ2_XXS / IQ2_S

2-bit imatrix quants. The right landing zone for 70B+ teachers on a single 24 GB consumer card.

edge
2.25bit

SpQR + sparse residual

Sparse-Quantized Representation. Outlier channels kept at 16-bit, the rest at 2-bit; the sparse residual lifts low-bit K-score at small zip-size cost.

edge
3.0bit

Q3_K_M / IQ3_M

k-quants 3-bit. Common landing zone for 32B teachers on 16 GB consumer cards.

ships
4.0bit

NF4 (default)

NormalFloat4 + double-quant via bitsandbytes 0.49.2. DeepSeek-R1 32B lands at 17.9 GB, 11.5 tok/s on a single RTX 5090.

ships
8.0bit

INT8 / Q8_0

SmoothQuant activation smoothing or k-quants Q8_0. Near-lossless on most tasks; the safe default when you have the VRAM.

ships

E4M3 / E5M2

Native FP8 on H100 and RTX 5090. vLLM and TRT-LLM serve direct from FP8 weights without dequantization.

ships

Blackwell-class FP4 with microscale blocks. Higher accuracy than INT4 at the same bit budget on B100-class hardware.

preview

Layer-wise mixed precision. Default for kolm compile --quantize int4; recovers ~1.5 K-score points over flat Q4.

ships

TurboQuant KV-cache

Per-channel KV-cache quantization to 4-bit. Cuts long-context VRAM ~3× with negligible K-score drift.

ships

BACKENDS

Nine quantization backends. One compile verb.

The compiler picks the right backend for the recipe and the runtime. Override with --backend. Cross-reference /distill for the upstream step and /k-score for the gate.

  • GPTQ

    3-BIT / 4-BIT · weight-only

    Hessian-based per-layer optimization. Strong on 3-bit / 4-bit weight-only quantization; the workhorse for vLLM and SGLang serving paths.

  • AWQ

    4-BIT · activation-aware

    Activation-aware Weight Quantization. Preserves the 1% of weight channels that handle the bulk of activations; better K-score than flat GPTQ at the same bit budget.

  • bitsandbytes NF4 / FP4

    NF4 · double-quant

    NormalFloat4 + double-quant. The default for the --quantize int4 flag. DeepSeek-R1 32B compiles to 17.9 GB in 125 seconds via bitsandbytes 0.49.2.

  • imatrix

    CALIBRATION · llama.cpp

    llama.cpp importance-matrix calibration. Reads a few hundred prompts to weight the quantizer per channel; lifts every k-quant tier (Q2_K through Q6_K).

  • Unsloth Dynamic 2.0

    DEFAULT · mixed precision

    Layer-wise mixed precision: keeps attention KQV at higher bits, drops MLP. Default for kolm compile; recovers ~1.5 K-score points over flat Q4.

  • TurboQuant

    KV-CACHE · 4-bit per chan

    KV-cache quantization to 4-bit per channel. Cuts long-context VRAM ~3x; the receipt records the KV scheme alongside the weight scheme.

  • AngelSlim

    TERNARY · BitNet

    BitNet ternary {-1, 0, +1} recipe. Matmul becomes additions; CPU-friendly. Honest scope: under 2-bit avg is research-tier, K-score gated.

  • SmoothQuant

    INT8 W/A · full-INT

    Activation smoothing that migrates scaling from activations into weights. Pairs with INT8 weight + INT8 activation for full-INT inference paths.

  • SpQR

    2-BIT + SPARSE · residual

    Sparse-Quantized Residual. Keeps outlier channels at 16-bit, the rest at 2- or 3-bit. The sparse residual lifts low-bit K-score at small zip-size cost.

MANIFEST

What lands in the artifact

The compile receipt names the backend, the bit-width, the K-score, and the signature. One zip, one signed manifest. The fields below are excerpted from a real receipt.

$ kolm receipt show ~/.kolm/artifacts/support-triage.kolm
{
  "format": "rs-1",
  "weights": { "format": "gguf", "quant": "Q4_K_M-dyn2" },
  "kscore": { "composite": 0.94, "...": "..." },
  "receipt": { "ed25519": "...", "jcs_sha256": "..." }
}

Every field above is reproducible from the inputs. /forge documents the backend label space; /k-score documents the composite formula; /forge drives the same pipeline from a browser console.

The artifact_cid binding

The artifact_cid is the field that ties everything together. It is a content-addressed id computed from the receipt, which itself names hashes of the weights, the eval pack, the teacher trace, and the K-score. Anything an attacker can edit shows up as a different cid.

  • Swap the weights, lose the signature

    TAMPER-EVIDENCE · live

    The receipt embeds artifact_bytes_sha256. Editing the GGUF or the safetensors file changes the hash. The cid is computed over the receipt, which embeds the bytes hash, so a swap is detectable in one verify call.

  • Rewrite the K-score, lose the signature

    TAMPER-EVIDENCE · live

    The K-score axes and verdict sit inside the receipt JSON. Rewriting them changes the JCS-canonical hash, which breaks the Ed25519 verify; no honesty service is needed in the loop.

  • Teacher trace is bound

    REPLAY-PROTECTION · live

    Each row labeled by the teacher writes an attribution line in trace/distill.jsonl. The file's hash flows into distill_run.seeds_sha256 in the receipt. An auditor can re-derive the same hash from the trace file.

  • Previous receipts stay verifiable

    KEY-ROTATION · live

    Rotating the tenant key with kolm keys rotate generates a new keypair; old receipts continue to verify because the public-key fingerprint is embedded in each receipt at compile time.

kolm verify, offline

The verifier is a static binary. No network, no registry lookup, no honesty service in the middle. Hand a .kolm file to a compliance reviewer on a USB stick and they can validate the chain on a clean laptop.

# Verify a .kolm artifact offline
$ kolm verify support-triage.kolm

artifact      support-triage.kolm
size          5.4 GB
artifact_cid  bafkreieq…4f7c

[bytes]       artifact_bytes_sha256 ok · 3f29…a01e
[canon]       JCS-canonical receipt hash ok · 2dba…77c4
[sign]        Ed25519 verify ok · key ed25519:2dba…77c4
[trace]       distill seeds hash ok · f48a…e2b1
[eval]        eval pack snapshot hash ok · bafkreif5…8a21
[k-score]     composite 86.3 / threshold 80 · verdict ship

ok: true
receipt:    ./out/receipt.json (extracted)
public-key: ed25519:2dba…77c4

A compliance reviewer can extract receipt.json alone and verify the signature without unpacking the weights; a regulated team can ship just the receipt to a downstream auditor and skip the multi-gigabyte weights download.

Optional attestation embed

For confidential-compute workloads, the compiler accepts a TEE attestation report and binds it into the receipt. The attestation hash becomes part of the artifact_cid so the receipt names both the weights and the hardware that produced them.

  • Intel PCCS

    SGX · live

    --attestation-kind pccs reads an Intel PCS report and embeds it into confidential_compute. The hash binds into artifact_cid.

  • AMD SNP report

    SEV-SNP · live

    --attestation-kind snp-report embeds an AMD SEV-SNP attestation. Used by cloud confidential VMs.

  • AWS Nitro attestation

    NITRO · live

    --attestation-kind nitro-attestation for AWS Nitro Enclaves. NSM token verification on the verifier side.

Honest scope: the compiler can bind an attestation report into the receipt, but it does not crypto-verify the report by itself. A tenant registers a real verifier with registerAttestationVerifier(); until then, the receipt records state:shape_ok, verified:false.

Where compile state lives

A compile run is reproducible. Every input the compiler read is recorded in the run directory; you can re-run with the same flags and (provided the base model and the eval pack are unchanged) get the same artifact_cid.

Default paths (override with KOLM_DATA_DIR)

~/.kolm/runs/<run-id>
One subdirectory per compile run. Contains run-meta.json, progress.jsonl, the staging weights, and the receipt draft.
~/.kolm/artifacts/<name>.kolm
The final signed artifact zip. Move it; copy it; archive it. The signature travels with the file.
~/.kolm/keys/ed25519
The tenant signing key. Back this up. Move to a KMS for multi-engineer teams via kolm sign --kms.
~/.kolm/eval-packs/<pack-id>
Local cache of eval pack snapshots. The compiler records the cid in the receipt; the pack content is reproducible from the cid.
~/.kolm/audit.log
Operational log of every compile and verify. Ship to your SIEM.

Reproducibility envelope

A compile run is reproducible inside an envelope. The same spec, the same base-model hash, and the same eval-pack cid produce the same artifact_cid. The compiler records which inputs are inside the envelope and which (CUDA driver version, BLAS kernel) sit outside; an auditor can see exactly what is determinism-locked and what is not.

  • Inside the envelope

    DETERMINISTIC · locked

    The spec, the seeds.jsonl, the base-model SHA-256, the quantize scheme, the eval pack snapshot, the K-score threshold. Two runs on the same machine with the same inputs produce the same artifact_cid.

  • Outside the envelope

    HONEST-SCOPE · recorded

    The exact CUDA driver, the BLAS kernel selection, the OS scheduler. These can shift floating-point rounding in the final per-axis K-score by under a tenth of a point; the verdict stays the same but the bytes hash can differ. The receipt records the toolchain so a third party can match it.
