Two commands. Your hardware.

Two-command install

A Helm install for the control plane; a Terraform apply for the data-plane VPC and the KMS wiring. An engineer can stand both up in an afternoon; an operator can re-stand them up in a release window.

# 1. control plane, on your existing Kubernetes (EKS, GKE, AKS, or self-managed)
helm repo add kolm https://helm.kolm.ai
helm install kolm kolm/kolm \
  --namespace kolm-system --create-namespace \
  --set tenant=$KOLM_TENANT \
  --set kms.provider=aws \
  --set kms.key_id=arn:aws:kms:us-east-1:<account>:key/<uuid> \
  --set teacher.endpoint=https://your-vllm.internal:8000/v1

# 2. data plane and KMS, via the Terraform module
terraform init
terraform apply -var="region=us-east-1" \
  -var="vpc_id=$KOLM_VPC" \
  -var="kms_alias=alias/kolm-signing"

# 3. verify the install end-to-end
kolm doctor                  # exit 0 = healthy
kolm verify ./sample.kolm    # offline signature check

Helm values and Terraform variables are documented inline in the chart and the module. The control-plane chart is the same binary that runs on kolm.ai; the difference is whose Kubernetes it runs in.

Operational footprint

What the on-call engineer sees once the install is in place. Latency, observability, drift response, and the things that wake people up at 03:00.

  • Sub-100 ms control plane

    latency · in-VPC p50

    Signup, registry lookup, and verify endpoint serve within 100 ms p50 when co-located with the data plane. Compile and distill latency depends on the teacher path; in-network teachers add no transit hop.

  • OpenTelemetry out of the box

    observability · OTLP

    Traces, metrics, and structured logs over OTLP. Ship into Datadog, Grafana Cloud, New Relic, Honeycomb, or your in-house OTel collector. The CLI emits ~/.kolm/audit.log as structured JSON for SIEM ingestion.

  • Drift detection on every release

    drift · K-score snapshot

    Each compile records a K-score snapshot. Drift detector compares the live snapshot to the last shipped artifact; verdict embeds in the receipt. Roll back by deploying the prior signed artifact; tombstone the failing one.

  • Object store on your account

    storage · S3 / GCS / Azure / S3-compat

    S3, GCS, Azure Blob, or any S3-compatible store. Receipts and artifacts live in your bucket with your lifecycle policy; kolm never deletes them. Hosted control plane indexes by CID, not by the underlying bytes.

  • Bucket lifecycle is the backup story

    backup · native object-lock

    Use S3 Object Lock or GCS Bucket Lock for receipts that must survive operator error. Receipts are kilobytes; artifacts are gigabytes. Most customers retain receipts indefinitely and age artifacts on the bucket's own schedule.

  • Runbook ships with the chart

    on-call · runbook

    Standard SRE runbook for the control plane: pod-level health, restart sequence, drift alert triage, signing-key rotation. Linked from the Helm chart README; reviewed against the SOC 2 system description.

Upgrade path

A new control-plane image is a helm upgrade. Artifacts do not churn; receipts already in your bucket remain verifiable against any future CLI release. Honest about what an upgrade touches and what it leaves alone.

  • Rolling upgrade in place

    helm upgrade · rolling deploy

    Pin the chart version (--version <semver>). The chart applies a rolling deployment; the existing signing keys, the registry index, and the captures in your bucket remain untouched. No data migration step.

  • Rollback to the prior revision

    helm rollback · no data touched

    If a release misbehaves, helm rollback kolm <rev> brings the previous control-plane image back online. The data plane and the bucket are not touched; in-flight compile jobs resume against the rolled-back control plane.

  • Receipts are forward-compatible

    receipt v1 · transition-safe

    The receipt schema is versioned (kolm-receipt-1). New compiler releases produce v1 receipts; older receipts in your bucket remain verifiable against any later CLI. If a future schema bump arrives, both versions verify side-by-side during the transition window.

  • CLI updates independent of the control plane

    CLI · operator cadence

    Operators upgrade the CLI on their cadence with kolm update. The CLI refuses to run from a repo clone (no silent global-install hazard) and announces a compatible control-plane version range in kolm doctor.

KMS integration

Signing keys live in your KMS. The CLI requests a signature; the private key never appears in process memory on the kolm side. Four KMS providers wired by default; bring-your-own via the same interface.

ProviderHelm valueWhat kolm uses it for
AWS KMS kms.provider=aws Signing receipts for compile events; key custody policies and grants follow the customer's IAM. CloudTrail records every sign call.
GCP KMS kms.provider=gcp Signing receipts via Cloud KMS asymmetric keys. Cloud Audit Logs record every sign call.
Azure Key Vault kms.provider=azure Signing receipts via Key Vault HSM keys. Azure Monitor captures the audit trail.
HashiCorp Vault kms.provider=vault Signing receipts via Vault Transit. For on-prem and air-gapped customers; no cloud dependency.
Local keypair (default) kms.provider=local Per-tenant Ed25519 keypair at ~/.kolm/keys. For small teams and air-gapped pilots; rotate with kolm keys rotate.

In every KMS-backed mode, the signing private key never leaves your network. The kolm control plane builds the canonical receipt body, submits it to your KMS for signature, and writes the signature back into the artifact zip. An auditor can confirm this via the KMS audit log on your side.

Air-gapped and sovereign deployments

If you cannot install from the Helm repo or the public Terraform registry, the chart and the module ship as offline tarballs against the Enterprise plan. Verify the bundle signature against the public key on the release notes; install behind your firewall without an outbound call.
