Verify

Three ways to verify a .kolm artifact.

Every .kolm carries a signature chain, a frozen-eval hash, and a K-score the receipt was issued against. Pick the verifier that matches what you have — a file, a receipt ID, or a regulated machine that can’t reach the public registry.

Paste a receipt CID

Receipt IDs start with rcpt_. We fetch the receipt, recompute the hash chain, and show every input that fed it.

Path 1

Browser drop-in.

Drag a .kolm file (or paste a content hash) into the in-browser verifier. Walks every signature, hash chain entry, and lifecycle transition. No account needed; fully offline-capable.

~30 s · no install Path 2

Standalone CLI.

The free kolm verify binary. Apache-2.0, no login, no telemetry, no network call required. Conforms to RS-1 and ships for Linux / macOS / Windows.

~60 s · air-gap friendly Path 3

Hosted registry.

Production-signed receipts from kolm.ai with full chain-of-custody. Useful for compliance reviewers who want a third-party attestation that the receipt was emitted by our key.

live · cryptographically signed

What gets checked. Every verifier runs the same four-step check against the RS-1 v2.1 reference spec. If any check fails, the verifier prints which one and exits non-zero.

signature chainDetached Ed25519 sig over the manifest, walked from leaf to root.
manifest schemaRequired fields, lifecycle stage, K-score gate, and provenance pins.
frozen-eval hashEval set hashes must match the value the artifact was trained against.
K-score recomputeReplay the held-out eval, compare to the issued K-score within tolerance.

No verifier sends anything to kolm.ai by default. Path 3 only contacts our registry to read the public receipt — the verification itself runs in your environment. See the security model and RS-1 spec for the full guarantee surface.
