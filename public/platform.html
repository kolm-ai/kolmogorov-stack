Platform Layers

Each plane has a job, a state boundary, and an exportable proof trail.

This is how Kolm stays more useful than an agent demo, an eval dashboard, or a model gateway: the platform owns the lifecycle between behavior and runtime.

Compile

Constrain with policy, gate with evals, emit a portable artifact.

Classify before reuse. Redaction, provider routing, data boundary, export scope, budget, and purge rules run before a trace becomes training or eval material. Failures become controls: labels, failure taxonomy, drift checks, thresholds, and locked regression sets decide whether a compile can proceed.

  • The compile run emits a signed artifact, manifest, target recipe, checksum, and verifier receipt
  • Every policy version is attached to downstream objects; unknown schemas stay opaque until mapped
  • Release-gate status blocks promotion until evals and protected slices pass

Backend Invariants

The backend standard is stricter than the website copy.

These invariants keep the platform credible when buyers inspect model gateways, observability stacks, agent tools, and enterprise control workflows.

Invariant · audit

Auditability

Every write emits an audit event: object ID, actor, tenant, environment, source, policy version, before/after state, and timestamp must be queryable and exportable.

Invariant · evidence

Evidence

Every claim-bearing object has proof: artifacts, receipts, reports, exports, readiness gates, and control mappings need source hashes, output hashes, versions, and owner state.

Invariant · exports

Exports

Every export records destination, payload class, policy scope, delivery status, checksum, retry state, and recipient context.

Invariant · schemas

Schemas

Unknown schemas stay opaque. Unmapped or sensitive payloads are stored with provenance and retention controls, but are not promoted into reusable datasets until classified.

Invariant · readiness

Readiness

Every open claim remains gated. Benchmark, certification, package, partner, standards, and adoption claims are visible as open gates until proof exists.

Invariant · operations

Operations

Every runtime target is explicit. Hosted, local, browser, edge, BYOC, offline, and restricted fleet targets carry compatibility, limits, and verification status.

Category Standard

To beat the category, Kolm has to be clearer than every adjacent tool.

The strongest position is not another agent, gateway, eval dashboard, or fine-tuning tool. It is the governed platform that connects them to runtime-ready artifacts and proof.

Adjacent toolWhere Kolm winsDurable object
agent toolsKolm should win after execution — capture what agents did, gate failures, produce artifacts, and export receipts instead of stopping at task automation.compare

model gatewaysKolm should control the lifecycle around providers — routing matters, but policy, eval, runtime target, receipt, and export state are the durable objects.lifecycle

fine-tuningKolm should stay runtime-neutral — the artifact, manifest, target, and receipt should survive provider and deployment changes.portable
