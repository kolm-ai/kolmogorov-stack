Platform / Architecture

The platform is the behavior-to-artifact control plane.

API behavior in. Device-fit models out. Kolm turns real traces, tool calls, policies, eval gates, and runtime requirements into signed artifacts, receipts, exports, and readiness-gated evidence that a buyer can inspect.

01 / Architecture Loop

The platform starts with observed behavior, not a blank model prompt.

A production AI/API workflow becomes useful only after its source, policy, eval, artifact, runtime, and evidence state are explicit.

1. source registered
Bind one production boundary. Organization, workspace, project, environment, API key, connector, source, and retention policy are created before data is reused.
source
2. behavior captured
Normalize the event stream. Provider logs, app events, agent traces, observability spans, warehouse rows, and custom records become trace and capture objects.
capture
3. policy applied
Classify before reuse. Unknown schemas stay opaque until mapped, redaction runs before export, and every policy version is attached to downstream objects.
policy
4. eval gate built
Failures become controls. Human labels, failure taxonomy, drift checks, thresholds, and locked regression sets decide whether a compile can proceed.
eval
5. artifact compiled
Output becomes portable. The compile run emits a signed artifact, manifest, target recipe, checksum, verifier receipt, and release-gate status.
artifact
6. proof exported
Evidence leaves the dashboard. Receipts, eval reports, governance packets, webhooks, SIEM/GRC events, warehouse exports, and CI status carry the same object IDs.
export
02 / Platform Layers

Each layer has a job, a state boundary, and an exportable proof trail.

This is how Kolm stays more useful than an agent demo, an eval dashboard, or a model gateway: the platform owns the lifecycle between behavior and runtime.

Data Plane

Capture without assuming shape

Accept structured, semi-structured, and opaque inputs. Store provenance, source hash, schema state, tenant, environment, and retention window with every capture.

Policy Plane

Constrain before processing

Apply redaction, provider routing, data boundary, export scope, budget, allowlist, and purge rules before a trace can become training or eval material.

Evaluation Plane

Turn failures into gates

Convert labels, edge cases, regressions, and policy violations into evals with thresholds, owners, versions, and release-blocking status.

Compiler Plane

Produce a portable artifact

Compile behavior and eval constraints into artifact, manifest, checksums, runtime target, verifier receipt, and approval state.

Runtime Plane

Deploy without provider lock-in

Target hosted, local, browser, edge, BYOC, offline, and restricted fleet paths with explicit compatibility and readiness caveats.

Evidence Plane

Export what buyers inspect

Emit audit events, evidence bundles, SIEM/GRC records, warehouse exports, webhooks, CI status, and readiness-gated claim state.

Admin Plane

Give operators control

Manage API keys, connector health, RBAC, environments, provider policy, cost limits, rate limits, retention, export destinations, and owner assignments.

Integration Plane

Move data both ways

Support inbound APIs, webhooks, streams, batches, queues, object storage, warehouses, CLI imports, and outbound evidence/export channels.

05 / Backend Invariants

The backend standard is stricter than the website copy.

These invariants keep the platform credible when buyers inspect model gateways, observability stacks, agent tools, and enterprise control workflows.

Auditability

Every write emits an audit event

Object ID, actor, tenant, environment, source, policy version, before/after state, and timestamp must be queryable and exportable.

Evidence

Every claim-bearing object has proof

Artifacts, receipts, reports, exports, readiness gates, and control mappings need source hashes, output hashes, versions, and owner state.

Exports

Every export has destination and receipt

Outbound packets should record destination, payload class, policy scope, delivery status, checksum, retry state, and recipient context.

Schemas

Unknown schemas stay opaque

Unmapped or sensitive payloads are stored with provenance and retention controls, but are not promoted into reusable datasets until classified.

Readiness

Every open claim remains gated

Benchmark, certification, package, partner, standards, and adoption claims are visible as open gates until proof exists.

Operations

Every runtime target is explicit

Hosted, local, browser, edge, BYOC, offline, and restricted fleet targets carry compatibility, limits, and verification status.

06 / Category Standard

To beat the category, Kolm has to be clearer than every adjacent tool.

The strongest position is not another agent, gateway, eval dashboard, or fine-tuning tool. It is the governed platform that connects them to runtime-ready artifacts and proof.

agent tools
Kolm should win after execution. Capture what agents did, gate failures, produce artifacts, and export receipts instead of stopping at task automation.
compare
observability
Kolm should convert logs into artifacts. Dashboards explain behavior; Kolm needs to compile it into portable outputs with evaluation and evidence.
artifact
model gateways
Kolm should control the lifecycle around providers. Routing matters, but policy, eval, runtime target, receipt, and export state are the durable objects.
lifecycle
eval tools
Kolm should make evals release gates. Scores are useful only when they block regressions, create receipts, and move into deployment decisions.
gate
fine-tuning
Kolm should stay runtime-neutral. The artifact, manifest, target, and receipt should survive provider and deployment changes.
portable
