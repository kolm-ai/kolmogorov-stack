How to contribute a measurement

If you have one of these devices and a kolm-compiled artifact, contribute a measurement back. The harness writes a JSON report that the benchmarks repo can ingest without manual cleanup.

Two commands

Run the harness, send the JSON

kolm eval --device <slug> --output ./benchmark.json && git pull-request kolm-ai/benchmarks ./benchmark.json

Replace <slug> with one of: pi5, jetson-orin-nx, mac-mini-m2, x86-nuc, web-gpu. The first command runs the holdout on your device, records tok/s and p95 latency, and emits a receipt. The second command opens a pull request against the public benchmarks repo with your JSON.

What we accept

  • A measurement with a signed receipt and a reachable model hash. We reject unsigned numbers.
  • A build hash for the runtime (llama.cpp commit, MLC commit, or WebGPU runtime version) so the measurement is reproducible.
  • A device label that includes silicon, memory, and OS so the matrix row is unambiguous.

What happens next

Maintainers verify the receipt, replay the eval on the same device class when we have one, and merge the PR. The new row replaces the “Not yet measured” cell within a week. Every contributor is credited in the row's notes column with a link to the PR.

Caveats / Limitations

  • Edge measurements vary by build. tok/s on Jetson depends heavily on the llama.cpp build flags (CUDA on vs off, batch size, kv-cache layout). The harness records the build hash so cross-device comparisons stay grounded.
  • WebGPU is a moving target. Chromium WebGPU shipped 113. Performance characteristics change with the underlying Dawn / Tint version. Pin the Chromium build in your benchmark row.
  • Numbers are a measurement, not a guarantee. Every cell on this page is replaceable. When we re-measure with a faster runtime or a better quant, we update the row and the previous number stays in the eval report linked from the receipt.
