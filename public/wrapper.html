For developers and platform teams

Drop-in OpenAI-compatible gateway, with capture and redaction built in.

Change one environment variable. Your application keeps the OpenAI SDK, Anthropic SDK, LangChain, Vercel AI SDK, Cursor, Continue, Aider, whatever. Behind the URL, kolm routes to the right provider, redacts PII at the edge, retries on failure, fails over, and signs every approved response into a tenant-fenced capture log you own.

01 · Gateway

One URL. Every model API.

Flip OPENAI_BASE_URL. Your app keeps the OpenAI SDK. Behind the scenes, kolm routes to OpenAI, Anthropic, vLLM, SGLang, TRT-LLM, TGI, Fireworks, Together, or your own kolm runtime. Set a per-namespace rate limit, retry budget, and failover order from the dashboard or the CLI.

  • OpenAI Chat Completions, drop-in compatible
  • Anthropic Messages, drop-in compatible
  • vLLM, SGLang, TRT-LLM, TGI, llama.cpp adapters
  • Per-namespace rate limits, retries, failover order
  • Stream and non-stream both supported
API reference → All integrations 
# one env var. that's the whole change.
$ export OPENAI_BASE_URL=https://kolm.ai/v1
$ export OPENAI_API_KEY=$KOLM_API_KEY

# your code is unchanged.
>>> from openai import OpenAI
>>> client = OpenAI()
>>> client.chat.completions.create(
...     model="claude-sonnet-4-6",
...     messages=[{"role":"user","content":"..."}]
... )
✓ routed to anthropic
✓ response captured: cap_8f3a1...
✓ receipt signed: rcp_92c4...

02 · Redact

PII never leaves your network.

Email, phone, address, name, SSN, account number, MRN. Detected and replaced before the request hits the model. Text, image, and audio redactors all share the same install_hint envelope, so if a redactor is missing the call refuses, it does not silently pass data through.

  • 13 PII classes detected pre-request
  • Image PII via YOLO ONNX (face, plate)
  • Audio voiceprint redaction via pyannote
  • HIPAA-grade audit trail per redacted field
  • Refuse, do not silent-pass, when detector missing
Security → HIPAA mode 
# redaction is on by default. one flag to turn off.
$ kolm redact text "Page Dr Lee at 415-555-2202 about MRN A1928"
{
  "redacted": "Page Dr [NAME] at [PHONE] about MRN [ID]",
  "hits": [
    {"class":"name", "range":[8,14]},
    {"class":"phone", "range":[18,30]},
    {"class":"id", "range":[42,47]}
  ],
  "detector": "kolm-rules-v3"
}

03 · Capture

Every approved response becomes your dataset.

One JSONL row per response: input, output, model, latency, scores, traces. Tenant-fenced at every read with defense-in-depth. Filter by namespace, tag, score, model, latency. Approve, reject, or tag rows from CLI, TUI, or web. Export as .kolm-dataset.jsonl for distill, anytime.

  • Tenant-fenced, defense-in-depth at every read
  • JSONL: input, output, scores, traces, latency
  • Approve, reject, tag from CLI / TUI / web
  • Export as dataset for distill
  • Stays in your namespace, ever

04 · Receipts

Signed receipts. Verifiable, anywhere.

Every approved response gets an Ed25519-signed receipt covering input hash, output hash, model id, timestamp, redaction summary, and provider response id. Verify offline. Re-verify in CI. The verifier is open-source and lives in 6 SDK languages.

  • Ed25519 sig per response, anchored to a public root
  • Inputs and outputs hashed, never raw on the wire
  • Offline verifier, no network call needed
  • 6 SDK languages: Node, Python, C, Rust, MCP, VSCode
  • Receipts survive rotate, deploy, restore
Receipt format → SDK reference 
# verify a receipt offline.
$ kolm verify rcp_92c4a8f...
{
  "ok": true,
  "cap_id": "cap_8f3a1...",
  "model": "claude-sonnet-4-6",
  "input_sha256": "3a9c...e1",
  "output_sha256": "b720...a4",
  "ts": "2026-05-22T17:18:09Z",
  "sig_alg": "ed25519",
  "anchor": "pki.kolm.ai/v1"
}

05 · Runtime

Same gateway, but in front of your own model.

Point the same wrapper at a .kolm artifact from Studio. Your client still uses the OpenAI SDK. Now requests run on your hardware, with the distilled and quantized model behind it. No code change in the app. Zero vendor coupling. Move it on-prem, into a VPC, onto an edge box, at any time.

  • Run a .kolm on CUDA, ROCm, Metal, CPU, or WASM
  • Same OpenAI Chat Completions surface, no rewrites
  • Air-gapped friendly, no phone-home, ever
  • kolm-runtime is open source, MIT
  • One artifact, every target, byte-reproducible build
Runtime targets → Offline mode 
# point the wrapper at a local artifact.
$ kolm runtime serve ./inbox-router.kolm --port 8080
✓ loaded inbox-router.kolm (124 MB, INT4, RTX 5090)
✓ listening on http://localhost:8080/v1

# the wrapper now points at your hardware.
$ export OPENAI_BASE_URL=http://localhost:8080/v1
$ python inbox.py
✓ 4.1 ms/token   1182 rows/min
✓ 0 phone-home, 0 outbound calls
✓ receipts signed locally with your tenant key

06 · SDKs

Six languages. The same shapes.

Node, Python, C, Rust, MCP, and a VSCode extension. Every SDK ships whoami, health, verify, changelog, intent.ask, and capture.log. C and Rust are single-file, no async runtime, no JSON parser bundled. Pair them with cJSON or serde_json on the caller side. The honesty contract: raw status, raw body, no is_success() sugar hiding a 4xx.

SDK reference → Source on GitHub 
// node, every call returns raw status and raw body.
import { Kolm } from '@kolm/sdk';

const k = new Kolm({
  base_url: process.env.KOLM_BASE_URL,
  api_key:  process.env.KOLM_API_KEY,
});

const { status, body } = await k.intent.ask({
  q: 'route this support email',
  modality: 'text',
});

if (status !== 200) throw new Error(body);
console.log(body.suggested_model);  // "qwen2.5-3b.kolm"
