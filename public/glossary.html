Reference · Glossary

The vocabulary of an agent security review.

When a buyer's security team vets an autonomous agent, both sides reach for the same words: signed evidence, offline verification, least privilege, prompt injection, provenance. Here is what each one means, defined plainly and linked to the page that shows it in practice. Start with how it works, see what we test, then verify a sample report yourself.

01 / Core terms

The artifact, and how it is trusted.

The five terms that describe what kolm produces and why a reviewer can believe it without taking anyone's word.

Term · Core

Agent Security Review

The process by which an enterprise buyer's security team evaluates an autonomous AI agent before approving it for production or signing a contract. It examines what the agent can access, what it records, where data flows, and how it resists manipulation, and it now asks agent-specific questions that a standard SOC 2 report or vendor questionnaire does not answer. kolm runs the same checks a reviewer runs and ships the results as signed evidence.

Related: Platform · How it works

Term · Core

Signed Evidence Report

A single canonical JSON artifact that records an agent's audit findings, each mapped to a named control, and sealed with an Ed25519 signature over its exact bytes. Because the signature covers the canonical payload, a downgraded finding or an inflated score breaks the seal in front of the reviewer. The buyer verifies it offline against the issuer's public key, so trust rests on the math, not on a vendor's claim.

Related: The report · Verify

Term · Core

Offline Verification

Checking a signed report's authenticity entirely in the verifier's own browser, using WebCrypto, with no account, no upload, and no kolm server in the trust path. The public key needed to verify travels inside the report, so the buyer needs nothing from the vendor or from us to confirm it. A first tier checks the signature against the bytes; a second tier confirms the signing key is the one the buyer expects.

Related: Verify a report · Trust center

Term · Core

Ed25519 Attestation

A statement of audit results signed with the Ed25519 elliptic-curve signature scheme (Curve25519, defined in RFC 8032). The signature is computed over the report's canonical, key-sorted, whitespace-free bytes, so any change to the content invalidates it. Verification is fast, deterministic, and runs without contacting an issuer, which is what makes a report checkable offline.

Related: How it works · Security

Term · Core wedge

Signed agent security record

A signed, timestamped, witnessed record that binds an agent's identity and provenance to its assessed security posture at a point in time. It carries the agent's control findings, the issuer's public key, an append-only log entry, and a trusted timestamp, so a buyer can confirm both what was attested and when. It is the durable evidence an agent presents when its security has already been reviewed, rather than re-answering a questionnaire for every deal.

Related: Research · Trust center · Transparency log

02 / The ASR control catalog

The controls a reviewer flags, named and tested.

Each ASR control is one thing a security reviewer checks, with the standard it maps to. Every finding in a signed report points back to one of these, so a reviewer traces each result to a framework, not to our word. The full catalog and crosswalk live on what we test.

ASR-1 · Permissions

Least-Privilege Audit

Compares the scopes an agent holds against the scopes it uses, and surfaces shared keys, standing grants, and high-impact actions reachable without a confirmation. Over-permissioning is the first thing a reviewer circles, so each finding ships with the calls that prove it.

Maps to SOC 2 CC6 · OWASP ASI03 · NIST MANAGE-1

ASR-2 · Audit trail

Audit-Trail Integrity

Tests whether agent activity is logged append-only, hash-chained, and retained, so the record a reviewer reads is the record that happened and not one rewritten after the fact. An editable log is not evidence, and this control is the regulatory anchor most reviewers cite first.

Maps to EU AI Act Art.12 · SOC 2 CC7

ASR-3 · Data egress

Data Egress Control

Enumerates every destination the agent can reach, the approved sub-processors behind them, and proof that sensitive fields are redacted before they cross the boundary. Where data goes, and what is masked before it goes, decides whether a reviewer signs off.

Maps to OWASP LLM02 · EU AI Act Art.10

ASR-4 · Injection

Prompt-Injection and Red-Team Testing

Runs a battery of direct and indirect prompt injection, guardrail bypass, and system-prompt extraction against the agent, with reproductions a reviewer can replay. Results are tested and reported, not warranted: the report states what held and what did not, scoped to exactly what was examined.

Maps to OWASP LLM01 · MITRE ATLAS

ASR-5 · Provenance

Model Provenance

Records the origin of the model and its dependencies: which base model, which version, and which supply-chain components the agent relies on. Provenance lets a reviewer trace a model back to a known source and detect an unvetted substitution before it reaches production.

Maps to ISO 42001 · NIST MAP-1

ASR-7 · Memory and retrieval

Memory and Retrieval Integrity

Examines an agent's long-term memory and retrieval-augmented (RAG) sources for poisoning, stale or unauthorized context, and untrusted content that can steer behavior. Because retrieved text can carry indirect injection, the integrity of what an agent remembers and retrieves is part of its security posture, not a separate concern.

Maps to OWASP LLM08 · relates to ASR-4

ASR-8 · Delegation

Multi-Agent Delegation

Assesses how an agent hands work to sub-agents or tools: whether delegated authority is scoped and revocable, whether a sub-agent can escalate beyond the caller's permissions, and whether the delegation chain is logged end to end. As agents call other agents, a single over-trusted hop can defeat least privilege everywhere upstream.

Relates to ASR-1 · ASR-2

Scope is contractual. Permission posture, redaction and audit-trail integrity are assessed. Injection is tested and reported, not warranted.

03 / Logging and time

How a report proves it was issued, and when.

A signature proves the bytes are intact. These two mechanisms add a public record and a trusted clock, so a buyer can also confirm a report was issued, never quietly replaced, and not back-dated.

Term · Provenance

Transparency Log and Inclusion Proof

A transparency log is an append-only, tamper-evident record of every report kolm issues. An inclusion proof is a compact cryptographic path showing that one specific report is recorded in the log without revealing the rest, the same idea behind certificate-transparency and Sigstore-style logs. Together they let a buyer confirm a report was issued and that the log it sits in cannot be silently rewritten.

Related: Transparency log · Trust center

Term · Provenance

RFC 3161 Timestamp

A trusted timestamp issued by a time-stamping authority under RFC 3161, which binds a hash of the report to a specific time so the attestation cannot be back-dated. Combined with the Ed25519 signature and the transparency-log entry, it answers not just who signed and what, but when. This matters when a buyer needs evidence that a posture was attested before a given deploy or deadline.

Related: How it works · Transparency log

04 / Frameworks and standards

The standards a reviewer already cites.

Every control maps to a framework a buyer's review group uses. kolm does not replace these; it produces the agent-specific evidence that slots into the work a team is already doing. See the full mapping on what we test.

Standard · Attestation

SOC 2 for AI Agents

Shorthand for the agent-specific evidence buyers now ask for on top of a SOC 2 report. SOC 2 attests an organization's controls, but not an autonomous agent's permissions, egress, or injection resistance, so reviewers ask a separate set of questions. kolm fills that gap and maps each finding back into the SOC 2, ISO 42001, and NIST AI RMF work a team is already doing. It complements a SOC 2 audit; it does not replace one.

Related: Platform · How it works

Standard · Regulation

EU AI Act Article 12

The Article of Regulation (EU) 2024/1689 that requires high-risk AI systems to automatically record events (record-keeping, or logging) over their lifetime, so the operation of the system is traceable. It is the regulatory anchor for ASR-2 audit-trail integrity: append-only, retained, and reviewable. A reviewer working an EU mandate looks here first.

Related: ASR-2 audit-trail · Platform

Standard · Framework

NIST AI RMF

The U.S. National Institute of Standards and Technology AI Risk Management Framework (AI RMF 1.0), organized around four functions: Govern, Map, Measure, and Manage. kolm maps controls such as least privilege (Manage) and provenance (Map) to its subcategories, so a finding traces to a recognized risk function rather than to a vendor's own scheme.

Related: ASR-1 · ASR-5 · Platform

Standard · Risk catalog

OWASP LLM Top 10

The Open Worldwide Application Security Project list of the most critical security risks for large-language-model and generative-AI applications, including prompt injection (LLM01), sensitive-information disclosure, supply-chain risks, and vector and embedding weaknesses (LLM08). kolm's injection, egress, and retrieval checks map directly to its entries.

Related: ASR-4 · ASR-3 · Research

Standard · Adversary model

MITRE ATLAS

The Adversarial Threat Landscape for Artificial-Intelligence Systems, a MITRE knowledge base of real-world tactics and techniques used against machine-learning systems, structured the same way as ATT&CK. kolm's red-team battery references ATLAS techniques so a reviewer can place each finding inside a shared, recognized adversary model.

Related: ASR-4 · Research
