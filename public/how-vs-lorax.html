kolm.ai  /  compare  /  how kolm relates to LoRAX

kolm builds the adapter. LoRAX serves it.

Predibase's LoRAX (github.com/predibase/lorax) is Apache-2.0 multi-LoRA serving infrastructure, heterogeneous continuous batching, adapter exchange scheduling, the SGMV kernel. It's the strongest adapter-serving infra in market. It is not a build tool, and kolm is not a serving cluster. The production-ready shape is both.

Wave 169 · reviewed 2026-05-17 · complement, not competitor

The stack with both.

kolm takes captured input/output pairs and emits a signed, eval-gated, receipt-bearing LoRA adapter. LoRAX takes one or more LoRA adapters and serves them with multi-tenant batching on a single GPU. The split is clean:

seeds.jsonl→kolm distill→signed .kolm + LoRA weights→LoRAX→OpenAI-compatible endpoint

The signed file is the contract between the two systems. LoRAX serves the LoRA bytes; kolm proves where those bytes came from, who signed them, and what eval gate they cleared.

What LoRAX does (well).

  • Multi-LoRA serving, thousands of fine-tuned models on a single GPU via Heterogeneous Continuous Batching + Adapter Exchange Scheduling + the SGMV kernel.
  • OpenAI-compatible API, existing OpenAI SDK consumers can switch to LoRAX with a base-URL change.
  • Base model breadth, Llama, CodeLlama, Mistral, Zephyr, Qwen. Loadable in fp16, bitsandbytes, GPT-Q, AWQ.
  • Production deployment, Docker + Kubernetes/Helm. Apache-2.0 license, no SaaS tax.

Where LoRAX is silent: it doesn't build the adapter, it doesn't sign anything, it doesn't gate on an eval, it doesn't ship offline / on-device / compiled native, and it doesn't carry a cryptographically verifiable provenance chain. Those are all kolm's surface.

The integration in one CLI session.

# 1. Build the signed adapter with kolm
$ kolm distill phone-normalizer \
 --teacher=anthropic:claude-opus-4-7 \
 --student-base=qwen2.5-0.5b \
 --redact=phi --eval-gate=0.85

 PASS K-score 0.917 (A=0.94 T=0.91 F=0.97 R=0.88)
 PASS PHI redactor: 18 Safe Harbor + 3 kolm extensions, no leakage
 PASS Ed25519 signature, Rekor transparency log entry
 WROTE phone-normalizer.kolm (LoRA weights + manifest + receipt chain)

# 2. Extract the LoRA for LoRAX
$ kolm extract phone-normalizer.kolm --target=lora-weights --out=./adapters/phone-normalizer

# 3. Serve with LoRAX as you already do
$ docker run --gpus all -v ./adapters:/data ghcr.io/predibase/lorax:latest \
 --model-id Qwen/Qwen2.5-0.5B --adapter-source local

# 4. Verify the served adapter still matches what kolm signed
$ kolm verify phone-normalizer.kolm
 PASS 24/24 checks · Ed25519 + Sigstore validated · receipt chain intact

When LoRAX alone is the right answer.

Your team already controls the LoRA training pipeline, the data isn't regulated, and you don't need a portable signed file. LoRAX is excellent and Apache-2.0, running it bare is the cheapest path to multi-tenant adapter serving. Don't pay kolm tax for a workflow that doesn't need the receipt.

When kolm + LoRAX is the right shape.

Your compliance team needs to know what each adapter was trained on, your auditor needs a third-party-verifiable signature on the eval, your tenant data can't leave its boundary, and you still want LoRAX's throughput. The signed .kolm is the contract between the build team and the serve team.
