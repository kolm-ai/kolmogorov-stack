Why a template

Three reasons most teams should start here.

You can always fork the spec later. Most teams discover they never need to.

01 · Hyperparameters

Stop bikeshedding hyperparameters

The templates ship known-good values for their workflow — learning rate, batch size, LoRA rank, context length, sampling temperature, KD loss weights. Tuned on real distill runs, not copy-pasted from the original transformer paper.

02 · Seed schemas

Seed schemas included

Each template ships a seed-generator with diversity buckets (paraphrase, edge-case, adversarial), a fail-closed PHI redactor config, and an explicit train / holdout split. You bring the domain knobs; the schema covers the structure.

03 · Eval harness

Eval harness wired

The template ships a holdout set and a wired eval harness. Run kolm eval after distill and you get teacher-vs-student agreement-rate, latency p50/p99, and a signed receipt that pins the holdout SHA to the run.

Build your own

A template is a directory. That is the whole spec.

The templates that ship with kolm are just the first four. You can author and share your own.

What goes in a template directory

A template is a directory with three things: a spec.json (base model, teachers, hyperparameters, output target), a seed-generator script (Python or Node) that emits training rows, and an eval set (JSONL with prompt + expected pairs). Drop it under ~/.kolm/templates/<name>/ and it shows up under kolm distill template <name>.

The full authoring guide — including how to ship a template as a Git repo and how to register one with the kolm recipe registry — is at /docs/recipes/authoring-a-template.
