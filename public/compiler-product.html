Capture → compile → compose → deploy

From live calls to a model on your own hardware.

Four steps from your API traffic to a signed model you own. Capture the behavior you already use, compile it into one portable file, combine your specialists, and run it where it fits. No rewrite to get here, no lock-in once you have it.

01 · /v1/route

Capture

Point your existing calls at Kolm. It keeps the prompts, tool calls, outcomes and receipts that define the behavior you want to own.

02 · /v1/compile

Compile

One behavior in, one signed model out. Compile (build one captured behavior into a model)To compile is to take one captured behavior and build it into a single deployable .kolm model: model weights, recipe, evals, and signature in one file. One behavior in, one model out. bundles the model, recipe, evals and signature into one .kolm file you can move anywhere.

03 · /v1/compose

Compose

Many models in, one system out. Compose (combine several models into one router)To compose is to combine several compiled models into one system. A router picks the right specialist per call, for example a redactor for one request and a classifier for the next. Compile makes the parts; compose assembles them. joins your narrow specialists, with routing that picks the right expert for each call.

04 · /v1/devices

Deploy

Run it on the smallest runtime that fits, then serve, install or export - on the hardware you already have.

Compile vs compose

Two different jobs, in plain terms.

People mix these up. Compile builds one model from one behavior. Compose wires several compiled models into one routed system. You compile first, then compose.

Inside a compile

A compile is a loop, not a button.

A behavior does not become a signed model on the first try. Kolm runs your evals, and any failure routes back through triage and a rebuild until every gate passes. Only then does it sign and seal the file.

A failed gate loops back to rebuild automatically. Nothing compiles until every gate is green.
