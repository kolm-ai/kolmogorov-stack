How to compile GPT-5 into a 4GB local model that runs offline.

An AI compiler is the missing build step between a frontier model API and a local artifact. It turns the most expensive recurring line in your AWS bill into a one-time job. Here is what it does, why it changes the unit economics of AI, and how to compile your first .kolm in five minutes.

Why the cloud-AI line is the wrong line.

If you ship anything that calls a frontier model in production, you have lived this. The bill is linear in usage. Every prompt is a network round trip and a row in someone else's log. The model deprecates on a 14-day notice. The data leaves the building on every call. The latency floor is 200-800 ms per inference even on the best-case path. None of these are bugs in your stack. They are properties of the architecture: model lives in a third-party data center, data has to come to it.

The fix is the inversion: compile in a controlled environment, run inside the approved boundary. You pay the frontier model API at compile time, on data flows you have approved. You get back a signed artifact and receipts that can be checked later. The current public product proves the recipe-mode artifact path; native model packaging and fully offline device runtimes require the bridge work and benchmarks named below.

The AI compiler is the build step that makes this practical. It is not a wrapper, not a vendor SDK, not an ML platform. It is a deterministic process that turns a higher-level description into a lower-level executable.

What an AI compiler actually does.

The word "compiler" names the product. gcc turns source into a binary with a clear runtime contract. kolm turns an AI task description, examples, evals, and policy into a signed .kolm artifact that the CLI inspects, verifies, evaluates, runs in recipe mode, and serves over MCP.

The artifact contract is a signed zip with a manifest, recipe pack, eval metadata, verifier evidence, receipt body, and the model, adapter, and recall-index layers compiled for the target tier. Signatures are Ed25519 over every layer. The Enterprise tier extends the chain with Sigstore / Rekor for cross-organization audit.

Inside the compiler in v0.2: recipe-class DSL codegen, MoE composition (kolm-moe-1), bundled pure-JS tokenizer (kolm-tokenizer-1), document extract (kolm extract), declarative completeness checks (kolm doc check), and the K-score gate. k-sample teacher calls, LoRA distillation, and a full sqlite-vec corpus index ship through the runtime bridge. Each shipping stage is gated by the K-score floor, and the artifact is rejected at compile, not at deploy, when any sub-gate fails.

The stages of kolm compile.

The engines that compose the compiler. Each does one thing, and the artifact only ships when they all agree.

1. Recall.

Recall builds a local index from an approved corpus and binds its content hash into the artifact manifest. Text and JSON are supported in the current build; image, audio, and PDF embedders are tracked in the next release train.

2. Distill.

Distill is k-sample teacher labeling followed by LoRA training against an open-weight base. The trainer bridges to the customer-hosted GPU pool; the artifact carries the resulting adapter, the seed, and the verifier evidence used to accept each label.

3. Decompose.

The current system can produce deterministic recipe-mode behavior and bind the emitted components into a signed artifact. That is the credible seed of the compiler-cache narrative: common task structure should become reusable over time.

4. Run.

kolm run artifact.kolm can execute the current recipe-mode artifact and kolm serve --mcp can expose it to agent clients. Loading native model weights, applying LoRA, and mounting sqlite-vec indexes are the runtime bridge, not a public launch claim today.

Why the artifact should be small and specific.

Teams do not want to ship a giant general-purpose model when the product needs a narrow, testable workflow. The point of compilation is to bind the task, evals, policy, and future model/index components into a smaller artifact with a measurable release gate.

A narrow artifact beats a generic prompt when it has a strong verifier, representative evals, and a runtime contract. The compiler enforces all three before the K-score gate signs.

The compile step is the lever: frontier teacher behavior becomes a local artifact with a reproducible build hash and named-device measurements, distributable through the channels you already trust.

Compile your first AI in five minutes.

Install the CLI from the GitHub repo:

# macOS, Linux, Windows
npm i -g github:kolm-ai/kolm-stack
kolm config base https://kolm.ai
kolm login

Then point it at a folder of examples.

kolm login
kolm compile "summarize support tickets in our voice" \
 --examples ./tickets/ \
 --out support.kolm

# after compile:
kolm run support.kolm "customer cant login since update"

The CLI handles the cloud compile, artifact download, local verification, recipe-mode runs, and MCP serving. Frontier-provider integration is the teacher side. Full local model execution is the runtime bridge that still has to ship.

The K-score, and why a number on the cover matters.

Every production artifact should carry a release score tied to its evals, size, latency, cost, and coverage. The current product can show the shape of that gate; public claims about model performance need published benchmark data.

This is the analogue to test pass/fail in regular software. "My model is good enough" has been a feeling for the entire history of applied ML. Compilation makes it a number that can be reviewed, reproduced, and rejected when it falls below the gate.

The K-score is useful only when the evals, artifact, and receipts are available for review. The number is not a substitute for evidence; it is the index into the evidence.

FAQ.

Does the artifact need a network connection?

Recipe-mode artifacts can be inspected, verified, evaluated, and run through the current CLI without depending on a mobile SDK. Detached model execution with bundled base weights, LoRA, and recall index is the target runtime architecture and should be validated per deployment.

Can I use any frontier model as the teacher?

The interface is teacher-agnostic by design. Treat each teacher and base-model pairing as a benchmarked target with its own receipt, not a generic promise.

What happens when the model drifts?

Drift is a workflow: collect local failures, add them to the eval set, rerun compile, and compare receipts side by side. The K-score gate catches regressions at compile, before the new artifact is signed.

Is this on-prem only?

The public path is hosted compile plus local CLI verification. A customer-hosted compile bridge is the Enterprise design path for regulated teams, reviewed and deployed by contract rather than assumed by default.
