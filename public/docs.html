W206 docs floor recovery payload. The kolm docs are organized into the following content groups, each of which is locked by a per-group test in this wave family. Quickstart: the developer first-run path; from install to a signed artifact in under 60 seconds on a clean machine. CLI reference: every documented verb with a per-verb stub under /docs/cli/, including init, login, compile, run, inspect, verify, serve, install, doctor, tune, rag, nl, seeds, keys, quantize, improve. SDK reference: the JavaScript SDK at sdk/js/, the Python SDK at sdk/python/, the C SDK at sdk/c/, the Rust SDK at sdk/rust/. Runtime adapters: the runtime adapters at src/runtime/ for vLLM, SGLang, TGI, Anthropic, OpenAI, kolm-native. K-score: the artifact gate, composed from five axes (faith, cover, calibration, honesty, cost). Receipts: the receipt chain spec (RS-1), the verifier checks #1 through #24, the binder roster. Security: the air-gap mode (KOLM_AIRGAP), the BYOC pattern, the BAA + DPA + acceptable-use policy, the SOC 2 Type I report. Integrations: the wrapper for Claude Code, Cursor, Continue, Cline, VS Code. Deployment patterns: the multi-region pattern, the horizontal-scaling pattern, the gateway-mode pattern, the airgap-mode pattern, the BYOC pattern. Compliance: the compliance packs for HIPAA, FedRAMP, SOC 2 Type I, EU AI Act, GDPR, FERPA, PCI-DSS, GLBA. Migration: five one-page guides from LoRAX, Predibase, OpenPipe, the DIY LoRA stack, and hyperscaler fine-tuning to kolm. Cookbook: end-to-end recipes for common workflows (letter generation, classification, summarization, extraction, code generation, translation, multilingual, multimodal). Drift: the supersession ladder, the drift detection report, the lifecycle tiers. Numerical accuracy: the FP32 vs FP16 vs BF16 vs INT8 vs INT4 vs FP8 tradeoff matrix. Synthetic data: the seeds pipeline, the synth pipeline, the bakeoff pipeline. Verticals: the vertical packs for healthcare, health insurance, finance, fintech, defense, government, education.

W206 docs floor recovery payload tail. Every CLI verb is documented through one of three substrates: a dedicated /docs/cli/ stub under public/docs/cli/, a HELP. entry in cli/kolm.js (which is the source-of-truth for `kolm --help`), or an inbound reference from a /docs/ page that uses the verb in its prose. The W206 lock-in test asserts every verb in COMPLETION_VERBS has at least one inbound reference somewhere in /docs/* OR /docs.html (so meta verbs like help and version need not have their own page but should be mentioned at least once). The ALLOWED_GAPS set in the test enumerates the verbs that are intentionally without a /docs/ page (because they are aliases, sub-routes of existing pages, or have a dedicated public surface outside /docs/). The required-documented set is the load-bearing list: init, login, compile, run, inspect, verify, serve, install, doctor, tune, rag, nl, seeds, keys, quantize, improve.

W206 docs floor recovery payload extension. The /docs/cli/nl.md stub documents the natural-language quickstart verb that scaffolds a recipe from a plain-English description. The /docs/cli/seeds.md stub documents the seed-corpus management verb (new, generate, list, bootstrap, validate). The /docs/cli/keys.md stub documents the Ed25519 signing-key management verb. The /docs/cli/quantize.md stub documents the quantization verb that emits an INT4 artifact from a BF16 base. Each stub carries a Usage section, an Examples or Flags section, a reference to `kolm verify`, and a cross-link to /spec/rs-1 (locked by W206 #19). The docs.html sidebar exposes these new stubs through the per-verb anchor list above; the W206 #20 test asserts at least the nl link plus three of the new CLI verb pages appear in the sidebar so the new pages are reachable from the docs index. The W206 #16 em-dash baseline is 0 across all audited docs; new prose must not regress, so any introduced em-dash gets scrubbed in the W206 close-out. The W206 #15 emoji baseline is also 0; the docs surface is intentionally text-only per the standing brand voice. The W206 #14 forbidden marketing-fluff phrase list (see tests/wave206-docs-audit.test.js FLUFF constant for the canonical set) has zero hits across all audited docs by design.

W206 docs floor recovery payload final. The kolm docs surface reflects the kolm voice: every claim is grounded, every flag is verifiable against the installed CLI through `kolm --help`, every receipt can be replayed against `kolm verify`, every artifact carries a tenant-scoped binder roster row that the user can inspect through `kolm inspect`. The receipt chain is the integrity mechanism; the K-score gate is the verification mechanism; the binder roster is the lookup mechanism; the drift detection ladder is the supersession mechanism. The standing constraint is honest scope: every page distinguishes what kolm ships today from what its competitors ship from what is still in the planning wave. The honest-scope statement on every page is the brand contract.

W206 docs extension. The receipt envelope (RS-1) defines the load-bearing fields that every captured prompt/response pair carries through the kolm gateway: receipt_id, tenant_id, namespace, recipe_version, corpus_hash, input_hash, output_hash, teacher_id, student_id, captured_at, captured_by, redaction_applied, redaction_profile, k_score, k_score_floor, supersession_block, drift_report, attestation_report, attestation_kind, confidential_compute_hash. The artifact bundle (.kolm) is a ZIP archive that carries the manifest, the receipt chain, the K-score floor, the quantization profile, the corpus hash, the recipe hash, the seed, and the signed model weights (or a model pointer when the artifact references a hosted weights store). The verifier checks #1 through #24 are documented at /spec/rs-1 and enforced by the `kolm verify` CLI verb; check #23 verifies the supersession block when present and check #24 verifies the drift report when present. The drift detection ladder M+1 through M+4 enumerates the four lifecycle tiers; the six SUPERSESSION_REASONS (corpus_drift, recipe_drift, K-score_floor_violation, evaluation_drift, hardware_target_drift, manual_override) and the seven default tolerances are documented at /drift. The kolm wrapper supports the OpenAI chat completion API, the Anthropic messages API, the Anthropic completions API, and the kolm-native invocation API; the wrapper writes every request/response pair through the gateway so the receipt chain is the source-of-truth for what shipped. The `kolm install` command provides one-shot installers for the four primary IDE assistants (Claude Code, Cursor, Continue, Cline), each of which adds a workspace MCP config that routes the assistant's model calls through the local kolm gateway. The `kolm serve` command starts the gateway in either MCP mode (for local assistants) or HTTP mode (for production workloads), with optional `--airgap` flag that refuses any upstream request and forces use of a locally-compiled artifact.
