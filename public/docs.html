01 · QUICKSTART

Start by routing one namespace.

Keep the first integration small. Route one OpenAI-compatible namespace, export captured examples, then compile only after the behavior has enough eval and regression coverage.

quickstart.sh · route · capture · compile
$ curl https://kolm.ai/v1/route/chat/completions \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"namespace":"billing-agent","model":"openai:gpt-4.1-mini","messages":[{"role":"user","content":"classify this ticket"}]}'

$ curl https://kolm.ai/v1/capture/export \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -o billing-agent.jsonl

$ curl https://kolm.ai/v1/compile \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"task":"billing-agent","examples_file":"billing-agent.jsonl","target":"local-cpu"}'

03 · API CONTROL CENTER

Every dashboard control needs an API contract.

The API Control Center is the enterprise surface for source registration, provider policy, credentials, retention, redaction, eval gates, compile state, runtime targets, exports, and readiness gates.

  • workspace, project, environment, source, connector, credential, and provider define the enterprise boundary.
  • capture policy, redaction policy, retention policy, eval suite, failure taxonomy, and regression set define the governance boundary.
  • compile run, artifact, manifest, target runtime, receipt, export, and readiness gate define the release boundary.

05 · LIFECYCLE ROUTES

The loop is the API surface.

lifecycle routes · behavior → artifact → runtime
POST /v1/route/chat/completions OpenAI-compatible traffic
POST /v1/gateway/dispatch capture · cache · fallback
GET /v1/capture/export log · list · bulk
POST /v1/compile .kolm
POST /v1/serve detect · recommend · install
  • POST /v1/route/chat/completions sends OpenAI-compatible traffic through a controlled route.
  • POST /v1/gateway/dispatch dispatches to configured providers with capture, cache, fallback, and quality-router controls.
  • POST /v1/capture/log, GET /v1/captures/list, GET /v1/capture/export, POST /v1/capture/bulk move behavior into review and training workflows.
  • POST /v1/compile/estimate, POST /v1/compile/preview, POST /v1/compile, GET /v1/compile/:id/.kolm move from data to signed artifact.
  • GET /v1/devices/detect, POST /v1/devices/recommend, POST /v1/devices/:id/install, POST /v1/serve move artifact state into runtime targets.

06 · ERRORS AND READINESS

Docs should show failure paths.

A serious enterprise API documents the edges: authentication, rate limits, idempotency, readiness states, unsupported targets, connector ownership, and recovery hints.

AUTH

Bearer keys

Use Authorization: Bearer $KOLM_API_KEY. Create a key at /signup or through the CLI.

ERRORS

Stable codes

Error responses should include stable error codes and a fix or next step.

IDEMPOTENCY

Safe writes

Write routes that can create work should accept idempotency or provide status lookup.

READINESS

Explicit gates

Open gates remain explicit for external partners, live certification, package release, and public benchmark data.

07 · EVIDENCE AND VERIFY

Every artifact ships a receipt you can check.

Compiled artifacts are signed and content-addressed. Export the receipts, hand them to a buyer, and let anyone verify the artifact against the public verifier — no access to your workspace required.

  • Signed .kolm artifacts with Ed25519 signatures and content-addressed receipts.
  • Export eval reports, manifests, and receipts as governance evidence.
  • Verify any artifact at /verify; browse the checks the compiler runs at /checks.

08 · CLI AND SDK

Local tools should match hosted docs.

Developers should be able to move between API, CLI, SDK, account console, and static contracts without learning a second product language.

kolm-cli · signup → compile → serve
$ kolm signup --email you@company.com
$ kolm login --key ks_...
$ kolm compile "billing-agent" --target local-cpu
$ kolm verify artifact.kolm
$ kolm serve artifact.kolm
