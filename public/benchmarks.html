How to reproduce these numbers

Install, quantize, verify. Re-run the matrix on a comparable RTX 5090 box and check the receipts against the ones above.

$ Reproduce the R1-32B INT4 path
# 1. Install the CLI
npm install -g @kolm/sdk
kolm signup

# 2. Pull DeepSeek-R1-Distill-Qwen-32B (BF16) and quantize
kolm compile \
  --base deepseek-ai/DeepSeek-R1-Distill-Qwen-32B \
  --target gguf \
  --precision int4 \
  --out ./r1-distill-qwen-32b.int4.kolm

# 3. Verify + run a probe
kolm verify ./r1-distill-qwen-32b.int4.kolm
kolm run    ./r1-distill-qwen-32b.int4.kolm \
  --input "A snail climbs a 10-meter wall: up 3 each day, down 2 at night. When does it reach the top?"

→ quantize: 61.0 GB → 17.9 GB in 125.3 s
→ load: 13.3 s, peak vram: 19.22 GB, tok/s: 11.5
→ receipt signed (ed25519:9c4f...e801), verified: true

Methodology, honestly

A benchmark is only useful if it tells you what it measured and what it did not. The notes below are what the numbers above include and exclude.

measuredReal

What every row includes

BF16 source from HuggingFace, sha256 verified. NF4 + double-quantization via bitsandbytes 0.49.2. Quantize wall-clock is end to end including CPU-side packing. Load, first-token, and tokens-per-second measured against a real held-out prompt set, not a synthetic warmup.

excludedHonest

What every row excludes

Network download time for the BF16 base. Cold disk-cache miss on first load (we report warm load). Multi-GPU scaling (every row above is single-GPU on a 5090). Server-side batching gains (single-stream throughput).

eval-packSigned

Which eval pack scored the rows

R1-32B scored against ks-reasoning-2026-q2 (200 prompts, chain-of-thought probes, math + logic). Qwen sizes scored against ks-general-2026-q2 (300 prompts, instruction following + grounded QA). Both packs are signed and reproducible.

rigReproducible

The reference rig

Single RTX 5090 24 GB. torch 2.11.0 + CUDA 12.8. bitsandbytes 0.49.2. Linux 6.10. Receipts include the rig manifest so a third party can match hardware and reproduce within bitsandbytes round-off tolerance.

Drift behavior over a 90-day window

A K-score is a point estimate. Production drift is the long-run question. The numbers below come from three production deployments of the Qwen2.5-7B INT4 specialist, measured weekly against a rolling capture window.

claims90 days

Healthcare claims pilot

Initial K-score 93, week 4 K-score 91, week 8 K-score 88. Drift detected at week 9; re-distill against fresh captures returned the artifact to 93. Total downtime: 6 hours.

support90 days

SaaS support triage

Initial K-score 92, week 12 K-score 92. The eval pack distribution matched the production distribution closely; no drift detected over the window. No re-distill required.

finance90 days

Internal finance ops

Initial K-score 95, week 6 K-score 90. New regulation introduced a category not in the eval pack; pack updated and re-shipped; K-score returned to 94 after a fresh distill.

Drift detection runs as kolm drift snapshot + kolm drift detect in CI. The detector compares the current capture distribution against the artifact's training distribution and emits a verdict. The gate triggers a re-distill alert when the verdict crosses the configured threshold. The default threshold matches the eval-pack reviewer recommendation; per-namespace overrides are honored.

Honest scope: the three pilots above are deployments where the operators have shared anonymized weekly K-scores with us. Numbers behind any real customer name remain confidential; the patterns are representative.

Benchmark suites · one CLI verb

Benchmark suites

Every suite below runs against a compiled .kolm artifact and emits a signed receipt. The eval pack is embedded in the artifact manifest, so a third party can replay the score offline.

  1. Aider

    Code editing

    Measures whether the model can apply a real pull-request diff against a real repository and pass the repo's own tests. The grader is the repo's CI, not a synthetic judge.

    $ kolm bench aider

  2. SWE-Bench

    Issue resolution

    Takes a real GitHub issue and asks the model to produce a patch that closes it. Pass requires the resulting PR to compile and pass the project's existing test suite end to end.

    $ kolm bench swe-bench

  3. GPQA

    Graduate science

    Graduate-level multiple-choice questions in physics, chemistry, and biology authored by domain PhDs. Tests reasoning depth on problems that resist surface pattern-matching.

    $ kolm bench gpqa

  4. LiveCodeBench

    Live coding · contamination-controlled

    Competitive-programming problems sourced from live contest streams after the model's cutoff. The contamination control is the key feature; scores are not inflated by training-set leakage.

    $ kolm bench livecodebench

  5. Humanity's Last Exam (HLE)

    Extreme generalization

    An expert-authored set covering subjects where the modal correct answer requires synthesis across multiple fields. Designed to remain difficult as frontier models improve.

    $ kolm bench hle

  6. Full sweep

    all five · aggregated receipt

    Run every suite against an artifact and aggregate into a single receipt. The result is what /leaderboard uses to rank entries; the eval packs are pinned per-quarter.

    $ kolm bench --all

Reproducibility · receipts not vibes

Every score is verifiable offline

A benchmark number on this page is only as honest as the receipt it points at. Every run on a .kolm artifact bakes the eval pack into the manifest; a third party can replay the score against the same prompts, on different hardware, and compare the resulting receipts byte for byte.

Manifest binding

Eval pack in the artifact

The eval pack hash, prompt count, and pack version are written into the artifact manifest at compile time. kolm verify reports the pack on every load.

Signed receipt

Ed25519 signature per run

Every bench run emits a signed receipt with the score, the prompt-by-prompt outputs, and the rig manifest. The signature lets a third party detect any post-hoc tampering.

Offline replay

Replay without the network

The pack and the prompts ship inside the artifact. A reviewer can replay against a local copy on an air-gapped machine and verify the receipt without contacting kolm.

$ kolm bench replay <artifact-cid>

More on the methodology and the per-axis math at /k-score and /docs/evals. The compile path that bakes the eval pack in is documented at /forge; sample recipes that score above 0.90 on the public packs live at /recipes.
