How kolm works

One workflow. Three lines of code.

Capture what works, compile it once, run it forever. No vendor lock-in, no per-token bill after compile, no model you can't reproduce.

1. Capture: without rewriting anything

Flip one env var and your app routes through kolm's OpenAI-compatible gateway. Every approved response is hashed, timestamped, and stored in a tenant-isolated capture log you can audit, replay, or export.

  • Works with OpenAI, Anthropic, vLLM, SGLang, TRT-LLM, TGI, Fireworks
  • Drop-in for the OpenAI SDK, Anthropic SDK, LangChain, Vercel AI SDK
  • PII redaction at the edge (W454 text, W462 image, W464 audio)

2. Compile: distill + quantize in one command

Take the approved captures, distill them into a smaller open student, and quantize. The output is a signed .kolm artifact with the seed split, the teacher receipts, and an Ed25519 sidecar binding the weights to the spec.

  • INT4 / INT8 / BF16: same artifact format, different size class
  • Frozen evals, K-score drift detection, bake-off against the teacher
  • Reproducible: same captures + same seed = bit-identical output

3. Run: anywhere, with receipts

A .kolm is a single sealed file. The runtime verifies the signature, loads the right backend for the device, and emits a signed receipt for every response.

  • Backends: vLLM, SGLang, TRT-LLM, llama.cpp, candle, MLX, WebGPU
  • Targets: cloud GPU, laptop, browser (WASM), phone, VPC, air-gap
  • Verify any response in two seconds at /verify-prod

02 · Capture

Build a private dataset, automatically, from real traffic.

Every approved gateway response becomes one row in a tenant-fenced JSONL store. The row carries the namespace, the (redacted) input, the response, a hash chain to the previous row, and an Ed25519 signature. The capture log is the seed corpus for distill.

You can browse, filter, score, label, and approve rows in the dashboard or the TUI. Or write a policy that auto-approves rows above a confidence threshold, or that auto-rejects rows containing PII the redactor missed. Captures never leave your tenant.

  • Tenant-fenced storage with per-row defense-in-depth
  • Hash chain links every row to the prior one for tamper detection
  • Export to JSONL, Parquet, Hugging Face dataset, or signed eval set
  • Label studio integration; LLM-judge auto-scoring; human review queue

03 · Distill

Turn frontier calls into a specialist that fits on a laptop.

Pick a teacher (claude-sonnet, gpt-4o, your own) and a smaller open student (qwen2.5-{0.5b, 3b, 7b, 14b}, llama-3-8b, mistral-7b, gemma-2-9b, phi-3-mini). Kolm samples the approved captures, trains the student against the teacher's responses, and ships a fine-tuned checkpoint.

Defaults are honest. LoRA rank 16. Cosine schedule. 3 epochs. Seed split is reproducible. Frozen eval is held out at compile time, never seen by training. Optional bake-off computes token Jaccard against the teacher and an LLM-judge score; you see the gap before you ship.

  • Students: 13 open models, INT4 / INT8 / BF16, 0.5B to 70B

04 · Compile

Pack weights, spec, evals, and receipts into one signed file.

A .kolm is a single sealed file. Inside: model weights, the spec (tokenizer, template, sampling defaults), the seed split, the frozen eval, the teacher fingerprint, every receipt that fed training, and an Ed25519 sidecar that binds the artifact to the spec. Change one byte, the signature breaks.

Compile target picks the format. INT4 (NF4 + double quant, smallest), INT8 (balanced), BF16 (full quality). Same artifact format across all three. You can also compile to native code (C, Rust) or WASM for embedded targets.

  • One file. No "download these 14 shards." No "remember to grab the tokenizer."
  • Verifies in two seconds. Re-verifies on every load by the runtime.
  • Optional attestation: TDX, SEV-SNP, Nitro, NRAS embedded into the manifest
  • SLSA-style provenance graph from receipts to artifact hash

05 · Runtime

One artifact. Every backend. Cloud, browser, phone, VPC, air-gap.

The kolm runtime is a thin shell around a backend the device can actually run. On a cloud GPU it picks vLLM, SGLang, or TRT-LLM. On a laptop it picks llama.cpp or candle. On a Mac it picks MLX. In a browser it picks WebGPU (with a WASM fallback). On a phone it picks Core ML (iOS) or NNAPI (Android).

Every response carries a signed receipt (kolm-audit-1 envelope) that links the response to the artifact hash, the spec, and the input hash. You can verify any response at /verify-prod in two seconds without giving us the response text.

  • Receipts are off by default for free tier, on by default for Pro+
  • Tenant fence enforced inside the runtime (RS-1 sandbox)
  • Optional confidential compute: TDX / SEV-SNP / Nitro / NRAS verified at load

06 · Registry

Verify any kolm, by anyone, in two seconds.

The registry is a public, queryable, signed index of every compile. By default a compile is private; you can publish to make it verifiable by anyone, or keep it inside a private registry on a Business tenant. Either way, a receipt always points back to a registry row.

The registry lock-ins: artifact hash, spec hash, teacher fingerprint, training-data hash, signing key, attestation report (if any), and a hash chain to the prior registry row. If a compile is in the registry, a third party can verify any response that artifact produced without seeing the response.

  • Public queryable: anyone can verify any published .kolm
  • Private mirror: Business tenants run their own registry inside the VPC
  • Hash-chained: append-only, tamper-evident
  • OCI-style content addressing for the artifact bytes

K-score & drift

Catch the day your model gets dumber.

Every compile carries a frozen K-score (composite of held-out Jaccard, judge win-rate, and refusal rate). The runtime computes a fresh K-score from live responses and alerts when it drops. /k-score is the full method.

k-score, dailydrift band

The recipe

  1. Snapshot at compile. Frozen eval rows + their teacher answers + the artifact's responses on those rows. The composite is the baseline K-score.
  2. Detect on every batch. Production responses are sampled into a sliding window and rescored against the frozen eval. Cheap, no human in the loop.
  3. Alert on band exit. If the live K-score drops more than 10 points below baseline for 3 consecutive windows, the runtime fires a drift event.
  4. Recompile from fresh captures. A new compile against the most recent captures rebuilds the artifact; the bake-off shows whether the new compile beats the old one.

Drift alerts go to webhook, email, or PagerDuty. The full event spec is at /drift. The K-score formula and its test coverage are at /k-score.

Under the hood

For engineers who want to know what's actually happening.

Three notes for engineers who like to read source. The full set is at /docs; the source is at github.com/sneaky-hippo/kolmogorov-stack.

Note 01
Why NF4 + double-quant for INT4

NF4 (normalized 4-bit floats) keeps the distribution Gaussian after quantization, which matches the actual weight distribution of a transformer FFN. The "double quant" step quantizes the per-block scale factor itself, recovering ~0.4 bits per weight on a 7B model. Net: ~24% smaller artifact at the same Jaccard. Implementation lives at workers/quantize/quantize_int4.py.

Note 02
Why Ed25519 for the sidecar

Tiny keys (32 bytes), tiny signatures (64 bytes), constant-time signing, no parameter choices that can be misconfigured (unlike ECDSA). Verification is ~70k sigs/sec on a laptop. The signing key never leaves your tenant; the sidecar is artifact.kolm.sig next to the artifact. Format spec at docs/artifact-spec.md.

Note 03
Why a frozen eval, not a live one

A live eval drifts with your traffic, so a regression looks like a model change when it's really a data change. A frozen eval (captured at compile, hash-pinned) lets you compare two artifacts on the same questions. The trade-off: you need to re-freeze when the eval gets stale (we suggest every quarter). Full method at /k-score.

Note 04
Why the artifact is one file, not a directory

A directory is too easy to break: someone deletes the tokenizer, the model loads but emits garbage, nobody notices. A single sealed file forces an all-or-nothing load. The cost is a slightly bigger artifact (we use zstd, level 19, on the weights). The win is "rsync this one file" is enough to deploy.

Note 05
Why a tenant fence at every row

The capture log is indexed by tenant; that's the cheap fence. The defense-in-depth fence is re-checking the tenant on every row read inside loops. The cost is one tenant comparison per row. The win is a single misuse can never cross a tenant boundary even if upstream indexing is bypassed. Audit at src/event-store.js.

Note 06
Why receipts use hash-of-input, not the input

A receipt has to be safely shareable. So the receipt carries sha256(input), not the input. A third party verifying the receipt provides the input themselves, and the verifier re-hashes. This lets you prove a response came from a specific compile without ever revealing the prompt that produced it. Spec at docs/receipt-spec.md.

Honest limits

What kolm doesn't do, and what we'd recommend instead.

If you came here for any of these, save yourself a phone call. We'd rather lose the deal than lose your trust.

Train a foundation model from scratch

Kolm specializes existing open models. It doesn't pretrain. If you need a brand-new foundation model from random init, look at the labs (Anthropic, OpenAI, Mistral) or roll your own with NeMo / Megatron.

Replace your RAG / vector DB

The compiler is a specializer, not a memory. If your job is "look stuff up in a private corpus and answer," pair kolm with a vector store (pgvector, Weaviate, LanceDB). The compiler then learns the answer shape your retrieval already supplies.

Vector-search your captures

We store captures for compile and replay, not for retrieval. There's no built-in "find me the 5 most similar past prompts" API. Use the export and index it in your vector store.

Run as an agent framework

Kolm is a model compiler. We're not LangChain / CrewAI / Autogen. You can run a kolm artifact inside any of those agent frameworks. We just don't provide the agent loop.

Host the open frontier models for you

The gateway routes to providers that host frontier models. We don't run our own pods serving Llama or Mixtral; that's Together, Fireworks, Groq, Modal. Pick one as your gateway upstream.

Magically make a 0.5B model as smart as Claude

Distillation has a ceiling. You can compress the part of a teacher that's relevant to your task; you can't compress the parts that aren't relevant or that the captures don't cover. Bake-off tells you the gap before you ship.

Promise zero hallucination

No compiler can. We add receipts so you can audit and reproduce hallucinations, and frozen evals so you can detect regressions. Those reduce the surprise; they don't eliminate the failure mode.

Be the only AI tool you need

Most teams pair kolm with one frontier API (frontier work) and one cloud GPU host (heavy serving). The kolm value is "the specialist runs on your hardware after compile," not "ditch every other vendor."

For data scientists

The training method, in plain language.

Kolm uses supervised fine-tuning on teacher responses, with LoRA by default, with a frozen held-out eval, with optional DPO and KTO post-passes for preference alignment. /training is the full method spec; /k-score is the eval method. Nothing here is novel; the value is that all of it is wired together and reproducible from the capture log.

Step a
Seed split: 90 / 10, deterministic

Captures are hashed and split by hash mod 10. Same captures + same seed = bit-identical split. The eval set is held out forever; it never moves into train, even on retrain.

Step b
SFT on teacher response

Standard supervised fine-tune. Loss is next-token cross-entropy on the teacher's output, conditioned on the input. LoRA rank 16 by default; full fine-tune available with --full-ft.

Step c
Optional DPO / KTO pass

If captures have approve / reject labels, kolm runs Direct Preference Optimization (Rafailov 2023) or KTO (Ethayarajh 2024) on the preference pairs. Off by default; opt in with --align dpo or --align kto.

Step d
Quantize with verified roundtrip

After fine-tune, kolm quantizes the merged weights to INT4 (NF4 + double) or INT8 and re-runs the frozen eval. The Jaccard drop from BF16 to INT4 is included in the compile receipt.

Step e
Bake-off vs teacher

On the held-out eval, kolm scores the student's response vs the teacher's. Two scores: token Jaccard (fast, cheap) and judge win/tie (a stronger model picks the better answer). Both go in the receipt.

Step f
Seal + sign

Weights, spec, eval, receipts, bake-off go into one .kolm, hashed, signed with Ed25519, registered. From the user's perspective: one command, one file out.

A scoring run that compares an artifact's responses to its teacher's on a held-out eval. Two scores: token Jaccard and judge win/tie.
capture
One row in the per-tenant log. Each row is one approved request/response, hashed and chain-linked to the prior row.
cid
Content identifier. ks_ prefix + sha256 of the canonical manifest. Lets a third party fetch + verify any artifact.
compile
The single command that distills, quantizes, evals, and seals. Input: a spec. Output: a .kolm.
distill
Train a smaller student against a teacher's responses on your captures. Output: a fine-tuned checkpoint, ready to quantize and seal.
drift
The condition where a deployed artifact's live K-score drops more than 10 points below baseline for 3 consecutive windows.
Ed25519
The signature scheme on the artifact sidecar. 32-byte keys, 64-byte sigs, constant-time, no parameter pitfalls.
frozen eval
A subset of captures pinned at compile time. The same eval rows produce the same K-score across artifact versions, so changes can be attributed to the model, not the data.
gateway
The OpenAI / Anthropic compatible proxy. Routes, redacts, logs approvals. Drop-in replacement for OpenRouter.
K-score
A composite quality score per artifact. Held-out Jaccard + judge win-rate + refusal rate, normalized 0-100. Frozen at compile, recomputed live for drift detection.
kolm-audit-1
The receipt envelope. Carries artifact_hash, signing key, input_hash, response_hash, and a kolm-1 wrapper signature.
namespace
A logical bucket per use case or app surface, scoped inside a tenant. E.g. support, onboarding, extract.
NF4
Normalized 4-bit float. The default INT4 format. Keeps the distribution Gaussian after quantization, matching the natural weight distribution.
receipt
A signed envelope emitted with every response. Lets a third party verify "this response came from this artifact" without seeing the prompt.
registry
The public, queryable, signed index of every published compile. Public mirror + per-tenant private mirror on Business tier.
spec
A .kolmspec file. Pins the teacher, student, training params, eval set, and quantization for a compile. Reproducible: same spec + same captures = bit-identical artifact.
student
The smaller open model that learns from a teacher. Currently 13 supported: Qwen2.5 0.5B / 3B / 7B / 14B / 32B, Llama-3 8B / 70B, Mistral-7B, Mixtral-8x7B, Gemma-2 9B / 27B, Phi-3 mini, Phi-3 medium.
teacher
The larger (usually closed) model whose responses train the student. Currently supported: Anthropic, OpenAI, vLLM, SGLang, TRT-LLM, TGI, Fireworks, Together, Groq, your own.
tenant
The top-level container. One tenant per company. Captures, datasets, compiles, registry rows, billing all fenced inside a tenant.

FAQ

The questions everyone asks first.

Is kolm a replacement for OpenAI?
No. kolm is in front of OpenAI (and Anthropic, vLLM, SGLang, TRT-LLM, etc.). You keep using the SDK you already use. We route, log approvals, and let you compile those approved calls into a smaller model you own. After compile, you don't have to pay per token anymore.
What is a .kolm artifact?
A single sealed file that contains the model weights, the spec it was compiled against, the seed split it trained on, the teacher receipts, and an Ed25519 signature binding them all. You can verify any response a .kolm produces at /verify-prod.
Where does the data live? Can I self-host?
Every capture, dataset, and artifact is tenant-isolated and tenant-fenced. Your data never crosses to another tenant. You can run the whole kolm stack on your own hardware (BYOC, VPC, on-prem, fully air-gapped) under Apache-2.0. See /self-host.
How real are the benchmarks on this page?
Run them yourself. The DeepSeek-R1 32B → 17.9 GB INT4 in 125s number was measured May 2026 on an RTX 5090 with bitsandbytes 0.49.2 (NF4 + double quant) and torch 2.11.0+cu128. The transcripts, including the snail-on-a-wall chain-of-thought, are linked from /benchmarks.
What about HIPAA, SOC 2, regulated workloads?
SOC 2 Type I is available today. HIPAA mapping (/hipaa-mapping) and a signed BAA (/baa) ship on Business. We have explicit doors for healthcare, finance, legal, defense, and EU/sovereign deployments. PII redaction (text, image, audio) is on by default at the gateway.
Will my model drift after I deploy it?
Every compile carries a K-score. The runtime computes a fresh K-score from production responses and alerts when it drops. /drift shows the recipe: snapshot at compile, detect on every batch, recompile from fresh captures when the verdict moves out of "within."
How do I integrate kolm into an existing CI/CD pipeline?
There's a first-party GitHub Action (kolm-ai/kolm-action) and a GitLab template. The compile step runs in the pipeline against the captures committed to the repo, the resulting .kolm is uploaded as a release artifact, and the Ed25519 signature is checked by the runtime on download. Examples at /docs/ci-cd.
Can I use my own teacher model?
Yes. Any OpenAI-compatible or Anthropic-compatible endpoint can be a teacher: a hosted vLLM, an SGLang deployment, a TRT-LLM server, an Ollama on a workstation, or your own internal proxy. Set --teacher http://your.endpoint/v1 and kolm will call it during compile. Anthropic-shape endpoints work too with --teacher-protocol anthropic.
What students do you support?
As of May 2026: Qwen2.5 0.5B, 3B, 7B, 14B, 32B; Llama-3 8B and 70B; Mistral-7B; Mixtral-8x7B; Gemma-2 9B and 27B; Phi-3 mini and medium. Adding a new student is a config entry and a tokenizer template; happy to add yours if the weights are publicly licensed.
How big is the .kolm for a 7B model?
5.2 GB at INT4 (NF4 + double), 8.1 GB at INT8, 14.2 GB at BF16. The artifact format is one zstd-compressed file with the weights at level 19, plus a few KB of spec, eval, receipts, and Ed25519 sidecar. Inspection of any artifact: kolm inspect file.kolm.
What's the smallest server that can run a 7B INT4?
A single A10G (24 GB VRAM) handles 4 concurrent requests at ~22 tok/s each, drawing about 130 W under load. A laptop with 16 GB RAM runs the same artifact at ~9 tok/s via llama.cpp. We've also run it on a Raspberry Pi 5 (16 GB) at ~2 tok/s, mostly for the meme.
How does kolm handle prompt injection or jailbreaks?
The gateway has a configurable injection filter (regex + ML classifier) you can turn on per-namespace. The compiler keeps the teacher's refusal behavior intact during distill, so a specialist doesn't lose its safety floor unless you specifically train it to. Full safety guide at /safety.
Can I expose a .kolm as an OpenAI-compatible server?
Yes. kolm serve artifact.kolm --port 8080 exposes a POST /v1/chat/completions endpoint that speaks the OpenAI Chat Completions shape. Drop-in for any client that points at OPENAI_BASE_URL. Streaming, tool-use, and JSON-mode all supported.
Does kolm support tool use and function calling?
Yes for runtime; OpenAI tools and Anthropic tool_use shapes both pass through the gateway and are emitted by kolm serve. For distill, you can include tool calls in the captures; the student learns to emit the same tool calls in the same shape. Bake-off scores tool-call-shape match separately from text match.
What's the upgrade path for an artifact when the format evolves?
Artifacts carry a version pin (version: kolm-1). The runtime supports current + previous major. When we bump to kolm-2, the CLI ships a kolm migrate verb that rewraps the weights into the new shell without re-training. Older artifacts keep working; new ones get new features.
How do I export my data if I leave kolm?
kolm account export writes a JSONL of every capture, every artifact, every receipt, every registry row, and every audit-log entry. Apache-2.0 license + open artifact format means the data is yours forever; you can rebuild the same compile elsewhere from the export. We make leaving easy on purpose.
What about RAG? Vector stores? Embeddings?
Kolm doesn't ship a vector store. The compiler is for "learn to answer my domain"; RAG is for "look up facts in my corpus." They compose: put the kolm artifact behind your RAG pipeline (LangChain, LlamaIndex, your own), so the retrieved chunks feed into a model that's already specialized to your shape. We pair well with pgvector, Weaviate, LanceDB, and Pinecone.
What happens if the gateway is down? What's the fallback?
The SDK has a built-in fallback list. If api.kolm.ai is unreachable, it can either fail closed (default for compile, doctor) or fall back to a configured direct provider (default for runtime). You can also self-host the gateway in your VPC and use the hosted one only as a backup. Status: /status.
Why "kolm"? What does it mean?
Short for Kolmogorov, after the mathematician who introduced algorithmic complexity (Kolmogorov complexity, 1965): the shortest program that produces a given output. Distillation is the same idea: compress the parts of a frontier model that matter for your job, and discard the rest.
Is kolm a single founder side project?
No. Kolm is a small team (currently 9 engineers + 1 designer + 1 GTM) backed by lead investors who let us publish a public roadmap and a public test count. We've shipped weekly since Wave 144 (December 2024) and write a public changelog. Headcount and funding details at /about.
What if I just want the gateway, not the compiler?
Use it. The gateway alone is a strong OpenRouter replacement: failover, edge redaction, OpenAI/Anthropic compat, per-namespace metering. You can ignore captures and compile entirely. The pricing tiers stay the same; you just don't use most of the surface.
What if I just want to verify someone else's .kolm?
Free, no signup. curl https://kolm.ai/v1/verify/<cid> returns the registry row. kolm verify file.kolm verifies an artifact locally. kolm verify --receipt <receipt.json> verifies a response. All three work without an API key.
Does kolm work with my model serving framework?
If your framework speaks OpenAI Chat Completions or Anthropic Messages, yes. Tested: vLLM, SGLang, TGI, TRT-LLM, Fireworks, Together, Groq, llama.cpp server, Ollama, Replicate, LM Studio. If yours isn't here, tell us and we'll add an adapter.
How is this different from "fine-tune a small model"?
Fine-tuning is one step of compile. Kolm wraps capture (where do the training rows come from?), distill (which student? which teacher? what loss?), quantize (verified roundtrip), eval (frozen at compile), package (one file), sign (Ed25519), and verify (registry + receipt). The value is the wiring, not any single piece.
Can I run kolm on Windows?
Yes. The CLI ships as a signed Windows binary (winget + direct download). Compile-on-Windows works with WSL or with native CUDA for NVIDIA GPUs. The runtime works in native PowerShell. Air-gap install for Windows-only environments is on the roadmap; today, Topology D is Linux-first.
How do I cancel my plan?
Dashboard: Settings → Plan → Cancel. CLI: kolm billing cancel. No phone tree, no retention specialist. Cancellation takes effect at the end of the current billing period; export your data first if you want it.
Principles

Six rules we hold ourselves to.

If a feature breaks one of these without a written exception in the changelog, it's a bug. Read them, and if we ever ship something that contradicts one, file an issue.

Principle 01
Receipts over claims

Every benchmark on our site links to a transcript. Every uptime number links to the status page. Every compliance claim links to evidence. If we say a thing, we make it verifiable.

Principle 02
Open by default

Apache-2.0 license, end-to-end. Open artifact format. Open SDKs. Open registry protocol. If we hide a piece, we say why on this page (today: nothing).

Principle 03
Tenants are sacred

Per-row fence with per-loop re-check. No "cross-tenant features." No "support engineer can browse customer data." Defense-in-depth, not just key indexing.

Principle 04
One artifact, every backend

The artifact is the contract. If a new backend can't load the existing format, we add the backend; we don't fork the format. New format versions ship with a migrate verb so old artifacts keep working.

Principle 05
Honest 503 over silent fallback

If a feature needs a dependency you don't have, we return 503 with an install hint. We never silently degrade or pretend a feature works when it doesn't. The redactor docs are the model: install missing means redaction missing means 503.

Principle 06
Make leaving easy

kolm account export writes everything. The artifact format is open and documented. The whole stack is Apache-2.0 and self-hostable. If you want to leave, we'd rather you leave on good terms than stay because of a lock-in.

Caught us breaking a principle?

Quickstart preview

Ten minutes from brew install to running .kolm.

The full /quickstart walks each step. Here is the shape so you know what you're committing to.

Minute 0 to 2 · Install
# macOS / Linux
$ curl -fsSL https://kolm.ai/install.sh | bash
$ kolm --version
  kolm 0.6.1 (2026-05-22)

# Windows
$ winget install kolm.kolm
$ kolm --version
Minute 2 to 4 · Sign up
$ kolm signup --email you@example.com
   tenant created
   key saved to ~/.kolm/auth
   free tier active

$ kolm doctor
   all checks green
Minute 4 to 8 · Capture & compile
$ export OPENAI_BASE_URL=https://api.kolm.ai/v1
$ # ... run your app, generate captures ...
$ kolm distill from-captures \
   --namespace demo --teacher gpt-4o \
   --student qwen2.5-3b --quantize int4
   sealed demo-2026-05-22.kolm
Minute 8 to 10 · Run
$ kolm run demo-2026-05-22.kolm \
   "summarize this in one sentence: ..."
  → Done. (28 tok/s on M2 Pro, 2.4 GB RAM)
  → receipt: ./last.receipt.json
$ kolm verify --receipt last.receipt.json
   response matches signed artifact

What "real" looks like in 10 minutes

At minute 10 you have a 1.9 GB file that does your specific job, an Ed25519 signature that proves it does, and a receipt that proves a specific response came from it. You can email the file to a colleague, drop it on a USB stick, or check it into a private git repo, and they can verify everything you just did.

The frontier model in the example above (GPT-4o) is a "teacher"; you only call it during the compile step, on a few thousand sampled prompts. After compile, every inference is local: no per-token cost, no rate limit, no vendor lock.

What "real" doesn't include

The demo above uses a small (3B) student to keep the compile fast (12 to 14 minutes on a laptop CPU; 90 seconds on a GPU). For production support copilots, you'd pick a 7B or 14B student and compile on a real GPU. The CLI is the same.

The demo uses the hosted gateway. If you'd rather route to a self-hosted gateway from minute 1, the /self-host guide gets you there in another 10 minutes.

Trust, in depth

What we do, what we don't do, and how to check.

The shortest possible version of "is kolm safe to deploy." Every claim links to evidence at /security.

Data residency
Where your bytes physically live

Topology A: US (Vercel + Cloudflare R2). Topology B / D: wherever you deploy. EU-only and APAC-only regions available on Business; FedRAMP-Moderate planned 2027.

Encryption
TLS 1.3 in flight, AES-256 at rest

All gateway, registry, and dashboard traffic on TLS 1.3 (no fallback). Captures and artifacts encrypted at rest with AES-256-GCM. BYO-KMS available on Business.

Tenant fence
Per-row, defense-in-depth

Every row in every table is keyed by tenant_id. Inside loops, we re-check the tenant_id on each read. A bug in the index can't cross a tenant boundary. Audit: src/event-store.js.

Auth
API key, SSO (Team+), SCIM (Team+)

API keys are 64-char hex prefixed ks_. SSO via OIDC or SAML 2.0 on Team and Business. SCIM 2.0 provisioning on Team and Business. WebAuthn second factor for admins.

Audit log
Hash-chained, exportable, tamper-evident

Every privileged action emits an audit row chained to the prior row. Export to JSONL, Splunk, Datadog, BigQuery. 7-year retention on Business; 90 days on Pro; 30 days on Free.

Secrets
Never logged, never shown, never echoed

API keys hashed in storage (Argon2id); shown once on signup, never echoed. Customer-side secrets in prompts get edge-redacted before any log line is written. BYO-KMS-wrapped key on Business.

Penetration testing
Trail of Bits, annually

Last engagement: March 2026 (Trail of Bits). Summary public at /security; full report under NDA on request. Two findings, both medium, both closed within 30 days.

Vuln disclosure
90 days, bounties up to $25k

Report at security@kolm.ai (PGP key on file). Coordinated disclosure, 90-day deadline. Critical RCE bounty up to $25k; full table at /security#bounty.

Incident response
