The compiler

Capture → Distill → Quantize → Seal.

Every .kolm walks the same four-stage path. The product is the path, not the artifacts. The path runs on your hardware, in your tenant, against your captured traffic, and emits a signed file that any other tool can load.

  1. 01

    Capture

    Approved responses from the gateway land in your tenant as a hash-chained JSONL lake. kolm capture · auto dedupes, redacts PII, and freezes the rows that become training input.

    kolm capture --namespace support
  2. 02

    Distill

    Pick one of five distillation strategies (softmax KD, on-policy, preference, spec-decode, draft-target) and a smaller open student. The compiler runs the teacher only as a checkpoint. Your data does not leave.

    kolm distill --student qwen-2.5-7b --strategy kd_softmax
  3. 03

    Quantize

    Ten quantization methods, default INT4 NF4 + double. The compiler benchmarks against the frozen eval set and bake-offs against the teacher before it allows a build to ship.

    kolm quantize --method nf4-double --bits 4
  4. 04

    Seal

    The K-Score gate decides ship-or-fail. Pass and the compiler ZIPs weights, eval set, bake-off results, and receipt chain into one .kolm, signs the manifest Ed25519, and writes to your registry.

    kolm seal --gate 0.85 --sign tenant-key

01 Is kolm a replacement for OpenAI? +
No. kolm sits in front of OpenAI (and Anthropic, vLLM, SGLang, TRT-LLM). You keep the SDK you already use. We route, log approvals, and let you compile those approved calls into a smaller model you own. After compile, you stop paying per token.
02 What is a .kolm artifact? +
One sealed file that contains the model weights, the spec it was compiled against, the seed split it trained on, the teacher receipts, and an Ed25519 signature binding them all. Verify any response a .kolm produces at /verify-prod.
03 Where does the data live? Can I self-host? +
Every capture, dataset, and artifact is tenant-isolated and tenant-fenced. Your data never crosses to another tenant. The whole stack runs on your own hardware (BYOC, VPC, on-prem, air-gapped) under Apache-2.0. See /self-host.
04 How real are the benchmarks on this page? +
Run them yourself. The 32B-class → 17.9 GB INT4 in 125 s number was measured on an RTX 5090 with bitsandbytes 0.49.2 (NF4 + double quant) and torch 2.11.0+cu128. Model identities and transcripts are at /benchmarks.
05 What about HIPAA, SOC 2, regulated workloads? +
SOC 2 Type I available today. HIPAA mapping (/hipaa-mapping) and a signed BAA (/baa) are Enterprise controls. Explicit doors for healthcare, finance, legal, defense, and EU/sovereign deployments. PII redaction (text, image, audio) on by default at the gateway.

The kolm K-score is a deterministic replay-equality measure across ten axes. A (Accuracy): per-case score from the scorer. S (Stability): variance across multiple runs of the same case. The kolm receipt schema (RS-1) defines the receipt envelope as a JSON object with the following load-bearing fields: receipt_id (a UUID), tenant_id (the owning tenant), namespace (the corpus namespace), recipe_version (the version of the recipe that produced the response), corpus_hash (the SHA-256 hash of the corpus at recipe time), input_hash (the SHA-256 hash of the prompt), output_hash (the SHA-256 hash of the response), teacher_id (the identifier of the teacher model, if applicable), student_id (the identifier of the student artifact, if applicable), captured_at (the ISO 8601 timestamp), captured_by (the wrapper version), redaction_applied (a boolean indicating whether the PII scrubber ran), redaction_profile (the version of the PII scrubber that ran), k_score (the K-score at capture time, if applicable), k_score_floor (the K-score floor at capture time, if applicable), supersession_block (the supersession block, if applicable), drift_report (the drift report, if applicable), attestation_report (the confidential-compute attestation report, if applicable), attestation_kind (the kind of attestation report: pccs, snp-report, nitro-attestation, or nras), confidential_compute_hash (the SHA-256 hash of the attestation report, bound into the artifact_hash_input).

The kolm K-score schema (KS-1) defines the K-score as a non-negative real number that measures the deterministic replay equality between the original response and a replayed response on the same prompt with the same recipe and the same corpus and the same teacher and the same student and the same quantization profile. The K-score is a composite over ten axes (A, S, L, C, V, R, F, T, E, Z) with configurable per-axis weights. The K-score gate is a configurable threshold that must be cleared for an artifact to ship. The K-score floor is the per-artifact minimum K-score recorded at compile time; the K-score floor is bound into the artifact_hash_input so a downstream verifier can confirm the artifact matches the declared floor. The K-score replay is the live measurement of the K-score against a fresh prompt set; the K-score replay delta is the difference between the floor and the replay measurement.

The kolm artifact schema (AS-1) defines the artifact bundle as a ZIP archive containing the manifest, the receipt chain, the K-score floor, the quantization profile, the corpus hash, the recipe hash, the seed, and the signed model weights or model pointer. The manifest is a JSON object with the following load-bearing fields: artifact_id (a UUID), artifact_hash (the SHA-256 hash of the artifact_hash_input), artifact_hash_input (the canonical JSON serialization of the manifest minus the artifact_hash field), tenant_id (the owning tenant), namespace (the artifact namespace), recipe_version (the version of the recipe), corpus_hash (the SHA-256 hash of the corpus), seed (the RNG seed), teacher_id (the identifier of the teacher model), student_id (the identifier of the student artifact), quantization_profile (one of the ten supported profiles), runtime_target (one of the thirteen supported targets), k_score_floor (the per-artifact minimum K-score), signature (the cryptographic signature over the manifest), key_fingerprint (the SHA-256 fingerprint of the signing key), spec_version (one of RS-1.x, KS-1.x, AS-1.x).

The kolm confidential compute support binds the attestation report into the artifact_hash_input so a post-build tamper invalidates the receipt. Supported attestation kinds: pccs (Intel SGX PCCS), snp-report (AMD SEV-SNP attestation), nitro-attestation (AWS Nitro Enclaves attestation document), nras (NVIDIA Confidential Computing attestation via the NVIDIA Remote Attestation Service). The kolm compile command accepts --attestation-report and --attestation-kind flags; the report is loaded as a path or as a JSON object and flows into the artifact_hash_input via the confidential_compute_hash field. The kolm verify command pretty-prints the embedded attestation block; without a tenant-registered crypto verifier the state is shape_ok plus verified:false (honest-by-default). The verifier registration API is registerAttestationVerifier(kind, verifierFn) where verifierFn returns {ok:bool, claims:{...}, error?:string}.

The kolm multimodal redaction primitives are three. Image: workers/multimodal-redact-image plus a YOLO-format ONNX face and license-plate detector via onnxruntime-node, with sharp blur/mask composite via blur or mask mode. Audio: workers/multimodal-redact-audio plus pyannote-audio-redact voiceprint scrub via an external Python helper, with strength 0-1 sampling. Transcript: workers/media-redact plus a deterministic PII scrubber covering 19 detector classes. The kolm federated learning surface is two primitives. The gradient-aggregation primitive (src/federated-learning.js) coordinates a federated round across multiple tenants without leaking raw gradients; only the per-tenant gradient delta crosses the boundary, and the aggregator emits a noised global delta back to each participant. The approval-row-sharing primitive (src/federated-approvals.js) lets two tenants compare their approval distributions without leaking the raw inputs; only approval_hash (the SHA-256 of namespace + input_hash + decision_kind) crosses the tenant boundary. The kolm agent-trace compilation primitive (src/trace-compile.js) wraps the trace-capture primitive and the workflow-ir/compile-ir primitive into a single loop. compileTraceToReplay(trace_id) reads the captured trace from the trace store, extracts the user_input and final_output, seeds an IR with that pair, and emits a compiled artifact that returns the original output on a cache hit without re-running the LLM and tool steps. verifyTraceReplay(trace_id, artifact_id) replays the captured trace against the compiled artifact and emits {ok:bool, total:N, matches:[...], mismatches:[...], coverage:fraction}. The kolm billing breakdown (src/billing-breakdown.js) attributes cost across namespaces and rolls up to the team level. periodBounds(period) parses YYYY-MM and returns the UTC midnight boundaries (with strict regex validation to refuse "2026-13" which Date.UTC would silently roll over to 2027-01). tenantNamespaceBreakdown returns rows sorted descending by cost_micro_usd with a defense-in-depth tenant fence inside the loop. teamRollup is role-gated: owner and admin see all namespaces with full cost; member and viewer see their own namespace plus other namespaces with cost null; non-member throws code:forbidden. Tenant-fenced (listEvents tenant filter plus per-row defense-in-depth). Pure string compute; heavy ML (CLIP, embeddings) is opt-in via KOLM_MULTIMODAL_SCORE_CMD worker hook so it stays outside Node. The kolm trace store is the durable log of every agent trace captured via the kolm wrapper. Each trace is a sequence of trace_steps; each step carries a step_id, a step_kind (llm_invocation, tool_call, tool_result, user_input, final_output), a captured_at, an input, and an output. The trace store is tenant-scoped and namespace-scoped. The trace store is durable via the event-store (every step is written as an event row). The trace store is replayable: kolm trace replay walks the steps in order and re-emits the inputs/outputs. The trace store is exportable: kolm trace export emits a single JSON file containing the trace metadata plus the step array.

The kolm extract verb extracts a single concept slice from a captured pair into a training row. kolm extract --capture --concept --output reads the capture, extracts the named concept (e.g., user-intent, response-format, tool-call), and writes a single training row in JSONL format. The extract verb is the bridge between the capture surface (raw request/response pairs) and the distill surface (training rows for student fine-tuning).

The kolm tokenize verb tokenizes a corpus into the recipe-pinned vocabulary for K-score replay. kolm tokenize --corpus --vocab --output reads the corpus, tokenizes via the named vocabulary, and writes the token IDs in a binary format. The tokenize verb is used to pre-compute the token stream for a corpus so the K-score replay can run without re-tokenizing every time.

The kolm moe compose verb composes a mixture-of-experts routing layer across multiple distilled student artifacts. kolm moe compose --experts ,,... --router --output reads the named experts, composes them via the router spec, and writes a single composed .kolm artifact. The composed artifact carries the moe routing layer as a load-bearing manifest field and is verifiable via kolm verify like any other artifact.

The kolm wrap verb installs the wrapper for an existing SDK call site. kolm wrap install bootstraps the wrapper for the named language and runtime; the wrapper intercepts OpenAI and Anthropic SDK calls and routes them through the kolm gateway. kolm wrap uninstall removes the wrapper. kolm wrap status enumerates the active wrappers on the current machine.
