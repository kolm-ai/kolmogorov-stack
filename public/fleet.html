Fleet management

One CLI. One .kolm. Every device on the rollout you choose.

kolm fleet deploy out.kolm --rolling 10% pushes to 10% of the fleet, watches success rates, advances to 25 / 50 / 100 — or rolls back automatically on the first device that regresses past your K-Score floor.

Caveats: rolling and canary deploys require devices registered via kolm devices add. SSH, agent, and MQTT transports supported. No agent daemon required for SSH mode.
The six verbs

Everything the CLI does for fleet, top to bottom.

The fleet is a flat list of device IDs with tags. Each verb takes --all, --tag <name>, or an explicit device list.

Inventory

kolm fleet status

Lists every device, current .kolm SHA, last heartbeat, K-Score on the most recent benchmark, and reachable status. JSON with --json.

kolm fleet status --tag prod
Rolling deploy

kolm fleet deploy --rolling

10% → 25% → 50% → 100% with a wait window between stages. Configurable per-stage success threshold. Stops on first device-class failure.

kolm fleet deploy out.kolm --rolling 10%
Canary

kolm fleet deploy --canary

Picks N tagged canary devices, runs them at the new build for a soak window, then promotes to fleet only if K-Score holds. --canary 3 --soak 24h typical.

kolm fleet deploy out.kolm --canary 3 --soak 24h
Watch

kolm fleet monitor

Streams per-device K-Score, latency, and error counts. Use during a rollout, or tail continuously for ongoing health. --json emits per-tick events.

kolm fleet monitor --tag prod --since 1h
Rollback

kolm fleet rollback

Reverts every device to the previous .kolm SHA recorded in the fleet ledger. Survives partial rollouts — devices on the new build get reverted, devices on the old build are skipped.

kolm fleet rollback --tag prod
Stop

kolm fleet stop

Halts an in-progress rolling deploy at the current stage. The fleet ledger records the partial state so a subsequent kolm fleet deploy --resume picks up where it stopped.

kolm fleet stop --tag prod
A rollout, end to end

Compile once. Canary three. Soak. Promote. Rollback if it ever slips.

This is the recipe one team uses to push a 7B specialist to 1,200 retail edge boxes every two weeks.

1

Compile + sign

in: distilled model · out: signed .kolm

kolm compile model/ --target-profile jetson-orin-nx -o out.kolm

2

Canary three devices

in: out.kolm · out: soak ledger entry

kolm fleet deploy out.kolm --canary 3 --soak 24h

3

Monitor the soak

in: fleet · out: live K-Score stream

kolm fleet monitor --tag canary --since 24h

4

Promote to rolling

in: passed canary · out: 10/25/50/100 rollout

kolm fleet deploy out.kolm --rolling 10% --gate k-score:0.92

5

Rollback on regression

in: failed gate · out: previous SHA on every device

kolm fleet rollback --tag prod (or automatic when --gate trips)

6

Receipts for audit

in: deploy id · out: signed deploy receipt

kolm verify deploy_01KYZ... — who pushed which SHA when to which device.

What the gate watches

A K-Score floor + p95 latency cap + error-rate ceiling.

The gate is what makes the rollout safe. Three numbers per device, evaluated every stage.

K-Score floor
--gate k-score:0.92

Min on-policy quality on your eval set. Computed from the most recent kolm test recorded for the device.

Latency cap
--gate p95-ms:800

p95 ms-per-request across the soak window. Trips on the first device that exceeds.

Error rate
--gate error-rate:0.01

Errors per 100 requests. Bounded against the device's baseline before deploy, not against zero.

Custom
--gate <name>:<op>:<value>

Plug in your own metric. The fleet ledger stores whatever kolm gate eval emits, so any custom check fits the same shape.
