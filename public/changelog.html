02 / Verification Gates

Release confidence comes from checks that target the actual failure modes.

A green page load is not enough. The release path checks routing, stale copy, broken references, claim scope, kernel readiness, and real screenshots.

Site tests

Routes render the intended pages

Direct route fetches assert titles and current product copy for trust, security, enterprise, contact, platform, capabilities, research, and changelog.

Rewrite tests

Aliases cannot hide stale pages

Vercel rewrite assertions prevent product surfaces from resolving to the generic compiler-product page by accident.

Stale-copy tests

Old packaging cannot return silently

Forbidden copy searches block old scan, package, pricing, verifier, overclaiming, and mojibake fragments from rebuilt pages.

Reference lint

Static links and assets resolve

The refs audit checks internal links, images, manifests, docs paths, public JSON, and route references across the static site.

Claim scope

Marketing cannot outrun proof

Claim verification keeps certification, benchmark, package, partner, and adoption statements tied to explicit evidence or open gates.

UI screenshots

Desktop and mobile must be readable

Full-page screenshots check spacing, stacking, footer integrity, hero density, and absence of smushing across desktop and mobile viewports.
