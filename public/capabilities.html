01 · Capability domains

Eight domains cover the full behavior-to-artifact path.

A capability is useful only when it changes the state of a real product object and leaves evidence behind.

Reg · 1 capture

Bring real behavior in

Register sources, ingest provider logs, gateway events, app traces, agent tool calls, labels, failures, warehouse rows, CI events, and custom API records.

Reg · 2 classify

Make data safe to reuse

Track source, schema state, tenant, environment, sensitivity, retention window, owner, and unknown-schema status before data moves downstream.

Reg · 3 govern

Apply policy before action

Enforce redaction, retention, purge, provider routing, model allowlists, export boundaries, scoped keys, budgets, and rate limits.

Reg · 4 evaluate

Turn failures into gates

Create evals, threshold checks, drift checks, human review queues, failure taxonomy, locked regression sets, and release-blocking gate state.

Reg · 5 compile

Build a portable output

Emit compile run, signed artifact, manifest, target recipe, source hash, output hash, approval state, and compatibility metadata.

Reg · 6 run

Target real environments

Support hosted, local, browser, edge, BYOC, offline, and restricted fleet runtime decisions with explicit readiness and limits.

Reg · 7 verify

Prove what was shipped

Attach verifier receipt, checksum, artifact version, policy version, eval result, runtime target, and open-gate caveats to the release path.

Reg · 8 operate

Keep control after launch

Expose audit events, admin analytics, connector health, cost controls, usage controls, exports, owner assignments, and incident-ready logs.

03 · Capability contract

Each capability carries the same proof contract.

This keeps product, API, backend, and website copy aligned. A capability is not done because it has a label; it is done when the object lifecycle can be inspected.

  • Input · what data enters, which tenant owns it, which environment, and how source provenance is retained
  • Control · which policy, role, retention, redaction, eval, budget, and release gate governs the action
  • Output · which artifact, report, receipt, export, runtime target, or audit event is created
  • Proof · which checksum, receipt ID, object version, route, test, or generated payload proves the claim

04 · Enterprise capabilities

The control center is where capabilities become operational.

Enterprise buyers need controls they can delegate, audit, export, and revoke. This is where Kolm outclasses generic AI tools.

Reg · identity

API keys and access

Create, scope, rotate, expire, and revoke API keys and service accounts by tenant, workspace, project, environment, and connector.

Reg · source

Connector governance

Track source health, connector owner, allowed channel family, ingress mode, retention window, redaction rule, and export scope.

Reg · policy

Model and provider policy

Control provider allowlists, model routing, cost budgets, rate limits, data-use boundaries, runtime compatibility, and fallback behavior.

Reg · export

Evidence destinations

Manage webhooks, warehouse exports, SIEM/GRC events, CI status, admin analytics, evidence bundles, and verifier receipts.

Reg · trust

Audit and readiness

Every write emits an audit event, every claim-bearing object has proof, and every unsupported external claim remains in the readiness ledger.

05 · Buyer map

Capabilities map to owners, not generic feature names.

This is the practical product spec: each owner gets the controls and evidence needed to approve the workflow.

Reg · connect

Platform owner

Needs connection and routing control. Sources, keys, environments, quotas, connector health, ingress modes, and runtime targets.

Reg · evaluate

AI owner

Needs behavior quality control. Traces, labels, evals, regression sets, drift checks, compile runs, and release gates.

Reg · govern

Security owner

Needs boundary control. Retention, redaction, unknown-schema handling, provider policy, access scope, audit events, and purge workflows.

Reg · export

Compliance owner

Needs proof export. Receipts, evidence bundles, object versions, audit trail, readiness status, and explicit open-gate caveats.

Reg · build

Developer

Needs implementation speed. API docs, OpenAPI, examples, test payloads, CLI path, clear errors, and repeatable local verification.
