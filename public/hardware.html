02 · why it matters

Quantize for what you have.

Most “will this fit?” questions get answered with a back-of-the-envelope params×bpp/8 calculation that ignores KV cache, activation memory, framework overhead, and the dtype your hardware actually supports natively. /v1/hardware + /v1/fit together answer the real question.

  • Native dtype list → pick the right kernel
  • Supported quant list → pick the right target
  • VRAM cap → reserve headroom for KV + activations
  • Vendor + driver → rule out incompatible toolchains
# the wrong way: guess.
$ python -c "print(7e9 * 4 / 8 / 1e9)"
3.5

# reality: 7B at 4bpp needs ~5.2 GB for weights *plus* KV.
# the right way: ask the box.
$ kolm hardware
device:   NVIDIA GeForce RTX 5090
vram:     32 GB
native:   bf16 fp8 nvfp4
supports: fp16 bf16 fp8 nvfp4 gguf-q8 ... gguf-q4km awq-4bit int4

$ kolm fit --params-b 7 --quant gguf-q4km --vram-gb 32
weights:  5.2 GB
kv:       3.4 GB  (context=8192)
total:    8.6 GB
headroom: +23.4 GB
fit:      fits comfortably

03 · targets

19 quant targets, ranked.

The probe also tells you the full ordered list of quant targets the host supports, from highest-quality (fp16) to most-aggressive (gguf-iq2xs). The bench operator at /account/bench runs every supported target for one base model and shows the recommended row — the highest-quality target that fits with headroom.

  • 19 targets in canonical quality order
  • Filtered to what this host actually supports
  • Bench iterates all of them in one pass
  • --pick-best picks the recommended row

04 · surfaces

One route, every face.

Just like merge and quantize, the hardware probe ships in all four surfaces. The CLI, the TUI, the Account dashboard, and the route all return the same payload — because they all call the same handler.

  • GET /v1/hardware — the route (public)
  • kolm hardware — the CLI verb
  • TUI :hardware (key H) — the live view
  • /account/hardware — the dashboard
