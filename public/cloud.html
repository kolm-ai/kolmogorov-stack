Cloud distill · W785 · managed training, local inference

Bring your captures. Get your .kolm artifact. Skip the GPUs.

You already have inference hardware. You do not have a rack of H100s for the LoRA pass. kolm cloud distill submit uploads a captured namespace, runs the distill on a managed pool, and returns a quantized .kolm you can verify, sign, and serve from your own runtime. Training meters per GPU-hour; inference stays on its own meter at $/1k-tokens.

Who this is for

If your operator can deploy a quantized model but cannot rationalize provisioning an H100 cluster for a one-time training pass, cloud distill removes the only step that needs the rack.

Edge & on-device shops

You ship inference on a customer's Mac Studio, RTX 4090 workstation, or a Cloudflare Worker. Training a 7B student on prem is overkill; one cloud distill run, then everything else stays local.

Regulated workloads with inference-only infra

Your inference runs inside the customer's VPC. Distillation happens against the captured-redacted namespace on managed infra; the resulting .kolm ships back signed for inference inside the VPC.

Solo operators

You have a serving box and a corpus. You don't have a training cluster. kolm cloud distill submit hands the GPU-bursty piece to managed infra and returns a deployable artifact.

Export, submit, download.

Export the captured namespace, submit the distill job, download the .kolm. Your local CLI never touches a training GPU; the managed pool never sees your inference traffic.

# 1. Export the captured rows you want to distill against
$ kolm capture export support-triage --since 7d --out support-triage.jsonl
  wrote 4812 rows · sha256 a3c1...

# 2. Submit the managed distill run
$ kolm cloud distill submit \
    --namespace support-triage \
    --capture-window 7d \
    --recipe support-distill-v1 \
    --gpu-sku H100-80GB \
    --vram-tier 1x

Honesty · queued vs running

When no managed pool is provisioned, cloud_backend_status returns no_pool_configured and the job sits in queued until an operator sets KOLM_CLOUD_DISTILL_ENDPOINT. We never simulate a finished run when no backend exists; kolm cloud distill status reports the truth. This is the same honesty contract as self-hosted distill and K-score reporting.

Pricing · training meter only

Cloud distill bills per GPU-hour. Inference traffic against the resulting .kolm is on the standard $/1k-token meter — the two ledgers never mix, so finance can split CapEx-ish training spend from OpEx inference spend without re-categorizing.

Job lifecycle

Terminal states (succeeded, failed, cancelled) cannot be re-entered; kolm cloud distill cancel is a no-op on terminal rows and returns invalid_transition.

QUEUED

Submitted, awaiting GPU

Job persisted to ~/.kolm/cloud-distill-jobs.jsonl. Meter row reserved at $0.

RUNNING

Backend has picked it up

Training loop streaming. Cancellable.

SUCCEEDED

.kolm available

artifact_url populated. Meter row finalized. Terminal.

FAILED

Training did not converge

error field populated with the gate reason. Meter row carries gpu_seconds spent. Terminal.

CANCELLED

Operator cancelled

Either explicit cloud distill cancel or backend-initiated. Meter row carries partial gpu_seconds. Terminal.

What stays local, what crosses the boundary

Captures cross the boundary going up; weights cross going down. Both are signed and verifiable. Inference NEVER crosses — the artifact runs on your runtime.

Goes to managed infra

Captured rows for the specified namespace + capture window, redacted by your local privacy membrane before upload. The recipe id (training recipe to apply). Billing token (not persisted on the worker).

Stays on your runtime

All inference traffic against the resulting .kolm. The artifact is signed Ed25519; you verify it with kolm verify <file>.kolm before serving. No phone-home at inference time.

Full CLI reference

Every verb is local-equivalent under kolm distill; kolm cloud distill just routes the GPU-bursty step off-box.

$ kolm cloud distill submit  [--namespace ns] [--capture-window 7d] \
                            [--recipe id] [--gpu-sku H100-80GB] [--vram-tier 1x]
$ kolm cloud distill status  <job_id>
$ kolm cloud distill list    [--status queued|running|succeeded|failed|cancelled] \
                            [--namespace ns] [--limit N]
$ kolm cloud distill cancel  <job_id> [--reason "text"]
$ kolm cloud distill meter   <job_id>
