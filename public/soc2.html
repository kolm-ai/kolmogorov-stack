Trust services criteria covered

We scope to the three criteria a regulated buyer actually needs from a compiler vendor. Processing Integrity and Privacy criteria are mapped but not in scope for Type I or the current Type II window; we will name the gap rather than imply coverage.

  • Security

    CC-series · in scope

    Common Criteria controls covering access management, change management, network protection, vulnerability management, and incident response for the hosted control plane and the open-source CLI release pipeline.

  • Availability

    A-series · in scope

    Capacity, backup, and recovery controls for the hosted registry, signing service, and verify endpoint. Customer-side compile workloads run on customer hardware and inherit the customer's availability posture.

  • Confidentiality

    C-series · in scope

    Controls covering data classification, retention, secure disposal, and key custody. Captures are tenant-scoped by namespace and the default deployment keeps them on customer hardware.

  • Processing Integrity

    PI-series · not in scope

    Not in scope for the current window. Verifiable processing integrity is delivered through the artifact receipt chain (Ed25519 signature, content-addressed CID, K-score axes) rather than as a SOC 2 criterion.

  • Privacy

    P-series · not in scope

    Not in scope for the current window. Privacy posture is documented separately in the DPA and the privacy statement; plain scope means we will not imply criterion coverage we do not have.

  • Receipt chain

    non-criterion · verifiable offline

    Every artifact ships with a signed receipt naming the teacher, base model, eval pack, and K-score verdict. An auditor can verify the chain offline without contacting kolm.ai; the same chain is referenced in the SOC 2 system description.

What is in the report

A SOC 2 reviewer expects four things; here is what the report carries and what we will say about each section before you ask.

  • Independent auditor's letter

    section i

    The CPA firm's opinion on the design (Type I) or design plus operating effectiveness (Type II) of the controls. Cited by name on the letter; firm name and engagement partner appear there rather than on this page.

  • Management's assertion

    section ii

    Our management's written statement that the system description fairly presents what we operate and that the controls are suitably designed for the criteria in scope.

  • System description and boundary diagram

    section iii

    The hosted control plane (signup, registry, signing service, verify endpoint), the open-source CLI release pipeline, and the supporting infrastructure. A boundary diagram shows what runs on our side versus on customer hardware in the default deployment.

  • Control descriptions and test results

    section iv

    Each control mapped to a trust services criterion, with the auditor's test procedure and the result. Any exceptions are listed verbatim; a real SOC 2 report shows the exceptions rather than hiding them.

How to request the report

Mutual NDA, then PDF over email. We do not gate the report behind a sales call; a compliance engineer can read it before talking to anyone in revenue.

  1. Email trust@kolm.ai with your company name, the audience for the report (your security team, an enterprise buyer, a downstream customer), and whether you need the Type I letter, a bridge letter, or both. Use the contact form if you prefer.
  2. Mutual NDA · we counter-sign a one-page mutual NDA the same business day. Your standard NDA is accepted with normal redlines; we do not require ours.
  3. Report delivery · PDF over email, watermarked with the requester organization and the date. Five-business-day turnaround for first-time requesters; same-day for repeat reviewers.
  4. Bridge letter · if you need coverage from the Type I audit date through today, we attach a bridge letter on request. The bridge letter names the period covered and any material control changes.
  5. Follow-up questions · a security engineer (not a salesperson) will answer questions about the report. We log questions and feed common ones back into the next system description.

Subprocessors

The list of third parties that process customer data on our behalf in the hosted control plane. In the default deployment, captures never leave the customer boundary; the only data that crosses to a subprocessor is the metadata required to operate the hosted control plane the customer chose to use.

CloudflareGlobal edgeEdge TLS termination and DDoS protection in front of kolm.ai. Customer data does not persist at the edge.
Anthropicus-east, us-westOptional hosted teacher path. Customers using their own teacher endpoint do not route any prompts through Anthropic; the default deployment does not invoke this path.
OpenAIus-east, us-westOptional hosted teacher path. Same opt-in posture as Anthropic; bring-your-own-key and self-hosted vLLM / SGLang / TGI keep traffic in your network.
Resendus-eastTransactional email for signup, password reset, BAA and SOC 2 request confirmations. No prompts or completions transit this path.
Stripeus-eastBilling and invoicing for paid plans. Free CLI and self-hosted deployments do not invoke this path.

In the default deployment, customer captures and weights stay on customer hardware. The hosted teacher subprocessors (Anthropic, OpenAI) are opt-in; the hosting subprocessors (AWS, GCP, Cloudflare) operate the kolm.ai control plane that issues API keys and serves the public verify endpoint. The plain scope: only the metadata the customer chose to put into the hosted control plane crosses the boundary.
