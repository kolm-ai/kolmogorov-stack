REST API reference · for developers, engineers, and security reviewers

The kolm /v1 surface. 708 operations, one honest envelope.

Every CLI verb, every SDK method, every integration adapter resolves to a route under /v1/*. Auth is a single x-kolm-api-key header. Every response carries the shape {ok, code, error, data}; HTTP status is never lied about. This page is the index. Group cards below link to per-surface deep references and code samples.

1auth header

The envelope, the auth header, the base URL

Three things to know before reading the rest of this page. The contract is intentionally small so a developer or security reviewer can audit it on first read.

Auth header
x-kolm-api-key: ks_4b7bc3b... on every request. Generated by kolm signup or kolm keys create. Tenant-scoped. No bearer prefix. Rotate with kolm keys rotate.
Base URL
Hosted: https://kolm.ai. Self-hosted: whatever you set KOLM_BASE_URL to inside the SDK or CLI. The path prefix /v1/* is identical in both modes.
Response envelope
Every success and every failure body uses {ok: boolean, code: string, error: string|null, data: any}. HTTP status mirrors ok: 2xx on success, 4xx for caller faults, 5xx for server faults.
Content type
Request: application/json. Response: application/json unless the route streams (NDJSON for log tails, SSE for completion). Streaming routes are flagged in their group below.
Versioning
Routes live under /v1/*. Breaking changes ship under a new prefix and the old surface continues to serve until announced retirement. No silent breaks.
Rate limits
Per-tenant, per-route. The hosted plan limits are in /pricing. Self-hosted installs set their own. Limit headers: x-ratelimit-remaining, x-ratelimit-reset.

Routes by surface

708 operations across 187 groups; the seven groups below cover the canonical surface a developer uses to ship a signed artifact. The full machine-readable index lives at /openapi.json and /api-routes.json.

auth8 routes

Auth and identity

Sign up, sign in, who am I, key management. The first call any client makes is /v1/whoami to confirm the API key is in scope.

  • GET/v1/whoami
  • Current tenant, plan, key fingerprint, logged-in flag.
  • POST/v1/signup
  • Public, rate-limited (10/IP/24h). Creates a tenant and returns an API key.
  • POST/v1/signin
  • Exchange email + magic-link for an API key. Used by kolm login.
  • POST/v1/keys/rotate
  • Issue a new API key and mark the prior one for retirement.
capture22 routes

Capture and gateway

Route a model call through the kolm gateway so the prompt, response, and metadata land in the local capture store. Captures are the input to distillation and to drift detection.

  • POST/v1/capture/log
  • Write a captured row. Tenant + namespace scoped. Honored by the CLI kolm capture log.
  • GET/v1/capture/list
  • List captured rows with cursor pagination. Filters: namespace, after, before, label.
  • POST/v1/capture/label
  • Attach a verifier-readable label to a row for the labeling queue.
  • DEL/v1/capture/delete
  • Right-to-delete. Removes raw rows and writes a tombstone receipt so auditors can prove the deletion.
distill31 routes

Distillation

Take captures and a teacher endpoint, train a student model. Resumable, tenant-fenced, teacher fallback supported.

  • POST/v1/distill/from-captures
  • Kick off a distill run against the local capture store. Returns a run_id and a progress URL.
  • POST/v1/distill/run
  • Lower-level entrypoint. Accepts an explicit dataset path, teacher id, base model id, hyperparameters.
  • GET/v1/distill/runs/:run_id
  • Status, step count, loss, attempted teachers, resume marker.
  • POST/v1/distill/cancel
  • Stop an in-flight run; partial artifacts retained under ~/.kolm/runs/.
compile27 routes

Compile and artifacts

Quantize a distilled candidate, package as a .kolm artifact, attach an attestation report, write a signed receipt. Output is a deterministic zip on your filesystem.

  • POST/v1/compile
  • Build the artifact. Accepts target (gguf, vllm, trt-llm, onnx, mlx) and precision (int4, int8, bf16).
  • GET/v1/artifacts
  • List artifacts for the tenant. Each row has CID, K-score, target, precision, signed_at.
  • GET/v1/artifacts/:cid
  • Fetch the receipt for a given CID. The artifact zip itself stays on your storage.
  • POST/v1/artifacts/sign
  • Re-sign an existing artifact with a rotated key. Receipt history preserved.
score19 routes

Score and eval packs

Run an eval pack against an artifact, compute the K-score, attach axes and weights to the receipt. The gate that decides ship-or-rebuild.

  • POST/v1/score
  • Score an artifact against an eval pack. Returns composite K-score and the five axes.
  • GET/v1/eval-packs
  • List installed eval packs. Each row has CID, signer, axis weights.
  • POST/v1/eval-packs/import
  • Import a signed eval pack from a URL or local file. Signature verified before install.
  • POST/v1/bakeoff/run
  • Score N artifacts against the same pack and pick a winner.
verify12 routes

Verify and registry

Resolve a content-addressed hash to a signed receipt. The C and Rust SDKs exist primarily so a regulated deployment can verify without a heavy runtime.

  • GET/v1/verify/:cid
  • Resolve a published receipt CID. Returns receipt JSON; signature checked server-side.
  • POST/v1/verify/artifact
  • Verify an uploaded .kolm artifact end-to-end: signature, receipt, eval pack hash, attestation.
  • POST/v1/registry/publish
  • Opt-in. Publish a receipt CID to the public registry. Weights and captures never leave.
  • DEL/v1/registry/takedown
  • Remove a CID from the public index. Downstream verifiers with the receipt continue to verify.
ops236+ routes

Operations, billing, integrations

The long tail: drift, bakeoff, billing breakdown, federated approvals, doctor, namespace management, audit log, attestation, and every first-party integration. The full surface is in the OpenAPI download.

  • GET/v1/drift/snapshot
  • Snapshot the current capture distribution; used to detect drift against the artifact training set.
  • GET/v1/billing/breakdown
  • Per-namespace cost attribution + team-level rollup.
  • GET/v1/doctor
  • Operational health: GPU, drivers, Python, Node, disk, credentials. Returns actionable hints.
  • POST/v1/multimodal/redact-image
  • Pixel-space face and license-plate scrub for captured images. Honest no_detector_installed envelope when deps missing.

Endpoint reference · the eight you will hit first

Every route in the catalog has the same shape. Below are the eight a new developer wires before anything else: provision, authenticate, capture, distill, compile, score, verify, and rotate. Body and response shapes are copy-paste ready. Every error is a stable string code, not a parsed message.

Error codes the surface returns

Stable string codes inside the envelope. Bindings and CLI scripts can branch on code without parsing the human error message.

HTTPcodeMeaning
401unauthorizedAPI key missing, malformed, or revoked. Run kolm login or rotate the key.
402plan_requiredRoute is gated to a higher plan (enterprise, business). See /pricing.
403tenant_mismatchThe resource belongs to another tenant. Always fails closed.
404not_foundCID, run, artifact, or capture id does not resolve in this tenant.
409idempotency_conflictReplay with a different body for the same idempotency key. Retry with a fresh key.
422validation_failedBody or query failed schema validation. data.errors[] names the offending fields.
429rate_limitedPer-tenant per-route ceiling. Respect x-ratelimit-reset.
500internalServer fault. The request id in data.request_id is what to send to support.
503not_yet_implementedSource-indexed route advertised but pending entitlement (e.g. SCIM SP config under SSO). Honest pre-ship signal.

End-to-end with nothing but curl

Six requests covering the canonical loop: sign up, capture, distill, compile, score, verify. Useful for a security reviewer or a CI script that does not want an SDK dependency.

Signed receipt from nothing but curl
# 1. Sign up (public, rate-limited)
curl -s -X POST https://kolm.ai/v1/signup \
  -H "content-type: application/json" \
  -d '{"email":"dev@example.com"}'

export KOLM_API_KEY=ks_4b7bc3b...

curl -s -X POST https://kolm.ai/v1/capture/log \
  -H "x-kolm-api-key: $KOLM_API_KEY" \

  -d '{"namespace":"claims","input":"...","output":"..."}'

# 3. Distill from captures
curl -s -X POST https://kolm.ai/v1/distill/from-captures \
  -H "x-kolm-api-key: $KOLM_API_KEY" \
  -d '{"namespace":"claims","base":"qwen2.5-3b-instruct"}'

# 4. Compile to a quantized .kolm artifact
curl -s -X POST https://kolm.ai/v1/compile \

curl -s https://kolm.ai/v1/verify/bafy2bzace...e8d0 | jq .data.k_score

Conventions across every route

Pagination, idempotency, webhooks, request ids. Same shape across every group so a developer or integrator can write one HTTP wrapper and reuse it for all 708 operations.

paginationcursor

Cursor pagination, not page numbers

Every list route returns data.items[] and data.next_cursor. Pass ?cursor=<value> on the next request. When next_cursor is null, the list is exhausted. Stable across writes; no skipped or duplicated rows.

idempotencyheader

Idempotency-Key on POST

Send Idempotency-Key: <uuid> on any POST that creates or mutates. Replaying the same key with the same body returns the original response. Replaying with a different body returns 409 idempotency_conflict. Keys retained for 24 hours.

request-idtrace

x-request-id on every response

A ULID returned in the x-request-id header and inside data.request_id. Send the same id back to support and the engineer can pull the matching trace from the OTLP stream without searching by timestamp.

webhookssigned

Webhooks for long-running jobs

Distill runs, compile jobs, and bake-offs emit completion webhooks. Payloads are JSON, signed with an HMAC of your webhook secret in the x-kolm-signature header. Replay-safe: every payload has a timestamp inside a 5-minute window.

streamingNDJSON + SSE

NDJSON for tails, SSE for completions

Log tails (/v1/distill/runs/:id/logs) stream NDJSON: one envelope per line, ready for jq -c. Hosted-completion routes stream Server-Sent Events with event: token and event: done frames; the envelope is the final frame.

namespacingheader

x-kolm-namespace for multi-app tenants

A single tenant typically runs many product surfaces. Every write route honors an x-kolm-namespace header (or a namespace body field) so captures, distill runs, artifacts, and billing all attribute cleanly. Default namespace is default.

Security posture on the REST surface

A security or compliance reviewer can audit the entire surface against a small list of guarantees. Below are the ones the bindings, the CLI, and the gateway all share.

tls1.2+

TLS 1.2 minimum, 1.3 preferred

The hosted gateway rejects TLS 1.1 and below. Cipher suites pinned to a hardened modern profile. kolm doctor reports the negotiated TLS version so a regulated install can verify the channel from the CLI.

scopeleast-priv

Tenant-scoped on every route

An API key resolves to exactly one tenant. Every read and every write goes through the same tenant filter; cross-tenant access returns 403 tenant_mismatch. Defense in depth: the filter runs both at the route boundary and inside the data layer.

piiredact

Captures and PII

Captures stay in ~/.kolm/events on your filesystem; the hosted control plane does not store them. For multimodal captures, /v1/multimodal/redact-image and /v1/multimodal/redact-audio apply pixel-space and voiceprint scrubs before any storage.

auditlog

Audit log on every state change

Sign-in, key rotation, namespace creation, distill kick-off, artifact publish, registry takedown. Every state change writes a row into ~/.kolm/audit.log (CLI side) and into the tenant audit ledger (server side). Both are tail-able into Splunk or Datadog.
