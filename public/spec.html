The .kolm spec.

The contract that makes one .kolm byte-identical to another given the same inputs, signed end-to-end, and runnable on any device that can read the format. Open, MIT-licensed, small enough to read in one sitting.

Signature scheme

RS-1 v0.1 is intentionally narrow on cryptography. One scheme is active in RS-1; the next scheme is tracked in the versioned roadmap.

VersionSchemeCoverageVerification
v0.1 · tenant-localHMAC-SHA256 over canonical JSONReceipt chain integrity inside one tenant boundaryIssuer, or a party that holds the shared tenant receipt secret
v0.2 · public distributionEd25519 over canonical JSONDetached public-key signature over manifest, receipt, and eval evidenceAnyone with the publisher's Ed25519 public key. No shared secret required.

Scroll horizontally →

The v0.1 chain remains reproducible and tamper-evident under the issuer-or-shared-secret trust model that fits a single org or a contractually bound pair. Ed25519 is the public distribution mode: a registry, marketplace, or partner network can verify an artifact without holding the issuer's tenant secret.

Detached signatures, deterministic canonical-JSON encoding, and key rotation through the registry are part of the public verifier contract.

RFC process

RS-1 is the floor; everything beyond gets a numbered RFC.

  • Open a discussion on GitHub Discussions. One paragraph: what changes, what breaks.
  • Draft a PR against public/docs/ with the new spec as rs-N.md and a JSON Schema if applicable. Mark the prose as draft in the front matter.
  • Iterate until the breaking-change column is empty or unanimous. Two reviewers sign off.
  • Promote by removing the draft marker and bumping the registry signature key version. The reproducibility floor (§6 of RS-1) holds across the bump.

If you ship a runtime that implements an RFC before it lands, that is fine. Flag the artifact with "spec": "rs-N-rcM" so legacy runtimes refuse it cleanly.

Why a spec at all

A signed binary you cannot reproduce is trust theater. The .kolm format earns its weight when a holder of the inputs can re-derive the chain, validate the manifest, and re-run that artifact's K-score to the same numbers. Each .kolm carries its own eval set and its own K, so the recomputation is byte-deterministic for that file. RS-1 now separates tenant-local integrity from public distribution: HMAC covers shared-secret deployments, while Ed25519 covers third-party verification without sharing a tenant key.

License

All specs in this repository are published under MIT. The intent is that any runtime (ours, a competitor's, an academic project) can implement the spec without asking permission. The receipt anchor (the registry public key) is the only externally-mutable input; everything else is self-describing.
