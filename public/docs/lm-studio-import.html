LM Studio import wizard.

Not Kolm therapeutics, Kolm band, or other unrelated organizations using similar names.

A signed .kolm archive carries enough metadata for LM Studio to register it as a local model. The path is drag-and-drop today; the wizard spec below covers what an LM Studio plugin would need to do this in one click.

Honesty contract: LM Studio does NOT enforce .kolm signature verification. The kolm CLI does. If you skip kolm verify before importing, you are trusting whoever handed you the file. The drag-and-drop path here is for convenience; the receipt path is for trust.

Step 1: download the .kolm artifact

1.

From kolm.ai

Browse /account/artifacts, find the artifact, click Download .kolm. The file lands in your default downloads folder.

2.

From the marketplace

Browse /marketplace, click Run with kolm on any listing. The marketplace returns a signed download URL; your browser downloads the artifact directly.

3.

From a USB stick or local file

Any .kolm file with the right signature is portable. Air-gapped, sneakernet, or shared via email attachment all work the same way. The artifact is self-contained.

Before you import, verify the signature on the CLI:

kolm verify ./my-artifact.kolm

Output ends with verify_ok: true when the signature checks out. If it doesn't, the kolm CLI refuses to install the artifact and prints the failure reason; do not import an unverified file into LM Studio.

Step 2: drag and drop into LM Studio

Open LM Studio. Drag the verified .kolm file onto the LM Studio window. LM Studio treats it as an unknown archive and prompts for the model name and template.

The kolm artifact carries that metadata inside its manifest. Today the operator types it in by hand. The wizard spec at the bottom of this page covers what LM Studio would need to read those fields automatically.

If LM Studio refuses the drop, the most likely cause is the weights.bin shard layout. LM Studio reads single-file GGUF natively; sharded artifacts need the kolm CLI to merge before import. Run kolm unpack --merge-shards ./my-artifact.kolm to produce a single-file output.

Step 3: LM Studio extracts and registers

Once LM Studio accepts the drop, it:

  1. Unzips the archive into its local model directory.
  2. Reads weights.bin as the model payload.
  3. Reads the operator-supplied (or, in a future plugin path, manifest-supplied) template, system prompt, and stop tokens.
  4. Lists the new model in the LM Studio model picker.

The model is now selectable like any other LM Studio model. Chat works, the inference parameters work, the local HTTP server works. You did not need to convert formats or run an extra build step.

Honest limitations

  • No signature enforcement. LM Studio does not check the kolm signature on import. If you want hard refuse-on-tamper, run the artifact under the kolm runtime (which does enforce) and use LM Studio as a UI layer over kolm's local server.
  • No attestation enforcement. If the manifest carries an attestation_block (TEE-issued report), LM Studio drops it. The kolm runtime is the only path that honours attestation gates today.
  • No automatic re-distill. LM Studio does not call back to kolm autopilot when the artifact drifts. K-Score watching, drift gates, and auto re-distill all happen in the kolm CLI; LM Studio is a downstream consumer.
  • Multi-modal artifacts not yet supported. The current LM Studio import path is text-only. Vision-language and audio artifacts need the kolm runtime.

Wizard spec section

These are the keys and values the LM Studio team would need from us to ship a native .kolm import flow. Posted here so it is reviewable in public.

An LM Studio plugin that reads the manifest, runs signature verification against the kolm public key chain, and pre-fills these fields would close the gap entirely. The wizard would still surface the values for the operator to confirm before the final "Register model" click.
