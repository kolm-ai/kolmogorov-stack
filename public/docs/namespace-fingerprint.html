Namespace fingerprinting.

A compact signature that identifies the distribution of captures in a kolm namespace without exposing any individual capture. Fingerprints make it possible to warm-start a brand-new namespace from a sibling that has already converged, and to detect distribution drift over time -- all without raw text leaving the tenant.

What it is

A namespace fingerprint is a fixed-length vector plus a small cluster signature. It is computed locally inside the tenant, signed, and stored alongside the namespace metadata. Two fingerprints can be compared in constant time to estimate how similar two namespaces look in their capture distribution. Comparing fingerprints does not require either tenant to expose raw captures.

How it is computed

The fingerprint is the HMAC-keyed embedding centroid of the namespace's recent captures, plus a small mixture signature that captures multi-modal distributions. The HMAC key is per-tenant and never leaves the tenant; this prevents an outside observer from reconstructing the centroid from leaked fingerprints.

  1. Sample the last N=10,000 captures in the namespace (or all of them, if fewer).
  2. Embed each capture text with the deterministic on-box embedder (sentence-transformers small).
  3. HMAC-rotate each embedding through the per-tenant key so the embedding space is tenant-private.
  4. Compute the centroid (mean vector) and a K=8-cluster K-means signature over the rotated embeddings.
  5. Concatenate centroid || cluster_means || cluster_weights and sign with the tenant key. Result is ~6 KB.
kolm fingerprint compute --namespace prod
# wrote ~/.kolm/fingerprints/prod-2026-05-24.json
# centroid dim: 384
# clusters:    8
# captures:    8,431
# fingerprint signed: ed25519 0x9a8b...

Cross-namespace transfer

The transfer flow (W715/W757) uses fingerprint distance to recommend a sibling namespace to warm-start from:

  1. Compute fingerprint for the new namespace once it has at least 100 captures.
  2. Compare against fingerprints of other tenant-owned namespaces; rank by cosine distance.
  3. If a sibling under cosine distance < 0.15 exists, propose to seed the new namespace's distill corpus with redacted, deduplicated captures from the sibling.
  4. The proposal is opt-in: the operator reviews the seed before promotion.

Cross-tenant transfer is not enabled by default. It is gated behind explicit opt-in via the W461 federated approval flow, and even then exchanges only fingerprints, never raw captures.

Privacy

  • No raw text leaves the tenant. The fingerprint is a low-dimensional aggregate; reconstructing any individual capture from a fingerprint is information-theoretically impossible at the dimensions used.
  • Tenant-private embedding space. The HMAC rotation by the per-tenant key means a fingerprint observed by an external party is meaningless without that key. Two tenants computing fingerprints of identical text will produce different fingerprints.
  • Differential-privacy noise (optional). For tenants that want a stronger guarantee, an opt-in Laplace noise layer can be added to the centroid with epsilon=1.0 before signing. The cost is slightly noisier sibling recommendations.
  • Audit. Every fingerprint compute, compare, and transfer proposal writes an audit row (see /docs/audit).

Use case: warm-starting a new namespace

A team launches a new product line and creates a new namespace prod-newline. After a week, the namespace has 200 captures but no distilled artifact -- not enough to converge. The team runs:

kolm fingerprint compute --namespace prod-newline
kolm fingerprint compare --namespace prod-newline --against prod
# cosine distance: 0.08  (very similar)
kolm fingerprint transfer --from prod --to prod-newline --propose
# proposed: seed prod-newline with 1,800 captures from prod
# review at /account/captures/proposed-transfers

Once approved, the proposed seed captures are redacted, deduplicated, and added to the new namespace's distill corpus. The first distill run on the new namespace converges meaningfully faster than starting from 200 captures alone.
