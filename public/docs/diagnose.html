Diagnose: per-category K-Score breakdown

A single K-Score number ("0.72") tells you the artifact is below the production gate. It does not tell you where the regression sits or what to do next. kolm diagnose <cid> turns that one number into a structured envelope: per-category buckets (each with its own Wilson 95% CI when n >= 30), the lowest-scoring sample cids, and a deterministic list of recommendations linked to the next CLI step. Brand-locked: kolm.ai — the AI workbench. Frontier AI on your own infrastructure.

Why a single K-Score is not enough

K-Score collapses accuracy, size, latency, cost, and coverage into one composite. That is exactly what you want when comparing two artifacts side by side — but it is also why "K-Score 0.72" is unhelpful when you need to ship next week. The headline number says "below gate" without saying which slice of traffic dragged it down. The diagnostic envelope un-collapses the composite along the axis that actually matters in production: which slice of the captured workload the artifact is failing on.

The canonical failure mode looks like this. A model passes 0.92 on single-turn support tickets, 0.94 on billing lookups, 0.62 on multi-turn refund negotiations, and 0.95 on PDF redaction. Unweighted mean: 0.72. Headline: "below gate." Real story: "the artifact is shippable on three of four slices; multi-turn needs 100 more captures." The diagnostic envelope spells that story out and links it to kolm capture --category multi-turn.

How captures get bucketed

Categorisation is a pure heuristic over fields already on the capture record — no LLM calls, no on-the-fly classifier. The rules below are checked in order; first match wins. Determinism is the load-bearing property: the same capture set always produces the same buckets, regardless of when the diagnostic ran.

If the capture has...Bucket
turn_count > 1 (or messages.length > 1)multi-turn
media_kind in {image,audio,video,pdf}the media kind itself
tool_calls array with len > 0tool-use
explicit category stringthat string
namespace stringthe namespace itself
everything elsegeneral

Wilson 95% CI — and the honest null floor

When a category has n >= 30 rows we report a Wilson 95% confidence interval on the per-bucket mean. The Wilson interval is the right answer for proportions on small samples because it does not blow up at the endpoints — an unweighted bucket of 47 rows with a mean of 0.62 honestly maps to [0.51, 0.72], not [0.62, 0.62].

When n < 30 we return k_score_ci_lo: null and k_score_ci_hi: null. We do not substitute a Bayesian prior, we do not fall back to the pooled estimate, we do not interpolate. The bucket is honestly under-sampled and the dashboard says so. This is the same honesty contract that drives W810 calibration's insufficient_data status and W258-ML-1's MIN_HONEST_A_FOR_RATIO floor — if the measurement is shaky, the envelope says shaky, not "0.6204 ± 0.0000."

Recommendation rules

Recommendations are deterministic and lock-in tested. The four rules below are the entire surface — no LLM picks them, no human curates them. A capture set that hits the same rule on a re-run produces the same recommendation list, in the same priority order.

TriggerActionPriority
per-category k_score < 0.85 AND n < 200 capture_more with target_count = max(100, 200 - n) high
overall k_score < 0.85 AND every per-category bucket is >= 0.85 inspect_captures (data-quality skew) medium
per-category k_score_stddev > 0.15 AND n >= 30 adjust_temperature from 0.7 to 0.4 medium
every per-category k_score >= 0.95 promote_to_production (emitted alone) info

Single-issue mode: when the promote_to_production rule fires, it is the only recommendation in the envelope — we do not pad it with low-signal warnings. The CI dashboard either sees "promote" or sees a list of specific fixes; there is no third state where both appear.

API + CLI

The CLI shells out to the API — same envelope, same fields, no client-side computation.

SurfaceInvocation
CLI (table)kolm diagnose <artifact_cid>
CLI (json)kolm diagnose <artifact_cid> --json
API (POST)POST /v1/diagnose body {"artifact_cid": "<cid>"}
API (GET)GET /v1/diagnose/:cid
In-app dashboard/account/diagnose?cid=<cid>

Honest absence: no_bakeoff_results_yet

The diagnostic depends on per-row bakeoff results. When no bakeoff has been run for the requested artifact, the route returns HTTP 200 with an honest { ok: false, error: "no_bakeoff_results_yet" } envelope — not a 404. The HTTP call itself succeeded; the actionable hint tells the caller what to do next: run `kolm bakeoff first` against this artifact_cid, then retry.

Next steps

Run kolm bakeoff first to produce per-row K-Score rows, then kolm diagnose <cid> to bucket them. Read K-Score methodology for the underlying composite and pipelines for how to compose specialists once you have shipped each per-category artifact.
