Open-source AI compiler

Cloud compile.

Run kolm compile against a hosted GPU when your local machine cannot host the source model. The kolm side is a thin driver; the cloud provider owns the GPU pool and the billing. Three providers are wired today: RunPod, Modal, and Together. Pick by GPU shape, by billing model, and by which one you already have an account with.

When you need it

Reach for cloud compile when any of these is true.

  • You do not have a local NVIDIA GPU (no 4090 / 5090 / A6000 / H100).
  • The source model is larger than ~24 GB in bf16 (DeepSeek-R1-32B, Llama-3.1-70B, Qwen-2.5-72B) and will not fit on consumer cards even at NF4.
  • You want a one-shot quantize for an artifact you will keep and serve forever, and you would rather rent an H100 for three minutes than buy hardware.

If your model is ≤ 7B and you have any modern GPU, run locally instead. It will be faster and free.

RunPod

Provider 1

What it is for

RunPod rents whole pods (a container plus a GPU plus a network volume) by the second. Best when you want spot-priced GPU time, a persistent volume that survives reboots, or a long-running interactive session for iterative compile work. RunPod gives you ssh into the pod, which means you can attach a debugger or run kolm verify in the same shell as the compile.

Command

kolm compile --cloud runpod \
  --model deepseek-ai/DeepSeek-R1-Distill-Qwen-32B \
  --quant nf4-int4 \
  --gpu a100-80gb \
  --out ./out/r1-32b-int4

The driver provisions a pod, mounts a persistent volume named kolm-compile-out, runs the quantize, and stitches the artifact back to your local --out directory. Pass --keep-pod if you want to leave the pod running for a follow-up bake-off.

When to pick it

Pick RunPod when you want the cheapest GPU-hour, when you need a persistent pod for back-to-back compiles, or when you want ssh access for debugging. Pick something else if you do not want to manage pod lifecycle.

Modal

Provider 2

What it is for

Modal is a function-as-a-service GPU platform. You define a Python function decorated with a GPU type, and Modal handles cold-start, autoscaling, billing per GPU-second, and artifact storage in a modal.Volume. Best when you want zero pod management, predictable cold-start under a minute, and per-second billing with no idle charge.

Command

kolm compile --cloud modal \
  --model deepseek-ai/DeepSeek-R1-Distill-Qwen-32B \
  --quant nf4-int4 \
  --gpu h100 \
  --out ./out/r1-32b-int4

The driver authenticates against ~/.modal.toml (or MODAL_TOKEN_ID + MODAL_TOKEN_SECRET env vars), runs the function, writes the artifact to a Modal Volume named kolm-compile-out, and pulls it back to your --out directory. For gated HuggingFace models, create a Modal Secret named huggingface containing HF_TOKEN.

A typical 32B quantize finishes in well under one GPU-hour, so the all-in Modal cost is usually in the single-dollar range, plus a few cents of network egress when the driver pulls the artifact back.

When to pick it

Pick Modal when you want the lowest operational overhead, when you only need the GPU for the duration of the quantize, or when you already have a Modal account for other workloads. Pick something else if you need ssh-into-pod debugging or persistent compute between runs.

Together

Provider 3

What it is for

Together runs a managed fine-tune and quantize API on their own GPU fleet. You ship the model name and a quant spec; Together runs the job and stores the artifact in their object store, then signs a presigned URL for the driver to download. Best when you want a single API call with no GPU shape selection, no pod lifecycle, and no Modal-style function authoring.

Command

kolm compile --cloud together \
  --model deepseek-ai/DeepSeek-R1-Distill-Qwen-32B \
  --quant nf4-int4 \
  --out ./out/r1-32b-int4

The driver reads TOGETHER_API_KEY from your environment, submits a quantize job to Together's API, polls until done, and stitches the artifact back to --out. Together selects the GPU shape based on the source-model size; you cannot pick the SKU.

When to pick it

Pick Together when you want one API call and a finished artifact with no infrastructure decisions, or when you already use Together for inference and want unified billing. Pick something else if you want spot pricing (RunPod is cheaper per GPU-hour), or if you want fine control over the GPU shape (Modal lets you pick).

How to pick

  • Cheapest GPU-hour — RunPod spot.
  • Lowest operational overhead — Modal.
  • One API call, no infra decisions — Together.
  • You already have an account with one of them — that one.

All prices indicative as of 2026-05; check the provider site for current rates. kolm does not bill or proxy. The provider bills your provider account directly.

Caveats / Limitations

  • GGUF profiles are deferred on the Modal and Together paths. V1 covers nf4-int4, int4, and int8 via bitsandbytes. GGUF (q4_k_m, q5_k_m, q8_0) needs llama.cpp convert.py plus quantize and ships on the RunPod path first.
  • No retry / resume. A cold-start timeout means you re-invoke the driver. The provider does not refund cold-start time.
  • No metering inside kolm. kolm does not bill or proxy. Treat the cost tables above as indicative, not contractual.
  • Gated-model auth is per-secret. The driver does not auto-forward your local HF_TOKEN. Set the provider's secret store as shown in each section.
