Failure modes: artifact-keyed cluster K-Score

An artifact's headline K-Score collapses dozens of slices into a single number. kolm failure-modes <cid> takes the bakeoff rows that artifact produced, clusters the captures behind them by topic, and tells you exactly which slice the student is failing on relative to the teacher. The canonical example: "your support bot scores 0.97 on refunds but 0.62 on billing disputes." Brand-locked: kolm.ai — the AI workbench. Frontier AI on your own infrastructure.

Why "K-Score 0.74" needs a slice view

A composite K-Score is the right way to rank two artifacts on the same dataset. It is the wrong tool for figuring out where the regression sits. A bot that ships 0.97 on the refund flow and 0.62 on the billing-disputes flow has a headline of around 0.80 — the average doesn't tell anyone what to fix, and the headline number alone won't even prompt the right question.

The artifact-keyed failure-modes envelope un-collapses the headline by partitioning captures into topical clusters. Each cluster gets its own K-Score (Wilson 95% CI when n >= 30; plain null below), and the report sorts them by delta_vs_overall so the panel literally reads "refunds 0.97 vs billing-disputes 0.62, the student diverges most on billing." Each row links directly to the diagnostic envelope for the next step.

vs. Both modules live in the same product. (kolm failure-modes with no positional arg) reads the tenant-wide event stream and clusters by topic + length + vendor class — useful for the routing-and-self-improvement loop. (kolm failure-modes <artifact_cid>) is CID-keyed and reads the bakeoff rows + captures attached to one artifact. Same verb, different question shape.

The clustering heuristic (and why it's plain about being a placeholder)

Today the clustering is pure-heuristic heuristic_keyword_v1 — tokenize the capture input, drop a small built-in stopword list, build top-3 1- and 2-grams, and cluster captures that share at least two n-grams. The envelope ships a clustering field that names the algorithm so downstream dashboards can branch on it. The envelope deliberately reports "heuristic_keyword_v1", NOT "w757_fingerprint", because the fingerprinting model from isn't shipped yet. When lands, this field flips and the cluster ids change shape; today it is frankly the placeholder.

  • Stopwords: a small built-in list of around 80 English function words. Larger lists drift into linguistic territory that ought to be a real embedding model.
  • N-grams: 1- and 2-grams, frequency-counted, top-3 per capture by count desc with alphabetical tiebreak for determinism.
  • Cluster rule: greedy merge into the existing cluster whose founder shares the most n-grams (must share >= 2); ties go to the lower-index existing cluster. Below min_cluster_size (default 10) a candidate is dropped — signal would be noise.
  • cluster_id: "cluster_" + FNV-1a-8(sorted-keyword-list). Deterministic and short enough to read in tables.

The same capture set always produces the same cluster ids. Adding one capture might merge two clusters or split one; the envelope's clustering field warns downstream tooling that this version of the heuristic is what was used.

Wilson 95% CI — and the plain null floor

Per-cluster CI is identical to the diagnostic floor: when a cluster has n >= 30 bakeoff rows we report a Wilson 95% interval on the per-cluster mean. Below 30, the CI is null for both endpoints — the bucket is frankly under-sampled and the dashboard says so. This is the same no-silent-pass contract that drives's recommendations and calibration's insufficient_data status — if the measurement is shaky, the envelope says shaky, not "0.6204 ± 0.0000."

Bridge to the diagnostic envelope

Every envelope ships a diagnostic_link pointing at /account/diagnose?cid=<artifact_cid> (-4). Each cluster row in the dashboard renders a "View diagnostic" button that follows the same link. The handoff is intentionally cheap: the failure-modes view tells you which slice is regressing; the diagnostic envelope tells you the deterministic next-step recommendation (capture_more / inspect_captures / adjust_temperature / promote_to_production) per category. Click through to /docs/diagnose for the recommendation rules.

API + CLI

The CLI shells out to the API — same envelope, same fields, no client-side computation.

In-app dashboard/account/failure-modes?cid=<cid>

Plain absence: no_bakeoff_results_yet

The report depends on per-row bakeoff results for the requested artifact. When no bakeoff has been run yet, the route returns HTTP 200 with an plain { ok: false, error: "no_bakeoff_results_yet" } envelope — not a 404. The HTTP call succeeded; the actionable hint tells the caller what to do next: run `kolm bakeoff first` against this artifact_cid, then retry. The envelope still ships diagnostic_link so the dashboard can offer the pivot even when the bakeoff is missing.

Next steps

Run kolm bakeoff first to produce per-row K-Score rows, then kolm failure-modes <cid> to bucket them. Click any cluster's diagnostic link to land in /docs/diagnose for the deterministic next-step recommendation. When fingerprinting ships, the clustering field in this envelope will flip from heuristic_keyword_v1 to w757_fingerprint and the cluster ids will change — downstream dashboards should branch on the clustering field to handle the migration frankly.

Diagnose →Per-category K-Score + recommendations for the next CLI step. K-Score →The composite the failure-modes report decomposes. In-app dashboard →Browse the envelope visually.
