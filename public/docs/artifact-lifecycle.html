Run & Govern / Lifecycle

Created, signed, deployed, monitored, superseded, revoked, archived.

Every artifact moves through a small set of states that mirror the publish workflow of a signed .kolm. Transitions are append-only and recorded with actor, reason, and an optional evidence pointer. Revoked artifacts return 410 Gone at the download endpoint and can never deploy again.

The seven states

StateMeaning
createdBytes exist; not yet signed (mid-build).
signedReceipt + signature attached; eligible to deploy.
deployedBound to a namespace and serving traffic.
monitoredDeployed AND drift/health monitoring is on.
supersededAnother artifact took its serving slot (successor_id set).
revokedWithdrawn. Pulls return 410 Gone and can never deploy again.
archivedTerminal cold storage. Not serving, not eligible.

An eighth label, undeployed, lives only in the transition graph as a transient when a namespace clears its artifact (deployed -> undeployed -> deployed or -> archived).

Valid transitions

Anything not listed is rejected. The full table is VALID_TRANSITIONS in src/artifact-lifecycle.js:

created    -> signed
signed     -> deployed, archived
deployed   -> monitored, superseded, revoked, undeployed
undeployed -> deployed, archived
monitored  -> deployed, superseded, revoked
superseded -> archived, revoked
revoked    -> archived
archived   -> (terminal)

Revoke is reachable from any non-terminal state via the dedicated path, but it is intentionally NOT in the casual transition table — callers route to it through transition(artifact, 'revoked', { reason }) so a typo cannot accidentally destroy serving state.

The history record

Every transition() appends one row to artifact.history:

{
  timestamp:    "2026-05-26T17:42:08.114Z",
  from:         "signed",
  to:           "deployed",
  actor:        "tenant_f33c240034c5",      // or "system"
  reason:       null,
  evidence_id:  "rcpt_01KYC1ZVTGDCW3FX06JQSC",
  successor_id: null                          // only on 'superseded'
}

The record is persisted atomically to data/artifacts/<artifact_id>/lifecycle.json using write-temp + rename, the same pattern store.js uses for json tables.

Hard rules

  • actor is required (tenant_id of the caller or the literal string system).
  • Revoking requires a non-empty reason — audit trail.
  • Superseding requires a non-empty successor_id so the dependency edge can be reconstructed.
  • Revoked artifacts are pull-blocked at the download endpoint via canPull(artifact). The router checks this BEFORE streaming bytes.

CLI

# Inspect current state + history.
kolm lifecycle <artifact_id>

# Move signed -> deployed (binds to a namespace).
kolm deploy <artifact_id> --namespace support

# Move deployed -> undeployed -> deployed (a different artifact takes the slot).
kolm undeploy <artifact_id>

# Move deployed -> superseded (atomic swap).
kolm deploy <successor_id> --namespace support --supersedes <artifact_id>

# Roll back to a prior deployment.
kolm rollback --namespace support --to <prior_artifact_id>

# Withdraw an artifact. Pulls return 410 Gone immediately.
kolm revoke <artifact_id> --reason "key compromised"

HTTP

GET  /v1/lifecycle/:artifact_id
# {
#   "current_state": "deployed",
#   "history": [ {timestamp, from, to, actor, reason, evidence_id}, ... ]
# }

POST /v1/lifecycle/:artifact_id/transition
{ "to": "revoked", "reason": "key compromised", "evidence_id": "..." }

GET  /v1/artifacts/:artifact_id/download
# 410 Gone when the artifact is revoked.
Caveats. Lifecycle state is per-artifact, not per-namespace binding. A single .kolm can be deployed to several namespaces; undeployed only fires when the LAST namespace releases the slot.
