Pipelines: composing kolm artifacts

Most real applications are not one model. They are a routing layer (the classifier) plus a handful of specialists. kolm.ai lets you describe that shape declaratively in kolm.pipeline.yaml — one small specialist per label, an escalation lane for the long tail, and a runtime that prints the per-phase latency so you can see exactly where the time went.

Why compose artifacts

A single 70B generalist gets you to "it works on the demo" fast. It gets you to "billing questions cost us $0.04 each and the latency is 8 seconds" just as fast. The kolm composition argument is the same one your favourite operating system makes: compose small, specialised, fast modules instead of one giant binary. An intake classifier that decides which of {support, billing, escalation} the prompt belongs to is a 0.5B model that runs in <100ms; a billing-specific specialist is a 32B INT4 model that ships in 1.9 GB; the escalation lane is a hosted teacher you only pay for on the long tail you actually need.

The win is not just cost. When billing logic changes you re-distill the billing artifact and bump its cid in the pipeline yaml. The classifier stays untouched, the support artifact stays untouched, and the W739 lineage chain shows the operator one line: billing.kolm: bafkrei… -> bafkrei…. Each artifact is independently versioned, independently re-distilled, independently evaluated against its own holdout. There is no "one giant model passed the eval" lie; the per-route K-Score is the actual K-Score for the actual route.

kolm.pipeline.yaml schema

The schema is small on purpose. A pipeline pins:

  • One classifier by cid (and optional version label) — whatever .run(input) returns is the route label.
  • A routes mapping of label -> target. Each target is exactly one of { artifact_cid } or { teacher }.
  • A name for the pipeline so dashboards and the W739 diff have something to print.
version: w738-v1
name: support-triage

classifier:
  artifact_cid: bafkreigh2akiscaildc3xy7p4nntwvjp7m5kw5kbsmm5kkkkkkkkkkkkkk
  version: v1

routes:
  support:
    artifact_cid: bafkreig5ssssssspport4qqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqq
  billing:
    artifact_cid: bafkreib1lling4qqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqqq
  escalation:
    teacher: claude-sonnet-4-6

Scaffold the file with kolm pipeline validate --file kolm.pipeline.yaml after you have copied the sample above; the validator is the same one the server runs on POST, so a clean validate locally guarantees a clean validate in CI.

Routing semantics

The runtime is intentionally boring: load the classifier, run it on the input, take the returned label, look it up in routes. If the route is an artifact_cid the runtime loads that artifact (tenant-fenced) and runs it on the same input. If the route is a teacher the runtime calls the configured teacher caller. The original prompt is passed through unchanged in both cases — specialists see exactly what the classifier saw.

Classifier outputResolution
"support"Load routes.support.artifact_cid and run.
"billing"Load routes.billing.artifact_cid and run.
"escalation"Call routes.escalation.teacher via the configured caller.
any other labelHonest { ok:false, error:"route_not_found", label, available_routes } envelope. No silent fallback — the operator either adds the label to the yaml or fixes the classifier.

"Escalation fallback" is a routing convention, not a runtime feature: name the escalation route whatever your classifier emits when it is not confident (we recommend literally escalation), and route that label to a hosted teacher. There is no implicit teacher fall-through when a label is missing — that would be exactly the silent-error footgun this design rules out.

Versioning

Every reference in the yaml is a cid. cids are content addresses: change the artifact and the cid changes. That means the pipeline yaml is the route table AND the version pin in one file. You can:

  • Re-distill one specialist without touching the others. Bump the billing cid, leave support + classifier alone, ship the new pipeline yaml.
  • Roll back one specialist by reverting the cid in the yaml. The artifact registry still holds the old cid; no rebuild required.
  • Evaluate per-artifact: each cid has its own K-Score, its own holdout, its own production-ready verdict. A pipeline is "ready" when every cid it references is ready.
  • Diff cleanly via kolm pipeline compile --file kolm.pipeline.yaml --out sidecar.json; the sidecar's parent_cids array is the W739 lineage chain entry-point.

Latency breakdown

Every runPipeline result — whether ok:true or an honest error envelope — includes a latency_ms_breakdown block. The numbers are real Date.now() deltas, not estimates, so a per-tenant p50/p99 dashboard sees the exact same numbers the runtime measured.

{
  "ok": true,
  "result": "Refunds are issued within 30 days...",
  "classifier_label": "billing",
  "route_taken": { "kind": "artifact", "cid": "bafkreib1lling4...", "label": "billing" },
  "latency_ms_breakdown": {
    "classify": 47,
    "route": 312,
    "total": 359
  },
  "version": "w738-v1"
}

The classify phase is load + run of the classifier. The route phase is load + run of the chosen specialist (or the teacher call). total is an independently-measured wall-clock from the top of the handler to just before the return; on a healthy host it equals classify + route plus a small amount of JS scheduling overhead. When it does not, the gap is your queueing time — surface it to your dashboard.

Honest limits

  • Numbers are wall-clock, not CPU time. A 312ms route on a saturated host might be 50ms of compute and 262ms of event-loop wait. The pipeline yaml is not a scheduler; pair this with load-queue metrics.
  • No retry at the routing layer. If a specialist throws, you get error:"route_artifact_failure" with the exception detail — the operator decides whether to retry, escalate, or surface the failure to the user.
  • Teacher escalations are non-deterministic. The same input through the same pipeline can give different teacher outputs because the teacher is a hosted model. The route_taken field is still deterministic — only the final result varies.

API + CLI

The same pipeline yaml drives the API and the CLI; they share the validator and the runner.

SurfaceInvocation
CLI validatekolm pipeline validate --file kolm.pipeline.yaml
CLI compile (sidecar)kolm pipeline compile --file kolm.pipeline.yaml --out sidecar.json
CLI runkolm pipeline run --file kolm.pipeline.yaml --input "<text>"
API runPOST /v1/pipeline/run with { pipeline_yaml, input } (auth-gated; tenant-fenced loader)
