Docs / Plugins

Open-source AI workbench

Frontier AI on your own infrastructure.

Extend the workbench. Bring your own quantization method, runtime adapter, capture processor, or eval metric — one contract, four kinds.

What you can plug in

Custom quantization

kind: quantization

Ships a quantization method into the forge alongside the built-in int4 / int8 / gptq / awq / aqlm / quip / exl2 / exl3 / hqq / qat menu.

Runtime adapter

kind: runtime

Wires an alternate execution backend into kolm run — e.g. a private inference server or a research runtime.

Capture processor

kind: capture-processor

Transforms capture rows in the gateway pipeline. Useful for domain-specific PII redaction, hashing, or enrichment before the row hits the lake.

Custom eval metric

kind: eval-metric

Scores rows in kolm bakeoff alongside K-Score / parse-failure-rate / context-faithfulness. Returns a per-row 0..1 score.

The contract

Every plugin is a directory under ~/.kolm/plugins/<name>/ with a plugin.json manifest and an entry script. The manifest is the same regardless of kind; dispatch happens via the kinds array.

plugin.json

{
  "name": "my-quant",
  "version": "1.0.0",
  "kinds": ["quantization"],
  "entry": "index.js",
  "description": "AWQ + W4A8 mixed-precision quantizer",
  "author": "Acme Research",
  "homepage": "https://github.com/acme/my-quant"
}

Fields

FieldRequiredDescription
nameyesLowercase alphanumeric + hyphen, 1–64 chars. Doubles as the directory name.
versionyesFree-form version string. Used to gate re-register (set KOLM_PLUGIN_OVERWRITE=1 to replace).
kindsyesArray of one or more of: quantization, runtime, capture-processor, eval-metric.
entryyesRelative path to the entry script. No absolute paths, no ...
descriptionnoSurfaced in kolm plugin list and the marketplace.
authornoSurfaced in kolm plugin info.
homepagenoLinked from the marketplace listing.

Install

# From a local checkout
kolm plugin register ./my-quant/plugin.json

# Inspect the registered set
kolm plugin list
kolm plugin list --kind=quantization
kolm plugin info my-quant

HTTP API

The same operations are available over HTTP for self-hosted dashboards and CI. All routes are auth-gated.

# List installed plugins
curl -H "Authorization: Bearer $KOLM_API_KEY" \
  https://kolm.ai/v1/plugins

# Filter by kind
curl -H "Authorization: Bearer $KOLM_API_KEY" \
  "https://kolm.ai/v1/plugins?kind=eval-metric"

# Register from a local manifest path (server reads it from its own disk)
curl -X POST -H "Authorization: Bearer $KOLM_API_KEY" \
     -H "Content-Type: application/json" \
     -d '{"manifest_path":"/srv/plugins/my-quant/plugin.json"}' \
     https://kolm.ai/v1/plugins

# Single plugin lookup
curl -H "Authorization: Bearer $KOLM_API_KEY" \
  https://kolm.ai/v1/plugins/my-quant

Worked example: an eval-metric plugin

// ~/.kolm/plugins/length-bias/plugin.json
{
  "name": "length-bias",
  "version": "0.1.0",
  "kinds": ["eval-metric"],
  "entry": "index.js",
  "description": "Penalizes overly verbose completions vs the reference"
}
// ~/.kolm/plugins/length-bias/index.js
export default function score({ prediction, reference }) {
  const p = (prediction || '').length;
  const r = (reference  || '').length || 1;
  const ratio = Math.min(p / r, r / p);
  return { score: ratio, units: 'ratio' };
}

After kolm plugin register, this metric shows up under kolm plugin list --kind=eval-metric. The bakeoff loop consults bakeoffMetricPlugins() in src/bakeoff.js and dispatches per row.

Security & trust

  • Plugins live under your own KOLM_HOME — nothing is fetched from a remote registry. You install what you wrote or what you cloned.
  • Manifest validation rejects path-traversal attempts (name matches a strict regex, entry must be relative and free of ..).
  • Entry scripts are not executed at load time. Each surface (forge, runtime, gateway, bakeoff) imports the plugin on demand under the caller's tenant context.
  • HTTP routes are auth-gated; on a multi-tenant host the routes only see the caller's own plugin directory.
