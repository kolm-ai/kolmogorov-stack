Studio / Benchmark

One eval. Many models. One table.

The benchmark harness (src/bench-harness.js) takes a list of models and an eval set, and produces a side-by-side comparison: K-Score per axis, mean latency, p95 latency, cost per call, and the judge's per-axis disagreement matrix. Models can be any mix of local .kolm files, Ollama models, GGUFs, vLLM endpoints, or remote API providers — the harness uses the same provider adapters as the Wrapper.

Why one harness

The unfair-comparison trap of LLM evals is that vendors pick different test sets, different system prompts, different sampling params, and different scoring rubrics — then publish numbers that disagree. The kolm bench harness eliminates the trap by fixing everything except the model under test: same prompts, same system prompt, same temperature, same judge, same rubric. The only variable is the model.

CLI

# Compare a distilled artifact against the base + two frontier APIs
kolm bench \
  --models trinity-support-7b-q4km.gguf,Qwen/Qwen2.5-7B-Instruct,anthropic/claude-haiku-4-5,openai/gpt-4o-mini \
  --eval data/support-holdout.jsonl \
  --judge google/gemini-2.5-pro \
  --rubric support-rubric-v2 \
  --output bench-2026-05-26.json

# Compare two versions of your own artifact
kolm bench --compare trinity-support-v2.kolm trinity-support-v3.kolm \
           --eval data/support-holdout.jsonl

# Smaller-and-faster: 50 prompts, single judge call per prompt
kolm bench --models a.kolm,b.kolm --eval data/support-holdout.jsonl --n 50 --judges 1

# Output formats
kolm bench --models a.kolm,b.kolm --eval e.jsonl --output bench.json --markdown bench.md

What the report contains

ColumnMeaning
modelThe id you passed (artifact path, Ollama name, or provider/model).
nNumber of prompts scored.
kscore_meanWeighted mean across rubric axes.
kscore_axes.<axis>Per-axis score, 0-1.
kscore_stdStd-dev across prompts (variability is information).
latency_ms_meanEnd-to-end generation latency.
latency_ms_p9595th-percentile latency (worst typical user).
cost_per_call_usdFor API teachers; for local models — amortized electricity per call.
output_chars_meanMean response length; useful for over-verbosity detection.
refusal_rateFraction the judge marked as refusal (off-policy + apology pattern).
error_rateFraction that errored (timeout / 500 / OOM).

How to read the result

  1. Headline number is K-Score mean — not the only number.
  2. Per-axis scores explain the headline — a model might tie on mean but lose on asks_first_question.
  3. Latency p95 is the user-facing one — mean hides tail.
  4. Cost includes electricity for local — the harness estimates from GPU model + token count.
  5. Std-dev between runs matters — high variance = you need more --n.

Bake-off mode

For an A/B between two student variants, use --compare to get a pairwise judge call (judge sees both responses and picks one), which is more statistically efficient than rubric scoring two models separately:

kolm bench --compare v2.kolm v3.kolm --eval support-holdout.jsonl --n 100
# emits win-rate, tie-rate, and a 95% bootstrap CI
Caveats. Latency numbers depend on your hardware (local) or network distance (API). Always re-run the bench on the box that will serve traffic before you commit to a deployment decision. Cost numbers for local include a default $0.12/kWh estimate; override with --cost-per-kwh 0.18 for your tariff.
