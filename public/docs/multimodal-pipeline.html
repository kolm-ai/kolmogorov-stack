Multimodal capture pipeline.

A filesystem capture lake for heterogeneous data: images, audio, tool-use spans, and full multi-turn conversations. One namespace per workload; deterministic path layout; the redaction receipt rides alongside the row. Stamped w829-v1.

Storage layout

The lake is a directory under ~/.kolm/captures/. One JSONL file per (namespace, kind, hash) tuple keeps every row append-only and content-addressable.

~/.kolm/captures/<namespace>/multimodal/<kind>/<hash>.jsonl
~/.kolm/captures/<namespace>/multi-turn/<conversation_id>.jsonl

The four supported kinds are image, audio, tool_use, and multi_turn. The hash is sha256 over the canonical payload; the same image submitted twice lands in the same JSONL file (one row per submission, dedup is the caller's job). The multi-turn file is keyed by conversation_id so an ongoing thread aggregates into one file.

Plain path safety

Namespace, kind, and hash are sanitized to closed alphabets before they touch the filesystem. A namespace containing ../ resolves to a sanitized literal token; the kind must be one of the four supported values or the call returns {ok:false, error:'kind_unsupported'}; the hash must be lowercase hex up to 64 chars. Path traversal is impossible by construction.

Capture routes

Two POST routes write to the lake: one for the single-row multimodal kinds, one for the append-to-conversation multi-turn kind. Both are tenant-fenced via req.tenant_record.id.

POST /v1/captures/multimodal

curl -X POST https://kolm.ai/v1/captures/multimodal \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "namespace": "vision-bot",
    "kind": "image",
    "payload": {
      "image_url": "https://example.com/screenshot.png",
      "prompt": "Describe what is on screen",
      "response": "A login form with email + password fields"
    },
    "redaction_receipt": {
      "redactor_version": "w462-v1",
      "redacted_regions": 2
    }
  }'

POST /v1/captures/multi-turn

curl -X POST https://kolm.ai/v1/captures/multi-turn \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "namespace": "support-thread",
    "conversation_id": "conv_2026_05_24_abc",
    "conversation": [
      {"role":"user", "content":"My export is stuck"},
      {"role":"assistant", "content":"What is the export ID?"}
    ],
    "parent_message_id": null
  }'

If the caller omits the hash on a multimodal capture, the route computes one from the canonical payload so the row always lands in a content-addressable file.

KOLM_NO_RAW_MULTIMODAL

When the env var KOLM_NO_RAW_MULTIMODAL=1 is set, the capture layer strips payload.data_uri before writing. The hash is computed against the original payload (so dedup still works), but the stored row carries no raw pixels or audio bytes.

This is the compliance posture for tenants who must answer "hash + redaction receipt only, never the raw media." The transcript redactor (text), image-PII redactor (pixels), and voiceprint redactor (audio) all process the row BEFORE the strip; the receipt fields are preserved.

The strip is opt-in via env so a development environment can keep the raw bytes for visual debugging. Production deployments that need the strict posture set the env var globally and rely on the/redactor receipts as the audit trail.

VLM distillation

The pipeline feeds into a VLM distillation entry point. The teacher options are gpt-4v, claude-3-vision, and gemini-vision. The route refuses to silently succeed: when the teacher API key is missing the envelope is ok:true but flagged real_run:false.

POST /v1/vlm-distill/run

curl -X POST https://kolm.ai/v1/vlm-distill/run \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "teacher": "claude-3-vision",
    "student_model": "Qwen2.5-VL-2B",
    "dataset_captures": "vision-bot"
  }'

# When KOLM_VLM_TEACHER_API_KEY is absent:
# {
#   "ok": true,
#   "status": "queued",
#   "real_run": false,
#   "missing_env": "KOLM_VLM_TEACHER_API_KEY",
#   "run_id": "vlm_a1b2c3d4..."
# }

The loud-fail envelope is deliberate: a queued job persists, the operator can set the env var, and the run resumes via the same run_id. The pipeline never claims a finished distillation against a teacher it could not actually reach.

GET /v1/vlm-distill/runs

curl -H "Authorization: Bearer $KOLM_API_KEY" \
  https://kolm.ai/v1/vlm-distill/runs
# -> { "ok":true, "runs":[...], "supported_teachers":["gpt-4v","claude-3-vision","gemini-vision"],
#      "supported_kinds":["image","audio","tool_use","multi_turn"] }

Redactor chain

Multimodal rows flow through three redactor workers, each isolated into its own package so heavy ML dependencies do not bloat the root install.

  • workers/media-redact - text transcript scrub for emails, phone numbers, account IDs.
  • workers/multimodal-redact-image - YOLO face + license-plate detection via onnxruntime-node, sharp blur/mask composite.
  • workers/multimodal-redact-audio - voiceprint scrub via pyannote-audio with priority chain over VOICEPRINT_REDACT_CMD, pyannote-audio-redact, ~/.kolm/scripts/voiceprint-redact.py.

No-silent-pass contract: when a redactor's optional deps are missing, the envelope returns {ok:false, error:'no_detector_installed', redacted_image:null} (or redacted_audio:null) plus an install hint. The pipeline never passes through unredacted media silently when a redactor was requested.

TUI and CLI

The TUI binds multimodal pipeline view to hotkey V (and command aliases :multimodal / :mm). The bare collection endpoint GET /v1/multimodal/pipeline aggregates the four routes for the dashboard.

kolm tui
# inside the TUI:
#   V        multimodal pipeline view
#   /        filter by kind
#   :mm      command palette alias

kolm media image-doctor       # check image redactor readiness
kolm media audio-doctor       # check audio redactor readiness
kolm bakeoff multimodal       # replay multimodal captures across artifacts
