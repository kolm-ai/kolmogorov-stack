Memory-aware runtime placement.

kolm runtime probes your VRAM, system RAM, and NVMe bandwidth before loading an artifact. It then picks full-GPU, hybrid GPU+RAM auto-split, or NVMe-mmap placement -- whichever fits your hardware and minimizes wallclock. Pre-load heuristics analyze your call pattern to warm the most likely next artifact. Probe failures fall back to a conservative placement with a logged envelope.

Memory hierarchy probe

VRAM, RAM, NVMe -- measured at load time.

The runtime probes the host before each artifact load. VRAM is queried via the driver. RAM is read from /proc/meminfo or platform-equivalent. NVMe bandwidth is measured with a short read-burst benchmark cached for the process lifetime so we do not penalize every load.

{ "probe": "ok", "vram_total_gb": 32.0, "vram_free_gb": 28.4, "ram_total_gb": 128.0, "ram_free_gb": 96.2, "nvme_read_gbps": 5.8, "artifact_size_gb": 17.9, "required_kv_cache_gb": 2.4 }
Placement decision tree

Three branches. Deterministic given a probe.

Given the probe, the placement decision is deterministic. We document the tree explicitly so operators can reproduce a placement choice from the probe alone, without running the binary.

if (artifact_size_gb + required_kv_cache_gb) <= vram_free_gb -> placement = "full_gpu" -> expected_tok_per_s = high (driver-direct) else if (artifact_size_gb + required_kv_cache_gb) <= (vram_free_gb + ram_free_gb) -> placement = "hybrid_auto_split" -> hot layers pinned to VRAM -> cold layers paged from system RAM else if nvme_read_gbps >= min_acceptable_gbps -> placement = "nvme_mmap" -> weights mmap'd from NVMe -> expected_tok_per_s = bandwidth-limited else -> placement = "refuse_with_envelope" -> reason = "host_below_minimum" -> suggest = upgrade_path_doc
Pre-load heuristic

Warm the artifact you are about to ask for.

If your inference traffic alternates between a small set of artifacts -- a router model plus a few task-specific students -- pre-loading hides the swap cost. The runtime tracks recent calls and warms the most likely next artifact when VRAM has headroom.

  • Markov-1 next-artifact predictor

    simple, interpretable, fast

    We use a first-order Markov chain over recent artifact IDs. Cheap to compute, easy to explain, and good enough when traffic has visible patterns. No deep model in the hot path.

  • Headroom gate

    never evict an in-use artifact

    Pre-loading is only attempted when VRAM has at least the predicted artifact's size in free headroom. We never evict an in-flight load to speculatively warm another.

  • Off by default

    opt-in per namespace

    Pre-loading is off by default. It is enabled per namespace via runtime.preload.enabled = true. We do not silently speculate against the operator's hardware budget.

Performance estimate before load

Know what to expect before you wait for the load.

Given the chosen placement, the runtime estimates tokens-per-second on your specific hardware before loading. The estimate uses a tiny per-placement lookup table calibrated on internal benchmarks and adjusted for the host's measured bandwidth.

{ "placement": "hybrid_auto_split", "estimated_tok_per_s": 25.2, "estimated_first_token_ms": 320, "load_seconds": 4.1, "basis": "calibrated_lookup_v1", "basis_confidence": "medium" }
~25 tok/s on your hardware
Honest envelope

Probe failures fall back conservatively.

If any probe fails -- driver returns nothing, /proc/meminfo is missing, NVMe benchmark times out -- we do not guess. The runtime falls back to a conservative placement (hybrid_auto_split when partial info is available, otherwise refuse-with-envelope) and logs a structured probe-failure record.

{ "probe": "partial", "vram_total_gb": null, "vram_probe_error": "driver_unreachable", "ram_total_gb": 64.0, "fallback_placement": "hybrid_auto_split", "fallback_reason": "vram_probe_failed", "logged_to": "/tmp/kolm/runtime-placement.log" }
Honest contract. The runtime never silently picks a placement when probes fail. Every fallback is logged with the failing probe and the reason. Operators can grep the log to see whether a slow load was a placement bug or a genuine hardware limit. If the host falls below documented minimums, the runtime refuses to load with an envelope rather than thrashing the swap file.
