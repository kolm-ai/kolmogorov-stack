Run & Govern

Run security.

The Run surface touches your production hosts. The defaults assume you do not trust the network and you do not want kolm to be the weakest link in the deploy chain. SSH keys are referenced by path and never copied. Every transferred artifact is sha256-checked and Ed25519-verified before it starts. Runtimes run unprivileged. Every action lands in an append-only audit trail.

SSH keys: paths only

When you kolm devices add --key ~/.ssh/id_ed25519_kolm, kolm stores the path in the device registry, not the key material. On every connection kolm reads the key from disk and hands it to ssh2; on disconnect the in-memory copy is dropped.

  • Key never leaves your control box. Removing a device from the registry deletes the row; it does not touch your key. The remote host is unaffected.
  • SSH agent supported. Omit --key and kolm will defer to SSH_AUTH_SOCK; useful for FIDO2 / hardware-backed keys.
  • Per-device key recommended. Use a dedicated key per device or per fleet (e.g. id_ed25519_fleet_prod) so revocation is surgical.
  • Password auth is supported but discouraged. Pass --password-stdin only when key auth is impossible. Stored as no part of the registry; re-prompted on every connect.

Integrity verify after transfer

Step 2 of the deploy pipeline (upload) emits the artifact and a sidecar .sha256+.sig. Step 3 (runtime ensure) blocks until both verify:

  • sha256 — the artifact bytes on disk match the hash in the manifest. Catches transport corruption.
  • Ed25519 — the artifact signature verifies against the publisher's pinned public key. Catches malicious substitution between the registry and the device.

If either check fails the deploy aborts before starting the pod, the previous artifact remains live, and an audit row is written with reason integrity-fail.

Non-root runtime

kolm never spawns the pod as root. On ssh device types the runtime starts as the SSH-login user; on docker as a non-root container user (--user 65532); on k8s with securityContext.runAsNonRoot: true in every generated manifest. If the runtime needs hardware access (e.g. /dev/kfd for ROCm, /dev/nvidia*) kolm requires the operator to grant the user-level cap-add in advance — never via sudo.

Audit trail

Every Run action writes one row to ~/.kolm/audit/run.jsonl (control box) and one row to your tenant's audit index (visible at /docs/audit). Rows are hash-chained: row_n.prev_hash == sha256(row_{n-1}), so silent edits in either log break the chain.

{
  "ts": "2026-05-26T17:22:01Z",
  "actor": "rodney",
  "action": "deploy",
  "artifact_id": "art_b2c3d4e5",
  "device_id": "prod-gpu-1",
  "namespace": "prod",
  "mode": "canary",
  "outcome": "ok",
  "deploy_id": "dpl_018f2a...",
  "prev_hash": "e1d4...",
  "row_hash": "9f7c..."
}

Export with kolm audit export --window 7d --format csv — same path as wrapper and studio audit exports, see audit export.

Optional mTLS (gateway ↔ pod)

By default a pod's OpenAI-compatible endpoint runs plain HTTP on a private network. For hostile networks add mTLS:

kolm devices add prod-gpu-1 \
  --type ssh --host 10.0.4.21 --user kolm --key ~/.ssh/id_ed25519 \
  --mtls-ca   ~/certs/fleet-ca.pem \
  --mtls-cert ~/certs/prod-gpu-1.pem \
  --mtls-key  ~/certs/prod-gpu-1.key

The pod requires a client cert signed by fleet-ca.pem; the gateway presents the matching cert when proxying. Cert files are referenced by path (never copied into the registry); rotation is a per-file update.

Firewall recommendations

DirectionPort / PathRecommendation
control box → device22/tcp (SSH)Allow from a fixed deploy network or jump host; rate-limit elsewhere.
device → tenant443/tcp to kolm.ai (or your self-host)Required for check-in + OTA. Block other egress when running closed-network.
gateway → podpod port (default 8000)Restrict to gateway IPs only. Pair with mTLS on hostile networks.
operators → podsame portBlock from operator workstations; route through gateway so every call lands in receipts.
pod → outboundegressBlock by default; allow only model/runtime dependency mirrors during runtime install (see runtimes).
Constraints. mTLS protects the in-fleet hop, not the operator-to-control-box hop — use your existing SSO/VPN there. The audit chain detects tampering but does not prevent it; for tamper-evident storage point ~/.kolm/audit/ at an append-only filesystem (e.g. WORM bucket, sigstore log). Air-gapped fleets cannot rotate the publisher pubkey via OTA — rotate the key by sneakernet, then re-deploy the artifact signed with the new key.
