Run & Govern

Troubleshooting.

Every Run-surface failure mode in one place, with the one-line remediation. The deploy pipeline always reports the failing step; the headings below mirror that step name verbatim so you can grep this page.

First line of defence

Before reading down the list, run the doctor. It probes every dependency the Run surface touches and prints the same fix lines this page does, scoped to your machine.

kolm doctor --json | jq '.checks[] | select(.status!="ok")'
kolm doctor --fix         # attempt the safe-to-automate fixes

SSH connection refused (preflight)

DeployError: preflight: ssh connect ECONNREFUSED 10.0.4.21:22

The SSH listener is not reachable on the host:port you registered. Either sshd is not running, a firewall is dropping the packet, or the registry has a stale address.

→ Run

kolm devices ping prod-gpu-1 --verbose          # confirms it is SSH, not artifact
ssh -p 22 kolm@10.0.4.21                        # reproduce the raw failure
# If wrong host/port:
kolm devices update prod-gpu-1 --host 10.0.4.99 --port 2222
# If host is up but firewall is blocking:
sudo ufw allow from <your-ip> to any port 22 proto tcp

SSH authentication failed (preflight)

DeployError: preflight: ssh authentication failed (publickey)

The key path is wrong, the agent does not have the key loaded, or the remote authorized_keys is missing the public half.

→ Run

kolm devices show prod-gpu-1 --json | jq .connection
# Re-register with the correct key path (never copied, only referenced):
kolm devices update prod-gpu-1 --key ~/.ssh/id_ed25519
# Or push the public half to the host:
ssh-copy-id -i ~/.ssh/id_ed25519.pub kolm@10.0.4.21

Hardware does not fit the artifact (preflight)

DeployError: preflight: artifact requires >= 24GB VRAM, device has 16GB

The artifact's runtime_passport.vram_required_gb is larger than what the device reported on the last hardware probe. The check is run before any bytes move so a misfit aborts in < 1s.

→ Run

# Option 1: deploy a smaller-quant variant of the same artifact
kolm test-quants <art> --device prod-gpu-1
kolm deploy <art-q4> --device prod-gpu-1

# Option 2: pick a fatter device
kolm devices list --select tag=a100
kolm deploy <art> --device prod-a100-2

# Option 3: refresh hardware (if you added VRAM)
kolm devices refresh prod-gpu-1

Upload or hash mismatch (upload)

DeployError: upload: sha256 mismatch (expected abc..., got def...)

The bytes on the device after transfer do not hash to the same sha256 as the local artifact. Usually a half-written transfer (disk full, network drop) or a tampered intermediate.

→ Run

kolm devices exec prod-gpu-1 -- "df -h /var/lib/kolm"   # is the disk full?
kolm devices exec prod-gpu-1 -- "rm -rf /var/lib/kolm/incoming/*"
kolm deploy <art> --device prod-gpu-1 --retry-upload  # second attempt; on second failure look at network

Runtime install failed (runtime-ensure)

DeployError: runtime-ensure: llama.cpp install failed (exit 127 on remote)

The bootstrap tried to install or upgrade the runtime on the remote and one of the package manager / source-build steps failed. The remote stderr is captured in the deploy record.

→ Run

kolm devices runtime status prod-gpu-1                 # what is actually installed
kolm devices runtime install prod-gpu-1 --runtime llama.cpp --version 0.3.4 --verbose
# Inspect the captured log:
kolm deploy logs <deploy-id> --step runtime-ensure --tail 200
# Fall back to a manual install per the runtimes doc, then mark as installed:
kolm devices runtime mark prod-gpu-1 --runtime llama.cpp --version 0.3.4 --path /usr/local/bin/llama-server

Pod failed to start (start)

DeployError: start: pod exited with code 137 before serving

The runtime started, the model loaded part way, then OOM-killed (exit 137 = SIGKILL, usually OOM). Or the runtime binary crashed; look at its stderr.

→ Run

kolm deploy logs <deploy-id> --step start --tail 200
kolm devices exec prod-gpu-1 -- "dmesg | tail -20"     # confirm oom-killer
# Drop to a smaller quant:
kolm test-quants <art> --device prod-gpu-1
# Or raise the device limit (only safe if the device is dedicated):
kolm devices config prod-gpu-1 --vram-reserve-gb 2

Port already in use (start)

DeployError: start: bind 0.0.0.0:8080 EADDRINUSE

Something on the device is already listening on the port kolm tried to bind. Usually a previous deploy that did not exit cleanly, or a non-kolm process.

→ Run

kolm devices exec prod-gpu-1 -- "ss -tlnp | grep :8080"
# If it is a stale kolm pod:
kolm fleet stop --device prod-gpu-1 --namespace <ns>
# Otherwise pick a different port:
kolm namespace config <ns> --port 8081
kolm deploy <art> --device prod-gpu-1

Smoke test failed (smoke)

DeployError: smoke: 3/10 prompts returned malformed JSON, k-score 0.42 < 0.70 floor

The pod started and served, but the on-device smoke test (the frozen 10-prompt eval set) failed to clear the k-score floor or returned malformed output. The deploy will auto-rollback if rollback-on-smoke is set; otherwise the bad pod is left running and traffic is not switched.

→ Run

kolm deploy logs <deploy-id> --step smoke --json
kolm test-device <art> --device prod-gpu-1 --verbose   # reproduce off the deploy path
# Likely causes:
# 1. Wrong runtime version -- check passport
kolm artifact show <art> --field runtime_passport
# 2. Quant too aggressive -- try a higher-precision build
kolm test-quants <art> --device prod-gpu-1
# 3. Genuine regression -- compare to the last green deploy
kolm deploy diff <deploy-id> <last-green-deploy-id>

Receipt write failed (record)

DeployError: record: receipt store unreachable (KOLM_RECEIPTS_URL timed out)

Pod is healthy and serving; the post-deploy receipt could not be written. Deploy is left in a recorded=false state but service is up. Worth fixing because audit chains depend on it.

→ Run

kolm config get receipts                              # confirm endpoint + tenant
kolm deploy record <deploy-id>                        # retry write only; does not touch the pod
# If the receipt store is genuinely down, queue and retry later:
kolm receipts queue list

OTA update stuck (per-namespace)

FleetWarning: namespace=prod has 3 devices on art_old, 2 on art_new for >30m

A rolling or canary update is partially propagated and stalled. Usually because new device joined mid-rollout, a canary device failed smoke, or the policy is set to notify and the operator has not approved yet.

→ Run

kolm fleet status --select namespace=prod
kolm namespace updates show prod                       # see policy + pending count
# Force-promote (skip notify gate):
kolm namespace updates promote prod
# Or roll back the partial cohort:
kolm fleet rollback --select namespace=prod,artifact=art_new

Device flagged offline by monitor

FleetAlert: prod-gpu-1 offline (3 consecutive health misses, last seen 14m ago)

The fleet monitor expected a heartbeat / health response and did not get one. The device may be down, the network path may be broken, or the runtime crashed.

→ Run

kolm devices ping prod-gpu-1
kolm devices health prod-gpu-1 --verbose
# If the host is reachable but the pod is down:
kolm fleet status --device prod-gpu-1
kolm deploy <last-good-art> --device prod-gpu-1
# If the host is unreachable, the monitor will keep retrying with backoff.

Kubernetes image pull or pod-pending failures

DeployError: k8s: pod stuck in ImagePullBackOff after 90s

The runtime image cannot be pulled (registry auth, network egress) or the pod cannot be scheduled (no node has the requested GPU). Common when a new cluster has not been wired with an image pull secret.

→ Run

kubectl describe pod -n <ns> <pod-name>            # exact reason
kubectl create secret docker-registry kolm-pull --docker-server=... --docker-username=... --docker-password=...
kolm devices update <k8s-device> --image-pull-secret kolm-pull
# If GPU scheduling is the issue:
kubectl get nodes -o json | jq '.items[].status.allocatable["nvidia.com/gpu"]'

Ollama device push fails

DeployError: ollama: Modelfile push failed (404 model not found after create)

The Ollama instance accepted the Modelfile but the model did not register. Usually a version skew (the GGUF format is newer than the Ollama runtime supports) or the artifact's GGUF blob was not uploaded before the Modelfile was applied.

→ Run

kolm devices runtime install <ollama-device> --runtime ollama --version latest
kolm deploy <art> --device <ollama-device> --verbose
# Look at the Ollama server logs on the host:
journalctl -u ollama -n 100 --no-pager

Docker daemon errors

DeployError: docker: dial unix /var/run/docker.sock: connect: permission denied

The user kolm ran devices add as does not have group access to the Docker socket on the host (or on the local box when type=docker, host=local).

→ Run

sudo usermod -aG docker $USER
# log out and back in for the group to take effect, then:
kolm devices ping <docker-device>
# Or run kolm as the docker group explicitly:
sg docker -c "kolm deploy <art> --device <docker-device>"

Rollback itself fails

RollbackError: previous artifact not on disk; upstream cache miss

The device evicted the previous artifact (small disk) and the upstream cache no longer has it. The rollback cannot proceed without bytes.

→ Run

kolm artifact list --remote                            # confirm the artifact is in the registry
kolm artifact pull <prev-art> --device prod-gpu-1      # re-seed the device
kolm deploy --rollback --device prod-gpu-1
# If the artifact is gone from the registry too, deploy the next-best known-good:
kolm deploy <art-from-3-deploys-ago> --device prod-gpu-1
Did this not cover your case? Run kolm deploy logs <deploy-id> --json and paste the failing step's stderr into a GitHub issue. The deploy id is printed at the start of every deploy attempt and stored in data/deploy-history.json.
