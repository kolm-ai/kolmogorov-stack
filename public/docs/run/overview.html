Run & Govern

Run a signed .kolm in production.

Run is the third surface of the compiler. Studio produces a signed .kolm; Run turns it into a serving pod that speaks the OpenAI Chat Completions API, exposes /health and /metrics, and writes a receipt for every request. The pod reports back to your tenant so the Serve dashboard can show what is live, where, and how it is performing.

Three ways in, one runtime

Run is reachable from the CLI, the TUI, and from the account UI. They all drive the same underlying kolm serve:

  • CLI. kolm serve <artifact> --port 8000 --runtime auto is the canonical entry point. Pair with kolm doctor first to verify the host has the runtime it needs.
  • TUI. kolm tui → key R opens a live pod table with the same fields as /account/serve.
  • Account UI. /account/serve lists every reporting pod across every host; /account/serve/new walks a new deployment.

What every pod emits

A reporting pod is one that calls home to your tenant on its check-in interval. The payload is the same regardless of where the pod runs:

{
  "artifact_id": "art_a1b2c3d4",
  "runtime": "vllm-0.6.3",
  "endpoint": "http://10.0.4.21:8000/v1",
  "health": "ok",
  "p50_ms": 92,
  "p99_ms": 318,
  "tokens_per_s": 41.2,
  "namespace": "prod"
}

The reporter ships in the runtime container and is enabled by default; opt-out with KOLM_REPORT=0 for fully air-gapped deployments. Air-gapped pods still write the same payload to $KOLM_DATA_DIR/serve-status.jsonl so a later sneakernet sync can roll the status up.

Hardware selection

The runtime auto-detects available accelerators (CUDA, ROCm, Metal, CPU) and picks the leanest runtime that fits the artifact's footprint. Override per-pod with --runtime vllm|sglang|tgi|llama-cpp|trt-llm. See /docs/run/hardware for the full matrix.

Speed flags

Five opt-in flags layer on top of the runtime to trade latency, throughput, and memory: --speculative, --kv-cache shard, --prompt-cache, --max-batch, and the bundled --optimize preset. Each is measurable with kolm bench --speed. Full reference at /docs/run/speed-optimization.

Health, metrics, receipts

  • /health. Liveness + readiness; returns the artifact id and runtime build so the orchestrator can pin the right scrape target.
  • /metrics. Prometheus exposition; includes kolm_requests_total, kolm_request_latency_ms, kolm_tokens_per_second, plus the standard runtime counters.
  • Receipts. Each request writes one row to the receipt log; the row's row_hash is returned in the x-kolm-receipt response header and is verifiable at /v1/verify/<rcpt_id>. See /docs/govern/receipts.
