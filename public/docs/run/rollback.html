Run & Govern

Rollback.

Every deploy records the artifact id it replaces. Rollback restores the previous artifact for the device or fleet in one command. Emergency stop bypasses graceful shutdown when a runaway pod is causing harm.

One-line example

kolm fleet rollback --select namespace=prod

Per-device rollback to the previous artifact for each device in the prod namespace. Runs the deploy pipeline in reverse: start old, smoke, swap traffic, stop new, write a rollback receipt.

Four shapes of rollback

VerbScopeUse when
kolm deploy --rollback --device <name>One device, one namespaceA specific device went bad after a partial deploy.
kolm fleet rollback --select <...>Many devices, per-device prevA bad artifact got propagated; restore each device's last-known-good.
kolm namespace rollback <ns>Every device in one namespaceYou promoted a bad artifact to prod; roll the whole namespace.
kolm fleet stop --emergency --select <...>Halt everythingActive harm in progress; stop now, fix offline, redeploy later.

Per-device rollback

# Rollback one device to its previous artifact (per its deploy history)
kolm deploy --rollback --device prod-gpu-1

# Rollback to a specific artifact (must be in the device's deploy history)
kolm deploy --rollback --device prod-gpu-1 --to art_a1b2c3d4

# Dry-run the rollback plan
kolm deploy --rollback --device prod-gpu-1 --dry-run

The same 6-step pipeline runs in reverse: preflight (is the previous artifact still on disk or cached upstream?) → download if needed → runtime ensure → start old pod → smoke → swap traffic in namespace → stop new pod → record. Total time per device is typically 5–30s depending on artifact size.

Fleet rollback

# Roll every device in the prod namespace
kolm fleet rollback --select namespace=prod

# Roll only the canary cohort (the devices that took the new artifact first)
kolm fleet rollback --select tag=prod-canary

# Per-device, in parallel (default 4 devices at once)
kolm fleet rollback --select tag=prod --parallel 8

# Output: per-device line + final summary
# prod-gpu-1   art_b2c3d4e5 -> art_a1b2c3d4  ok    4.1s
# prod-gpu-2   art_b2c3d4e5 -> art_a1b2c3d4  ok    3.8s
# prod-gpu-3   art_b2c3d4e5 -> art_z9y8x7w6  warn  9.2s (stale prev, downloaded)
# rollback: 3/3 ok / 0 fail

Namespace rollback

When you promoted a bad artifact to a namespace and want to revert the namespace pointer for every device:

kolm namespace rollback prod
# moves the namespace pointer back to the previous artifact id
# then runs fleet rollback for every device in the namespace
# emits one combined namespace-rollback receipt + per-device deploy receipts

Emergency stop

For active-harm scenarios (a pod is leaking PII, returning malformed responses to safety-critical clients, or crashing the host). Emergency stop skips graceful drain and SIGKILLs after 5s if SIGTERM is not honored.

# One device
kolm fleet stop --device prod-gpu-1 --emergency

# Whole tag
kolm fleet stop --select tag=prod --emergency

# Everything in the fleet -- last resort
kolm fleet stop --all --emergency

After an emergency stop the namespaces are unbound: traffic to that namespace returns HTTP 503 until a new deploy lands. Use kolm fleet status to confirm; restore service with kolm fleet deploy <previous-good-artifact>.

What gets logged

Every rollback writes:

  • One rollback receipt per device (rbk_018f2a...) with from_artifact_id, to_artifact_id, reason (if supplied via --reason), and the operator's identity.
  • One audit row per rollback action with the hash-chain pointer to the prior audit row.
  • For namespace and fleet rollbacks, one parent rollout receipt that references the per-device children.

Preventing accidental rollback

Use --reason + --confirm for production. Pair with kolm namespace config <ns> --rollback-require-reason true to make it a fleet-wide rule.

kolm fleet rollback --select namespace=prod \
  --reason "incident-2026-05-26-bad-tokenizer" \
  --confirm

kolm namespace config prod --rollback-require-reason true
Limitations. Rollback restores the binary previous artifact in deploy history; it does not roll back the runtime version or device config. If the rollback target requires an older runtime than the device currently has, pair with kolm devices runtime rollback (see runtimes). Emergency stop is for an active fire; it is not a clean shutdown and will drop in-flight requests.
