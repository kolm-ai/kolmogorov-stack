Run & Govern

Devices.

A device is any target kolm can deploy a signed .kolm artifact to. Register a device once, then any artifact or fleet command can reach it by id or tag. Hardware is auto-detected on first connect, refreshed on demand, and persisted in data/devices.json.

One-line example

kolm devices add prod-gpu-1 --type ssh --host 10.0.4.21 --user kolm --key ~/.ssh/id_ed25519 --tag prod --tag a100

That single command opens an SSH session, runs the hardware probe (nvidia-smi / rocm-smi / system_profiler), writes the result to the registry, and reports back. From now on kolm deploy <artifact> --device prod-gpu-1 just works.

Device types

TypeBacked byUse when
sshsrc/device-ssh.js (ssh2)Any Linux/Mac host you can reach over SSH. The default for bare-metal GPU servers and edge boxes.
k8ssrc/device-k8s.js (kubectl shell-out)You have a kubeconfig pointing at a cluster. Treats a context+namespace pair as one device.
dockersrc/device-docker.js (docker CLI)Local or remote Docker daemon. Pods become containers; cleanup is one docker rm.
ollamasrc/device-ollama.js (HTTP)You already run Ollama somewhere. kolm pushes the artifact via Modelfile, no SSH required.
localsrc/device-local.jsThe machine running the CLI. Always registered as local, no flags required.

kolm devices add — full flag surface

kolm devices add <name> --type <ssh|k8s|docker|ollama> [flags]

# SSH
  --host <ip-or-fqdn>         required for ssh
  --port <n>                  default 22
  --user <username>           default current user
  --key <path>                path to private key (never copied, only referenced)
  --password-stdin            read password from stdin (not recommended; key auth preferred)
  --jump-host <name>          chain through another registered ssh device

# Kubernetes
  --kubeconfig <path>         default ~/.kube/config
  --context <name>            kubeconfig context
  --namespace <name>          default 'default'

# Docker
  --docker-host <url>         e.g. tcp://10.0.4.21:2375 or unix:///var/run/docker.sock
  --tls-cert <path> --tls-key <path> --tls-ca <path>

# Ollama
  --ollama-url <url>          e.g. http://10.0.4.21:11434

# All types
  --tag <label>               repeatable; used by fleet selectors (--tag prod --tag a100)
  --namespace <name>          logical namespace this device serves (default 'default')
  --no-probe                  skip hardware auto-detect on add (defer to next 'devices refresh')
  --json                      print the resulting device row as JSON

Hardware auto-detect

On every add and every devices refresh, kolm runs a probe matched to the device OS and reports back:

  • NVIDIA. nvidia-smi --query-gpu=name,memory.total,driver_version --format=csv → GPU model, VRAM, driver.
  • AMD ROCm. rocm-smi --showproductname --showmeminfo vram.
  • Apple Silicon. system_profiler SPDisplaysDataType → chip family, unified memory total.
  • Intel + CPU fallback. lscpu / sysctl → cores, ISA, RAM.
  • Disk. df -BG on the deploy target path (default /var/lib/kolm).

The result lands at devices[].hardware in data/devices.json and feeds the runtime-selector in runtimes.

Tags and namespaces

Tags are a free-form label set per device. The fleet layer uses them as selectors:

kolm fleet deploy art_a1b2c3d4 --select tag=prod --select tag=a100
kolm fleet deploy art_a1b2c3d4 --select namespace=prod
kolm fleet status --select tag=edge

A namespace is the logical routing target. One namespace can have many devices; the gateway routes by namespace and the namespace fans out to whichever devices are healthy.

Four worked examples

1. Production GPU server

kolm devices add prod-gpu-1 \
  --type ssh --host 10.0.4.21 --user kolm \
  --key ~/.ssh/id_ed25519_kolm \
  --namespace prod --tag prod --tag a100

# Output:
# device prod-gpu-1 added
# hardware: NVIDIA A100-SXM4-80GB | 80GB VRAM | driver 535.183.01
# disk: 1.8TB free at /var/lib/kolm
# status: ready

2. Edge Jetson

kolm devices add edge-jetson-7 \
  --type ssh --host 192.168.50.7 --user nvidia \
  --key ~/.ssh/edge_fleet \
  --namespace edge --tag edge --tag jetson --tag orin

# Probe detects: NVIDIA Orin (8GB shared), JetPack 6.0, 256GB NVMe.
# kolm will prefer Q4/Q5 GGUFs on this device per the quant Pareto map.

3. Mac Mini (M-series)

kolm devices add mac-mini-dev \
  --type ssh --host 192.168.1.41 --user rodney \
  --namespace dev --tag mac --tag arm64

# Probe: Apple M4 Pro, 24GB unified, mlx-lm available.
# Runtime selector will pick mlx-lm for inference.

4. Kubernetes cluster

kolm devices add prod-k8s \
  --type k8s --kubeconfig ~/.kube/prod \
  --context prod-us-east-1 --namespace kolm \
  --tag prod --tag k8s

# Probe lists nodes + GPU pools via kubectl describe nodes.
# Deploys generate a Job + Service per artifact in the chosen ns.

Listing, health, removal

kolm devices list                  # tabular, --json available
kolm devices status <name>         # one device
kolm devices health                # ping all, report unreachable
kolm devices refresh <name>        # re-run hardware probe
kolm devices remove <name>         # purge from registry; no remote side-effect
Caveats. Removing a device only deletes the local registry row. Artifacts already deployed there keep running and keep reporting until you also kolm fleet stop --device <name>. SSH keys are never stored in the registry — only paths are referenced — so deleting a device does not touch your key material.
