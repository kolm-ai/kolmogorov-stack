Run & Govern

Monitoring.

A live pod exposes /health and /metrics; kolm scrapes them on each check-in interval and rolls the result into one device row. Configurable alerts fire on offline, crash, VRAM, and drift conditions. Everything is Prometheus-compatible.

One-line example

kolm devices health --watch
# refreshes every 30s; reports OFFLINE / OK / WARN / FAIL per device

/health

Every pod exposes /health on its bound port. Liveness + readiness in one endpoint, no separate /ready:

curl http://10.0.4.21:8000/health
{
  "status": "ok",
  "artifact_id": "art_a1b2c3d4",
  "runtime": "vllm-0.6.3",
  "uptime_s": 4218,
  "pending_requests": 0,
  "passport_match": true
}

kolm uses this for orchestrator probes, fleet status, and OTA pre-flight. kolm devices health rolls /health on every device into one table.

/metrics (Prometheus)

Standard Prometheus exposition. Scrape with any Prometheus-compatible agent — or let kolm's src/remote-metrics.js do it for you through an SSH tunnel.

# HELP kolm_requests_total HTTP requests served by the pod
# TYPE kolm_requests_total counter
kolm_requests_total{artifact="art_a1b2c3d4",namespace="prod"} 184127

# HELP kolm_request_latency_ms End-to-end request latency in ms
# TYPE kolm_request_latency_ms histogram
kolm_request_latency_ms_bucket{le="100",...} 91204
kolm_request_latency_ms_bucket{le="200",...} 162840
...

# HELP kolm_tokens_per_second Live throughput
# TYPE kolm_tokens_per_second gauge
kolm_tokens_per_second{artifact="art_a1b2c3d4"} 41.2

# HELP kolm_vram_used_mb VRAM in use by the pod
# TYPE kolm_vram_used_mb gauge
kolm_vram_used_mb{artifact="art_a1b2c3d4"} 5040

Continuous monitoring

The fleet monitor loop runs in the CLI when you start kolm fleet monitor, or as a long-lived daemon you can leave running. It scrapes each device on its check-in interval, evaluates alert rules, and writes results to ~/.kolm/events/events.sqlite (table device_health).

kolm fleet monitor --select tag=prod --interval 30s --json > health.ndjson &
# one NDJSON row per device per scrape

Alert configuration

Alerts are declared per device or per tag-selector. Set thresholds with kolm alerts set; list with kolm alerts list; pipe matches to webhooks (Slack, PagerDuty, email) via kolm config set alerts.webhook.

AlertDefault ruleWhat it catches
offlinemissed_checkins ≥ 3Device unreachable. SSH refused, host down, network partition.
crashpod exit ne 0Runtime panic, OOM, signal. The reporter logs the exit code before tearing down.
vramused > 95% for 60sMemory pressure. Likely an over-aggressive batch size or runaway request.
latencyp95 > 2× rolling 24h baselineLatency regression. Usually paired with a deploy — check kolm fleet status for the last cutover.
driftk_score drop > 3pp vs deploy baselineQuality regression from data drift. See drift detection.
error_rate5xx > 1% over 5mBackend errors. Often runtime mismatch or artifact corruption.

Setting alerts

# Per-device override
kolm alerts set prod-gpu-1 vram --threshold 85 --duration 30s

# Per-tag default
kolm alerts set --select tag=prod latency --p95-multiplier 1.5

# Disable a default
kolm alerts disable --select tag=edge offline   # noisy on intermittent edge links

# Route alerts somewhere
kolm config set alerts.webhook https://hooks.slack.com/services/...
kolm config set alerts.email   ops@example.com

Metrics collection from running artifacts

Two ways to get metrics off devices into your stack of choice:

  • Direct scrape. Your own Prometheus agent hits http://<device>:<port>/metrics. Works for any device that exposes the port to your monitoring network.
  • SSH tunnel scrape. kolm metrics scrape --device <name> opens a one-shot SSH tunnel, pulls /metrics, and prints to stdout — useful when the device port is firewalled from your monitoring network but reachable via SSH.
kolm metrics scrape --device prod-gpu-1 > prod-gpu-1.prom
kolm metrics scrape --select tag=prod --output dir ./scrape-out/
Limitations. /health answers liveness + readiness in one line; if you need separate startup vs readiness gates (k8s pattern), the orchestrator should map them itself. Drift alerts depend on the gateway sending a labeled traffic sample back to the device; air-gapped pods only catch drift if you wire a local eval cron. Webhooks are best-effort: on send failure kolm logs the failure and continues (no retry loop).
