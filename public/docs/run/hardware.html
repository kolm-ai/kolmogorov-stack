Run & Govern

Hardware & runtime matrix.

Pods served from a signed .kolm bind to the best runtime the host has installed. kolm doctor reports what is detected; kolm serve --runtime <name> overrides. This page documents the matrix the auto-selector walks.

What gets detected

On pod start the runtime probe walks the host in this order: NVIDIA CUDA → AMD ROCm → Apple Metal → Intel oneAPI → CPU. The first present accelerator wins; CPU is the always-available fallback. Probe results are cached in $KOLM_DATA_DIR/runtime-cache.json for 24h.

When the artifact's required format is not in the host's accepted set the pod fails closed with an actionable message: quant mismatch: artifact ships AWQ-INT4, host runtime llama-cpp expects GGUF; re-export with kolm export <art> --format gguf-q4_k_m or run on a CUDA host.

VRAM-first sizing

The selector also weighs the artifact's footprint against host VRAM. Artifacts manifest a min_vram_gb field; the selector subtracts kv-cache overhead per the runtime, then picks the lightest runtime that still fits:

kolm doctor --runtime
# host:       linux/x86_64
# accel:      cuda 12.4, 1 device (RTX 5090, 32 GB)
# detected:   vllm 0.6.3, sglang 0.3.5, llama-cpp 0.3.2
# selection:  vllm (artifact 18 GB + kv 8 GB = 26 GB, fits)

When to override

  • Single-GPU multi-pod. Run two artifacts on one card by forcing --runtime llama-cpp on the smaller one to free vllm's kv-cache reservation.
  • Latency floor. sglang often wins p99 on long contexts; vllm wins p50 on short turns.
  • BYOC parity. Mirror production by pinning the same runtime across dev/staging/prod even if the auto-selector would pick a different one locally.
