Run & Govern

Deploy.

One command pushes a signed .kolm to a registered device, ensures the runtime is present, starts the pod, runs a smoke test, and writes the result to the audit trail. The same command supports rolling, canary, and dry-run modes.

One-line example

kolm deploy art_a1b2c3d4 --device prod-gpu-1

That call does the full 6-step pipeline below. Every step is logged with a row id; the deploy record is the single source of truth that the Serve dashboard and kolm fleet status read from.

The 6-step pipeline

1

Preflight

Re-probe device hardware, check disk space against the artifact size, verify the artifact's runtime passport matches what the device can run, confirm the target port is free. Failure here exits with HTTP-style status and a → Run X suggestion.

2

Upload

The .kolm file is streamed over the device's transport (SSH/SFTP, kubectl cp, docker cp, or Ollama HTTP push). Resumable for SSH targets via partial-hash retry.

3

Runtime ensure

Check that the runtime named in the artifact's passport (vLLM, llama.cpp, Ollama, mlx-lm, TGI, TRT-LLM) is installed at a compatible version. If not, install per runtimes — or abort if --no-install.

4

Start

Spawn the pod as a systemd unit, k8s Job, Docker container, or Ollama model entry — whichever the device type calls for. The runtime binds the port and registers the namespace.

5

Smoke

Issue a hard-coded eval prompt over /v1/chat/completions, check the response shape, measure first-token latency. Failure rolls back to the previous artifact for the namespace.

6

Record

Write a deploy receipt: artifact_id, device_id, started_at, ended_at, status, smoke_latency_ms, passport_match. The row is signed and verifiable at /v1/verify/<deploy_id>.

Flags

FlagDefaultWhat it does
--device <name>—Required for single-target deploy. Use kolm fleet deploy for multi-target.
--namespace <name>artifact defaultOverride the namespace the pod registers under.
--port <n>auto-pickPin the port; defaults to the lowest free port at or above 8000.
--runtime <name>passportOverride the runtime; use only when the passport allows it.
--rollingoffZero-downtime cutover. Drains the previous pod before stopping it.
--canary <pct>offSend pct of traffic to the new pod for the monitor window; auto-rollback on regression.
--monitor <duration>10mCanary monitor window. Accepts s/m/h suffixes.
--dry-runoffRun preflight only; do not upload or start. Prints the plan as JSON.
--no-installoffFail step 3 instead of installing a missing runtime.
--no-smokeoffSkip the smoke test. Not recommended outside of recovery flows.
--jsonoffStream NDJSON per step; final line is the deploy receipt.

Three deploy modes

Plain replace

The default. Stop the previous pod for the namespace, start the new one. Traffic blip is bounded by step 4 + step 5 (typically 2–15 seconds depending on runtime warm-up).

kolm deploy art_b2c3d4e5 --device prod-gpu-1

Rolling

Use when the device runs multiple replicas of the same artifact (k8s, multi-pod ssh hosts). The new pod is brought up, smoke-tested, registered into the namespace, then the previous pod is drained and stopped. Zero requests dropped.

kolm deploy art_b2c3d4e5 --device prod-k8s --rolling

Canary

The new pod takes a fraction of traffic for the monitor window. The gateway compares K-Score / error rate / p95 against the previous pod; on regression it auto-routes back and the new pod is stopped.

kolm deploy art_b2c3d4e5 --device prod-gpu-1 --canary 10 --monitor 15m
# 10% traffic to new pod for 15 minutes
# auto-rollback if k_score drops >2pp OR p95 +30% OR error_rate > 1%

Dry-run

Validates the deploy plan without touching the device. Useful in CI to gate a release.

kolm deploy art_b2c3d4e5 --device prod-gpu-1 --dry-run --json
# emits a JSON plan: preflight checks, upload size, runtime delta, smoke prompt.

What gets logged

Every step writes one row to the local lake (~/.kolm/events/events.sqlite, table deploys) and one row to the audit trail (~/.kolm/audit/deploys.jsonl). For tenant-bound CLIs both also go to your tenant's deploy index, visible at /account/serve.

{
  "deploy_id": "dpl_018f2a...",
  "artifact_id": "art_b2c3d4e5",
  "device_id": "prod-gpu-1",
  "namespace": "prod",
  "mode": "canary",
  "started_at": "2026-05-26T17:22:01Z",
  "ended_at":   "2026-05-26T17:23:17Z",
  "status": "ok",
  "smoke_latency_ms": 184,
  "passport_match": true,
  "rollback_from": "art_a1b2c3d4"
}
Limitations. Canary requires that the gateway is in front of the namespace (not direct-to-pod clients). For direct-bound clients only rolling and plain replace are meaningful. The smoke test runs the artifact's declared evaluation prompt — not a tenant-supplied prompt — so a green smoke is necessary but not sufficient; rely on your own eval suite for promote/hold decisions.
