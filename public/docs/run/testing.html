Run & Govern

On-device testing.

Bench-style numbers from a developer laptop will mislead you on a Jetson. kolm test-device ships the artifact to the real target, runs a frozen eval, and reports tok/s, VRAM, K-Score, and latency from the device you actually plan to deploy to. kolm test-quants sweeps the quant ladder and returns the Pareto-optimal level for that device.

One-line example

kolm test-device art_a1b2c3d4 --device prod-gpu-1

That call uploads the artifact (if missing), boots a throwaway pod on an unused port, runs the frozen eval set, tears down, and prints a one-table report.

kolm test-device

Runs the same eval set every time per device so results are comparable across artifact versions. The set is the frozen-eval bundle shipped in the artifact's manifest, paired with kolm's standard multi-context tests (1k / 8k / 32k context windows).

kolm test-device <artifact> --device <name> [flags]
  --eval-set <name>        override the frozen set (rare)
  --context <n>            override context windows; default 1024,8192,32768
  --samples <n>            samples per prompt; default 5
  --warmup <n>             warm-up requests before measurement; default 3
  --port <n>               override the throwaway port
  --keep                   leave the pod running after the test
  --json                   machine output

What you get back:

MetricWhat it measures
tok_per_sSustained generation throughput at the device's optimal batch size.
ttft_p50_ms / ttft_p95_msTime-to-first-token, median + 95th percentile.
e2e_p50_ms / e2e_p95_msEnd-to-end request latency for the eval prompts.
vram_peak_mbPeak device memory used during the run.
k_scoreK-Score against the frozen eval (a quality number, see spec).
passport_okBoolean: did the runtime passport check pass on this device.

kolm test-quants — the Pareto frontier

test-quants takes one model and runs test-device for each quant level the device can host. It plots the K-Score vs tok/s vs VRAM Pareto frontier and recommends the level that maximises K-Score per byte under the device's VRAM budget.

kolm test-quants art_a1b2c3d4 --device prod-gpu-1 --ladder q4_k_m,q5_k_m,q6_k,q8_0,fp16

QUANT      K_SCORE   TOK/S   VRAM_MB   FITS  PARETO
q4_k_m     0.81      62.0    4180      yes   *
q5_k_m     0.83      54.1    5040      yes   *
q6_k       0.84      48.7    6020      yes
q8_0       0.85      32.9    8740      yes   *
fp16       0.86      18.4    16100     yes

recommended: q5_k_m  (K-Score 0.83 at 54.1 tok/s, 5GB VRAM)
reason: highest tok/s on the Pareto frontier above K-Score 0.82 budget

The ladder defaults to the device-appropriate set (e.g. a Jetson skips fp16; an A100 considers fp16 and bf16; a Mac Silicon adds the mlx 4-bit variant).

Cross-device comparison: --all-devices

Run the same artifact against every registered device in parallel; useful when picking deployment targets, sizing fleet, or sanity-checking that an artifact works everywhere.

kolm test-device art_a1b2c3d4 --all-devices --json > matrix.json

DEVICE          TOK/S   TTFT_P95  E2E_P95   VRAM_MB  K_SCORE
prod-gpu-1      54.1    98ms      612ms     5040     0.83
prod-gpu-2      55.0    96ms      598ms     5040     0.83
edge-jetson-7   12.4    310ms     1820ms    3120     0.81
mac-mini-dev    18.7    240ms     1100ms    4810     0.82
prod-k8s        49.8    104ms     640ms     5040     0.83

What gets measured (and what does not)

  • Measured. Throughput, latency (TTFT + e2e at p50/p95), VRAM peak, K-Score against the frozen eval set, runtime passport match.
  • Not measured. Long-tail behavior beyond the eval set, multi-tenant interference, network-path latency from real clients (use kolm bench for that).
Caveats. Numbers are reproducible per device (frozen eval set, fixed seeds, three warm-up requests) but absolute throughput on a shared host depends on other workloads; schedule on quiet windows for promotion decisions. K-Score is a relative quality signal on the frozen set, not a guarantee on your traffic — pair with your own evals on captured prompts.
