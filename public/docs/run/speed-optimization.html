Run & Govern

Speed optimization.

Five flags, one bench. Speculative decoding, Shard KV compression, prompt caching, continuous batching, and an NVFP4 path that gates itself on Blackwell silicon. Each one is opt-in and measurable. Defaults stay safe.

Scope. Every number here you can reproduce with kolm bench --speed against your own pod. Caveats are listed next to each flag rather than buried at the bottom — some of these (NVFP4 on non-Blackwell, Shard on vLLM) are net negative for throughput and the page says so plainly.

TL;DR

kolm serve art_a1b2c3d4 --port 8000 --optimize
kolm bench --speed --endpoint http://localhost:8000 --model art_a1b2c3d4 --n 30 --json

--optimize applies the hardware-aware defaults (speculative auto, Shard KV on if available, prompt cache on, batch size keyed to VRAM). Override any single dimension with the explicit flag below.

The five speed flags

FlagDefaultWhat it doesWhen to skip
--speculative auto|<model> off Runs a small draft model in lockstep with the target; accepted drafts skip the target forward pass. 1.5–2.4× tok/s on accepting workloads. Workload has near-zero overlap with the draft distribution (acceptance < 30% measured by kolm bench --speed · spec_accept_rate).
--kv-cache shard|standard standard Wraps the HF past_key_values in a Shard cache subclass; K-PCA + V-VQ256 yields ~10× KV memory compression at <1% quality loss in our measurements. Runtime is vLLM/SGLang/TRT-LLM. They use PagedAttention or their own KV; Shard only attaches to the bare HF generate loop. The serve loop logs a stderr warning and falls back to standard.
--prompt-cache auto|on|off auto Enables vLLM enable_prefix_caching. Repeated system prompts and few-shot prefixes hit the cache instead of recomputing. Every request is fresh and prefix overlap is near zero (RAG with rotating context). Cache fills RAM for no gain.
--max-batch N 8 vLLM max_num_seqs — the continuous-batching depth. Higher = more concurrent requests share one GPU pass; tail latency goes up at the head. Single-user latency-critical paths. Set to 1 for strict per-request TTFT.
--optimize off Probes hardware via kolm forge hardware and writes the four flags above to safe defaults. Explicit per-flag overrides win. You already know your tuned numbers and want full control. --optimize never overrides a flag the user passed.

Speculative decoding

The student-draft trick: a small fast model proposes k tokens, the target runs them in one forward pass, accepts the prefix that matches, and rolls back the rest. On accepting workloads you trade one large pass per k tokens for one large + several small. Net win: ~1.5–2.4× tok/s when acceptance > 60%.

# Auto-pick a draft from the same family
kolm serve art_qwen7b --speculative auto --port 8000

# Or name the draft explicitly
kolm serve art_qwen7b --speculative qwen2.5-0.5b-instruct --port 8000

# Tune the draft window
KOLM_NUM_SPECULATIVE_TOKENS=5 kolm serve art_qwen7b --speculative auto

# Measure acceptance
kolm bench --speed --endpoint http://localhost:8000 --json | jq .spec_accept_rate

Caveats. (a) The draft must share the same tokenizer family as the target — cross-tokenizer drafts silently corrupt output. --speculative auto refuses with an explanatory error rather than pick a wrong one. (b) Acceptance is workload-dependent; chat workloads typically land 55–75%, structured-output workloads 70–90%, free-form creative writing 30–50%. (c) The draft model must fit in VRAM alongside the target; kolm forge hardware reports headroom.

Shard KV-cache compression

Shard is a transformers.Cache subclass that compresses key tensors via per-head PCA and quantizes values to 256 centroids per layer. The serve loop allocates a fresh ShardCache(model.config) per request for concurrent safety; no cross-request key leakage. The info() envelope reports kv_cache_backend_requested and kv_cache_backend_active separately so a silent fallback (Shard missing, vLLM in use) is visible at /info.

# Opt in
kolm serve art_qwen7b --kv-cache shard --port 8000

# Verify it actually attached
curl -s http://localhost:8000/info | jq '{requested:.kv_cache_backend_requested, active:.kv_cache_backend_active}'
# { "requested": "shard", "active": "shard" }   <-- ok
# { "requested": "shard", "active": "standard" } <-- fell back, check serve stderr

Caveats. (a) HF transformers only. vLLM, SGLang, TGI, and TRT-LLM each ship their own KV manager and ignore past_key_values from outside; Shard cannot attach. (b) ~10× compression is from our measurements on Qwen2.5-7B at 4k context; long-context decay is non-monotonic in some layers and the next minor release will publish a per-layer fidelity matrix. (c) For very short prompts (< 256 tokens) the cache barely fills, so the compression overhead is a net loss; set --kv-cache standard for short-prompt workloads.

Prompt caching / KV reuse

vLLM's prefix cache stores computed KV slices keyed by token prefix. A request that shares the first n tokens with a cached entry starts decoding from token n instead of recomputing. Big win for fixed system prompts, few-shot examples, and tool-use templates.

# Default: auto (enabled when the runtime supports it)
kolm serve art_qwen7b --port 8000

# Force on (vLLM only)
kolm serve art_qwen7b --prompt-cache on --port 8000

# Force off (debug or memory-constrained)
kolm serve art_qwen7b --prompt-cache off --port 8000

Caveats. (a) vLLM only; the flag is a no-op on llama.cpp/TGI/SGLang/TRT-LLM (each has its own caching story, surfaced separately in the runtime adapter). (b) The cache lives in GPU memory and competes with model weights and the active KV. For tight-VRAM deployments, switch to off and recover the bytes. (c) The cache is per-process; an A/B with two pods does not share prefix work.

Continuous batching

vLLM's max_num_seqs is the upper bound on concurrent in-flight sequences. Raising it lets more requests share one forward-pass batch; the trade is per-request tail latency goes up when the batch is full and a new request has to wait for the next decode step.

# Default 8 — safe for 12-24 GB VRAM
kolm serve art_qwen7b --max-batch 8 --port 8000

# For 80 GB H100s with many lightweight requests
kolm serve art_qwen7b --max-batch 64 --port 8000

# Single-user latency-critical
kolm serve art_qwen7b --max-batch 1 --port 8000

Caveats. (a) Higher batch == more peak VRAM (KV per sequence). The serve loop refuses to start if the projected peak exceeds available VRAM and points you at --max-batch with a suggested number. (b) --max-batch only takes effect for vLLM; other runtimes have their own concurrency knobs (HF: --max-concurrency, llama.cpp: -np).

NVFP4 on Blackwell

NVFP4 is NVIDIA's 4-bit float format with native tensor-core support on sm_100+ (B100, B200, RTX 5090). On older silicon (sm_89 Ada, sm_90 Hopper) the modelopt path will export but the runtime emulates FP4 on FP16 cores — slower than FP8 in our measurements. The export gate refuses to run on non-Blackwell hardware unless you pass --force.

# Preview the gate without running the export
kolm export --target nvfp4 --preview art_qwen7b
# { "blackwell_gate": { "ok": true, "blackwell_native": true,
#                       "compute_capability": "12.0", "reason": "Blackwell native (cc 12.0)..." } }

# On Hopper / Ada this returns ok:false with hint to use FP8 instead
kolm export --target nvfp4 art_qwen7b
# error: not_blackwell_native — use --target fp8 or pass --force

# Override (for calibration / regression testing on non-Blackwell)
kolm export --target nvfp4 --force art_qwen7b

Caveats. (a) The gate is a Blackwell-native check via nvidia-smi --query-gpu=compute_cap; if nvidia-smi is missing the gate fails closed with vendor:unknown. (b) --force still produces a valid artifact, but the runtime passport records blackwell_gate.forced:true so downstream verifiers can see the artifact was calibrated outside its design envelope. (c) On Blackwell, expect NVFP4 to land between FP8 and INT4 on accuracy and ~1.5–2× faster than FP8 for matmul-bound workloads.

Measuring — kolm bench --speed

The benchmark hits any OpenAI-format SSE endpoint, captures time-to-first-token at the first non-empty content delta, and reports mean / p50 / p95 / p99 for TTFT, total latency, and tokens/sec.

kolm bench --speed \
  --endpoint http://localhost:8000 \
  --model art_qwen7b \
  --n 30 --warmup 3 --concurrency 1 \
  --max-tokens 64 \
  --prompt 'Explain backpropagation in two sentences.' \
  --json

Envelope:

Use it to A/B a single flag at a time. kolm bench --speed --compare baseline.json optimized.json emits a delta table.

Picking flags for your hardware

HardwareRecommendedNotes
RTX 5090 (Blackwell, 32 GB)--optimize --target nvfp4Native NVFP4. --optimize picks --max-batch 16.
RTX 4090 (Ada, 24 GB)--optimizeNVFP4 emulates — stay on FP8 or INT4. Shard on for HF, off for vLLM.
H100 (Hopper, 80 GB)--optimize --max-batch 32Plenty of VRAM for big batch + prompt cache. FP8 native.
L40S (Ada, 48 GB)--optimizeFP8 native. --optimize picks --max-batch 16.
Jetson AGX Orin (64 GB unified)--kv-cache shard --max-batch 4HF + Shard. vLLM does not run on Tegra.
Apple M2/M3 Max--runtime mlxSpeculative / Shard / NVFP4 are NVIDIA-path features. MLX has its own KV reuse.
CPU only--runtime llama-cpp -t $(nproc)Speed flags above are GPU-only. GGUF + llama.cpp is the speed path on CPU.
