Run & Govern

OTA updates.

A new artifact lands in your registry; what should the fleet do? The OTA update policy answers that per namespace. Pick from manual, notify, canary, rolling, or immediate — pinned in config and overridable per fleet command.

One-line example

kolm namespace config prod --auto-update canary

From now on, any new artifact promoted to the prod namespace runs the canary flow automatically across every device in that namespace: 10% of devices, 15-minute monitor, auto-rollback on regression, expand on green.

The five policies

PolicyWhat it doesUse when
manualkolm does nothing. The new artifact sits in the registry until an operator runs kolm fleet deploy.Critical production, regulated environments, anything where a human must sign the deploy receipt.
notifykolm notifies via the configured webhook/email and creates a pending update entry. No traffic moves.You want a human-in-the-loop but with low-friction triggers (e.g. one-click promote from a Slack alert).
canaryRuns the canary flow automatically: fraction first, monitor window, auto-rollback on regression, expand on green.Most production fleets. Catches regressions without operator attention.
rollingZero-downtime rolling deploy across the fleet. No canary fraction; the assumption is the artifact is pre-vetted.Pre-vetted artifacts (e.g. promoted from a staging fleet that already ran canary).
immediateReplace on every device in parallel; no rolling, no canary. Brief traffic blip per device.Dev or edge fleets where blip is tolerable and rollout speed matters.

The update flow

  1. New artifact arrives. A new .kolm is published to your tenant's registry (either via kolm publish, a build job, or a fleet sync).
  2. Policy evaluates. For each namespace, kolm checks auto-update and matches the new artifact against the namespace's last deployed artifact id.
  3. Action fires. manual → nothing. notify → webhook+email. canary/rolling/immediate → the appropriate kolm fleet deploy with the right flags.
  4. Receipts written. Every device gets a deploy receipt; the rollout itself gets a rollout receipt with the policy name and outcome.

Setting the policy

# Per-namespace
kolm namespace config prod    --auto-update canary
kolm namespace config staging --auto-update rolling
kolm namespace config edge    --auto-update notify
kolm namespace config dev     --auto-update immediate
kolm namespace config gov     --auto-update manual

# View
kolm namespace config prod
# auto-update: canary
# canary-fraction: 10
# canary-monitor: 15m
# rollback-on: k_score-2pp, p95+30, error-1pct

# Per-rollout override (one-off, does not change the persisted policy)
kolm fleet deploy art_b2c3d4e5 --namespace prod --policy immediate

Tuning canary defaults

The canary policy is configurable per namespace. Pick a fraction (devices, not traffic, for fleet canary), a monitor window, and regression thresholds.

kolm namespace config prod \
  --auto-update canary \
  --canary-fraction 20 \
  --canary-monitor 30m \
  --rollback-k-score 3 \
  --rollback-p95-pct 30 \
  --rollback-error-pct 1

When to use each

  • manual — regulated production where the deploy itself is the audit event.
  • notify — small ops teams who want one-click control from chat.
  • canary — default for most production fleets — cheapest insurance against silent regressions.
  • rolling — staging or post-canary environments where the artifact is already trusted.
  • immediate — dev and edge fleets where rollout speed matters and a few seconds of blip per device is fine.
Constraints. OTA fires when a new artifact lands in your tenant registry. Air-gapped fleets do not receive OTA notifications — you sneakernet the artifact in, then run kolm fleet deploy on the air-gapped operator host. Canary needs at least one device to remain on the previous artifact for the monitor window; on a single-device namespace canary degrades to per-device split-traffic if the gateway is in front, otherwise to rolling.
