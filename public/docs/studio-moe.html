Studio / MoE

Mixture-of-experts is a different shape. Studio handles it.

MoE models (Mixtral, DeepSeek-V3, Qwen3-MoE, Llama-4-Maverick) replace the dense FFN with a router that picks top-K of N experts per token. Distilling, quantizing, and serving them needs MoE-aware code paths — treating them as dense costs you 4-8 K-Score and 2× the VRAM. The MoE pipeline lives in src/moe.js + src/moe-support.js.

Detection

Studio detects MoE from config.json at inspect time. The fields the detector reads:

  • num_experts or n_routed_experts — total experts per MoE layer.
  • num_experts_per_tok or n_active_experts — top-K activated per token.
  • shared_expert_intermediate_size — if present, model has a dense shared expert in parallel.
  • router_aux_loss_coef — load-balance loss coefficient.
kolm inspect mistralai/Mixtral-8x7B-Instruct-v0.1 --json
# {
#   "architecture": "moe",
#   "num_experts": 8,
#   "num_experts_per_tok": 2,
#   "total_params_b": 46.7,
#   "active_params_b": 12.9,
#   "shared_expert": false,
#   ...
# }

MoE-aware LoRA

For dense models, the LoRA target list is q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj. For MoE, that target list misses the expert-specific projections and over-targets the router. Studio's MoE-aware default targets:

  • Attention — q_proj, k_proj, v_proj, o_proj (same as dense).
  • Router — not targeted (the router is small and fragile; LoRA tends to break load balance).
  • Per-expert — experts.<i>.{w1, w2, w3} or experts.<i>.{gate_proj, up_proj, down_proj} for top-N most-active experts.
  • Shared expert (if present) — shared_expert.{gate_proj, up_proj, down_proj}.

Set the active-expert threshold with --moe-lora-top-pct 50 — target only experts that fire on at least 50% of training tokens. The default is 100% (every expert), but on Mixtral-8x7B-Instruct, two experts handle ~70% of code tokens; targeting only those cuts adapter size by ~75% with no quality loss on a code recipe.

MoE-aware quant policy

Uniformly quantizing an MoE model to Q4_K_M drops ~5 K-Score because the router and the gate projections are sensitive. Studio's router-high policy keeps routers + gate at Q6_K minimum while experts go to the requested bpw.

Layer classDefault policy
Router (gate)Q6_K minimum (never quantize below)
Attention QKVOSame as user bpw (Q4_K_M, etc.)
Expert w1 / gate_projSame as user bpw
Expert w2 / down_projSame as user bpw (most numerically sensitive)
Expert w3 / up_projSame as user bpw
Shared expert (if present)One step higher than expert bpw
 
# Default: router-high
kolm compile mixtral-distill.kolm --target gguf-q4km --output mixtral-distill-q4km.gguf

# Override: uniform Q4 everywhere (smallest file)
kolm compile mixtral-distill.kolm --target gguf-q4km --moe-policy uniform --output small.gguf

# Override: router-very-high (Q8 routers)
kolm compile mixtral-distill.kolm --target gguf-q4km --moe-policy router-very-high --output safe.gguf

Expert activation profiling

kolm experts <artifact> runs the eval set through the model and emits the activation distribution per expert per layer. Use this to spot dead experts (never fire), star experts (fire on >50% of tokens in their domain), and load-balance pathologies.

kolm experts mixtral-distill.kolm --eval data/code-eval.jsonl --json
# {
#   "layer_0": { "expert_0": 0.143, "expert_1": 0.071, "expert_2": 0.024, "expert_3": 0.581, ... },
#   "layer_1": { ... },
#   ...
#   "dead_experts": [],
#   "star_experts": [{ "layer": 0, "expert": 3, "fire_rate": 0.581 }, ...]
# }

Pruning low-traffic experts

If an expert fires on <1% of your domain's tokens, pruning it (replacing its weights with the mean of the other experts, or removing it from the router) saves 1/N of the expert VRAM with negligible quality impact. The pruner reports K-Score before-and-after; if ΔK-Score > 0.3, the prune is rolled back.

# Prune experts firing on <0.5% of eval tokens
kolm experts mixtral-distill.kolm --prune-below 0.005 --output mixtral-distill-pruned.kolm
# wrote: pruned 2 experts (layer_0.expert_6, layer_15.expert_2), 1/4 VRAM saved, ΔK-Score = -0.04

MoE memory estimation

VRAM for an MoE model is the full parameter count, not the active parameter count — all experts need to be resident (CPU offload is possible but slow). The memory estimator accounts for this:

ModelTotal paramsActive paramsVRAM at Q4_K_M (router-high)
Mixtral-8x7B46.7 B12.9 B~28 GB
Mixtral-8x22B141 B39 B~84 GB
DeepSeek-V3671 B37 B~420 GB (multi-GPU only)
Qwen3-MoE-30B-A3B30 B3 B~18 GB
Caveats. Expert pinning (forcing specific experts active per call) is research-grade and not enabled by default; it usually breaks the model. Pruning is safer and almost always preferable. If you genuinely need a single-expert path (e.g., for compliance reasons), open an issue — the API is in src/moe-support.js but the validator currently warns.
