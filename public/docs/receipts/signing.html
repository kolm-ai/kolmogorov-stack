Receipt signing

Ed25519 keys with a 90-day rotation cadence.

Generate the keypair with kolm key generate. The private half lives at ~/.kolm/signing-key.pem with mode 0600 by default; override via env var. Rotation is 90 days with a 30-day overlap window so verifiers do not break.

Generating a key

The CLI generates an Ed25519 keypair, writes the private key (PKCS#8 PEM) to disk with mode 0600, and registers the public key with the hosted gateway. For self-hosted gateways, the public key needs to be served at /v1/keys/<key_id>; the CLI writes it to ~/.kolm/public-keys/<key_id>.pub for upload to your gateway.

# Generate (interactive: confirms overwrite if a key exists)
kolm key generate

# Generate with a specific key id (defaults to <tenant>_<short-random>)
kolm key generate --key-id tenant_acme_v3

# Generate to a specific path
kolm key generate --out /etc/kolm/signing.ed25519 --key-id tenant_acme_v3

# Inspect the current key (public half + metadata only; never prints private)
kolm key show
# key_id:     tenant_acme_v3
# algorithm:  Ed25519
# created:    2026-04-15T12:00:00Z
# rotates:    2026-07-14T12:00:00Z (90 days)
# public key: base64url:M+oS...vN

Storage location and permissions

The default path is ~/.kolm/signing-key.pem on Linux / macOS. Windows: %USERPROFILE%\.kolm\signing-key.pem. Mode is 0600 on POSIX (owner read/write only); the CLI refuses to load a key with looser permissions unless --allow-loose-perms is passed. Windows: the CLI sets the ACL to current user only.

PlatformDefault pathMode
Linux / macOS~/.kolm/signing-key.pem0600
Windows%USERPROFILE%\.kolm\signing-key.pemACL: current user only
Docker self-hostmount to /etc/kolm/signing.ed25519image expects 0400

Environment overrides

Two env vars override the disk file. Useful for cloud-native deployments where the key comes from a secret manager (AWS Secrets Manager, GCP Secret Manager, Vault).

# Cloud deployment: inject key from secret manager
export KOLM_ED25519_PRIVATE_KEY="$(aws secretsmanager get-secret-value --secret-id kolm/signing --query SecretString --output text)"

# Or by path
export KOLM_ED25519_PRIVATE_KEY_PATH=/var/run/secrets/kolm/signing-key.pem
Env varTypeEffect
KOLM_ED25519_PRIVATE_KEYPEM string (literal)Used directly; precedence over file.
KOLM_ED25519_PRIVATE_KEY_PATHfilesystem pathRead from this path instead of the default.

The 90-day rotation cadence

NIST SP 800-57 recommends rotating asymmetric signing keys on a periodic basis; the kolm default is 90 days. The rotation generates a new keypair, marks the old key as superseded but still valid for the 30-day overlap window, and starts signing new receipts with the new key. Receipts signed during the overlap remain verifiable against the old public key for the lifetime of the receipt.

# Manual rotation
kolm key rotate
# Generated tenant_acme_v4 (active 2026-07-14T00:00:00Z onwards)
# Overlap window: tenant_acme_v3 valid for verifiers until 2026-08-13T00:00:00Z
# New receipts will be signed with tenant_acme_v4.

# Scheduled rotation (cron-style)
kolm key rotate --schedule "90d --overlap 30d"
# Sets a self-rotation cadence; the gateway issues a new key automatically on the cadence.

The 30-day overlap window

When key v3 retires at day 90, key v4 has been live for 60 days. Verifiers can request either public key from /v1/keys/<key_id>; each receipt carries its signing_key_id so the right key is always retrievable. The overlap protects long-lived deployments where some verifier hasn’t yet refreshed its cached key list.

# Timeline
day 0:    v3 active.   v4 not yet generated.
day 60:   v3 active.   v4 generated, dormant.
day 90:   v3 retired.  v4 takes over (new receipts use v4).
day 120:  v3 still valid for verifying old receipts (overlap window).
day 121+: v3 public key removed from the keys endpoint. Old receipts no longer verifiable.

# Operator action at day 120: confirm no live verifier still has v3 cached past expiry

Key revocation

If a key is compromised, revoke it immediately. Revocation removes the public key from the keys endpoint and adds the key id to a published revocation list. Receipts signed before the revocation timestamp remain verifiable if the revoker chooses; new verifications against revoked keys fail by default.

kolm key revoke tenant_acme_v3 --reason "suspected compromise" --revoke-pre true
# Revocation effective 2026-06-12T10:30:00Z
# Pre-revocation receipts: --revoke-pre true (no longer verifiable, fail-closed)
# Pre-revocation receipts: --revoke-pre false (still verifiable; document the trust assumption)
Caveats. Lose the private key, lose the ability to sign new receipts under that key id. Past receipts remain verifiable as long as the public key is published. Back up the private key to your secret manager before rotation; the CLI refuses to delete a key file unless --force is passed. The rotation overlap of 30 days is the longest live verifier should ever need to converge; if you have longer-tail verifiers (regulatory archive readers), lengthen the overlap explicitly.
