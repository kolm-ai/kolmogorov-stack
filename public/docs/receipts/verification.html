Receipt verification

Online or offline, the same gates fire.

kolm receipts verify <id> walks five gates: schema match, CID match, signature validity, rate-card reconciliation, capture-row linkage. --offline uses a cached public key and skips the network fetch.

The verb

One verb covers both modes. The online mode fetches the receipt and the public key from kolm.ai (or your self-hosted gateway); the offline mode reads both from local files.

# Online: fetch receipt + public key over HTTPS, verify
kolm receipts verify bafy2bzaceabc...

# Offline: receipt body on disk, public key cached in ~/.kolm/keys/
kolm receipts verify --offline ./receipt.json

# Verify a batch
kolm receipts verify --batch ./receipts.jsonl --json | jq '.summary'
# { "total": 1248, "ok": 1248, "failed": 0, "elapsed_ms": 4231 }

The five gates

Each gate runs in order. The verb exits non-zero on the first failure and reports which gate failed and why. --continue-on-fail runs all gates and reports a summary.

GateWhat it checksFailure mode
1. schemaReceipt schema is one the verifier understands (kolm-audit-1 or a compatible shim).Unknown schema or version drift.
2. cidreceipt_id equals the recomputed CID over the unsigned body.Receipt body was edited after signing.
3. signatureEd25519 signature validates against the public key resolved from signing_key_id.Body tampered or wrong key.
4. rate-cardcost_usd reconciles with tokens × rate card at timestamp.Cost drifted from the rate card.
5. capture-linkageWhen capture_id is set, the capture row exists and its hash matches output_hash.Capture row missing or modified.

Expected output

kolm receipts verify bafy2bzaceabc...
# Fetching receipt... OK (1247 bytes)
# Fetching public key tenant_acme_v3... OK (cached for 24h)
# [OK] gate 1/5: schema = kolm-audit-1
# [OK] gate 2/5: receipt_id matches CID over unsigned body
# [OK] gate 3/5: Ed25519 signature valid (key_id=tenant_acme_v3)
# [OK] gate 4/5: cost_usd reconciles with rate card 2026-05-15 (delta=0.00000)
# [OK] gate 5/5: capture row cap_01HXYZ... present, output_hash matches
# Receipt verified. Exit code: 0

Online mode details

Online mode is the default. The verb resolves signing_key_id by GET /v1/keys/<id>; the response is cached in ~/.kolm/keys/ for 24 hours by default (override via --key-cache-ttl). The capture-linkage gate fetches the capture by id from the lake.

Network failures degrade gracefully: gate 3 (signature) still runs against a cached key when available; gate 5 (capture-linkage) is skipped with a warning when the lake is unreachable.

Offline mode details

Offline mode reads everything from disk. The public key must already exist in ~/.kolm/keys/<key_id>.pub (run kolm keys fetch <key_id> once in advance). Gate 5 (capture-linkage) requires a local capture-lake snapshot; without it, the gate is skipped with a warning.

# One-time key prefetch (online)
kolm keys fetch tenant_acme_v3
# Saved ~/.kolm/keys/tenant_acme_v3.pub (32 bytes)

# Now verify offline anywhere (CI, air-gapped audit, etc.)
kolm receipts verify --offline ./receipt.json
# [OK] gates 1-4 of 5
# [SKIP] gate 5: capture row lookup requires lake; pass --lake ./lake-snapshot.db
kolm receipts verify --offline --lake ./lake-snapshot.db ./receipt.json
# [OK] gates 1-5 of 5

Common failure modes

SymptomLikely causeAction
Gate 2 fails (cid mismatch)Receipt body was reformatted or edited.Re-fetch from /v1/receipts/<cid>; do not edit receipts manually.
Gate 3 fails (signature)Wrong key, or body tampered.Confirm signing_key_id; if right, body was modified.
Gate 4 fails (rate-card)Provider rate-card change after the call; verifier rate card is newer.Pin verifier rate card with --rate-card <path>; rate cards are versioned.
Gate 5 fails (capture)Capture row was purged, deleted, or its hash drifted.If retention purge expected, ignore with --allow-purged; if not, investigate the lake.

Verifier in CI

A common pattern is to verify a sampled batch of receipts in CI nightly. The exit code drives the build status; the JSON output feeds a dashboard.

# In your CI pipeline
kolm receipts list --namespace customer-support --since 24h --sample 100 --json | jq -r '.[].receipt_id' > /tmp/sample.txt
kolm receipts verify --batch /tmp/sample.txt --json > /tmp/verify.json
# Fail the build on any failure
test "$(jq '.summary.failed' /tmp/verify.json)" -eq 0
Caveats. Verification proves the receipt is consistent; it does not validate the response semantics. A signed receipt with a bad response is still a valid receipt. Pair with bake-offs and held-out evals for quality. The verifier itself is reproducible: same receipt + same public key + same rate card yields the same gate output.
