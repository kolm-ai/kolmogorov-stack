Receipts

One signed receipt per call. Verifiable forever.

A receipt is the cryptographic record of one gateway call: who, when, what was asked, what was returned, which model, how much it cost, and the Ed25519 signature that proves none of it was tampered with after the fact.

What a receipt is

When a request finishes the wrapper pipeline, stage 8 computes a receipt and signs it with the tenant Ed25519 key. The receipt is stored on the capture row, returned to the caller as response header X-Kolm-Receipt-CID, and made fetchable at https://kolm.ai/v1/receipts/<cid>.

The receipt is the unit of verifiability. Anyone with the tenant public key can fetch a receipt and confirm: this exact request was handled by this exact model at this exact time, the response bytes match the recorded hash, the cost was computed against this rate card.

The schema name

Every receipt is tagged schema: "kolm-audit-1". Future schema versions will bump the suffix and remain readable through verifier compatibility shims. See format for the full 19-field reference.

Receipt lifecycle

  1. Pipeline stage 8 builds the receipt body from the in-progress call (provider, model, tokens, cost, route decision, redaction summary).
  2. The receipt body is canonicalized (sorted JSON, NFC) and signed with the tenant Ed25519 key.
  3. The receipt is stored on the capture row in the lake.
  4. The receipt CID is returned to the caller as X-Kolm-Receipt-CID header.
  5. A copy is fetchable at /v1/receipts/<cid> for the lifetime of the capture (subject to retention).

What is signed

The signature covers every field in the receipt except the signature itself. That includes the input hash and the output hash, so any post-hoc edit to the prompt or response stored on the capture row breaks the receipt verification.

Verify URL

The receipt body includes verify_url pointing to a public verification page. Paste a CID into verify.kolm.ai and the page fetches the receipt, fetches the tenant public key from /v1/keys/<key_id>, and confirms the signature in the browser. Useful for sharing a receipt with an auditor who does not have the CLI installed.

# Header on every response
X-Kolm-Receipt-CID: bafy2bzaceabcdefghijklmnopqrstuvwxyz1234567890

# Fetch the receipt body
curl -s https://kolm.ai/v1/receipts/bafy2bzaceabc... | jq .

# Verify on the public page
open https://verify.kolm.ai/?cid=bafy2bzaceabc...

What receipts enable

  • Compliance audits — export a slice of receipts for any reviewer (SOC 2, HIPAA, internal). They can verify without access to the lake.
  • Customer disputes — "show me what the model actually told user X on date Y" answered by fetching the receipt CID for that capture.
  • Regulatory disclosure — the receipt is the ground truth for any "what did the AI say" question.
  • Cost reconciliation — the signed cost on the receipt is the source of truth for chargeback.
Caveats. A receipt proves the call happened and the recorded fields are intact. It does not prove the upstream provider returned the answer correctly — the gateway records what it received, not whether the model was right. Pair with bake-offs (replay against another provider) and held-out evals for quality assurance.
