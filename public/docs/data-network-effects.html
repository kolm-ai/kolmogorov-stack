Data network effects — opt-in pattern lake.

Every kolm installation produces signal about how AI is used in the real world. The pattern lake lets opted-in installations contribute that signal back as hash-only bigram fingerprints, never raw text. The collective fingerprint helps the vertical foundation students (kolm-legal-7b, kolm-medical-7b, kolm-code-7b, kolm-finance-7b, kolm-support-7b) learn the shape of each industry without anyone shipping prompts off-prem.

w757-v1 disabled by default

1. Why a hash-only lake (privacy by design)

The classic "network effect" pitch — everyone benefits when everyone contributes data — collapses the moment privacy enters the room. Most enterprise prompts contain PII, contract drafts, patient summaries, or proprietary code. Shipping those to a central aggregator is a non-starter.

The lake takes the opposite shape: it never sees raw text. The only payload that crosses your tenant boundary is a list of sha256(bigram) hashes — a one-way fingerprint that lets the aggregator count how often a given two-word pattern appears across opted-in installations, without ever being able to reverse the hash back to the prompt.

Consent is binding. Every contribution must carry an explicit consent:true flag. The lake throws consent_not_granted on missing consent and refuses the write. There is no default-on path; opting in is an explicit operator action, per namespace.

2. What enters the lake

  • Bigram SHA256 hashes — adjacent token pairs from your capture, hashed before persistence. Never the raw token, never the joined prompt.
  • Capture id + source namespace — so duplicate writes are idempotent (we never re-hash the same row twice).
  • Timestamp + lake version stamp (w757-v1).

What does not enter the lake: prompt text, completion text, tool calls, user identity, request hash, or any payload that could be used to re-identify the original capture. The contribution row in the event-store is a tight envelope; tests/wave757-pattern-lake.test.js pins that no raw substring survives the tokenizer.

4. Differential privacy guarantee

Cross-tenant aggregates pass through a Laplace mechanism before they leave the aggregation surface. The default epsilon is 1.0; the floor is 0.1. An aggregate query with epsilon < 0.1 throws epsilon_below_floor — we never publish a weaker noise level than the floor.

Mechanism: laplace_v1. Noise distribution is Laplace(0, sensitivity / epsilon) where sensitivity defaults to 1 (count queries). Negative counts are clamped at zero. The mechanism stamp travels with every aggregate response so consumers can branch on the exact algorithm. A future Gaussian or RAPPOR mechanism would carry a different stamp and a different contract.

A separate, stronger privacy claim — the hash-only contribution surface above — means the DP layer is the belt over the suspenders. The lake's primary guarantee is that raw text never crosses the tenant boundary; DP is the second-order protection against count-based reconstruction attacks.

5. How vertical fingerprints unlock

The five vertical foundation students each need a fingerprint of how their industry actually uses AI — what bigrams dominate in legal prompts versus medical prompts versus support prompts. With the lake disabled, the fingerprint surface returns an plain insufficient_lake_data envelope. With the lake enabled and at least 5 opted-in contributors per vertical, the fingerprint surface returns the top-K bigram hashes the vertical's student should learn to predict.

6. Caveats note

Lake disabled by default. Set KOLM_W757_LAKE_ENABLED=1 to opt your installation in. No data crosses tenant boundaries without consent. The opt-in registry is append-only and per-namespace; opting a namespace in does NOT implicitly opt other namespaces in. Opt-out is durable and observed by all future reads.
