One-click marketplace import.

Browse /marketplace, click Run with kolm on any listing. The SDK downloads the signed .kolm archive, verifies the signature against the publisher's key chain, and starts the runtime. The model is callable from your code in three lines.

Signature verification is mandatory and refuses to fall through. If the marketplace listing's signature does not match the publisher's chain, the SDK throws before any weights are mmaped. There is no "open anyway" path; the runtime never serves an unverifiable artifact.

Step 1: browse the marketplace

1.

Find a listing

Open /marketplace, filter by use case (support, billing, doc-search, etc.), inspect the publisher's K-Score history, latency profile, and license. Each listing carries its full provenance receipt; if anything is missing, it is missing for a reason and the listing says so.

2.

Click Run with kolm

The button returns a stable kolm://sha256-... URI. Copy it once or hand it straight to the SDK as shown below.

Step 2: SDK downloads + verifies

The SDK pull call resolves the kolm:// URI against the kolm registry, downloads the archive, and verifies the signature against the publisher's cert chain. The same call returns the artifact path on disk and a verify_ok boolean.

Both SDKs use environment variables to read the API key. The fields are deliberately the same so a config block can be shared across languages.

Step 3: SDK starts the runtime

Once the artifact is verified, the SDK starts the kolm runtime in-process and binds the model to a stable handle. Subsequent calls are normal function calls; there is no separate "start server" step.

The runtime sits in the same process as your code, so latency is local-call latency (typically <1ms overhead on top of the model's own inference time). For long-running services, the runtime is kept warm across requests; for one-shot scripts it spins down when the SDK handle goes out of scope.

Node sample

Install the SDK once:

npm install @kolmogorov/sdk

Then in your code:

import KolmClient from '@kolmogorov/sdk';

const kolm = new KolmClient({
  apiKey: process.env.KOLM_API_KEY,
  baseUrl: process.env.KOLM_BASE_URL || 'https://kolm.ai',
});

// Pull + verify + cache in ~/.kolm/artifacts/.
// Throws KolmError if signature verification fails.
const artifact = await kolm.pull('kolm://sha256-abc123...');
console.log(artifact.verify_ok);  // => true

// Start the in-process runtime for this artifact.
const runtime = await kolm.runtime(artifact);

// Plain function call from here on.
const out = await runtime.run({ input: "user can't log in" });
console.log(out.text);
console.log(out.latency_us);

Environment variables follow the kolm convention. KOLM_API_KEY is required for private listings; public listings work without it. KOLM_BASE_URL override is honoured for self-hosted installs.

Python sample

Install the SDK once:

pip install git+https://github.com/kolm-ai/kolmogorov-stack@main#subdirectory=sdk/python

Then in your code:

import os
from kolm import Kolm

k = Kolm(
    api_key=os.environ.get("KOLM_API_KEY"),
    base_url=os.environ.get("KOLM_BASE_URL", "https://kolm.ai"),
)

# Pull + verify + cache in ~/.kolm/artifacts/.
# Raises KolmError if signature verification fails.
artifact = k.pull("kolm://sha256-abc123...")
print(artifact.verify_ok)  # => True

# Start the in-process runtime for this artifact.
runtime = k.runtime(artifact)

# Plain function call from here on.
out = runtime.run(input="user can't log in")
print(out.text)
print(out.latency_us)

The Python client uses the same envelope shape as the Node client: artifact.verify_ok, artifact.path, artifact.manifest, and out.text, out.latency_us, out.kscore. Non-2xx responses raise KolmError(status, body) verbatim; nothing is silently swallowed.

Caching

The first pull() downloads. Subsequent pull() calls for the same hash are no-ops; the SDK reads from ~/.kolm/artifacts/<hash>/. The cache survives process restart and is shared across SDKs (the Node, Python, Rust, and C SDKs all read the same on-disk layout).

For air-gapped workflows: kolm pull on the connected machine, copy the ~/.kolm/artifacts/<hash>/ directory to the air-gapped target, and kolm runtime there. The SDK never re-fetches when the cached hash matches the requested URI.

Honest failure modes

  • Signature mismatch. The pull call throws KolmError(code: 'signature_invalid') before any bytes are loaded. The artifact stays unpacked but is never made callable. Investigate the publisher chain via kolm verify --explain.
  • Network unreachable. The pull call throws KolmError(code: 'pull_network') with the upstream error verbatim. The SDK does not retry; the operator decides whether to back off or fail loud.
  • Unknown URI scheme. Only kolm://, file://, and github:// are recognised. Anything else throws KolmError(code: 'unknown_scheme').
  • Private listing without API key. The marketplace returns HTTP 401; the SDK surfaces it as KolmError(401, 'auth_required'). There is no anonymous fallback for private listings.
The "verify ok" boolean is the gate. The SDK refuses to expose runtime() on an artifact where verify_ok is false; trying it raises immediately. Treat the boolean as a hard precondition rather than a hint.
