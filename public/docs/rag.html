RAG-aware distillation

Most LLM apps that work are RAG apps. Capturing only prompt and response throws away the most load-bearing input: the chunks the retriever pulled from your source-of-truth corpus. kolm.ai lets you capture all three so the distilled student learns to attend to the retrieved context, not just memorise canned answers.

Why RAG-aware distillation

A distill pipeline that sees only {prompt, response} learns a closed-book version of your task. The student memorises that "What is our refund policy?" maps to a specific paragraph, then drifts the moment the underlying policy changes. The retrieved context is the bridge: with {prompt, retrieved, response} the student learns the shape of "given these chunks, produce a response grounded in them" and generalises across new versions of the corpus.

The W734 capture path persists the retrieved chunks alongside prompt + response on every capture row. At distill time, the W734-2 training-data formatter prefixes each chunk as a <RETRIEVED> block so the student sees the same in-context shape the teacher did. Privacy: kolm.ai never logs raw retrieved text — only len(retrieved) and per-item source URLs appear in operator logs / audit trails. The text content lives only on the persisted capture row (where it belongs — it is training data).

Capture with retrieved context

The two first-party SDKs ship a captureWithContext helper. Both encode the retrieved array as base64 JSON onto the kolm-retrieved-context request header so structured chunks survive HTTP escaping.

Python

from kolm.client import Kolm

client = Kolm()  # picks up KOLM_KEY from env

retrieved = [
    {"source": "kolm.ai/docs/refunds", "text": "Refunds are issued within 30 days of purchase.", "score": 0.94},
    {"source": "kolm.ai/docs/refunds#partial", "text": "Partial refunds for unused months.", "score": 0.81},
]

client.capture_with_context(
    prompt="Can I get a refund after 45 days?",
    retrieved=retrieved,
    response="Refunds are issued within 30 days of purchase, so after 45 days a refund is not available.",
    namespace="customer-support",
)

JavaScript / Node

import { RecipeClient } from "@kolm/kolm-sdk";

const client = new RecipeClient();

const retrieved = [
  { source: "kolm.ai/docs/refunds", text: "Refunds are issued within 30 days of purchase.", score: 0.94 },
  { source: "kolm.ai/docs/refunds#partial", text: "Partial refunds for unused months.", score: 0.81 },
];

await client.captureWithContext({
  prompt: "Can I get a refund after 45 days?",
  retrieved,
  response: "Refunds are issued within 30 days of purchase, so after 45 days a refund is not available.",
  namespace: "customer-support",
});

Header format

The SDK helpers serialize retrieved as JSON and base64-encode it onto the kolm-retrieved-context request header. If you are calling the API from a custom client, the contract is:

FieldSpec
Header namekolm-retrieved-context (case-insensitive)
ValueBase64-encoded UTF-8 JSON
PayloadJSON array of {source: string, text: string, score?: number}
Max size256 KiB pre-decode, 500 items post-decode
Required fieldssource (non-empty string), text (string)
Optionalscore (any finite number, retriever-specific)

Example raw payload before base64 encoding:

[
  {"source": "kolm.ai/docs/refunds", "text": "Refunds are issued within 30 days.", "score": 0.94},
  {"source": "kolm.ai/docs/refunds#partial", "text": "Partial refunds for unused months."}
]

On a malformed header the server returns HTTP 400 with the honest envelope:

{
  "ok": false,
  "error": "invalid_retrieved_context_header",
  "hint": "must be base64 JSON array of {source, text, score?}"
}

Absence is not an error — non-RAG requests skip the header and the capture row stores retrieved_context: null. The training-data formatter falls through to the legacy USER: ... ASSISTANT: ... shape for those rows, so existing distill flows keep working untouched.

Training-data format

At distill time, formatCaptureForTraining(capture) emits one row per capture. When retrieved_context is present each chunk is rendered as a <RETRIEVED> block before the USER / ASSISTANT turn:

<RETRIEVED source=kolm.ai/docs/refunds score=0.94>
Refunds are issued within 30 days of purchase.
</RETRIEVED>
<RETRIEVED source=kolm.ai/docs/refunds#partial score=0.81>
Partial refunds for unused months.
</RETRIEVED>

USER: Can I get a refund after 45 days?
ASSISTANT: Refunds are issued within 30 days of purchase, so after 45 days a refund is not available.

The student sees the retrieved chunks in the same shape at every training step, which teaches it to attend to whatever you place between the <RETRIEVED> tags at inference time. At serve time you wrap real retriever output in the same tags — the student generalises across new corpus versions because it learned the structural pattern, not the specific text.

Bakeoff axis: context-faithfulness

The W734-5 bakeoff axis adds a context_faithfulness column to kolm bakeoff results. For every contestant call that carries a retrieved-context payload, the bakeoff computes the fraction of response tokens that appear in the retrieved corpus.

ValueMeaning
1.0Every response token appears in the retrieved chunks — maximally grounded.
0.5Half the response tokens are grounded; the rest are model priors or hallucinations.
0.0No grounding — the response ignores the retrieved chunks entirely.
nullHonest absence. The captures had no retrieved_context, so the axis does not apply. Distinct from 0.

Honest limits

  • This is a token-presence heuristic. It rewards "the response says words that appeared in the chunks" — not semantic agreement. A model that paraphrases honestly will score lower than one that quotes verbatim, even if the paraphrase is more faithful.
  • Token comparison is case-folded, punctuation-stripped, and drops tokens shorter than 3 characters so stop-words like "a / an / of / to / the" don't dominate the score.
  • Semantic faithfulness (sentence-transformers cosine similarity) is on the roadmap and will be a sibling axis — not a replacement — so operators can read both signals.
  • Honest null when no call carried retrieved_context. The bakeoff display and recommendation logic distinguish null from 0 — null means "axis not applicable", not "model failed".

Honesty contract

  • Header absence is a no-op, never an error — non-RAG requests still capture cleanly.
  • Malformed header returns HTTP 400 with error:'invalid_retrieved_context_header' so misconfigured callers fail loud.
  • Raw retrieved text is NEVER logged. Operator logs and the audit trail see only len(retrieved) and per-item source URLs.
  • The capture row stores retrieved_context: null when absent (not []) so downstream filters can distinguish "no RAG" from "RAG with empty result-set".
  • The version stamp is pinned by RAG_CAPTURE_VERSION (currently w734-v1) so distill consumers can detect a schema migration.
