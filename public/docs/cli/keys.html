kolm keys

Signing-key rotation lifecycle. Generates Ed25519 keypairs, records rotation receipts, and emits KMS wrap intents. The receipt-chain verifier (kolm verify) re-checks the active fingerprint against the manifest's signing block.

Usage

kolm keys list list keys + status + KMS target
kolm keys rotate [--kms <target>] [--overlap-days <N>]
kolm keys fingerprint print active key short fingerprint
kolm keys export emit the rotation manifest as JSON

Flags

FlagEffect
--kms <target>One of local, aws-kms, gcp-kms, azure-kv, hashicorp-vault. Default local. Hosted targets emit a wrap intent; the customer KMS hook applies the wrapping.
--overlap-days <N>How long the old key stays valid during rotation. Default 30 days; spec lock at DEFAULT_OVERLAP_DAYS.
--jsonMachine-readable output. Default is human-readable.

Examples

kolm keys list
kolm keys rotate
kolm keys rotate --kms aws-kms --overlap-days 60
kolm keys fingerprint

Honest scope

kolm emits the rotation receipt and writes the new key into KOLM_HOME for the local target. For hosted KMS targets the customer's KMS hook applies the wrapping; kolm does not call AWS, GCP, or Azure APIs from the CLI. Verify the wrap landed in your KMS before rotating production traffic.

The overlap window matters. During those days both keys verify, so a manifest signed by the old key still passes kolm verify while you re-sign in-flight artifacts. Set --overlap-days shorter than your shortest receipt-validity window and you can lock yourself out of past artifacts. Set it longer than your rotation cadence and you defeat the point.

What lands in the receipt

Every rotation appends a row to the export manifest with the active fingerprint, the previous fingerprint, the KMS target, and the timestamp. Re-run kolm keys export after each rotation to capture the new state. The receipt-chain verifier reads the active fingerprint from the signing block and checks it against the live registry, so a rolled key with no overlap window will fail verification on every prior artifact.
