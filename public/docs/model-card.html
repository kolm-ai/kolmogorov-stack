kolm.ai / docs / model-card

Open-source AI workbench

Auto-generated model cards for every .kolm artifact.

Every kolm compile emits a fresh Hugging Face Model Card (v0.3) alongside the artifact. Ten sections covering intended use, limitations, training data summary, eval results, ethical considerations, and environmental impact — built from the same manifest that signs the receipt, so the card and the bytes can never drift. Fields that the manifest does not yet carry come through as the plain sentinel not_yet_disclosed: an auditor can grep for the exact gap. Stamped w768-v1. Built by src/model-card-emit.js, src/model-card-schema.js, and apps/export/model_card.py.

Why a model card for every artifact

A signed receipt proves a .kolm artifact's bytes match what was minted. A model card is what a procurement, legal, or compliance team needs before they will deploy those bytes inside a regulated workflow. The HF Model Card v0.3 schema is the de facto standard for that description; OneTrust, ServiceNow AI Governance, and IBM OpenPages all consume some variant of it. generates the card on demand from the manifest, so it stays in lockstep with the artifact you just built — no out-of-band Word document to keep up to date.

Ten sections (HF Model Card v0.3)

The card renders these ten sections in canonical order. Byte-stability matters: downstream tools diff sequential card emissions, and we never reorder without bumping the w768-v1 stamp.

1. Model details

Name, version, developer, model type, license, base model, framework, languages.

2. Intended use

Primary uses, primary users, out-of-scope uses. Plain about what the model is NOT for.

3. Factors

Relevant factors (demographic, domain, environmental) the model's performance varies across.

4. Metrics

Performance measures, decision thresholds, variation approaches, raw measured values.

5. Evaluation data

Datasets, motivation for selection, preprocessing applied before scoring.

6. Training data

Datasets, size, preprocessing, capture count (from kolm event-store when distilled from captures).

7. Quantitative analyses

Unitary results (per-factor) and intersectional results (factor combinations).

8. Ethical considerations

Sensitive data, human-life implications, mitigations applied, risks & harms surfaced, intended use cases.

9. Caveats & recommendations

Caveats the user should know, recommendations for safe deployment.

10. Environmental impact

Compute hours, GPU class, estimated CO2. Always stamped static_grid_average_w768_v1 + estimate_not_measured — see the next section.

Plain sentinel: any field the manifest does not yet carry comes through as the literal string not_yet_disclosed. We deliberately do NOT silently substitute an empty string or list — the auditor must be able to grep for not_yet_disclosed and see every gap in the card.

Environmental impact estimation

The environmental_impact section is an estimate, not a measurement. We compute it as:

estimated_co2_kg = compute_hours * gpu_class_kw * grid_co2_kg_per_kwh
                 = compute_hours * gpu_class_kw * 0.475

# 0.475 kg CO2 / kWh is the IEA 2024 World Energy Outlook global grid average.
# gpu_class_kw is the vendor TDP under sustained training load, single-card draw.

Every emission carries two stamps that prevent the estimate from being mistaken for a measurement:

  • methodology: "static_grid_average_w768_v1" — surfaces the methodology version so a future revision can be diffed.
  • honest_caveat: "estimate_not_measured" — the loud-fail envelope value, so downstream tooling and auditors can never confuse the static estimate with a measured datacenter bill.

Known GPU classes today: a100, h100, h200, b200, rtx-5090, rtx-4090, l40s, mi300x, tpu-v5p. When the manifest's gpu_class is unknown to us, estimated_co2_kg is set to not_yet_disclosed with a reason: "unknown_gpu_class" field and the full list of known classes in known_gpu_classes. We never silently fall back to a different class.

Plain scope: Measured datacenter billing is the gold standard for environmental impact. Until the artifact's compile loop ingests an actual energy meter reading, the value here is a static estimate. Treat it as a floor for compliance reporting and a direction for comparing across artifacts — not as an audited figure.

Governance platform integration

The card schema is published as a JSON Schema draft-07 document at /v1/model-card/schema and re-emitted statically below. For the three named GRC suites we also ship a field-name mapping table so an integrator can move a generated card into the platform's expected shape without hand-rewiring.

kolm card field OneTrust AI Governance ServiceNow AI Governance IBM OpenPages
model_details aiInventory.modelMetadata sn_aigov_inventory.model_metadata OPModel.ModelOverview
intended_use aiInventory.purposeOfProcessing sn_aigov_inventory.intended_use OPModel.IntendedUse
factors aiInventory.modelFactors sn_aigov_inventory.relevant_factors OPModel.ModelFactors
metrics aiInventory.modelPerformance sn_aigov_inventory.performance_metrics OPModel.PerformanceMetrics
evaluation_data aiInventory.evaluationDatasets sn_aigov_inventory.evaluation_data OPModel.EvaluationData
training_data aiInventory.trainingDatasets sn_aigov_inventory.training_data OPModel.TrainingData
quantitative_analyses aiInventory.quantitativeAnalyses sn_aigov_inventory.quantitative_analyses OPModel.QuantitativeAnalyses
ethical_considerations aiInventory.ethicalRiskAssessment sn_aigov_risk.ethical_considerations OPModel.EthicalConsiderations
caveats_and_recommendations aiInventory.modelLimitations sn_aigov_risk.model_limitations OPModel.CaveatsAndRecommendations
environmental_impact aiInventory.environmentalImpact sn_aigov_inventory.environmental_impact OPModel.EnvironmentalImpact

Where a platform's top-level path is documented publicly, we use it verbatim. Where the platform exposes a customer-defined custom-field surface, we use a clearly-namespaced placeholder (for instance custom.intendedUse) so an integrator knows to wire it at deploy time. The mapping is exposed as frozen reference data via /v1/model-card/governance-mappings?platform=<name>.

CLI usage

Three paths: the local CLI for offline / CI use, the HTTP API for tenant-scoped builds, and a stdlib-only Python script for runners that don't have Node.

CLI (offline, pure-compute)

# Generate a JSON model card from a manifest:
kolm mc generate ./my.kolm/manifest.json

# Markdown body (HF-standard):
kolm mc generate ./my.kolm/manifest.json --format markdown

# Hugging Face Hub-ready (YAML frontmatter + markdown body):
kolm mc generate ./my.kolm/manifest.json --format huggingface --out README.md

# Include the environmental estimate:
kolm mc generate ./my.kolm/manifest.json --include-environmental

# Fetch the JSON schema (auth-gated):
kolm mc schema

# Fetch a governance platform mapping (auth-gated):
kolm mc governance-map onetrust
kolm mc governance-map servicenow_ai_governance
kolm mc governance-map ibm_openpages

HTTP API (authenticated)

POST /v1/model-card/generate
{ "manifest": { ...kolm manifest... },
  "format":   "json",
  "include_environmental": true }
→
{ "ok": true, "version": "w768-v1", "format": "json",
  "generated_at": "2026-05-24T...Z",
  "card": { "model_details": {...}, ..., "environmental_impact": {...} } }

GET /v1/model-card/schema
GET /v1/model-card/governance-mappings?platform=onetrust

Python (stdlib-only)

python3 apps/export/model_card.py \
  --manifest ./my.kolm/manifest.json \
  --format huggingface \
  --include-environmental \
  --out README.md
