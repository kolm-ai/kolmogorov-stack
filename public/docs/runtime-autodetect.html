Run & Govern / Serve auto-detect

Pick the runtime from the artifact shape and the host hardware.

kolm serve artifact.kolm reads the artifact file extension and a fresh hardware probe, then picks one of llama.cpp, vLLM, mlx, or ollama. The decision is a pure routing table in src/serve-autodetect.js — no I/O beyond the file suffix and the optional manifest, so the same inputs always produce the same spawn invocation.

Detection priority

First match wins. The full ladder is implemented in detectRuntime():

#ConditionRuntime
1--runtime overridecaller-pinned, no auto-pick
2.gguf + --runtime ollamaollama (Modelfile + ollama serve)
3.gguf + CUDA GPUllama.cpp with -ngl <layers>
4.gguf + Apple Siliconllama.cpp with --metal
5.gguf + CPU onlyllama.cpp CPU mode
6.safetensors + CUDA GPUvLLM OpenAI-compat server
7.safetensors + Apple Siliconmlx-lm.server
8.mlx + Apple Siliconmlx-lm.server
9anything elseunsupported with a reason field

GPU class

The probe is reduced to three classes:

  • cuda — NVIDIA or AMD with at least 4 GB VRAM. AMD groups with cuda for llama.cpp paths (HIP); vLLM coverage on ROCm is shakier.
  • metal — Apple Silicon (any unified memory).
  • cpu — no GPU or unknown vendor.

GPU layer offload

For llama.cpp on a CUDA host, the -ngl flag tells the runtime how many transformer layers to push onto the GPU. pickGpuLayers() picks a conservative default per VRAM band:

  • VRAM ≥ 16 GB — -ngl -1 (offload everything).
  • VRAM ≥ 8 GB — -ngl 32 (partial offload).
  • VRAM ≥ 4 GB — -ngl 20 (minimal offload).
  • VRAM < 4 GB — -ngl 0 (CPU only).

Override with --gpu-layers 60 at the CLI or pin gpu_layers in the artifact manifest. auto follows the table above.

The spawn invocation

The decision returns a command: { bin, args } tuple so callers can introspect (--dry-run prints it) without re-parsing. Examples:

// gguf + cuda + 24 GB VRAM
{
  bin: 'llama-server',
  args: [
    '--model', './artifact.gguf',
    '--ctx-size', '4096',
    '--host', '0.0.0.0',
    '--port', '8788',
    '-ngl', '-1'
  ]
}

// safetensors + cuda
{
  bin: 'python',
  args: [
    '-m', 'vllm.entrypoints.openai.api_server',
    '--model', './artifact-dir',
    '--host', '0.0.0.0',
    '--port', '8788',
    '--max-model-len', '4096'
  ]
}

Dry run

kolm serve ./artifact.kolm --dry-run
# detected runtime: llama.cpp (gguf + cuda GPU (RTX 5090) -> llama.cpp with -ngl -1)
# spawn: llama-server --model ./artifact.gguf --ctx-size 4096 --host 0.0.0.0 --port 8788 -ngl -1
# health: http://0.0.0.0:8788/health

Overrides

FlagEffect
--runtime <name>Force one of llama.cpp / vllm / mlx / ollama. Bypasses auto-pick.
--port <n>HTTP port. Default 8788.
--host <ip>Bind address. Default 0.0.0.0.
--context-length <n>Token window. Default 4096.
--gpu-layers <n|auto>llama.cpp -ngl. Default auto.
--dry-runPrint the runtime + command and exit without spawning.
--dockerEmit a docker-compose.yml instead of spawning. See Docker Compose.
--k8sEmit Kubernetes manifests instead of spawning. See Kubernetes.

/metrics + /health

The runtime exposes an OpenAI-compatible /v1/chat/completions natively. The metrics sidecar (src/serve-metrics-sidecar.js) runs on port+1 and surfaces request count, p50 latency, and tok/s in Prometheus text exposition format, parsed from the runtime's stdout/stderr log lines.

curl http://localhost:8789/metrics
# kolm_serve_request_count{runtime="llama.cpp"} 142
# kolm_serve_latency_p50_ms{runtime="llama.cpp"} 28.4
# kolm_serve_tok_s_p50{runtime="llama.cpp"} 41.2
# kolm_serve_memory_mb{runtime="llama.cpp"} 5232
Constraints. The router never invents a runtime. .safetensors on a CPU-only host returns unsupported with the hint to install CUDA or quantize to GGUF first. .mlx on Linux returns unsupported because mlx-lm requires Apple Silicon.
