Run & Govern / Deploy

Air-gap deployment bundle.

generateAirgapBundle from src/deploy-generators.js writes a real tar.gz to disk. The bundle contains the artifact byte-for-byte, a runtime installer that reads from a local wheelhouse, an offline kolm-audit-1 receipt verifier, and a sha256 manifest covering every entry. Tar headers are USTAR-formatted and deterministic so the same inputs always produce a byte-identical tarball — useful for repro builds and supply-chain attestation.

Invocation

import { generateAirgapBundle } from './src/deploy-generators.js';

const result = generateAirgapBundle({
  artifact_path: '/tmp/qwen3-8b-support-q4km.kolm',
  runtime: 'vllm',                // 'vllm' or 'llama.cpp'
  output_dir: '/tmp/dist',
});

// { ok: true,
//   bundle_path: '/tmp/dist/kolm-airgap-qwen3-8b-support-q4km-2026-05-26T11-25-30-123Z.tar.gz',
//   manifest_sha256: 'a3b1...',
//   sha256: 'd91f...',
//   size_bytes: 3812944822,
//   file_count: 9,
//   artifact_id: 'qwen3-8b-support-q4km',
//   runtime: 'vllm',
//   created_at: '2026-05-26T11:25:30.123Z',
//   version: 'r4-v1' }

Bundle layout

/artifact.kolm           - the artifact (copy of input)
/runtime/install.cjs      - installs the chosen runtime from local wheelhouse
/runtime/wheels/.keep    - drop offline pip wheels here BEFORE bundling (vllm only)
/runtime/bin/.keep       - drop llama-server binary here BEFORE bundling (llama.cpp only)
/config/vllm.json        - vLLM config pointing at /opt/kolm/artifact.kolm
/config/llama-cpp.cmd    - llama-server startup command
/verify.cjs              - offline kolm-audit-1 receipt verifier
/MANIFEST.sha256         - sha256 of every file in this bundle
/README.md               - offline-install instructions

Install on the air-gapped host

tar -xzf kolm-airgap-qwen3-8b-support-q4km-*.tar.gz -C /opt/kolm
cd /opt/kolm
bash runtime/install.cjs

# vLLM is now installed from the local wheelhouse. Start the server.
vllm serve --config-file config/vllm.json

Verify the bundle is intact

Two complementary checks. The first is a transport-integrity check (did the bytes survive the USB / mirror / WORM media?). The second is a receipt-chain check (does this receipt match the bundled artifact under the local secret?).

# Per-file sha256 check
sha256sum -c MANIFEST.sha256

# Replayable kolm-audit-1 receipt verification
node verify.cjs path/to/receipt.json --manifest ./MANIFEST.sha256 --secret $KOLM_ARTIFACT_SECRET
# Exit 0 on {ok: true}, exit 1 otherwise

The bundled verifier

verify.cjs is a minimal CommonJS file that depends only on Node's built-ins (crypto, fs, path). It checks a kolm-audit-1 envelope's HMAC signature against the local secret and, when invoked with --manifest, additionally walks the manifest to verify every payload file's sha256 still matches.

node verify.cjs receipt.json --secret $KOLM_ARTIFACT_SECRET
{
  "ok": true,
  "cid": "rcpt_01KYC1ZVTGDCW3FX06JQSC"
}

# A tampered file makes the manifest check fail loudly:
node verify.cjs receipt.json --secret $SECRET --manifest ./MANIFEST.sha256
{
  "ok": false,
  "cid": "rcpt_01KYC1ZVTGDCW3FX06JQSC",
  "manifest": {
    "ok": false, "files_ok": 7, "files_bad": 1,
    "bad_files": [{"path": "artifact.kolm", "expected": "a3b1...", "got": "9d2e..."}]
  }
}

Return value

FieldTypeNotes
okbooleanTrue on a successful bundle write. False with error + hint on recoverable input problems.
bundle_pathstringAbsolute path to the written .tar.gz file.
manifest_sha256stringsha256 of the MANIFEST.sha256 body (the manifest covers payload files; this hash covers the manifest itself).
sha256stringsha256 of the gzipped tarball itself. Also written to a sibling .sha256 file next to the bundle.
size_bytesintegerCompressed bundle size on disk.
file_countintegerNumber of entries inside the tarball (including the README and the manifest itself).

Preparing the wheelhouse / binary

Two of the bundle's directories are empty placeholders on first emit. You fill them on an internet-connected workstation before re-bundling for transport.

vLLM (wheelhouse):

# On a connected workstation, build the offline wheel cache.
pip wheel vllm -w /path/to/bundle/runtime/wheels/

# Then re-run generateAirgapBundle so the wheels land inside the tarball.

llama.cpp (binary):

# On a connected workstation, build llama.cpp.
git clone https://github.com/ggml-org/llama.cpp.git && cd llama.cpp
make -j llama-server
cp llama-server /path/to/bundle/runtime/bin/llama-server

# Re-run generateAirgapBundle.
Caveats. The bundle does NOT carry TLS certificates (supply via your reverse proxy), Secrets (inject via your secrets manager), or the base model weights (point your runtime config at your local model cache). The README at /opt/kolm/README.md lists what is not in the bundle so an operator does not discover it on the air-gapped host.
Determinism. Tar headers fix uid / gid / checksum to deterministic values. The same inputs produce a byte-identical tarball across runs, so the sibling .sha256 file is stable across builds when the inputs are stable. The bundle's mtime field uses the current wall clock; pass an explicit clock via the test seam if you need bit-for-bit reproducibility across hours.
