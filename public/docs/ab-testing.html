Open-source AI workbench

A/B testing for artifacts.

Configure variant A and variant B on a single namespace; the kolm gateway splits traffic, records outcomes, runs bootstrap significance, and either auto-promotes B when it wins decisively or auto-rolls back when fallback rate or p95 latency regresses. Six routes, one CLI verb, one dashboard. Stamped w822-v1.

Mental model

An A/B test is a configuration row plus a feedback log plus a decision gate. The configuration row pins which two artifacts split a namespace. The feedback log is the canonical event-store. The decision gate is the bootstrap + threshold function called by POST /v1/ab/promote.

Variant A is the incumbent; variant B is the challenger. The split field is the fraction of traffic sent to A (so split=0.5 is 50/50, split=0.2 sends 80% to B, split=1.0 sends 100% to A which is the rollback shape). Every request that lands on a configured namespace is hashed deterministically and assigned to a variant; the assignment is logged with the K-Score, latency, fallback flag, and optional thumbs-up/down feedback.

The decision gate runs over the rolling window (default seven days). It needs at least min_samples per variant (default 30) before it returns anything other than hold. Below the floor, the answer is always "wait for more data" - kolm.ai will not promote on thin samples.

Chi-square and bootstrap

The decision engine combines two tests: a chi-square for categorical outcomes (fallback rate, error rate) and a bootstrap percentile test for continuous outcomes (K-Score, latency).

The bootstrap is the load-bearing test for K-Score. We resample each variant's K-Score samples 1000 times (configurable via thresholds.bootstrap_iters), compute the difference of means at each resample, and report the two-tailed p-value as the fraction of bootstrap differences whose sign disagrees with the observed difference. The bootstrap is preferred over Welch's t because K-Score distributions are typically non-normal and bounded in [0, 1].

Welch's t-test is still available via kolm stat-sig test for the case where you have raw sample arrays in hand and want a Welch p-value plus confidence interval without going through the event-store. The stat-sig module sits at src/stat-sig.js; it is intentionally pure-math and unit-testable in isolation.

Holdout split and assignment

Traffic is split deterministically. Two requests with the same request_hash always land on the same variant - that is the contract that lets you A/B at the user or session granularity without a sticky cookie store.

The split function is a uniform hash on the request_hash field; the variant boundary is split * 2^32. A request hash of user_42 with split=0.5 lands deterministically on A or B and stays there across reconfigurations as long as the split fraction does not move. When you re-issue POST /v1/ab/configure with a new split, the assignment boundary moves and a fraction of previously-A users move to B (and vice versa); that is the cost of changing the split mid-flight.

The rollback path uses split=1.0 (all traffic on A). The promote path uses split=0.0 (all traffic on B). Both are honest config moves, not separate "rollback config" rows - the same /v1/ab/configure surface drives all three decisions.

Auto-promote and auto-rollback

The decision engine has three outcomes: promote (flip to B), rollback (force back to A), and hold (keep current split).

Promote rule

Triggered when the bootstrap p-value is below promote_p (default 0.05) AND the K-Score delta is above promote_k_delta (default +0.02). Both conditions must hold; a low p-value on a hairline win does not promote, and a large delta with a noisy sample does not promote.

Rollback rule

Triggered when fallback rate delta exceeds rollback_fallback_delta (default +0.05) OR p95 latency delta exceeds rollback_latency_p95_pct (default +0.25, i.e. 25% slower). The rollback path is OR-gated (any regression triggers) and runs BEFORE the promote check, so a fallback regression always wins over a marginal K-Score uplift.

Hold rule

Returned when neither the rollback nor the promote conditions are met, or when sample counts are below min_samples. The hold reason is one of insufficient_samples, no_signal, or metrics_missing; the dashboard renders the reason verbatim so an operator can tell why nothing happened.

Every promote and rollback writes a row to the audit chain via tryAppendAudit with op ab.promoted or ab.rolled_back. The audit row carries the previous split, the new split, the deltas, and the bootstrap envelope - sufficient to reconstruct the decision later.

HTTP API

Five routes under /v1/ab/*, all auth-required and tenant-fenced via req.tenant_record.id. The kolm ab CLI verb wraps the same surface.

method + pathpurpose
POST /v1/ab/configureSet up an A/B test: {namespace, version_a, version_b, split?, idempotency_key?}. Returns the config; idempotent on key.
GET /v1/ab/statusLatest config + sample counts + metrics for a namespace; without ?namespace=X returns the rollup across the tenant.
POST /v1/ab/feedbackRecord one outcome: {namespace, variant, request_id?, k_score?, latency_ms?, thumb?, comment?}. Also fans the row to the W720 self-improvement queue.
POST /v1/ab/promoteRun the decision gate. {namespace, force?, thresholds?}. With force:true skips the gate and flips to B; otherwise returns one of promote/rollback/hold.
GET /v1/ab/metricsAggregated per-variant rollup: {a: {count, k_score_mean, latency_p95, fallback_rate, ...}, b: {...}}.

One curl call

curl -X POST https://kolm.ai/v1/ab/configure \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "namespace": "support-reply-v4",
    "version_a": "support-reply-v3.kolm",
    "version_b": "support-reply-v4-candidate.kolm",
    "split": 0.5
  }'

The response carries the persisted config plus the version stamp. Subsequent feedback events for namespace support-reply-v4 are tagged with their variant and counted toward the rolling window.

CLI verbs

The kolm ab verb wraps the configure/list/status/assign/promote/rollback/stop subcommands; kolm stat-sig covers the standalone significance tests.

One CLI call

# Start a 50/50 test
kolm ab start --namespace support-reply-v4 \
  --arm-a support-reply-v3 \
  --arm-b support-reply-v4-candidate \
  --split 0.5 \
  --sample-target 1000

# After ~30+ samples per arm, evaluate
kolm stat-sig gate --ab-test-id abt_a1b2c3 --alpha 0.05 --min-effect 0.02
# {
#   "ok": true,
#   "decision": "pass",
#   "p": 0.018,
#   "effect_size": 0.034,
#   "version": "w778-v1"
# }

# Promote the winner
kolm ab promote abt_a1b2c3 --arm b --reason "p=0.018, K-Score lift +3.4%"

The kolm stat-sig test sub-verb takes raw comma-separated samples and runs Welch's t-test without touching the event-store - useful in CI on a fixture file. The kolm stat-sig gate sub-verb pulls samples from a live A/B test ID and runs the decision against the configured alpha / min-effect.

Dashboard

The /account/ab-tests page renders the live test rollup: per-namespace configurations, sample counts, deltas, current decision, and the promote/rollback buttons that drive POST /v1/ab/promote with force:true.

Each row carries the same shape the metrics route returns: counts per variant, K-Score mean, latency p50/p95, fallback rate, thumb-up rate. The "Decision" column renders the current gate output (promote / rollback / hold) with the reason underneath - so an operator can tell whether the test is waiting for samples, holding on insufficient signal, or actively recommending a flip.
