Open standard

.kolm format v1.0.

The .kolm format is a signed, verifiable bundle for model artifacts. It packages weights, recipe, eval set, and provenance behind a single Ed25519 signature so anyone can stand up, inspect, and verify the artifact without trusting the publisher. This page is the canonical specification.

Why a separate format

A .kolm is not a model. It is the bundle that lets a third party stand up, inspect, and verify a model someone else compiled, without trusting the publisher. Six things have to ride together for that promise to hold:

  1. The model weights (or a pointer to them).
  2. The recipes that produced the model.
  3. The evaluation set the model was graded against.
  4. The signed receipt that anchors the whole bundle to a key.
  5. The provenance / evidence DAG that names every upstream source.
  6. The runtime metadata (passports) declaring which runtimes have loaded and benchmarked the bundle.

A .kolm v1.0 is a single ZIP container holding exactly these things, with deterministic byte layout, sorted entries, and a sha256-anchored manifest.

File container

A v1.0 .kolm artifact is a ZIP archive (PKZIP, RFC 1951 deflate) with the file extension .kolm. Implementations MAY also accept the bundle as a flat directory tree containing the same entries; the validator script scripts/dotkolm-validate.cjs accepts either layout.

Compression: deflate. No password, no encryption layer, no nested archives. ZIP entries SHOULD be sorted alphabetically by filename so two byte-identical builds produce byte-identical archives. Readers MUST NOT depend on entry order — they MUST locate entries by name.

Required entries

Every v1.0 .kolm MUST contain these top-level entries:

EntryPurpose
passport.jsonMachine-readable provenance + manifest. Schema in section 4.
README.mdHuman-readable summary. Free-form text.

The remaining entries are conditional. A reader MUST be able to validate the artifact identity (passport.json) from these two alone.

passport.json schema

passport.json is a single UTF-8 JSON object. The authoritative machine-readable manifest. Top-level identity fields (all required):

FieldTypeConstraint
specstringMUST be "kolm-format-1.0".
format_versionstringMUST be "1.0".
artifact_idstringURL-safe slug, 1-128 chars matching [A-Za-z0-9._-]+.
artifact_hashstringHex sha256 (64 chars).
cidstringDeterministic content id over hashes.
created_atstringISO-8601 UTC timestamp.
taskstringFree-form task slug, 1-200 chars.
artifact_classstringOne of rule, synthesized_rule, compiled_rule, distilled_model.
runtime_targetstringOne of js, gguf, onnx, wasm, native.
base_modelstringFree-form base-model identifier.
licensestringSPDX-style identifier (e.g. Apache-2.0).

Provenance (required): seed_provenance ({eval_source, comparator, production_ready, holdout_ratio}) and hashes (per-file sha256 map).

The hashes object always carries passport_json and weights. When the corresponding payload is bundled, tokenizer, eval_set, receipts, evidence_dag, compile_args, runtime_passport may also be present. The constant EMPTY_SHA is the sha256 of zero bytes (e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855).

Signature (required): a signature object of shape

{
  "algorithm": "ed25519",
  "public_key": "<base64-pad-stripped Ed25519 public key>",
  "signature":  "<base64-pad-stripped Ed25519 signature>",
  "key_fingerprint": "<hex sha256 of public_key>",
  "signed_at": "<ISO-8601 UTC>",
  "payload_canonical_sha256": "<hex sha256 over canonical body>"
}

The signature is computed over the canonical-JSON encoding of passport.json with the signature key removed. Canonical JSON means UTF-8 throughout, object keys sorted lexicographically at every level, no insignificant whitespace, no NaN / Infinity / -Infinity.

Standard bundle layout

PathContains
passport.jsonThis manifest. REQUIRED.
README.mdHuman-readable summary. REQUIRED.
weights/<file>Model weights (e.g. weights/model.gguf).
tokenizer/<file>Tokenizer files.
eval/eval_set.jsonlEval cases the build was graded against.
receipts/receipt.jsonBuild receipt (Ed25519 + HMAC chain).
evidence_dag.jsonProvenance DAG (capture lineage, teacher rollouts, gates).
compile_args.jsonReproducibility arguments.

Verification chain

A conforming reader MUST execute these steps in order. A failure at any step is a hard verification failure — the reader MUST refuse to load the artifact further.

  1. Container parse. Open the ZIP, enumerate entries. Reject if passport.json is missing.
  2. Schema validate. Parse passport.json as JSON. Validate against the companion JSON Schema.
  3. Version gate. Confirm format_version is accepted by the reader.
  4. Hash recompute. For each entry declared in hashes, sha256 the bytes inside the container and compare. Mismatch fails.
  5. Signature verify. Re-canonicalize passport.json with the signature key removed, sha256 it, and verify the Ed25519 signature against the declared public_key. Mismatch fails.
  6. Optional registry cross-check. If network reach is available and the operator has not disabled it, fetch https://kolm.ai/v1/verify/{cid} and compare the returned manifest hash against the local recompute.

Compression and signing rationale

ZIP + deflate was chosen over tar+gz, tar+zstd, and OCI layouts. ZIP supports random access without streaming the whole archive (a reader that only needs passport.json reads ~1 KB regardless of bundle size), off-the-shelf ZIP libraries exist in every language (Node, Python, Go, Rust standard libraries parse ZIP without external deps), and sorted-entry determinism gives byte-identical archives for byte-identical builds.

Ed25519 was chosen over RSA + ECDSA. Smaller signatures (64 bytes vs 256+), fast verify on hardware without dedicated crypto units (edge devices), and no nonce — re-signing the same payload produces a stable signature byte-for-byte (this matters for reproducibility).

Versioning rules

  • Major bump (1.0 → 2.0): breaking change. A v1.0 reader MAY refuse to load 2.x artifacts.
  • Minor bump (1.0 → 1.1): backwards-compatible addition. A v1.0 reader MUST load 1.x artifacts, ignoring unknown optional fields, and MUST round-trip unknown fields without modification.
  • Patch bump (1.0.0 → 1.0.1): editorial corrections. No on-the-wire change.

Reference implementation

The reference encoder lives at src/forge-export.js (Node) and apps/export/run.py (Python). The reference validator lives at scripts/dotkolm-validate.cjs and accepts either a .kolm file or a flat directory. Exit code 0 on success, non-zero on any failure.

node scripts/dotkolm-validate.cjs tests/fixtures/dotkolm/valid-minimal.kolm
# pass: kolm-format-1.0 valid-minimal-001 cid=...

node scripts/dotkolm-validate.cjs tests/fixtures/dotkolm/invalid-missing-passport.kolm
# fail: passport.json missing from bundle

Test vectors

FixturePurpose
valid-minimal.kolmSmallest valid .kolm: dummy weights + minimal passport.
valid-full.kolmFull-featured: passport + weights + tokenizer + eval + receipts + evidence DAG.
invalid-missing-passport.kolmFails validation. The validator names the missing entry.

These are synthetic test vectors, not real models. File sizes are intentionally tiny so the test suite runs offline.

Ecosystem readers

Reader-side adoption is tracked at docs/spec/ecosystem-prs.md. A reader that intends to claim v1.0 compliance MUST pass the test-vector suite: accept valid-minimal.kolm and valid-full.kolm, reject invalid-missing-passport.kolm with a clear error message naming the missing required entry.
