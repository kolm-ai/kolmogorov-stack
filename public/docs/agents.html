Agent / tool-use distillation

Agent workloads — calls to get_weather, search_docs, create_ticket, query_database — are mostly repetitive. kolm.ai captures the tool-use trace alongside the prompt and response, then teaches a smaller student model to make the same tool calls locally. The runtime adapter executes the calls against your real APIs with your auth; the distilled agent handles the routine patterns without pinging a frontier model on every turn.

Tool-use distillation

A distill pipeline that sees only {prompt, response} learns a closed-book version of an agent. The student memorises which words go in — it never learns to plan, to fetch, or to react to a tool result that surprised it. The tool-call trace is the missing signal: with {prompt, tool_calls, tool_results, response} the student learns the shape of "given this question, call get_weather(city=...); given the result, format the final answer". It generalises across new cities, new questions, and new tool versions because it learned the structural pattern.

The W735 capture path parses tool calls out of upstream LLM response bodies in three shapes: Anthropic's content[].type === 'tool_use', OpenAI's message.tool_calls[], and a generic fallback (function_call at top level). The parsed calls are folded onto the capture row alongside the existing prompt + response + retrieved_context fields, so a single capture can be RAG-aware, tool-aware, or both — the formatter handles all combinations.

Capture format

At distill time, formatToolUseCapture(capture) emits one row per capture. When tool_calls is present, each call is rendered as a paired ASSISTANT_TOOL_CALL + TOOL_RESULT block before the final ASSISTANT line:

USER: What's the weather in San Francisco?
ASSISTANT_TOOL_CALL: {"name":"get_weather","arguments":{"city":"SF"}}
TOOL_RESULT: {"temp_f":62,"conditions":"foggy"}
ASSISTANT: The weather in San Francisco is 62 degrees Fahrenheit and foggy.

When the capture has no tool_calls, the formatter falls through to the legacy USER: / ASSISTANT: format. This is the additive, non-breaking contract — non-agent captures keep looking exactly like W734 / pre-W735 rows. The student model can be fine-tuned on a mixed corpus (plain Q&A + RAG + tool-use) without the formatter switching contracts mid-stream.

Multi-call traces are emitted in order. A capture with three sequential tool calls produces three ASSISTANT_TOOL_CALL + TOOL_RESULT pairs before the final synthesis line. The student sees a deterministic structural shape regardless of how many steps the agent took.

Runtime adapter

The W735-3 scaffold ships src/tool-runtime.js. kolm.ai does not ship a built-in tool registry — the tenant registers their own tools with their own auth. This is a security and scope decision (we don't want to be the one who breaks your Stripe key by shipping a buggy adapter; you don't want to wait for us to add the SaaS API you actually need).

import { registerTool, executeToolCall } from 'kolm/tool-runtime';

const tools = new Map();

registerTool(tools, {
  name: 'get_weather',
  handler: async ({ arguments: args, auth_context }) => {
    const r = await fetch(`https://api.weather.example/v1/current?city=${encodeURIComponent(args.city)}`, {
      headers: { authorization: `Bearer ${auth_context.weather_api_key}` },
    });
    return await r.json();
  },
  auth_schema: { required: ['weather_api_key'] },
});

// At inference time, parse the student's response, then execute each call:
const out = await executeToolCall({
  tool_call: { name: 'get_weather', arguments: { city: 'SF' } },
  tool_registry: tools,
  auth_context: { weather_api_key: process.env.WEATHER_KEY },
});
// out => {ok: true, result: {temp_f: 62, conditions: 'foggy'}}

Every executeToolCall returns a discriminated-union envelope and never throws. Handler exceptions are caught and surfaced as tool_threw:

EnvelopeWhen
{ok:true, result:...}Handler returned a value.
{ok:false, error:'tool_not_found'}The tool name is not registered in this tenant's registry.
{ok:false, error:'invalid_tool_call'}The tool_call argument was missing name or malformed.
{ok:false, error:'auth_failed'}The auth_schema.required fields were missing from auth_context.
{ok:false, error:'tool_threw', detail:...}The handler raised an exception; detail carries the message.

Honest acceptance target

The W735-4 spec target is "distilled agent handles 90% of tool-calling patterns locally". We measure this honestly. When the observed sample size is <100 captures, local_handling_rate is reported as null with reason:'insufficient_signal_n<100' — we do not extrapolate from a tiny sample. Below 100 calls, any rate you compute is noise.

The measurement function accumulateAcceptanceMetrics returns the local-handling rate plus a Wilson 95% confidence band when the sample size is large enough:

{
  sample_size: 142,
  n_handled_locally: 128,
  n_escalated_to_teacher: 14,
  target: 0.90,
  local_handling_rate: 0.901,
  honest_acceptance: 0.901,
  confidence_band_95: { lo: 0.842, hi: 0.940 }
}
ValueMeaning
local_handling_rate ≥ 0.90The distilled agent met the W735-4 spec target on this sample.
local_handling_rate < 0.90Honest miss — do not ship as "agent-grade" yet. Capture more before re-measuring.
local_handling_rate: nullSample size <100. Insufficient signal. Keep capturing.
confidence_band_95: nullSample size <30. The Wilson normal approximation isn't reliable yet.

You can run the local report at any time:

kolm tool acceptance --namespace customer-support
kolm tool patterns   --namespace customer-support

Limitations

  • No built-in tool registry. kolm.ai does not ship default get_weather / search / send_slack adapters — tenants register their own via registerTool. Shipping defaults would be a security hazard (your auth context is yours) and a scope hazard (every SaaS would want a built-in adapter).
  • The runtime does not sandbox handler code. Handlers run with whatever permissions the host process has. Treat each registered tool as part of your trust boundary; review the same way you review server-side route handlers.
  • Argument validation is shape-only at registration time. The handler is expected to revalidate — defense in depth. validateToolSchema checks the load-bearing JSON Schema fields (type:"object", properties, required[]) but does not run full JSON Schema validation.
  • The 90% local-handling rate is a goal, not a claim. The reported rate is null until you have ≥100 captures. We will not pretend a 4/4 (100%) early sample is meaningful.
  • Tool calls and responses are persisted on the capture row. Treat the persisted store as you would a server-side log of agent traces — redact PII at capture time if your tools handle user data. The privacy contract for raw tool arguments mirrors the W734 contract for retrieved text: the data lives on the row (where it belongs — it IS training data) and never appears in console / audit-log output.
