Open-source AI workbench

Cost optimization.

Record teacher API spend during a baseline window. Deploy the kolm gateway. Record post-deployment spend over the same window. The savings tracker computes the dollar delta; kolm.ai charges 12.5% on the savings, zero on regression. Default 30-day window, 7-day minimum, frozen quarterly rate-card.

The bargain

Savings-based pricing makes the vendor share the outcome. If kolm.ai does not reduce your teacher spend, kolm.ai does not bill on the savings dimension. The fee is a flat 12.5% within the 10-15% band published in SAVINGS_FEE_RATE_DEFAULT; the per-tenant rate can be negotiated for very large or very small deployments.

Two windows: the baseline window is your pre-kolm spend (typically 30 days of normal usage), and the post-deployment window is the same length, starting after the gateway is in path. The dollar saving is baseline_total_usd - post_total_usd, the fee is max(0, saving) * fee_rate, the net to you is saving - fee.

Regression handling is explicit: if post spend exceeds baseline the saved_usd is negative and the fee_usd is exactly zero. We do not bill against negative savings. The envelope still includes the negative figure plus regression:true so the operator sees the regression rather than a silent zero.

Frozen rate-card

The rate-card in src/savings-tracker.js is frozen at module load and refreshed quarterly. Volume discounts, batch-API discounts, and cached-input discounts are explicitly NOT modelled. Those are negotiated; baking them in would let the savings tracker overstate per-call savings against a list-price baseline.

Baseline window

Start the baseline before you deploy kolm. The marker row is a zero-cost event in the event-store, so the same /v1/lake/export endpoint will surface it for audit.

POST /v1/savings/baseline

curl -X POST https://kolm.ai/v1/savings/baseline \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "namespace": "support-bot" }'

# -> { "ok":true, "namespace":"support-bot", "started_at":"2026-04-24T...",
#      "min_baseline_days": 7 }

GET /v1/savings/baseline

curl -H "Authorization: Bearer $KOLM_API_KEY" \
  "https://kolm.ai/v1/savings/baseline?namespace=support-bot&period_days=30"
# -> { "ok":true, "namespace":"support-bot", "started_at":"...",
#      "days_elapsed": 18, "accumulated_usd": 142.37,
#      "status": "in_progress" }

Below the 7-day minimum the status is insufficient_baseline and downstream summary calls refuse to report a saving. Above 7 days the status flips to ready and computeSavings has enough signal.

Record teacher spend

Every teacher API call lands in the event-store via the standard capture layer. For tenants who run the teacher outside the kolm gateway (a legacy direct call), the savings/record endpoint accepts an explicit cost report so the savings calculator can see the row.

POST /v1/savings/record

curl -X POST https://kolm.ai/v1/savings/record \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "namespace": "support-bot",
    "provider": "anthropic",
    "model": "claude-sonnet-4-7",
    "input_tokens": 2300,
    "output_tokens": 540
  }'

# -> { "ok":true, "recorded": { "event_id":"...", "cost_micro_usd":15000,
#      "cost_usd":0.015, "input_tokens":2300, "output_tokens":540 } }

The cost is computed against the frozen rate-card. Unknown provider returns a 400 with error:'unknown_provider'; unknown model returns error:'unknown_model'. The 400 is honest, the operator can either add the provider/model to the rate-card OR record the cost manually upstream.

Savings summary

The summary endpoint computes the dollar savings, the fee, and the net for the requested period. The default fee_rate is 12.5%; override per-call via ?fee_rate=0.10 if your tenant has a negotiated rate.

GET /v1/savings/summary

curl -H "Authorization: Bearer $KOLM_API_KEY" \
  "https://kolm.ai/v1/savings/summary?namespace=support-bot&period_days=30"

# -> {
#   "ok": true,
#   "namespace": "support-bot",
#   "period_days": 30,
#   "baseline_usd": 1284.50,
#   "post_usd": 218.10,
#   "saved_usd": 1066.40,
#   "fee_rate": 0.125,
#   "fee_usd": 133.30,
#   "net_savings_usd": 933.10,
#   "regression": false,
#   "status": "ready",
#   "defaults": {
#     "min_baseline_days": 7,
#     "baseline_period_days_default": 30,
#     "savings_fee_rate_default": 0.125
#   }
# }

Honest sub-states

  • status:"no_baseline_started" - call POST /v1/savings/baseline first.
  • status:"insufficient_baseline" - fewer than 7 days of baseline observed; the saved_usd is null, never fabricated from thin data.
  • status:"ready" - the full envelope with saved_usd, fee_usd, net_savings_usd populated.
  • regression:true - post spend exceeded baseline; saved_usd is negative, fee_usd is zero.

CLI

The CLI exposes kolm savings baseline and kolm savings summary as the operator-facing surface. Useful for one-off "how am I doing this month" reads without scripting against the HTTP route.

kolm savings baseline --namespace support-bot
# -> baseline started; days_elapsed:0, status:in_progress

kolm savings summary --namespace support-bot --period-days 30
# -> pretty-prints baseline_usd, post_usd, saved_usd, fee_usd, net_savings_usd

kolm savings summary --namespace support-bot --period-days 30 --json
# -> raw JSON envelope for piping into jq / your billing dashboard
