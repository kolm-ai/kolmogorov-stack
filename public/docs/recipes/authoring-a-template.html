Open-source AI compiler

Author your own recipe.

You shipped something that works. Publish it as a kolm template and the next team starts a step ahead instead of re-deriving the same hyperparameters from scratch. We host the CI, you get attribution in the registry, and your recipe earns a free run on every model upgrade so it doesn't bit-rot.

No bureaucracy. The bar is: your recipe runs end-to-end on the hardware floor you claim, the front-matter is filled in, and the artifact verifies. If that holds, we merge.

If you are not sure your recipe is ready, open a draft PR anyway. We will review and suggest the smallest set of changes to get it across the line.

Step-by-step

Step a

Fork the templates repo

Templates live in their own repo so the registry can pin every recipe to a commit. Fork from GitHub.

git clone https://github.com/kolm-ai/kolm-templates.git
cd kolm-templates
git checkout -b add-your-recipe
Step b

Copy the scaffold

Every recipe starts from templates/_template.md. Copy it under your slug.

cp templates/_template.md templates/your-recipe.md

The scaffold contains the five recipe sections (inputs, steps, artifact, caveats, how to verify) with placeholder content. Replace the placeholders with your real run.

Step c

Fill in the front-matter

The block at the top of the file is what the registry reads to index your recipe. Every field is required.

---
slug: your-recipe
title: Distill a thing that does the job
base_model: Qwen/Qwen2.5-3B-Instruct
hardware_floor: RTX 4090 / 24GB
teachers:
  - anthropic:claude-sonnet-4-6
  - openai:gpt-4o-mini
wall_clock_estimate: ~90 min
license: Apache-2.0
---

Tips on the fields:

  • slug — kebab-case, matches the filename. Becomes /docs/recipes/<slug>.
  • title — imperative, short. “Distill X”, “Migrate Y”, “Deploy Z”.
  • base_model — an exact HuggingFace id. The registry will not accept a vague family name.
  • hardware_floor — the smallest box your recipe actually ran on. Be conservative; we will test it.
  • teachers — list of vendor:model ids. Match the strings kolm distill --list-catalog prints.
  • wall_clock_estimate — rough total, including any data prep and the final kolm verify.
  • license — SPDX id. Apache-2.0 is the recommended default; we accept any OSI-approved license.
Step d

Validate locally

The same check the registry runs in CI. Catches missing front-matter fields, unresolved teacher ids, broken cross-links, and missing recipe sections.

kolm template validate ./templates/your-recipe.md

If the validator complains, it prints the exact field or section that is missing and where to put it. Re-run until the output is ok: true.

Step e

Open a pull request

Push the branch and open a PR against main. The registry CI runs your recipe end-to-end on a fresh runner and posts the K-Score, wall-clock, and artifact size as a check.

git add templates/your-recipe.md
git commit -m "templates: add your-recipe"
git push origin add-your-recipe
gh pr create --title "templates: add your-recipe" \
             --body  "Adds a recipe for <short description>."

A maintainer reviews within a couple of business days. Merges land in the registry on the next deploy and appear at /docs/recipes.

Front-matter schema

The full schema the validator enforces. Required fields are listed; optional fields are listed in the scaffold.

---
# required
slug:                 <kebab-case-id>          # matches filename; URL slug
title:                <imperative short title>
base_model:           <hf-org/hf-name>         # exact HuggingFace id
hardware_floor:       <label>                  # e.g. "RTX 4090 / 24GB"
teachers:                                       # 1+ entries
  - <vendor:model>                              # e.g. anthropic:claude-sonnet-4-6
wall_clock_estimate:  <rough total>             # e.g. "~90 min"
license:              <SPDX id>                 # e.g. Apache-2.0

# optional
author:               <name or handle>          # appears in registry attribution
contact:              <email or url>
tags:                 [distill, support, ...]   # discoverability
min_corpus_size:      <int>                     # capture-row floor
gpu_vram_gb:          <int>                     # numeric form of hardware_floor
last_verified:        <YYYY-MM-DD>              # CI updates on each run
---

Why publish a template?

Shared distill seeds

Your recipe joins the public seed pool. Other authors can fork your seed schema instead of writing one from scratch, and your seeds get tested by every distill run that picks them up.

Free CI runs

The registry runs your recipe end-to-end on every model and toolchain upgrade. If a base model or teacher version drifts and breaks your numbers, we open the issue against the template — not against you.

Attribution in the registry

The recipe page on kolm.ai credits you by name (or handle), links your contact, and surfaces your name in the registry API. Teams that pick up your recipe see who wrote it.

How to verify

Before opening the PR, run the same gates the registry CI will run.

  • Front-matter passes. kolm template validate ./templates/your-recipe.md prints ok: true.
  • The recipe actually runs. Execute every step in a clean checkout, end-to-end, on the hardware floor you claimed. Time it and confirm your wall_clock_estimate is within 25%.
  • The artifact verifies. The final step of your recipe produces a .kolm artifact. kolm verify <artifact> must pass with a K-Score above the gate you specify in the recipe (default 0.85).
  • Cross-links resolve. Any link to /docs/..., /sdks/..., or /recipes/... in your recipe body must point at a page that already exists or that ships in the same PR.
