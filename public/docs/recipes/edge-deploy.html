Docs / Recipes / Edge deploy

Open-source AI compiler

Deploy a distilled model to the edge.

Take a .kolm artifact you already trust, compile it to a Q4_K_M GGUF, and ship it to the device class you actually run. The same artifact lands on a Raspberry Pi 5, a Jetson Orin, a Mac mini, an x86 NUC, or a browser via WebGPU/WASM. The runtime path never round-trips to a cloud, and the signed manifest travels with the model so the device can prove what is running.

What you need

  • A .kolm artifact you trust. Distill one using a recipe from the recipe library (customer support, doc Q&A, classify, or agent). The export step below assumes the artifact path is ./artifacts/your-model.kolm.
  • kolm CLI on your build box. npm i -g @kolmogorov/cli or the Python SDK path from the Python quickstart.
  • llama.cpp on every target device. Per-device install commands are inline below.

Step-by-step

Step a

Export the artifact to GGUF

The compile-to-GGUF path is the same one covered in the distill recipes — see customer-support or doc-qa for the end-to-end version. Once you have a .kolm artifact, the export is one command:

kolm export ./artifacts/your-model.kolm \
  --backend gguf \
  --quant q4_k_m \
  --out ./out

This drops a model.gguf file (typically under 1 GB for a 3B-class student at Q4_K_M) under ./out/, alongside the manifest the device runtime checks at load.

Step b

Raspberry Pi 5 (CPU, NEON)

The Pi 5 runs llama.cpp directly on its Cortex-A76 cores. No accelerator needed. SCP the GGUF and launch a single command.

# on the build box
scp ./out/model.gguf pi@pi5.local:~

# on the Pi (one-time install)
ssh pi@pi5.local
sudo apt-get update && sudo apt-get install -y build-essential cmake
git clone https://github.com/ggerganov/llama.cpp && cd llama.cpp
cmake -B build && cmake --build build --config Release -j

# every-run
./build/bin/llama-cli -m ~/model.gguf -p 'hello'

The 8GB Pi 5 SKU has enough headroom for a 3B-class model at Q4_K_M with a small context window. Drop to IQ4_XS or a smaller base model if you need longer contexts on the 4GB SKU.

Step c

Jetson Orin (CUDA)

Build llama.cpp with the JetPack-bundled CUDA toolkit. Use the nvidia-l4t-jetson-multimedia-api stack for camera / IO pipelines if you need them; the LLM runtime itself only depends on CUDA.

# on the Jetson (JetPack 6+)
sudo apt-get update
sudo apt-get install -y build-essential cmake nvidia-l4t-jetson-multimedia-api
git clone https://github.com/ggerganov/llama.cpp && cd llama.cpp
cmake -B build -DGGML_CUDA=ON
cmake --build build --config Release -j

# copy the artifact
scp user@build-box:/path/to/out/model.gguf ~/

# run
./build/bin/llama-cli -m ~/model.gguf -p 'hello' -ngl 99

-ngl 99 offloads every layer to the Orin's iGPU. Orin Nano (8GB) is enough for a 3B-class model at Q4_K_M; Orin NX / AGX comfortably handle 7B and above.

Step d

Mac mini M2 (Metal)

Apple Silicon runs llama.cpp through Metal with no extra flags. Homebrew ships a current build.

brew install llama.cpp

# copy the artifact
scp user@build-box:/path/to/out/model.gguf ~/

# run
llama-cli -m ~/model.gguf -p 'hello'

If you want the MLX-native path instead of GGUF on Apple Silicon, re-export with kolm export ./artifacts/your-model.kolm --backend mlx --quantize and run via mlx_lm. MLX wins for tok/s on M-series chips; GGUF wins for cross-device parity.

Step e

x86 NUC (Vulkan or CPU)

If the NUC has an Intel Arc / iGPU with Vulkan support, build llama.cpp with the Vulkan backend. Otherwise fall back to the CPU build.

# with Vulkan (iGPU / Arc)
sudo apt-get install -y build-essential cmake libvulkan-dev vulkan-tools
git clone https://github.com/ggerganov/llama.cpp && cd llama.cpp
cmake -B build -DGGML_VULKAN=ON
cmake --build build --config Release -j
./build/bin/llama-cli -m ~/model.gguf -p 'hello' -ngl 99

# CPU fallback (no GPU)
cmake -B build && cmake --build build --config Release -j
./build/bin/llama-cli -m ~/model.gguf -p 'hello'

The CPU path lands roughly where the Pi 5 does for tok/s on a modern x86 chip, but a Vulkan-backed iGPU usually gets you a 3-5x speedup over CPU on the same box.

Step f

Browser via WebGPU / WASM

The browser path uses the WASM target. The same artifact compiles to a WASM bundle that runs in-page via WebGPU on browsers that support it, with a WASM-CPU fallback elsewhere.

kolm compile ./artifacts/your-model.kolm \
  --target wasm \
  --out ./public

# serve and open
python3 -m http.server -d ./public 8080
# browse to http://localhost:8080/

The output drops a model bundle plus a tiny JS shim into ./public/. Open the page in a recent Chrome/Edge/Safari Tech Preview to get the WebGPU path; older browsers fall back to WASM-CPU at lower tok/s.

Other limitations to know before you ship

  • Context window vs RAM. Long contexts blow up KV-cache size linearly. On the 4GB Pi SKU, keep the context under 2K tokens. On Orin Nano, 4K is comfortable. On Mac mini and above, 8K+ is fine.
  • WebGPU availability. WebGPU shipped to Chrome stable in 2023 and Safari in 2024, but coverage on older devices is uneven. Always test the WASM-CPU fallback path before relying on WebGPU.
  • Quant level vs quality. Q4_K_M is the sweet spot for most distill outputs. If your K-Score drops more than 3 points after quantization, try Q5_K_M (larger, slower) or re-distill with more training data.
  • Manifest portability. The signed manifest travels with the artifact, but llama.cpp itself only reads the GGUF header. Keep the original .kolm file alongside the GGUF if you want the device to be able to re-verify the chain via kolm verify.

How to verify

  • On the build box. kolm verify ./artifacts/your-model.kolm — checks the manifest signature, the K-Score gate, and the receipt chain. Run this once before you start copying the GGUF anywhere.
  • On the device. Confirm the GGUF loads and produces a non-empty completion with llama-cli -m model.gguf -p 'hello'. If the load fails on a low-RAM device, drop to a smaller quant (IQ4_XS) or a smaller base model.
  • Round-trip the manifest. If you copied the .kolm file to the device, re-run kolm verify model.kolm there. The CID and the receipt chain should match the build-box run byte-for-byte.
