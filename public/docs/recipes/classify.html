Open-source AI compiler

Distill a multi-class classifier.

Compile a fast, small classifier from a labeled JSONL or an OpenAI fine-tuning dataset. The recipe targets Qwen2.5-3B-Instruct (3B is plenty for narrow label-set work) and ships a Q4_K_M GGUF plus a holdout macro-F1 number signed into the artifact receipt — so the next reviewer does not have to re-prove the model's accuracy on your fixture.

What you need

  • kolm CLI — npm i -g github:kolm-ai/kolm or build from source. Confirm with kolm version.
  • API key — kolm login or sign up at /signup.
  • Teacher provider keys — ANTHROPIC_API_KEY, OPENAI_API_KEY, and CEREBRAS_API_KEY exported in the shell. A missing key drops that teacher from the council; the run continues.
  • GPU for the training step — one card with 8 GB+ VRAM is enough for a 3B QLoRA run (RTX 3090 / 4090 / 5090 all comfortable). The rest is CPU-bound.
  • Dataset shape — either an OpenAI fine-tuning JSONL or a JSONL of {"text":..., "label":...} rows. Aim for at least 50 examples per class. Pair with a labels file: ./labels.json = {"classes":["billing","tech","sales","other"]}.
  • Holdout — a separate JSONL of {"text":..., "label":...} rows that did not appear in training, for the eval step.
Why 3B is the right size. Classification is a narrow output (one of N labels) on a narrow input (one short text). The base model only needs enough capacity to encode the label boundaries; it does not need open-ended generation muscle. Qwen2.5-3B-Instruct in Q4_K_M is small, fast (sub-50 ms first token on a 4090), and routinely matches a 7B teacher on macro-F1 once distilled on a clean label set.

Step-by-step

Step 1

Import your labeled data

Pick the path that matches what you already have. The OpenAI fine-tuning JSONL importer reads both chat ({"messages":[...]}) and legacy completions ({"prompt":..., "completion":...}) shapes; the label is read from the assistant turn or the completion text. For raw labeled JSONL, use kolm capture log --label-field to tell the importer which JSON key holds the class.

# Path A — existing OpenAI fine-tuning JSONL
kolm import --from openai-finetune ./classifier-finetune.jsonl \
  --out ./captures.jsonl \
  --namespace intent

or, if you have a labeled JSONL already:

# Path B — labeled JSONL with an explicit label field
kolm capture log \
  --namespace intent \
  --import ./labeled.jsonl \
  --label-field label

Inspect counts and label distribution with kolm capture status --namespace intent. The distill spec in step 2 will warn if any class is below the recommended 50-example floor.

Step 2

Generate a classify-task distill spec

The --task classify hint switches the spec into classification mode: the loss is cross-entropy on the label head plus SeqKD on the teacher's brief rationale, the prompt template enforces "respond with the class label only," and the eval block is wired for macro-F1. The --label-set file pins the class vocabulary into the artifact's manifest so the runtime can reject out-of-set labels.

kolm distill spec \
  --base qwen2.5-3b-instruct \
  --task classify \
  --label-set ./labels.json \
  --out spec.json

The default council for classification is openai:gpt-4o,cerebras:llama-3.3-70b with a Claude judge — two cheap, fast labelers with a third lineage as the judge keeps cost low. Override with --teachers if you have a reason.

Step 3

Collect council labels

The collector samples your imported captures, asks each teacher to label every row, and records every teacher's label plus a one-sentence rationale. Conflicts (teachers disagree) are routed to the judge for resolution and tagged for review.

kolm distill collect \
  --spec spec.json \
  --out captures.jsonl

A 2000-row corpus across two labelers + one judge typically completes in 5–8 minutes and costs $1–$3.

Step 4

Train the student with QLoRA, 3 epochs

Classification benefits from a few extra epochs — the label head needs to converge tightly. QLoRA on Qwen2.5-3B-Instruct fits in roughly 8 GB of VRAM; even a laptop 4070 handles it. The trainer signs the resulting adapter with your tenant's Ed25519 key and pins the spec hash + label-set hash + base-model commit into the receipt chain.

kolm distill train \
  --spec spec.json \
  --captures captures.jsonl \
  --qlora \
  --epochs 3

Three epochs on 2000 council pairs takes about 80 seconds on an RTX 5090 and around 4–5 minutes on an RTX 4090. The signed .kolm artifact is written to ~/.kolm/artifacts/; the path is printed on success.

Step 5

Evaluate on the holdout for macro-F1

Run the artifact against the holdout JSONL. The eval harness computes macro-F1 (per-class precision/recall averaged across classes, so an imbalanced class set is not hidden by a frequent-class win), prints a confusion matrix, and writes a signed eval report next to the artifact.

kolm eval ~/.kolm/artifacts/<artifact>.kolm \
  --examples ./holdout.jsonl \
  --json > eval-report.json

to compare against the GGUF before you ship:

kolm eval ~/.kolm/artifacts/<artifact>.kolm \
  --examples ./holdout.jsonl \
  --trace

The default ship gate is macro-F1 ≥ 0.85 on a balanced holdout. Inspect failing rows with --trace; the report names the predicted label, the expected label, and the model's rationale for each miss.

Step 6

Export to GGUF for local serving

Convert the signed adapter to a Q4_K_M GGUF for Ollama, LM Studio, llama.cpp, or any other GGUF-aware runtime. At 3B parameters in Q4_K_M, the file is around 2 GB — small enough to ship in a container image.

kolm export ~/.kolm/artifacts/<artifact>.kolm \
  --backend gguf \
  --quant q4_k_m \
  --out intent-classifier.gguf

Pair with --backend mlx for Apple Silicon, --backend onnx for browser WebGPU or ONNX Runtime, --backend tensorrt for NVIDIA serving, or --backend coreml for iPhone/iPad. Q5_K_M is a safer choice if your holdout macro-F1 is right at the gate.

What you ship

Artifactintent-classifier.gguf (~2 GB Q4_K_M on Qwen2.5-3B) + signed .kolm manifest with spec, label-set hash, teacher provenance, and Ed25519 signature.
Eval reporteval-report.json with macro-F1, per-class precision/recall, confusion matrix, and the holdout SHA pinned into the receipt.
ModelfileOne Ollama Modelfile that pins the chat template and stop tokens; drops straight into any Ollama-aware tool.
ReceiptHMAC chain (task → seeds → recipes → evals → package) verifiable offline with kolm verify intent-classifier.kolm.
Deploy targetsOllama (local + macOS), vLLM (production GPU serving), llama.cpp (edge + CPU), LM Studio (desktop UI), WebGPU via kolm export --backend onnx, CoreML for iOS.

Caveats

The runtime rejects out-of-set predictions by default. If the model produces a label string that is not in ./labels.json, the call returns a structured "unknown" with the raw output attached — treat that as a routing signal, not a silent miss.
Macro-F1 averages across classes evenly, so a tiny class with poor recall is not hidden behind a frequent-class win. If your label distribution is heavily skewed, weight your holdout to reflect production traffic and read both macro-F1 and weighted-F1.
Label drift is the usual failure mode. If a class definition changes over time, re-import the updated labels and re-train; the prior receipt becomes a historical record, not a guarantee.
A 3B classifier costs roughly 50x less per call than a hosted frontier model on the same workload. The trade is a narrower model: it predicts labels well; it cannot answer arbitrary questions.
