Docs / Recipes / Customer support replier

Open-source AI compiler

Distill a customer-support replier.

Take a directory of closed tickets (or your existing OpenAI fine-tuning JSONL) and compile a small student model that drafts replies in your team's voice. The output is a signed GGUF you own, runnable in Ollama, vLLM, llama.cpp, or any GGUF-aware runtime — with a verifiable receipt that ties the artifact back to its training spec.

What you need

  • kolm CLI — npm i -g github:kolm-ai/kolm or build from source. Confirm with kolm version.
  • API key — kolm login or sign up at /signup. The key authorizes capture, distill, and signing.
  • Teacher provider keys — ANTHROPIC_API_KEY, OPENAI_API_KEY, and CEREBRAS_API_KEY exported in the shell. Missing keys cause the matching teacher to be skipped (with a warning), not fail the run.
  • GPU for the training step — one card with 16 GB+ VRAM (RTX 4090 / 5090) or a $0.39/hr RunPod community GPU. Import, spec, collect, and export are CPU-bound.
  • Dataset shape — either an OpenAI fine-tuning JSONL ({"messages":[...]} chat or {"prompt":..., "completion":...} legacy) or a JSONL of closed-ticket pairs with at least 800–2000 rows. A thumbs-up / CSAT field is recommended but not required.

Step-by-step

Step 1

Import your historical conversations

Pick the path that matches what you already have. If you ran an OpenAI fine-tune (now sunsetting January 2027), use the importer to land its training set into a kolm capture lake. If your tickets live in a CSV/JSONL export from Zendesk, Front, or Help Scout, use kolm capture log --import directly.

# Path A — existing OpenAI fine-tuning JSONL
kolm import --from openai-finetune ./historical-finetune.jsonl \
  --out ./captures.jsonl \
  --namespace support

or, if you have raw closed tickets:

# Path B — closed-ticket JSONL export
kolm capture log \
  --namespace support \
  --import ./closed-tickets.jsonl

Both paths produce capture rows in the support namespace that the distill pipeline can read. Inspect counts with kolm capture status --namespace support.

Step 2

Generate a distill spec

The spec pins the base model, the teacher council, the per-teacher row budget, and the hyperparameters. The defaults below come from the Trinity-500 council recipe and have shipped real artifacts on the same vocabulary.

kolm distill spec \
  --base qwen2.5-7b-instruct \
  --teachers anthropic:claude-sonnet-4-6,openai:gpt-4o,cerebras:llama-3.3-70b \
  --rows 800,600,600 \
  --namespace support \
  --out spec.json

The --rows 800,600,600 flag caps how many rows each teacher will be asked to produce, in declared teacher order. Cerebras is the fastest of the three (sub-second per row), so it can carry weight without slowing the run.

Step 3

Collect the council's responses

Run every imported prompt through every configured teacher. The output is a single JSONL with prompt, three teacher responses, the council-merged response, and per-teacher provenance (request id, model id, latency).

kolm distill collect \
  --spec spec.json \
  --out captures.jsonl

Expect a few minutes for a 2000-row corpus across three teachers. The collector backs off automatically on rate-limit and resumes with --resume. Cost on a 2000-row council typically lands at $4–$8.

Step 4

Train the student with QLoRA

QLoRA on Qwen2.5-7B-Instruct fits in roughly 16 GB of VRAM with grad-checkpointing and microbatch 1. The trainer signs the resulting adapter with your tenant's Ed25519 key and pins the spec hash + capture hash + base-model commit into the receipt chain.

kolm distill train \
  --spec spec.json \
  --captures captures.jsonl \
  --qlora

One epoch on 2000 council pairs takes about 90 seconds on an RTX 5090 and around 4–5 minutes on an RTX 4090. The signed .kolm artifact is written to ~/.kolm/artifacts/; the path is printed on success.

Step 5

Export to GGUF for local serving

Convert the signed adapter to a Q4_K_M GGUF that runs in Ollama, LM Studio, llama.cpp, and any other GGUF-aware runtime. Q4_K_M is the right starting point for replier workloads — small enough to fit a 7B on an 8 GB card, large enough to keep tone fidelity.

kolm export ~/.kolm/artifacts/<artifact>.kolm \
  --backend gguf \
  --quant q4_k_m \
  --out support-replier.gguf

Pair with --backend mlx for Apple Silicon, --backend onnx for ONNX Runtime, --backend tensorrt for NVIDIA serving, or --backend coreml for iPhone/iPad.

Step 6

Serve with Ollama and smoke-test

Wrap the GGUF in an Ollama Modelfile, register it, and run a one-shot prompt to confirm the artifact is responsive.

cat > Modelfile <<'EOF'
FROM ./support-replier.gguf
TEMPLATE """{{ if .System }}<|im_start|>system
{{ .System }}<|im_end|>
{{ end }}{{ if .Prompt }}<|im_start|>user
{{ .Prompt }}<|im_end|>
{{ end }}<|im_start|>assistant
"""
PARAMETER stop "<|im_end|>"
EOF

ollama create support-replier -f Modelfile
ollama run support-replier "test"

From here, point your support tool's webhook at http://localhost:11434/api/generate (or wrap it in a thin proxy) and ship in shadow mode first — let the model draft, the human review, and log the diff back as new captures with kolm capture log --namespace support.

What you ship

Artifactsupport-replier.gguf (~4 GB Q4_K_M on Qwen2.5-7B) + signed .kolm manifest with spec, captures hash, teacher provenance, and Ed25519 signature.
ModelfileOne Ollama Modelfile that pins the chat template and stop tokens. Drops straight into any Ollama-aware tool.
ReceiptHMAC chain (task → seeds → recipes → evals → package) verifiable offline with kolm verify support-replier.kolm.
Deploy targetsOllama (local + macOS), vLLM (production GPU serving), llama.cpp (edge + CPU), LM Studio (desktop UI), WebGPU via kolm export --backend onnx.

Caveats

The distilled student inherits its teachers' blind spots. If only one provider's key is configured, the council collapses to one lineage and the student's tone narrows. Re-run with at least two teachers from different lineages for production-grade output.
Closed-ticket corpora skew toward fast resolutions. To keep the model useful on hard cases, oversample escalated tickets (CSAT ≤ 3, or tier-2+) when you build the import JSONL.
The receipt verifies the artifact's signature and chain, not the underlying customer data. Run kolm import --redact phi,pii in step 1 if your tickets contain PHI/PII you do not want crossing teacher boundaries.
Re-distill on a cadence. Frontier teachers update every few weeks. A weekly cron of kolm distill collect --since 7d followed by kolm distill train keeps the student in step with drifting production.
