Docs / Recipes / Tool-use agent

Open-source AI compiler

Distill a tool-use agent.

Compile a small agent that calls a fixed tool registry — the same set of book-meeting, search-orders, file-ticket functions your team already exposes. The distilled student learns the registry's call schema, the planning style of your teachers, and the error-handling conventions that keep an agent from spinning forever. The tools stay the source of truth; the model just calls them.

What you need

  • kolm CLI — npm i -g github:kolm-ai/kolm or build from source. Confirm with kolm version.
  • API key — kolm login or sign up at /signup.
  • Teacher provider keys — ANTHROPIC_API_KEY, OPENAI_API_KEY, and CEREBRAS_API_KEY exported in the shell. A missing key drops that teacher from the council; the run continues.
  • GPU for the training step — one card with 16 GB+ VRAM (RTX 4090 / 5090) or a $0.39/hr RunPod community GPU. The rest is CPU-bound.
  • Tool registry — a directory of JSON files, one per tool, each containing a name, a description, a JSON-schema parameters block, and (optional) a worked usage example. Compatible with the OpenAI / Anthropic tool-call JSON schema.
  • Trajectory data — either an OpenAI fine-tuning JSONL of tool-call traces (preserves the assistant-turn tool_calls shape) or a JSONL of {"user":..., "plan":..., "tool":..., "args":..., "observation":..., "final":...} rows. 200–1000 traces usually converges a 7B.
What the model learns vs. what it does. The distilled agent learns when to call a tool, which tool to pick, how to shape the arguments, and how to weave the observation back into the user-visible answer. It does not learn the tools themselves — those execute in your runtime. Swap a tool implementation later and the agent is unaffected as long as the JSON schema is stable.

Step-by-step

Step 1

Register the tool registry

The --tool-spec flag tells the corpus ingester to parse each JSON file as a tool definition (name, description, parameters schema) rather than as a doc chunk. The registered tools become available to kolm distill spec --tools in step 3 and to kolm run --tools at inference.

kolm corpus add ./tools/*.json --tool-spec

Confirm with kolm corpus stats --tool-spec: it lists every registered tool by name, dumps the parsed parameter schema, and warns on any duplicate name or invalid schema.

Step 2

Import trajectory data

If you have OpenAI fine-tuning JSONL with tool-call traces, the importer preserves the tool_calls schema from the assistant turn so the student trains on the same shape the runtime will emit. Mixed plain-chat + tool-call rows are handled correctly — the schema is only invoked when a row contains a tool call.

kolm import --from openai-finetune ./agent-finetune.jsonl \
  --out ./captures.jsonl \
  --namespace agent

Inspect counts and trace-step distribution with kolm capture status --namespace agent. The spec generator in step 3 warns if any registered tool has zero example calls in the corpus.

Step 3

Generate a tool-use distill spec

The --task tool-use hint wires a tool-aware prompt template, a multi-step trajectory loss (per-call exact-match on the tool name and arguments, plus SeqKD on the surrounding plan), and an eval block that scores tool-arg exact-match and end-to-end task completion. The --tools ./tools/ path is required so the spec embeds the schema set the student is allowed to call.

kolm distill spec \
  --base qwen2.5-7b-instruct \
  --task tool-use \
  --tools ./tools/ \
  --teachers anthropic:claude-sonnet-4-6,openai:gpt-4o,cerebras:llama-3.3-70b \
  --out spec.json

Qwen2.5-7B-Instruct is the right starting point for tool-use because it ships strong native tool-call support. For agents with a very narrow tool set (3–5 tools) you can drop to qwen2.5-3b-instruct and save the VRAM.

Step 4

Collect council trajectories

The collector samples your imported captures, replays the user turn through each teacher with the registered tools in scope, and records every teacher's full trajectory (plan, tool calls, observations, final response). Multi-turn rows are collected end-to-end, not just per turn.

kolm distill collect \
  --spec spec.json \
  --out captures.jsonl

A 500-trace corpus across three teachers (some multi-call) typically completes in 8–15 minutes and costs $5–$15. Tool calls in the trace are simulated against the registered schemas; no live tool execution at collect time.

Step 5

Train the student with QLoRA

QLoRA on Qwen2.5-7B-Instruct fits in roughly 16 GB of VRAM with grad-checkpointing and microbatch 1. The trainer signs the resulting adapter with your tenant's Ed25519 key and pins the spec hash + tool-registry hash + base-model commit into the receipt chain.

kolm distill train \
  --spec spec.json \
  --captures captures.jsonl \
  --qlora

One epoch on 500 multi-step traces takes about 2–3 minutes on an RTX 5090 and around 8–10 minutes on an RTX 4090. The signed .kolm artifact is written to ~/.kolm/artifacts/; the path is printed on success.

Step 6

Export to GGUF for local serving

Convert the signed adapter to a Q4_K_M GGUF for Ollama, LM Studio, llama.cpp, or any other GGUF-aware runtime. The Q4_K_M default preserves tool-arg exact-match in our test runs; if your eval shows arg drift, step up to Q5_K_M or Q8_0.

kolm export ~/.kolm/artifacts/<artifact>.kolm \
  --backend gguf \
  --quant q4_k_m \
  --out tool-agent.gguf

Pair with --backend mlx for Apple Silicon, --backend onnx for browser WebGPU, --backend tensorrt for NVIDIA serving, or --backend coreml for iPhone/iPad.

Step 7

Run the agent end-to-end

kolm run wires the GGUF, attaches the tool registry, and drives the agent loop — emit a plan, call the tools as JSON, feed observations back in, return a final answer. Tool execution happens in your runtime; the agent emits the call and consumes the result.

kolm run tool-agent.gguf \
  --tools ./tools/ \
  "Book me a meeting with Jane at 3pm"

to serve as a long-running OpenAI-compatible endpoint:

ollama create tool-agent -f Modelfile
ollama run tool-agent "Search the order log for refunds last week"

The Ollama path needs your own tool-call bridge to actually invoke functions; kolm run --tools includes a built-in bridge that calls a shell hook per tool name (configurable in the registry JSON).

What you ship

Artifacttool-agent.gguf (~4 GB Q4_K_M on Qwen2.5-7B) + signed .kolm manifest with spec, tool-registry hash, teacher provenance, and Ed25519 signature.
Tool registryThe ./tools/ directory, pinned by SHA into the receipt. Re-pinned automatically on every distill run.
ModelfileOne Ollama Modelfile that pins the chat template, stop tokens, and tool-call delimiter; drops straight into any Ollama-aware tool.
ReceiptHMAC chain (tools → seeds → recipes → evals → package) verifiable offline with kolm verify tool-agent.kolm.
Deploy targetsOllama (local + macOS), vLLM (production GPU serving), llama.cpp (edge + CPU), LM Studio (desktop UI), WebGPU via kolm export --backend onnx.

Caveats

The tool-call schema must be stable. The distilled agent expects every registered tool's parameter shape to match what it saw at training time. Add a tool without re-distilling and the agent ignores it; rename a parameter and the call drifts. Re-run the recipe whenever the registry changes substantively.
The distilled agent will not invent new tools at inference time. If a user asks for something no registered tool covers, the agent says so — it does not hallucinate a function. That is the design, and it is the reason the registry is the source of truth.
Tool execution happens in your runtime, not in the model. Misconfigured tools (failing network, missing credentials, schema mismatch) surface as observations the agent sees; budget for a retry loop and a clear error-handling convention in the training trajectories.
For agents that need broader generation muscle (long-form synthesis, multi-document reasoning), pair this artifact with a larger frontier model behind a router rather than swapping the base. The 7B student is tuned for fast, faithful tool-use; the router decides when to hand off.
