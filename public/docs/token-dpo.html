Token-level DPO.

Sequence-level DPO assigns one preference reward to the whole completion. Token-level DPO assigns a reward to every token, weighted by the attention mask, so the gradient lands on the exact spans that earned the preference flip. Built on the Rafailov 2023 loss with beta=0.1, accessible behind one trainer flag.

Sequence-level vs token-level

Sequence-level DPO collapses a long answer into one scalar. If only the last sentence was the actual win, the gradient still updates every token in the response uniformly. Token-level DPO assigns the loss per token so the update concentrates where the preference signal actually lives.

The Rafailov 2023 DPO loss compares the log-probability of a preferred response against a rejected response under both the trained policy and a frozen reference policy. The classic implementation reduces both log-probs to a scalar (one number per pair) and applies F.logsigmoid(beta * (log_pi_pos - log_pi_neg)). Token-level changes the reduction: each token contributes its own log-prob delta, masked by the attention mask so padding never enters the gradient.

When to use it: the preference signal is concentrated (a single hallucinated number, a refusal that should have been a yes, the wrong tool call). When NOT to use it: the entire response style is the preference axis (long-vs-short, formal-vs-casual). Sequence-level DPO handles the latter cleanly; token-level over-weights the high-entropy tokens at the start.

The loss function

The token-level DPO loss lives at apps/trainer/contrastive_distill.py. It accepts paired logits for the preferred and rejected continuations, the target token IDs, an attention mask, and the beta temperature (default 0.1).

Reference signature

def token_level_dpo_loss(
    logits_pos,       # [batch, seq, vocab] - logits for the preferred completion
    logits_neg,       # [batch, seq, vocab] - logits for the rejected completion
    target_ids,       # [batch, seq]        - ground-truth token IDs (shared)
    attention_mask,   # [batch, seq]        - 1 for real tokens, 0 for padding
    beta=0.1,         # Rafailov 2023 default
):
    # Per-token log-prob of the target under each policy
    log_pi_pos = log_prob_of_target(logits_pos, target_ids)  # [batch, seq]
    log_pi_neg = log_prob_of_target(logits_neg, target_ids)  # [batch, seq]

    # DPO term per token, masked to real tokens
    per_token = -F.logsigmoid(beta * (log_pi_pos - log_pi_neg))
    per_token = per_token * attention_mask
    return per_token.sum() / attention_mask.sum().clamp(min=1)

Two design decisions worth flagging. First, the mask is multiplicative rather than additive (we never subtract a sentinel), so the loss is exactly zero on padded positions instead of "small and noisy." Second, the divisor is the count of real tokens, not the sequence length, so the gradient norm is independent of batch padding.

Captured input format

A token-level DPO run reads a captured.kolm artifact whose rows pair a preferred and rejected completion. The simplest source is the approvals queue: every thumbs-up is the preferred response, every thumbs-down is rejected, the input is shared.

Each row in the JSONL stream must carry an input (the shared prompt), a preferred string, and a rejected string. The trainer tokenizes both completions, pads to the longest, builds an attention mask, and packs into the DPO batch. Rows without both arms are dropped at preprocessing time.

Minimal row

{
  "namespace": "support-bot",
  "input": "What is the refund window for annual plans?",
  "preferred": "Annual plans can be refunded within 30 days, prorated for unused months.",
  "rejected": "Annual plans are non-refundable. Please contact support.",
  "feedback_source": "w458_approvals"
}

The trainer ignores all other fields. The feedback_source hint is kept in the manifest for audit (so a downstream reviewer can re-derive the preference labels from the same queue). Provenance is bound to the K-Score receipt.

Training CLI

The contrastive distill driver accepts --contrastive-token-level as the toggle. Without the flag, the loss reduces to standard sequence-level DPO. The environment variable KOLM_CONTRASTIVE_TOKEN_LEVEL=1 is equivalent.

kolm distill contrastive \
  --student qwen2.5-3b \
  --teacher anthropic/claude-sonnet-4-5 \
  --captured ~/.kolm/captures/support-bot/approvals.jsonl \
  --contrastive-token-level \
  --beta 0.1 \
  --epochs 1 \
  --learning-rate 5e-7 \
  --output ./student-tokendpo.kolm

Hyperparameter table

flagdefaultmeaning
--beta0.1DPO temperature; lower means tighter to the reference policy

Bench scaffold

The token-level vs sequence-level comparison is wired as a deterministic benchmark so the loss change can be regression-tested without spending teacher tokens.

The scaffold at apps/trainer/bench_contrastive_token.py builds a synthetic preference dataset with a known sparse signal (one specific token in the rejected completion is the wrong answer; the rest matches). It then trains for one step with each reduction and reports the L2 norm of the gradient at the marked token vs everywhere else. Token-level concentrates the gradient on the marked position; sequence-level smears it.

Run the bench locally to verify the loss is behaving as documented before you spend a real preference dataset on a 7B run.

cd apps/trainer
python bench_contrastive_token.py --batch 8 --seq 64 --marked-position 47
# expected: token-level gradient norm at position 47 is >3x norm at unmarked positions

HTTP and TUI surface

Token-DPO runs surface in two places: the bare HTTP collection at GET /v1/training/token-dpo and the TUI view bound to hotkey T (or commands :dpo / :token-dpo).

The HTTP endpoint returns an loud-fail envelope today: until the trainer queue is mounted through the daemon, the response is {ok:true, view:'token-dpo', status:'pending', items:[]} with a note pointing at the trainer module. The endpoint is still useful as a probe in dashboards (so a tenant can poll for "is the trainer wired yet?" without parsing source).

The TUI view (hotkey T) is the operator surface. It will list completed runs once the queue is wired; today it shares the same pending envelope so the view always renders.
