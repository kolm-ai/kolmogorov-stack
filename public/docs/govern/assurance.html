Run & Govern

Assurance case.

An assurance case is the human-readable view of the evidence DAG. It states what the artifact claims to do safely, breaks the claim into sub-claims, and points each sub-claim at the evidence that supports it. The format follows GSN (Goal Structuring Notation) so it slots into existing model-risk-management workflows without translation.

Anatomy

Every assurance case has one top goal — the highest-level claim the artifact is making — broken into sub-goals via explicit strategies, each backed by solutions (evidence pointers into the DAG). A context node bounds where the case applies; an assumption node makes external dependencies legible.

G1: art_a1b2c3d4 is acceptably safe for prod customer support
  S1: refute primary risks
    G1.1: does not leak PII upstream
      Sn1.1: receipts/redactor coverage report (DAG -> eval:e83f...)
    G1.2: does not regress task quality vs incumbent
      Sn1.2: shadow window k-score delta (DAG -> eval:c11a...)
    G1.3: hardware footprint fits target host class
      Sn1.3: runtime passport report (DAG -> artifact:art_a1b2c3d4.manifest)
  Context: support tier 1, English, business hours
  Assumption: upstream LLM provider remains reachable for fallback

Generation

The case is generated automatically from the artifact's spec and DAG by kolm assurance build. The author of the spec writes the goals and strategies in spec.toml; the builder fills in the solution pointers and verifies that every solution is reachable in the signed DAG.

kolm assurance build art_a1b2c3d4 --output assurance.json
kolm assurance build art_a1b2c3d4 --output assurance.pdf  --format gsn
kolm assurance verify art_a1b2c3d4
# OK: 11 goals, 11 sub-goals all backed by signed DAG nodes

Binding the case to compliance

Each sub-goal can be tagged with one or more compliance controls (NIST AI RMF, ISO/IEC 42001, EU AI Act). The compliance export rolls the tags up into a single matrix per regime — see /docs/govern/compliance.
