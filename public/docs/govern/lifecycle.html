Run & Govern

Artifact lifecycle.

A .kolm moves through six states. Every transition is a deliberate act — either a CLI command, a TUI confirmation, or an approval in the account UI — and every transition writes one row to the audit log and one row to the lifecycle history table. There are no silent state changes.

The six states

StateEnters viaWhat it means
candidatekolm distill completesWeights exist and have cleared local sanity checks. Not yet signed.
signedkolm compile clears evalsManifest, evidence DAG root, and weights are signed. Ready to publish.
stagedkolm deploy --stageLive on a non-prod namespace; receives shadow traffic; metrics accumulate.
livekolm deploy --promoteReceives production traffic per the routing table.
recalledDrift signal or manual recallPulled from production routing; pods may still serve until the swap completes.
retiredkolm retireNo longer eligible for routing; weights remain in cold storage for replay only.

Transition diagram

candidate -- compile + evals pass --> signed
signed    -- deploy --stage         --> staged
staged    -- deploy --promote       --> live
live      -- drift signal | recall  --> recalled
recalled  -- distill new            --> (new candidate)
recalled  -- retire                 --> retired
live      -- retire (no recall)     --> retired

Every arrow is gated by a check: signed requires k-score ≥ floor; staged → live requires the shadow window to clear a delta threshold against the incumbent; recalled → retired requires the recall review to be closed.

Approvals

Each transition can be gated by an approval policy in spec.toml. Default is one approver from the artifact's owners list; high-risk artifacts can require multiple approvers or a specific role:

[govern.lifecycle]
candidate_to_signed = "automatic"
signed_to_staged    = ["owner"]
staged_to_live      = ["owner", "risk_reviewer"]
live_to_recalled    = "any_owner_or_oncall"
retire              = ["owner", "owner"]   # two distinct owners
