RBAC — org roles and permission matrix

kolm organizations have four roles. Each role maps to a scoped capability set. This page is the full reference; the members page hides controls a role cannot use.

The four roles

owner

Owner

Exactly one per org. Can do everything, including transfer ownership and delete the org. The user who runs kolm org create becomes the first owner.

admin

Admin

Manages members, invites, and audit logs. Cannot transfer ownership or change billing. Multiple admins per org.

member

Member

Day-to-day user. Reads and writes tenant data (captures, distills, deploys). No member or billing controls.

billing

Billing

Reads invoices, edits payment method, changes plan. Cannot read tenant data or manage members.

Capability matrix

Y = permitted, — = forbidden. Owner always has every capability; the table shows the difference between the other three roles.

CLI reference

The same operations are available from the CLI for users who script their org operations:

kolm org list
kolm org create <name>
kolm org members <org_id>
kolm org invite <org_id> <email> --role=admin
kolm org role <org_id> <user_id_or_email> <new_role>
kolm org remove <org_id> <user_id_or_email>
kolm org transfer-owner <org_id> <new_owner_email>

Exit code 4 means the calling user lacks the capability; exit code 5 means the user is not signed in (kolm login).

Auditing

Every role change, member add or remove, invite creation, and ownership transfer emits a signed audit event. Owners and admins can read them from /account/audit-log or via kolm org audit <org_id>. Audit events sit in the same Ed25519-signed ledger as inference receipts — the receipt is the proof.

Design notes

  • Roles are stored in the data/orgs.json document and validated by src/rbac.js on every mutating call.
  • Capability checks happen at the API and CLI boundary; internal calls trust each other.
  • There is no role hierarchy at the data layer — a user's capabilities come from a single explicit role, not an inheritance chain.
  • Owners cannot be removed; the only way to release ownership is owner:transfer to another existing member.
