Docs · gateway region lock

region_lock = true. Fallback to non-pinned provider refused.

When set, the gateway will only fall back to providers whose configured region matches the namespace's pinned region. A request that would otherwise cross the boundary returns HTTP 451 region_lock_violation.

Caveats: region pinning is a routing constraint, not an encryption-in-transit guarantee. Pair with TLS pinning + your own VPC peering for end-to-end region custody.
The config

Two lines in gateway.toml.

Place at the top of the file, outside any [provider.<name>] block. Applies to every namespace served by this gateway.

gateway.toml

Region lock + EU pin

region        = "eu-west-1"
region_lock   = true

[provider.anthropic]
endpoint = "https://api.anthropic.com"
region   = "us-east-1"   # cross-region — fallback will be refused

[provider.local-vllm]
endpoint = "http://10.0.0.7:8000"
region   = "eu-west-1"   # in-region — fallback allowed
What happens

Fallback paths the gateway evaluates.

The ConfidenceRouter still runs. The only difference is that out-of-region providers are removed from the candidate list before scoring.

In-region primary OK
HTTP 200 + receipt

Primary serves the call. Same as without region_lock.

Primary down, in-region fallback OK
HTTP 200 + receipt + provider_used

Fallback to an in-region provider. provider_used in the receipt records the actual provider.

Primary down, no in-region fallback
HTTP 451 region_lock_violation

Out-of-region providers are skipped even if they are healthier. The error body lists the candidates and why each was rejected.

Audit
receipt.region_decision = "locked"

Every receipt records whether the region lock was enforced for that call. Use kolm captures stats --region-locked to count.
