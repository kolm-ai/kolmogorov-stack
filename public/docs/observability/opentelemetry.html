OpenTelemetry semantic conventions

kolm emits a stable set of kolm.* OpenTelemetry attributes and a four-phase inference span timeline (queue → load → prefill → decode) so token-confidence, routing decisions, and K-Score drift show up natively in Honeycomb, Grafana Tempo, Datadog, and Jaeger. The exporter version is pinned by OTEL_W733_VERSION (currently w733-v1). When no OTel tracer is registered, kolm is an plain no-op — it never blocks the request path on tracing.

Attributes

Every kolm inference span carries a subset of the table below. Names are namespaced under kolm.* per the OTel semantic-conventions guidance for vendor-specific attributes. None of these collide with the public OTel http.* or llm.* namespaces.

AttributeTypeDescription
kolm.token.confidence double (nats) Mean Shannon entropy across the response tokens (or per-token gauge when sub-spans are emitted). Lower means higher confidence.
kolm.routing.decision string One of student, teacher, mixed. Reflects the W709 confidence-router outcome for the span.
kolm.routing.entropy_nats double (nats) Entropy threshold that fired for the routing decision. Compare against per-token kolm.token.confidence to see which tokens triggered escalation.
kolm.kscore.value double K-Score for the artifact at the time of inference. Bounded [0, 1]; higher is better.
kolm.kscore.drift_24h double Delta from the 24h-ago K-Score baseline. Negative means quality regression — alert on this directly.
kolm.artifact.cid string Content-id of the compiled artifact (immutable). Use it to join spans to the W144 receipt store.
kolm.tenant.id_hash string (12 hex) sha256(tenant_id) first 12 hex chars. Never the raw tenant id (see Privacy note below).
kolm.namespace string Capture / artifact namespace. Already public — safe to emit raw.

Span structure

Every /v1/route/chat/completions request emits one parent kolm.inference span plus four child sub-spans laid out on a monotonic timeline. Buyers see this as a stacked horizontal bar in any OTel UI:

parent span ── kolm.inference [POST /v1/route/chat/completions] ├── kolm.inference.queue ▇ (W729 load-queue wait time) ├── kolm.inference.load ▇ (model paged into VRAM, or 0 if cached) ├── kolm.inference.prefill ▇▇▇ (prompt → KV cache compute) └── kolm.inference.decode ▇▇▇▇▇▇▇ (KV cache → output tokens)

The four sub-spans are emitted in order with zero overlap. Phase durations are also attached as kolm.inference.phase_ms on each child so a histogram query in Tempo/Honeycomb can slice latency by phase without needing the parent.

Span namePhaseNotes
kolm.inference.queueQueue waitTime spent in the W729 load queue before the request was picked up. Sustained non-zero means the runtime is GPU-bound — provision more capacity.
kolm.inference.loadModel loadTime to page the artifact into VRAM. 0 ms on cache hit. Watch this on cold starts to size your warm pool.
kolm.inference.prefillPrefillPrompt → KV cache compute. Scales with prompt length. Spike here means the caller is overpaying for token budget — consider prompt distillation.
kolm.inference.decodeDecodeKV cache → output tokens. Scales with output length. The W727 speculative-decoding kernel shrinks this when accept-rate is high.

Privacy note

tenant.id_hash vs raw tenant_id. kolm never writes a raw tenant id into an OTel attribute. The kolm.tenant.id_hash field is the first 12 hex characters of sha256(tenant_id) — enough to link spans across one tenant's traces without leaking the identifier into the buyer's tracing backend. If you need the raw tenant id for a support case, look it up in the kolm dashboard against the same hash; we never persist a reverse map server-side.

The same contract applies to capture content: kolm OTel spans carry the artifact CID and the routing decision, but never the raw prompt or response text. Capture-level inspection lives in /account/captures, which is auth-gated and tenant-scoped — OTel is for telemetry, not for log replay.

No-silent-pass contract

  • If no tracer is registered (globalThis.__OTEL_TRACER__ missing AND @opentelemetry/api not importable), createInferenceSpans and setRoutingAttributes are no-ops — they return false, never throw, and never block the request path.
  • KOLM_OTEL=0 (the default) keeps the native OTLP exporter disabled. Setting it to 1 starts a 5-second flush loop that posts spans + metrics in OTLP/HTTP JSON to OTEL_EXPORTER_OTLP_ENDPOINT.
  • Span attributes are sorted alphabetically when emitted so wire-format diffs are deterministic across runs.
  • The exporter version is pinned by OTEL_W733_VERSION (currently w733-v1) so a downstream consumer can detect a schema migration.
  • The kolm-native exporter caps the queue at 1 MB; oldest spans are dropped first to prevent unbounded memory growth on a collector outage.
