Multi-region gateway

Three-region launch plan (US, EU, APAC). Region-aware capture routing wired through W769 data residency. Cloudflare Workers + Lambda@Edge adapters ship as starter templates.

w780-v1

Launch plan

kolm.ai launches in three regions. Each region terminates TLS at its own POP, proxies to a region-local origin, and never crosses a regional boundary at request time. A capture sent to eu.kolm.ai stays in the EU.

RegionShort nameCanonicalHostname
United States (East)usus-east-1us.kolm.ai
European Union (West)eueu-west-1eu.kolm.ai
Asia Pacific (Singapore)apacap-southeast-1ap.kolm.ai

The apex record kolm.ai resolves to the closest region via GeoDNS. The W780 module honors this geo-routing decision and uses $KOLM_REGION as the runtime override for operators who need to pin a request to a specific gateway.

DNS layout

The full DNS schema lives at infra/dns-schema.json. It describes the Route53 + Cloudflare records required to launch:

  • Apex GeoDNS - kolm.ai resolves to the closest region by client IP. Weight 100 per geo bucket (NA -> us, EU/AF -> eu, APAC/OCEANIA -> apac).
  • Per-region hostnames - us.kolm.ai, eu.kolm.ai, ap.kolm.ai. Each terminates its own TLS cert (SAN-per-region by default; rotation is region-independent).
  • Health checks - every 30 seconds against /v1/health; three consecutive failures pull a region out of rotation. Same path src/multi-region.js #testFailover probes.
  • Fallback region - when no geo match (e.g., satellite uplinks reporting as anycast), the apex falls back to us.

Environment variables

VariableRequiredDescription
KOLM_REGIONnoPin the daemon to a region. Defaults to us-east-1. Accepts short names (us/eu/apac) or canonical codes.
KOLM_REGION_GATEWAY_URLSyes (for routing)JSON map of short_name -> URL. Example: {"us":"https://us.kolm.ai","eu":"https://eu.kolm.ai","apac":"https://ap.kolm.ai"}. When empty, routeRequest returns {ok:false,error:"region_not_configured"}.

Cloudflare Workers adapter

Starter template at tools/edge/cloudflare-worker.js. Drop into a Workers project, wire the gateway map via wrangler.toml vars, deploy behind the per-region hostname.

# wrangler.toml fragment
[vars]
KOLM_REGION = "eu-west-1"
KOLM_REGION_GATEWAY_URLS = '{"us":"https://us.kolm.ai","eu":"https://eu.kolm.ai","apac":"https://ap.kolm.ai"}'

Routing decision honors x-kolm-residency as a HARD constraint (returns HTTP 451 when no matching gateway) and x-kolm-prefer-region as a soft hint. Response carries x-kolm-routed-region and x-kolm-routed-reason so downstream code can audit the decision.

Lambda@Edge adapter

Starter template at tools/edge/lambda-at-edge.js. Attach as a viewer-request trigger so the routing decision lands at the edge POP closest to the user.

Lambda@Edge does NOT support env vars at execution time. Bake the gateway map into the GATEWAY_MAP constant at deploy time. The handler rewrites request.origin so CloudFront proxies to the chosen region's backend without re-fetching the body.

Region-aware capture routing

W780 joins with W769 data residency to decide which region a capture should land in. The resolution ladder:

  1. The W769 namespace default region (from configureNamespaceRegion) - if a namespace is pinned to EU_WEST, every capture in that namespace routes to eu.
  2. The W769 tenant inference from ISO country code - a tenant signed up from DE maps to eu.
  3. The runtime current region from $KOLM_REGION - the final fallback.

The join logic is honest about each rung: the returned envelope carries reason (one of w769_namespace_default, w769_tenant_inference, w769_global_uses_current_region) so callers can audit which rung made the decision.

CLI

$ kolm region status
$ kolm region gateways
$ kolm region route <request_hash> --residency eu --prefer apac
$ kolm region test-failover

The kolm region status and kolm region gateways calls hit the hosted /v1/region/* routes when $KOLM_API_KEY is set. Otherwise they fall back to the LOCAL module so an operator can sanity-check the gateway map before the daemon comes up. test-failover always runs locally because cross-region probing from a hosted daemon would amplify a flap into a thunder.

API

GET  /v1/region/status     -> { region, gateway_url, capture_residency, version }
GET  /v1/region/gateways   -> { gateways: {us, eu, apac}, region_count, version }
POST /v1/region/route      -> { region, gateway_url, reason, version }
  body: { request_hash, residency_requirement?, prefer_region? }

All auth-gated. residency_requirement is a HARD constraint - a request tagged eu against a gateway map without an eu entry returns HTTP 400 with {ok:false,error:"no_gateway_for_residency_requirement",available_regions:[...]}. We NEVER silently downgrade.
