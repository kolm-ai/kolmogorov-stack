Studio / Export / GGUF

Export to GGUF. Run on llama.cpp, Ollama, LM Studio, anything.

GGUF is the dominant on-device format. It packs weights + tokenizer + chat template + metadata into a single file the llama.cpp family can load directly. kolm's GGUF backend (in src/export-gguf.js) wraps the canonical llama.cpp convert-hf-to-gguf.py + llama-quantize pipeline, adds an imatrix step for IQ-quants, and embeds the kolm passport fields into the file's metadata block.

The export pipeline

  1. Stage merged weights — the .kolm's weights/ dir is staged to a scratch directory readable by convert-hf-to-gguf.py.
  2. Convert — convert-hf-to-gguf.py --outtype f16 --outfile staged.f16.gguf weights/. Produces the fp16 GGUF baseline.
  3. Imatrix (if target is IQ-quant) — llama-imatrix -m staged.f16.gguf -f calibration.txt -o staged.imatrix.
  4. Quantize — llama-quantize [--imatrix staged.imatrix] staged.f16.gguf out.gguf Q4_K_M.
  5. Embed metadata — llama-gguf-set writes general.name, general.quantized_by=kolm-forge, general.license, kolm.kscore, kolm.artifact_hash, llm.context_length into the output GGUF.
  6. Coherence test — load with llama-cpp-python, generate 100 tokens against a fixed prompt, check no NaN / empty / repetition-collapse. If the test fails, the exporter refuses to emit the file.
  7. Quality delta — runs the artifact's eval holdout against the quantized file; compares K-Score per axis to the fp16 baseline; writes quality_delta_vs_fp16 into the runtime passport.

CLI

# From a .kolm artifact
kolm export trinity-support.kolm --format gguf-q4km --output trinity-support-q4km.gguf

# Compile + export in one shot from a recipe
kolm compile --recipe support-trinity-500 --target gguf-q4km --output trinity-support-q4km.gguf

# Multi-target: emit several quants from the same fp16 baseline (re-uses staged.f16.gguf)
kolm export trinity-support.kolm --format gguf-q4km,gguf-q5km,gguf-q8 --output-dir gguf/

# Inspect the resulting GGUF
kolm inspect trinity-support-q4km.gguf --json

GGUF metadata fields kolm writes

KeyValue
general.nameArtifact name from spec.toml
general.authorspec.toml meta.authors
general.licensespec.toml meta.license
general.quantized_bykolm-forge v<CLI_VERSION>
general.quantization_versionllama.cpp commit hash used
general.base_model.nameBase model id (e.g. Qwen/Qwen2.5-7B-Instruct)
kolm.artifact_hashsha256 of the source .kolm
kolm.kscoreMean K-Score from artifact passport
kolm.kscore_per_axisPer-axis JSON object
kolm.recipe_hashsha256 of spec.toml
kolm.signature_pubkeyEd25519 public key (base64)
llm.context_lengthFrom spec.toml [student]
tokenizer.chat_templateJinja2 chat template

You can read these fields anywhere with llama-gguf-info file.gguf or kolm inspect file.gguf --json.

Split-file output (>50 GB models)

For models larger than 50 GB (typically 70B+ at Q8_0 or 405B at Q4_K_M), GGUF supports the split-file convention: name-00001-of-00003.gguf, name-00002-of-00003.gguf, etc. kolm enables split-file automatically when the output would exceed 48 GB, or you can force it with --split-shard-size 24G.

# Force splits at 24 GB each
kolm export big-distill.kolm --format gguf-q4km --split-shard-size 24G --output big-distill-q4km.gguf

# Load split files in llama.cpp — just point at the first shard
llama-cli -m big-distill-q4km-00001-of-00003.gguf

Ollama Modelfile bridge

Pair the GGUF output with an auto-generated Modelfile via src/export-ollama.js:

kolm export trinity-support.kolm --format ollama-modelfile --output trinity-support.Modelfile

# Then in Ollama
ollama create trinity-support -f trinity-support.Modelfile
ollama run trinity-support "hello"
Caveats. The chat template embedded in GGUF is the canonical Jinja2 from your spec.toml. Some clients (older LM Studio, some Open WebUI builds) ignore the embedded template and apply their own — double-check the prompt format if quality looks wrong in a third-party UI.
