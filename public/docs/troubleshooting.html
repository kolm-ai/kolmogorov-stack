Troubleshooting.

Eleven common failures with the kolm.ai CLI, API, and runtime. Each entry names the symptom, the root cause, and three to five concrete steps to recover. If a fix is not here, open an issue or email support@kolm.ai with the receipt id from kolm logs.

11 entries

  1. HTTP 401 – unauthorized
  2. HTTP 403 – forbidden
  3. HTTP 429 – rate limited
  4. HTTP 503 capture_store_unavailable
  5. HTTP 503 receipt_unverifiable
  6. BYOC attestation failure
  7. Trainer timeout or OOM
  8. Compile says dataset empty
  9. Compile K-score below gate
  10. Quantize says torch not installed
  11. Receipt verify returns verified false
  12. SSE capture stream disconnects
HTTP 401

Unauthorized. The API key is missing, expired, or wrong tenant.

What you see

{"error":"unauthorized","message":"missing or invalid api key"}

Root cause

The Authorization: Bearer kolm-xxx header is absent, the key was rotated, or the request hit a different tenant base. The CLI reads the key from ~/.kolm/config.json, written by kolm login.

What to do

  1. Run kolm whoami --json. If logged_in is false, run kolm login and paste a fresh key from /account/keys.
  2. Confirm the base URL matches the tenant you expect; switch with kolm profile use <name> (see kolm profile).
  3. If you are using a CI key, re-issue from /account/keys and update the secret store.
  4. If 401 persists after re-login, the user may have been removed from the workspace. Check with kolm team list.
HTTP 403

Forbidden. The key is valid but lacks the required role or scope.

What you see

{"error":"forbidden","message":"role 'viewer' cannot POST /v1/compile"}

Root cause

kolm.ai uses role-based access. viewer can read receipts and run kolm verify but cannot compile, distill, or modify captures. Service keys can be further scoped per-namespace.

What to do

  1. Ask a workspace admin to elevate your role via /account/overview (viewer to builder, or builder to admin).
  2. If you are using a scoped service key, re-issue with the namespace and verb you need.
  3. For BYOC compile, confirm the policy.yaml at the cluster allows the verb. Run kolm doctor --byoc.
  4. For air-gapped installs, re-check the seat-license file under ~/.kolm/license.
HTTP 429

Rate limited. You exceeded the per-key per-minute quota.

What you see

{"error":"rate_limited","retry_after_ms":12000}
Retry-After: 12

Root cause

Each key gets a per-minute call budget by plan tier. Bursts above the budget return 429 with a Retry-After header. Capture endpoints have a separate higher budget so log writes do not starve compute calls.

What to do

  1. Honor the Retry-After header. The CLI already backs off automatically; if you are calling the API directly, sleep for retry_after_ms then retry.
  2. Batch small calls in the shipped endpoint by sending multiple items to POST /v1/capture/log instead of one POST per row.
  3. Move heavy training calls to the cloud backend or self-hosted compute via kolm cloud train.
  4. If you need a higher steady rate, request a quota bump at /account/billing.
HTTP 503 capture_store_unavailable

The durable capture store is offline. The request was rejected, not silently dropped.

What you see

HTTP/1.1 503 Service Unavailable
x-kolm-capture-durable: false
{"error":"capture_store_unavailable","retry_after_ms":5000}

Root cause

W212 changed every capture handler to refuse writes when the durable backing store (vercel_postgres, vercel_kv, or sqlite) is not reachable, instead of silently swallowing the row. The response header x-kolm-capture-durable: false is the canary.

What to do

  1. Probe the backing store: GET /v1/capture/health returns {"durable":true|false,"driver":"vercel_postgres","latency_ms":...}.
  2. If self-hosting, restart the storage driver. Check KOLM_STORE_DRIVER env var and confirm ~/.kolm/data/kolm.sqlite is writable.
  3. The CLI buffers retries on the client side. Re-send with the same Idempotency-Key header to avoid double-write.
  4. For status, watch status.kolm.ai.
HTTP 503 receipt_unverifiable

The receipt could not be cross-checked with the witness in time.

What you see

HTTP/1.1 503 Service Unavailable
{"error":"receipt_unverifiable","reason":"rekor_unreachable","timeout_ms":4000}

Root cause

The build receipt has a sigstore Rekor entry id, but the public log was unreachable within the verifier timeout. The artifact bytes are still valid; the witness leg of the chain just could not be confirmed online.

What to do

  1. Retry the verify call. Transparency logs are eventually consistent; retry typically succeeds within 60 seconds.
  2. Run kolm verify --offline path/to/artifact.kolm to skip the witness leg if you only need signature plus Ed25519 receipt chain.
  3. For air-gapped installs, point the verifier at your private witness via KOLM_WITNESS_URL.
  4. If Rekor is down for an extended period, check status.sigstore.dev.
BYOC

BYOC attestation failure. The cluster nodes did not produce a TEE quote the host trusts.

What you see

kolm cloud deploy --byoc
[error] attestation_failed: node-7 SEV-SNP quote rejected (PCK chain mismatch)

Root cause

The node failed to produce a valid AMD SEV-SNP, Intel TDX, or Nitro attestation quote. Usually the host firmware is too old, the PCK certificate chain is stale, or the node booted outside the expected secure-boot policy.

What to do

  1. Run kolm doctor --byoc to dump the per-node attestation report and the expected policy hash.
  2. Update the host BIOS or the cloud-provider AMD SEV firmware on the affected node.
  3. Refresh the PCK chain: kolm cloud byoc refresh-pck --node node-7.
  4. If a node cannot attest, drain it: kolm cloud byoc drain node-7. The artifact will not deploy until the cluster meets the minimum attested-node count.
Trainer

Trainer timeout or OOM. The student model did not fit or the step budget was exhausted.

What you see

[trainer] OOM at step 142 of 200 (peak 23.8 GiB on 24 GiB device)
[trainer] killed by watchdog after 1800s wall time

Root cause

Either the student base is too large for the device, the batch size is too high, or the wall-clock budget is too tight. The runtime kills training instead of paging to disk to avoid corrupting half-written shards.

What to do

  1. Drop one student tier: kolm distill --student-base auto picks the largest student that fits in doctor --detect-hw output.
  2. Halve the batch: --batch 8. K-score impact is usually under 1 percent on specialist tasks.
  3. Raise the wall budget: --max-steps 500 --max-wall 3600.
  4. If still OOM, switch to a smaller student via --student-base qwen-3.5b or move to kolm cloud train on an A100.
Compile

Compile refuses with "dataset empty". The capture filter matched zero rows.

What you see

kolm compile --namespace claims-router
[error] dataset_empty: 0 rows match {namespace:'claims-router', since:'2026-05-01'}

Root cause

The compile filter resolves to no captured pairs. Most often the namespace name is misspelled, the --since window is too tight, or the captures landed under a different namespace because the proxy header was set wrong.

What to do

  1. List what is on disk: kolm what --json shows the namespaces and per-namespace pair counts.
  2. Confirm the proxy capture is writing the right namespace: kolm tail captures --namespace claims-router.
  3. Widen the time window: --since 2026-01-01.
  4. If the namespace is genuinely empty, run the capture step longer or import a JSONL with kolm compile --from-jsonl ./pairs.jsonl.
Compile

Compile blocks on K-score below gate. The artifact would not pass production gates.

What you see

[gate] frozen_set K-score 0.78 below threshold 0.85
[gate] refuse_to_ship (use --force to override)

Root cause

The frozen-set gate failed. Either the student is too small, the recipe is underspecified, or the captures are mixing two tasks under one namespace.

What to do

  1. Inspect failing rows: kolm fix <artifact.kolm> surfaces the top failing cases and suggests seed rows.
  2. Add or relabel captures: kolm fix <artifact.kolm> --apply writes <basename>.fix-seeds.jsonl to feed back to compile.
  3. Lower the gate intentionally during exploration: --k-target 0.75. Do not lower below the production threshold for a release.
  4. Switch the student up one tier or extend training: --student-base qwen-7b --max-steps 500.
Quantize

Quantize says torch not installed. The Python sidecar is missing.

What you see

kolm quantize --to int4 phi-redactor.kolm
[error] python_sidecar_missing: torch not importable (python_ok=true, torch_ok=false, bnb_ok=false)

Root cause

kolm's quantizer (scripts/quantize.py) needs torch plus bitsandbytes for int4 and gptq, or auto-gptq for awq. The host has Python but the wheels are not installed in the kolm venv.

What to do

  1. Run kolm doctor --quantize to print which wheels are missing.
  2. Install into the kolm venv: kolm quantize install (downloads matching torch + bnb wheels for the local CUDA or ROCm version).
  3. On macOS or CPU-only, use --to int8 (no bnb required) or skip quantization.
  4. For air-gapped installs, copy a pre-warmed venv from a mirror node; see /airgap.
Verify

kolm verify returns 200 with verified: false. The chain has a broken link.

What you see

{
 "verified": false,
 "signature": "ok",
 "receipt_chain": "ok",
 "frozen_set": "ok",
 "witness": "rekor_uuid_not_found"
}

Root cause

One leg of the chain failed. Common cases: the Rekor entry id was rotated out of the test log, the artifact was rebuilt without re-signing the manifest, or the frozen-set hash on disk does not match the receipt.

What to do

  1. Read the per-leg breakdown in the JSON response. Whichever field is not "ok" is the failing link.
  2. If signature failed, the artifact was modified after sign. Re-fetch from the source.
  3. If witness failed, retry; transparency logs are eventually consistent. Or run --offline.
  4. If frozen_set failed, the in-artifact frozen samples diverged from the receipt hash. Re-compile.
SSE

SSE /v1/capture/stream disconnects every few minutes. Idle timeout from a proxy or CDN.

What you see

event: open
data: {"ok":true}

... (3 minutes silence) ...

(connection closed by remote)

Root cause

Most front proxies (Vercel edge, Cloudflare, nginx default) cut idle HTTP connections at 60 to 300 seconds. The SSE endpoint emits a keepalive comment every 25 seconds, but a proxy stripping comments or buffering responses will still cut the stream.

What to do

  1. The CLI re-connects automatically. kolm tail captures handles the gap and de-dupes by capture id.
  2. If using the API directly, listen for onerror and re-open the EventSource with the last seen Last-Event-ID.
  3. Disable proxy buffering: add X-Accel-Buffering: no response header (already set server-side) and confirm your reverse proxy honors it.
  4. For long-running streams behind a corporate proxy that buffers regardless, switch to kolm tail captures --poll 2s which uses short HTTP polls instead.
