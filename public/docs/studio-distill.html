Studio / Distill

Teacher generates. Student learns. You keep the weights.

Distill is the act of turning a frontier teacher (or council of teachers) into a smaller student model fine-tuned on the slice of behaviour you actually need. The student fits in a fraction of the VRAM, runs at frontier-comparable quality on its trained domain, and ships with a passport that names every teacher response that produced it.

The training loop

  1. Collect pairs — either from captured Wrapper traffic (kolm captures list --status approved) or from a synth recipe (kolm distill --recipe support-trinity-500).
  2. Query teacher — each prompt is sent to one or more teachers in spec.toml. Responses are stored verbatim with full provenance (model, version, timestamp, sampler params).
  3. Judge — an optional separate-lineage judge filters out responses that don't pass the rubric. See studio-judges.
  4. Train — the student receives the surviving pairs via LoRA (rank 16-32 default), QLoRA on 4-bit base, or full fine-tune (rare; only for <1B).
  5. Eval — held-out 10% split scored against the same judge for K-Score axes.
  6. Compile — the adapter merges into the base, the chat template + tokenizer + spec.toml + passport pack into a .kolm file. See studio-compile.

Loss shape

The default loss is DistiLLM-2 contrastive SKL + SRKL on sequence-KD outputs. In plain terms: the student isn't just trying to match the teacher's next token, it's trying to make the teacher's sequence more likely than the base model's sequence under its own distribution. This is what closes the ≈5-point K-Score gap that vanilla SFT leaves on the table.

Provenance & receipts

Every distill run writes a distill.receipt.json next to the artifact. It contains:

  • recipe_hash — sha256 of the spec.toml that drove the run.
  • teacher_council[] — model id, provider, version, sampler.
  • judge_id — the model + rubric used to filter.
  • pair_count — after judge filtering.
  • train_seconds, train_pps (pairs per second), peak_vram_gb.
  • kscore_pre, kscore_post, and per-axis delta.
  • artifact_hash — sha256 of the resulting .kolm.
  • signature — Ed25519 over the canonical receipt body.

You can re-verify the receipt locally with kolm verify distill.receipt.json, or remotely against the public verifier at https://kolm.ai/v1/verify/<cid>.

CLI

# Full pipeline from a captured namespace
kolm distill \
  --namespace support \
  --since-last-distill \
  --teacher claude-haiku-4-5 \
  --judge gemini-2.5-pro \
  --student qwen-2.5-7b \
  --method distillm-2 \
  --output trinity-support-v3.kolm

# Run a built-in recipe (no captures required)
kolm distill --recipe support-trinity-500 --output trinity-support.kolm

# Inspect a recipe before running
kolm distill --recipe support-trinity-500 --dry-run --json

# Resume an interrupted run
kolm distill --resume runs/trinity-support-v3/
Caveats. Distill loss numbers above are measured on internal eval sets (W869 Trinity-500). Your delta will depend on how representative your captures are of the deployment distribution. Always read the K-Score per-axis breakdown, not the headline number.
