Numerical accuracy in compiled artifacts.

Numerical content is the failure mode where a distilled model most often regresses against its teacher. 23 * 7 = 162 is one token away from 23 * 7 = 161 and the language model has no incentive to choose one over the other — the loss function does not know which is correct. The W759 surface gives you (1) a pure recursive-descent calculator that cannot run arbitrary code, (2) an extractor that pulls numbers out of model outputs in the seven shapes that actually appear in production captures, (3) a namespace warning flag so a distillation run on numeric-heavy traffic gets the calculator tool wired in before it starts. Version stamp: w759-v1.

Honesty note. The calculator runs ONLY pure + - * / arithmetic with parentheses — (no eval, no Function, no vm) no eval, no new Function(), no vm.runInNewContext, no **, no %, no bitwise operators, no identifiers, no function calls. Anything else is rejected at the tokenizer with error:'unsupported_operator'. Use kolm numeric calc <expr> to verify the boundary — kolm numeric calc "process.exit(1)" exits with syntax_error, not with the running process exiting.

Not a proof of correctness. A passing W759 eval means the response contained no numbers, OR the numbers/equations the response did contain were arithmetically consistent. It does NOT mean the response is true. A model that confidently says "The Eiffel Tower is 12345 feet tall" passes W759 because there is no equation to verify; the truth of the claim is the caller's problem.

Why numerical content needs special handling

Language model loss is character-level (technically token-level), not arithmetic. The model is rewarded for predicting the token "5" after the token "=" in the training corpus — not for the proposition that 2+3 equals five. When the corpus distribution shifts at inference time — the user asks "6 + 4 =" — the model has to generalize from token co-occurrence statistics to actual arithmetic, and it fails in predictable ways:

  • Off-by-one on multi-digit multiplication (23 * 7 → 162 when the right answer is 161)
  • Decimal-place drift (0.1 + 0.2 → 0.30000000000004 when the user-facing answer should be 0.3)
  • Unit confusion (5 kg + 5 lb → 10 kg; the units do not unify)
  • Sign flips on subtraction (3 - 7 → 4 instead of -4)
  • Scientific notation rounding (6.022e23 → 6.022e22; a 10x error)
  • Percentage compounding (a 10% discount on a 20% markup is not a 10% markup)

These failures are silent in standard distillation pipelines because the teacher-student loss only sees the token stream. Calculator-augmented distillation closes the gap by treating arithmetic as a TOOL call — the teacher emits <calculator>23 * 7</calculator>, the runtime evaluates, and the student learns the tool-call shape rather than memorizing the arithmetic. This is the W735 contract; W759 is the eval + namespace-flag side of the same coin.

How K-Score's E axis tracks numerical errors

K-Score's E (Edge cases) axis — one of the five letters in the score — tracks the rate of numerical / boundary failures on the holdout eval. A regression on E with the other axes flat is the canonical signal that the artifact has a numerical problem rather than a general quality problem. The W759 evaluator feeds the E axis through three signals:

SignalSourceWhat it means
equations_verified / equations_foundPOST /v1/numeric/evalOf the <lhs> = <rhs> patterns the response emitted, what fraction are actually true.
match_with_expectedPOST /v1/numeric/eval with expected_answerBoolean: did any extracted number match the ground truth within tolerance.
mean_ratio > thresholdGET /v1/numeric/namespace-flag/:namespaceNamespace-level signal: distillation will under-perform on numeric content unless the calculator tool is wired in.

The default tolerance is 0.001 (0.1%). For a calculation result of 1000, that is a tolerance of 1.0. For a result of 0.001, the floor at tolerance_pct itself kicks in to avoid impossible-to-meet tolerances near zero. The implementation is in verifyEquation() in src/eval-numeric.js.

Calculator tool integration (W735-compatible spec)

The CALCULATOR_TOOL_SPEC exported from src/calculator-tool.js matches the W735 tool-use distillation contract verbatim. Register it by reference; no adapter shim is required.

// src/calculator-tool.js
export const CALCULATOR_TOOL_SPEC = Object.freeze({
  name: 'calculator',
  description: 'Pure arithmetic only: + - * / parentheses. Returns numeric value.',
  input_schema: {
    type: 'object',
    properties: {
      expression: {
        type: 'string',
        description: 'An arithmetic expression using only + - * / and parentheses.'
      }
    },
    required: ['expression']
  },
  version: 'w759-v1'
});

The W735 distillation path consumes the spec, prompts the teacher to emit <tool>calculator</tool> invocations on its scratchpad, evaluates each invocation through evalSafeArithmetic(), and trains the student on the resulting tool-call traces. The student model never sees the calculator; it learns the call shape, and the runtime middleware intercepts the calls and routes them through the same safe evaluator at inference time.

Runtime middleware

runtimeCalculatorMiddleware({ response_text, auto_eval }) finds <lhs> = <rhs> patterns in a response, evaluates the LHS through the safe evaluator, and either (a) appends correction notes when the RHS disagrees, or (b) returns the expression list for the caller to surface in the UI:

const { runtimeCalculatorMiddleware } = require('./src/calculator-tool.js');

const r = runtimeCalculatorMiddleware({
  response_text: 'The order total is 23 * 7 = 162 dollars.',
  auto_eval: true,
});

// r.corrections = [
//   { expr: '23 * 7', claimed: 162, computed: 161, correction: '23 * 7 = 161' }
// ]
// r.augmented_text = 'The order total is 23 * 7 = 162 dollars.\n\n[calc: 23 * 7 = 162 -> 161 (correction)]'

Namespace warning flag — when distillation is high-risk

Before you start a distillation run, you usually want to know: is this namespace numeric-heavy enough that I need the calculator tool? The W759 namespace flag answers that with a single call:

kolm numeric flag-namespace --namespace pricing-quote --threshold 0.10
# {
#   "ok": true,
#   "namespace": "pricing-quote",
#   "mean_ratio": 0.182,
#   "threshold": 0.10,
#   "flagged": true,
#   "sample_n": 100,
#   "captures_seen": 4327,
#   "hint": "Distillation of numerical content requires calculator tool  -  see W759 docs",
#   "version": "w759-v1"
# }

The implementation in flagHighNumericNamespace():

  1. Loads up to sample_n captures from the namespace via findByTenant('observations', tenant_id) with a defense-in-depth tenant fence inside the loop (never trust the upstream table filter alone — W465 trap).
  2. For each capture, concatenates prompt + response and runs numericContentRatio() — the fraction of whitespace-separated tokens that match the numeric-token regex.
  3. Averages the per-capture ratios. Compares against the threshold. Returns flagged:true when the mean is above the threshold.

Empty-namespace handling is the standard W462/W464 honest-envelope pattern: when no captures exist we return ok:true with captures_seen:0 and a note:'empty_namespace' field. The caller distinguishes "no data" from "low-numeric" via captures_seen, not via the flagged flag.

mean_ratioUI signalRecommended action
< 0.05proseStandard distillation. Calculator tool is unnecessary overhead.
0.05 — 0.10mixedMixed content. Calculator tool optional; bench against a held-out numeric eval.
> 0.10numeric-heavyWire the calculator tool. Run W759 eval before promotion.

Number extraction shapes

The extractNumbers(text) function recognizes seven shapes, dispatched in priority order so a value matching multiple shapes (e.g. $1,234.56 matches both currency and thousands-separator) gets the most-specific tag:

ShapeExampleExtracted
Integer-42{ value: -42, kind: 'int', unit: null }
Float3.14{ value: 3.14, kind: 'float', unit: null }
Scientific1.5e-3{ value: 0.0015, kind: 'sci', unit: null }
Percent15%{ value: 0.15, kind: 'pct', unit: '%' }
Currency$1,234.56{ value: 1234.56, kind: 'currency', unit: '$' }
Thousands1,234,567{ value: 1234567, kind: 'int', unit: null }
Unit5 kg{ value: 5, kind: 'int', unit: 'kg' }

Each hit also carries span:[start,end] so the caller can highlight the source range. The extractor is conservative on the RHS — 5x does NOT become {value:5, unit:'x'} because x is far more often an algebraic variable than a unit. Three-or-more-letter words after a space are accepted as units verbatim; two-letter words are accepted only from a small allow-list (kg, mg, lb, ft, mi, km, etc.).

API + CLI surface

SurfaceInvocation
CLI (eval response)kolm numeric eval --text "<text>" [--expected N] [--tolerance-pct P]
CLI (local calc)kolm numeric calc "<expr>"
CLI (flag namespace)kolm numeric flag-namespace --namespace ns [--threshold 0.10]
API (eval)POST /v1/numeric/eval body {response_text, expected_answer?, tolerance_pct?, confirm:true}
API (calculator)POST /v1/numeric/calculator body {expression}
API (namespace flag)GET /v1/numeric/namespace-flag/:namespace

CLI exit codes are honest: 0 when the eval passes (no mismatches, namespace not flagged), 3 when a mismatch is found OR the namespace is flagged (informational — the operator should look), 2 on real errors (auth missing, bad arguments, network failure, syntax error in kolm numeric calc). The mismatch-found path is intentionally not exit-1 so CI can keep "review needed" separate from "build failed".

Quickstart

# Local arithmetic check — no network, no auth.
kolm numeric calc "23 * 7"
# → {"ok":true,"value":161,"version":"w759-v1"}

# Same expression with the unsupported `**` operator — rejected at tokenize.
kolm numeric calc "2 ** 10"
# → {"ok":false,"error":"unsupported_operator",
#         "detail":"unexpected_character:\"*\"",...}

# Eval a model response against an expected answer.
kolm numeric eval --text "23 * 7 = 162" --expected 161
# → {"ok":false,"equations_found":1,"equations_verified":0,
#         "errors":[{"kind":"equation_mismatch","lhs_value":161,"rhs_value":162,...}],
#         "match_with_expected":true, ...}
# exits 3 (mismatch found, informational)

# Flag a namespace as numeric-heavy before launching distillation.
kolm numeric flag-namespace --namespace pricing-quote --threshold 0.10
