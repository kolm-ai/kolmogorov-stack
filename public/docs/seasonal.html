Seasonal: capture tagging, variants, auto-select

A capture from black-friday week and a capture from a sleepy Tuesday in March are answering the same prompt against very different distributions. Seasonal gives you three primitives to put numbers on that fact: a per-row season + event tag stamped at capture time, a variant registry (support/seasonal-black-friday) so you can distill specialised students, and an auto-recommendation engine that picks the right variant from the calendar at query time. Frontier AI on your own infrastructure.

N-hemisphere default. The four-season carve-up is biased toward meteorological convention (Dec-Feb winter, Mar-May spring, etc.). Tenants in the southern hemisphere flip via a per-tenant setting in a follow-up. The envelope always echoes hemisphere:'north' so callers can detect the bias. Likewise the event registry (black-friday, cyber-monday, tax-season-us, back-to-school) is US-retail anchored on purpose — it is not exhaustive, and per-region overrides are a follow-up rather than more hardcoded ranges in the module.

Why the calendar belongs in the capture row

The student is trying to learn today's conditional distribution. When today is in a tight calendar window (the holiday rush, tax season, back-to-school), the conditional distribution shifts — query mix, expected response style, latency tolerance — and the same student model that handles "sleepy Tuesday" is the wrong instrument. Two routes out: (1) train one big student on the union and hope the loss averages well, or (2) distill a variant per window and route queries to the variant whose calendar matches.

Seasonal supports route 2. The work splits into three pieces: tag the rows so you can train a variant, register the variant in the namespace so the recommendation engine knows it exists, and recommend the variant at query time based on today's date. None of the three pieces are clever; the value is that they are explicit, idempotent, and plain about edge cases.

Per-row season + event tag

Every capture row carries season (winter|spring|summer|fall|null) and seasonal_events (a sorted array of event names active when the row was captured). Tagging happens server-side on the POST /v1/capture/log path BEFORE the row hits the event-store, so the tag travels with the row through every downstream consumer (datasets, label queues, bakeoffs).

The date considered is captured_at → created_at → Date.now() (first non-null wins). Unparseable dates produce season:null and seasonal_events:[] — plain, never guessed.

// Server adds these fields automatically:
{
  ...,
  season: "winter",
  seasonal_events: ["black-friday", "cyber-monday"]
}

Tagging is idempotent. A row that already carries a non-empty season string is left alone — the closer-to-source value always wins (matches teacher-version contract).

Byte-stability safety hatch

If a deploy needs to fall back to pre-capture behavior in a hotfix, set KOLM_W748_SEASONAL_TAGGING=off. With the flag off, POST /v1/capture/log writes rows without season or seasonal_events — byte-stable with the shape. Default is on.

Event registry

Events are tight calendar windows where the conditional distribution shifts hard. The registry is small on purpose — it exists so the recommendation engine has SOMETHING to match against on day 1, not so we replicate every public holiday catalog. Cross-year ranges (holiday: Dec 15 - Jan 5) are honoured.

EventWindowTypical use
black-fridayNov 21 - Nov 30Pre-Thanksgiving retail ramp
cyber-mondayDec 1 - Dec 2Post-Thanksgiving e-commerce peak
holidayDec 15 - Jan 5Year-end gift + returns season
tax-season-usJan 15 - Apr 15US tax prep / advisory queries
back-to-schoolAug 1 - Sep 15Education + parental query spike

Adding holidays for non-US tenants is a follow-up that should land per-region overrides — not bake more hardcoded ranges in the module.

Seasonal-variant namespaces

A seasonal-variant namespace is a sub-namespace named after the season or event it serves. Registering one tells the recommendation engine "this variant exists; if today matches, route to it":

kolm seasonal create-variant support black-friday
# → {"ok":true,"namespace":"support","variant":"black-friday",
#      "variant_namespace":"support/seasonal-black-friday",
#      "kind":"event","version":"w748-v1"}

The variant name must be a known season (winter, spring, summer, fall) or a known event key (black-friday, etc.). Free-form labels are rejected with HTTP 400 unknown_variant — preventing typos that silently never match.

From there, captures and distills against support/seasonal-black-friday behave like any other namespace. The seasonal layer just gives you the registry + auto-routing on top.

Recommendation engine

The recommendation engine looks at today's date and the variants you've registered, then picks one with explicit priority:

  1. Active event match. If today falls in an event window (e.g. Dec 1 is cyber-monday) AND a variant matching that event is registered AND has captures > 0, that wins. Event windows are tighter than seasons, so they win the tie.
  2. Season match. If no event matched, fall back to today's season (e.g. winter). Same gate: variant must be registered AND have captures > 0.
  3. Plain null. No variant matched. recommended:null + a reason string that names the seasons + events checked + the variants registered. The dashboard renders this as an actionable empty state.
kolm seasonal recommend support
# → {
#   "ok": true, "namespace": "support",
#   "season": "winter",
#   "active_events": ["black-friday"],
#   "recommendation": {
#     "recommended": "black-friday",
#     "reason": "active_event_match: black-friday (window 11/21..11/30)"
#   },
#   "version": "w748-v1"
# }

The reason field is always populated, even on success — so the dashboard can show why the engine picked what it picked, not just what. Caveats over guessing.

API surface

GET /v1/seasonal/:namespace

Returns the seasonal distribution + active events + recommendation for the namespace.

curl -H "Authorization: Bearer $KOLM_API_KEY" \
  https://kolm.ai/v1/seasonal/support

# → {
#   "ok": true,
#   "namespace": "support",
#   "total_captures": 142,
#   "hemisphere": "north",
#   "season": "winter",
#   "active_events": ["holiday"],
#   "distribution": {
#     "by_season": {"winter":61,"spring":18,"summer":22,"fall":37,"_unknown":4},
#     "by_event": {"black-friday":9,"holiday":12},
#     "total": 142
#   },
#   "registered_variants": ["holiday","winter"],
#   "recommendation": {"recommended":"holiday","reason":"active_event_match: holiday (window 12/15..1/5)"},
#   "version": "w748-v1"
# }

POST /v1/seasonal/variant

Registers a seasonal-variant namespace. Body: {namespace, variant}. Variant must be a known season or event key.

curl -X POST -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"namespace":"support","variant":"black-friday"}' \
  https://kolm.ai/v1/seasonal/variant

# → {
#   "ok": true,
#   "namespace": "support",
#   "variant": "black-friday",
#   "variant_namespace": "support/seasonal-black-friday",
#   "kind": "event",
#   "version": "w748-v1"
# }

CLI

kolm seasonal show <namespace>
# → full envelope (distribution + events + recommendation)

kolm seasonal create-variant <namespace> <variant>
# → registers a seasonal-variant namespace

kolm seasonal recommend <namespace>
# → just the recommendation block (for automation)

Version pin

Every seasonal envelope echoes version: "w748-v1". The version string bumps to w748-v2 only when the envelope shape, season vocabulary, or event registry changes in a non-additive way. Adding new optional fields keeps the version. Removing or renaming fields, OR changing the seasons array, OR adding/removing/changing an event window's start/end day — those bump the version.
