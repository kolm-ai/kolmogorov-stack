Verify

kolm verify <artifact.kolm> re-hashes every bundled file, re-checks the Ed25519 signature, refuses on train/hold-out leakage, missing binary, or synthetic-only seeds. One CLI command, six closed-enum failure codes, offline by default.

What verify checks

A .kolm bundle ships with a signed receipt that pins every load-bearing file by sha256, plus the train/hold-out hashes, source-seed count, runtime target, and Ed25519 signature. kolm verify re-derives all of those and refuses to proceed unless every check passes.

  1. File hash, re-sha256 each file in the bundle, compare to the receipt’s pinned hash. A single bit-flip aborts.
  2. Signature, re-check the Ed25519 signature over the canonical receipt JSON. Refuses if the signing key is unknown or the signature does not validate.
  3. Hold-out disjointness, re-derive the holdout sha256-mod buckets from the dataset id, reject if any training row falls inside the holdout slice.
  4. Runtime binary, resolve the runtime declared in manifest.runtime. Refuses if the binary is missing or its hash does not match the manifest pin.
  5. Source seeds, refuse on a seeds.jsonl that contains zero rows traceable to a real captured event (synthetic-only artifacts fail the production gate).
  6. Schema, refuse on a manifest that does not validate against the RS-1 receipt schema.

Run it

kolm verify ./phi-redactor.kolm
# verify phi-redactor.kolm
# files 17 / 17 sha256 ok
# signature ed25519 ok key=kolm-prod-2026-04
# holdout buckets 90..99 disjoint from train (4,217 rows)
# runtime gguf 0.3.7 sha256 ok
# seeds 2,140 real / 0 synthetic
# schema rs-1 ok
# OK receipt=r_2f93... generated_at=2026-05-19T17:42:08Z

The six closed-enum failure codes

Every refusal sets exit code 2 and prints a single failure code. Failure codes are stable across releases so CI gates can pin on them.

CodeWhat it means
file_hash_mismatchA bundled file’s sha256 does not match the receipt’s pinned hash. Most common cause: the bundle was edited after signing.
signature_invalidEd25519 signature does not validate against the receipt JSON, or the signing key is not in the trusted set.
holdout_leakA training row falls inside the holdout sha256-mod buckets. The artifact saw its own eval set during training, reject.
runtime_missingThe runtime binary declared in manifest.runtime is missing or its hash does not match the pinned hash.
seeds_synthetic_onlyseeds.jsonl has zero rows traceable to a real captured event. Synthetic-only artifacts cannot ship through the production gate.
schema_invalidThe manifest does not validate against the RS-1 receipt schema.

Wire it into CI

Verify is the gate. Every deploy pipeline should re-verify the artifact before running it, and every promotion should re-verify before the bake-off. The signature + holdout disjointness checks are the load-bearing guarantees, everything else can be re-derived from them.

# reject any artifact that fails verify
kolm verify ./candidate.kolm || exit 1

# re-verify after every download
curl -sSfLO https://example.com/builds/v1.2.3.kolm
kolm verify ./v1.2.3.kolm || { rm v1.2.3.kolm; exit 1; }

Offline by default

Verify makes zero network calls. The trusted key set lives in ~/.kolm/keys/, rotation is explicit. The receipt schema lives in the bundle itself. Air-gapped builds verify the same way production builds do.

Next steps

If verify passes, runtime dispatches the artifact to the runtime declared in the manifest. If verify fails, the failure code tells you exactly which gate refused, fix it at the source, re-bundle, re-sign, re-verify.

Security →Receipts, signatures, threat model. RS-1 spec →Canonical receipt schema (JSON + markdown). Runtime →What verify hands off to.
