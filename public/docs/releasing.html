Releasing.

Release a new artifact version the way every honest release engineer wishes they could: split production traffic 50/50 between the old and new artifact, let the W778 Welch t-test decide whether the new version is actually better, promote the winner, and let the auto-rollback hook revert any promotion the gate later turns against. Two CLI verbs, two modules, four route prefixes, one audit log.

Why an explicit release path

Every team eventually invents this pipeline themselves: a feature flag, a logging table, a stats script, a Slack alert, a runbook. kolm packages the loop directly because every artifact build SHOULD see traffic before it sees all traffic, and every promotion SHOULD have a reversible audit row. kolm ab is the orchestrator. kolm stat-sig is the math. Together they replace ~400 lines of glue code per release pipeline.

Quickstart

Three commands. Assume you've already built two artifacts (model-v1 and model-v2).

# 1. Open the test - 50/50 split, advisory sample target 1000 per arm
kolm ab start --namespace prod --arm-a model-v1 --arm-b model-v2 --split 0.5

# Returns: { ok:true, ab_test_id:"abt_<hex>", record:{...}, version:"w777-v1" }

# 2. Route per-request via the deterministic FNV-1a hash. Same request_hash
#    always lands on the same arm so a retried call stays on its arm.
kolm ab assign abt_<hex> --request-hash user_42
# Returns: { arm:"a", artifact_id:"model-v1", ab_test_id:"...", version:"w777-v1" }

# 3. After ~30+ outcomes per arm, ask the gate what to do
kolm stat-sig gate --ab-test-id abt_<hex>
# Returns: { decision:"pass"|"fail"|"insufficient", p:0.0023, effect_size:0.18, ... }

When the gate returns decision:"pass", promote arm B:

kolm ab promote abt_<hex> --arm b --reason "won_the_test"
# All future kolm ab assign calls now freeze to arm B (artifact=model-v2).

Recording outcomes

Outcomes feed the gate. Every time you finish serving a request from arm A or arm B, record the K-Score and latency so the W778 gate can score it later. The outcome event lands in the canonical event-store under workflow_id="w777:ab"; W411 tenant fence enforces ownership.

# From the SDK / your service layer
POST /v1/ab-tests/<ab_test_id>/assignments?request_hash=<h>
→ { arm, artifact_id }

# After your call returns
POST /v1/capture/log
{
  "workflow_id": "w777:ab",
  "ab_test_id": "abt_...",
  "arm": "a"|"b",
  "kscore": 0.93,
  "latency_ms": 220,
  ...
}

The gate (W778)

The gate combines three checks. All three must pass to return decision:"pass":

CheckDefaultOverride
Sample sizemin_n >= 30 per arm--min-n
Statistical significance (Welch's t-test, two-tailed)p < alpha = 0.05--alpha
Effect size|mean_b - mean_a| >= 0.01--min-effect

Decisions:

  • pass — arm B beats arm A, statistically significant, and the effect is large enough to be worth shipping. Promote.
  • fail — either p >= alpha (no signal) OR arm B underperforms arm A (regression). If arm B was previously promoted, this is the auto-rollback trigger.
  • insufficient — sample size or effect size below floor. Keep collecting outcomes; do not act.

The Welch t-test uses Welch-Satterthwaite degrees of freedom (no equal-variance assumption), Hill's series approximation / regularized incomplete beta for the Student's t CDF, and Lanczos approximation for log gamma. Zero-variance arms return p = 1.0 instead of NaN.

Auto-rollback

After promotion, call kolm ab rollback on any schedule that fits your operational cadence (e.g., hourly cron, post-deploy hook, on-alert from drift-alert). It re-runs the gate and reverts only when BOTH conditions hold:

  1. The promoted arm is b (rollback is only meaningful in the "old vs new" direction).
  2. The gate decision is fail AND arm B's mean is strictly less than arm A's mean.
kolm ab rollback abt_<hex>
# rolled_back=yes  reason=arm_b_underperforms_and_was_promoted
# — OR —
# rolled_back=no   reason=arm_b_winning|arm_b_not_clearly_worse|insufficient_data

After a rollback, kolm ab assign returns arm: "a" for ALL request hashes regardless of bucket math, with reason: "rolled_back_to_arm_a". The audit log captures the rollback under workflow_id="w777:ab:rollback".

API surface

MethodPathPurpose
POST/v1/ab-tests/createOpen a new test
GET/v1/ab-testsList tests for the authenticated tenant
GET/v1/ab-tests/:idStatus envelope: counts, means, Welch t-test
GET/v1/ab-tests/:id/assignments?request_hash=XResolve a request to an arm
POST/v1/ab-tests/:id/promoteFreeze future routing to the chosen arm
POST/v1/ab-tests/:id/rollbackTrigger the auto-rollback check
POST/v1/ab-tests/:id/stopMark stopped (no future assignments)
POST/v1/stat-sig/testWelch t-test on raw samples_a + samples_b
GET/v1/stat-sig/gate?ab_test_id=XGate decision pulling samples from W777

All routes auth-gated. Tenant fence forced from session (never from request body) per W411 conventions.
