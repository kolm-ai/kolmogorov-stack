Open-source AI workbench

Regulatory toolkit.

Seven routes that translate an artifact manifest into the compliance shape your auditor expects. EU AI Act Annex IV technical documentation, intended-use risk classifier, per-namespace human-in-loop thresholds, data-governance report, extended model card, GRC payloads for OneTrust, ServiceNow, IBM OpenPages.

Seven routes

All routes are auth-required and tenant-scoped via req.tenant_record.id. Each route takes an inline artifact_manifest in the body (a follow-up wave will add artifact_id resolution). Modules are imported lazily so cold daemons that never touch the regulatory surface do not pay for the imports.

routeoutput
POST /v1/reg/eu-aiact-docsAnnex IV technical-docs blob (six sections)
POST /v1/reg/classify-riskIntended-use risk class (high / limited / minimal)
POST /v1/reg/hil/thresholdSet per-namespace human-in-loop confidence threshold
GET /v1/reg/hil/thresholdRead back the threshold
GET /v1/reg/data-governanceData-governance markdown report
POST /v1/reg/model-cardExtended model card (HuggingFace + EU AI Act extensions)
POST /v1/reg/grc-exportVendor-shaped GRC payload (OneTrust / ServiceNow / IBM OpenPages)

EU AI Act Annex IV docs

Annex IV of the EU AI Act requires high-risk system providers to maintain a technical documentation packet. The generator emits the six required sections from the inline artifact manifest plus optional tenant metadata (legal name).

POST /v1/reg/eu-aiact-docs

curl -X POST https://kolm.ai/v1/reg/eu-aiact-docs \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "artifact_manifest": {
      "artifact_id": "art_...",
      "model": { "name":"qwen-2.5-7b-support", "version":"1.0.0" },
      "training": { "dataset_size":120000, "k_score":0.87 },
      "privacy": { "dp_epsilon": 2.3, "dp_delta": 1e-5 }
    },
    "tenant_metadata": { "legal_name": "Acme GmbH" },
    "format": "json"
  }'

# -> { "ok":true, "annex_iv":{ "section_1_general":..., "section_2_design":...,
#      "section_3_data":..., "section_4_validation":..., "section_5_post_market":...,
#      "section_6_changes":... }, "version":"w834-v1" }

The six sections map 1:1 to Annex IV: general description, design and development, data and data governance, monitoring and validation, post-market monitoring, changes log. The output is a JSON tree by default; pass "format":"markdown" to receive a rendered markdown blob suitable for archival.

Risk classifier

The EU AI Act tiers systems by intended use into high-risk, limited-risk, and minimal-risk. The classifier reads intended_use + deployment_context off the manifest and returns the inferred class plus the rule chain that produced it.

POST /v1/reg/classify-risk

curl -X POST https://kolm.ai/v1/reg/classify-risk \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "artifact_manifest": {
      "intended_use": "credit-decision",
      "deployment_context": "regulated-banking"
    }
  }'

# -> { "ok":true, "risk_class":"high", "rules_fired":[
#      "credit-decision is an Annex III high-risk use case",
#      "regulated-banking context triggers the conformity assessment requirement"
#   ] }

The classifier is deterministic and the rules are stamped on the response so an auditor can see the exact chain that produced the verdict. A manifest without enough context returns {ok:true, risk_class:'unclassified', hint:'add intended_use'} rather than guessing.

Human-in-loop threshold

For limited- and high-risk uses, the EU AI Act requires meaningful human oversight. The HIL threshold endpoint sets the per-namespace confidence cutoff below which a response is queued for human review instead of returned to the caller.

POST /v1/reg/hil/threshold

curl -X POST https://kolm.ai/v1/reg/hil/threshold \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "namespace": "credit-decision", "threshold": 0.85 }'

# -> { "ok":true, "namespace":"credit-decision", "threshold":0.85, "set_at":"..." }

GET /v1/reg/hil/threshold

curl -H "Authorization: Bearer $KOLM_API_KEY" \
  "https://kolm.ai/v1/reg/hil/threshold?namespace=credit-decision"
# -> { "ok":true, "namespace":"credit-decision", "threshold":0.85 }

The threshold compares against the model's calibrated confidence per response. Below threshold the gateway returns the response with hil_required:true and queues the row in the approvals queue for a human reviewer.

Data governance report

A markdown report summarizing the data lineage for the tenant: where captures originate, how they are redacted (/ / receipts), retention policy, deletion mechanics. Suitable as an auditor handover.

GET /v1/reg/data-governance

curl -H "Authorization: Bearer $KOLM_API_KEY" \
  https://kolm.ai/v1/reg/data-governance
# -> { "ok":true, "report_markdown":"# Data governance report\n\n## Capture origins\n...",
#      "version":"w834-v1" }

Extended model card

The extended model card combines the HuggingFace model-card fields with the EU AI Act-required disclosures (intended use, out-of-scope use, limitations, training data summary, evaluation methodology, environmental impact).

POST /v1/reg/model-card

curl -X POST https://kolm.ai/v1/reg/model-card \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "artifact_manifest": { ... } }'

# -> { "ok":true, "model_card_markdown":"---\nlanguage: en\nlicense: apache-2.0\n...",
#      "version":"w834-v1" }

GRC export

Three vendor schemas supported: onetrust, servicenow, ibm_openpages. Each emits the artifact metadata in the shape the vendor's import API expects, so the compliance team can drop the JSON straight into their GRC workflow.

POST /v1/reg/grc-export

curl -X POST https://kolm.ai/v1/reg/grc-export \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "artifact_manifest": { ... }, "vendor": "onetrust" }'

# -> { "ok":true, "vendor":"onetrust", "payload":{ ...vendor-shaped JSON... },
#      "version":"w834-v1" }

The exporter is deterministic and stamps the source artifact_id + manifest hash on every payload so the compliance team can re-derive the export later. Unrecognized vendors return {ok:false, error:'vendor_unsupported', supported:['onetrust','servicenow','ibm_openpages']}.
