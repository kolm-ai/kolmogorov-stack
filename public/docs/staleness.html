Staleness: capture decay, TTL, teacher-version drift

A capture from last week and a capture from 18 months ago do not deserve the same vote in your training mix. And once your teacher upgrades from claude-opus-4-6 to claude-opus-4-7, the rows that came before silently belong to a different conditional distribution than the rows that came after. Staleness gives you three primitives to put numbers on those facts: a recency weight, a per-namespace TTL, and a teacher_version tag on every row. Frontier AI on your own infrastructure.

Why a capture's age is load-bearing

The student is trying to learn today's conditional distribution. The corpus you train on is a mixture of every age band, weighted equally by default. If 60% of your captures came from a 90-day stretch where the teacher was a different model version, the unweighted student averages over a distribution that no longer exists in production.

The fix is two-sided. First, down-weight older rows in the sampler so recent behavior dominates the loss. Second, tag each row with the teacher that produced it, so you can audit "what fraction of the loss is being computed against an obsolete teacher" before you ship.

Recency weighting - exponential decay

The recency weight is a pure exponential decay in age (days), parameterised by a half-life. Same-day captures get weight 1.0. After one half-life of age, weight is 0.5. After two half-lives, 0.25. Etc. Future-dated captures (clock skew, replay-from-archive) are clamped to 1.0 so a skewed client never inflates a row above present-day mass.

weight = 0.5 ** (age_days / half_life_days)

# Defaults
half_life_days = 30  # tune via `kolm staleness sampler-weights --half-life N`

The 30-day default is a starting point, not a religious commitment. Domains with rapid drift (news, code, market data) want a shorter half-life (7 or 14 days). Domains with slow drift (legal templates, medical coding) want longer (60 or 90). The CLI lets you see the resulting distribution before you hard-code a choice:

kolm staleness sampler-weights production --half-life 14

TTL policy — configurable per-namespace

The TTL knob is separate from the weight knob. Weights down-weight a row; TTL deletes it. A 90-day TTL on a namespace evicts every row older than 90 days, permanently. The two together let you say "weight aggressively toward recency in the sampler AND drop rows past a known cliff."

Destructive. kolm staleness apply-ttl permanently deletes capture rows from the event-store. Run with --yes only after you have looked at kolm staleness show projections (the freshness distribution + projected eviction counts at 30d/90d). The CLI requires you to retype the namespace when run interactively. There is no undo.

Defaults are deliberately conservative. A namespace with no TTL policy configured is treated as ttl_days = null, meaning no rows are evicted — ever — until you opt in. We never invent a default retention period: a user who has not configured a TTL must never lose rows by surprise.

Setting a TTL

Owner-only. The dashboard at /account/staleness exposes a dropdown for {none, 30d, 90d, 365d}; the CLI exposes the same via kolm staleness apply-ttl <namespace> <days>. Both paths route through POST /v1/staleness/apply-ttl, which is gated to owner-equivalent privilege (admin, local-daemon, or workspace owner). Viewer and member roles get HTTP 403 forbidden.

Teacher version tagging

Every capture row carries a teacher_version string (e.g. claude-opus-4-7, gpt-4o-2024-08) and a teacher_provider normalised name (e.g. anthropic). Tagging happens server-side on the POST /v1/capture/log path BEFORE the row hits the event-store, so the tag travels with the row through every downstream consumer (datasets, label queues, bakeoffs).

The version is resolved at insert time using a strict priority order:

  1. Provider-specific env: KOLM_TEACHER_VERSION_<UPPER(provider)>
  2. Generic env: KOLM_DISTILL_TEACHER (the existing repo env)
  3. Documented per-provider default (anthropic → claude-opus-4-7; openai → gpt-4o)
  4. Plain fallback: unknown_teacher_v0 — queryable string, never null

Tagging is idempotent. A row that already carries a non-empty teacher_version (e.g. a passthrough capture that already knew the exact upstream model id) is left alone. This means a capture stamped at proxy time with claude-opus-4-7-20251007 survives intact even if the env default later changes — the closer-to-source value always wins.

Byte-stability safety hatch

If a deploy needs to fall back to pre-capture behavior in a hotfix (e.g. a downstream consumer doesn't tolerate the new fields), set KOLM_W746_TEACHER_TAGGING=off. With the flag off, POST /v1/capture/log writes rows without teacher_version or teacher_provider — byte-stable with the shape. Default is on.

Freshness distribution — the timeline view

The /account/staleness page renders the per-bucket freshness distribution as a horizontal stacked bar:

BucketSemanticTypical use
≤1dCaptures from the last 24 hoursSpotting incident-traffic spikes
≤7dLast weekSampler weight near 1.0 at 30d half-life
≤30dLast monthWeight ~0.5 at 30d half-life
≤90dLast quarterWeight ~0.125 at 30d half-life
≤365dLast yearLong tail; commonly the TTL target
>365dOlder than a yearOverflow bucket; usually safe to TTL

The overflow bucket also catches rows with unparseable captured_at timestamps — we never silently drop a row from the chart. If >365d looks larger than expected, suspect a serialization bug in a capture writer, not normal aging.
