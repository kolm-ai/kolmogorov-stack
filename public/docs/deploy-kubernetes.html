Run & Govern / Deploy

Kubernetes manifests for a compiled artifact.

generateKubernetesManifests from src/deploy-generators.js emits six YAML documents in one string: a ConfigMap, a PersistentVolumeClaim, an init Job that pulls the artifact, a Deployment with GPU resource limits and probe configuration, a Service, and a HorizontalPodAutoscaler. The order is intentional — kubectl apply -f processes top-down and the PVC, ConfigMap, and Job must be ready before the Deployment binds to them.

Invocation

The function takes four parameters and returns a string. Write that string to a single .yaml file or pipe it into kubectl apply -f -.

import { generateKubernetesManifests } from './src/deploy-generators.js';

const yaml = generateKubernetesManifests({
  artifact: 'qwen3-8b-support-q4km',
  runtime: 'vllm',         // 'vllm' or 'llama.cpp'
  gpu_count: 1,            // limits AND requests on the Deployment
  namespace: 'kolm-prod',  // sanitized to [a-z0-9-]
});

fs.writeFileSync('kolm-deploy.yaml', yaml);

Parameters

FieldTypeDefaultNotes
artifactstringrequiredArtifact id used in image command args, ConfigMap data, and the download URL the init Job hits.
runtime'vllm' | 'llama.cpp''vllm'Chooses image (vllm/vllm-openai:latest vs ghcr.io/ggml-org/llama.cpp:server) and the args block.
gpu_countinteger1Sets both resources.requests.nvidia.com/gpu and resources.limits. Coerced to max(1, n).
namespacestring'default'Sanitized to lowercase and [a-z0-9-] characters. Used in every metadata block.

The six documents

Each YAML doc is separated by ---. They are emitted in dependency order so a single kubectl apply sees the dependencies first.

OrderKindNamePurpose
1ConfigMapkolm-<artifact>-configStores artifact_id, runtime, port, served_model_name. Used by sidecars and external tools.
2PersistentVolumeClaimkolm-<artifact>-models100Gi ReadWriteOnce. Shared between the init Job (writes) and the Deployment (reads readOnly).
3Jobkolm-<artifact>-init-downloadOne-shot curl from $KOLM_REGISTRY_URL/artifacts/<id> using optional kolm-registry-secret Secret. Idempotent — skips if file exists.
4Deploymentkolm-<artifact>One replica by default. Mounts PVC readOnly: true. Liveness and readiness probes hit /health on port 8000.
5Servicekolm-<artifact>ClusterIP on port 8000. Switch to NodePort or LoadBalancer after pasting if you need external exposure.
6HorizontalPodAutoscalerkolm-<artifact>1 to 5 replicas, scales on 70% CPU. Hand-edit to add custom metrics (request queue depth, GPU utilization).

Liveness and readiness

Both probes hit /health on the container port. The values reflect what vLLM and llama.cpp actually need to come up: vLLM cold-loads weights and the optimizer for the first request, so the liveness probe sleeps 60 seconds before its first poll.

livenessProbe:
  httpGet: { path: /health, port: 8000 }
  initialDelaySeconds: 60
  periodSeconds: 30
  timeoutSeconds: 5
  failureThreshold: 5
readinessProbe:
  httpGet: { path: /health, port: 8000 }
  initialDelaySeconds: 30
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 3

Init Job — downloader contract

The Job pulls the artifact from your registry on first apply. Two env vars sourced from a Secret named kolm-registry-secret:

env:
  - name: KOLM_REGISTRY_URL
    valueFrom:
      secretKeyRef: { name: kolm-registry-secret, key: url, optional: true }
  - name: KOLM_API_KEY
    valueFrom:
      secretKeyRef: { name: kolm-registry-secret, key: api_key, optional: true }

The downloader image is curlimages/curl:8.10.1 so it works in air-gap deployments when the image is mirrored. The Job sets ttlSecondsAfterFinished: 600 so finished Job pods are reaped automatically.

Apply

kubectl create namespace kolm-prod         # if it does not exist
kubectl create secret generic kolm-registry-secret \
  --namespace kolm-prod \
  --from-literal=url='https://registry.example.com' \
  --from-literal=api_key='secret_xxx'

# Apply all six documents in one shot.
kolm deploy --target k8s --artifact qwen3-8b-support-q4km --namespace kolm-prod \
  | kubectl apply -f -
Caveats. The HPA uses CPU utilization out of the box because that metric ships in every cluster. For a GPU-bound workload you almost certainly want to swap the metrics: block for a custom metric — queue depth from /metrics on the runtime sidecar or GPU memory from dcgm-exporter. The generator emits a placeholder you can edit; it does not auto-detect cluster metric availability.
Limitations. The manifests are operator-facing — no auto-generated Ingress, no NetworkPolicy, no PodDisruptionBudget. Those are environment-specific and would be wrong to generate without knowing your ingress controller class, namespace network policy stance, and maintenance window contract. The generator emits the minimum that runs; you bolt on the cluster-specific surface.
