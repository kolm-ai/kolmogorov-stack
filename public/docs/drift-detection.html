Namespace-scoped drift detection.

computeDriftSignals from src/drift-detector.js walks the capture lake and receipt log for one tenant and namespace, slices them into a baseline window and a lookback window, computes three drift signals, and grades each against its in-window standard deviation. The SPC (Statistical Process Control) ladder is 2 sigma for warn, 3 sigma for alert. Self-calibrating per namespace — a steady namespace trips on smaller absolute deltas than a noisy one.

The three drift signals

SignalComputationWhat it catches
fallback_rate_deltaFraction of routed calls in the lookback window where route_decision === 'frontier_fallback', minus the same fraction in the baseline window. Receipts without route_decision are excluded so the rate is never mis-counted.Confidence drop: the artifact is increasingly unable to answer locally and the router is falling back to a frontier model.
distribution_distanceProject each capture to a 256-dim sha256-bucketed pseudo-embedding, L2-normalize, take the centroid of each window, return cosine distance.Topic drift: the kind of question the namespace is being asked has shifted away from the training corpus.
volume_ratioPer-day rate in the lookback over the per-day rate in the baseline.Traffic shift: the workload is rising or tapering enough that the steady-state assumption no longer holds.

The SPC ladder

For each signal we compute a baseline standard deviation by chunking the baseline window into K=6 equal-time slices and treating each slice's signal value as one sample. The lookback signal is then graded:

if |signal_delta| >= 3 * baseline_sigma : alert
if |signal_delta| >= 2 * baseline_sigma : warn
otherwise                               : ok

Each signal has a noise floor that prevents an artificially steady baseline from tripping on micro-deltas:

FloorValueMeaning
SIGMA_FLOOR_FALLBACK_RATE0.022 percentage points minimum noise for fallback rate.
SIGMA_FLOOR_DISTRIBUTION0.022% cosine-distance minimum noise.
SIGMA_FLOOR_VOLUME0.1010% per-day rate minimum noise.

Invocation

import { computeDriftSignals } from './src/drift-detector.js';

const r = computeDriftSignals({
  tenant_id: 'tenant_4d3d8db6c820',
  namespace: 'support',
  lookback_days: 7,    // the window being scored
  baseline_days: 30,   // the reference window
});

// {
//   ok: true, version: 'r7-v1',
//   tenant_id, namespace, lookback_days: 7, baseline_days: 30,
//   fallback_rate_delta: 0.07,        // +7pp vs baseline
//   distribution_distance: 0.13,
//   volume_ratio: 1.45,
//   status: 'warn',
//   recommendation: 'kolm distill --namespace support --priority-captures fallback_eligible --limit 200',
//   details: { lookback:{...}, baseline:{...}, per_signal_status:{...}, thresholds:{...} }
// }

Insufficient data

The detector refuses to fabricate a signal from too little data. When either constraint is unmet, the offending signal returns null and is excluded from the overall status grading:

  • MIN_BASELINE_RECEIPTS = 5 — below this, fallback_rate_delta is null.
  • MIN_BASELINE_SAMPLES = 8 — below this, distribution_distance is null.

Recommendation generator

On warn or alert the envelope carries a literal CLI command the operator runs. We never auto-execute — the operator stays in the loop. The recommendation lifts the captures that landed on frontier_fallback so the next distill round focuses on the weakness the router already surfaced:

kolm distill --namespace support --priority-captures fallback_eligible --limit 200

Details block

The details field surfaces every input the SPC ladder used. A dashboard or audit log can render "why" without re-running the math.

"details": {
  "lookback": {
    "receipts_count": 1240, "captures_count": 1240,
    "fallback_rate": 0.18, "fallback_sample_size": 1240
  },
  "baseline": {
    "receipts_count": 4980, "captures_count": 4980,
    "fallback_rate": 0.11, "fallback_sample_size": 4980,
    "fallback_mean": 0.10, "fallback_sigma": 0.025,
    "distribution_mean": 0.05, "distribution_sigma": 0.03,
    "volume_mean": 166.0, "volume_sigma": 0.18
  },
  "per_signal_status": { "fallback": "warn", "distribution": "warn", "volume": "ok" },
  "thresholds": { "sigma_warn": 2.0, "sigma_alert": 3.0, "min_baseline_samples": 8, "min_baseline_receipts": 5 }
}

The pseudo-embedding

The distribution signal is intentionally model-free. No torch, no transformers, no embedding API call. pseudoEmbed(text) tokenizes lowercase alphanumerics, hashes each token to one of 256 buckets via sha256, increments the bucket count, then L2-normalizes the vector. The bucket assignment is stable across runs so the same capture text always lands in the same bucket.

import { pseudoEmbed, centroid, cosineDistance } from './src/drift-detector.js';

const a = pseudoEmbed('How do I reset my password?');
const b = pseudoEmbed('How do I delete my account?');
cosineDistance(a, b);  // ~0.62
Caveats. The pseudo-embedding is cheap and stable, but it is a bag-of-words approximation — it cannot tell apart two prompts that share vocabulary but mean different things ("How do I delete my account?" vs "How do I delete a user's account?"). For high-stakes drift signals pair this with the W813 KL-on-real-embeddings comparator in src/drift-detect.js, which accepts pre-computed embeddings from your own pipeline.
