kolm.ai / docs / migrate

Open-source AI workbench

Frontier AI on your own infrastructure.

brings the model libraries you already manage in Ollama and LM Studio into kolm without copying a single GGUF. Discovery walks the on-disk catalogs those tools maintain, then wraps each model through the kolm import path so the not_kolm_compiled:true truthfulness lock flows transitively. Nothing leaves the box.

From Ollama

Ollama writes its models in a content-addressed layout: a JSON manifest under <root>/manifests/registry.ollama.ai/library/<name>/<tag> points at GGUF blobs in <root>/blobs/sha256-<hex>. We read the manifest, pick the layer whose mediaType is application/vnd.ollama.image.model (or fall back to the largest layer on older builds), and resolve the blob path. The blob is the same GGUF that ollama run uses; we never modify it.

PlatformDefault root
linux / macOS~/.ollama/models
windows%USERPROFILE%\.ollama\models
anyhonors OLLAMA_MODELS env
 
# Discover everything Ollama has on disk (no parse, no wrap).
kolm migrate from-ollama --dry-run

# Override the root if Ollama is somewhere non-default.
kolm migrate from-ollama --path /mnt/models/.ollama/models --dry-run

# Wrap a specific model by <name:tag>. Emits the not_kolm_compiled manifest.
kolm migrate from-ollama --name llama3.2:3b

From LM Studio

LM Studio writes models in a <publisher>/<repo>/<file>.gguf tree (sometimes with a per-variant subdirectory). We walk the configured root and pick up every *.gguf we find; the parent directory chain becomes the publisher + repo context on the wrapped manifest. The default-path candidate list covers the layouts LM Studio has shipped on each platform; --path overrides the entire list.

PlatformDefault roots
macOS~/Library/Caches/lm-studio/models
~/Library/Application Support/lm-studio/models
linux~/.cache/lm-studio/models
windows%LOCALAPPDATA%\LM Studio\models
%LOCALAPPDATA%\LMStudio\models
%USERPROFILE%\.cache\lm-studio\models
anyhonors LMSTUDIO_MODELS env
 
# Discover everything LM Studio has on disk.
kolm migrate from-lmstudio --dry-run

# Override the root.
kolm migrate from-lmstudio --path "/Volumes/Models/lm-studio/models" --dry-run

# Wrap a specific file (matched by basename or full path).
kolm migrate from-lmstudio --name llama-3.2-3b-instruct-q4_k_m

Health check

kolm migrate doctor reports which default roots exist on this host and how many models we found in each - useful for confirming the right Ollama / LM Studio install is being picked up before you start wrapping things.

kolm migrate doctor

Plain manifests, every time

A migrated model inherits the-2 truthfulness lock:

{
  "ok": true,
  "manifest_version": "w740-v1",
  "not_kolm_compiled": true,
  "k_score": null,
  "holdout": null,
  "source_tool": "ollama",
  "source_name": "llama3.2:3b",
  "source_format": "gguf",
  "source_sha256": "<64-hex>",
  "source_path": "/abs/path/blobs/sha256-...",
  "imported_at": "2026-05-24T12:34:56.789Z",
  "migrate_version": "w743-v1",
  "version": "w740-v1"
}

Migration alone never earns a K-Score. The wrap exists so the migrated model can ride through catalog / inspect / signature flows immediately - but to claim it is kolm-compiled, you have to run kolm distill on real seeds and let the artifact earn a K-Score on a holdout, exactly like a fresh distill would.

# 1. Migrate the existing Ollama model  -  wrapped not_kolm_compiled manifest.
kolm migrate from-ollama --name llama3.2:3b

# 2. Re-distill against your captured traffic (this is where the K-Score is earned).
kolm distill --base ~/.ollama/models/blobs/sha256-<hex> --seeds ./seeds.jsonl --out llama-distilled.kolm

# 3. Optional: export back to GGUF for Ollama / LM Studio / llama.cpp.
kolm export llama-distilled.kolm --backend gguf --quant q4_k_m --out ./exports

Privacy: migration is local

Discovery walks the filesystem only. Manifests, blob digests, and the path of every GGUF stay on the box; no model name, no SHA, and no file content is sent anywhere. The /v1/migrate/{discover,wrap} routes accept paths from the caller and run the same code; they exist so a self-hosted dashboard or CI runner can drive the migration without shelling out, but they still touch only the local filesystem.

API + CLI

The same engine drives the CLI and the API.

SurfaceInvocation
CLI discoverykolm migrate from-ollama [--path <p>] [--dry-run] [--limit N]
CLI discoverykolm migrate from-lmstudio [--path <p>] [--dry-run] [--limit N]
CLI wrapkolm migrate from-ollama --name <name:tag>
CLI doctorkolm migrate doctor
API discoverPOST /v1/migrate/discover body {source, path?} (auth-gated)
API wrapPOST /v1/migrate/wrap body {source, model_name, path?} (auth-gated)

Plain error envelopes

Every failure mode is a named error code with a one-line hint. No silent passthrough, no fake parses, no zero-byte manifests.
