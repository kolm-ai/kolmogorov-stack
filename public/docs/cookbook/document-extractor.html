Distill a document extractor.

Turn invoices, contracts, or clinical notes into structured JSON validated against your schema, using a model small enough to run on a Jetson-class edge device. The recipe pairs schema-constrained decoding with distillation so the extractor not only produces valid JSON every time, but also matches the field-level decisions your reviewers would have made on the same document.

What you will build

A 32B INT4 extractor that reads a document, returns JSON conforming to your schema, and ships an attached confidence score per field. The example artifact runs at INT4 on a Jetson Orin or a small GPU, with the schema bound into the deployment so the runtime refuses to emit malformed output.

The example use case is a finance team extracting vendor, line items, totals, and payment terms from PDF invoices. In a representative run the team saw schema-valid output on roughly 98% of held-out invoices and field-level agreement with reviewers on ~93%. Your numbers will differ — these are example values from one representative run.

What you need

  • kolm CLI — npm i -g @kolmogorov/cli or the Python SDK install path.
  • API key — kolm login or sign up at /signup.
  • Schema — a JSON Schema describing the target structure (invoice schema, contract clause schema, clinical-note schema, etc.). The compiler binds this into the artifact so the runtime can constrain decoding.
  • Capture source — roughly 500 to 3000 documents already extracted by the teacher (frontier vision-or-text extractor) plus the reviewer's gold extractions for at least 200 of them.
  • GPU for compile — an RTX 4090 or better. Compile takes around two hours for a 2000-document corpus.
  • Optional: OCR — for image-only PDFs, pair with the kolm OCR worker (Tesseract or PaddleOCR backends).

Step-by-step

Step 1

Capture teacher extractions against the schema

Run the teacher over your documents and capture the schema-validated outputs into a namespace.

kolm capture log \
  --namespace doc-extract \
  --schema ./schemas/invoice.schema.json \
  --import ./invoices/*.pdf \
  --ocr paddleocr

The schema is attached to every capture so the bake-off and compile steps know what valid output looks like.

Step 2

Bake-off candidate students

Score Qwen2.5-3B-Instruct, Phi-3.5-mini, and Gemma-2-2B against a 200-document holdout. The judge is the reviewer's gold extractions.

kolm bake-off \
  --namespace doc-extract \
  --candidates qwen2.5-3b-instruct,phi-3.5-mini,gemma-2-2b \
  --holdout 200 \
  --judge gold \
  --metric schema-valid,field-accuracy,field-coverage

Field-accuracy is the key metric. Schema-valid catches malformed output. Field-coverage flags students that silently drop optional fields.

Step 3

Compile with schema-constrained decoding

Distill and bind the schema into the artifact. The runtime will refuse to emit invalid JSON.

kolm compile \
  --namespace doc-extract \
  --target qwen2.5-3b-instruct-int4 \
  --schema ./schemas/invoice.schema.json \
  --constrain-decode \
  --epochs 5 \
  --output ./artifacts/doc-extract-v1.kolm

The 5-epoch budget biases toward field accuracy at the cost of some general capability — the right tradeoff for an extractor.

Step 4

Verify

Print the signed manifest. Confirm schema-valid rate, field-accuracy, and the bound schema hash.

kolm verify ./artifacts/doc-extract-v1.kolm --pretty

We ship at schema-valid ≥ 0.98 and field-accuracy ≥ 0.90. Iterate the corpus if either gate fails — usually the fix is more captures of the rare layouts.

Step 5

Deploy to the edge or a small GPU

Ship to TensorRT-LLM on a Jetson Orin, or to vLLM on a small cloud GPU. The runtime enforces the bound schema.

kolm deploy ./artifacts/doc-extract-v1.kolm \
  --runtime trt-llm \
  --device jetson-orin \
  --endpoint http://edge-extractor.local:8000

Run for two weeks behind a human-review queue: every extraction draft goes to a reviewer, every correction lands back in the lake as a fresh capture, and the next compile closes the loop.

Real numbers (example run)

From a representative 2000-invoice compile on a single RTX 5090. These are example values from one finance team's run, not a generic benchmark.

  • Compile time — example: ~2 hours for 2000 invoices, 5 epochs.
  • K-Score — example: 0.91 against reviewer gold on a 200-invoice holdout.
  • Schema-valid rate — example: 98% on holdout, 100% with constrained decode enabled at runtime.
  • Field-level agreement — example: 93% across vendor, line items, totals, and payment terms.
  • Footprint — example: 1.9 GB VRAM at INT4 on Jetson Orin.
  • p95 latency — example: ~640ms per invoice on Jetson Orin including OCR.

All numbers above are example values from one representative run. Treat them as a worked example, not a guarantee.

Common pitfalls

  • Capture diversity skewed toward the common layout. Eight vendors often produce 80% of the volume. The student over-fits to those layouts and the long-tail vendors degrade. Force diversity with kolm capture filter --max-per-vendor 50.
  • K-Score above your schema-valid gate. A high K-Score on captured outputs can mask a malformed-JSON regression because the score is computed on parsed fields. Always read schema-valid and K-Score independently.
  • Schema drift after a teacher upgrade. When the teacher gains a new field (e.g., "vat_rate") and starts emitting it, your student silently omits it. Bind the schema version into the artifact (the manifest already does this) and re-compile when the teacher's schema bumps.
