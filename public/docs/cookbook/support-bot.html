Distill a support bot.

Replay your closed-ticket corpus into a small replier that sounds like your team, never leaks customer data offsite, and answers in roughly the time your fastest agent takes to read the ticket. The recipe trades the cost and risk of sending tickets to a hosted frontier model for a 32B INT4 model you own end-to-end. Every step uses the standard kolm CLI.

What you will build

A 32B INT4 support replier compiled from your closed tickets and your team's review labels. The model handles tier-1 replies, classifies escalation paths, and drafts in your house voice. The example deployment is a single GPU box behind your support tool's webhook, returning a draft reply within the average agent read-time so the human reviews instead of writes.

In a representative team's run we have seen roughly 50x lower per-call cost than the frontier API the team was previously using, p95 latency around 180ms on an RTX 4090, and the privacy property that customer ticket text never leaves the team's network. Your numbers will differ — these are example values from one representative run.

What you need

  • kolm CLI — npm i -g @kolmogorov/cli or the Python SDK install path from the Python quickstart.
  • API key — kolm login or sign up at /signup.
  • Capture source — an existing closed-ticket dataset (CSV, JSONL, or pulled live from Zendesk/Front/Help Scout via the kolm connector). You need roughly 2000 to 10000 ticket/reply pairs with at least a thumbs-up signal.
  • GPU for compile — an RTX 4090 or better is enough for a 32B INT4 student. Compile takes around 90 minutes for a 5000-ticket corpus on a 5090.
  • PII redaction policy — the recipe redacts emails, phone numbers, order IDs, and free-form addresses before any capture lands in the lake. Defaults are HIPAA-grade; customize via kolm capture redact --policy ./policy.json.

Step-by-step

Step 1

Import closed tickets

Load existing tickets into the lake. The CLI accepts CSV and JSONL, or pulls directly from a support tool via the connector.

kolm capture log \
  --namespace support-bot \
  --import ./closed-tickets-2025.jsonl \
  --redact emails,phones,order-ids,addresses \
  --label-field csat

The --label-field csat flag tells the bake-off to treat customer satisfaction score as the thumbs-up signal during eval.

Step 2

Bake-off candidate students

Score Qwen2.5-3B-Instruct, Llama-3.2-3B, and Phi-3.5-mini against a 500-ticket holdout. The judge is the original teacher (the frontier model the team previously used).

kolm bake-off \
  --namespace support-bot \
  --candidates qwen2.5-3b-instruct,llama-3.2-3b,phi-3.5-mini \
  --holdout 500 \
  --judge teacher \
  --metric reply-similarity,intent-accuracy,tone-match

The leaderboard surfaces tone-match per candidate — the critical metric for support voice.

Step 3

Compile to INT4

Distill onto the winning student and quantize in one pass. The output is a signed .kolm artifact with the redaction manifest attached.

kolm compile \
  --namespace support-bot \
  --target qwen2.5-3b-instruct-int4 \
  --epochs 4 \
  --max-seq-len 4096 \
  --output ./artifacts/support-bot-v1.kolm

Expect ~90 minutes on an RTX 5090 for a 5000-ticket corpus. The 4-epoch budget biases toward tone match over generality.

Step 4

Verify

Inspect the signed manifest. Confirm the redaction policy applied, the K-Score against the holdout, and the attestation block.

kolm verify ./artifacts/support-bot-v1.kolm --pretty

For a tier-1 support deployment we ship at K-Score ≥ 0.80 and tone-match ≥ 0.85. Iterate the corpus if either gate fails.

Step 5

Deploy behind your support webhook

Ship to vLLM or SGLang. Point your support tool's webhook at the deployed endpoint. The artifact's redaction manifest travels with it — the runtime refuses to start if the deployment target doesn't preserve the same redaction guarantees.

kolm deploy ./artifacts/support-bot-v1.kolm \
  --runtime sglang \
  --endpoint https://support-llm.internal:8000 \
  --guard pii-strict

Roll out in shadow mode for a week — the bot drafts every reply, the human ships it, and you log the diff between the bot draft and the final human reply as new captures.

Real numbers (example run)

From a representative 5000-ticket compile on a single RTX 5090. These are example values from one team's run, not a generic benchmark.

  • Compile time — example: ~90 minutes for 5000 tickets, 4 epochs, max-seq-len 4096.
  • K-Score — example: 0.83 against the teacher on a 500-ticket holdout with CSAT as the label.
  • Tone match — example: 0.87 against the team's house voice rubric.
  • Cost reduction — example: ~50x lower per-call cost than the previously used frontier API on the same workload.
  • p95 latency — example: ~180ms first token on an RTX 4090.
  • Privacy — ticket text never leaves the team's network. Redaction manifest attached to the artifact for SOC 2 evidence.

All numbers above are example values from one representative run. Treat them as a worked example, not a guarantee.

Common pitfalls

  • Capture diversity skewed toward easy tickets. Closed-ticket corpora are biased toward fast resolutions. Boost diversity by sampling at least 30% from tickets that escalated past tier 1, even if they were eventually closed. Use kolm capture filter --min-tier 2 on a sub-corpus.
  • K-Score above your tone-match gate. A high K-Score can hide a tone regression because reply similarity averages out occasional off-voice replies. Always read tone-match independently; never let one metric stand in for the other.
  • Teacher drift mid-quarter. Frontier replier models update every few weeks. Snapshot the teacher hash at compile time (it's already in the manifest) and re-compile when the rolling K-Score on fresh tickets drops more than 5 points.
