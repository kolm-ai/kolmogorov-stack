Multilingual K-Score and per-language thresholds.

A composite K-Score can hide a 0.42 in Spanish behind a 0.91 in English. The pooled number is plain about the average, but it's a lie about per-language quality. partitions captures by detected language and reports a K-Score per partition, with Wilson 95% CI gated at n≥30 per language. The fallback router then uses per-language confidence thresholds — weaker languages get a lower threshold so more requests route to the teacher. Stamped w760-v1. Built by src/lang-detect.js + src/kscore-per-language.js + src/multilingual-augment.js.

Why per-language K-Score matters

A distilled student that scored 0.91 on a corpus that's 95% English and 5% Spanish is reporting two different students under one number. The English K-Score is genuinely 0.91; the Spanish K-Score is almost certainly far lower — the model never saw enough Spanish to learn it. The pooled composite hides that gap because the small Spanish denominator can't move the average.

This shows up in production when a Spanish-speaking user gets a confidently-bad answer. The composite gate (K≥0.85) passed; the language-specific gate would have caught it. surfaces the per-language breakdown so you can:

  • See which languages are under-served BEFORE you ship
  • Generate synthetic multilingual rows from teacher (-2)
  • Lower the confidence threshold per-language so weaker languages route to the teacher more aggressively (-3)

How language is detected

Pure JS, no model file, no Python, no native deps. detectLang(text) runs two stages and picks the higher-confidence answer:

  1. Script-class scoring. Count code points that hit each Unicode block: CJK ideographs for zh, hiragana+katakana for ja, Hangul for ko, Cyrillic for ru, Arabic block for ar, Devanagari for hi, Thai block for th. If ≥20% of non-whitespace chars hit a target block, that's a high-confidence signal — a language genuinely written in that script. Japanese vs Chinese: any kana present ⇒ lean Japanese (kana doesn't exist in Chinese).
  2. Stopword frequency. For Latin-script text we count whole-word hits against tiny per-language stopword tables (5-8 diagnostic words each: "the/and" ⇒ en, "el/la/que" ⇒ es, "le/la/et" ⇒ fr, "der/die/und" ⇒ de). Two or more hits in one language wins; otherwise the detector returns fallback:true.

Trade-off: we sacrifice absolute accuracy for zero dependencies and sub-millisecond latency. The detector is for routing decisions and capture analytics, not for forensic translation work. When confidence is below min_confidence (default 0.4), we return {lang:null, fallback:true} — never silently pick a wrong language.

Per-language confidence thresholds for fallback

The runtime router currently uses one global confidence threshold (0.7 by default) — below that, the request routes to the teacher instead of the local student.-3 scales this threshold per-language so weaker languages fall back more aggressively:

The clamp ensures a single very-weak language can't drop the threshold to zero (which would route everything to teacher and rack up cost), and a very-strong language can't push it to 1.0 (which would always-fallback). The router reads this per-call so the routing decision sees the language of THIS request, not a stale snapshot.

Synthetic multilingual augmentation

If your corpus is 95% English, you can't fix that by re-training harder — you need more non-English rows.-2 fans out source rows through a translation backend (dependency-injected, same lineage as the synthetic capture loop) so the student sees parallel examples in every target language:

CODE25

Without confirm:true, the route returns a dry run plan — row count, target count, estimated cost — without calling the translator. With confirm:true AND a wired translator (req.app.locals._w760_translator_caller in tests, KOLM_TRANSLATOR_CMD in production), the augmenter fans out and returns the augmented rows.

Every augmented row carries source_type:'synthetic' and synthetic_kind:'translation' packed into the feedback JSON blob. The source_type enum is the same canonical value — we don't invent a new "translated" enum so downstream consumers (datasets, bakeoffs, sampler weights) can filter synthetic-vs-real with one rule.

No-silent-pass contract

Wilson 95% CI gated at n≥30 per language. Below 30 captures, the per-language K-Score is null — not estimated, not interpolated, not silently substituted with the pooled value. The insufficient_per_lang_samples envelope tells you which languages are under-sampled and points at-2 synthetic augmentation as the fix.

Mirrors (per-category diagnose) and (failure-mode clustering) — same n≥30 floor everywhere we report a confidence band.

Next steps

Start with kolm lang kscore --namespace <ns> to see your per-language K-Score breakdown. If you see insufficient_per_lang_samples, the report tells you exactly which languages need more captures. Run kolm lang augment --namespace <ns> --target-langs es,fr,de to preview a synthetic augmentation plan; re-run with --confirm to actually translate. Re-run kolm lang kscore afterward to verify the per-language n's crossed 30.

Cross-lingual distillation, English teacher, multilingual student.

A language-balanced sampler picks captures by detected lang under one of four strategies. Per-language eval gates every score at Wilson n≥30 — never a number-shaped lie.

Balance strategies

Four frozen strategies determine how the sampler re-weights captures across target languages. Pick by intent — uniform if downstream eval weights every language equally; traffic_weighted if you want the sample to mirror production language mix; sqrt_inverse for the canonical NMT balance; log_inverse for an even softer rare-language tilt.

StrategyWeight ruleBest for
uniform1.0 per target langPer-language K-Score parity
sqrt_inverse1 / √trafficNMT canonical balance (Arivazhagan et al, 2019)
log_inverse1 / log(1+traffic)Soft tail-language tilt
traffic_weightedtraffic countProduction-mirror sampling

Per-language eval (Wilson 95% CI gated at n≥30)

Every per-language score is gated at the same Wilson n≥30 floor as 's per-language K-Score. Below the floor the language is moved into languages_skipped_below_n30:[] — never reported with a CI. compareLanguageDelta() diffs two eval outputs and flags significant per-language deltas; the heuristic is conservative (delta wider than either CI band).

Multilingual bake-off

Head-to-head compare two artifacts on multilingual captures. runXlangBakeoff() partitions captures by detected language, judges both outputs per row, then emits a per-language winner plus a multilingual_consistency_score (0..1) reflecting how uniformly the overall winner wins across languages. Requires ≥2 distinct detected languages — otherwise returns no_multilingual_captures (loud-fail envelope; use /v1/bakeoff for single-language head-to-head).
