Studio

Studio overview.

Studio is the no-code browser studio for the kolm compiler. The CLI, the TUI, and Studio are three faces over the same backend: the same compile graph, the same recipes, the same signed artifact at the end. Studio exists for people who want a visual wizard for the parts of the pipeline that are easier to see than to type, and a hub for the rest.

What lives in Studio

SurfaceWhat it doesCLI equivalent
distillPick a capture set, a teacher council, and a recipe. Watch the loss curve, k-score gate, and per-stage receipt.kolm distill
compileBind weights to a spec, run evals, write the signed manifest, anchor the evidence DAG.kolm compile
exportQuantize and project: GGUF, AWQ, EXL2, MLX, NVFP4, FP8.kolm export
publishPush to Hugging Face, a private model registry, or both. Writes a publish receipt.kolm publish
recipesBrowse the recipe catalog; pick one for a wizard or fork into spec.toml.kolm recipes list
jobsOne pane of all running and recent jobs across the four surfaces above.kolm jobs

CLI / TUI / Studio parity

Every job started in Studio writes the same row to the jobs table that the CLI writes. The job receipt is identical. A job started in Studio can be inspected from the CLI (kolm job inspect <id>) and a job started from the CLI shows up live in /studio/jobs. There is no "Studio-only state".

When to use what

  • Use Studio when you want a guided wizard, when you are exploring recipes for the first time, or when a non-engineer needs to inspect a job.
  • Use the CLI for scripting, CI, batch runs, and any job that wants to be deterministic across machines.
  • Use the TUI when you are on a headless host and want a live dashboard without a browser.
