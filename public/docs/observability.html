Observability and metrics.

The kolm runtime exports OpenTelemetry spans for every routed request and a Prometheus endpoint for time-series metrics. Drop them into Grafana, Datadog, Honeycomb, or any OTLP-compatible backend. The dashboards and alerts on this page are the reference set we ship; copy them, adapt them, or replace them with your own.

OpenTelemetry span attributes

Every routed request emits one span named kolm.runtime.route with the attributes below. Tools and capture writes nest spans under the root with their own attribute set.

AttributeDescription
artifact_idThe .kolm artifact ID that served the response. Empty if the request was routed to the teacher path (fallback).
routing_decisionOne of local, teacher, spliced. Indicates which path the runtime chose for this request.
token_confidence_p50Median per-token confidence over the response sequence. Range [0, 1]; lower values indicate the local model was less sure.
token_confidence_p9595th percentile per-token confidence. Used by drift gates to detect tail-quality degradation.
kscore_driftThe K-Score delta between the artifact baseline and the rolling window for this namespace. Positive means quality improved; negative means quality drifted down.
namespace_idThe tenant namespace that issued the request. Used for per-namespace dashboards.
splice_latency_msFor routing_decision=spliced requests, the latency of the local-to-teacher splice in milliseconds.
fallback_reasonFor routing_decision=teacher requests, one of confidence_low, schema_violation, token_budget, operator_force.

Grafana dashboard template

The reference Grafana dashboard ships in the kolm repository at tools/grafana/kolm-dashboard.json. The file lives in the repo and is not served as a URL; download it from your local checkout and import via Grafana's dashboard import flow.

The dashboard includes four panels: K-Score over time (timeseries), local-vs-teacher token ratio (gauge), fallback rate (stat), p95 splice latency (timeseries). Each panel uses the Prometheus queries documented in the dashboard JSON; replace the data source UID with your own before saving.

The dashboard is a starting point, not a final product. Customize the panel queries for your namespace labels, your alert thresholds, and the dimensions your team cares about. The shipped version assumes a single namespace; multi-tenant deployments SHOULD add a namespace template variable.

Alert templates

Three alert templates we recommend for any kolm production deployment. Each carries a Prometheus rule example you can drop into your Alertmanager config.

K-Score drift

Fires when the rolling K-Score for a namespace drifts more than 0.05 below the artifact baseline for 30 minutes. Drift this large indicates the response distribution has shifted enough that the artifact is no longer well-calibrated; the operator SHOULD investigate or trigger a re-distill.

- alert: KolmKScoreDrift
  expr: kolm_kscore_drift < -0.05
  for: 30m
  labels:
    severity: warning
  annotations:
    summary: "K-Score drift below threshold for namespace {{ $labels.namespace }}"
    runbook: "https://kolm.ai/docs/runbooks/kscore-drift"

Fallback spike

Fires when the fallback rate (requests routed to teacher because the local model failed a gate) exceeds 20% for 10 minutes. A spike usually means the response distribution has changed faster than drift detection caught; immediate investigation is warranted.

- alert: KolmFallbackSpike
  expr: rate(kolm_fallback_total[5m]) / rate(kolm_requests_total[5m]) > 0.20
  for: 10m
  labels:
    severity: critical
  annotations:
    summary: "Fallback rate above 20% for namespace {{ $labels.namespace }}"
    runbook: "https://kolm.ai/docs/runbooks/fallback-spike"

Latency regression

Fires when p95 splice latency exceeds 250ms for 15 minutes. Splice latency is the local-to-teacher handoff time; a regression here usually means network conditions changed or the teacher backend is degraded.

- alert: KolmLatencyRegression
  expr: kolm_splice_latency_p95_ms > 250
  for: 15m
  labels:
    severity: warning
  annotations:
    summary: "p95 splice latency above 250ms for namespace {{ $labels.namespace }}"
    runbook: "https://kolm.ai/docs/runbooks/latency-regression"

Prometheus endpoint

The kolm runtime exposes a Prometheus-compatible metrics endpoint at /metrics. The endpoint is reachable from any process that can reach the runtime; configure your Prometheus scrape job to point at the runtime address.

Exported metric families:

  • kolm_kscore_point - current K-Score point estimate per namespace.
  • kolm_kscore_drift - delta from the artifact baseline per namespace.
  • kolm_local_token_ratio - fraction of tokens served by the local model per namespace.
  • kolm_fallback_rate - rate of fallback-to-teacher routing per namespace.
  • kolm_splice_latency_p95_ms - 95th percentile splice latency in milliseconds per namespace.
  • kolm_requests_total - counter of requests per namespace + routing decision.
  • kolm_fallback_total - counter of fallback events per namespace + fallback reason.

All metrics carry a namespace label. The runtime never exports raw prompt or completion text in metrics; if you need that level of detail, use the OpenTelemetry trace export instead.

Datadog / Honeycomb compatibility

The kolm runtime emits OTLP-format traces and OpenMetrics-format counters. Any backend that accepts those formats can consume kolm's observability stream without translation.

For Datadog: set OTEL_EXPORTER_OTLP_ENDPOINT to the Datadog OTLP receiver URL (typically http://localhost:4318 with the Datadog agent running locally). The agent forwards traces and metrics to Datadog with the standard provider config.

For Honeycomb: set OTEL_EXPORTER_OTLP_ENDPOINT to https://api.honeycomb.io and add the Honeycomb API key header via OTEL_EXPORTER_OTLP_HEADERS=x-honeycomb-team=<key>. Traces and metrics arrive in the dataset you configure.

For New Relic, Lightstep, Splunk Observability, and any other OTLP-compatible backend, follow the same pattern: point the OTLP endpoint at the receiver, set the authentication headers, and the runtime exports without further configuration.
