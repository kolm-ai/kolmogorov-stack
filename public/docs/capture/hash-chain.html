Capture hash chain

Tamper-evident by construction.

Every capture row carries an HMAC-SHA256 hash that links it to the previous row. Any single-byte mutation anywhere in history breaks the chain; kolm captures verify-chain finds the break in O(n).

The construction

Each row stores two hashes: prev_hash (the previous row’s row_hash, or the genesis seed for the first row) and row_hash (HMAC-SHA256 over the canonical serialization of this row, keyed by the tenant signing key).

# Genesis seed (per tenant, set at lake init)
genesis = HMAC-SHA256(tenant_key, "kolm-capture-genesis-v1" || tenant_id)

# Per-row computation
row_bytes = canonical_json(row_minus_hash_fields)
row_hash  = HMAC-SHA256(tenant_key, prev_hash || row_bytes)

# Storage
row.prev_hash = previous_row.row_hash   # or genesis for the first row
row.row_hash  = row_hash

Why a keyed chain, not plain SHA

Plain SHA-256 is content-addressed: anyone with the row bytes can recompute the hash. HMAC-SHA256 binds the hash to the tenant key — an attacker who tampers with rows and recomputes plain hashes cannot do so for HMAC without the key. The hash chain is therefore both tamper-evident and tamper-resistant against an attacker who can write but cannot read the key file.

The same key is used for the receipt Ed25519 signature in a different role; see signing. Compromise of one compromises the other.

Canonical serialization

The row hash must be reproducible from the stored row. We canonicalize before HMAC: JSON keys sorted lexicographically, integers as NumberLong (no leading zeros), floats as RFC 8259 grisu3, no whitespace, UTF-8 NFC. The row_hash and prev_hash fields themselves are excluded from the canonical form so the hash describes the rest of the row.

# Canonical form excerpt
{"capture_id":"01HXYZ...","cost_usd":0.000041,"model":"gpt-4o-mini","namespace_id":"customer-support","prompt":{...},"provider":"openai","receipt_cid":"bafy2bzace...","response":{...},"route_decision":"teacher","state":"approved","tenant_id":"tenant_xxx","timestamp":"2026-05-20T14:32:01Z","tokens_in":48,"tokens_out":120}

Verifying the chain

The verify-chain verb walks the lake in capture_id order, re-derives row_hash for each row, and compares against the stored value. The first mismatch is the tamper point; the verb reports it and exits non-zero.

# Verify the full chain for a tenant
kolm captures verify-chain
# Walking 1,248,301 rows...
# OK: 1,248,301 rows hashed and linked.
# Genesis: a3f4...91c2
# Tip:     8e2b...f17a
# Elapsed: 14.2s

# Verify a slice (last 7 days)
kolm captures verify-chain --since 7d
# OK: 18,427 rows.

# Fail mode (simulated tampering)
kolm captures verify-chain --since 1d
# FAIL at row cap_01HXYZ... (#18019)
#   stored row_hash:  4f2a...09c1
#   derived row_hash: 88b1...e342
#   prev_hash chain:  4f2a... (matches previous row.row_hash)
#   diagnosis: row body was modified after capture; chain breaks here.
# Exit code: 2

Where the chain is verified

  • Before export — kolm captures export --verify walks the slice first and aborts on mismatch.
  • Before distill — kolm distill --verify-chain refuses to use a slice that fails verification.
  • Scheduled audit — the self-hosted gateway can be set to verify nightly and emit a telemetry event on failure.
  • External auditor — the verb runs against a read-only replica with the tenant public key (verifier mode skips HMAC, validates by re-deriving over the canonical form).

Genesis seed and key rotation

The genesis seed is fixed for the lifetime of the lake; it does not rotate with the signing key. When the signing key rotates (90-day cadence; see signing), new rows use the new key for HMAC and the chain stores a key_id on each row. verify-chain uses the correct key per row.

Caveats. Hash chains protect against undetected mutation, not against deletion of the tail. An attacker who can truncate the file can roll back recent history; the verifier sees a shorter but internally consistent chain. Pair with off-site capture-id watermarks (e.g. nightly snapshot of tip_capture_id to S3 or to the public anchor at verify.kolm.ai) so truncation is also detectable.
