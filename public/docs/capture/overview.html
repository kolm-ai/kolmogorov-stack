Capture lake

Every call, captured. Every row, chained.

The capture lake is an append-only ledger of every request that walks the wrapper pipeline. Same schema across four storage backends, keyed hash chain between rows, indexed by tenant + namespace + timestamp.

What the lake is

The capture lake is the storage layer behind pipeline stage 9. Every request that completes the pipeline produces one capture row containing the redacted prompt, the response, the routing decision, the cost, the receipt CID, and a hash chain link to the previous row.

The lake is the seed dataset for distillation, the truth set for bake-offs, the audit record for compliance reviews, and the input to the active-learning loop. One write path; many readers.

The four storage backends

All four backends share the same row schema and the same hash-chain construction. Pick the backend that matches the volume and the operational model you already run.

BackendBest forThroughputCompaction
jsonlDev, single-host, <100 req/sHigh (append-only file)Manual rotate
sqliteSingle-node prod, <500 req/sHighVACUUM nightly
postgresMulti-node, queryableMediumTimescaleDB recommended
s3Long-term retention, audit exportStreamingS3 lifecycle to Glacier

Why hash-chained

Each row carries the HMAC-SHA256 of (previous_hash, row_bytes) using the tenant signing key. A single mutated byte anywhere in history breaks the chain and is detected by kolm captures verify-chain in O(n). The chain is independent of the storage backend; you can dump JSONL, reload into Postgres, re-verify, and get the same result.

See hash-chain for the construction details and the verifier walkthrough.

Row schema (summary)

Each capture row is a single JSON document. The schema is stable; new fields are additive. The receipt CID links the row to the signed audit receipt (see receipt format).

Three downstream consumers

  • Distillation — kolm distill reads approved captures as the teacher-labeled seed set.
  • Bake-off — kolm bakeoff replays captures across providers to score alternatives on real traffic.
  • Audit — kolm receipts verify walks the chain and re-derives every row hash.

Why append-only

Rows are never updated in place. Approval transitions (pending → approved / rejected / quarantined) write a new state row that supersedes the old one for the same capture_id; the original row stays in the chain. The same is true for retention deletion — the chain records the tombstone, not the absence.

Caveats. The hash chain protects integrity, not confidentiality. The lake stores prompts and responses in cleartext by default. If your workload contains regulated data, enable redaction mode redact_captures or redact_all (see redaction) so the cleartext never reaches the lake.
