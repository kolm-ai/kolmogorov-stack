kolm.ai / docs / synthetic

Open-source AI workbench

Frontier AI on your own infrastructure.

Synthetic augmentation closes the loop between coverage analysis and training-set hygiene. Look at the distribution of your captures, find the under-served buckets ("200 refund queries but zero escalation"), and ask the teacher to generate the gap-filler — with every synthetic row frankly tagged kolm_synthetic:true and tied back to its parent seed CIDs. Stamped w749-v1. Built by src/synthetic-augment.js.

No-silent-pass contract

Synthetic captures are always tagged kolm_synthetic:true. The flag is written at the event-store boundary by POST /v1/synthetic/commit and survives every downstream consumer (datasets, bakeoffs, sampler weights). Every row also carries parent_seed_cids — the capture CIDs the teacher saw as in-context examples — so downstream audits can answer "which real captures did this synthetic row descend from?" without guessing. We never silently mix synthetic with real.

Why this matters: a distilled student that scored 0.94 K-Score on a mix that was 30% synthetic and 70% real may be over-fitting the teacher's style on the synthetic rows. The plain tag lets the bakeoff stratify by kolm_synthetic and report two K-Scores — one on real-only, one on synthetic-only — so you can spot that gap before promotion.

Coverage report + Gini coefficient

Coverage analysis buckets your captures and reports a count + percentage per bucket. The default bucket strategy is category, using the same inferred categorizer as diagnose (multi-turn, media kind, tool-use, namespace, or general). The keyword strategy uses keyword clustering for finer-grained topical buckets.

The Gini coefficient summarises distribution inequality into one number:

GiniMeaningAction
0.00Perfectly uniform — every bucket has the same countNo coverage debt; ship.
0.20 – 0.40Slight skew — common in healthy production trafficOptional rare-bucket up-weighting in sampler.
0.60 – 0.80Heavy concentration — one or two buckets dominateRecommend synthetic gap-fill on the rare end.
1.00One bucket holds everythingThe student has never seen anything else — synthesise.
 
GET /v1/synthetic/coverage/billing?bucket_strategy=category
{
  "ok": true,
  "namespace": "billing",
  "total": 247,
  "buckets": [
    {"name": "refund", "count": 200, "pct": 0.81, "rarity_score": 0.04},
    {"name": "general", "count": 45, "pct": 0.18, "rarity_score": 0.32},
    {"name": "escalation", "count": 2, "pct": 0.008, "rarity_score": 0.87}
  ],
  "rare_buckets": [
    {"name": "escalation", "count": 2, "rarity_score": 0.87},
    ...
  ],
  "gini_coefficient": 0.661,
  "bucket_strategy": "category",
  "version": "w749-v1"
}

Rarity score + up-weighting math

Each bucket carries a rarity_score in roughly [0, 1]:

rarity_score(bucket) = -log(pct + 1e-6) / log(total)

# pct ∈ [0, 1]    -  fraction of captures in this bucket
# total           -  total capture count
# epsilon (1e-6)  -  guards log(0) for zero-count buckets when target_categories is used

The training sampler can then up-weight rare buckets so the model sees them proportionally more often than their raw frequency would dictate:

importance_weight(capture) = 1.0 + rarity_score(capture.bucket) * 2

# Common bucket  (rarity ~= 0): weight ~= 1.0  (no boost)
# Rare bucket    (rarity ~= 1): weight ~= 3.0  (3x boost)
# Capped at 5.0  to prevent gradient explosion on degenerate corpora.

This is a sampler weight, not a loss weight — rare captures appear in the training batches more often, but each example contributes the same gradient. (Loss-weighting can be added by multiplying the gradient by the same factor; we keep the two knobs separate so practitioners can tune them independently.)

Gap detection + the "Generate 50?" UI

Gap detection answers a focused question: "which categories have fewer than N captures?" The default N is 50 — below this threshold a category is considered under-served and surfaced for synthetic top-up. You can pass either a list of target_categories (forces missing categories to appear with count=0) or omit it (only shows categories the corpus already has, under-served).

GET /v1/synthetic/gaps/billing?target_categories=refund,escalation,billing,general
{
  "ok": true,
  "namespace": "billing",
  "total_captures": 247,
  "gaps": [
    {"category": "escalation", "current_count": 2,   "suggested_count": 50, "gap": 48, "present": true},
    {"category": "billing",    "current_count": 0,   "suggested_count": 50, "gap": 50, "present": false},
    {"category": "general",    "current_count": 45,  "suggested_count": 50, "gap": 5,  "present": true}
  ],
  "version": "w749-v1"
}

The dashboard at /account/synthetic renders one card per gap with a "Generate 50?" button that opens the confirm modal — matching the spec verbatim ("I see you have 200 refund queries but zero escalation — generate 50 escalation examples from teacher?").

Spend warning — the confirm gate

Synthetic generation costs money. Each row calls the teacher (currently Anthropic Claude when wired) and bills against your hosted-inference budget. POST /v1/synthetic/generate deliberately refuses to incur charges without confirm:true in the body. Without confirm, the route returns 200 OK with { ok: false, error: "synthetic_costs_money", estimated_cost_usd } — a preview, not a charge. Re-submit with confirm: true to actually run the batch.
# Preview (no charge):
POST /v1/synthetic/generate
{
  "namespace": "billing",
  "category": "escalation",
  "target_count": 50
}
→ 200 OK
{
  "ok": false,
  "error": "synthetic_costs_money",
  "hint": "synthetic generation calls the teacher and incurs cost; re-send with {confirm:true} to proceed",
  "namespace": "billing",
  "category": "escalation",
  "target_count": 50,
  "estimated_cost_usd": 0.10,
  "cost_per_row_usd": 0.002,
  "version": "w749-v1"
}

# Actually run (charged):
POST /v1/synthetic/generate
{
  "namespace": "billing",
  "category": "escalation",
  "target_count": 50,
  "confirm": true
}
→ 200 OK
{
  "ok": true,
  "namespace": "billing",
  "category": "escalation",
  "generation_id": "a3b9f7c1e2d4a8e0",
  "generated": [ /* 50 rows, each kolm_synthetic:true */ ],
  "actual_count": 50,
  "cost_usd_est": 0.10,
  "version": "w749-v1"
}

The CLI mirrors this protocol verbatim. Without --confirm, the dispatcher prints a yellow ⚠ warning to stderr AND sends the request without the confirm flag, so the user sees the cost estimate in the envelope before they commit. With --confirm, the request fires for real.

Commit — staging to event-store

Generation does NOT immediately write to the event-store. A successful generate returns a generation_id and stages the batch in-process. The caller must explicitly POST /v1/synthetic/commit with the same id to persist:

POST /v1/synthetic/commit
{
  "namespace": "billing",
  "generation_id": "a3b9f7c1e2d4a8e0"
}
→ 200 OK
{
  "ok": true,
  "namespace": "billing",
  "generation_id": "a3b9f7c1e2d4a8e0",
  "persisted_count": 50,
  "persisted_event_ids": [ "ev_...", ... ],
  "errors": [],
  "version": "w749-v1"
}

The staging Map is in-process only by design. If the server restarts before commit, the generated rows are dropped — no orphaned synthetic data lands silently in your corpus. Generate + commit are expected to happen in the same session.

On commit, each row is written via appendEvent with provider:"kolm-synthetic", model:"kolm-synthetic-w749", kolm_synthetic:true, parent_seed_cids:[...], source_category, and generation_id. The event-store carries these fields verbatim through the canonical event row, so any downstream consumer that reads via listEvents sees the synthetic provenance without modification.

Next steps

Start with kolm synthetic coverage <ns> to see the Gini and rare-bucket table. If Gini is above 0.6, run kolm synthetic gaps <ns> to enumerate the under-served categories, then preview a batch with kolm synthetic generate <ns> --category <c> --count N. Re-run with --confirm to actually call the teacher, then kolm synthetic commit <ns> <generation_id> to persist. Re-run coverage to verify the Gini moved.

Diagnose →Per-category K-Score breakdown that points you at the buckets to fill. Failure modes →Keyword clustering that powers the bucket_strategy=keyword option. In-app synthetic →Coverage card + Gini bar + "Generate 50?" gap cards.
