Run & Govern / Deploy

Standalone vLLM config.

generateVllmConfig from src/deploy-generators.js emits a JSON file for vllm serve --config-file <path>. The fields match the contract vLLM 0.6.x and later read. The function round-trip parses its own output before returning — a malformed key or value never leaves the module. Use this when you do not want a container around the runtime: bare-metal, systemd unit, single-machine experiment.

Invocation

import { generateVllmConfig } from './src/deploy-generators.js';

const config = generateVllmConfig({
  artifact: '/models/qwen3-8b-support-q4km',
  gpu_memory_utilization: 0.9,
  tensor_parallel_size: 1,
  quantization: null,        // 'awq' | 'gptq' | 'fp8' | null
  dtype: 'auto',             // 'auto' | 'bfloat16' | 'float16' | 'float32'
  max_model_len: 4096,
});

fs.writeFileSync('vllm.json', config);
// $ vllm serve --config-file vllm.json

Parameters

FieldTypeDefaultValidation
artifactstringrequiredNon-empty string. Used both as the model path and as served_model_name.
gpu_memory_utilizationnumber0.9Must satisfy 0 < n <= 1. Lower values leave room for other GPU workloads on the same card.
tensor_parallel_sizeinteger1Positive integer. Set to the GPU count for multi-GPU sharding.
quantizationstring | nullnullOne of vLLM's quantization names (awq, gptq, fp8, etc.) or null to let vLLM read it from the artifact.
dtypestring'auto'Lets vLLM pick by default. Override only when the artifact's compiled dtype must be forced.
max_model_leninteger4096Integer >= 128. Match this to the artifact's training context length; exceeding it is a runtime error in vLLM.

Output

Pretty-printed JSON, two-space indent. Every field present whether you passed it or not — defaults explicit so a reader can diff two configs without ambiguity.

{
  "model": "/models/qwen3-8b-support-q4km",
  "quantization": null,
  "dtype": "auto",
  "max_model_len": 4096,
  "gpu_memory_utilization": 0.9,
  "tensor_parallel_size": 1,
  "enforce_eager": false,
  "swap_space": 4,
  "kv_cache_dtype": "auto",
  "trust_remote_code": false,
  "download_dir": null,
  "served_model_name": "/models/qwen3-8b-support-q4km",
  "served_model_aliases": [],
  "disable_log_stats": false,
  "generator": "r4-v1"
}

Field reference

FieldvLLM behaviour
enforce_eagerfalse — lets vLLM compile CUDA graphs. Flip to true only when debugging odd shapes.
swap_space4 GiB — CPU swap for KV cache overflow. Bump on long contexts.
kv_cache_dtype"auto" — vLLM picks based on model. Set to "fp8" on Hopper for ~2x throughput.
trust_remote_codefalse — opt-in to running model code from disk. Refuses by default for supply-chain safety.
download_dirnull — tells vLLM to use its default HuggingFace cache. Override to pin a local cache directory.
served_model_aliases[] — extra names the OpenAI-compatible endpoint will answer to. Hand-edit after emit.
disable_log_statsfalse — emits per-step throughput logs. Flip to true in CI to silence the firehose.
generator"r4-v1" — the schema version the file was written against. Stays as the file ages.

Validation errors

The generator throws on bad input rather than emitting a config that would fail to load. The throws are explicit and actionable:

generateVllmConfig({ artifact: 'foo', gpu_memory_utilization: 1.5 });
// Error: generateVllmConfig: gpu_memory_utilization must be in (0, 1], got 1.5

generateVllmConfig({ artifact: 'foo', tensor_parallel_size: 0 });
// Error: generateVllmConfig: tensor_parallel_size must be a positive integer, got 0

generateVllmConfig({ artifact: 'foo', max_model_len: 64 });
// Error: generateVllmConfig: max_model_len must be an integer >= 128, got 64
Caveats. This generator only writes a config; it does not check that vLLM is installed, that CUDA is on the host, or that the artifact at model exists. Pair with kolm doctor (which probes for Python, CUDA, and the model directory) before vllm serve.
