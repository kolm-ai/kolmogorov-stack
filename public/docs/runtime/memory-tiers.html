Memory tiers and Auto-Placement

Every kolm artifact has to live somewhere. The runtime picks where, then tells you the expected decode throughput before the model loads. Four tiers, one decision, one reasoning line. No manual GPU layers, no offload ratios, no mmap flags.

The four memory tiers

Auto-Placement walks the tiers from fastest to slowest and stops at the first one that fits the artifact with headroom. Each tier has a predictable throughput band. The estimator surfaces both the chosen tier and the bucket it lives in.

Tier 1 · Fastest

VRAM

GPU device memory. Whole-artifact-in-VRAM is the best case: no host-device transfers on the decode path, headroom left over for activations and the KV cache. Detection is nvidia-smi --query-gpu=memory.free across every visible GPU, summed. Throughput band: roughly 80 to 120 tok/s.

Tier 2 · Spillover

RAM

Host system memory. When the artifact spills VRAM, part of the weights sit in RAM and PCIe-stream into the device on each forward pass. Detection is os.freemem(): free memory, not total, so the estimator never promises capacity the OS already holds. Mixed VRAM+RAM throughput drops sharply: roughly 20 to 50 tok/s.

Tier 3 · Last local resort

NVMe

Local SSD. When even RAM cannot hold the artifact, the runtime mmaps from disk and pays page-cache cost on every decode step. Detection is fs.statfsSync(KOLM_DATA_DIR): free bytes available to unprivileged callers, walked up to the nearest extant parent directory. Mostly-NVMe throughput is slow but still functional: roughly 2 to 8 tok/s.

Tier 4 · Last resort

network

Remote weight pull. When nothing local fits, the runtime can stream weights over the wire if you wire KOLM_NETWORK_GBPS with your link speed. Throughput is bounded under 1 tok/s because the network roundtrip dominates every decode step. If KOLM_NETWORK_GBPS is unset the field reports null instead of guessing.

Auto-Placement

Auto-Placement is the decision rule that picks the highest tier where the artifact fits. It runs before the model loads, prints a one-line reasoning summary, and surfaces a structured envelope for machine consumers. The flag is on by default for kolm run.

kolm run my-artifact.kolm --input "your prompt"
# Auto-Placement is on by default. Equivalent to:
# kolm run my-artifact.kolm --auto-place --input "your prompt"

The decision rule

  1. Try VRAM. If vram_gb - headroom >= artifact_size_gb, place the whole artifact in VRAM. We hold back ~1 GB of headroom for activations and the KV cache so the model does not OOM on the first decode step.
  2. Try mixed VRAM+RAM. If (vram_gb - headroom) + ram_gb >= artifact_size_gb and at least some VRAM is available, split the layers proportionally. The reasoning line reports the split (e.g. 40% layers in VRAM, 60% in RAM).
  3. Try RAM-only. If ram_gb >= artifact_size_gb but no VRAM is available, serve from host RAM. CPU-bound throughput, but no spilling to disk.
  4. Try NVMe. If nvme_gb >= artifact_size_gb, mmap from disk. Slow but functional.
  5. Fall back to network. When nothing local fits, surface the network-fetch path so the user knows what to expect.

Example reasoning lines

HostArtifactReasoning line
RTX 5090 (32 GB VRAM) 17.9 GB INT4 fits in 32GB VRAM with 14.1GB headroom — expected ~95 tok/s
RTX 3060 (12 GB VRAM) + 64 GB RAM 30 GB BF16 spills VRAM (12GB avail, artifact 30GB): 37% layers in VRAM, 63% in RAM — expected ~30 tok/s
CPU-only laptop (16 GB RAM) 5 GB INT4 no VRAM available; serving from 16GB RAM — expected ~20 tok/s
Edge box (8 GB RAM, 1 TB NVMe) 200 GB BF16 RAM too small for 200GB artifact; mmap from 1000GB NVMe — expected ~5 tok/s

Placement-only dry-run

You can preview the decision without loading the artifact. Useful in CI, capacity planning, and operator scripts that need to validate a host can run a given model before scheduling work.

kolm run my-artifact.kolm --auto-place --placement-only
# placement: vram
# expected_tok_per_s: 95
# tiers: { vram_gb: 32, ram_gb: 64, nvme_gb: 512, network_gbps: null }
# reasoning: fits in 32GB VRAM with 14.1GB headroom — expected ~95 tok/s

The --placement-only flag exits with status 0 after printing the decision. No artifact resolution, no recipe load, no inference. Add --json if you want the full envelope for machine consumers.

Flags

FlagBehavior
--auto-placeEnable Auto-Placement. ON by default for kolm run. Pass explicitly when scripting.
--placement-onlyPrint the placement decision and exit 0 without loading the artifact. Pure dry-run.
--artifact <path>Path to the .kolm artifact. If the file exists the estimator uses its on-disk size; otherwise it falls back to a 0 GB stub for the dry-run.
--jsonEmit the full placement envelope as JSON instead of the human-readable reasoning line.

Environment variables

VariableEffect
KOLM_DATA_DIROverride the path probed for NVMe free bytes. Defaults to the parent of ~/.kolm.
KOLM_NETWORK_GBPSPin your link speed in gigabits per second. Typical home gigabit is 1.0; 10GbE office LAN is 10.0. Unset means the network tier reports null instead of guessing.

Honesty contract

  • Every probe is wrapped in try/catch. A failing probe yields 0 (for the GB tiers) or null (for network). Never a throw, never a negative number, never undefined.
  • Hosts without nvidia-smi report vram_gb: 0 and the estimator routes around the GPU tier accordingly. CPU-only laptops and CI runners get a real placement decision, not an error.
  • The tok/s numbers are a FIRST-PASS estimator. They come from rough buckets and will be recalibrated against the W721-3 bench data when that lands. We err on the conservative end of each bucket so users are not over-promised.
  • The decision is exposed via MEMORY_TIER_VERSION (currently w724-v1) so a verifier comparing two run-meta records can detect schema-incompatible placement plans.
