kolm.ai / docs / runtime / horizontal-scaling

Open-source AI workbench

Frontier AI on your own infrastructure.

When traffic outruns local capacity, kolm sheds load gracefully. Stateless workers behind a priority-aware queue, HTTP 429 with Retry-After for backpressure, and an optional teacher-overflow path that absorbs spillover into the hosted W709 router. No silent drops, no OOM, no opaque proxy errors.

Stateless workers

Every kolm worker is stateless on the request path. The auth lookup, the rate counter, the audit append, and the placement decision all read from a shared store; nothing about a worker's local memory determines whether the next request succeeds. That means you can scale workers horizontally by adding processes behind a load balancer without coordinating drain windows or sticky sessions.

Worker boot is fast because the heavy compile-time work happens during kolm compile, not at request-time. A fresh worker reads the artifact from local cache (or pulls from the registry on cold start), warms the placement decision, and starts serving in the time it takes to bind a port. The first request pays the warm-up cost; every subsequent one is hot.

The runtime keeps zero per-tenant state in worker memory. Tenant quotas, cached embeddings, and rate-limit counters live in the durable store (or a shared cache when configured). Killing any worker mid-flight loses nothing the caller cares about: the next retry against the same load balancer lands on a different worker that sees the same authoritative state.

Queue depth as load signal

The W729 load queue is the load signal you autoscale on. Every inference route checks in with the queue before doing work; if a slot is free the request runs immediately, if all slots are busy the request waits in a priority lane. Enterprise traffic dequeues before business, business before starter, starter before free. Within a lane the order is strict FIFO.

Read the queue depth from any worker with kolm load queue --stats. The output is a single JSON snapshot with depth (waiting + in-flight), capacity (configured slot count), and a by_priority breakdown so an operator can see whether the contention is concentrated in one tier. The same numbers are exposed on the server as the source of truth - every worker reports them identically because the queue is per-worker, not shared.

Saturation has two failure modes, both honest. A request that waits past its timeout_ms resolves to HTTP 429 with Retry-After set to a value the SDK respects on retry. A request that arrives when the queue is already at four times capacity (the hard backstop against unbounded memory growth) immediately returns 429 with queue_full. The client sees one envelope shape in both cases: {ok:false, error:'queue_full'|'queue_timeout', retry_after_seconds, queue_depth}.

Horizontal autoscale playbook

Wire the queue depth into your orchestrator's autoscale signal. On Kubernetes, an HPA reading the queue_depth / capacity ratio works well: scale up when the ratio crosses 0.7 sustained for thirty seconds, scale down when it stays below 0.3 for two minutes. On a fleet manager, the same metric drives a worker-count target. Pin the lower bound at two workers so a single restart never empties the pool, and the upper bound at whatever your inference budget allows.

Configure overflow before you autoscale. Set KOLM_TEACHER_OVERFLOW_URL on every worker; when the queue saturates, the W729 enqueue path forwards the request to the W709 hosted teacher instead of rejecting. The caller sees a real response, the worker sheds load, and the autoscale trigger fires on the queue-depth metric before the overflow path becomes the steady-state.

Tune the queue capacity per host. The default is sixteen concurrent slots, which fits a single-GPU box running an INT4 distilled artifact at roughly ninety tokens per second. For a larger box raise it with kolm load queue --capacity 64 (requires KOLM_CLI_ROLE=admin); for a CPU-only worker drop it to four. The right number is whichever value lets the artifact's expected throughput keep up with the slot count without backing up the queue. Read memory tiers for the placement decision that picks the expected tokens-per-second band.

Environment variables

VariableEffect
KOLM_LOAD_QUEUE_DISABLEDSet to 1 to disable the queue entirely. enqueue() resolves immediately and the production hot path is preserved bit-for-bit.
KOLM_TEACHER_OVERFLOW_URLForward overflow requests to a hosted teacher URL (W709 routing plumbing). Unset means the queue rejects with HTTP 429 when full.
KOLM_CLI_ROLESet to admin to authorize kolm load queue --capacity <N>. Non-admin callers see a forbidden envelope and exit 3.

Honesty contract

  • The queue never silently drops. Timeouts and full-queue conditions both surface a structured 429 envelope with Retry-After so the SDK can back off correctly.
  • The queue is per-worker. Stats reported from one process do not reflect the fleet; aggregate at the load balancer or push queue_depth into your observability stack to see the fleet view.
  • The kill-switch is real. KOLM_LOAD_QUEUE_DISABLED=1 makes enqueue() a no-op so the pre-W729 hot path is preserved when ops decides the queue is causing more harm than good.
  • The version stamp is exposed via LOAD_QUEUE_VERSION (currently w729-v1) so a verifier comparing two queue-meta records can detect schema-incompatible queue plans.
