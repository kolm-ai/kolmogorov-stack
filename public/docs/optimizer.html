Optimizer

The optimizer is a read-only scanner over the event lake. It runs on demand or on a schedule, scores every detected pattern by expected dollar savings, and emits a ranked list of opportunities. You pick which ones to act on. The optimizer never modifies traffic on its own.

Run it

kolm optimize --window 7d
# scanning 18,442 events ... 11 detectors ... 7 opportunities found
# total addressable savings: $312.40 / month
#
# 1. [local_replacement] classifier on gpt-4o-mini, 4,200 calls/day -> $186/mo
# 2. [cache] 81% duplicate prompts in /summarize -> $58/mo
# 3. [cheaper_model] gpt-4o on simple_translate -> $41/mo
# ...

The 11 opportunity types

Each detector looks for a specific cost or quality regression in the lake. Each finds a pattern that has a concrete remediation.

TypePatternRemediation
cacheIdentical prompt seen N+ times in the window with identical responseTurn on the proxy response cache for that template hash
cheaper_modelCalls where a cheaper model would have scored within tolerance on the bakeoffSwap the upstream model id; gate behind a holdout
local_replacementRepetitive narrow task with stable structureDistill into a local .kolm artifact, serve via runtime
privacy_leakCalls where a detector class fired but the policy was set to allowFlip the per-class action to redact or block
prompt_compressionVerbose system prompts that compress losslesslyReplace with a shorter equivalent; verify on holdout
repeated_extractionSame JSON-schema extraction over and over on similar inputsCompile to an extractor artifact
repeated_classificationSingle-label outputs from a finite class setCompile to a classifier artifact
log_triageUnstructured logs being summarized one at a timeBatch through a triage artifact
routing_policyCalls that crossed providers but did not need toPin the cheaper provider, gate with a fallback policy
dataset_readyNamespace has crossed the per-task minimum row countBuild a labelled dataset
training_readyDataset is split, labelled, and within budget for a training runKick off the training plan

JSON output

For piping into your own dashboard, ticket system, or CI gate.

kolm optimize --window 30d --json | jq '.[0]'
# {
# "type": "local_replacement",
# "namespace": "ticket-classifier",
# "evidence_event_count": 12640,
# "addressable_spend_usd": 186.42,
# "confidence": 0.91,
# "remediation": {
# "command": "kolm distill ticket-classifier --kind classifier",
# "estimated_run_minutes": 22
# }
# }

Schedule

The recommended cadence is daily. Wire it into your scheduler of choice and pipe high-confidence findings to a channel.

0 9 * * * kolm optimize --window 24h --json --min-confidence 0.8 | tee /var/log/kolm/optimizer.json

Thresholds

Tune the per-type minimums in ~/.kolm/config.json so you do not get noise on a small lake.

{
 "optimizer": {
 "cache_min_repeats": 5,
 "local_replacement_min_events": 500,
 "dataset_ready_min_rows": 1000
 }
}

Next steps

The most common follow-on is distillation for local_replacement, or runtime to set up routing for routing_policy. Start from /docs/quickstart if your lake is still empty.

Lake →The 35-field event row the optimizer scans. Distillation →Act on local_replacement findings. Datasets →Act on dataset_ready findings.
