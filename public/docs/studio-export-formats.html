Studio / Export / Format picker

Pick the format by where the model will run.

There is no "best" quant format — only the right format for the deployment target. GGUF is the most portable. EXL2 has the best single-GPU throughput. GPTQ and AWQ run on vLLM in clusters. FP8 unlocks Hopper, NVFP4 unlocks Blackwell. MLX is Apple Silicon native. The table below picks one in 30 seconds.

Format-by-format

GGUF (src/export-gguf.js)

Pick when: the model will run on a laptop, phone, Raspberry Pi, edge box, or anywhere llama.cpp / Ollama / LM Studio runs. Most portable format in 2026; widest ecosystem. Self-contained single file. See studio-export-gguf.

Don't pick when: you need maximum vLLM throughput (use AWQ/GPTQ) or you have a Hopper / Blackwell rack (use FP8 / NVFP4).

EXL2 (src/export-exl2.js)

Pick when: you run on a single consumer GPU (3090/4090/5090) and need every last token/sec. ExLlamaV2 is the throughput king on a single CUDA card. Supports arbitrary bpw (3.0 / 4.0 / 4.5 / 5.0 / 6.0 / 8.0 — you can fine-tune the size to the available VRAM exactly).

Don't pick when: the model is going to vLLM (EXL2 isn't supported there). Don't pick for on-device.

GPTQ (src/export-gptq.js)

Pick when: you deploy via vLLM and quality matters more than speed. GPTQ with g128 Hessian calibration has the highest quality of the 4-bit vLLM formats on most models.

Don't pick when: throughput matters more (AWQ is faster). Don't pick if you have no calibration set (use HQQ).

AWQ (src/export-awq.js)

Pick when: you deploy via vLLM and throughput matters. AWQ kernels are usually 1.3-1.6× faster than GPTQ at the same bpw. Activation-aware quantization — preserves the salient weights better than uniform schemes.

Don't pick when: quality is the binding constraint (use GPTQ).

FP8 (src/export-fp8.js)

Pick when: you have Hopper (H100/H200) or Ada (L40S) or Blackwell. FP8 E4M3 gives near-fp16 quality at half the memory and double the throughput on Hopper. Native first-class in TensorRT-LLM and vLLM.

Don't pick when: the deployment GPU is Ampere or older (no FP8 hardware path).

NVFP4 (src/export-nvfp4.js)

Pick when: you have Blackwell (B100/B200/RTX 5090). NVFP4 is the new 4-bit microscaling format that the Blackwell Transformer Engine supports natively — 2× the throughput of FP8 at near-fp8 quality. Bleeding edge as of 2026.

Don't pick when: you're not on Blackwell.

HQQ (src/export-hqq.js)

Pick when: you need a 4-bit quant fast and don't have a representative calibration set. HQQ is calibration-free, so produce time is <1 min on a 7B vs ~15 min for GPTQ/AWQ. Quality is usually 0.5-1 K-Score below AWQ but close to GPTQ on instruction-tuned models.

Don't pick when: you have a representative calibration set (use GPTQ/AWQ).

MLX (src/export-mlx.js)

Pick when: the model will run on Apple Silicon (M1-M4). mlx-lm is Apple's native ML framework — it sees the unified memory architecture and runs ~1.5-2× faster than GGUF on the same Mac.

Don't pick when: the model needs to be portable (use GGUF).

Multi-target builds

You usually want 2-3 formats per artifact: a portable GGUF (Q4_K_M + Q8 for HuggingFace), a vLLM target (AWQ or FP8), and the fp16 .kolm archive. Compile once, export many:

kolm export trinity-support.kolm \
  --format gguf-q4km,gguf-q5km,gguf-q8,awq-4bit,fp8 \
  --output-dir release/
Caveats. Every quantization format eats a different slice of K-Score. The compile receipt records quality_delta_vs_fp16 per format; the right format for you is the one that fits your VRAM and stays within the K-Score budget your eval defines. Don't ship a format you haven't benchmarked.
