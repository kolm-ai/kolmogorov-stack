Provider failover

Ordered fallback on every retryable failure.

Declare a primary plus an ordered fallback chain in gateway.toml. On 429, 502, 503, 504, or network timeout the gateway walks the chain in order until one succeeds or the retry budget is exhausted.

What counts as retryable

The gateway distinguishes retryable from terminal failures. Only retryable failures trigger the fallback chain; terminal failures (400 bad request, 401 unauthorized, 422 unprocessable) propagate to the caller because retrying the same request against a different provider will produce the same error.

Status / eventRetryable?Default backoff
429 rate limitYesRespect Retry-After header.
500 upstream errorYes100ms + exponential.
502 bad gatewayYes100ms + exponential.
503 service unavailableYes500ms + exponential.
504 upstream timeoutYesImmediate next provider.
network timeoutYesImmediate next provider.
TCP resetYesImmediate next provider.
400 bad requestNo— (propagates).
401 unauthorizedNo— (propagates).
403 forbiddenNo— (propagates).
404 not foundNo— (propagates).
422 unprocessableNo— (propagates).

Retry budget

The chain has a budget that caps total attempts. Defaults to retry_budget = 3. If the chain is longer than the budget, the gateway walks the chain in order until the budget runs out. A failed attempt counts against the budget even if it returns immediately (network refused).

[[namespace]]
name           = "production-rag"
primary        = "openai/gpt-4o"
fallback       = [
  "anthropic/claude-sonnet-4-5",   # tried on primary failure
  "deepseek/deepseek-v3.1",        # tried if claude also fails
  "local-vllm/qwen-3.6-27b"        # last-ditch local fallback
]
retry_budget   = 4
retry_backoff  = "exponential"     # exponential | linear | none
retry_jitter   = "0.1"             # 10% jitter on backoff

Fallback receipts

Every fallback attempt is recorded on the receipt under attempts[]. The final attempt that succeeded is the canonical response; previous attempts include the upstream status, latency, and reason. Use this to debug flaky providers and to feed the active-learning loop.

kolm captures get <receipt_cid> --json | jq '.attempts'
[
  {"provider":"openai","model":"gpt-4o","status":429,"reason":"rate_limit","latency_ms":12},
  {"provider":"anthropic","model":"claude-sonnet-4-5","status":200,"reason":"ok","latency_ms":847}
]

Per-request override

A caller can disable fallback for one request (useful when the application has its own retry policy and does not want a slower fallback masking a primary outage).

curl https://kolm.ai/v1/wrap/openai/chat/completions \
  -H "X-Kolm-No-Fallback: true" \
  ...

Health-based pre-emptive skip

Optionally, the gateway can skip a fallback entry that has failed N times in the last M seconds (circuit breaker). Off by default; opt in per namespace.

[[namespace]]
name                 = "production-rag"
primary              = "openai/gpt-4o"
fallback             = ["anthropic/claude-sonnet-4-5", "deepseek/deepseek-v3.1"]
circuit_breaker      = true
circuit_window       = "60s"
circuit_threshold    = 5            # 5 failures in 60s opens the circuit
circuit_reset        = "30s"        # try again after 30s
Caveats. Circuit breakers can mask transient flapping that you would want to see. Pair with telemetry alerts on kolm_circuit_open events so you know when a provider is being skipped.
