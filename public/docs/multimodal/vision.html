kolm.ai / docs / multimodal / vision

Open-source AI workbench

Vision-language distillation, end-to-end.

Capture image_url content blocks from any chat turn, distill a smaller VLM student against the captured teacher output, and replay vision pairs through your compiled .kolm artifact with bake-off scoring.

Supported teacher VLMs

anthropic

Claude 3.5 Sonnet vision

Anthropic Messages API vision input. content[].type === 'image' with base64 or URL source. Strong on chart/diagram reasoning; the default teacher for screenshot triage workloads.

openai

GPT-4o vision

OpenAI Chat Completions vision input. content[].type === 'image_url' with either remote URL or data:image/...;base64,... inline payload.

google

Gemini 1.5 Pro vision

Google Generative AI / Vertex AI. content[].fileData for GCS-pointer images or content[].inlineData for inline base64. Largest context window of the three frontier vision teachers.

open-source

Qwen2-VL · LLaVA-1.6

Qwen/Qwen2.5-VL-{3,7,32,72}B-Instruct and the LLaVA-NeXT family. Wired through apps/trainer/vlm.py's AutoProcessor + AutoModelForVision2Seq path. Stay local-only for privacy-sensitive distill runs.

Capture pattern

Vision detection runs at the capture chokepoint — src/vision-capture.js recognizes the three vendor schemas and stamps has_vision:true plus a per-row vision_block_count on the persisted observation. Raw image bytes never enter the store; only the URL string (for URL-source blocks) and a sha256 of the URL (for dedup across captures) land on disk.

openai / openrouter

image_url

Two shapes accepted. Object form carries an optional detail field for hint-grade sizing.

{
  "role": "user",
  "content": [
    {"type": "text", "text": "What's in this image?"},
    {"type": "image_url",
     "image_url": {"url": "https://...png", "detail": "high"}}
  ]
}

anthropic

image (source: base64 | url)

Anthropic separates the source type from the payload. The vision-capture chokepoint hashes the URL or hashes a base64 descriptor (mime + estimate + source) so dedup works without ever storing the bytes.

{
  "role": "user",
  "content": [
    {"type": "image",
     "source": {"type": "base64",
                "media_type": "image/png",
                "data": "iVBORw0KGgo..."}},
    {"type": "text", "text": "Describe."}
  ]
}

google

fileData / inlineData

Gemini distinguishes URL-pointer (fileData, often gs://) from inline (inlineData) image payloads. Both are normalized into the canonical capture shape.

{
  "role": "user",
  "parts": [
    {"text": "What's this?"},
    {"fileData": {
       "mimeType": "image/jpeg",
       "fileUri": "gs://my-bucket/photo.jpg"}}
  ]
}

Trainer pipeline

The distill entry point is apps/trainer/vlm_distill.py — a stdlib-only Python wrapper that consumes a JSONL of captured vision turns. The --dry-run path NEVER imports torch; it validates input shape, counts vision rows, and emits a run-meta scaffold with trainer_not_invoked:true. Live mode imports torch + transformers + PIL lazily; missing deps exit 3 with an honest envelope rather than silently degrading.

# dry-run (stdlib-only; never imports torch)
python apps/trainer/vlm_distill.py \
  --captures vision_captures.jsonl \
  --out run-meta.json \
  --dry-run

# live mode (requires torch + transformers + pillow)
python apps/trainer/vlm_distill.py \
  --captures vision_captures.jsonl \
  --out run-meta.json \
  --model claude-3-5-sonnet-vision \
  --max-samples 5000

The trainer NEVER fabricates results. Dry-run emits a scaffold tagged trainer_not_invoked:true. Live mode either runs a real distill loop or exits with error:'torch_not_available' / error:'live_trainer_not_wired' — we will not silently no-op.

Bake-off

Once the .kolm artifact is built, replay captured vision turns through it via src/vlm-bakeoff.js. The replay is tenant-fenced under W411 defense-in-depth (per-row tenant_id filter), scores by Jaccard token overlap by default (heuristic judge), and groups results by image_kind (photo, screenshot, diagram, chart, other). A callable judge can be wired through the opts.judge DI seam for embedding-similarity scoring without changing the bake-off envelope.

FieldMeaning
count_totalTotal vision capture rows for the tenant + namespace.
count_vision_pairs_evaluatedRows that produced a score (after row-level run errors are dropped).
by_image_kindPer-bucket {count, avg_score} across photo / screenshot / diagram / chart / other.
avg_scoreMean Jaccard token-overlap across all evaluated pairs.
judge_kindheuristic (default) or callable (when opts.judge is wired).

CLI

The kolm vlm verb (alias: kolm vision) wraps the three auth-gated routes. Bake-off requires --confirm — matches the W411 spend-protection pattern so a misclick or stray curl cannot fire the replay against every captured row.

kolm vlm captures --namespace prod                    # list vision rows
echo '{"role":"user","content":[...]}' | kolm vlm detect

kolm vlm bakeoff ./my.kolm --namespace prod --confirm
kolm vlm bakeoff ./my.kolm --namespace prod --max 50 --confirm
kolm vision captures                                  # alias works the same

API

EndpointWhat it does
POST /v1/vision/capture-detectPure detector: body {message} returns {is_vision, image_url_blocks, base64_blocks, total_images}. No persistence.
POST /v1/vision/bakeoffReplays vision captures through the named .kolm artifact. confirm:true REQUIRED. Returns the bake-off envelope.

Tenant fence (W411 defense-in-depth)

Every vision read does store.all('observations').filter(r => r.tenant === tenant_id || r.tenant_id === tenant_id) rather than the shortcut findByTenant. The defense-in-depth filter runs even after the all() read so a future store schema change cannot silently flip "no leak" to "all leak". The captureVisionMessage row stamps BOTH tenant and tenant_id fields so the fence fires no matter which key the index ends up using.

Privacy honesty

Raw image bytes never enter the capture store. URL-source blocks land as the URL string + a sha256 digest of the URL. Base64-source blocks land as a synthetic digest of (mime + estimate + source) — dedup still works without storing the payload. The byte_count_estimate field is capped at 50MB so an adversarial caller cannot blow up the column with a 4GB base64 payload estimate.
