Open-source AI workbench

Video distillation — frame sampling and caption pipeline.

Capture inbound video URLs paired with teacher responses, sample frames via uniform / keyframe / scene-change / adaptive strategies, and distill a smaller student grounded on per-frame captions plus a per-content-kind bake-off.

Supported video MIME types

The capture chokepoint (src/video-capture.js) normalizes five MIME types covering every video shape the major frontier vision-video vendors accept as of 2026-Q2. The set is Object.freeze()-d — a refactor cannot push a sixth MIME without bumping VIDEO_CAPTURE_VERSION.

video/mp4

MP4 (H.264 / H.265)

ISO base media file format. The most-accepted video shape across Gemini 1.5 Pro, GPT-4o video, and Claude 3.5 Sonnet video. Default for streamed uploads.

video/webm

WebM (VP9 / AV1)

Royalty-free container. Default output of browser MediaRecorder in Chromium. Smaller wire size than MP4 at equivalent fidelity.

video/quicktime

QuickTime (.mov)

Apple container, frequently produced by iOS screen recordings and ProRes captures. Mostly identical to MP4 internally.

video/x-msvideo

AVI

Microsoft container. Widely produced by older screen recorders even when not accepted by every vendor — the capture detects + frankly rejects before upload rather than after a 500.

video/x-matroska

Matroska (.mkv)

Open container often used for archive footage. Same handling as AVI: detect locally + reject frankly when the downstream vendor does not accept it.

Frame sampling strategies

The frame-sampler (src/frame-sampler.js) emits a portable JSON SPEC the Python trainer consumes; the heavy frame-extraction (ffmpeg + cv2) lives outside Node. SAMPLING_STRATEGIES is a frozen 4-entry closed enum — the density-multiplier table in the JS estimator and the Python trainer are mirrored, so the dry-run estimate never diverges from the live run.

uniform

Uniform fps_target

Evenly-spaced midpoint sampling at fps_target across the clip duration. Deterministic, cheap, good for surveillance and continuous footage.

density mult: 1.0x

keyframe

Container I-frames

Extract I-frames per the container GOP metadata. Cheap and deterministic, but density varies wildly by encoder (some screen recorders emit one I-frame per minute).

density mult: 0.35x

scene_change

Pixel-diff scene cuts

Run a pixel-diff threshold across decoded frames looking for cut points. More accurate for tutorials and presentations, more expensive than keyframe.

density mult: 0.20x

adaptive

Motion-densified uniform

Start uniform, then densify around high-motion regions using a motion histogram. Most accurate, slowest. Spec rides 1.2x density to reflect the densification budget.

density mult: 1.2x

Every spec carries a HARD_FRAME_CAP of 1024 frames — a 24-hour surveillance clip at fps_target=30 cannot emit 2.5M sampling indices. Trainers that genuinely need more must chunk.

Capture pattern

Video detection runs at the capture chokepoint — src/video-capture.js recognizes three block shapes: Anthropic-style {type:'video', source:{type:'base64',...}}, OpenAI-style {type:'video_url', video_url:{url:...}}, and generic {url:...} with a recognized video/* MIME. The detector NEVER reads bytes off URLs; it is a SHAPE inspector only.

Raw video bytes never enter the capture store. URL-source blocks land as a sha256 digest of the URL. Base64-source blocks land as a sha256 hash of a synthesized placeholder (data:video/<mime>;base64,<elided>) so dedup still works without storing the payload. The capture row carries raw_video_bytes_persisted:false explicitly — an auditor can grep for the flag and confirm.

// Anthropic-style base64 block
{
  "role": "user",
  "content": [
    {"type": "text", "text": "What is happening in this clip?"},
    {"type": "video",
     "source": {
       "type": "base64",
       "media_type": "video/mp4",
       "data": "AAAAGGZ0eXBtcDQy..."
     }}
  ]
}

// OpenAI-style URL block
{
  "role": "user",
  "content": [
    {"type": "text", "text": "Summarize the screencast."},
    {"type": "video_url",
     "video_url": {"url": "https://cdn.example.com/clip.mp4"}}
  ]
}

The byte_count_estimate field is capped at 1 GiB (MAX_BYTE_COUNT_ESTIMATE). A hostile or accidentally-huge Content-Length header cannot force a 100 GiB downstream allocation; when the cap fires, the envelope sets byte_count_capped:true so the clamp is observable and never silent.

Trainer pipeline

The distill entry point is apps/trainer/video_distill.py — a stdlib-only Python wrapper that consumes a JSONL of captured video turns. The --dry-run path never imports torch, ffmpeg, or cv2; it parses the captures file, counts video rows by source (url / base64 / pre-captioned), and emits a structured envelope tagged trainer_not_invoked:true. Live mode imports torch + transformers + cv2 lazily; missing deps exit 3 / 5 / 6 with an loud-fail envelope rather than silently degrading.

# dry-run (stdlib-only; never imports torch / cv2 / ffmpeg)
python apps/trainer/video_distill.py \
  --captures video_captures.jsonl \
  --out run-meta.json \
  --dry-run

# live mode (requires torch + transformers + opencv-python + ffmpeg-python)
python apps/trainer/video_distill.py \
  --captures video_captures.jsonl \
  --out student-weights/ \
  --teacher-model gemini-1.5-pro \
  --frame-sampler-spec spec.json \
  --max-samples 1000

The trainer NEVER fabricates results. Dry-run emits a scaffold tagged trainer_not_invoked:true with total_frames_estimated sourced from the SAME density math the live run uses. Live mode either runs a real frame-extract-plus-distill loop or exits with error:'torch_not_available' (exit 3), error:'no_video_captures' (exit 4), error:'cv2_not_available' (exit 5), or error:'teacher_load_failed' (exit 6) — we will not silently no-op.

Bake-off — per-content-kind score breakdown

Once the .kolm artifact is built, replay captured video turns through it via src/video-bakeoff.js. The replay is tenant-fenced under defense-in-depth (per-row tenant_id filter even after listEvents pre-filters), scores by Jaccard token overlap against the captured teacher response (heuristic judge), and buckets per-row by content kind. A tutorial-trained student should win on tutorial captures and lose on surveillance captures; without the per-kind breakdown that signal would be averaged away.

tutorial

"How to", "step by step", "walkthrough", or "lesson" keyword hits in the captured response head or URL path.

screencast

"Screencast", "screen recording", or "demo" keyword hits.

presentation

"Slide(s)", "slidedeck", "webinar", or "keynote" keyword hits.

surveillance

"Surveillance", "CCTV", "camera-feed", "security camera", or "dashcam" keyword hits.

other

Catch-all for rows that match none of the four keyword buckets.

FieldMeaning
count_totalTotal video capture rows for the tenant + namespace.
count_video_pairs_evaluatedRows that produced a score after row-level run errors are dropped.
by_content_kindPer-bucket avg score + count across tutorial / screencast / presentation / surveillance / other. Empty kinds report 0/null rather than being omitted.
avg_scoreMean Jaccard token-overlap across all evaluated pairs.
median_scoreMedian Jaccard across evaluated pairs. Less sensitive to outlier replays than the mean.
judge_kindjaccard (default) or external (when opts.judge is wired). NEVER fabricated.

CLI usage

The kolm video verb wraps the three auth-gated routes plus a local sampling-spec generator. Bake-off requires --confirm — matches the spend-protection pattern so a misclick or stray curl cannot fire the replay against every captured video row.

# local  -  pure JS, no network, no auth required
kolm video sampling-spec --duration-s 600 --strategy adaptive --fps-target 1 --max-frames 64

# auth'd
kolm video captures --namespace prod
echo '{"role":"user","content":[...]}' | kolm video detect --message-file -

kolm video bakeoff --artifact-path ./student.kolm --namespace prod --confirm
kolm video bakeoff --artifact-path ./student.kolm --namespace prod --max-n 50 --confirm

Tenant fence

Every video read does a per-row tenant_id filter even after the canonical event-store has already filtered. The defense-in-depth filter runs after the listEvents read so a future store schema change cannot silently flip "no leak" to "all leak". The captureVideoMessage chokepoint stamps the canonical event with media_kind:'video' + the tenant fence column so reads land deterministically.
