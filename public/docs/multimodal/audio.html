Open-source AI workbench

Audio distillation — transcript and intent.

Capture audio inputs as whisper transcripts paired with teacher responses, distill a smaller student grounded on the transcript text, and replay audio captures through your compiled .kolm artifact with intent-bucketed bake-off scoring.

Supported audio MIME types

The capture chokepoint (src/audio-capture.js) normalizes seven mime types covering every audio shape both major chat providers accept on the inbound side as of 2026-05. The set is Object.freeze()-d — a refactor cannot push a new mime without bumping AUDIO_CAPTURE_VERSION.

audio/wav

WAV (RIFF / LPCM)

Uncompressed 16-bit PCM, the default openai-whisper ingestion format. Largest on the wire but lossless — preferred for transcription accuracy.

audio/mp3 · audio/mpeg

MP3

MPEG-1/2 Audio Layer III. Compact, lossy. Two mimes accepted because both providers and browsers disagree on the canonical spelling.

audio/flac

FLAC

Free Lossless Audio Codec. About half the size of WAV at the same fidelity — used for archived call-center transcripts.

audio/ogg

OGG (Vorbis / Opus)

Free container with either Vorbis or Opus codec. The default WebRTC mic-capture format in Firefox.

audio/webm

WebM

Matroska-derived container that carries Opus audio. The default MediaRecorder output in Chromium browsers.

audio/m4a

M4A (AAC)

ISO-BMFF container with AAC audio. The default voice-memo format on iOS and macOS.

Capture pattern

Audio detection runs at the capture chokepoint — src/audio-capture.js recognizes the OpenAI input_audio content block, the Anthropic future audio block, base64 data URLs in plain text content, and pre-transcribed whisper_transcript fields on the message. Raw audio bytes never enter the store; only the transcript text plus a sha256 of the audio URL (or data-URL head) land on disk.

openai

input_audio

Chat Completions audio input. The format field maps directly into the supported MIME table.

{
  "role": "user",
  "content": [
    {"type": "text", "text": "What did the caller say?"},
    {"type": "input_audio",
     "input_audio": {
        "data": "UklGRhYAAABXQVZF...",
        "format": "wav"}}
  ]
}

pre-transcribed

whisper_transcript

When the caller has already run openai-whisper upstream (browser, mobile, on-prem), the message may carry the transcript directly. The detector picks it up and the trainer skips re-transcription.

{
  "role": "user",
  "whisper_transcript": "Schedule a follow-up
   with the patient in two weeks.",
  "content": []
}

Trainer pipeline

The distill entry point is apps/trainer/audio_distill.py — a stdlib-only Python wrapper that consumes a JSONL of captured audio turns. The --dry-run path never imports torch or openai-whisper; it parses the captures file, counts audio rows by source (transcript / url / base64), and emits a run-meta.json envelope tagged trainer_not_invoked:true. Live mode imports whisper + transformers lazily; missing deps exit 3 / 5 / 6 with an honest envelope rather than silently degrading.

# dry-run (stdlib-only; never imports torch / whisper)
python apps/trainer/audio_distill.py \
  --captures audio_captures.jsonl \
  --out run-meta.json \
  --dry-run

# live mode (requires openai-whisper + torch + transformers)
python apps/trainer/audio_distill.py \
  --captures audio_captures.jsonl \
  --out run-meta.json \
  --whisper-model base \
  --teacher-model anthropic/claude-opus-4-7 \
  --max-samples 5000

The trainer NEVER fabricates results. Dry-run emits a scaffold tagged trainer_not_invoked:true. Live mode either runs a real transcribe-plus-distill loop or exits with error:'torch_not_available' (exit 3), error:'whisper_load_failed' (exit 5), or error:'teacher_load_failed' (exit 6) — we will not silently no-op.

Bake-off — intent classification + transcript fidelity

Once the .kolm artifact is built, replay captured audio turns through it via src/audio-bakeoff.js. The replay is tenant-fenced under W411 defense-in-depth (per-row tenant_id filter), scores by Jaccard token overlap against the captured teacher response (heuristic judge), and bucketizes per-utterance by intent.

question

Trailing ? OR leading wh-word OR auxiliary-verb prefix.

command

Imperative leading verb (open, send, deploy, train, ...).

conversation

Greeting, first-person pronoun, or short utterance.

dictation

Long single sentence (>35 words) without trailing question mark.

other

Catch-all for utterances that match none of the four heuristics.

FieldMeaning
count_totalTotal audio capture rows for the tenant + namespace.
count_audio_pairs_evaluatedRows that produced a score (after row-level run errors are dropped).
by_intent_kindPer-bucket counts across question / command / conversation / dictation / other.
avg_scoreMean Jaccard token-overlap across all evaluated pairs.
transcript_coverage_pctShare of rows that carried a whisper transcript (0–100).
judge_kindtranscript_jaccard (default) or di_judge (when opts.judge is wired).

CLI usage

The kolm audio verb wraps the three auth-gated routes. Bake-off requires --confirm — matches the W411 spend-protection pattern so a misclick or stray curl cannot fire the replay against every captured audio row.

kolm audio captures --namespace prod                    # list audio rows
echo '{"role":"user","content":[...]}' | kolm audio detect

kolm audio bakeoff ./my.kolm --namespace prod --confirm
kolm audio bakeoff ./my.kolm --namespace prod --max 50 --confirm

Tenant fence (W411 defense-in-depth)

Every audio read does a per-row tenant_id filter even after the canonical event-store has already filtered. The defense-in-depth filter runs after the listEvents read so a future store schema change cannot silently flip "no leak" to "all leak". The captureAudioMessage chokepoint stamps the canonical event with media_kind:'audio' + the tenant fence column so reads land deterministically.

Privacy honesty

Raw audio bytes never enter the capture store. URL-source blocks land as a sha256 digest of the URL only. Base64-source blocks land as a sha256 digest of the first 256 characters of the base64 payload — dedup still works without storing the audio. The transcript text is capped at 128 KiB; longer transcripts surface a transcript_truncated:true flag so the operator sees the harness clipped instead of silently passing the full blob through.
