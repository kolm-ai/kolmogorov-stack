Eval harness.

kolm's eval harness measures recipe-level pass/fail, not vibes. You hand it a task, a holdout file of input/expected-output pairs, and a model. It returns multi-step pass rate, exact-match for classification, latency p50/p95, cost per call, and a drift score against the previous run. Every eval emits a signed receipt that links back to the model hash and the recipe hash.

Overview

An eval is a contract between a recipe and a holdout. The recipe says how the model should behave; the holdout is the test set the model never saw during distill. The harness scores the model against the holdout, prints per-row pass/fail, and rolls up to a single number you can gate on.

What the harness measures, in one breath:

  • Task-level pass/fail — multi-step pass rate for chat and tool-use, exact-match for classification.
  • Latency — p50 and p95 first-token and end-to-end, against the deployment runtime you targeted.
  • Cost per call — computed from the runtime's per-token rate (cloud) or the GPU-second rate you supply (on-prem).
  • Drift over time — delta against the previous run's pass rate, latency, and cost, with a configurable alert threshold.

Command

Run an eval

Single-line invocation

kolm eval --task <task> --pairs ./holdout.jsonl --model <model>

Worked example for a support replier compiled in the cookbook:

kolm eval \
  --task chat \
  --pairs ./holdout/support-500.jsonl \
  --model ./artifacts/support-bot-v1.kolm \
  --metric reply-similarity,intent-accuracy,tone-match \
  --judge teacher \
  --report ./reports/support-v1.json

The harness streams per-row pass/fail to stdout, writes a full JSON report to --report, and emits a signed receipt to ./receipts/. The report includes the holdout hash, the model hash, the runtime hash, and the recipe hash so you can re-run the same eval six months later and prove the comparison is apples to apples.

Supported task types

TaskPair shapeDefault scorersUse it for
chat {messages, expected} reply-similarity, tone-match, multi-step pass Support replier, agent assistant, generic conversational recipe.
classify {input, label} exact-match, F1, per-class confusion Intent classification, ticket routing, content moderation, label cleanup.
doc-qa {question, context, expected} answer-faithfulness, citation-precision, exact-match Retrieval-augmented question answer, internal wiki Q&A, policy lookup.
tool-use {prompt, tools, expected_call} tool-name match, argument exact-match, multi-step pass Function-calling, agent loop with structured tools, browser automation.

Each task ships with default scorers; override with --metric. Add a custom scorer with --scorer ./my-scorer.js.

Edge-aware mode

For models you intend to deploy on-device (a Raspberry Pi 5, a Jetson Orin, a Mac mini, a browser via WebGPU), measure tok/s and p95 on the target device before you ship. The harness runs the holdout against the model loaded on the device and rolls up the same pass-rate plus device-specific latency and watt budget.

kolm eval \
  --device pi5 \
  --target gguf-q4km \
  --task chat \
  --pairs ./holdout/support-500.jsonl \
  --model ./artifacts/support-bot-v1.kolm \
  --report ./reports/support-v1-pi5.json

Supported device slugs today: pi5 (Raspberry Pi 5, 8GB, CPU only), jetson-orin-nx (Jetson Orin NX 16GB), mac-mini-m2 (Mac mini M2 with 10-core GPU), x86-nuc (x86 NUC with iGPU), web-gpu (Browser WebGPU, Chromium 113+ desktop). See /benchmarks/edge for the current measurement matrix.

The eval connects via the kolm doctor's device registry. Pair the harness with /docs/recipes/edge-deploy to walk the full flow: distill on a 5090, quantize to GGUF Q4_K_M, push to the edge fleet, measure with kolm eval --device, gate the rollout on the report.

Gate on a number

The harness exits non-zero when any of the configured gates fail. Wire it into CI to block a regression from shipping.

kolm eval \
  --task chat \
  --pairs ./holdout/support-500.jsonl \
  --model ./artifacts/support-bot-v1.kolm \
  --gate pass-rate>=0.85,p95-ms<=250,cost-per-call<=0.002 \
  --drift-against ./reports/support-v0.json \
  --max-drift 0.03

The --drift-against flag compares this run to a baseline report; --max-drift sets the maximum allowed regression in pass rate. Any gate failure exits 3 and prints the failing rows.

Caveats / Limitations

  • The judge is the bottleneck on chat-style tasks. When the scorer asks a teacher model to judge similarity, the eval is only as reliable as the judge. Snapshot the judge hash into the report so you can re-judge with the same teacher version when you re-run.
  • Edge measurements vary by build. tok/s on Jetson depends heavily on the llama.cpp build flags (CUDA on vs off, batch size, kv-cache layout). The harness records the build hash so cross-device comparisons stay grounded.
  • Cost per call is estimated. Cloud cost comes from the runtime's published per-token rate at eval time. On-prem cost uses a GPU-second rate you supply with --gpu-cost. Neither is a contractual quote.
