Distillation approval queue.

An optional gate that holds a distillation run until a qualified owner or admin approves it. Useful for high-cost runs, sensitive namespaces, and regulated tiers.

When approvals are required

Approvals fire on a per-tenant policy. The policy is OFF by default; enable it on namespaces where one of the following applies:

  • High-cost runs. Estimated teacher spend above the namespace cost ceiling (configured at /account/policy).
  • Sensitive namespaces. Any namespace tagged with sensitive:true -- typically those handling PHI, PII, or regulated content.
  • Regulatory tier. Any namespace under the medical, legal, or financial vertical (the kolm artifact stamps disclaimers automatically for these; the approval gate gives a human one more checkpoint).

Queue lifecycle

StateMeaning
requestedA member or viewer has requested a distill run; awaiting submission to the approver pool.
pendingRequest is in the approver pool; visible to owners and admins.
grantedAn owner or admin approved the request; the distill pipeline picks it up on the next tick.
rejectedAn owner or admin rejected the request. A reason string is required; the request stays in the audit log.
expiredRequest sat in pending longer than the namespace timeout (default 72 hours). The requester may re-submit.

Roles and permissions

  • Owner. Can approve, reject, or revoke any request. Can change the namespace approval policy.
  • Admin. Can approve or reject requests. Cannot change the namespace approval policy.
  • Member. Can request a distill run. Cannot approve or reject.
  • Viewer. Can view the queue. Cannot request, approve, or reject.

CLI

kolm approvals list --namespace prod-medical --state pending
kolm approvals request --namespace prod-medical --reason "weekly retrain"
kolm approvals approve <request_id> --reason "spend within Q2 budget"
kolm approvals reject  <request_id> --reason "wait for HIPAA audit close"

API routes

The same surfaces are exposed as auth-gated routes; see /account/approvals for the dashboard.

  • POST /v1/approvals/request
  • POST /v1/approvals/:id/approve
  • POST /v1/approvals/:id/reject
  • GET /v1/approvals?namespace=&state=
Honest contract. Rejection requires a reason string -- the API returns HTTP 400 reason_required if reason is missing or empty. We never silently reject. The reason text is written to the audit log alongside the requester id, the approver id, and the request hash.
