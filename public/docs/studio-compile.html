Studio / Compile

One file. Every runtime. Signed at the bottom.

The compile wizard takes the adapter your distill run produced, merges it into the base, packs the tokenizer + chat template + spec.toml + passport + receipt chain, optionally quantizes, and seals the whole thing with an Ed25519 signature. The result is a single .kolm file you can ship to any of seven runtimes — or first export to GGUF, EXL2, GPTQ, AWQ, FP8, NVFP4, MLX.

What's inside a .kolm file

A .kolm is a zip with a strict layout (see spec/kolm-format-v1).

my-artifact.kolm/
├─ manifest.json          # name, base, method, sizes, hashes
├─ weights/                # safetensors shards (merged base+adapter)
├─ tokenizer/              # tokenizer.json + special tokens
├─ chat-template.jinja     # chat template (Jinja2)
├─ spec.toml               # the recipe that built this artifact
├─ passport.json           # provenance + K-Score
├─ receipts/               # ordered chain of receipts
│   ├─ 001-distill.json
│   ├─ 002-eval.json
│   └─ 003-compile.json
├─ signature.ed25519       # signs everything above
└─ verify.py               # 30-line self-verifier (no kolm CLI needed)

The compile wizard stages

  1. Validate — check adapter exists, base is locally cached, spec.toml parses, all referenced teachers/judge versions resolvable.
  2. Merge — peft merges the LoRA adapter into the base; output is safetensors shards in fp16.
  3. Tokenizer + template — copy + canonicalize tokenizer.json, render chat-template.jinja from the recipe.
  4. Passport — build passport.json from the receipt chain: model card, K-Score per axis, training cost, teacher council, hardware fingerprint, runtime memory estimate.
  5. Quantize (optional) — if --target gguf-q4km (or any quant target) is set, the merged fp16 is handed to the matching exporter. See studio-quantization.
  6. Sign — canonical-JSON the manifest + passport + receipts, sign with the keypair from ~/.kolm/keys. Public key is embedded in the manifest.
  7. Verify — the compile driver immediately re-runs verify on the produced file. If sign ≠ verify, the file is deleted before write.

CLI

# Compile the canonical recipe with default target (fp16 .kolm)
kolm compile --recipe support-trinity-500 --output trinity-support.kolm

# Compile + export to GGUF Q4_K_M in one step
kolm compile --recipe support-trinity-500 --target gguf-q4km --output trinity-support-q4km.gguf

# Compile from an existing adapter directory
kolm compile --adapter runs/trinity-support-v3/adapter --base Qwen/Qwen2.5-7B-Instruct \
             --spec trinity-support-500.toml --output trinity-support-v3.kolm

# Just re-verify an existing artifact
kolm verify trinity-support.kolm

# Inspect without unpacking
kolm inspect trinity-support.kolm --json

Supported compile targets

TargetOutputWhere it runs
kolm (default)fp16 .kolmAnywhere with the kolm runtime adapter.
gguf-q2km..q8.ggufllama.cpp, Ollama, LM Studio, kolm runtime.
exl2-3.0..8.0bpwEXL2 dirExLlamaV2 (best single-GPU throughput).
gptq-{3,4,8}bitGPTQ dirvLLM, AutoGPTQ.
awq-4bitAWQ dirvLLM, AutoAWQ.
fp8FP8 safetensorsHopper (H100/H200) + TensorRT-LLM, vLLM.
nvfp4NVFP4 safetensorsBlackwell (B100/B200/RTX 5090) + TensorRT-LLM.
hqq-{2,3,4,8}bitHQQ safetensorsHuggingFace + HQQ runtime (calibration-free).
mlx-{2,3,4,8}bitMLX dirApple Silicon native (mlx-lm).

See studio-export-formats for help picking.

Caveats. Compile is RAM-bound, not VRAM-bound — the merge step holds the full fp16 base in CPU RAM (e.g., 14 GB for a 7B). The wizard prints the RAM forecast in the validate stage; abort early if you don't have it. Use --low-ram-merge to merge per-shard at the cost of ~2× wall time.

Implementation

The compile driver lives in src/compile-spec.js. Each stage is a function that takes the previous stage's output and emits a receipt. The export backends live in src/export-gguf.js, src/export-ollama.js, plus the S-6 modules src/export-{exl2,gptq,awq,fp8,nvfp4,hqq,mlx}.js. The signing logic lives in src/sigstore-attest.js.
