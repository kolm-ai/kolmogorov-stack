Run & Govern / Reference

Ten questions a risk team will ask.

The questions a buyer's security and risk team raises before approving the artifact, with the literal answer the platform gives. Where a control is package-gated or certification-gated, that is named — not paved over.

1. How do I know a runtime passport row is real?

Every row carries a status field that is one of three values: tested, estimated, or unsupported. Tested rows require a measured_at timestamp plus measured tokens-per-second, p50 and p99 latencies, GPU memory used, and the concurrent-request count under which the measurement was taken. Estimated rows are computed from model parameter count and the quantization formula params * bytes_per_param; they never overwrite a tested row.

The full schema is at /docs/runtime-passport.

2. Can the evidence DAG be tampered with after the artifact is signed?

The DAG rides inside the artifact manifest. It is NOT bound into artifact_hash_input by design — the DAG is operational fingerprint, not the bytes themselves. So a re-prover can legitimately re-walk and re-emit the DAG (for example, adding a new validated_by edge after an eval re-run) without invalidating the receipt chain on the bytes.

What this means in practice: a signature on an artifact attests to its weights and recipes. The DAG is the provenance graph behind those bytes — tamper-evident through the signature on the artifact root, not through a separate DAG signature.

3. What happens when we revoke a capture? Does the whole graph re-sign?

No. revoke() emits a verdict object: {revoked: [capture_id], needs_review: [...descendants]}. The verdict is persisted as a new invalidates edge on the DAG (append-only) so the audit trail records what was revoked, when, and why. Downstream artifacts that depended on the revoked capture are not auto-deleted — they are flagged needs_review, and the operator decides whether to re-distill, re-train, or accept the residual risk.

See /docs/evidence-dag#revoke for the fan-out math.

4. Does the assurance case constitute SOC 2 certification?

No. An implemented status on SOC2 CC6.1 means the workspace has the control wired up and the vault row points at the supporting evidence. It does not mean a SOC 2 auditor has signed off on the implementation. The taxonomy has an explicit fourth value — certification-gated — that names the gating certification when one is in progress.

For a fully-certified SOC 2 Type II report, request the latest report from the trust center; the assurance case envelope cross-references the report id when one exists.

5. How does drift detection avoid false positives on noisy namespaces?

Each signal is graded against its in-window standard deviation, not against a global threshold. The baseline window is chunked into 6 equal-time slices and each slice's signal value is one sample for the sigma calculation. A steady namespace with low variance trips on smaller absolute deltas; a noisy namespace requires a larger excursion. The standard SPC ladder of 2-sigma warn and 3-sigma alert means false-positive rate stays roughly constant per namespace.

There is a floor on each sigma (2 percentage points for fallback rate, 2% for distribution distance, 10% for volume ratio) so an artificially steady baseline does not trip on micro-deltas.

6. The cost-displacement number says $1,678 saved this month. How do you know?

Every receipt carries a cost_usd field stamped by the router at decision time — that is the actual cost. For the counterfactual we walk each receipt and ask: would this call have gone to frontier if local routing did not exist? If route_decision is local, we re-price the call using the configured frontier model's per-token rate from PROVIDER_RATE_CARD (the same frozen rate table the kolm fee calculation uses). If route_decision is frontier or frontier_fallback, the call would have gone to frontier anyway — no displacement.

When a local row has no matching rate-card entry, the row is excluded and ok_status reports partial:N_unrepriced. We never invent a price.

7. What runs in the air-gap bundle? Is anything missing?

The bundle contains: the artifact byte-for-byte, a runtime installer that reads from a local wheelhouse (vLLM) or local binary (llama.cpp), the rendered runtime config, an offline kolm-audit-1 receipt verifier, and a sha256 manifest covering every entry. Plus a README that names exactly what is NOT in the bundle.

Not in the bundle: TLS certificates (you supply via your reverse proxy), Secrets like KOLM_ARTIFACT_SECRET and registry tokens (you inject via your secrets manager), and the base model weights (your runtime config points at your local model cache).

8. The lifecycle says "deployed". Does that mean the artifact is currently serving traffic?

The lifecycle records the canonical state of the artifact in your workspace; it is not a liveness probe. deployed means the artifact has been promoted past signed and is the namespace's current canonical choice. monitored means drift detection is enabled on its namespace. Whether a specific replica is currently answering requests is a runtime question — check the runtime sidecar at /metrics (per runtime autodetect / metrics).

9. We are in a regulated jurisdiction. Do you ever leak our captures across tenants?

Cross-tenant queries are structurally impossible. Every lake read (receipts, captures, observations) goes through store.findByTenant(table, tenant_id). The drift detector and cost-displacement modules never fall back to an all() read if findByTenant returns nothing — they return an empty envelope with ok_status: 'no_receipts' instead.

The tenant binding lives on the API key. Even an internal route handler bug cannot cause a cross-tenant read because the tenant id is the primary index on the storage tables and no code path in the Run & Govern surface reads without it.

10. Where do I file a security disclosure?

Email security@kolm.ai. We accept disclosures via signed email (PGP key at /security#pgp) and respond within one business day. Critical vulnerabilities are patched and disclosed via the changelog; non-critical findings ship in the next regular release.

Bug bounty: not currently formalized. We pay for clearly-reproducible findings that bypass auth, leak across tenants, or break the receipt chain.
