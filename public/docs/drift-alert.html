kolm.ai / docs / drift-alert

Open-source AI workbench

Frontier AI on your own infrastructure.

Drift alert compares the token distribution of your training captures against the token distribution of live production traffic. When the two diverge past a threshold, the route fires registered webhooks and stamps drift_warning:true onto the next routing decision — the small student model can then fall back to the teacher on novel queries. Same envelope is returned by POST /v1/drift-alert/snapshot, GET /v1/drift-alert/:namespace, and the kolm drift-alert CLI path.

Why distribution-shift matters more than accuracy

A distilled student that scored 0.94 K-Score on your training captures last month is now answering 15% more billing queries than it ever saw at training time. The K-Score on production is silently dropping — but you will not know until either a customer escalation lands or the next bake-off runs. Drift alert closes that gap: it compares the shape of the workload in real time, not the quality of any individual answer. When the shape moves, you find out the same day, not the same quarter.

The canonical failure mode looks like this. A support agent gets distilled on 2000 captures — mostly password resets and shipping lookups. Marketing pushes a campaign, and overnight 23% of inbound is now refund-policy questions, which were 4% of training. K-Score on the unchanged distilled model has not budged because the bake-off dataset is still the training mix. But every refund-policy answer is now coming from a head of the distribution the student barely saw. Drift alert catches the JSD crossing 0.15 within minutes of the campaign launch.

KL divergence + Jensen-Shannon divergence

Drift alert builds two sketches: P (training distribution, recorded once per snapshot via POST /v1/drift-alert/snapshot with kind:"training") and Q (production distribution, recorded periodically with kind:"production" or computed on-the-fly from the last N captures). Both sketches are top-K count tables over monograms, bigrams, and trigrams of the captured prompt + response text. A _other bucket holds the tail.

Comparison computes two divergence numbers:

QuantityFormulaRange
kl (KL divergence)sum_i P(i) * log(P(i) / Q(i))0 to ∞ (nats), asymmetric
jsd (Jensen-Shannon)0.5 * KL(P || M) + 0.5 * KL(Q || M) where M = 0.5*(P+Q)0 to ln(2) ≈ 0.693, symmetric

The alert threshold is on jsd — symmetric, bounded, and well-behaved at the endpoints. Default threshold is 0.15. Both numbers use additive Laplace smoothing with epsilon = 1e-6 on every probability before the log, so missing tokens do not produce -Infinity or NaN. The smoothing is small enough that it does not move the alert decision, but eliminates the divide-by-zero failure mode entirely.

Envelope shape

The same envelope is returned by GET /v1/drift-alert/:namespace and the kolm drift-alert check CLI path. Field stability is part of the contract.

CODE20

Plain absence: missing_snapshot

When the requested namespace has only a training snapshot (no production yet) or only a production snapshot (no training reference), the route returns HTTP 200 with { ok: false, error: "missing_snapshot", missing: "production", hint: "POST /v1/drift-alert/snapshot {kind:'production'}" } — not a 404. The HTTP call itself succeeded; the hint tells the caller exactly which side is missing and what to do next.

Webhook delivery

Register a webhook to be notified when jsd >= threshold. Each webhook URL has its own threshold; when multiple webhooks are registered for the same namespace, the lowest threshold wins — the most conservative caller sets the alert floor. A second registration of the same URL UPDATES the existing row (threshold change), it does not duplicate.

CODE23

Best-effort, no retries. Delivery uses a 5-second AbortController timeout and fires once. The route does not block the response on webhook completion — the per-webhook fetch runs in a fire-and-forget async IIFE. If your endpoint is down, the alert is dropped. The durable signal lives in the snapshot ledger; the webhook is the optional push channel.

routing-decision tie-in

When shouldAlert fires, drift-alert calls registerDriftWarning(tenant_id, namespace, {jsd}) — an in-memory Map keyed by tenant_id|namespace. The next time the router records a routing decision for that tenant + namespace via recordRoutingDecision in src/routing-events.js, it consumes the warning (one-shot semantics: read + clear) and stamps the row with:

Row fieldValue
drift_warningtrue
drift_warning_atISO timestamp the warning was registered
drift_warning_jsdthe JSD that crossed threshold

The pragmatic use: when you see drift_warning:true on a routing decision, escalate to the teacher (or to a more conservative student) for that one query. The signal does not re-fire until the next GET /v1/drift-alert/:namespace pulls a fresh compare that crosses threshold. It is intentionally a soft signal whose value decays quickly — the durable record is the snapshot ledger, the in-memory bridge is just the cross-module hop from "alert tripped" to "next decision sees it."

Plain fallback: if src/drift-alert-store.js fails to import (future refactor decouples the modules), the path silently continues without the warning. The warning is best-effort — never load-bearing.

Plain-English shift suggestions

When shouldAlert fires, the envelope includes a suggestions array — up to 3 actionable strings (configurable via top_n). Each suggestion follows a fixed template, so a downstream dashboard can pattern-match on it if needed:

Your student sees ${pct}% more "${token}" queries than trained. Capture ${rounded} more "${token}" examples.

The percentage is the relative lift from p_train to p_prod (rounded to nearest 1%). The capture count is derived from the absolute delta in probability mass: Math.round(delta * 1000) rounded to the nearest 50, with a floor of 50. The suggestions are sorted by absolute delta in probability mass — the most-shifted tokens come first.

How this differs from drift-detector

Both surfaces use the same KL/JSD math, but they answer different questions. drift-detector compares a baseline window against a current window within the live capture stream — it tells you "the last 24 hours look different from the prior week." drift-alert compares a long-lived training sketch against a live production sketch — it tells you "production looks different from what you trained on." Use for intra-day anomaly detection; use for "is my distilled model still fit for purpose."

Next steps

Run kolm drift-alert snapshot <ns> --kind training right after a fresh distill to anchor the baseline, then kolm drift-alert snapshot <ns> --kind production on a cron (hourly is plenty) to keep the production sketch current. Register a webhook with kolm drift-alert webhooks. Read diagnose for the per-category breakdown that drift-alert points you toward when production starts shifting.
