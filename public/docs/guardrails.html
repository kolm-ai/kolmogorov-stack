Docs / Guardrails

kolm.ai · w736 · guardrail compilation

Brand-safety rules that ship inside the artifact.

Declare rules at capture time in kolm.yaml. They bake into the .kolm manifest as hard constraints, not training signal. Runtime enforces them on every response; kolm verify --guardrails replays them against the artifact's example traces.

Why guardrails ship with the artifact

Most teams treat brand safety as a prompt-engineering problem: pin a system prompt, hope the model holds the line, retrain when it doesn't. That works until it doesn't. A sufficiently adversarial prompt, a model swap, or a hosted-provider rollout can quietly erode the policy because the policy was never a contract — it was a suggestion the model could ignore.

W736 puts the policy inside the artifact. manifest.guardrails is bound into artifact_hash via the same conditional-slot pattern the W460 attestation chain uses, so any post-build mutation of a rule breaks every signature down the chain. At runtime, /v1/chat/completions calls enforceGuardrails on the response before it reaches the wire — a block action returns HTTP 403 with a blocked_by_guardrail envelope, never the offending text. That's the distinction: training signal is soft, guardrails are hard.

kolm.yaml syntax

Add a top-level guardrails: array. Each entry has name, pattern, and action. Rules apply in order; the first block match terminates evaluation.

version: w732-v1

namespaces:
  - name: support-bot
    teacher: claude-sonnet-4-6
    min_captures: 1000

guardrails:
  - name: no-competitor-recommendations
    pattern: keyword:competitor-product-x
    action: block
  - name: warn-on-pricing-claims
    pattern: regex:\$\d+\.\d{2}
    action: warn
  - name: redact-ssn-shaped-strings
    pattern: regex:\d{3}-\d{2}-\d{4}
    action: rewrite
    replacement: '[REDACTED-SSN]'

quality_gates:
  min_kscore: 0.85

Absent guardrails is a no-op — the artifact ships without a runtime fence and stays byte-identical to a pre-W736 build. Empty guardrails: [] collapses to the same no-op state (W460 byte-stability rule).

Pattern types

Three forms, picked by prefix:

PrefixFormExample
keyword:Case-insensitive substring match. Special regex characters in the keyword are escaped, so keyword:price.list matches the literal string.keyword:competitor-x
glob:Shell-style glob. * matches any chars, ? matches a single char, [a-z] is a character class. Anchored to the whole response.glob:*price*$*
(none)Raw regex, compiled with the case-insensitive flag. Use this for credit-card / SSN / token patterns.\d{3}-\d{2}-\d{4}

Bad regex syntax is caught at kolm yaml validate time so a typo never reaches production. The error code is guardrail_pattern_invalid with the failing rule path.

Actions

  • block — refuse the response. Runtime returns HTTP 403 with {ok:false, error:'blocked_by_guardrail', rule_name, hint}. The offending text never leaves the proxy.
  • warn — pass through, but annotate the response envelope with kolm_guardrail_warnings: [{rule_name, action:'warn', matched_at}]. Useful for sampling rules or pre-launch tuning where you want to observe without blocking.
  • rewrite — substitute the matched span with replacement (default [redacted]). The response envelope carries kolm_guardrail_rewrites: [{rule_name, action:'rewrite', bytes_changed}] so the caller sees that a substitution happened.

Verify-time check

kolm verify --guardrails <artifact.kolm> replays every rule in manifest.guardrails against every entry in manifest.example_traces (or manifest.evals.cases as a fallback). The verdict downgrades to fail when any block rule fires against a real example trace — that means the rule contradicts a fixture the artifact was built from, which is almost always a build error.

$ kolm verify --guardrails ./support-bot.kolm
verdict: pass
cid:     bafy...

guardrails:
  ok:       true
  total:    24
  violations: 1
    - rule=warn-on-pricing-claims action=warn trace_idx=12

Warn-only violations are surfaced but do not flip the verdict — they are observational by design. CI gates that want to fail on any guardrail hit can grep the JSON output for "action":"warn" and exit non-zero locally.

CLI

  • kolm guardrails validate [--file kolm.yaml] — lints the guardrails block in the named file and prints the W736 validation envelope.
  • kolm guardrails test --response "<text>" --file kolm.yaml — runs enforceGuardrails against the supplied response so you can dry-run a rule against a fixture string before re-baking the artifact.

HTTP API

Two header / body forms feed the runtime fence. Both produce identical enforcement; pick whichever your SDK exposes more cleanly.

  • Header: x-kolm-guardrails: [{"name":"no-x","pattern":"keyword:x","action":"block"}]
  • Body: include kolm_guardrails: [...] alongside messages in the POST body.

When the W736 module is missing on a deploy (older stacks), the response header x-kolm-guardrails-skipped: module_missing surfaces the honest fall-through — no silent enforcement-bypass.
