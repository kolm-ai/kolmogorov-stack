Capture anomaly detection.

A poisoned or drifted capture can quietly corrupt your distill corpus. kolm.ai runs a per-tenant per-namespace 3-sigma statistical detector across four signal channels, holds outliers in quarantine, requires a review action before promotion, and back-stops the whole loop with a post-distillation regression gate.

Statistical detector

For every incoming capture, the gateway maintains a running mean and standard deviation per (tenant, namespace) across four signals:

  • output_length -- token count of the model response.
  • vocab_entropy -- Shannon entropy of the response token distribution.
  • response_time -- end-to-end teacher latency in milliseconds.
  • token_overlap -- Jaccard overlap of the response against the last K=20 captures in the same namespace.

State is held in a per-namespace ring of the last N=10,000 observations. Statistics update in O(1) using Welford's online algorithm.

3-sigma threshold

Any capture whose value on any channel exceeds three standard deviations from the running mean is flagged. The default threshold is 3.0; tenants on Pro and Enterprise tiers can dial this between 2.0 (paranoid) and 4.0 (permissive) per namespace.

z = abs((value - mean) / stddev)
if z > 3.0: stage_for_review

Staging and quarantine flow

A flagged capture lands in the staged_captures table with status quarantined. The capture is held outside the distill corpus for a 24-hour quarantine window. During that window:

  • The capture is not visible to kolm distill or to any read of /v1/captures with status filter clean.
  • A reviewer can allow, block, or escalate the row from /account/captures/review.
  • If no review action is taken in 24 hours, the row is auto-blocked and aged out to cold storage. The default behaviour is fail-closed.

Cryptographic origin binding

Every capture stores a origin_binding field computed as sha256(teacher_request_headers || teacher_response_body_sample). This binds the row to the specific upstream call that produced it. If a row appears in the lake whose origin binding does not match any recorded teacher call for the namespace, it is auto-quarantined regardless of statistical signal. This catches both injection attacks and accidental cross-namespace bleed.

Manual review queue

The review UI at /account/captures/review presents each quarantined capture with:

  • The four 3-sigma z-scores side by side.
  • A side-by-side diff against the median response in the namespace.
  • One-click allow, block, and escalate buttons.
  • An allow with annotation path that lets you mark a row as known-anomalous (e.g. a new product line) so future similar rows are auto-allowed.

Post-distillation regression gate

Even if a poisoned row slips past the per-capture detector, the distill loop has a final gate. After every distill run, the new artifact is bake-off-scored against the previous artifact on the namespace eval pack. If the new K-Score is more than 0.02 lower than the previous, the distill auto-rolls back to the previous artifact and the run is flagged for human review. The threshold is configurable per namespace.

kolm distill --namespace prod --gate-threshold 0.02
# pre:  K-Score 0.871
# post: K-Score 0.844  (delta -0.027, gate failed)
# rollback to previous artifact: art_a1b2c3d4
# flagged: capture-poisoning-suspected

CLI

kolm capture status --namespace prod
kolm capture review --namespace prod
kolm capture allow <capture_id>
kolm capture block <capture_id>
kolm capture escalate <capture_id>
