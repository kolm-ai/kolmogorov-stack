Open-source AI workbench

Federated consortium.

Multiple tenants pool gradients on a shared training target without crossing raw-data boundaries. Opt-in / opt-out membership, per-consortium privacy budget in epsilon, membership-inference attack verification on the shipped artifact. Hash-only approval sharing keeps the prompt text on the source tenant.

How it works

A consortium is a named set of opted-in tenants who agree on a shared training target. Each tenant runs the training on its own captures; only the gradients (plus the optional Laplace-noised approval counts) are aggregated.

Membership is per-tenant and per-consortium_id. A tenant calls POST /v1/federated/consortium/opt-in with a consortium_id and an optional scope (which namespaces are eligible to contribute). The member row records opted_in_at, contribution_count, and last_share_at so the consortium can spot dormant members and "who is pulling their weight."

Aggregation rounds consume the consortium-wide privacy budget. Every round is logged with its epsilon spend and the participant list; an aggregation that pushes the consortium over its allocated budget is rejected at the aggregator before raw values are touched.

Opt-in / opt-out

Membership is explicit. A tenant must POST to opt in; opt-out is symmetric. Both routes are tenant-fenced via req.tenant_record.id; the body's consortium_id cannot impersonate another tenant.

POST /v1/federated/consortium/opt-in

curl -X POST https://kolm.ai/v1/federated/consortium/opt-in \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "consortium_id": "support-bots-2026",
    "scope": ["support-bot", "billing-bot"],
    "epsilon_allocated": 10.0,
    "note": "joining for the Q2 cohort"
  }'

# -> {
#   "ok": true,
#   "consortium_id": "support-bots-2026",
#   "member": { "tenant_id":"tenant_...", "scope":[...], "opted_in_at":"...", ... },
#   "epsilon_allocated": 10.0
# }

Default epsilon allocation is 10.0; pass any positive number to override (only the first opt-in for a consortium can set the allocation, subsequent opt-ins read whatever was set). The scope is optional; an empty scope means "any of my namespaces can contribute when I run rounds."

POST /v1/federated/consortium/opt-out

curl -X POST https://kolm.ai/v1/federated/consortium/opt-out \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "consortium_id": "support-bots-2026", "reason": "ending Q2 participation" }'

Privacy budget

Differential privacy is enforced as a budget. Every aggregation round spends some epsilon; the consortium has an allocated cap. The budget endpoint returns the totals plus the caller's personal contribution.

GET /v1/federated/consortium/budget

curl -H "Authorization: Bearer $KOLM_API_KEY" \
  "https://kolm.ai/v1/federated/consortium/budget?consortium_id=support-bots-2026"

# -> {
#   "ok": true,
#   "consortium_id": "support-bots-2026",
#   "epsilon_allocated": 10.0,
#   "epsilon_spent": 3.7,
#   "epsilon_spent_by_self": 1.2,
#   "epsilon_remaining": 6.3,
#   "pct_spent": 0.37,
#   "n_aggregations": 4
# }

The epsilon_spent_by_self field is computed by intersecting the round participant lists with the caller's tenant_id. A consortium that wants to enforce a per-tenant cap reads this field and refuses local round participation when the personal spend crosses a threshold.

Members and aggregations

Two read endpoints expose the consortium state: the member roster and the recent aggregation runs.

GET /v1/federated/consortium/members

curl -H "Authorization: Bearer $KOLM_API_KEY" \
  "https://kolm.ai/v1/federated/consortium/members?consortium_id=support-bots-2026"

# -> { "ok":true, "consortium_id":"support-bots-2026", "total":4, "members":[
#      { "tenant_id":"tenant_a...", "scope":["support-bot"], "opted_in_at":"2026-04-12T...",
#        "contribution_count":12, "last_share_at":"2026-05-23T...", "is_self":true },
#      ...
#   ] }

GET /v1/federated/consortium/aggregations

curl -H "Authorization: Bearer $KOLM_API_KEY" \
  "https://kolm.ai/v1/federated/consortium/aggregations?consortium_id=support-bots-2026&limit=20"

# -> { "ok":true, "consortium_id":"...", "total":4, "aggregations":[
#      { "aggregation_id":"agg_...", "round_id":"round_...", "status":"completed",
#        "started_at":"...", "completed_at":"...", "epsilon_spent":0.9,
#        "n_participants":3, "participant_in_round":true },
#      ...
#   ] }

Defense in depth: a tenant only sees aggregation rows where they were a participant OR they are a current member. A foreign tenant who guesses the consortium_id still cannot read other tenants' contributions.

Membership-inference verification

A shipped artifact should resist the "was this row in your training set" attack. The W830-2 verifier scores an attacker's AUC against a configured threshold (default 0.55, near-chance) and returns passing or leaking.

POST /v1/federated/consortium/verify-mia

curl -X POST https://kolm.ai/v1/federated/consortium/verify-mia \
  -H "Authorization: Bearer $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "artifact_id":"art_...", "test_inputs":[...], "shadow_models":[],
        "p_threshold": 0.55 }'

# When shadow_models is empty (the honest HTTP case):
# -> { "ok":false, "error":"mia_requires_shadow_models",
#      "install_hint":"Provide >=3 shadow models trained on disjoint shards" }

Honest scope: this module is the protocol contract. The actual shadow-model attack runs on the operator side because callable models cannot cross the JSON wire. When the route returns the missing-shadow-models envelope, the SDK runs the attack locally and re-posts the AUC for verdict scoring.

The verdict envelope tags the audit method: a 'passing' verdict bounds the attacker AUC observed in YOUR test setup; it does not bound a more sophisticated adversary. The verdict is honest within its scope and the scope is stamped.
