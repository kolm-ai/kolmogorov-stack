Gateway self-host

Self-host in 10 minutes with Docker Compose.

Three containers: gateway, Postgres, MinIO. One docker compose up -d. The same image runs in Kubernetes via the published Helm chart.

When to self-host

  • Regulated data that cannot leave your VPC.
  • Latency-sensitive workloads where the hosted gateway adds a round-trip.
  • Air-gapped deployments (artifact + image work fully offline).
  • Multi-tenant SaaS where you want one gateway per customer namespace.

Docker Compose layout

The published compose file ships three services. gateway runs the proxy; postgres stores captures and routing decisions; minio provides S3-compatible storage for long-term capture archives.

# docker-compose.yml
services:
  gateway:
    image: ghcr.io/kolm-ai/kolm-gateway:latest
    ports:
      - "8080:8080"
    environment:
      KOLM_GATEWAY_CONFIG: /etc/kolm/gateway.toml
    volumes:
      - ./gateway.toml:/etc/kolm/gateway.toml:ro
      - ./signing.ed25519:/etc/kolm/signing.ed25519:ro
    depends_on: [postgres, minio]
    restart: unless-stopped

  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: kolm
      POSTGRES_PASSWORD: change_me
      POSTGRES_DB: kolm
    volumes:
      - pg-data:/var/lib/postgresql/data
    restart: unless-stopped

  minio:
    image: minio/minio:latest
    command: server /data --console-address ":9001"
    environment:
      MINIO_ROOT_USER: kolm
      MINIO_ROOT_PASSWORD: change_me_too
    volumes:
      - minio-data:/data
    ports:
      - "9000:9000"
      - "9001:9001"
    restart: unless-stopped

volumes:
  pg-data:
  minio-data:

Boot sequence

  1. Generate a signing key: kolm key generate > signing.ed25519 (chmod 0400).
  2. Write a gateway.toml with the provider blocks you need (see configuration).
  3. Run docker compose up -d.
  4. Apply the Postgres schema: docker compose exec gateway kolm db migrate.
  5. Verify the health endpoint: curl -s http://localhost:8080/v1/health | jq ..
  6. Point a test SDK at http://localhost:8080/v1/wrap/openai; fire a call; check kolm captures list.

Production hardening

  • TLS termination — run nginx, Caddy, or your cloud load balancer in front of the gateway. Do not expose port 8080 directly.
  • Postgres backups — the capture lake is the audit trail; pg_dump nightly into the MinIO bucket.
  • Resource limits — set deploy.resources.limits in compose or use a process manager. The gateway uses ~80 MB RSS idle plus ~2 KB per in-flight request.
  • Log shipping — structured JSON logs to stderr; ship via your existing collector (Vector, Fluent Bit, Loki).
  • Key rotation — kolm key rotate generates a new signing key and keeps the old public key valid for 30 days for verifier compatibility.

Kubernetes via Helm

The same image runs in Kubernetes with the published chart. The chart includes a StatefulSet for Postgres, a Deployment for the gateway (with HPA on request rate), and a Service of type ClusterIP. Pair with cert-manager for TLS and external-dns for DNS.

helm repo add kolm https://kolm-ai.github.io/kolmogorov-stack/charts
helm repo update

helm install gateway kolm/gateway \
  --namespace kolm --create-namespace \
  --set image.tag=latest \
  --set config.signingKeySecret=kolm-signing \
  --set ingress.host=gateway.internal.example.com \
  --set postgres.enabled=true \
  --set minio.enabled=true

Air-gapped install

Pull the image once on a connected host, save it as a tar, transfer to the air-gapped network, load with docker load. All provider blocks should target on-network endpoints (typically local-vllm, local-ollama, or local-kolm). The gateway has zero outbound calls when no hosted providers are configured.

Caveats. The image and Helm chart are tagged by version; latest is fine for dev but pin to a specific tag for production. The first install requires the Postgres schema migration; subsequent restarts do not. Hosted (kolm.ai) and self-hosted deployments share the same capture schema, so you can migrate either direction by exporting / importing captures.
