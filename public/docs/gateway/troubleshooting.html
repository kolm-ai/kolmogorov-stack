Wrapper troubleshooting

Diagnose, fix, re-verify.

Every error from the gateway carries a stable error code and a remediation pointer. The most common four are listed below with the exact command to surface the root cause.

401 unauthorized

The gateway rejected your X-Kolm-Key or your upstream provider rejected the passthrough credential. The error body distinguishes which side failed.

  • kolm_auth_failed — your kolm key is wrong. Run kolm whoami; if it says logged_in: false, re-run kolm login or paste a fresh key.
  • upstream_auth_failed — your provider key (OpenAI / Anthropic / Gemini) is wrong. The body includes upstream_status and upstream_body verbatim.
  • tenant_disabled — the tenant is suspended (billing or abuse). Contact support.
{
  "error": {
    "code": "kolm_auth_failed",
    "message": "X-Kolm-Key invalid or revoked",
    "doc": "https://kolm.ai/docs/gateway/troubleshooting#401-unauthorized"
  }
}

429 rate limit

Two sources: the gateway’s own per-namespace quota, or the upstream provider’s rate limit. Headers distinguish them.

# Check current usage
kolm usage --namespace default --since 1d
# Free-tier monthly remaining
kolm whoami --json | jq .free_tier
HeaderSourceAction
X-Kolm-Quota-ResetGateway namespace quota hitWait until midnight UTC or raise quota_captures_per_day.
Retry-AfterUpstream provider rate limitBackoff per the header; fallback chain will try the next provider if configured.
X-Kolm-Free-Tier-Remaining: 0Hosted free-tier monthly cap hitUpgrade or wait for monthly reset.

502 upstream timeout / failure

The upstream provider did not respond within [provider] timeout, or returned 5xx, or the TCP connection failed. If a fallback chain is configured the gateway will have tried it; the 502 you see is after the entire chain exhausted. The receipt records every attempt with fallback_reason.

  • Raise [provider] timeout if your prompts genuinely take longer than the default.
  • Add a closer-region fallback (e.g. self-hosted vLLM) if hosted providers are flaky in your region.
  • Check provider status pages — the gateway does not retry status-page outages.
# Inspect what was tried
kolm captures get <receipt_cid> --json | jq '.attempts[]'
# [
#   {"provider":"openai","model":"gpt-4o","status":504,"latency_ms":30021,"reason":"timeout"},
#   {"provider":"anthropic","model":"claude-sonnet-4-5","status":502,"latency_ms":1240,"reason":"upstream_5xx"},
#   {"provider":"local-vllm","model":"qwen-3.6-7b","status":200,"latency_ms":840,"reason":"ok"}
# ]

503 capture write failure

The pipeline finished but stage 9 (capture) could not persist the receipt. The response is still returned to the caller; the 503 surfaces only on the next request if the capture backend stays unhealthy. Default behavior is non-blocking capture ([capture] async = true), so capture failures do not block traffic.

# Backend-specific checks
# JSONL: disk full?
df -h /var/kolm

# SQLite: locked?
sqlite3 /var/kolm/captures.db "PRAGMA integrity_check;"

# Postgres: connection?
psql $CONNECTION_URL -c "SELECT 1;"

# S3: credentials?
aws s3 ls s3://your-capture-bucket --endpoint-url $MINIO_URL
Caveats. When capture is non-blocking and the backend is down, receipts are buffered in memory up to [capture] max_buffer = 10000. Beyond that, new requests get HTTP 503 with capture_buffer_full. Set async = false if you require capture durability before responding (adds ~3 ms p50).

Diagnostic command

The CLI ships a one-shot diagnostic that probes every dependency.

kolm doctor --gateway
# checks:
#   [OK]  gateway /v1/health reachable
#   [OK]  signing key loaded, key_id=tenant_acme_v3
#   [OK]  capture backend (postgres) reachable, rows=1248301
#   [OK]  provider openai reachable, p95=420ms
#   [WARN] provider anthropic returned 429 in last 5m (12 events)
#   [OK]  telemetry exporter (otlp) reachable

Where to file a bug

Include the receipt CID (visible in X-Kolm-Receipt-CID) and the output of kolm doctor --gateway --json. Open an issue at github.com/kolm-ai/kolm/issues.
