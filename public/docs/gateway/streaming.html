Wrapper streaming

SSE streaming, normalized across providers.

Set stream: true in the request body and the gateway forwards chunks as they arrive. Receipt and capture are written when the final chunk lands; latency to first token is identical to direct upstream.

How chunks flow

The gateway reads the upstream SSE stream and re-emits each data: line to the caller. No buffering — chunks reach the caller as soon as TCP delivers them. The pipeline stages 7 (output PII scan), 8 (sign), 9 (capture), and 10 (meter) run after the final chunk; they do not delay the stream.

request → [auth, namespace, input-PII, route, forward]
          ↓
     upstream SSE stream
          ↓
   chunks forwarded verbatim → caller
          ↓ (final chunk)
     [output-PII, sign, capture, meter, telemetry]

Provider chunk shape normalization

Different providers emit different chunk shapes. The wrap endpoint passes through each provider’s native chunks unchanged. Use the route endpoint (/v1/route/chat/completions) if you want a single normalized shape across all providers.

Streaming a request

curl -N -s https://kolm.ai/v1/wrap/openai/chat/completions \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "X-Kolm-Key: $KOLM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o-mini",
    "stream": true,
    "messages": [{"role":"user","content":"Count to five."}]
  }'

# data: {"id":"chatcmpl-...","choices":[{"delta":{"content":"One"}}]}
# data: {"id":"chatcmpl-...","choices":[{"delta":{"content":", two"}}]}
# ...
# data: [DONE]
#
# After [DONE], inspect response headers:
#   X-Kolm-Receipt-CID: bafy2bzace...
#   X-Kolm-Tokens-Out: 23

Receipt timing

The receipt is signed and the capture is written after the final chunk. The X-Kolm-Receipt-CID header arrives at the start of the response (trailer-style is not used; the CID is reserved at request time and the receipt body is filled in on completion). Polling GET /v1/captures/<cid> returns 425 (Too Early) until the capture flushes; then 200.

Failure during stream

If the upstream connection drops mid-stream, the gateway closes the downstream connection at the same byte boundary and writes a partial-capture receipt with terminated_early: true and the bytes that did arrive. The fallback chain does NOT activate mid-stream — it only fires on pre-first-byte failures.

Caveats. Mid-stream failures cannot be retried transparently because the caller has already received bytes. Applications should idempotently re-issue the request on terminated_early receipts. The route endpoint optionally buffers the entire response before forwarding (X-Kolm-Buffer-Stream: true) which restores fallback at the cost of time-to-first-token.

Tool / function-calling streams

Tool-call deltas are forwarded unchanged. The capture stores the full reassembled tool-call payload on the receipt; intermediate deltas are not stored individually (the capture is per-call, not per-chunk).
