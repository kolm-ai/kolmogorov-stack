Gateway

A signing gateway in front of every LLM call.

The gateway is a thin reverse-proxy that turns every chat-completion request into a signed, captured, metered event. Eleven providers behind one OpenAI-compatible interface. No vendor lock, no SDK rewrite.

What it does

Drop the gateway between your application and any LLM provider. Point your existing OpenAI / Anthropic SDK at https://kolm.ai/v1/wrap/<provider> (or a self-hosted endpoint) and every call becomes a row in your capture lake with an Ed25519-signed receipt.

Captures feed three downstream loops: distillation (turn high-volume prompts into a local .kolm specialist), routing (send the easy 80% to the cheap path), and audit (prove what the model saw and emitted, for any compliance regime).

The 11-stage pipeline

Every request walks the same deterministic pipeline. Each stage is a separate function with a documented contract; you can disable any stage that does not apply (telemetry, capture, redaction modes) but the order is fixed.

#StagePurpose
1authValidate the inbound API key or signed JWT, attach tenant identity.
2namespaceResolve the namespace (header, path, or default) for routing and quota.
3input-PII-scanDetect emails, phone numbers, SSNs, API tokens, credit cards in the prompt.
4routePick provider + model based on namespace rules, confidence threshold, or pin.
5forwardTranslate request to provider native shape, attach upstream credentials.
6receiveBuffer or stream the response, normalize chunk shape if SSE.
7output-PII-scanDetect leaked secrets or contaminated text on the response side.
8signCompute kolm-audit-1 receipt, sign with Ed25519 tenant key.
9captureAppend to the capture lake (JSONL / SQLite / Postgres / S3).
10meterRecord tokens + USD against the tenant + namespace bucket.
11telemetryEmit OpenTelemetry spans with the receipt id as trace attribute.

Who it is for

  • Teams already paying $5k+/mo to a single LLM vendor who want to see, attribute, and route that spend.
  • Regulated workloads (health, finance, legal) that need a tamper-evident audit trail for every model output.
  • Distillation candidates — if you have one workflow that fires the same prompt shape 50,000 times a month, the captures are the seed dataset for a local specialist.
  • Multi-provider shops who want one billing dashboard and one failover policy across OpenAI, Anthropic, Google, and self-hosted vLLM.

What it is not

The gateway is not a model. It does not host inference unless you point a route at local-vLLM, local-Ollama, or a .kolm artifact. It is not a prompt cache; it forwards every request to the routed provider. It does not modify prompts beyond optional PII redaction in the captured copy.

Performance overhead

Target p50 latency overhead is under 5 ms when running co-located with your application (same VPC, same region as the upstream provider). Signing is a single Ed25519 operation (~50 us). Capture writes are append-only and buffered; the response is released to the caller before the capture is durable.

Caveats. The 5 ms target assumes co-location. If you route through the hosted kolm.ai endpoint from a different region, add the round-trip. Self-host (Docker Compose or Kubernetes) for production workloads where overhead matters.
