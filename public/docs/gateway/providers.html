Wrapper providers

Eleven providers, one wrap endpoint.

Every provider accepts its native request shape on the wrap URL. The gateway translates only what the provider requires; nothing else is rewritten. Your existing SDK keeps working.

Full provider table

The wrap path is https://kolm.ai/v1/wrap/<provider>/<native_path>. The upstream credential is passed through in the header the provider expects. The X-Kolm-Key header authenticates to kolm itself.

Hosted vs local distinction

The first eight (openai through openrouter) are hosted SaaS providers; the gateway forwards your credentials and bills against your account at each provider. The three local-* slugs forward to inference you control: a vLLM cluster, an Ollama daemon, or a .kolm artifact mounted into the gateway container. Local providers cost zero per-call USD (the meter records GPU-seconds instead).

Request shape compatibility

The gateway does not normalize request shapes across providers. An OpenAI chat/completions body sent to /v1/wrap/anthropic will be rejected by Anthropic with a 400. If you want one request shape that fans out, use the /v1/route/chat/completions endpoint instead — that one accepts the OpenAI shape and translates to whichever provider the route picks.

local-kolm: serving compiled artifacts

Point the gateway at a .kolm artifact path and it serves the artifact via the bundled llama.cpp / vLLM backend depending on the artifact format. The slug appears in receipts as local-kolm/<artifact_id> so routing decisions and cost attribution work uniformly.

# Provider block for a compiled artifact
[[provider]]
name        = "local-kolm-support"
base_url    = "file:///var/kolm/artifacts/support-v3.kolm"
auth_scheme = "none"
backend     = "llama-cpp"           # or "vllm"
gpu_layers  = 35                    # llama.cpp offload count

Adding a custom provider

Any OpenAI-compatible HTTP endpoint works as a custom provider. Declare a [[provider]] block with the base URL and pick the closest auth_scheme (bearer_passthrough, x_api_key_passthrough, or none). The slug you choose becomes the wrap path.

Caveats. The wrap path is verbatim forwarded after the provider segment. Per-provider request limits (token caps, image size, tool count) still apply — the gateway does not inflate them. If a 400 surprises you, replay the request directly against the provider native URL and compare; the gateway does not modify request bodies.
