Run & Govern / Runtime passport

Every artifact carries a capacity row per runtime.

The runtime passport is the contract that pins what an artifact actually does on a given runtime: how much memory it uses, how fast it decodes, and how its quality compares to the reference. Tested rows require real measurements; estimated rows are clearly labelled; unsupported rows refuse to lie with a 0 ms latency.

The shape

Schema version kolm-runtime-passport-1. Every .kolm manifest carries a runtime_passports: [] array of these objects, one per (runtime, target_id) combination the artifact supports:

FieldWhat it is
target_idString, required. Stable identifier like gguf-q4_k_m-llama.cpp or mlx-fp16.
statusOne of tested, estimated, unsupported. Controls what numeric fields are required.
runtimeOne of llama.cpp, vllm, ollama, mlx, transformers, tensorrt-llm, sglang, tgi.
runtime_versionFree-form string. Example: b3415, 0.6.4, mlx-lm 0.20.0.
precisionString. Example: fp16, bf16, q4_k_m, iq4_xs.
memory_mbWorking-set memory. Required for tested and estimated.
latency_p50_msPer-token decode latency, 50th percentile.
latency_p95_msPer-token decode latency, 95th percentile.
tok_sSustained throughput, tokens/sec.
quality_deltaEval-score delta vs the reference run. 0.0 = parity; negative = worse.
fallbackString target_id of the recommended fallback, or null.

Status rules

The status field is enforced by validatePassport in src/runtime-passport.js:

  • tested — every numeric field MUST be a real finite non-negative measurement. The pipeline must have actually loaded the artifact, run a probe, and pulled the numbers from the runtime's own counters. No interpolation, no scaling, no inference from sibling runs.
  • estimated — same shape, but the numbers come from a compile-time model (param count, precision, runtime constant). memory_mb is required; latency_*, tok_s, and quality_delta may be null. UI surfaces a status pill so a buyer never confuses an estimate for a measurement.
  • unsupported — the (runtime, precision) combination is incompatible with this artifact. Every numeric field MUST be null. The fallback field names the recommended alternative.

Example row

{
  "target_id":        "gguf-q4_k_m-llama.cpp",
  "status":           "tested",
  "runtime":          "llama.cpp",
  "runtime_version":  "b3415",
  "precision":        "q4_k_m",
  "memory_mb":        5232,
  "latency_p50_ms":   28.4,
  "latency_p95_ms":   41.7,
  "tok_s":            41.2,
  "quality_delta":    -0.014,
  "fallback":         "gguf-q5_k_m-llama.cpp"
}

The compile-time estimator

When ExportForge cannot hand off to the target runtime to take a measurement, it calls estimatePassport(spec) so the manifest always carries a row. The estimator multiplies params_b by a precision-specific bytes-per-param table and adds a 256 MB baseline for tokenizer + scheduler + small caches.

import { estimatePassport } from 'kolmogorov-stack/src/runtime-passport.js';

const row = estimatePassport({
  target_id:       'gguf-q4_k_m-llama.cpp',
  runtime:         'llama.cpp',
  runtime_version: 'b3415',
  precision:       'q4_k_m',
  params_b:        7.0,
});
// status: 'estimated', memory_mb: ~4252

If the estimator cannot resolve memory_mb (unknown precision or missing params_b), the row is promoted to unsupported rather than shipping a passport that fails its own validator.

Recording a tested row

recordTestedPassport(measurements) is what the post-export probe calls after loading the bytes into the target runtime and pulling actual numbers. It throws if you try to ship a tested row with missing measurements — the failure mode the schema exists to prevent.

import { recordTestedPassport } from 'kolmogorov-stack/src/runtime-passport.js';

const row = recordTestedPassport({
  target_id:       'mlx-fp16',
  runtime:         'mlx',
  runtime_version: 'mlx-lm 0.20.0',
  precision:       'fp16',
  memory_mb:       14_800,
  latency_p50_ms:  18.2,
  latency_p95_ms:  26.4,
  tok_s:           54.7,
  quality_delta:   0.003,
});
// throws if any of those numeric fields is missing or non-finite

Reading a passport from a built artifact

kolm inspect ./artifact.kolm --runtime-passport --json
# {
#   "spec": "kolm-runtime-passport-1",
#   "rows": [
#     { "target_id": "gguf-q4_k_m-llama.cpp", "status": "tested", ... },
#     { "target_id": "mlx-fp16",              "status": "estimated", ... },
#     { "target_id": "fp8-tensorrt-llm",      "status": "unsupported", "fallback": "fp16-vllm" }
#   ]
# }
Limitations. The estimator's bytes-per-param table is derived from public docs (llama.cpp quant ladder, MLX precision table, vLLM weights-only stats). It is intentionally conservative and lands within ~10% of the actually-loaded working set; tighter numbers come from a real probe.
