Open-source AI workbench

Kubernetes deployment.

The kolm runtime ships with a reference Helm chart, a readiness endpoint that waits for the artifact to warm, a Prometheus exporter, an HPA template keyed on inference-queue depth, an init container that pulls the .kolm artifact from your registry, and a rolling-update annotation that supports zero-downtime artifact swap. This page is the operator's playbook; the chart is the source of truth.

Helm chart location

The reference chart lives in the kolm repository at tools/helm/kolm/. Clone the repo or vendor the chart into your platform repo; helm registry hosting is on the roadmap but not live today.

git clone https://github.com/kolm-ai/kolmogorov-stack
cd kolmogorov-stack/tools/helm/kolm
helm install kolm . --namespace kolm-runtime --create-namespace --values my-values.yaml

The chart is intentionally small. It ships a Deployment, a Service, a ConfigMap for the runtime config, an HPA template, a ServiceMonitor for Prometheus Operator (optional, behind a feature flag), and the init container that fetches the artifact. No CRDs, no operators, no custom controllers; the chart targets standard upstream Kubernetes.

/ready endpoint semantics

The runtime exposes a /ready endpoint that returns 200 only when the artifact is loaded AND warmed. Use it as the Kubernetes readiness probe.

The endpoint returns 503 in three cases: the artifact has not been loaded yet (init container still running or failed); the artifact loaded but warm-up is still in progress (first 30 seconds typically); the runtime detected a fatal error during load (signature mismatch, format version unsupported, weights checksum mismatch).

The endpoint returns 200 only after the artifact is fully resident, the warm-up forward pass completed, and the runtime accepted at least one self-test request. A 200 means the pod is ready to serve real traffic; a 503 means it is NOT.

The distinction matters for rolling updates: if the readiness probe returns 200 before warm-up completes, the first real traffic to the new pod hits cold weights and pays the cold-start tax. The runtime's gating on warm-up is the right default.

/metrics Prometheus exporter

The runtime exposes a Prometheus-compatible /metrics endpoint on the same port as the inference API. Scrape with a standard Prometheus job or the Prometheus Operator's ServiceMonitor.

The chart includes an optional ServiceMonitor template enabled by setting serviceMonitor.enabled=true in values.yaml. The default scrape interval is 30 seconds; the runtime tolerates lower intervals down to 5 seconds without performance degradation.

For the full list of exported metric families, see the observability page at /docs/observability. The kolm.ai/runtime metric prefix is stable; new families MAY be added in minor releases, existing families WILL NOT be removed without a major release and a deprecation cycle.

HPA config example

The reference HPA scales on a custom metric (inference-queue-depth) rather than CPU. Inference latency is dominated by queue wait time, not by per-request CPU; scaling on queue depth gets you the right number of replicas faster.

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: kolm-runtime
  namespace: kolm-runtime
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: kolm-runtime
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Pods
      pods:
        metric:
          name: inference_queue_depth
        target:
          type: AverageValue
          averageValue: "5"

The custom metric requires a metrics adapter (typically prometheus-adapter or Datadog's custom-metrics adapter) wired to the kolm_inference_queue_depth Prometheus metric. The chart includes an annotated example in tools/helm/kolm/examples/prometheus-adapter.yaml.

Init container example

The init container pulls the .kolm artifact from your registry into an emptyDir volume that the runtime container then loads. Decoupling pull from load means a slow registry does not block the runtime once the artifact is local.

initContainers:
  - name: artifact-pull
    image: ghcr.io/kolm-ai/kolm-puller:v1
    args:
      - pull
      - --registry=$(KOLM_REGISTRY)
      - --artifact=$(KOLM_ARTIFACT_ID)
      - --out=/artifacts/current.kolm
    env:
      - name: KOLM_REGISTRY
        value: registry.acme.internal/kolm
      - name: KOLM_ARTIFACT_ID
        value: prod-2026-05-24
    volumeMounts:
      - name: artifacts
        mountPath: /artifacts
volumes:
  - name: artifacts
    emptyDir: {}

The puller image verifies the artifact signature before writing it to the emptyDir. A signature mismatch causes the init container to exit non-zero; the pod never reaches the runtime stage. This is the supply-chain guard at deploy time.

Rolling update annotation

The runtime supports zero-downtime artifact swap when the Deployment carries the annotation kolm.ai/zero-downtime-swap: "true". The annotation changes the rolling update behavior to keep the old artifact serving until the new artifact has warmed on the replacement pod.

With the annotation set, the Deployment controller waits for the new pod's /ready to return 200 (which includes warm-up) before terminating the old pod. Without the annotation, the default Kubernetes rolling update may terminate the old pod as soon as the new pod is Ready, which can include the cold-start window if your readiness probe is too eager.

The annotation also enables the runtime's connection-drain behavior on SIGTERM: in-flight requests complete (up to terminationGracePeriodSeconds) before the pod exits. The default grace period in the chart is 60 seconds, which is comfortable for most response sizes.

Sample values.yaml

A minimal values.yaml that exercises the readiness probe, the HPA, the init container, and the zero-downtime annotation.

image:
  repository: ghcr.io/kolm-ai/kolm-runtime
  tag: v1.0.0
  pullPolicy: IfNotPresent

replicaCount: 2

artifact:
  registry: registry.acme.internal/kolm
  id: prod-2026-05-24

resources:
  requests:
    cpu: 2
    memory: 8Gi
    nvidia.com/gpu: 1
  limits:
    memory: 16Gi
    nvidia.com/gpu: 1

readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
  failureThreshold: 60

hpa:
  enabled: true
  minReplicas: 2
  maxReplicas: 20
  metric: inference_queue_depth
  targetAverageValue: "5"

serviceMonitor:
  enabled: true
  interval: 30s

annotations:
  kolm.ai/zero-downtime-swap: "true"

terminationGracePeriodSeconds: 60
