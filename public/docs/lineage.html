kolm.ai / docs / lineage

Open-source AI workbench

Frontier AI on your own infrastructure.

Every .kolm artifact carries a parent_cid pointer to the artifact it descended from. The chain is content-addressed, the hash is part of the receipt seal, and a tamperer cannot rewrite history without invalidating every downstream signature. Walk the chain with kolm lineage <cid>, compare two versions with kolm diff, and roll back from the same terminal you shipped from.

Why lineage ships inside the file

Most ML platforms record lineage in a separate metadata service: a database somewhere remembers that model v3.2.1 descended from v3.2.0. The footgun is obvious - when the metadata service goes down or the row gets corrupted, lineage is gone. kolm.ai bakes the parent pointer into the manifest itself, so the file IS the source of truth. The chain is verifiable offline; the chain is verifiable years after the metadata service was decommissioned; the chain is verifiable on an air-gapped reproducer who has never heard of your control plane.

The byte-stability law is preserved: artifacts that do not declare a parent (root / first-of-line) carry NO parent_cid key, so their bytes are byte-identical to pre-artifacts rebuilt today. Only artifacts that explicitly descend from a parent carry the new field - and when they do, the parent_cid is folded into artifact_hash_input immediately adjacent to guardrails_hash, so swapping the pointer post-build breaks the receipt chain loudly.

The parent_cid field

The field is a string: 64 lowercase hex characters (a sha256 digest) or absent. There is no third option - a partial cid, an upper-case cid, or a non-hex string throws at build time so a malformed lineage pointer never reaches a deployed artifact. The validator lives in src/artifact-lineage.js#setParentCid and the build-time guard lives in src/artifact.js#buildPayload.

Walking the chain

The walk follows parent_cid pointers one hop at a time, capped at max_depth (default 10). Each row carries (cid, parent_cid, k_score, created_at) so operators see the K-Score trajectory at a glance.

$ kolm lineage e9b07c4d3aab6f1c5e72d4ba3b1d7e8c4f5a1b2c3d4e5f6a7b8c9d0e1f2a3b4c

lineage chain for e9b07c4d…3b4c  (depth=3, truncated=false)
depth  cid           k_score  parent_cid    created_at
    0  e9b07c4d3aab… 0.9412   f1a2b3c4d5e6… 2026-05-24T18:22:01Z
    1  f1a2b3c4d5e6… 0.9337   a7b8c9d0e1f2… 2026-05-22T09:15:43Z
    2  a7b8c9d0e1f2… 0.9201   null          2026-05-19T14:03:11Z

The CLI loader scans ~/.kolm/artifacts/*.kolm for a matching manifest. The same chain is available over HTTP via POST /v1/artifact/lineage with body {cid, max_depth?}; the server-side loader is tenant-fenced so a tenant can only walk their own catalog. Cycle-safe: a chain that loops back on itself terminates with truncated:true instead of running forever.

Diff workflow

kolm diff <a.kolm> <b.kolm> compares two artifacts: their lineage relation (sibling / ancestor / descendant / unrelated), per-axis K-Score delta, and a roll-back recommendation pill. The recommendation is one of three values and operators can wire it directly into CI without re-implementing the threshold:

RecommendationTriggerWhat to do
roll_backANY axis dropped by >0.02 between A and BPin the prior artifact (see roll-back section below)
promoteEVERY axis improved or unchangedB is safe to ship as the new production artifact
inconclusiveMixed-but-not-bad-enough OR insufficient dataRerun kolm score against a larger holdout before deciding
 
$ kolm diff ~/.kolm/artifacts/v3.2.0.kolm ~/.kolm/artifacts/v3.2.1.kolm

The envelope shape is identical when the same comparison is run server-side via POST /v1/artifact/diff with body {a_cid, b_cid} - the only difference is the route resolves cids through the tenant-fenced loader instead of file paths.

Roll-back via kolm pin

When the recommendation pill is roll_back, the playbook is one command: pin the prior artifact's cid so the runtime serves it instead of the new build. kolm pin <cid> writes the chosen cid into the tenant's active-artifact pointer; the next inference request resolves through the pinned artifact instead of the latest publish. The original artifact bytes are still in the registry - nothing has been deleted - so re-pinning forward is also one command when the regression is fixed.

# roll back to the parent artifact when the new version regresses
$ kolm diff ~/.kolm/artifacts/v3.2.0.kolm ~/.kolm/artifacts/v3.2.1.kolm
recommendation: [roll_back]  regression detected; pin the prior artifact via `kolm pin f1a2b3…` to roll back

$ kolm pin f1a2b3c4d5e6...
pinned: f1a2b3c4d5e6...  (now active for tenant_id=...)

The roll-back has zero downtime because the pinned artifact is already in the registry - there is no rebuild, no redeploy, no compile job. Forward-fixing the regression follows the same loop: distill a new candidate, kolm diff against the pinned baseline, and kolm pin the new cid when the recommendation flips back to promote.

A/B testing

The routing fabric wires kolm pin into a percentage-weighted A/B split: pin {cid_a: 80, cid_b: 20} and the runtime serves the new build to 20% of traffic. The lineage chain is what makes the A/B safe - both arms are content-addressed, both are signed, both descend from the same parent so a per-axis K-Score delta is the right measurement. For the chain + diff land first; the routing fabric ships in.
