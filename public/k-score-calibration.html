K-score methodology · external calibration

What does K-score 0.91 actually mean to a human?

K-score is a per-artifact gate composed from five (or ten) axes. It is calibrated, every quarter, against a pack of pairwise human preferences. This page documents the fit, the per-category curves currently in production, and the cases where the data is too thin to publish a curve at all.

What this page is

Calibration, not contest.

K-score is the per-artifact ship gate. Calibration is the translation from that internal gate to a number a stakeholder cares about: "if you ship at K=0.91, what fraction of users will prefer this artifact's response over the reference?". This page documents how that translation is built and where it does not yet apply.

Quarterly

One pack per quarter

Every quarter, a fresh pack of pairwise human preferences ships into ~/.kolm/calibration-pack-YYYY-MM.jsonl. Each row carries a prompt, two responses, a human preference (a / b / tie), and a task category. The pack's anonymized hash is published below.

Bradley-Terry

Latent skill, fit in pure JS

We fit a Bradley-Terry model (gradient ascent on the logistic log-likelihood with backtracking line search) to estimate a latent skill per response. The fit is bundled into the CLI so anyone can recompute it from the pack without a Python toolchain.

Per-category

Coding, writing, analysis, support

The fit is run separately for each task category. A category-specific curve only ships when its pack contains at least 500 pairs. Below that, the envelope reports calibration_status:insufficient_data and the K-score is surfaced without a human-preference-rate block.

CI95

Wald 95% interval, not a point

Each surfaced human_preference_rate carries a 95% Wald interval derived from the slope's standard error in the calibration regression. A point estimate without an interval is a confidence claim without uncertainty; we do not ship those.

End to end

From pack rows to a calibrated envelope.

01 / Pack
~/.kolm/calibration-pack-YYYY-MM.jsonl

JSONL rows: {pair_id, prompt, response_a, response_b, human_preference, task_category}. KOLM_DATA_DIR overrides ~/.kolm for testing.

02 / Fit
Bradley-Terry per category

Centered theta, max 1000 iters or |grad|inf below 1e-6. Tiny L2 ridge keeps the Hessian invertible on rare items.

03 / Regress
Logit-WLS calibration

10-bin weighted least squares of logit(observed-rate) on logit(BT-predicted-rate). Slope, intercept, slope SE, Wald 95% CI persisted.

04 / Persist
~/.kolm/kscore-calibration.json

Per-category {slope, intercept, ci95_low, ci95_high, n_pairs, status} + pooled. Status is either ok or insufficient_data.

05 / Surface
K-score envelope

Every computeKScore response carries human_preference_rate when calibration_category is supplied and the category fit is ok; null with a status code otherwise.

06 / Recalibrate
scripts/recalibrate-kscore.cjs

Cron-friendly: exit 0 success, exit 2 pack missing, exit 3 no category reached threshold. --dry-run available.

The math

Bradley-Terry, then logit-WLS.

# Bradley-Terry: latent skill from pairwise prefs
P(a beats b)         = sigmoid(theta_a - theta_b)
loss(theta)          = - sum_{pairs} log P(observed | theta) + (ridge/2) ||theta||^2
fit                  = gradient ascent + backtracking line search
identifiability      = mean(theta) == 0  (re-centered each step)
convergence          = |grad|_inf below 1e-6, or 1000 iters
ci95 (per response)  = Wald, sqrt(1 / diag(neg Hessian))

# Calibration regression: logit(rate) over 10 bins of logit(predicted)
x_b                  = logit(mean predicted prob in bin b)
y_b                  = logit(mean observed rate  in bin b)
weight_b             = n in bin b
slope, intercept     = weighted least squares
ci95(slope)          = slope +/- 1.96 * SE(slope)
mapping(K)           = expit(slope * logit(K) + intercept)
mapping_ci95(K)      = [expit(low * logit(K) + intercept),
                        expit(high * logit(K) + intercept)]
Honest contract

What the envelope says, line by line.

Calibration is a separate axis from the K-Score itself. Never silently substituted, never inflated, always typed.

calibration_status: ok
The supplied calibration_category has a category fit on file with n ≥ 500 pairs. human_preference_rate is populated with a point estimate and a Wald 95% interval derived from the slope's standard error in the calibration regression.
calibration_status: insufficient_data
The category exists in the mapping file but has fewer than 500 pairs. human_preference_rate is null. We do NOT fall back to the pooled estimate; the per-category measurement basis would be wrong.
calibration_status: unknown_category
The calibration_category supplied by the caller is not one of coding / writing / analysis / support. human_preference_rate is null; the K-Score composite itself is unchanged.
calibration_status: no_calibration_mapping
No fitted mapping file is installed (~/.kolm/kscore-calibration.json missing or unreadable). Every calibration field is null. The K-Score still ships; we just cannot translate it to a human-preference rate.
calibration_status: no_category_supplied
A mapping is installed but the caller did not pass calibration_category. The mapping_id is still surfaced so logs are auditable; human_preference_rate is null until the caller declares which category to translate to.
calibration_status: no_composite_to_calibrate
Defensive code path: the K-Score envelope did not produce a numeric composite (e.g., the v2 axis math degenerated). Surfaces as null so downstream telemetry can detect and alert.
Reproduce on your machine

From pack to mapping in one command.

# 1. Drop a quarterly pack at ~/.kolm/calibration-pack-2026-04.jsonl
#    Row shape: {pair_id, prompt, response_a, response_b,
#                human_preference:'a'|'b'|'tie', task_category}

# 2. Fit + persist the mapping (cron-friendly: exit 0/2/3/4)
node scripts/recalibrate-kscore.cjs --period 2026-Q2 --json

# 3. Inspect ~/.kolm/kscore-calibration.json
#    {
#      "version": "w810-v1",
#      "calibration_pack_id": "2026-Q2",
#      "by_category": {
#        "coding":   {"status":"ok", "slope":..., "intercept":...,
#                     "ci95_low":..., "ci95_high":..., "n_pairs":...},
#        "writing":  {"status":"insufficient_data", "n_pairs":...},
#        ...
#      },
#      "pooled":   {...},
#      "n_pairs": ...,
#      "fitted_at": "..."
#    }

# 4. K-Score now surfaces a human_preference_rate block when the
#    caller supplies calibration_category
node -e "
  import('./src/kscore.js').then(({computeKScore}) => {
    const out = computeKScore({
      accuracy: 0.97, size_bytes: 32 * 1024 * 1024,
      p50_latency_us: 1200, cost_usd_per_call: 0.0001, coverage: 0.92,
      calibration_category: 'coding'
    });
    console.log(out.human_preference_rate, out.calibration_status);
  });
"
