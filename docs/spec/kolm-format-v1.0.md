# kolm format specification, v1.0

STATUS: DRAFT, 2026-05-24

This document specifies version 1.0 of the .kolm artifact format. A .kolm artifact is the durable, signed unit that the kolm compiler produces and that downstream runtimes consume. The format is designed to be auditable years after the fact: every byte that influences trust is named, hashed, and bound to the artifact identity.

## 1. Overview

A .kolm artifact is a zip container with three required entries (a manifest, a weights blob, a provenance journal) and one optional entry tree (signatures). Readers MUST be able to verify the artifact identity and the signing chain from the zip alone, with no network access.

The artifact identity is the sha256 of the canonical zip stream (entries in alphabetical order, no compression, no extra fields). That sha256 is the artifact_hash referenced in the manifest and the receipt.

The format is intentionally simple. There is no nested archive, no custom compression, no proprietary index. A reader written in plain C, Python, or Rust can implement the full spec in under 500 lines of code without external dependencies beyond a zip library and a sha256 implementation.

## 2. File layout

A v1.0 .kolm artifact MUST contain these top-level entries:

- `manifest.json` (required) - the JSON manifest, schema in section 3.
- `weights.bin` (required) - the quantized weights blob, format opaque to the spec; bound by sha256 in the manifest.
- `provenance.jsonl` (required) - append-only JSONL of provenance rows (teacher calls, distill steps, calibration events).
- `signatures/` (optional) - directory of detached signature files; one file per signer.

Readers MUST reject artifacts missing any of the three required entries. Readers MAY accept additional top-level entries for forward-compatibility but MUST NOT use them when computing identity (the canonical hash covers only the four named entries; see section 6).

## 3. manifest.json schema

The manifest is a single JSON object. The required fields are:

- `artifact_hash` (string) - sha256 of the canonical zip stream, hex-encoded.
- `teacher_id` (string) - opaque identifier for the teacher model used during distillation.
- `student_arch` (string) - opaque identifier for the student architecture (e.g., "qwen2.5-7b-int4").
- `k_score` (object) - calibrated quality score with two sub-fields:
  - `point` (number, range [0, 1]) - the point estimate.
  - `ci95` (array of two numbers) - the 95% confidence interval as [low, high].
- `calibration_pack_id` (string) - identifier of the calibration pack used to compute k_score.

Optional fields:

- `output_schema` (object) - a JSON Schema describing the expected response shape; runtimes MAY validate against it.
- `confidential_compute_hash` (string) - sha256 of the attestation block when the artifact was produced inside a TEE.
- `attestation_block` (object) - the raw attestation report (PCCS, SNP, Nitro, NRAS); content opaque to the spec.

Readers MUST reject manifests missing any required field. Readers MUST ignore unknown fields (forward-compatibility).

## 4. Versioning policy

The format follows semantic versioning:

- Major bump (1.x -> 2.x) signals a parser break. v2 readers MUST also implement a v1 fallback when the artifact declares a v1 format version, and v1 readers SHOULD warn but MAY refuse to load v2 artifacts.
- Minor bump (1.0 -> 1.1) adds new optional fields. v1.0 readers MUST tolerate v1.1 artifacts by ignoring unknown optional fields.
- Patch bump (1.0.0 -> 1.0.1) is reserved for spec clarifications that do not change wire format. A patch bump never changes parser behavior.

The format version is declared in the manifest under the `format_version` field (string, semver). v1.0 artifacts MUST declare `"format_version":"1.0"`.

## 5. Test vectors

Reference fixtures for conformance testing live under `tests/fixtures/format-v1/` in the kolm source tree. Each fixture is a .kolm file whose sha256 is published in `docs/spec/test-vectors/MANIFEST.md`.

The five v1.0 fixtures cover:

- `tests/fixtures/format-v1/minimal.kolm` - smallest valid artifact, no optional fields. Expected sha256: PENDING.
- `tests/fixtures/format-v1/with-attestation.kolm` - includes confidential_compute_hash + attestation_block. Expected sha256: PENDING.
- `tests/fixtures/format-v1/with-output-schema.kolm` - includes output_schema. Expected sha256: PENDING.
- `tests/fixtures/format-v1/with-confidential-compute.kolm` - includes a full TEE attestation block. Expected sha256: PENDING.
- `tests/fixtures/format-v1/with-multimodal.kolm` - declares multimodal output_schema with image and audio fields. Expected sha256: PENDING.

Binary fixtures are not generated by this spec document. The expected sha256 columns in the test vector manifest will be filled when the v1.0 status moves from DRAFT to ACTIVE.

## 6. Compatibility table

| Reader version | Accepts v1.0.x | Accepts v1.x | Accepts v2.x |
| -------------- | -------------- | ------------ | ------------ |
| v1.0           | MUST           | SHOULD warn  | MAY refuse   |
| v1.x           | MUST           | MUST         | SHOULD warn  |
| v2.x           | SHOULD         | SHOULD       | MUST         |

Readers SHOULD log the artifact format_version on load. When a reader chooses to refuse an artifact for version reasons, it MUST surface the declared version in the rejection message so the operator can debug the mismatch without reading the artifact.
