HIPAA-grade specialist models. PHI never leaves your VPC.

kolm is an open-source AI compiler. Angle Health deploys it inside an Angle-owned AWS account; PHI is captured, distilled, quantized, and served entirely inside that perimeter. Every inference produces an Ed25519-signed receipt your auditor can verify offline.

1. The architecture in one sentence

An OpenAI-compatible gateway sits in front of a base model. Member-facing calls hit the gateway; (input, output) pairs are redacted and persisted to an Angle-owned S3 bucket; a periodic kolm compile job distills a smaller specialist model from those pairs; the artifact is signed and served back through the same gateway. PHI does not traverse the kolm.ai control plane at any step.

For diagram, see architecture.svg in this directory.

2. PHI lifecycle, step by step

  1. Capture. A member request enters the gateway. The redactor runs HIPAA Safe Harbor profile on the request body and the model response before either is written to disk. Original payloads are never persisted in cleartext; only the redacted, hashed canonical form lands in the lake.
  2. Storage. Redacted pairs live in an Angle-controlled S3 bucket with SSE-KMS using an Angle-owned CMK. The bucket policy denies access to the kolm.ai principal; only the in-VPC compute role can read it.
  3. Distill. A scheduled batch job runs kolm compile on an Angle-owned GPU instance. Teacher calls (Claude, GPT-4, or an in-VPC base model) execute through the same gateway, so any teacher traffic is also subject to the redactor profile. The compiler writes a signed .kolm artifact to Angle S3.
  4. Serve. The runtime loads the .kolm from Angle S3 and serves it through the gateway. Every inference returns a per-call receipt: { artifact_cid, input_hash, output_hash, k_axes, signed_at, signature }. The receipt is the audit log entry; it is appended to a Postgres table in Angle's account.
  5. Verify. An auditor exports the receipt table and runs kolm verify on any laptop. The verifier recomputes the artifact hash, re-signs the canonical receipt, and checks the Ed25519 signature against the embedded public key. No network call to kolm.ai is required.

4. Signed receipts: what is in one, what they prove

Every inference returns a receipt. The receipt names the inputs that produced the answer and binds them under a single Ed25519 signature.

artifact_cid content address of the .kolm that served the call
input_hash sha-256 of the redacted, canonicalized prompt
output_hash sha-256 of the redacted, canonicalized response
k_axes per-call K-score (faithfulness, coverage, calibration, groundedness, cost)
signed_at RFC 3339 timestamp inside the signature scope
signature Ed25519 over JCS-canonical receipt body

Editing any field invalidates the signature. The verifier requires no network call; the public-key fingerprint is embedded in the artifact and pinned at deploy time.

5. Revocation and key custody

Revocation

Per-artifact revocation: an Angle operator publishes a revocation entry naming the artifact_cid; the gateway refuses to load any artifact whose CID matches. Per-key revocation: an Angle operator rotates the signing key; all artifacts signed by the rotated key fail verification until re-signed.

Key custody

The Ed25519 signing key lives in AWS KMS in the Angle account. The kolm compiler calls kms:Sign; the raw key material never leaves KMS. Public-key fingerprints are pinned per environment (sandbox, staging, production) so a compromised lower-environment key cannot mint production artifacts.

7. What we do not do

  • kolm.ai does not see PHI. The data plane is in your AWS account. The control plane (license, telemetry opt-out, version pin) sees only the artifact CID and a per-tenant counter.
  • We do not store an external copy of any .kolm artifact. Artifacts live in the customer S3 bucket. The kolm.ai marketplace shows only signed metadata for artifacts that opt-in to publication.
  • We do not have any signing key for any customer. The Ed25519 key is generated in the customer KMS and never copied.
