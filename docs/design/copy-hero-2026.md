## REUSABLE SECTION-HEADLINE BANK

Drop these into any page. Each is benefit-first, 3-5 word imperative where it fits, period-terminated, with a one-line earn beneath. Grouped by the four strategy pillars so every page maps cleanly.

### Pillar 1 — Cut the bill (cost)
- **Cut the bill.** — *Re-running it shouldn't cost as much as inventing it. Run it on your hardware and stop paying per call.*
- **Stop renting the AI you already paid for.** — *Once Kolm has captured how you use the API, you pay to run that behavior, not to rent it.*
- **Pay to run your model, not to rent someone else's.** — *Every call after capture is yours.*
- **Re-running it shouldn't cost as much as inventing it.** — *Compile the behavior once; run it cheap from then on.*

### Pillar 2 — Own the model (ownership)
- **Own the model.** — *One portable file — model, recipe, evals, receipt — that's yours.*
- **One file you own.** — *Model, recipe, evals, and receipt, compiled into a single artifact you can move anywhere.*
- **Buy the model once. Run it forever.** — *Not an endpoint you rent — a portable artifact you keep.*
- **Your API calls, compiled into a model you own.** — *Export it, move it, keep it. No provider lock-in.*

### Pillar 3 — Ships anywhere (portability + drop-in)
- **Ships anywhere.** — *Drop it in, run it on the hardware you already have.*
- **Point your existing calls at Kolm. No rewrite.** — *Keep your SDK exactly as it is.*
- **Run it on the laptop you have, the cloud you trust.** — *Pick the smallest runtime that fits — with the limits shown.*
- **Take your model and go — it's one file.** — *Laptop, your private cloud, the edge, or a server.*

### Pillar 4 — Provable trust (signed + verifiable)
- **Provable trust.** — *Every artifact is signed. Verify it yourself — no trust required.*
- **Verify it yourself.** — *Check the signature without trusting us — verification built in, not bolted on.*
- **Signed, sealed, yours.** — *Every artifact carries proof of exactly what it is and how it behaves.*
- **Every model signed, so anyone can verify it.** — *Proof anyone can check — no trust required.*

### Process / how-it-works (speed-control)
- **From live calls to a shipped model in four steps.** — *Capture, compile, sign, deploy.*
- **Capture, compile, ship.** — *Your behavior in one portable file, on your own hardware.*
- **Four steps from your API traffic to a model on your own hardware.** — *No rewrite to get here, no lock-in once you have it.*

### Headline construction rules (so the bank stays coherent)
1. Outcome leads the line; the spec earns it in the sub-line ("so you can…").
2. 3-5 word imperative where it fits, period-terminated, parallel rhythm.
3. Numbers as proof, not adjectives (a real limit/count beats "powerful").
4. Kill a buyer fear in the differentiator line — name the dread (lock-in, opaque cost, unverifiable models) and state it dead.
5. Never name the architecture, the backend, or our own honesty as a headline.
