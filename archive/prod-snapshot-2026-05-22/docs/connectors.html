Connectors

Every framework you already use speaks an OpenAI-compatible or Anthropic-compatible API. Point its base URL at the local kolm proxy on http://127.0.0.1:8787 and every round-trip is captured, the response streams through unchanged, and you build a private corpus you can later compile into a smaller, cheaper artifact.

One env var, eight runtimes

The proxy speaks both API shapes (OpenAI Chat Completions and Anthropic Messages), and translates between them when a client expects one and an upstream serves the other. You set the base URL, the proxy handles the rest. Streaming, function calling, vision, and tool use all pass through.

Start the proxy

kolm capture --provider openai --as ticket-classifier --namespace tickets
# kolm proxy listening on http://127.0.0.1:8787
# capture config written to ~/.kolm/capture/ticket-classifier.json

The first form picks a namespace and the upstream provider, and writes a small JSON file with the headers your app should send. Pass your real OpenAI, Anthropic, or OpenRouter key in the x-upstream-api-key header on each request; the Kolm API key goes in Authorization: Bearer ks_* or Authorization: Bearer kao_*.

Verify the proxy sees traffic

kolm capture status --namespace tickets
# pairs captured: 38 / 1000
# pairs until distill: 962

Once you cross the threshold (default 1000 pairs), kolm distill --namespace tickets auto-runs the LoRA trainer against the namespace.

Streaming and tool use

Server-sent events are forwarded byte-for-byte from the upstream provider. Tool use, function calling, and structured outputs pass through with no schema rewrite. The only thing the proxy adds is a captured row in the local lake; payload shape is whatever the upstream emitted.

Honest scope

The proxy is local. Captured pairs land in ~/.kolm/events.sqlite on the machine running the proxy. Nothing is sent to kolm.ai unless you flip on cloud-sync (off by default) or run a hosted compile. The 19-class privacy membrane tags each captured prompt with which classes it touched, and the per-class blocklist on the cloud-sync settings page acts as a kill-switch.
