kolm.ai / integrations / openrouter

OpenRouter-compatible routing through kolm.

If your app already talks to OpenRouter, point it at kolm and keep the routing semantics. The HTTP-Referer and X-Title headers travel with the request, get stamped onto the receipt, and show up in your captures inbox so app-attribution still works end to end.

How it works in one paragraph

kolm.ai exposes the OpenRouter routing conventions at https://kolm.ai/v1. The request body is OpenAI-shaped chat completions, but kolm honors the two OpenRouter-specific headers: HTTP-Referer (used by OpenRouter to identify the calling app) and X-Title (the app's display name). Both are persisted with the capture row so downstream analytics see the same attribution string they always did. Provider preferences and fallback semantics map onto kolm's router config.

Python – routing headers preserved

Send the same headers you would send to OpenRouter. kolm forwards them to the upstream model and stamps them onto the receipt.

python – openai >= 1.0
from openai import OpenAI

client = OpenAI(
 base_url="https://kolm.ai/v1",
 api_key="ks_live_...",
 default_headers={
 "HTTP-Referer": "https://your-app.example.com",
 "X-Title": "your-app",
 },
)

resp = client.chat.completions.create(
 model="anthropic/claude-3.5-sonnet",
 messages=[{"role": "user", "content": "Summarize this PR: ..."}],
)

print(resp.choices[0].message.content)

JavaScript / TypeScript – routing headers preserved

Same surface in Node. Set defaultHeaders at client construction.

node – openai >= 4
import OpenAI from "openai";

const client = new OpenAI({
 baseURL: "https://kolm.ai/v1",
 apiKey: process.env.KOLM_API_KEY,
 defaultHeaders: {
 "HTTP-Referer": "https://your-app.example.com",
 "X-Title": "your-app",
 },
});

const resp = await client.chat.completions.create({
 model: "anthropic/claude-3.5-sonnet",
 messages: [{ role: "user", content: "Summarize this PR: ..." }],
});
