Script-everything recipe

A shell script is the most durable interface a tool can ship. It runs on every Unix going back to 1971, it lives in source control, it survives every framework rewrite. kolm exposes profiles, a doctor, and a kolmrc convention so every machine you touch can be configured, validated, and reproduced from a script in your repo.

Why this matters

"Works on my machine" is the most expensive bug in any AI stack. The base model is on a path that does not exist on the other box. The GPU driver is the wrong version. The capture endpoint points at staging on one machine and prod on another. Solving this with documentation does not work; solving it with a Docker image is heavy and slow. Solving it with a 30-line shell script that kolm doctor validates is the right size of solution.

kolm ships three primitives for this: kolm profile for env snapshots, kolm doctor for environment validation, and the ~/.kolmrc convention for shell auto-source. Together they make "set up a fresh box for this fleet" a single sourced script.

One-time setup

  1. On each box: run kolm doctor once. It reports missing env vars, wrong base-model paths, GPU driver mismatches, capture endpoint reachability. Fix what it flags before you save a profile.
  2. Save a per-environment profile: kolm profile save prod snapshots KOLM_BASE / KOLM_KEY (redacted) / KOLM_TENANT / KOLM_DATA_DIR / KOLM_STORE_DRIVER.
  3. Add a ~/.kolmrc file with the env vars your shell should always have. Source it from ~/.bashrc or ~/.zshrc.
  4. Commit the kolmrc to your private dotfiles repo (NOT the per-machine profile, which may contain machine-specific paths). The shape is portable; the values are not.

The kolmrc convention

# ~/.kolmrc
# Sourced from .bashrc/.zshrc on every shell start.
# Per-machine paths go in `kolm profile save`; this file is the shape.

export KOLM_BASE="${KOLM_BASE:-https://kolm.ai}"
export KOLM_DATA_DIR="${KOLM_DATA_DIR:-$HOME/.kolm/data}"
export KOLM_PROFILE="${KOLM_PROFILE:-default}"

# Aliases for the verbs you call most.
alias kjobs="kolm jobs list"
alias ktail="kolm tail captures"
alias kdoc="kolm doctor"

# Function: run the doctor at every login. Fails loud, not silent.
kolm_check() {
 if command -v kolm >/dev/null 2>&1; then
 kolm doctor --quiet || echo "warn: kolm doctor flagged issues; run 'kolm doctor' for details"
 fi
}

The profile flow

# save the current env as a named profile
kolm profile save prod

# list every saved profile (* marks the active one)
kolm profile list
# default
# * prod
# staging

# switch active profile (records intent; you still apply via export)
kolm profile use staging

# show what a profile contains
kolm profile show staging
# {
# "created_at": "2026-05-18T14:33:12.881Z",
# "env": {
# "KOLM_BASE": "https://staging.kolm.ai",
# "KOLM_TENANT": "acme",
# "KOLM_DATA_DIR": "/srv/kolm/staging",
# ...
# }
# }

The doctor flow

# verbose check (the default; suitable for fresh-box setup)
kolm doctor

# quiet, exit-code only (suitable for shell prompts or CI)
kolm doctor --quiet || echo "kolm env broken"

# detect hardware tier and recommend a base model
kolm doctor --detect-hw

What this gets you

  • Reproducible env in one file you can commit to dotfiles. No new tool to learn.
  • Per-machine drift caught at login, not at compile time. A failed compile because of a wrong env var costs hours; a doctor warning at login costs seconds.
  • Profile-per-environment makes context switching explicit. "Why is my eval reporting against staging" stops being a mystery.
  • Scriptable from CI: cron a kolm doctor --quiet on every consumer box, alert on non-zero exit.
  • No vendor abstraction: it is just env vars and a bash function. Every tool in your stack already speaks this.
The W229 jobs registry stores its log files under $KOLM_JOB_LOG_DIR (default ~/.kolm/job-logs). If you override KOLM_DATA_DIR in a profile, set KOLM_JOB_LOG_DIR with it; otherwise kolm watch <id> on a profile-switched shell will not find the log.

Failure modes worth knowing

Profile values fall out of date. Profiles snapshot env at save time; they do not auto-refresh. Re-save when the underlying env changes (new tenant ID, rotated key, moved data dir). The doctor catches drift but does not fix it.

Sourcing kolmrc twice. If ~/.kolmrc is sourced from both ~/.bashrc and a parent shell, env exports stack but the aliases redefine cleanly. Verify with type kjobs.

Quiet mode hides real failures. kolm doctor --quiet suppresses output; if you wire it into a shell prompt or CI, also wire the non-zero exit to an alert.
