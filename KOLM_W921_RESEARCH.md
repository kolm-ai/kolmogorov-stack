# KOLM W921 — Frontier Research → Execution Roadmap

> 60 verified specs span 8 product surfaces plus the demo rebuild. Two source files dominate the contention map and force a SEQUENTIAL LANE: src/router.js (touched by ~17 specs: all of Gateway, all of Govern, Autopilot bandit + sequential-AB, plus Website pricing/trust/hero/error-boundary) and cli/kolm.js (touched by ~33 specs: all of CLI, Studio, Run, Data Engine, plus Autopilot and several Govern). Everything that creates NEW isolated modules (semantic-router.js, provider-health.js, gateway-guardrail.js, conformal.js, drift-stats.js, etc.) or NEW worker/python files (workers/data, apps/trainer, apps/runtime) or NEW build scripts (scripts/build-*.cjs) or NEW client assets (public/account-*.js, public/*.css) can be authored 100% in parallel; only the final WIRE step that edits router.js or cli/kolm.js must serialize through the big-file lane. The demo rebuild is fully parallel-safe (it touches only public/demo-live*, data/demo, scripts/capture-demo-timeline.mjs, scripts/demo-claim-verify.cjs, scripts/release-verify.cjs, public/sw.js — none of the two big files). The highest-leverage, most-differentiating work concentrates in the Gateway (semantic routing + injection guardrail + circuit breaker + true streaming + semantic cache all stamped into the existing non-signed receipt block — turning resilience, safety and savings into AUDITABLE proof no competitor offers) and in Studio/Run (a real logit-level distillation objective + multi-LoRA serving + EAGLE speculative decoding, the engine kolm sells but has not fully wired). CRITICAL SEQUENCING NOTE: three pre-existing w920 P0 defects on the gateway hot path (output-redaction never reaching the client, fallback_reason enum drift that can THROW and abort a call, cost_usd silently defaulting to 0) must be fixed FIRST because four W921 gateway specs build on that path and one (semantic router) explicitly depends on the cost_usd fix and on a quality-label/prompt-text capture pipeline the lake does not yet store. Recommended order: Phase 0 fix the gateway P0 trust bugs + build the additive non-signed receipt-field convention once; Phase 1 author all NEW-module/worker/script/asset groups in parallel; Phase 2 run the two big-file wire lanes serially (router lane, then/alongside cli lane) integrating each completed module; Phase 3 demo rebuild + release-verify gates + sw.js bump.

## Top leverage

1. **Cost/latency/quality-aware semantic router (Avengers-Pro cluster/KNN over kolm's own captured outcomes) stamped into the non-signed receipt.router_decision block** _(Gateway / Route & Capture)_ — kolm's central pitch — 'cheapest model that clears the bar, with a receipt to prove it' — is today UNREALIZED: routing is a static chain plus a caller-supplied confidence number kolm never computes. This is the single biggest gap-to-pitch on the flagship revenue route, and stamping route_score + rejected candidates into the existing additive non-signed receipt field makes the savings AUDITABLE — a differentiator RouteLLM/LiteLLM/OpenRouter cannot match. Caveat: the verdict flags a real prerequisite (the lake stores no quality/win-loss label and no prompt text), so v1 ships as a cost+latency reorderer and the quality term needs a capture-pipeline addition (adjusted 13d).
2. **Inline prompt-injection / jailbreak guardrail wired into the dispatch input stage, verdict stamped into the signed-receipt-adjacent non-signed block** _(Gateway / Route & Capture)_ — OWASP LLM01:2025, the #1 LLM risk and table-stakes vs Cloudflare/Portkey. The classifier already exists, is tested, and never throws — it is merely UNWIRED from the live path. 3-day, zero-new-dependency win that blocks an attack BEFORE any upstream token is spent ($0 rejection) and lets kolm uniquely ATTEST the screening verdict in a verifiable receipt. Lowest-risk/highest-impact item in the whole wave (verdict: scopeCorrect+depsComplete, 3d).
3. **Health-aware adaptive failover: per-provider circuit breaker (Resilience4j 3-state + Envoy exponential ejection) + health-aware load balancing, with a fail-open panic invariant** _(Gateway / Route & Capture)_ — Today a hard-down primary is re-fired on EVERY request at full timeout (60s default, up to 300s), cascading one provider outage into a tenant-wide latency cliff and inflating the very wrapper-tax the receipts measure. A breaker that skips known-dead providers (and stamps circuit_open as a verifiable route decision) converts resilience into the same audit-trail differentiator. Must enforce never-eject-the-last-provider so the breaker can only improve availability.
4. **Trainer-stack unification + a real logit-level KD objective (DistiLLM-2 contrastive SKL+SRKL / KL) promoted into the shipping path** _(Studio / Distill & Compile)_ — kolm sells distillation, but the actual distillation OBJECTIVE (logit-level KD loss) is the critical w920 Studio gap (#0, 0.85) and the two trainer stacks diverge. Unifying them and shipping the real contrastive loss is what makes the Trinity story technically true rather than a SeqKD approximation. Two critical-impact specs converge here (unification + DistiLLM-2).
5. **Multi-LoRA serving (one base, N adapters, per-request switch on one GPU) wired into serve.py + deploy generators** _(Run / Serve & Deploy)_ — Critical-impact (run.frontier #0, 0.9): the Python is implemented but NOT wired into the serve path, so the marquee 'distill many small adapters, serve them cheaply' economics are not deliverable. Unlocks the per-skill adapter business model and pairs directly with the Studio model-merge + LoRA-variant specs.
6. **Fix fallback_reason enum drift (latent P0 that can THROW and abort a gateway call) + adopt the additive non-signed / RFC-8785 receipt discipline, then Sigstore/Rekor per-call receipt anchoring** _(Govern / Receipts & Compliance)_ — The enum-drift bug (govern.frontier #0, critical 0.9) is a live abort risk that the new gateway specs (which add 'circuit_open' to the enum) would otherwise trip. Once the receipt path is hardened, transparency-log anchoring of per-call receipts turns kolm's signed-receipt story into independently-verifiable provenance — the deepest moat in the product.
7. **Demo rebuild (re-root the cinematic engine in a captured REAL CLI run with an in-browser Ed25519 verify climax) + single-source-of-truth pricing coherence + the live hero demo** _(Website / Marketing / Conversion + Demo)_ — The current /demo-live is 100% fabricated string literals with a climax that points at a 404 — the #1 'canned demo' credibility-killer. Re-rooting it in a checked-in real run with a self-contained, zero-network, in-browser receipt verification makes the product's core claim impossible to fake. Pricing coherence is critical-impact (incoherent numbers across homepage/pricing/compare actively erode trust). These are parallel-safe and convert the proof story into the conversion story.
8. **Sequential always-valid A/B testing (GAVI + mSPRT) + replace the toy stump-GBM meta-model with a real gradient-boosted regressor + conformal-calibrated K-Score intervals** _(Autopilot / Autonomous Agent)_ — The autopilot deploy guardrail currently gates on a point estimate from a depth-1 stump and a fixed-horizon test — statistically indefensible for an autonomous re-distill/deploy loop. Anytime-valid stats (critical-impact) + a real GBM + conformal lower-bound gating make the self-improving loop trustworthy enough to actually run unattended, which is the entire autopilot value proposition.

**How it works.** PIPELINE PLACEMENT. The meta-model sits at the bottom of the autopilot intelligence stack. Every economic decision the daemon makes — simulateCompile (compile vs skip on delta_k), cost-optimizer.rankStrategies (delta_K / dollar), _evaluateDeploy (promote only when the prediction clears the ship gate), student-arch-recommender (which backbone) — is gated on a single number: predicted K-Score, plus an interval. quality-predictor.predictKScore is the entry point; below MIN_ROWS_FOR_META (1000) it uses a hand-calibrated heuristic, at/above it flips to the learned GBM in src/kolm-meta-trainer.js (W832). That GBM is the toy this spec replaces.  WHY GBT IS THE RIGHT FAMILY (math + evidence). The meta-task is small-n tabular regression: a feature matrix X (n_pairs, dup_fraction, coverage_score, avg_quality, cot_contam_fraction, teacher_diversity, plus the W832 META_FEATURES) -> scalar target y = observed K in [0,1], with n in the hundreds-to-few-thousand range. The 2024 'Comprehensive Benchmark of ML and DL across diverse tabular datasets' (arXiv:2408.14817) shows tree ensembles win the MAJORITY of tabular tasks and KEEP winning even when large sets are down-sampled to 1000 rows (CatBoost best on 19/111 = 17.1% vs best DL 9.9%; on <1000-row sets H2O-GBM 16.6% vs best DL ResNet 13.9%). This is exactly kolm's regime, so a real GBM (not a NN, not a single tree) is the correct upgrade.  WHAT 'REAL GBM' MEANS vs THE CURRENT STUMP. Gradient boosting fits an additive model F_M(x) = sum_{m=1..M} eta * h_m(x), where each h_m is a regression tree fit to the negative gradient (for squared loss, the residual r_i = y_i - F_{m-1}(x_i)). The current code (_trainOneTargetGBM) restricts EVERY h_m to a depth-1 stump (one split, one feature) with NO regularization, NO row subsampling, NO column subsampling, and a constant 0 baseline. Depth-1 stumps cannot represent ANY feature interaction (e.g. 'more pairs only helps when dup_fraction is low'), which is the dominant structure in data-quality->K. The fix is the textbook GBT recipe (Friedman 2001 'Greedy Function Approximation'; Chen & Guestrin 2016 'XGBoost'): (1) depth-D trees (D=3 captures up to 3-way interactions); (2) the regularized split objective Gain = 1/2 * [ G_L^2/(H_L+lambda) + G_R^2/(H_R+lambda) - (G_L+G_R)^2/(H_L+H_R+lambda) ] - gamma, where for squared loss g_i = -(y_i - F(x_i)) and h_i = 1 (so leaf weight w = -G/(H+lambda) = mean residual shrunk by L2 lambda); (3) Friedman stochastic subsampling (row fraction subsample=0.8, feature fraction colsample=0.7) which both regularizes and decorrelates trees; (4) baseline F_0 = mean(y) instead of 0 so the first tree models deviation not level; (5) min_child_weight (min samples per leaf) and early stopping on a holdout to pick M. All of this is implementable in pure JS at the N<=50k scale the file already targets — it is the same brute-force best-split scan, just recursed to depth D over a column/row subsample with the regularized gain.  WHY UNCERTAINTY MUST BE CALIBRATED, NOT HEURISTIC. Today inferKolmMeta returns confidence = n_train_rows/1000 (a SUPPORT proxy, explicitly 'NOT a posterior') and quality-predictor turns it into a fixed band half = 0.155 - 0.11*confidence. An autonomous system that SPENDS GPU and DEPLOYS models on an uncalibrated band is fragile: when the true conditional variance is high (sparse region of feature space) the band can be far too tight, so _evaluateDeploy promotes on false confidence. Two production-grade ways to get real uncertainty: (A) NGBoost / CatBoost-RMSEWithUncertainty — natural-gradient boosting that fits the parameters (mu, sigma) of a conditional Normal by minimizing NLL, giving a native predictive distribution. BUT the parametric-Normal assumption is fragile: Manokhin's published evaluation shows NGBoost's nominal 95% interval covered only 91% and the 80% interval only 74%, and HPO flipped it from over- to under-confident — i.e. NOT reliably calibrated. (B) CONFORMALIZED QUANTILE REGRESSION (Romano, Patterson, Candes 2019, arXiv:1905.03222) / split-conformal — wrap ANY point regressor (our real GBM) and produce intervals with a DISTRIBUTION-FREE finite-sample marginal coverage guarantee P(y in C(x)) >= 1 - alpha under exchangeability, with NO distributional assumption. Concretely: fit two quantile GBMs q_lo (alpha/2) and q_hi (1-alpha/2) via pinball loss on a training split, then on a held-out calibration split compute conformity scores E_i = max(q_lo(x_i) - y_i, y_i - q_hi(x_i)), take Q = the ceil((1-alpha)(n_cal+1))/n_cal empirical quantile of {E_i}, and emit the calibrated interval [q_lo(x) - Q, q_hi(x) + Q]. This is the technique Manokhin and the MAPIE library (scikit-learn-contrib, arXiv:2207.12274) recommend over NGBoost precisely because coverage tracks the nominal level regardless of the K-distribution's shape. DECISION: ship the REAL GBM as the point predictor and CQR (or simple split-conformal absolute-residual when quantile heads aren't available) for the interval. ### Conformal / calibrated uncertainty for the K-Score prediction (split-conformal + locally-weighted/CQR intervals, Mondrian per-basis grouping, ACI online drift correction, and conformal selective prediction / abstention) replacing the confidence-scaled band in quality-predictor.js — _Autopilot / Autonomous Agent — quality predictor + compile simulator + cost optimizer + auto-deploy guardrail_  (high, 4d)

SPLIT (INDUCTIVE) CONFORMAL — REGRESSION, exact algorithm (Lei et al. 2018; Angelopoulos & Bates 2021): we already have a point predictor f_hat(x) = predictKScore over a feature vector x and a growing pool of (features -> observed K) labelled rows. (1) Hold out a CALIBRATION set of n rows DISJOINT from anything used to fit f_hat. (2) For each calibration row compute the nonconformity score s_i = |y_i - f_hat(x_i)| (absolute residual of observed K vs predicted K). (3) Sort the scores; take the conformal quantile q_hat = s_(b) where b = ceil((n+1)*(1-alpha)) (the b-th smallest score). (4) For a NEW vector x return the interval C(x) = [f_hat(x) - q_hat, f_hat(x) + q_hat], clamped to [0,1]. THEOREM (exchangeability): 1-alpha <= P(Y_new in C(x_new)) <= 1-alpha + 1/(n+1). This is a finite-sample marginal coverage guarantee with NO distributional assumption on K.   SMALL-n GUARD: if b > n (equivalently n < 1/alpha - 1, e.g. n < 19 for alpha=0.10, n < 99 for alpha=0.01) the quantile is undefined and the only valid interval is [0,1] (infinite / unbounded). The module MUST detect this and either return the trivial [0,1] interval with basis:'conformal_undercalibrated' OR fall back to the existing heuristic band — never silently return a too-tight conformal interval. LOCALLY-WEIGHTED (ADAPTIVE) VARIANT (Lei et al. 'split-localized'; SLCP): fixed split-conformal intervals are CONSTANT width for every x, which is wrong for heteroskedastic K (cold-start vectors are far noisier than Trinity-like ones). ### Data-scaling-law / Chinchilla-style data-budget model: fit a parametric law of K-Score (or fine-tune loss) vs n_pairs per task/namespace using the Rectified Scaling Law form (Lin et al. ICLR-track 2024, arXiv:2402.02314), then differentiate it to get marginal-quality-per-row, a budget-to-target estimate, and a principled stopping rule. — _Autopilot / Autonomous Agent (decision-intelligence layer): quality prediction sub-system that feeds the cost-optimizer ingest-more / gap-fill deltas, the compile-simulator compile/skip decision, and the lifecycle data-acquisition stopping rule._  (high, 4.5d)

**How it works.** GOAL: replace the hand-tuned, log-saturating, single-anchor pairs heuristic with a DATA-DRIVEN scaling curve that the autopilot refits as runs accumulate, so it can answer 'how many MORE labeled pairs to reach target K, and is the next batch worth the spend?' with a defensible number instead of a fixed +500/+150 prior.  MODEL FORM (core): use the RECTIFIED SCALING LAW (arXiv:2402.02314, the only published law that captures the small-data 'pre-power phase' that IS kolm's regime — sub-1000 pairs). In loss space: L_hat(D) = B / (D_l + D^beta) + E, where D = n_pairs (fine-tune data size), B>0 sets the initial-loss scale, D_l>=0 is the 'pre-learned data size' (how much downstream signal the base model already carries from pretraining — large D_l => student starts already-good, curve is flat-then-bends; this term is what makes the law fit at small n where a vanilla power law fails), beta>0 is the learning-difficulty exponent (smaller beta => harder task, later phase transition), E>=0 is the irreducible loss at infinite data. kolm tracks K-Score in [0,1] (ship gate 0.85, higher=better), NOT loss. We bridge with a monotone link L = -ln(K) (pseudo-loss: K=1 -> L=0, K->0 -> L->inf, strictly decreasing), fit the law in L-space (where the power-law math holds), then map predictions back via K_hat(D) = exp(-L_hat(D)). The K-domain curve is K_hat(D) = exp(-(B/(D_l + D^beta) + E)), which is monotone-increasing and saturating in D — exactly the shape the heuristic hand-approximates, but now fitted.  FITTING (Hitchhiker's-Guide recipe, arXiv:2410.11840 + scipy-cookbook robust-regression practice): we have a set of observed (n_pairs_i, K_i) points per (tenant, namespace) from kscore-timeseries + backfillFromRuns. (1) Convert each to (D_i, L_i=-ln(clamp(K_i, 1e-3, 1-1e-3))). (2) Minimize HUBER loss (delta=1e-3) of the LOG-residual r_i = ln(L_hat(D_i)) - ln(L_i) over theta=(ln B, ln D_l, beta, ln E) (log-parameterize the positive params so the optimizer is unconstrained and scale-robust — standard scaling-law practice). (3) Optimize with a small pure-JS BFGS / Nelder-Mead from a GRID of initializations (the loss surface is non-convex; literature uses a grid of seeds and keeps the best) — e.g. beta in {0.05,0.1,0.2,0.4}, D_l in {0, n_min, mean(D)}, B/E from a closed-form linear seed. ### Multi-armed bandit strategy selection — budgeted, non-stationary Thompson sampling (Normal-Gamma posterior with discounting + warm-start priors) over the 5 improvement strategies, with cost-aware reward/cost-ratio arm selection — _Autopilot / Autonomous Agent — the improvement-loop decision layer (src/cost-optimizer.js rankStrategies, invoked once per cron tick by src/autopilot-lifecycle.js tickAutopilotFull -> recommended strategy -> simulateCompile -> re-distill)_  (high, 5d)

**How it works.** PROBLEM FRAMING. Each autopilot tick must answer: which of the 5 improvement strategies (dedup / ingest-more / gap-fill / preference / evol) buys the most K-Score per dollar on THIS namespace's corpus, right now? Today this is a one-shot greedy argmax on a FIXED predicted ratio (predicted_delta_k / max(cost, eps)) using frozen characteristic-prior feature deltas — it never learns which strategy actually paid off, never explores under uncertainty, and clamps regressions to 0. This is exactly a BUDGETED, NON-STATIONARY multi-armed bandit: arms = strategies; reward = realized post-compile DeltaK; cost = teacher-token spend; reward distribution drifts because the corpus state changes every round (dedup stops paying once the corpus is clean; ingest-more saturates per the data-scaling law). Thompson sampling is the textbook policy: it has near-optimal Bayesian regret, adapts exploration to data without hand-tuned schedules, and self-corrects instantly when an arm degrades (Stitch Fix's stated reason for choosing TS in production).  MODEL. Reward per strategy a is the realized DeltaK (continuous, can be negative), so the conjugate model is a Normal-Gamma (Normal-Inverse-Gamma) per arm: unknown mean mu_a and unknown precision tau_a. Hyperparameters per arm: (mu0_a, kappa_a, alpha_a, beta_a). Posterior after observing rewards with discounted effective count n_eff, discounted mean xbar, discounted sum-of-squares S:   kappa_n  = kappa0 + n_eff   mu_n     = (kappa0*mu0 + n_eff*xbar) / kappa_n   alpha_n  = alpha0 + n_eff/2   beta_n   = beta0 + 0.5*S + 0.5*(kappa0*n_eff*(xbar-mu0)^2)/kappa_n SAMPLING one posterior draw of the mean reward for arm a:   tau ~ Gamma(shape=alpha_n, rate=beta_n)            (precision)   mu_tilde ~ Normal(mu_n, 1/(kappa_n * tau))          (mean reward draw) This is the standard NIG posterior-predictive-mean sampler (van den Burg; Honda & Takemura 2014).  NON-STATIONARITY (DISCOUNTED TS). Because reward distributions drift round-to-round, raw counts over-trust stale rewards. We apply a discount gamma in (0,1] (Qi et al. 2023, Discounted Thompson Sampling): when folding the j-th most recent reward r_j (j=0 newest), weight it by gamma^j. So n_eff = sum_j gamma^j, xbar = (sum_j gamma^j * r_j)/n_eff, S = sum_j gamma^j*(r_j - xbar)^2. Equivalently maintain decaying sufficient statistics incrementally: on each new reward, multiply the running (count, sum, sumsq) by gamma then add the new point. Recommended gamma ~ 0.9 (forget over ~10 effective rounds; tunable). gamma=1 recovers stationary TS. A sliding-window variant (Beta-SWTS / gamma-SWGTS, Fiandri et al. 2024) is the alternative; discounting is chosen here because it needs no window buffer and folds into a single incremental update.  BUDGETED / COST-AWARE SELECTION (Xia et al. IJCAI 2015, Budgeted Thompson Sampling). The autopilot optimizes DeltaK PER DOLLAR, not raw DeltaK, so we do not pick argmax of the sampled mean — we pick argmax of the sampled-reward-over-cost RATIO. Cost is also a (lower-variance, near-deterministic) posterior: model est_cost_usd per arm with a small Gamma posterior (or, since cost is well-modeled analytically by estimateBatchCost, use the analytic point cost as a near-deterministic draw and only Bayes-track the realized/estimated cost discrepancy). Per tick: for each arm draw mu_tilde_a (NIG) and cost_tilde_a; compute ratio_a = mu_tilde_a / max(cost_tilde_a, EPSILON); recommend argmax ratio_a among budget-feasible arms. dedup (cost=0) keeps its today-behavior of floating to the top whenever its sampled reward is positive — preserved exactly.  WARM-START / COLD-START PRIOR (Empirical-Bayes / Dynamic-Prior TS). The hard part of any bandit on a fresh namespace is the n=0 cold start, which is precisely kolm's honesty theme. Rather than an uninformative prior (which over-explores), we WARM-START each arm's prior mean mu0_a from the EXISTING frozen characteristic delta: project STRATEGIES[a].delta through predictKScore on the current features to get prior_delta_k_a (the exact number rankStrategies computes today), and set mu0_a = prior_delta_k_a with a WEAK prior strength kappa0 (e.g. kappa0=1, alpha0=1, beta0 tuned so prior variance ~ (0.1)^2). Result: at zero observed rewards the sampled mean concentrates near today's heuristic ranking (graceful degradation to current behavior — no regression risk on day 0), and as realized rewards accumulate the data overrides the prior. This is the bandit analog of the predictor's 'low capped confidence' contract.  DATA FLOW / PIPELINE PLACEMENT. ### Sequential / always-valid A/B testing (GAVI confidence sequence + mSPRT always-valid p-value) to replace the fixed-horizon Welch t-test gate that the autopilot peeks every cron tick — _Autopilot / Autonomous Agent_  (critical, 4d)

**How it works.** PROBLEM (statistics). A fixed-horizon test (Student/Welch t-test) controls the Type-I error rate alpha ONLY if you look at the data exactly once, at a pre-committed sample size. The kolm autopilot is the textbook anti-pattern: src/autopilot-daemon.js says the tick is 'cron-driven POST /v1/autopilot/tick every minute (or whatever cadence the operator picks)', and on each tick the A/B path (src/ab-router.js autoRollback -> stat-sig.gate) and the deploy path (src/autopilot-lifecycle.js _evaluateDeploy -> compareAndDecide) re-evaluate the SAME accumulating samples. Re-applying a fixed-horizon p<0.05 rule at every one of N peeks inflates the realized false-positive rate toward ~1 as N grows (Johari-Pekelis-Walsh show ~5x at modest N; with minutely cron over days, the gate is effectively guaranteed to fire a false 'promote' or false 'rollback' eventually). This is a correctness bug on the highest-blast-radius path (autonomous model promotion).  FIX (anytime-valid inference). An always-valid / anytime-valid procedure produces a test statistic that is valid at EVERY sample size simultaneously, so you may peek (and stop, promote, or roll back) at any time, any number of times, for any reason, while still controlling Type-I error at alpha. Two production-proven, closed-form, dependency-free families fit kolm's continuous K-Score-mean comparison:  (A) mSPRT (mixture Sequential Probability Ratio Test; Robbins 1970; Johari et al. 2019; the method Statsig ships). Define the mixture likelihood ratio of the true effect theta vs theta0=0 under a normal mixing prior N(0, tau^2) on theta. For two-sample normal data the mixture LR has a CLOSED FORM (verified verbatim from the canonical R implementation): given n paired observations, running means mu_a_n and mu_b_n, pooled variance V_n, and mixing variance tau_sq:     Lambda_n = sqrt( V_n / (V_n + n*tau_sq) ) * exp( ( n^2 * tau_sq * (mu_a_n - mu_b_n)^2 ) / ( 2 * V_n * (V_n + n*tau_sq) ) ) The ALWAYS-VALID p-value is the running infimum of the reciprocal: p_0 = 1; p_n = min(p_{n-1}, 1/Lambda_n). Ville's inequality guarantees P(inf_n p_n <= alpha) <= alpha under H0, so 'reject (B beats A) the first tick p_n < alpha' controls Type-I error under unlimited peeking. tau_sq is the only tuning knob; convention (Johari) is tau_sq on the order of the outcome variance / the squared minimum-effect-of-interest (Statsig sets it from the MDE; we anchor it to DEFAULT_MIN_EFFECT_SIZE^2 with a variance fallback). p-value monotone-decreasing means the decision is STICKY: once significant, stays significant -> peeking-safe by construction.  (B) GAVI confidence sequence (Howard-Ramdas-Sekhon-McAuliffe 2021 'Time-uniform, nonparametric, nonasymptotic confidence sequences', Annals of Statistics 49(2):1055-1080; the method Eppo ships as its default). A confidence sequence (CS) is a sequence of intervals (L_t, U_t) with the time-UNIFORM guarantee P(forall t>=1 : Delta in (L_t,U_t)) >= 1-alpha (vs a fixed-horizon CI which only covers at one t). For the difference of means Delta = mu_b - mu_a with cumulative sample size t and per-observation standard-error estimate sigma_hat_Delta, the closed-form half-width (verbatim from Eppo's production docs) is:     B = (1/sqrt(t)) * sqrt( (t + rho) * log( (t + rho) / (rho * (1-c)^2) ) )     interval = mu_hat_Delta +/- sigma_hat_Delta * B where c = 1-alpha (the coverage) and rho is the single tuning parameter set to minimize width near a target sample size N_tune:     rho = N_tune / ( log( log( e / (1-c)^2 ) ) - 2*log(1-c) ) Eppo uses N_tune=10000. DECISION via the CS: promote arm B iff the CS lower bound clears the effect threshold (L_t > min_effect_size); roll back iff the CS upper bound is below it (U_t < min_effect_size, i.e. B is significantly NOT better / worse); otherwise 'continue' (keep sampling). The CS also yields a calibrated, anytime-valid interval the dashboard can render directly, and it is variance-adaptive.  WHICH AND WHY. We implement BOTH as pure functions and expose mSPRT as the default decision (matches kolm's existing p-value-shaped gate() contract and Statsig's production choice for streaming/continuous data) while the GAVI CS supplies the anytime-valid INTERVAL (replacing stat-sig.js's fake 95%-CI that uses a normal 1.96 quantile with NO peeking correction) and is the natural feed for the conformal-uncertainty candidate (w921 sibling). ### DistiLLM-2: contrastive asymmetric SKL+SRKL distillation with on/off-policy curriculum — _Studio / Distill & Compile_  (critical, 9d)

**How it works.** DistiLLM-2 (Ko et al., ICML 2025 Oral, arXiv:2503.07067) is the current SOTA white-box LLM-distillation objective. Its core insight: prior KD applies the SAME loss to teacher-generated and student-generated data, which is suboptimal because the two data types want opposite skew. DistiLLM-2 uses a CONTRASTIVE asymmetric loss that simultaneously raises the likelihood of teacher responses and lowers the likelihood of student responses.  MATH (verified against the reference impl src/distillm_trainer.py):   Let p = teacher next-token dist, q_theta = student. Skewed-KL with skew alpha:     SKL^alpha(p||q) = KL( p || alpha*p + (1-alpha)*q )   -- the mixture lives in the SECOND slot so log-ratios never blow up; gives stable gradients (DistiLLM v1 result, arXiv:2402.03898).     SRKL^alpha(p||q) = KL( q || (1-alpha)*p + alpha*q )   -- reverse direction, student in numerator.   In the repo these are computed in log-space per token. For the TEACHER (chosen) responses:     mix = logsumexp( log(alpha)+tea_logps , log(1-alpha)+stu_logps )  ; tea_pos_kl = sum( exp(tea)*(tea - mix) )   == SKL.   For the STUDENT (rejected) responses (student side gets weight alpha, student logps DETACHED on the teacher-anchor term):     mix = logsumexp( log(1-alpha)+tea_logps , log(alpha)+stu_logps_detached ) ; ref_pos_kl = sum( exp(stu)*(stu - mix) ) == SRKL.   Final per-batch loss (loss_type=='distillm_v2'):     L = (2 - beta)*chosen_position_kl + beta*rejected_position_kl     where chosen_position_kl=SKL on teacher-generated y_t, rejected_position_kl=SRKL on student-generated y_s.  CURRICULUM (two schedules, both in the reference impl):   (1) Adaptive per-sample alpha (CALD / curriculum): alpha_t = clip( 1 - (1-base_alpha)*anchor / (sentence_logp_gap + eps), 1e-2, base_alpha ). Hard samples (large teacher-student gap) get larger skew, easy samples smaller. base_alpha default 0.1.   (2) Gradual beta ramp: beta = 1.0 + 0.5*min(1, 2*global_step/max_steps). Starts emphasizing the teacher term (SKL on chosen), ramps the student-correction term (SRKL on rejected) as the student improves.  DATA FLOW (off-policy + on-policy contrastive pairs): DistiLLM-2 reuses the trl DPO data shape: each row = {prompt, chosen=teacher_generated_response, rejected=student_generated_response}. The student responses are produced ON-POLICY by sampling from the current student (vLLM, temperature 0.8) once before/per stage, then reformatted into chosen/rejected pairs (generate/generate_vllm.py + generate/reformat.py). The trainer needs BOTH a trainable student AND a frozen teacher resident to compute teacher logits over both response branches (the teacher is the DPO 'ref_model').  PIPELINE PLACEMENT in kolm: this is the DISTILL stage objective, slotting where workers/distill/distill.mjs --mode=full hands off to a python trainer, and is the substantive upgrade to the apps/trainer/distill.py logit-KD engine. It sits AFTER CURATE/AUGMENT produce the prompt set + teacher responses, and AFTER (or interleaved with) an on-policy student-generation step. Its output (LoRA adapter + run-meta loss receipt) feeds COMPILE -> K-score gate -> .kolm artifact.
