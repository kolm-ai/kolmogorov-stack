# KOLM W921 — 100x Demo Spec (/demo-live)

**Target experience.** A first-time visitor lands on /demo-live and watches what is unmistakably a recording of a real terminal session: the kolm loop run end-to-end on real hardware, paced with human-imperfect timing. Four beats, one idea each - 1. Capture, 2. Compile (distill->quantize INT4), 3. Verify, 4. Run. Every visible value is real and sourced: the model is Qwen2.5 distilled from the real Claude 3.5 Sonnet + GPT-4o + DeepSeek-R1-32B council; the INT4 numbers are the X04-blocking measurements (61.0 GB -> 17.9 GB in 125.3s, 11.5 tok/s on an RTX 5090, named as such, no M3 Max sleight of hand); the captures are realistically messy support tickets with a real shape, PII-redacted on screen. The climax is proof-as-performance: the demo prints a REAL receipt CID and the /verify URL, then the viewer clicks 'Verify this receipt' and watches the page recompute the canonical hash and check the Ed25519 signature IN THE BROWSER against the public key embedded in the receipt - no login, no network, no trust required. The exact commands shown are the copy-pasteable quickstart that actually runs. One primary CTA at the end. The whole thing is keyboard-operable, screen-reader-narrated via the caption track, falls back to a static poster under prefers-reduced-motion, and presents as one focused pane per beat on mobile. It feels finished, reviewed, and impossible to fake.

**Data strategy.** Replace the invented CHAPTERS data source with a CAPTURED, checked-in timeline produced by an actual kolm run, and make the climax verifiable client-side with zero network dependence. Concretely:  (1) GROUND TRUTH FILES that already exist and are X04-blocking: public/benchmarks/sota-quantize-matrix.json (DeepSeek-R1-32B 61.03 GB BF16 -> 17.9 GB INT4 in 125.26s, 11.53 tok/s, 19.22 GB VRAM, 13.3s load on RTX 5090, CUDA 12.8, torch 2.11.0+cu128, bitsandbytes 0.49.2 NF4+double; Qwen2.5-7B 14.2->5.18 GB in 29.2s @ 24.5 tok/s; Qwen2.5-3B 5.76->1.92 GB in 13.85s; Qwen2.5-0.5B 0.93->0.44 GB in 11.59s) and public/benchmarks/trinity-500-benchmark.json (council = Claude 3.5 Sonnet + GPT-4o + DeepSeek-R1-32B distilled to Qwen2.5-7B via LoRA r=16 alpha=32, 410 pairs, 79.18s, n=57 holdout: 96.5% asks-1Q, 100% no-inventions, 100% judge-clarify, 100% judge-on-policy, 1.24s mean latency, 210 mean chars - beats base-qwen, claude-haiku-4-5, ties gpt-4o-mini at fewer chars) and public/benchmarks/wave887-wrapper-prod-benchmark.json (gateway mean 2478ms, +703ms / +40% overhead vs direct, $0.307 vs $0.301 per 1k, 10/10 receipts, example receipt rcpt_01KYC1ZV98HBEHW0NFC5DB signed by key c2e942e24061e9314790d73ce7b3f351). EVERY headline number in the new demo MUST be read from one of these files (or its on-disk source) and MUST match an x04-claim-fixtures.json row. No literal that isn't sourced.  (2) CAPTURE PIPELINE (new, under scripts/, the only product-code file this spec authorizes touching besides the HTML): scripts/capture-demo-timeline.mjs runs the REAL CLI (cli/kolm.js) against committed fixtures and serializes a timeline. Steps it captures: `kolm capture` / gateway proxy over a fixtures file of realistically-messy support tickets (data/demo/support-tickets.jsonl - committed, PII-redacted shape) -> real capture rows with real cap_ ids and real per-call token/cost; `kolm distill` / `kolm compile` of that namespace against the council teachers -> real step/loss lines and a real compile receipt; `kolm quantize ... int4` -> the real sota-quantize-matrix figures for the chosen model; `kolm serve` + one real `/v1/chat/completions` reply -> real tok/s + latency for that model on the capturing machine; `kolm receipts verify --offline <receipt>` -> the real Ed25519 verification transcript. The script writes public/demo-live-timeline.json: an array of beats, each with {label, events:[{at, type, text|html, klass}]} in the SAME event vocabulary the existing engine already understands (type/out/page/progress/receipt/capture/prompt/enter/subtitle/end), PLUS the real artifacts (a full signed receipt blob, the captures, the metrics). The HTML's CHAPTERS array is deleted and replaced by `fetch('/demo-live-timeline.json')` at boot (with the last-captured timeline inlined as a fallback so the page never blanks).  (3) THE VERIFIABLE CLIMAX (no network, no trust): commit the FULL real signed receipt JSON (a complete kolm-audit-1 receipt with its signature_ed25519 block, captured from the run) into public/demo-live-timeline.json. In the Verify beat, the page does what src/gateway-receipt.js verifyReceipt does, but in the BROWSER: rebuild the canonical payload = JSON.stringify over receipt-schema.js ALL_FIELDS order with signature_ed25519 stripped (the exact 20-field order: schema, receipt_id, timestamp, namespace_id, route_decision, provider, model, artifact_id, confidence, fallback_reason, input_hash, output_hash, capture_eligible, capture_id, redaction_applied, input_tokens, output_tokens, cost_usd, signing_key_id, verify_url), then call crypto.subtle.verify('Ed25519', publicKey, signature, canonicalBytes) using the public_key embedded in the receipt's signature_ed25519 block. Because the public key travels with the receipt, this verifies with no secret and no server. The viewer clicks 'Verify this receipt' and watches the canonical hash recompute and the signature check go green in front of them - proof-as-performance. The receipt's verify_url (https://kolm.ai/v1/verify/<id>) is shown and clickable as a PROGRESSIVE ENHANCEMENT: if prod still has the row it resolves; if prod's DB was wiped (the known Railway constraint - signing_key currently 'missing' and both candidate receipts return receipt_not_found today) the in-browser check still passes, so the climax can NEVER 404. This is the single most important design decision in the spec: the proof is self-contained in the committed receipt, not dependent on live prod state.  (4) THE REAL RECEIPT ID: use the receipt that the capture pipeline actually produces and commits. Do NOT reuse rcpt_compile_71F2A6 (invented) or rcpt_01KYC1ZVTGDCW3FX06JQSC (a verify.html placeholder that does not resolve). If a fresh capture is not run before ship, fall back to rcpt_01KYC1ZV98HBEHW0NFC5DB which is the X04-blocking example_receipt_id in wave887-wrapper-prod-benchmark.json (signing key c2e942e24061e9314790d73ce7b3f351) - but ONLY if its full signed blob is committed so in-browser verification works; the bare ID alone is not enough.  (5) NO NUMBER WITHOUT A SOURCE: add a release-verify gate (extend scripts/x04-claim-verify.cjs coverage or a sibling scripts/demo-claim-verify.cjs) that walks public/demo-live-timeline.json and asserts every numeric/metric literal it emits matches an x04-claim-fixtures.json row or a benchmarks/*.json field. The demo can then never out-claim a measurement. Reconcile or remove the fabricated arithmetic (DEMO-7) by deriving spend/cost from the captured token counts, not by typing a round number.

## Diagnosis (what is canned today)
- The whole timeline is fabricated: CHAPTERS (demo-live.html:288-439), PAGES (510-521), and the three receipt builders (442-507) are hand-authored string literals fired on a requestAnimationFrame timer. No command runs, no API is hit, no receipt resolves. This is the exact DOM-manipulation-theatre / faked-terminal anti-pattern the W921 best-in-class research names as the #1 canned tell for dev tools.
- DEMO-1 climax is a dead end: the Verify chapter (356-382) invites 'audit it - no login required' against kolm.ai/verify/rcpt_compile_71F2A6, but that ID exists ONLY in demo-live.html and resolves nowhere. The one moment that should prove authenticity points at a 404.
- DEMO-2 fabricated model: claude-3.7-haiku is used 7x as the teacher (331,351,367,454,500,514,517,520) and appears nowhere else in the repo. Anthropic never shipped a 3.7 Haiku. The real council teachers are Claude 3.5 Sonnet + GPT-4o + DeepSeek-R1-32B (trinity-500-benchmark.json method.scheme).
- DEMO-3 dead install line: 'curl -fsSL kolm.ai/get | sh' (433-435, PAGES.home 511). There is no /get route in vercel.json and no public/get file. The one line a viewer might copy 404s. Real install is npm i -g @kolm/cli or pip-from-source.
- DEMO-4 mis-attributed headline number: '11.5 tok/s on M3 Max' (386-399, end-card 244). 11.5 tok/s is REAL but it is DeepSeek-R1-32B INT4 on an RTX 5090 (sota-quantize-matrix.json row 32B; x04-claim-fixtures.json id sota-r1-32b-inference-tok-per-sec, blocking). A 1.5B Q4_K_M model on an M3 Max would run at hundreds of tok/s, not 11.5. The number is lifted from a different machine+model and re-skinned, violating the X04 'every number traces to a measurement' gate.
- DEMO-5 every terminal line and browser page is a scripted literal: 'gateway listening on :7421', '19 calls / 19 receipts / $0.022', 'distill done in 78s', 'step 56/56 loss=0.93 perplexity=2.54', 'k-score 0.91 PASS', 'loaded 0.9s / vram 1.9 GB / latency 217ms'. The browser pane is fake chrome swapping static HTML.
- DEMO-6 invented IDs/hashes throughout: cmp_72a9, sha256:d4ce91...8f0a (truncated, never resolvable), ed25519:7f3a8e4b... (a literal, not a real signing output), chain prev 0x91ce4f8a3b2e7d1c, order numbers #38214/#38421/etc with placeholder UPS tracking 1Z999 AA10 1234 5678 90. gpu_hours 0.0217 on RTX 5090 silently contradicts the 'M3 Max serve' claim - the demo mixes two machines without saying so.
- DEMO-7 internally inconsistent arithmetic: types 'for i in {1..18}' + 1 = 19 captures, but only 8 rows are scripted and renderCaptures caps at 8 (575) so the dashboard never shows 19; captures page says 19 total / 14 approved / 5 pending (515) while Compile says 'approving 19 captures' (330); end-card spend $0.07 (242) vs itemized 0.0478+0.0218=0.0696 (372) double-counts the $0.022 capture spend as both 'captures spent' and 'gpu compute'.
- DEMO-8 engine is shallower than its comments claim: typeText (620-637) drops remaining characters permanently if paused mid-type (closure exits, never resumes); header comments (280-286) document 'wait' and 'browserPatch' event types never implemented in applyEvent (677-689); 'human speed' typing is a fixed uniform 26 cps (TYPING_CPS) with zero variance - the uniform-typing canned tell.
- DEMO-9 recorder double-toggles play: demo auto-plays at 500ms (795) AND record-demo-w905.mjs clicks #playBtn (28-29) which is a toggle (765-767), so a late networkidle click PAUSES the demo and records ~90s of a frozen timeline. No play-state readiness check.
- DEMO-10 recorder targets live prod (record-demo-w905.mjs:8 default https://kolm.ai/demo-live), so the asset depends on whatever is deployed and on network at record time - the opposite of reproducible-by-construction; RECORD_MS 90000 vs a 70s timeline records 20s of dead tail.
- DEMO-12 'measured' values are wired to UI events, not computed: setProgress hardcodes the k-score label to '0.91 (gate 0.85)' when the distill bar hits 100% (596-599); hallu_rate 0.00 on 6 holdout prompts is a literal with a sample too small to credibly assert 0.00.
- A11Y: scrub role=slider never updates aria-valuenow / has no aria-valuetext / no :focus-visible (A11Y-1); end-card role=dialog has no focus move, no trap, no Esc, underlying stage not inert (A11Y-2); chapter chips are non-semantic spans with no role=button/tabindex/keydown/aria-current (A11Y-3); auto-plays on load even for reduced-motion users, with no static poster fallback (A11Y-4); terminal aria-live=polite floods the screen reader with ~80 fabricated lines while the actual narration (subtitle) is NOT in a live region (A11Y-5).

- ROOT CAUSE: the engine's data source is invented. Polishing the animation only makes the fakery more convincing. The fix is to replace the data source - invented -> captured - while reusing the cinematic engine the user explicitly wants kept.

## Benchmarks to steal
